# Release Notes v1.5.1

- **Дата:** 2026-05-28
- **Тип релиза:** patch

## Кратко

Исправлена доставка push-уведомлений на несколько устройств одного клиента. Теперь вход с десктопа больше не затирает подписку iPhone — все устройства сосуществуют в хранилище и получают уведомления независимо.

## Что нового

### Исправления

- **Мультидевайс push (Worker, корень бага).** Cloudflare Worker хранил одну запись подписки на клиента (`sub:${clientId}`), поэтому второе устройство перезаписывало первое. Исправлено переходом на схему ключей `sub:${clientId}:${endpointHash}` — каждое устройство получает отдельный ключ в KV.
- **Адресная отправка на все устройства.** `/send` с `targetClientId` теперь перебирает все подписки клиента через `list({ prefix: 'sub:${clientId}:' })` с пагинацией. Счётчик `sent` достоверно отражает число устройств, получивших push.
- **Cron-напоминания на все устройства.** `handleCron` использует `list({ prefix: 'sub:${clientId}:' })` вместо точечного `get`. Клиент с записью на завтра получает напоминание на каждое устройство.
- **Отписка конкретного устройства.** `handleUnsubscribe` теперь требует `endpoint` в теле запроса и удаляет только запись этого устройства; другие подписки клиента не затрагиваются. Без `endpoint` → 400 `missing_fields` (безопасный отказ, ничего не удаляет).
- **Клиент передаёт endpoint при отписке.** `unsubscribePush` в `push.js` читает `sub.endpoint` до вызова `sub.unsubscribe()` и передаёт его в тело запроса.
- **Удалён небезопасный эндпоинт `GET /debug-auth`.** Эндпоинт выдавал фрагменты YCLIENTS-токена в ответе — удалён.
- **Точечная очистка по 410.** При ответе 410 (подписка недействительна) удаляется только ключ конкретного устройства; остальные подписки остаются.
- **Логирование хоста, не полного endpoint.** В логах Worker (`[send]`, `[cron]`) фиксируются хост и статус; полный URL endpoint в лог не попадает.

## Принятые поведения

1. **Cron шлёт на все устройства** — намеренно. Клиент с несколькими записями на завтра получает по напоминанию на запись на каждом устройстве. Это ожидаемое поведение.
2. **Деградация для старых подписок** (до переподписки). Если клиент не перелогинился после деплоя, его ключ в KV остаётся в старом формате `sub:${clientId}`. Адресная отправка (`targetClientId`) и cron по префиксу `sub:${clientId}:` не найдут эту запись. Broadcast (`/send` без `targetClientId`) подхватывает её через общий префикс `sub:`. Восстанавливается автоматически при следующем входе клиента (переподписка записывает ключ в новом формате).
3. **Порядок деплоя важен**: сначала Worker, затем PWA (см. раздел «Деплой»).

## Изменения публичного контракта

- **`/unsubscribe` (внутренний, клиент↔Worker):** тело меняется с `{ clientId }` на `{ clientId, endpoint }`. Без `endpoint` → 400 `missing_fields`. Клиент и Worker обновлены синхронно.
- **`/subscribe`, `/send` (внешне):** тело запроса и формат ответа не изменились.
- **Схема ключей KV `SUBSCRIPTIONS`:** `sub:${clientId}` → `sub:${clientId}:${endpointHash}`. Миграция не выполняется; старые ключи самоустаревают (переподписка при логине, точечная очистка по 410, допустим разовый ручной сброс неймспейса).

## Известные ограничения

- Пока клиент не перелогинился после деплоя, адресный push на него не попадёт (деградация к broadcast). Восстанавливается при следующем входе.
- YCLIENTS API-токен по-прежнему захардкожен в `prototype.html` — известный технический долг, в этот релиз не входит.

## Деплой

**Порядок обязателен:**

1. **Worker**: `cd cf-worker && npx wrangler deploy` (требует `CLOUDFLARE_API_TOKEN`). Сначала Worker — чтобы новая схема ключей и безопасный `/unsubscribe` были активны до обновления клиентов.
2. **PWA**: `git push origin main` → GitHub Pages CD публикует `docs/` (~1 мин). Новый SW (`v39`) обновит `push.js` у клиентов.
3. **Проверка на реальном устройстве**: вход одним клиентом с десктопа и iPhone; отправка push; push приходит на оба устройства; счётчик `sent` соответствует числу устройств. `wrangler tail` — видны хост+статус по каждой подписке.
