# ADR-015: Инбокс уведомлений на главном экране — персистентность истории push через IndexedDB (запись из SW, чтение из приложения), отдельный экран `s-inbox`, дедупликация, лимит, бейдж непрочитанных

- **Дата:** 2026-05-29
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-016.md` (колокольчик, инбокс, счётчик непрочитанных, очистка истории; блокирующих открытых вопросов нет; §8 №1 — архитектурная развилка персистентности, закрывается этим ADR; §8 №2,4–9,11 — значения по умолчанию, фиксируются этим ADR)
- **Связанные ADR:** `adrs/adr-006.md` (мультидевайс push-подписки `sub:${clientId}:${endpointHash}`, `handleSend`/`handleCron`, KV `SUBSCRIPTIONS`, криптография Web Push — **доставка не пересматривается**, история строится как клиентская надстройка над приходящими push); `adrs/adr-005.md` (PWA install, оффлайн — почва для подписчиков; не затрагивается); `adrs/adr-009.md`, `adrs/adr-010.md` (регистрация экранов через `SCREENS` + `registerOnEnter` — переиспользуется модель навигации). Прямых конфликтов с ADR-001…ADR-014 не выявлено.

## Контекст

Studio PWA — устанавливаемое статическое приложение онлайн-записи (студия «Реснички, сестрички»), отдаётся из `docs/` на GitHub Pages; стек vanilla JS/HTML/CSS без рантайм-зависимостей. Push-инфраструктура существует и стабильна (ADR-006): клиентский `docs/modules/push.js`, бэкенд — Cloudflare Worker (`cf-worker/src/index.js`, KV `SUBSCRIPTIONS`), `handleSend` (админские рассылки) и `handleCron` (напоминания о визитах).

Подтверждено кодом: приходящие push **нигде не сохраняются**. SW-обработчик `push` (`docs/sw.js:158-170`) формирует системное уведомление ОС (`showNotification(title, opts)`) из payload `{title, body, icon?}` и на этом завершается; `notificationclick` (`docs/sw.js:172-182`) при тапе фокусирует/открывает приложение. Внутри приложения есть тост `#pushNotif` (`docs/prototype.html:25`), тап по которому (`handlePushTap()`, `docs/modules/notifications.js:11-14`) ведёт в `s-history` — это «История записей» (визитов), не уведомлений. Отдельного экрана истории уведомлений нет. Состояние приложения хранится в `localStorage` (`docs/modules/storage.js`, `docs/modules/state.js`).

BA-016 требует «инбокс» на главном экране `s-home`: колокольчик с бейджем непрочитанных, список **всех** полученных уведомлений (включая пришедшие при закрытом приложении), пометка «прочитано» при открытии инбокса, очистка истории с подтверждением, пустое состояние, оффлайн-просмотр, дедупликация, лимит хранения, экранирование контента через `esc()`. Существующий поток push (системное уведомление, `notificationclick`, тост, подписка) не должен ломаться.

Силы, действующие на решение:

- **Ключевое ограничение: `localStorage` недоступен из Service Worker, а push прилетает именно в SW — в том числе при закрытом приложении.** Обработчик `push` (`docs/sw.js:158`) исполняется в контексте SW без доступа к `localStorage`/`window`. Полнота истории (ФТ-5/КП-4 — «все уведомления, включая закрытое приложение») достижима только хранилищем, доступным **из контекста SW**. BA отверг вариант «писать только то, что получено при открытом окне через `postMessage` в `localStorage`» как теряющий закрытое приложение (§8 №1в).
- **Доступные из SW механизмы персистентности.** В среде Service Worker доступны: IndexedDB (асинхронный, доступен и из окна, и из SW), Cache Storage (для ответов, не для произвольных JSON-записей), и серверная дочитка (хранить отправленное на Worker и читать историю). `localStorage` и `sessionStorage` — недоступны.
- **Минимальность и стек.** Vanilla JS/HTML/CSS, код в `docs/`, **без новых рантайм-npm-зависимостей** (НФТ-1). Изменения строго в пределах объёма: колокольчик/инбокс/счётчик/история (НФТ-5). Доставка push, схема подписок ADR-006, криптография — не трогаются (НФТ-8). YCLIENTS-токен не усугубляется.
- **PWA/оффлайн и `CACHE_VERSION`.** Оффлайн-просмотр обязателен (ФТ-9/КП-8). При изменении файлов `APP_SHELL` (включая `docs/sw.js`, новые JS-модули в shell, `prototype.html`, `style.css`) поднимается `CACHE_VERSION` (НФТ-2). Поскольку для записи истории правится SW-обработчик `push`, подъём `CACHE_VERSION` обязателен.
- **Безопасность/XSS.** Контент уведомления (`title`/`body`) приходит из admin push (внешний источник) — при рендере в инбоксе только через `esc()` (`docs/modules/utils.js:3`); вставка непроверенного HTML недопустима (НФТ-3/КП-9).
- **Дедупликация без серверного id.** Payload push сейчас `{title, body, icon?}` — устойчивого сквозного `id` в нём нет. Дубль возможен при повторной доставке. Нужен ключ дедупликации (ФТ-11/КП-10).
- **Навигационная модель проекта.** Экраны регистрируются в `SCREENS` (`docs/modules/constants.js:3-9`) и через `registerOnEnter` в `docs/app.js`; переход — `go(id)` / `go(id,'tab')`, возврат — `back()` (`docs/modules/navigation.js`). Это устоявшийся паттерн (ADR-009, ADR-010).

Решение охватывает развилки: **(A)** механизм персистентности истории (доступный из SW); **(B)** структура записи уведомления и ключ дедупликации; **(C)** лимит хранения и автовытеснение; **(D)** инбокс — отдельный экран vs накладная панель; **(E)** обновление бейджа непрочитанных (в т. ч. при push в открытом окне); **(F)** влияние на `CACHE_VERSION`.

## Рассматриваемые варианты

> Значения по умолчанию, принятые BA и не являющиеся развилками реализации: уведомление = push-сообщения `title`/`body`(+`icon?`), элемент информационный без deep-link (§8 №2); бейдж только визуальный, без Badging API/`setAppBadge` (§8 №3); «прочитано» — всё целиком при открытии инбокса (§8 №4); «очистить» — удаляет все записи с подтверждением, удаление одного вне рамок (§8 №5); колокольчик только на `s-home` (§8 №10); вид бейджа — кружок справа внизу, «9+» при больших числах (§8 №11). Эти решения принимаются как есть и не порождают альтернатив ниже.

### Вопрос A. Механизм персистентности истории, доступный из контекста Service Worker

#### Вариант A1 (выбран): IndexedDB — запись в SW-обработчике `push`, чтение/мутации из приложения; общий тонкий модуль доступа

Создаётся тонкий модуль доступа к IndexedDB (одна БД, одно объектное хранилище уведомлений), пригодный к импорту **и из окна, и из Service Worker** (ES-модуль без зависимостей от `window`/`localStorage`/DOM). В SW-обработчик `push` (`docs/sw.js:158-170`) добавляется запись уведомления в IndexedDB **до/параллельно** `showNotification` (через `event.waitUntil`), так что уведомление сохраняется даже при закрытом приложении. Приложение читает историю из IndexedDB при открытии инбокса и для подсчёта бейджа, помечает прочитанным, очищает.

- **Плюсы:**
  - IndexedDB доступен из контекста SW (`self.indexedDB`) — единственный из стандартных JSON-хранилищ, покрывающий «push при закрытом приложении» (прямо устраняет корень — ФТ-5/КП-4).
  - Доступен и из окна, и из SW → одно хранилище, единый источник истины; чтение/мутации из приложения тривиальны.
  - Без рантайк-зависимостей (НФТ-1) — IndexedDB встроен в браузер; модуль доступа пишется на голом API.
  - Оффлайн-просмотр бесплатен: чтение из локального IndexedDB не требует сети (ФТ-9/КП-8).
  - Не трогает доставку push, схему подписок ADR-006, криптографию (НФТ-8) — это чисто клиентская надстройка.
- **Минусы:**
  - IndexedDB-API многословный (open/onupgradeneeded/transaction) — но инкапсулируется в одном тонком модуле; разовая стоимость.
  - Запись в SW добавляет асинхронную работу в `push`-обработчик — оборачивается в `event.waitUntil`, чтобы SW не завершился до записи; при ошибке записи системное уведомление всё равно показывается (история не должна ломать показ).
  - Модуль доступа должен быть импортируемым из SW (нет `window`/`document`/`localStorage`) — ограничение учитывается при написании (только `self`/IndexedDB API).
- **Последствия:** новый модуль `docs/modules/inboxStore.js` (open БД, add с дедупом+лимитом, getAll, markAllRead/getUnreadCount, clear); правка `docs/sw.js` (`importScripts`/динамический импорт логики записи в `push` + `CACHE_VERSION`); новый модуль UI инбокса/бейджа; правка `prototype.html` (колокольчик в шапке `s-home` + экран `s-inbox`), `style.css`, `constants.js` (`SCREENS`), `app.js` (`registerOnEnter`).

#### Вариант A2 (отвергнут): только `postMessage` SW→клиент + `localStorage`

SW при `push` шлёт сообщение в открытые окна (`clients.matchAll` + `postMessage`); клиент пишет в `localStorage`.

- **Минусы:** при **закрытом приложении** открытых окон нет — `postMessage` некому доставить, уведомление теряется. Прямо нарушает ФТ-5/КП-4 (полнота истории — жёсткий критерий готовности BA). **Отвергнут BA в §8 №1в** и подтверждается здесь.
- **Последствия:** не удовлетворяет обязательному критерию. Отвергнут.

#### Вариант A3 (отвергнут): серверная история на Cloudflare Worker + дочитка

Хранить отправленные уведомления в KV/хранилище Worker, приложение дочитывает историю по запросу.

- **Минусы:** требует серверной разработки (новый эндпоинт/хранилище, схема, фильтрация по клиенту, согласование с ADR-006) — выходит за «минимальность» (НФТ-5) и за объём (BA раздел 2 явно выносит «серверную историю/синхронизацию устройств» **вне рамок** как продуктовую функцию); добавляет сетевую зависимость, ухудшая оффлайн без локального кэша; `handleSend`/`handleCron`/cron-напоминания пришлось бы расширять записью истории — пересмотр доставки, чего BA запрещает (НФТ-8). Несоразмерно ради локального инбокса.
- **Последствия:** широкая зона изменений на бэкенде, против объёма. Отвергнут. (BA §8 №1 допускает его как решение архитектора, но при наличии IndexedDB он избыточен.)

#### Вариант A4 (отвергнут): Cache Storage как хранилище JSON-записей

Складывать синтетические `Response`-объекты с JSON в `RUNTIME_CACHE`.

- **Минусы:** Cache Storage спроектирован под HTTP-ответы, не под список структурированных записей; запрос/сортировка/дедуп/лимит реализуются вручную поверх синтетических ключей — хрупко и неестественно; смешивает кэш ресурсов и данные истории (риск, что `activate`-очистка кэшей по `CACHE_VERSION` снесёт историю). IndexedDB — штатный инструмент для этой задачи.
- **Последствия:** хрупкость, риск потери истории при смене `CACHE_VERSION`. Отвергнут.

### Вопрос B. Структура записи уведомления и ключ дедупликации

#### Вариант B1 (выбран): запись `{id, title, body, icon?, ts, read}`; `id` = детерминированный ключ из `ts`-минуты + контента; дедуп по `id`

Каждое уведомление сохраняется как `{ id, title, body, icon, ts, read }`, где `ts` — время получения (`Date.now()` в момент `push`), `read` — флаг прочитанности (по умолчанию `false`), `id` — устойчивый ключ дедупликации. Поскольку устойчивого сквозного id в payload нет (`{title, body, icon?}` — `docs/sw.js:159-167`), `id` строится детерминированно из **нормализованного контента + огрублённого времени** (например, `title|body` + временное «ведро», скажем минута получения), чтобы повторная доставка того же уведомления в близкое время не создавала дубль (ФТ-11/КП-10), но два **разных** уведомления с одинаковым текстом в разное время сохранялись как разные. Дедуп — на уровне `add`: при совпадении `id` запись не дублируется (перезапись/пропуск). `IDBObjectStore` с `keyPath: 'id'`.

- **Плюсы:** не требует менять payload push и Worker (доставка ADR-006 не трогается — НФТ-8); дедуп детерминированный, без сервера; структура минимальна и покрывает рендер (заголовок/текст/время — ФТ-4) и логику прочитанности (ФТ-6).
- **Минусы:** два **идентичных** уведомления, отправленных в пределах одного временного «ведра», схлопнутся в одно — приемлемый компромисс (повторная доставка одного push — частый кейс; намеренная отправка двух одинаковых в одну минуту — редкий). Размер «ведра» (минута) — деталь реализации в этих рамках.
- **Последствия:** структура записи и функция вычисления `id` — в `docs/modules/inboxStore.js`; используется и при записи из SW, и (при желании) при чтении.

#### Вариант B2 (отвергнут): сквозной `id` в payload push (расширить Worker)

Добавить уникальный `id`/`tag` в payload на стороне `handleSend`/`handleCron`.

- **Минусы:** меняет формат payload и затрагивает Worker (`handleSend`/`handleCron`) — пересмотр доставки, который BA запрещает (НФТ-8, раздел 2 «состав/семантика/доставка push не пересматриваются»). Несоразмерно: клиентский детерминированный ключ (B1) решает дедуп без серверных правок.
- **Последствия:** правки бэкенда вне объёма. Отвергнут.

#### Вариант B3 (отвергнут): автоинкремент-ключ без дедупликации

`autoIncrement` keyPath, дубли не выявляются.

- **Минусы:** повторная доставка одного push создаёт дубль в инбоксе — прямо нарушает ФТ-11/КП-10.
- **Последствия:** не удовлетворяет критерию дедупликации. Отвергнут.

### Вопрос C. Лимит хранения и автовытеснение

#### Вариант C1 (выбран): хранить последние 50 записей, при превышении вытеснять самые старые по `ts`

После добавления новой записи, если число записей превышает лимит (BA §8 №6 — предложение 50), удаляются самые старые по `ts` сверх лимита. Реализуется внутри `add` модуля `inboxStore.js` (после вставки — подсчёт/обрезка) либо по курсору, отсортированному по `ts`.

- **Плюсы:** хранилище не растёт неограниченно (НФТ-4/КП-14); 50 — разумный предел для клиента студии; обрезка детерминирована (по времени получения).
- **Минусы:** очень старые уведомления исчезают — соответствует ожиданиям BA («последние N»); точное N может скорректировать заказчик (вынесено как настраиваемая константа).
- **Последствия:** константа лимита + логика обрезки в `inboxStore.js`.

#### Вариант C2 (отвергнут): без лимита / по времени (TTL)

Хранить всё либо удалять старше N дней.

- **Минусы:** «без лимита» нарушает НФТ-4 (неограниченный рост); TTL требует периодической чистки и менее предсказуем, чем «последние N» (BA дал значение по умолчанию именно «последние N»). Избыточно.
- **Последствия:** против НФТ-4 / сложнее. Отвергнут.

### Вопрос D. Инбокс — отдельный экран vs накладная панель

#### Вариант D1 (выбран): отдельный экран `s-inbox` в существующей навигационной модели

Инбокс — новый экран `s-inbox`, зарегистрированный в `SCREENS` (`docs/modules/constants.js`) и через `registerOnEnter('s-inbox', …)` в `docs/app.js`; открытие — `go('s-inbox')` по тапу на колокольчик, возврат — `back()` (стрелка/жест). Рендер списка из IndexedDB на входе; при входе все непрочитанные помечаются прочитанными и бейдж обнуляется.

- **Плюсы:** консистентно с остальными экранами проекта (ADR-009, ADR-010), переиспользует анимацию навигации, `back()`, `SCREENS`-регистрацию; «понятный возврат назад» (ФТ-3) — штатный `back()`; не вводит новый UI-примитив.
- **Минусы:** добавляется экран в `SCREENS` (требование ADR-009: иначе навигация/сброс сценариев его не учтут) — учтено.
- **Последствия:** блок `<div class="screen" id="s-inbox">` в `prototype.html`; запись в `SCREENS`; `registerOnEnter` в `app.js`.

#### Вариант D2 (отвергнут): накладная панель/модалка поверх `s-home`

- **Минусы:** новый паттерн оверлея, ручное управление показом/скрытием и историей назад вне модели `navHistory`; риск рассинхрона с системной «назад»; BA допускает оба, но отдельный экран дешевле и консистентнее. Не запрещён, но проигрывает D1 по консистентности.
- **Последствия:** дублирование навигационной логики. Отвергнут в пользу D1.

### Вопрос E. Обновление бейджа непрочитанных, в т. ч. при push в открытом окне

#### Вариант E1 (выбран): бейдж пересчитывается из IndexedDB на входе в `s-home`; SW уведомляет открытое окно через `postMessage` для немедленного обновления

Счётчик непрочитанных вычисляется из IndexedDB (`getUnreadCount`) при входе на `s-home` (в `registerOnEnter('s-home', …)`, `docs/app.js:53`) и при возврате из инбокса. Дополнительно: SW-обработчик `push` после записи в IndexedDB шлёт `postMessage` в открытые окна (`clients.matchAll` + `postMessage`), а клиент слушает `navigator.serviceWorker` `message` и обновляет бейдж немедленно, если приложение открыто. Это **дополнение** к персистентному источнику (IndexedDB), а не замена: при закрытом приложении `postMessage` некому доставить, но запись уже в IndexedDB, и бейдж появится при следующем входе на `s-home` (ФТ-5/КП-4 покрыт хранилищем, не сообщением).

- **Плюсы:** бейдж достоверен из единого источника (IndexedDB) — нет рассинхрона; немедленное обновление при открытом окне улучшает UX, но не является обязательным для полноты (полноту даёт IndexedDB); `postMessage` не нарушает доставку push.
- **Минусы:** небольшой объём кода для слушателя сообщений в приложении — приемлемо; при отсутствии поддержки `postMessage`/закрытом окне деградирует на пересчёт при входе (без потери данных).
- **Последствия:** функция обновления бейджа (читает `getUnreadCount`), вызов в `registerOnEnter('s-home')`; слушатель `message` в `app.js`/UI-модуле; `postMessage` в SW `push`-обработчике (в том же `event.waitUntil`).

#### Вариант E2 (отвергнут): бейдж только из `postMessage`, без пересчёта из IndexedDB

- **Минусы:** при закрытом приложении сообщение теряется — бейдж не покажет уведомления, пришедшие при закрытом окне (нарушает ожидание КП-4 «учтено в бейдже до открытия инбокса»). IndexedDB как источник обязателен.
- **Последствия:** недостоверный бейдж. Отвергнут.

### Вопрос F. Влияние на `CACHE_VERSION` и состав `APP_SHELL`

#### Вариант F1 (выбран): поднять `CACHE_VERSION` v59 → v60; добавить новые JS-модули в `APP_SHELL`

Правятся `docs/sw.js` (запись истории в `push` + `postMessage`), `docs/prototype.html` (колокольчик + экран `s-inbox`), `docs/style.css` (стили колокольчика/бейджа/инбокса), `docs/modules/constants.js` (`SCREENS`), `docs/app.js` (импорт/регистрация). Добавляются новые модули `docs/modules/inboxStore.js` и UI-модуль инбокса — они должны попасть в `APP_SHELL` (для оффлайн-доступности и кэширования). По правилу проекта при изменении файлов `APP_SHELL` (в т. ч. правке SW-обработчика `push`) поднимается `CACHE_VERSION`: `v59` → `v60`.

- **Плюсы:** соблюдено правило проекта (НФТ-2/КП-16); новые модули кэшируются и доступны оффлайн; клиенты получат новый `sw.js` с записью истории.
- **Минусы:** нет (разовая операция). Модуль доступа к IndexedDB, импортируемый из SW, не должен зависеть от `window` — учтено в Вопросе A1.
- **Последствия:** `CACHE_VERSION` v59 → v60; в `APP_SHELL` добавить `./modules/inboxStore.js` и UI-модуль инбокса. `version.json` обновляет мейнтейнер при релизе (вне объёма реализации фичи).

## Решение

- **Вопрос A — A1:** история push хранится в **IndexedDB**; запись — в SW-обработчике `push` (через `event.waitUntil`), чтение/мутации — из приложения; общий тонкий модуль `docs/modules/inboxStore.js`, импортируемый и из окна, и из SW, без зависимостей от `window`/`localStorage`/DOM (ФТ-5, ФТ-9, КП-4, КП-8).
- **Вопрос B — B1:** структура записи `{id, title, body, icon?, ts, read}`; `id` — детерминированный ключ из нормализованного контента + огрублённого времени получения; дедуп по `id` на уровне `add` (ФТ-11, КП-10).
- **Вопрос C — C1:** лимит — последние 50 записей (настраиваемая константа), автовытеснение самых старых по `ts` (НФТ-4, КП-14).
- **Вопрос D — D1:** инбокс — отдельный экран `s-inbox` в `SCREENS` + `registerOnEnter`; открытие `go('s-inbox')`, возврат `back()`; при входе непрочитанные помечаются прочитанными, бейдж обнуляется (ФТ-3, ФТ-6, КП-3, КП-5).
- **Вопрос E — E1:** бейдж непрочитанных пересчитывается из IndexedDB на входе в `s-home` и при возврате из инбокса; SW дополнительно шлёт `postMessage` для немедленного обновления при открытом окне (источник истины — IndexedDB) (ФТ-2, ФТ-6, КП-2, КП-4).
- **Вопрос F — F1:** `CACHE_VERSION` v59 → v60; новые модули `inboxStore.js` и UI-модуль инбокса добавляются в `APP_SHELL` (НФТ-2, КП-16).
- **Существующий поток push не меняется:** `showNotification` (`docs/sw.js:169`), `notificationclick` (`docs/sw.js:172-182`), тост `#pushNotif`, подписка `docs/modules/push.js` работают как прежде; запись истории — **дополнительный** шаг в `push`-обработчике, его сбой не должен ломать показ системного уведомления (ФТ-12, КП-11). Тексты на русском, через `esc()` (ФТ-10, ФТ-15, НФТ-3, КП-9, КП-15).

Отвергнуты: A2 (`postMessage`+`localStorage` — теряет закрытое приложение), A3 (серверная история — вне объёма, пересмотр доставки), A4 (Cache Storage под JSON — хрупко, риск потери при смене `CACHE_VERSION`), B2 (id в payload — пересмотр Worker/доставки), B3 (без дедупа — дубли), C2 (без лимита/TTL — против НФТ-4 / избыточно), D2 (накладная панель — менее консистентно), E2 (бейдж только из `postMessage` — недостоверен при закрытом окне).

### Границы решения

- Изменения **только** в пределах: новый локальный store истории (IndexedDB), запись из SW `push`, колокольчик на `s-home`, экран `s-inbox`, бейдж непрочитанных, очистка истории. Каталог услуг, выбор мастера, слоты, бронирование, профиль, лента, отзывы, история визитов `s-history` — **не рефакторятся** (НФТ-5).
- **Доставка push не пересматривается:** `handleSend`/`handleCron`, схема подписок `sub:${clientId}:${endpointHash}` (ADR-006), криптография Web Push, формат payload, тумблеры предпочтений (BA-007) — **не трогаются**. Worker (`cf-worker/`) в этом релизе **не изменяется**.
- Колокольчик — **только на `s-home`** (§8 №10). Deep-link/действия по тапу на конкретный элемент инбокса — **вне рамок** (§8 №2): элемент информационный. Удаление одного уведомления — вне рамок (§8 №5). Синхронизация с системным значком (Badging API/`setAppBadge`) — вне рамок (§8 №3).
- Тап по тосту `#pushNotif` (`handlePushTap`, `docs/modules/notifications.js:11-14`), сейчас ведущий в `s-history`, **допустимо** перенаправить в `s-inbox` (§8 №2 — необязательно). Решение реализации; не обязательное требование. Прочее поведение тоста и `s-history` — без изменений.
- YCLIENTS-токен не трогается (известный долг). Операции с веткой `backup` запрещены.

## Последствия

- **Положительные:**
  - Клиент получает единое место — инбокс — где видит **все** полученные push, включая пришедшие при закрытом приложении (IndexedDB пишется из SW).
  - Бейдж непрочитанных достоверен (источник — IndexedDB), обновляется немедленно при открытом окне (`postMessage`).
  - Оффлайн-просмотр истории бесплатен (чтение из локального IndexedDB).
  - Дедупликация и лимит хранения предотвращают дубли и неограниченный рост без серверной разработки.
  - Доставка push, подписки ADR-006, Worker, криптография — неизменны; история чисто клиентская надстройка.
  - Без новых рантайм-зависимостей; навигация переиспользует существующую модель (`SCREENS` + `registerOnEnter` + `back()`).
- **Отрицательные / компромиссы:**
  - Введение IndexedDB — новый класс хранилища в проекте (раньше только `localStorage`); инкапсулируется в одном тонком модуле, импортируемом из окна и SW.
  - Дедуп по «контент + огрублённое время» схлопнет два идентичных уведомления в пределах одного временного «ведра» — осознанный компромисс ради дедупликации повторной доставки без серверного id.
  - SW-обработчик `push` становится сложнее (запись + `postMessage`) — оборачивается в `event.waitUntil`; сбой записи не должен ломать показ системного уведомления.
  - `postMessage`-обновление бейджа — best-effort при открытом окне; при закрытом приложении бейдж появляется при следующем входе на `s-home` (полноту обеспечивает IndexedDB, не сообщение).
- **Технический долг:**
  - Deep-link/действия по тапу на элемент инбокса, удаление отдельного уведомления, синхронизация с системным значком приложения — отложены (вне объёма; при отдельном запросе заказчика).
  - Серверная история/синхронизация между устройствами — вне объёма; при необходимости — отдельный релиз поверх Worker (без пересмотра ADR-006).
  - Захардкоженный YCLIENTS-токен — известный долг, не усугубляется.

## Риски

- **Риск (СРЕДНИЙ): модуль доступа к IndexedDB, импортируемый из SW, случайно зависит от `window`/`localStorage`/DOM** → при импорте в SW падает. **Митигация:** `inboxStore.js` пишется только на `self`/IndexedDB-API, без обращений к `window`/`document`/`localStorage`; в SW логика записи подключается совместимым способом (динамический `import()` в обработчике или `importScripts`/инлайн логики записи в `sw.js`); разработчик проверяет, что SW регистрируется и пишет историю в DevTools → Application → IndexedDB.
- **Риск (СРЕДНИЙ): запись в IndexedDB из `push` не успевает до завершения SW** → уведомление не сохранилось. **Митигация:** запись оборачивается в `event.waitUntil(Promise.all([showNotification, writeHistory, postMessage]))`; ошибка записи перехватывается и **не** блокирует `showNotification` (показ системного уведомления приоритетен — ФТ-12/КП-11).
- **Риск (СРЕДНИЙ): дедуп-ключ схлопывает легитимно разные уведомления или, наоборот, плодит дубли** → потеря/дублирование записей. **Митигация:** ключ = нормализованный `title|body` + временное «ведро» (минута); протестировать повторную доставку (нет дубля — КП-10) и два разных уведомления в разное время (оба сохраняются).
- **Риск (НИЗКИЙ): старый кэш SW отдаёт прежний `sw.js`/разметку/CSS/JS** → история не пишется, колокольчик не появляется. **Митигация:** поднять `CACHE_VERSION` v59 → v60; новые модули в `APP_SHELL`; `controllerchange`-перезагрузка (`docs/app.js:325`) подхватит новый SW (НФТ-2, КП-16).
- **Риск (НИЗКИЙ): XSS через `title`/`body` push** → инъекция HTML в инбоксе. **Митигация:** рендер только через `esc()` (`docs/modules/utils.js:3`); никакого `innerHTML` с непроверенным контентом (НФТ-3, КП-9).
- **Риск (НИЗКИЙ): IndexedDB недоступен/заблокирован (приватный режим, квота)** → история не ведётся. **Митигация:** все обращения к store обёрнуты в try/catch; при недоступности инбокс показывает пустое состояние без ошибок (ФТ-13/КП-12), приложение не падает (НФТ-6).
- **Риск (НИЗКИЙ): колокольчик ломает шапку `s-home`/строку поиска** → регрессия вёрстки. **Митигация:** колокольчик размещается в свободном правом верхнем углу строки логотипа (`docs/prototype.html:38-44`), не задевая `#homeSearchBar` ниже (`:46-52`); проверить на узких экранах и тач (КП-1, ФТ-14).
