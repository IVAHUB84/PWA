# ADR-004: Отправка отзыва клиента через серверный прокси на Cloudflare Worker

- **Дата:** 2026-05-28
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-004.md`
- **Связанные ADR:** ADR-003 (дополняет, не отменяет; меняется только транспорт отправки `POST /comments`), ADR-002 (не конфликтует), ADR-001 (не конфликтует)

## Контекст

В релизе v1.3.0 по ADR-003 реализована отправка внутреннего отзыва клиента: `submitReview` (`docs/modules/review.js`) вызывает `postComment` (`docs/modules/api.js`), который делает `POST /comments/{company}` напрямую в YCLIENTS из браузера, переиспользуя `YC.post` и захардкоженные admin-токены (`YC.token` partner + `YC.userToken` user).

На практике этот запрос стабильно возвращает **401 Unauthorized**. Причина — архитектурная (BA-004, раздел 1):

- YCLIENTS `user_token` для записи отзыва от лица клиента получается только через сайт yclients.com (логин/пароль сотрудника) или мобильное приложение YPLACES (SMS клиенту). Из браузерного партнёрского API получить клиентский `user_token` нельзя.
- Статический `YC.userToken` в `api.js` принадлежит admin-аккаунту и либо протух, либо не имеет права `client_comments_add_access` на запись комментария от лица клиента.
- `POST /book_code/{company}` шлёт SMS для подтверждения брони, а не для авторизации пользователя — его код не принимается в `POST /user/auth` (см. исходное требование `requirements/req-client-auth-token-004.md`).

Открытые вопросы BA закрыты: единственный работающий подход — выполнять `POST /comments` **на серверной стороне** с валидными серверными токенами, а не из браузера клиента.

Силы, действующие на решение:

- **Бэкенд частично уже есть.** Cloudflare Worker (`cf-worker/src/index.js`, URL `https://studio-push.ivahub84.workers.dev`) уже задеплоен для Web Push. Он уже хранит YCLIENTS-токены в Cloudflare Secrets (`YC_TOKEN`, `YC_USER_TOKEN`, `YC_COMPANY`) и уже ходит в YCLIENTS из cron (`handleCron` → `GET /records`). То есть серверный канал к YCLIENTS с серверными токенами **существует** — нужно лишь добавить маршрут записи отзыва.
- **Токены не должны попадать в браузер.** НФТ-1/КП-3 требуют, чтобы YCLIENTS-токены оставались в Cloudflare Secrets и отсутствовали в коде PWA (`docs/`). Прямой `POST /comments` из браузера это нарушает (токены и так в `api.js`, но усугублять нельзя; целевое состояние — отзыв уходит без токенов в коде PWA).
- **Стек PWA неизменен.** Vanilla JS/HTML/CSS, статический хостинг из `docs/`, без рантайм-npm-зависимостей (НФТ-3). Worker — отдельный артефакт, деплоится своим конвейером (`wrangler deploy`), не входит в `APP_SHELL` PWA.
- **Fail-safe из ADR-003 обязателен сохранить.** При ошибке отправки отзыв не теряется «молча»: клиент видит понятное сообщение, поля не очищаются, можно повторить (ФТ-4, НФТ-5, КП-4). Меняется только транспорт, контракт исходов `postComment` (`{ok:true}`/`{ok:false}`) сохраняется.
- **Worker URL для клиента должен работать «из коробки».** Существующий push-модуль (`docs/modules/push.js`) берёт Worker URL из `localStorage` (`yc_worker_url`), задаваемого вручную в админ-панели. Для отзыва рядового клиента ручная настройка недопустима — URL должен быть доступен любому клиенту без действий.
- **CORS уже разрешён.** Worker отдаёт `Access-Control-Allow-Origin: *` и обрабатывает `OPTIONS` (НФТ-4 выполняется существующим кодом).

Решение охватывает три связанных вопроса: где выполнять `POST /comments` (транспорт), как PWA узнаёт адрес серверного endpoint-а, и какие токены использует Worker. Конкретные имена функций, маршрут, тело запроса и тексты — в спецификации релиза, не здесь.

## Рассматриваемые варианты

### Вопрос 1. Где выполнять `POST /comments` (транспорт отправки отзыва)

#### Вариант 1A (выбран): новый endpoint `POST /review` в существующем Cloudflare Worker — прокси в YCLIENTS

PWA шлёт данные отзыва (`rating`, `text`, `staff_id`, `record_id`) на новый маршрут Worker. Worker валидирует тело, делает `POST /comments/{YC_COMPANY}` в YCLIENTS с серверными токенами из Secrets и возвращает PWA различимый исход (2xx success / ошибка). Браузер токенов YCLIENTS для записи не получает.

- **Плюсы:**
  - Решает первопричину 401: запись идёт с серверными токенами, недоступными браузеру (ФТ-1, ФТ-2, НФТ-1, КП-3).
  - Переиспользует уже задеплоенный Worker, его Secrets (`YC_TOKEN`, `YC_USER_TOKEN`, `YC_COMPANY`) и CORS — минимум новой инфраструктуры (НФТ-3, НФТ-4).
  - Контракт исходов `postComment` сохраняется → fail-safe из ADR-003 и UI `submitReview` не меняются (ФТ-4, НФТ-5, КП-4).
  - Worker — только прокси, без хранилища отзывов (BA «Вне рамок»): отзыв сразу уходит в YCLIENTS и появляется на карточке мастера через уже существующее чтение `GET /comments` (ФТ-5, КП-2).
- **Минусы:**
  - Появляется сетевой хоп PWA → Worker → YCLIENTS (на практике пренебрежимая задержка, НФТ-2; Worker рядом с edge).
  - Контракт `POST /comments` YCLIENTS по-прежнему публично не подтверждён (наследие ADR-003) — но теперь запрос идёт с правильными серверными токенами, что и было недостающим звеном; остаточный риск закрыт fail-safe.
- **Последствия:** в Worker добавляется маршрут и handler; в `api.js` `postComment` бьёт на Worker URL вместо `YC.post`; контракт возврата неизменен.

#### Вариант 1B (отвергнут): получать клиентский `user_token` через YCLIENTS auth и слать `POST /comments` из браузера

Авторизовать клиента в YCLIENTS (SMS/пароль), сохранить личный `user_token`, подставлять его в `POST /comments`.

- **Минусы:** технически невозможно — YCLIENTS не предоставляет API для получения клиентского `user_token` из браузера через партнёрское API (BA-004, раздел 1 и вопрос 8, вариант A). Код подтверждения брони (`book_code`) не принимается в `user/auth`. Тупиковый путь, подтверждён исследованием.

#### Вариант 1C (отвергнут): обновить статический `YC.userToken` в `api.js` на свежий admin-токен

Перевыпустить admin `user_token` и заменить значение в коде PWA.

- **Минусы:** не решает проблему — даже свежий admin `user_token` может не иметь права `client_comments_add_access` для записи от лица клиента (BA-004, вопрос 8, вариант C). Усугубляет хранение токенов в коде PWA (нарушает НФТ-1/КП-3 — токены остаются в браузере). Хрупко: токен снова протухнет.

### Вопрос 2. Как PWA получает адрес серверного endpoint-а

#### Вариант 2A (выбран): захардкодить базовый Worker URL константой в коде PWA

Базовый URL Worker (`https://studio-push.ivahub84.workers.dev`) задаётся константой в PWA (в `constants.js` или рядом с `postComment`), `postComment` шлёт на `<base>/review`. URL — публичный адрес, не секрет.

- **Плюсы:** работает у любого клиента без настройки (КП-1 выполняется для всех пользователей, а не только для админа, настроившего `localStorage`). URL Worker не является секретом (секреты — токены внутри Worker). Один источник адреса.
- **Минусы:** при смене домена Worker нужно править константу и релизить PWA. Приемлемо: домен стабилен, смена — редкое событие.

#### Вариант 2B (отвергнут): брать Worker URL из `localStorage` (`yc_worker_url`), как делает push-модуль

- **Минусы:** `yc_worker_url` задаётся вручную в админ-панели и существует только на устройстве админа. У рядового клиента он пуст → отзыв не отправится. Не годится для пользовательского сценария отзыва (противоречит ФТ-3/КП-1 для обычного клиента).

### Вопрос 3. Какие токены использует Worker для `POST /comments`

#### Вариант 3A (выбран): переиспользовать существующие Secrets `YC_TOKEN` / `YC_USER_TOKEN` / `YC_COMPANY`

Worker уже хранит эти Secrets и применяет их в `handleCron` для `GET /records`. Тот же заголовок `Authorization: Bearer <YC_TOKEN>, User <YC_USER_TOKEN>` используется и для `POST /comments`.

- **Плюсы:** нет новых Secrets, единый формат авторизации к YCLIENTS внутри Worker, токены остаются в Cloudflare (НФТ-1, КП-3, ФТ-2). Если у текущего `YC_USER_TOKEN` нет права записи комментариев — Secrets перевыпускаются через `wrangler secret put` без изменения кода.
- **Минусы:** если для записи комментариев нужен иной user-токен, чем для чтения записей, его придётся обновить в Secrets (операционная, не кодовая задача). Митигируется: имена Secrets неизменны, значение меняется через `wrangler secret put`.

#### Вариант 3B (отвергнут): завести отдельные Secrets специально для `/comments`

- **Минусы:** дублирование без выигрыша — компания и формат авторизации те же. Усложняет конфигурацию. Если потребуется другой токен — достаточно обновить значение существующего Secret (Вариант 3A это покрывает).

## Решение

- **Вопрос 1 — Вариант 1A:** добавить `POST /review` в существующий Cloudflare Worker; Worker проксирует отзыв в YCLIENTS `POST /comments/{YC_COMPANY}` с серверными токенами и возвращает различимый исход. Это единственный работающий транспорт (1B невозможен, 1C не решает проблему).
- **Вопрос 2 — Вариант 2A:** базовый Worker URL — захардкоженная публичная константа в PWA; `postComment` шлёт на `<base>/review`. Не из `localStorage` (2B не работает у клиента).
- **Вопрос 3 — Вариант 3A:** переиспользовать существующие Secrets `YC_TOKEN`/`YC_USER_TOKEN`/`YC_COMPANY`; значения при необходимости перевыпускаются через `wrangler secret put`.

Отвергнуты: 1B (технически невозможно), 1C (не решает 401, усугубляет хранение токенов в коде), 2B (URL из `localStorage` отсутствует у рядового клиента), 3B (избыточные Secrets без выигрыша).

### Границы решения

- Меняется **только транспорт** отправки внутреннего отзыва: `POST /comments` переезжает с браузера на Worker. Контракт `postComment` (`{ok:true}`/`{ok:false}`) и весь UI `submitReview` (валидация звёзд, индикатор, fail-safe, пометка `yc_reviewed_ids`, навигация) — **без изменений** (наследуются из ADR-003).
- **Не затрагиваются:** чтение и рендер отзывов на `s-master` (`GET /comments` из браузера — остаётся как есть), Web Push (`/subscribe`, `/unsubscribe`, `/send`, cron, VAPID — без изменений), любые другие запросы к YCLIENTS из PWA (`/records`, `/book_staff`, `/clients` и т.д. — на Worker не переносятся, BA «Вне рамок»), внешний вид экрана `s-review`, хранение отзывов в Worker (не нужно — только прокси).
- Захардкоженный `YC.userToken` в `api.js` для прочих запросов — известная проблема (CLAUDE.md), в этом релизе **не трогается**; задача только убирает токены из пути отправки отзыва.

## Последствия

- **Положительные:**
  - Устранена первопричина 401: `POST /comments` идёт с серверными токенами → отзыв реально попадает в YCLIENTS и виден на карточке мастера (ФТ-1, ФТ-5, КП-2).
  - YCLIENTS-токены для записи отзыва не передаются в браузер и не добавляются в код PWA (НФТ-1, ФТ-2, КП-3).
  - Минимум новой инфраструктуры: переиспользованы задеплоенный Worker, его Secrets и CORS (НФТ-3, НФТ-4).
  - Fail-safe и UI отзыва из ADR-003 сохранены без изменений: предсказуемая деградация при недоступности Worker/ошибке YCLIENTS (НФТ-5, КП-4).
- **Отрицательные / компромиссы:**
  - Добавлен сетевой хоп PWA → Worker (пренебрежимая задержка, НФТ-2).
  - Worker становится критичным для отправки отзыва: при его недоступности отзыв не уйдёт (закрыто fail-safe — клиент уведомлён, может повторить).
  - При смене домена Worker нужно править константу URL и релизить PWA (редкое событие).
- **Технический долг:**
  - Контракт `POST /comments` YCLIENTS публично не подтверждён (наследие ADR-003): точная форма тела/обязательные поля проверяются эмпирически; при отклонении API закрыто fail-safe. Возврат — при подтверждении контракта.
  - Захардкоженный admin `YC.userToken` в `api.js` для остальных запросов — отдельная задача (вынос всех приватных вызовов на Worker или ротация токена). Вне рамок этого релиза.
  - Серверный дедуп отзывов по `record_id` (ADR-003, техдолг) — без изменений, остаётся отложенным.

## Влияние на код

- **`cf-worker/src/index.js`** — новый handler (например `handleReview`) и регистрация маршрута `POST /review` в `fetch`. Handler парсит тело (`rating`, `text`, `staff_id`, `record_id`), валидирует обязательные поля (как минимум `rating`), делает `POST /comments/${env.YC_COMPANY}` к YCLIENTS с заголовком `Authorization: Bearer ${env.YC_TOKEN}, User ${env.YC_USER_TOKEN}` (формат как в `handleCron`), возвращает `json({ ok:true })` при 2xx/`success:true` и ошибку (4xx/5xx или `{ ok:false }`) иначе. Использует существующие `CORS`/`json`. OPTIONS уже обрабатывается.
- **`docs/modules/api.js`** — `postComment` перестаёт вызывать `YC.post('/comments/...')`; вместо этого `fetch(<WORKER_BASE>/review, { method:'POST', headers:{'Content-Type':'application/json'}, body })`. Контракт возврата (`{ok:true}`/`{ok:false}`) **сохраняется** — `review.js` не меняется. Параметр `_ycPost` заменяется на инъекцию fetch для тестируемости (детали — в спецификации).
- **Worker URL** — публичная константа (в `docs/modules/constants.js` или рядом с `postComment` в `api.js`). Не из `localStorage`. Не секрет.
- **`docs/modules/review.js`** — **без изменений** (контракт `postComment` неизменен). Перечислен только для подтверждения границ.
- **`docs/sw.js`** — `CACHE_VERSION` поднимается на одну ступень, т.к. изменён `api.js` в `APP_SHELL` (и `constants.js`, если константа там). Состав `APP_SHELL` не меняется. Worker в `APP_SHELL` не входит.
- **`cf-worker/wrangler.toml`** — без изменений: используются существующие Secrets и binding. Комментарий о Secrets уже описывает `YC_TOKEN`/`YC_USER_TOKEN`/`YC_COMPANY`.
- **Новые рантайм-зависимости:** нет. Новые Secrets: нет (переиспользуются существующие).
- **Что ломается:** меняется транспорт отправки отзыва (браузер → Worker, намеренно). Чтение отзывов на `s-master`, Web Push, прочие запросы к YCLIENTS, UI `s-review` — не ломаются.
