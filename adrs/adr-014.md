# ADR-014: Карусель фото услуги — полная галерея из YCLIENTS, скругления, peek-аффорданса, зацикливание (transform + клоны), stale-while-revalidate для хоста фото

- **Дата:** 2026-05-29
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-015.md` (все блокирующие вопросы РЕШЕНЫ; §8 №1 «зацикливание» и №10 «источник галереи» — инженерные развилки, закрываются этим ADR)
- **Связанные ADR:** `adrs/adr-013.md` (детальная карточка `s-service`, карусель на CSS scroll-snap + точки, маппинг `photos`/`image_group`, нормализация `serviceImageUrls`, деградация 0/1/N — **прямо дорабатывается этим ADR**; вариант E1 расширяется, варианты A1/B1/C1/D1/F1 сохраняются как есть); `adrs/adr-012.md` (диапазон цен `priceStr` — переиспользуется, не пересматривается); `adrs/adr-001.md` (категорийные плейсхолдеры — переиспользуются). Прямых конфликтов с ADR-002…ADR-011 не выявлено.

> **ПОПРАВКА (v1.12.1, 2026-05-29):** решение G1 ниже опиралось на неверное предположение, что `GET /company/{company}/services/` (СПИСОЧНЫЙ management-эндпойнт, без `{id}`) отдаёт `image_group.images` массивом. На живых данных подтверждено: **списочный** management-эндпойнт **схлопывает галерею до 1 фото** (`image_group.images` = объект `{basic}`); массив отдаёт только **одиночный** `GET /company/{company}/services/{id}` (с `{id}`). Из-за этого в v1.12.0 карусель не появлялась (всегда 1 фото). **Фактический источник галереи изменён на `GET /book_services/{company}` → `data.services[].images` (массив URL-строк), читается одним запросом для всего каталога.** Эндпойнты `/services/`, `/service_categories/`, `/book_staff/` и маппинг прочих полей каталога — без изменений. См. `docs/app.js` (initApp).

## Контекст

ADR-013 ввёл детальный экран услуги `s-service` (`docs/modules/services.js:156-216`) с галереей фото работ: при `photos.length >= 2` рендерится горизонтальная карусель на нативном CSS scroll-snap (`docs/style.css:820-848`, слайд `flex:0 0 100%`) + точки-индикаторы; активная точка обновляется по событию `scroll` (`services.js:204-213`). При 1 фото — одиночное изображение, при 0 — категорийный плейсхолдер. Маппинг каталога (`docs/app.js:160-179`) загружает услуги из `GET /services/{company}` (`app.js:163`) и нормализует фото хелпером `serviceImageUrls(s.image_group)` (`docs/modules/serviceImages.js:23-51`, уже умеет и массив, и объект размеров).

BA-015 ставит четыре доработки этой галереи, две из которых — архитектурные развилки. Силы, действующие на решение:

- **Каталог получает только одно фото.** `GET /services/{company}` отдаёт `image_group.images` **объектом** `{basic}` (одно фото), даже когда в галерее YCLIENTS у услуги несколько фото (подтверждено на живых данных компании 1940850: услуга id 29196117 — 3 фото в галерее, путь `gallery_service_image/...`). Карусель ADR-013 на проде физически не получает >1 слайда. BA ФТ-0/КП-0 требуют доставить **все** фото (до 8), не сломав существующие поля каталога (название, категория, длительность, диапазон цен per-master, описание) и поток записи.
- **Два эндпойнта отдают галерею массивом, но по-разному.** (а) `GET /company/{company}/services/` (management, тот же `Bearer+User` токен, та же база `api.yclients.com/api/v1`) → `image_group.images` **массивом** объектов с `.path` — совместим с уже существующим `serviceImageUrls()`, набор полей услуги тот же, что у `/services/`. (б) `GET /book_services/{company}` (публичный booking) → готовое поле `images[]` (массив URL-строк) + `image` (одиночный), но **иной набор полей**: длительность `seance_length` в секундах (нет `duration`), нет `image_group`, нет `comment`, цены приходят иначе. Источник списка URL фото берётся из API `api.yclients.com`, помеченного `NETWORK_ONLY_HOSTS` (`sw.js:43,76`) — список URL всегда свежий.
- **Карусель «не продаёт» эстетику.** Изображения прямоугольные (`height:280px; object-fit:cover`, без скруглений — `style.css:812-835`), слайд занимает 100% ширины (`flex:0 0 100%`) — соседнее фото не «выглядывает», единственный сигнал «листается» — мелкие точки. В проекте нет переменной `--radius`; карточки скруглены на 16px, кнопки/инпуты — 14px.
- **Нативный scroll-snap «честно» не зацикливается.** На краях прокрутка упирается; перехода с последнего фото на первое (и обратно) нет. ADR-013 (вариант E1) сознательно избегал рукописного touch-свайпа (вариант E2 отвергнут как несоразмерный). BA ФТ-3/КП-3 требуют бесшовное зацикливание в обе стороны, BA §8 №1 даёт варианты и просит «минимальный по коду подход без рантайм-зависимостей, при котором переход через край визуально бесшовен и сохраняется плавный свайп».
- **Фото «застревают».** Байты картинок отдаются с кросс-origin хоста `assets.yclients.com` (пути `main_service_image/...` и `gallery_service_image/...`), который попадает в ветку Service Worker «не-same-origin → cache-first без ревалидации» (`sw.js:108-118`). Закэшированный URL отдаётся из `RUNTIME_CACHE` до смены `CACHE_VERSION` — обновлённое студией фото не доходит до клиента. Имя файла содержит таймстемп (`..._20260529180821.png`) — при перезаливке URL, вероятно, меняется. BA §8 №7 зафиксировал стратегию **stale-while-revalidate** точечно для этого хоста (§8 №8), не задевая APP_SHELL, JS-модули, EmailJS, jsdelivr и оффлайн.
- **Минимальность, PWA/оффлайн, безопасность.** Vanilla JS/HTML/CSS, код в `docs/`, без рантайм-зависимостей (НФТ-1); правки строго в пределах детальной карточки `s-service` и точечной кэш-стратегии (НФТ-5); при изменении файлов `APP_SHELL` (включая `sw.js`) — поднятие `CACHE_VERSION` (НФТ-4); внешние данные — только через `esc()` (НФТ-6); не добавлять запросов за списком фото на каждое открытие карточки (ФТ-12).

Решение охватывает четыре развилки: **(G)** источник полной галереи фото; **(H)** скругления + peek-аффорданса; **(I)** механизм зацикливания и синхронизации индикатора; **(J)** стратегия свежести фото в Service Worker.

## Рассматриваемые варианты

> Аффорданса (peek ~14px + точки, без стрелок — §8 №2), радиус (~16px под карточки — §8 №3), скругление плейсхолдера/одиночного фото (§8 №4), высота 280px + cover (§8 №5), без автопрокрутки (§8 №6), стратегия свежести = SWR (§8 №7), хост фото = `assets.yclients.com` (§8 №8) — **закрыты BA**. Ниже — инженерные развилки реализации.

### Вопрос G. Источник полной галереи фото (до 8) без поломки полей каталога

#### Вариант G1 (выбран): оставить каталог на `/services/`, точечно дочитать галерею из `GET /company/{company}/services/` и слить только поле `photos` по `id`

Основная разовая загрузка каталога (`app.js:160-179`, `Promise.all` из `/service_categories`, `/services`, `/book_staff`) **сохраняется как есть** — все существующие поля (`id, name, cat, dur, price_min, priceStr, comment, photos`) маппятся из `/services/{company}` без изменений. Дополнительно в тот же `Promise.all` добавляется запрос `GET /company/{company}/services/`; из его ответа строится словарь `id → serviceImageUrls(image_group)` (полная галерея массивом). После маппинга `SERVICES_DATA` поле `photos` каждой услуги **перезаписывается** значением из словаря галереи, **если** там есть непустой массив; иначе остаётся то, что дал `/services/` (одно фото или пусто). Хелпер `serviceImageUrls()` переиспользуется без изменений (он уже обрабатывает массив `image_group.images`).

- **Плюсы:** все поля каталога (название, категория, длительность `duration`, диапазон цен per-master, описание `comment`) и поток записи **остаются на проверенном `/services/`** — нулевой риск регрессии маппинга (ФТ-0/КП-16); галерея сливается **только в поле `photos`** по `id`, точечно; `serviceImageUrls()` уже совместим с `image_group.images` массивом (ADR-013, вариант F1) — нового парсинга нет; +1 запрос **на старте каталога** (разово), без поштучных запросов на открытие карточки (ФТ-12/КП-11); деградация: если management-эндпойнт недоступен/пуст — каталог работает на одиночных фото из `/services/` (graceful).
- **Минусы:** один дополнительный сетевой запрос при старте (параллельно в `Promise.all`, не на критическом пути ниже существующих трёх); услуга, скрытая в management-выдаче, но видимая в `/services/`, останется с одним фото (приемлемо — слияние по `id`, отсутствие в словаре = сохранить текущее).
- **Последствия:** правка `app.js` (добавить 4-й запрос в `Promise.all`, построить словарь галереи, слить `photos`); `serviceImageUrls` — без изменений; `services.js`/каталог — без изменений.

#### Вариант G2 (отвергнут): полностью перевести каталог на `GET /book_services/{company}`

Заменить `/services/` на `/book_services/`, брать готовое `images[]`.

- **Минусы:** **иной набор полей** — длительность `seance_length` (секунды) вместо `duration` (минуты) → переписать `dur`; нет `image_group`, нет `comment` → потеря описания (ADR-013) или новый источник; цены приходят иначе → риск рассинхрона с диапазоном per-master (ADR-012). Это переписывание всего маппинга каталога и риск регрессии фильтров/поиска/баннера/потока записи — прямо против ФТ-0/КП-16/НФТ-5. Несоразмерно ради одного поля фото.
- **Последствия:** широкая зона изменений, высокий риск регрессии. Отвергнут.

#### Вариант G3 (отвергнут): полностью перевести каталог на `GET /company/{company}/services/` (management) вместо `/services/`

Заменить `/services/` на management-эндпойнт целиком.

- **Минусы:** хотя набор полей у management-эндпойнта близок, это смена основного источника каталога без необходимости — `/services/` уже даёт все нужные поля кроме полной галереи; полная замена расширяет зону регрессии (диапазон цен, видимость услуг, порядок) без выигрыша против точечного слияния G1. Против НФТ-5 (минимальность).
- **Последствия:** избыточная зона изменений. Отвергнут в пользу точечного слияния G1.

### Вопрос H. Скругления и peek-аффорданса

#### Вариант H1 (выбран): peek чистым CSS (слайд < 100% + gap + боковой padding контейнера), скругление на каждом слайде/изображении ~16px

Peek реализуется **без JS** (BA §8 №2): контейнер карусели получает горизонтальный `gap` и боковой `padding`, слайд — ширину `< 100%` (например `flex:0 0 calc(100% - 28px)`), так что по краям выглядывают соседние слайды (~14px с каждой стороны). `scroll-snap-align: center` сохраняется. Скругление (~16px, под радиус карточек) применяется к **каждому изображению/слайду** (а не к контейнеру), чтобы при peek у выглядывающего соседа тоже были видны скруглённые края. Тот же радиус — у одиночного фото (`.svc-card-single-img`) и у плейсхолдера (`.svc-card-placeholder`). Высота 280px и `object-fit: cover` сохраняются (§8 №5).

- **Плюсы:** peek — чистый CSS, без JS (НФТ-1); самый сильный сигнал «листается»; скругление на каждом слайде корректно работает с peek (видны скруглённые края соседей); консистентность с карточками проекта; деградация 1/0 фото скругляется тем же радиусом.
- **Минусы:** при `padding` контейнера и `scroll-snap-align:center` нужно выверить, что первый/последний слайд центрируется без «недокрута» — решается `scroll-padding`/боковыми отступами; геометрия peek взаимодействует с зацикливанием (Вопрос I) — оба решаются согласованно.
- **Последствия:** правка `style.css` (`.svc-carousel`, `.svc-carousel-slide`, `.svc-card-single-img`, `.svc-card-placeholder`, добавить `border-radius`, `gap`, боковой `padding`, ширину слайда). Файл `APP_SHELL` → поднятие `CACHE_VERSION`.

#### Вариант H2 (отвергнут): стрелки влево/вправо + скругление на контейнере

- **Минусы:** стрелки прямо отвергнуты BA (§8 №2 — приложение мобильное, свайп — основной жест); скругление контейнера обрезало бы peek-края соседей некрасиво. Против решённого UX.
- **Последствия:** противоречит BA. Отвергнут.

### Вопрос I. Механизм зацикливания и синхронизация индикатора

#### Вариант I1 (выбран): JS-карусель на `transform: translateX` с клонами крайних слайдов + индекс ведётся в JS

Карусель N>1 рендерится как лента слайдов с **клонами**: перед первым слайдом вставляется клон последнего, после последнего — клон первого ([clone_last, s0, s1, …, sN-1, clone_first]). Лента двигается через `transform: translateX` с CSS-`transition`. Текущий логический индекс ведётся в JS-переменной (не вычисляется из `scrollLeft`). Свайп на тач отслеживается лёгкими `touchstart`/`touchmove`/`touchend` (порог + направление), на десктопе — без обязательных стрелок (свайп/трекпад; точки как индикатор). При переходе **за край** (с последнего → клон первого, с первого → клон последнего) после завершения `transitionend` выполняется **бесшовный сброс** позиции на реальный слайд **без анимации** (снять `transition`, выставить `translateX` на реальный индекс, вернуть `transition`). Активная точка подсвечивается по логическому индексу из JS (синхронизация при зацикленном переходе через край — ФТ-4/КП-4). Деградация: 1 фото — одиночное изображение без клонов/ленты/точек; 0 — плейсхолдер (как ADR-013).

- **Плюсы:** **визуально бесшовное** зацикливание в обе стороны (приоритет BA §8 №1 — без заметного «скачка»); индекс ведётся явно в JS → синхронизация точек надёжна и при переходе через край (нет гонки с `scrollLeft`); peek (Вопрос H1) совмещается естественно — ширина слайда `< 100%` + `gap` задаются в той же ленте; без рантайм-зависимостей (НФТ-1); touch-логика минимальна (порог свайпа + transitionend-сброс), это не полноценный физический инерционный движок.
- **Минусы:** отказ от нативного scroll-snap ADR-013 (вариант E1) в пользу transform-ленты — больше JS, чем в ADR-013 (рукописный свайп, который E2 отвергал); нативная инерция прокрутки теряется (заменяется CSS-transition на смену слайда — для пошаговой галереи это приемлемо и предсказуемо); нужно аккуратно снимать слушатели при ре-рендере экрана (DOM `#serviceCardContent` пересоздаётся при каждом `openServiceCard`, старые слушатели уходят с DOM-узлами).
- **Последствия:** переписать ветку `photos.length >= 2` в `openServiceCard` (`services.js:167-213`): рендер ленты с клонами, состояние индекса, touch-хендлеры, transitionend-сброс, подсветка точек по индексу. Стили — transform-лента вместо scroll-snap в `style.css`. Это **расширение варианта E1 ADR-013** (зафиксировать смену механизма).

#### Вариант I2 (отвергнут): сохранить scroll-snap, на краю мгновенно переставлять `scrollLeft` на противоположный край

Оставить нативный scroll-snap, в обработчике `scroll`/`scrollend` при достижении границы прыгать `scrollLeft` на противоположный конец.

- **Минусы:** BA §8 №1 прямо предупреждает о **риске визуального «скачка»**; `scrollend` слабо поддержан, эмуляция по `scroll`+таймеру ненадёжна; «прыжок» во время инерционной прокрутки на iOS часто виден и сбивает жест; синхронизация индикатора через `scrollLeft` на краю — гонки. Приоритет BA — бесшовность; этот вариант его не гарантирует.
- **Последствия:** риск заметного скачка и рассинхрона. Отвергнут.

#### Вариант I3 (отвергнут): нативный scroll-snap без клонов + только программная «телепортация» позиции при загрузке

- **Минусы:** не даёт зацикливания вообще без скачка (см. I2) либо требует тех же клонов, но тогда scroll-snap+клоны+peek+`scrollLeft`-математика сложнее и хрупче, чем явная transform-лента I1 с индексом в JS. Хуже по предсказуемости синхронизации.
- **Последствия:** сложность без выигрыша. Отвергнут.

### Вопрос J. Стратегия свежести фото в Service Worker (точечно для `assets.yclients.com`)

#### Вариант J1 (выбран): отдельная ветка stale-while-revalidate ДО общей не-same-origin ветки, только для хоста фото

В `sw.js` (`fetch`-обработчик) перед общей веткой «не-same-origin → cache-first» (`sw.js:108-118`) добавляется **узкая ветка**: если `url.hostname === 'assets.yclients.com'` (и, для строгости, путь начинается с `main_service_image/` или `gallery_service_image/`) — применить **stale-while-revalidate**: вернуть закэшированный ответ немедленно (если есть), параллельно в фоне сходить в сеть, при `res.ok` положить свежий ответ в `RUNTIME_CACHE`; если кэша нет — вернуть сетевой ответ (и закэшировать); при сетевой ошибке — вернуть кэш (оффлайн-фолбэк). Остальные кросс-origin ресурсы (EmailJS, jsdelivr) продолжают идти по существующей cache-first ветке. `CACHE_VERSION` поднимается (`v57 → v58`).

- **Плюсы:** точно соответствует BA §8 №7/№8 — кэш мгновенно, свежее в фоне, обновление при следующем открытии карточки (ФТ-8/КП-7); **точечно** только для хоста фото (ФТ-10/КП-9) — APP_SHELL, JS, EmailJS, jsdelivr, общий оффлайн не затронуты; не ломает оффлайн (кэш — основной источник показа — ФТ-9/КП-8); покрывает и стабильный, и меняющийся (таймстемп) URL; экономнее network-first (НФТ-8 — показываем кэш, не блокируем загрузку сети).
- **Минусы:** фоновая ревалидация делает сетевой запрос при каждом показе фото (трафик на фон) — приемлемо по BA (приоритет: показать кэш мгновенно, обновлять в фоне); ветка должна стоять **до** общей не-same-origin ветки, иначе фото уйдут в cache-first.
- **Последствия:** правка `sw.js` (новая ветка + поднятие `CACHE_VERSION`). Файл `APP_SHELL`.

#### Вариант J2 (отвергнут): network-first для хоста фото

- **Минусы:** дороже по трафику и медленнее на слабой сети (фото ждёт сеть перед показом); BA §8 №7 явно отверг network-first в пользу SWR. Против НФТ-8.
- **Последствия:** деградация UX/трафика. Отвергнут.

#### Вариант J3 (отвергнут): cache-busting (версия/таймстемп в URL) или TTL кэша

- **Минусы:** cache-busting избыточен — SWR покрывает и меняющийся, и стабильный URL (BA §8 №7); TTL менее предсказуем, требует хранения метаданных времени. Оба сложнее SWR без выигрыша.
- **Последствия:** избыточная сложность. Отвергнут.

## Решение

- **Вопрос G — G1:** каталог остаётся на `GET /services/{company}` (все поля без изменений); в `Promise.all` (`app.js:161-165`) добавляется `GET /company/{company}/services/`, из его ответа строится словарь `id → serviceImageUrls(image_group)`; после маппинга `SERVICES_DATA` поле `photos` перезаписывается значением из словаря **только при непустом массиве**, иначе сохраняется текущее. `serviceImageUrls()` — без изменений. Прочие поля и поток записи не трогаются (ФТ-0, КП-0, КП-16).
- **Вопрос H — H1:** peek чистым CSS (слайд `< 100%` + `gap` + боковой `padding` контейнера, ~14px выглядывания); скругление ~16px на каждом изображении/слайде, одиночном фото и плейсхолдере; высота 280px + `cover` сохранены (ФТ-1, ФТ-2, ФТ-6, КП-1, КП-2, КП-5).
- **Вопрос I — I1:** зацикливание — transform-лента с клонами крайних слайдов; логический индекс ведётся в JS; touch-свайп с порогом; бесшовный сброс позиции по `transitionend`; точки подсвечиваются по индексу из JS (ФТ-3, ФТ-4, КП-3, КП-4). Это **расширяет/заменяет** механизм карусели варианта E1 ADR-013 (нативный scroll-snap) на детальной карточке; деградация 1/0 фото сохраняется как в ADR-013.
- **Вопрос J — J1:** в `sw.js` отдельная SWR-ветка для `assets.yclients.com` (пути `main_service_image/`, `gallery_service_image/`), размещённая **до** общей не-same-origin ветки; `CACHE_VERSION` поднимается `v57 → v58` (ФТ-8, ФТ-9, ФТ-10, КП-7, КП-8, КП-9, НФТ-4).
- **Без стрелок** (ФТ-11/КП-10); точки остаются индикатором (кликабельность не обязательна; если добавляется — не мешает свайпу). Автопрокрутки нет (§8 №6). Тексты на русском (ФТ-13). Внешние данные через `esc()` (НФТ-6, КП-13).

Отвергнуты: G2 (перевод каталога на `book_services` — переписывание полей, риск регрессии), G3 (полная замена на management-эндпойнт — избыточно), H2 (стрелки + скругление контейнера — против решённого UX), I2 (`scrollLeft`-телепортация — риск скачка), I3 (scroll-snap без клонов — хрупко), J2 (network-first — дороже трафик/медленнее), J3 (cache-busting/TTL — избыточно).

### Границы решения

- Изменения **только** в пределах детальной карточки `s-service` (галерея/карусель), точечного слияния поля `photos` при загрузке каталога и точечной SWR-ветки `sw.js`. Каталог (вёрстка, поиск, фильтр категории, баннер предвыбранного мастера, обложка `resolveServiceImage` → `photos[0]`), выбор мастера, расчёт слотов, бронирование, подтверждение, логика цен ADR-012 — **не рефакторятся** (НФТ-5).
- Обложка каталога продолжает использовать `photos[0]` (первое фото). При слиянии галереи G1 `photos[0]` остаётся первым фото из management-галереи — порядок фото определяется ответом API; визуально консистентно с детальной карточкой.
- Не вводятся новые рантайм-зависимости; YCLIENTS-токен не трогается (известный техдолг — не усугублять); операции с веткой `backup` запрещены.
- Зум/лайтбокс, автопрокрутка, счётчик «N/M» — вне объёма (§8 №2, №6; ADR-013).

## Последствия

- **Положительные:**
  - Клиент видит **всю** галерею услуги (до 8 фото), а не одно — карусель ADR-013 наконец получает реальные данные (ФТ-0).
  - Галерея считывается как премиальный листаемый компонент: скругления + peek + зацикливание (ФТ-1, ФТ-2, ФТ-3).
  - Обновлённое студией фото доходит до клиента автоматически при следующем открытии карточки, оффлайн сохранён (ФТ-8, ФТ-9).
  - Поля каталога и поток записи неизменны — слияние галереи точечно по `id` (нулевой риск регрессии маппинга).
  - SWR применён узко к хосту фото — прочее кэширование и оффлайн не затронуты (ФТ-10).
- **Отрицательные / компромиссы:**
  - Отказ от нативного scroll-snap ADR-013 на детальной карточке в пользу transform-ленты — больше JS и потеря нативной инерции прокрутки (компенсируется предсказуемой пошаговой анимацией и надёжной синхронизацией индекса).
  - +1 сетевой запрос при старте каталога (management-эндпойнт) — параллельно в `Promise.all`, разово; деградирует мягко при недоступности.
  - Фоновая ревалидация фото делает сетевой запрос при каждом показе — осознанный выбор BA ради свежести (приоритет: кэш мгновенно, обновление в фоне).
- **Технический долг:**
  - Унификация механизма карусели (transform-лента) с возможными будущими галереями — не выносится в общий компонент в этом релизе.
  - Зум/лайтбокс — при отдельном запросе заказчика.
  - Возможная двойственность источника фото (services + management) — приемлема; консолидация при следующем касании маппинга.

## Риски

- **Риск (СРЕДНИЙ): `GET /company/{company}/services/` отдаёт иную форму/недоступен или меняет состав услуг.** → **Митигация:** словарь строится защищённо через `serviceImageUrls()` (любая иная форма → `[]`); слияние только при **непустом** массиве, иначе сохраняется `photos` из `/services/` (graceful degrade — каталог работает на одиночных фото); запрос в `Promise.all` не блокирует остальные (обрабатывать `success === false` мягко). Разработчик подтверждает на живых данных, что `image_group.images` приходит массивом и `id` совпадают с `/services/`.
- **Риск (СРЕДНИЙ): transform-лента с клонами даёт видимый «скачок» при сбросе позиции на краю.** → **Митигация:** сброс делается **без анимации** (снять `transition`, переставить `translateX`, форсировать reflow, вернуть `transition`) строго в `transitionend`; протестировать на тач (iOS/Android) — приоритет BA «без заметного скачка» (§8 №1). Если на каком-то устройстве скачок виден — эскалация.
- **Риск (СРЕДНИЙ): peek + центрирование первого/последнего слайда** (недокрут/полоса по краю). → **Митигация:** выверить боковой `padding`/`scroll-padding` и ширину слайда; в transform-ленте позиция считается явно, центрирование детерминировано.
- **Риск (НИЗКИЙ): SWR-ветка перехватит лишние ресурсы или не сработает (попадёт в cache-first).** → **Митигация:** условие узкое (`hostname === 'assets.yclients.com'` + префикс пути), ветка размещается **до** общей не-same-origin ветки; проверить, что EmailJS/jsdelivr/APP_SHELL/JS идут прежними ветками (КП-9).
- **Риск (НИЗКИЙ): старый кэш SW отдаёт прежнюю разметку/CSS/JS/SW.** → **Митигация:** при изменении `prototype.html`(если нужен)/`style.css`/`services.js`/`app.js`/`sw.js` **поднять `CACHE_VERSION`** `v57 → v58` (НФТ-4, КП-14).
- **Риск (НИЗКИЙ): XSS через URL фото/`alt`.** → **Митигация:** все URL и `alt` — через `esc()`; клоны строятся из тех же экранированных данных; вставка непроверенного HTML недопустима (НФТ-6, КП-13).
