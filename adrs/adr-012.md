# ADR-012: Диапазон цен услуги в каталоге — агрегация min/max по ценам мастеров через единый проход `book_services` с кэшированием

- **Дата:** 2026-05-29
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-013.md` (блокирующий открытый вопрос №1 РЕШЁН заказчиком 2026-05-29: диапазон вычисляется приложением из цен мастеров; вопросы №2–№6 — значения по умолчанию зафиксированы)
- **Связанные ADR:** прямых пересечений с ADR-001…ADR-011 не выявлено. Затрагивает схему загрузки каталога услуг (`app.js`, `initApp`), которая ранее ни одним ADR не фиксировалась.

## Контекст

В каталоге услуг (`s-services`, `_renderList`, `docs/modules/services.js:86-99`) для каждой услуги показывается строка `{priceStr} · {dur} мин`. `priceStr` формируется один раз при загрузке услуг из YCLIENTS `/services/{company}` через `_fmtPrice(s.price_min, s.price_max)` (`docs/app.js:172`). Сейчас при различии min и max `_fmtPrice` **намеренно отбрасывает верхнюю границу** и выводит «от {min} ₽» (`docs/modules/utils.js:17-21`).

**Ключевое подтверждение заказчика (BA §8 №1, обязательное входное ограничение):** поля `price_min`/`price_max` услуги из `/services/` **не отражают разброс цен по мастерам**. Одну услугу разные мастера выполняют по разной цене (например, «Стрижка»: Иван — 2 000 ₽, Мария — 5 000 ₽), и **минимум/максимум приложение должно вычислять само** по ценам мастеров. Тривиальная правка `_fmtPrice` (показывать `price_min`–`price_max` из `/services/`) **отменена как недостаточная** — нужен сбор цен по мастерам и агрегация. Это меняет схему загрузки данных каталога и требует настоящего ADR.

**Бизнес-цель (BA, раздел 1):** показывать клиенту понятный диапазон стоимости «от минимальной до максимальной», когда цены у разных мастеров отличаются — для прозрачности ценообразования и снижения расхождения ожиданий с итоговой ценой записи.

**Решения заказчика по открытым вопросам BA (§8) — обязательные входные ограничения:**

1. **(РЕШЕНО, блокирующий) Источник диапазона — цены мастеров, не `/services/`.** Диапазон строится агрегированием цен по мастерам, выполняющим услугу. Конкретную схему и стоимость запросов определяет **настоящий ADR**.
2. **Формат диапазона по умолчанию — «от {min} до {max} ₽»** (словесная форма), форматирование сумм `ru-RU`, символ «₽». Допустима замена на «{min} – {max} ₽» на усмотрение реализации; формат **единый во всех местах** показа диапазона.
3. **Подтверждение и сохранение записи — конкретная цена выбранного мастера** (не диапазон): экран подтверждения «Стоимость» (`slots.js:85`) и поле `price` визита (`booking.js:30`).
4. **Режим предвыбранного мастера в каталоге — конкретная цена этого мастера** (не диапазон), `services.js:106-129`.
5. **Поиск на главной — тот же диапазон, что и в каталоге** (`search.js:27`), для единообразия.
6. **Граница «диапазон vs единичная цена» — диапазон только при различии min и max**; при равенстве/единственном мастере — единичная сумма; при отсутствии цены — «По запросу».

Силы, действующие на решение:

- **Доступные эндпоинты YCLIENTS (подтверждены в коде).**
  - Цена мастера на его услуги: `GET /book_services/{company}?staff_id={id}` → `data.services[]` с полями `id`, `title`, `duration`, `price_min` (используется в `masters.js:146` и в режиме мастера `services.js:111`).
  - Список мастеров услуги: `GET /book_staff/{company}?service_ids[]={id}` → `data[]` мастеров (`api.js:132`, `fetchStaffByService`).
  - Конкретная цена выбранного мастера на услугу доступна через тот же `/book_services?staff_id=` (поле `price_min` элемента `services[]` для нужного `service.id`).
- **Малый объём данных (BA §6, НФТ-2).** Число мастеров и услуг невелико (в текущих данных ~4 мастера, ~13 услуг). CORS разрешён (`api.js`, прямые запросы к YCLIENTS работают). Это делает сбор «по мастерам» дешёвым: количество запросов = число мастеров, а не число услуг.
- **Текущая загрузка каталога — `Promise.all` из трёх запросов (`app.js:160-174`).** `service_categories`, `services`, `book_staff` грузятся параллельно; затем `setServicesData` + `renderServices`. Сбор цен мастеров должен встроиться сюда, **не блокируя первичную отрисовку каталога** (НФТ-2: «не деградировать время загрузки каталога», «прокрутка/отрисовка не должны деградировать»).
- **Деградация при недоступности данных мастера (BA §8 №1, явно требуется учесть).** Если запрос цен мастера упал/пуст, агрегация по этой услуге должна опираться на доступные данные; при полном отсутствии собранных цен по услуге — корректный фолбэк на имеющуюся цену из `/services/` (или «По запросу»), без ошибки в консоли.
- **Конкретная цена нужна после выбора мастера (BA №3, №4).** На подтверждении (`slots.js:85`) и при сохранении визита (`booking.js:30`) мастер уже выбран — там логична **его** цена, а не диапазон. В режиме предвыбранного мастера (`services.js:106-129`) — тоже конкретная цена. Это значит: помимо агрегата «диапазон по услуге» нужна структура «цена мастера M на услугу S».
- **`_fmtPrice` — общая утилита (НФТ-3).** Вызывается из `app.js:172`; её прямое изменение поведения затронуло бы все вызовы. Нельзя менять её семантику без учёта всех мест; безопаснее добавить отдельную функцию форматирования диапазона, а `_fmtPrice` сохранить для единичной цены.
- **Vanilla JS, без рантайм-зависимостей (НФТ-1); минимальность (НФТ-3); PWA/офлайн (НФТ-4).** Правки — в существующих файлах `APP_SHELL`; при их изменении — поднятие `CACHE_VERSION` (единое на релиз v1.10.0).

Решение охватывает: (A) схему сбора цен мастеров и агрегации диапазона (ключевой вопрос); (B) когда выполнять сбор относительно отрисовки каталога (блокирующе vs фоном); (C) откуда брать конкретную цену выбранного мастера для подтверждения/сохранения/режима мастера; (D) форматирование диапазона и единичной цены без поломки общей `_fmtPrice`; (E) поведение при недоступности данных мастера.

## Рассматриваемые варианты

> Вопросы №1 (источник — цены мастеров), №2 (формат «от {min} до {max} ₽»), №3/№4 (конкретная цена после/при выборе мастера), №5 (поиск — тот же диапазон), №6 (граница диапазон/единичная) — **закрыты решениями заказчика** (BA §8). Архитектор не пересматривает их; ниже — инженерные развилки реализации в рамках принятых решений.

### Вопрос A. Схема сбора цен мастеров и агрегации диапазона по услуге

Цель — для каждой услуги получить min/max по ценам мастеров минимальным числом запросов, не деградируя загрузку (НФТ-2).

#### Вариант A1 (выбран): один проход по всем мастерам — `book_services?staff_id=` на каждого мастера, агрегация min/max по услуге, кэш в памяти

При загрузке каталога после получения списка мастеров (`book_staff`, уже грузится в `app.js:163`) выполнить по запросу `GET /book_services/{company}?staff_id={ycId}` на **каждого** мастера (параллельно, `Promise.all`). Для каждого ответа пройти `data.services[]` и для каждого `service.id` накопить `price_min` мастера в агрегат услуги. Получить две структуры в памяти (модуль `state` / отдельный кэш-модуль):
- **`servicePriceRange[serviceId] = { min, max }`** — агрегат min/max по всем мастерам, выполняющим услугу (только по валидным `price_min > 0`).
- **`staffServicePrice[staffId][serviceId] = price`** — конкретная цена мастера на услугу (для Вопроса C: подтверждение, сохранение, режим мастера).

Число запросов = **число мастеров** (N≈4), а не число услуг (M≈13). После сбора пересчитать `priceStr` каждой услуги через форматтер диапазона и перерисовать каталог (`renderServices`).

- **Плюсы:** минимум запросов — O(N мастеров), при малом N это дёшево (BA §6, НФТ-2); один проход даёт **обе** нужные структуры (диапазон по услуге **и** цена мастера на услугу) — закрывает и каталог, и Вопрос C (подтверждение/сохранение/режим мастера) **без дополнительных запросов**; переиспользует уже применяемый в коде эндпоинт `book_services?staff_id=` (`masters.js:146`, `services.js:111`) — формат ответа известен; кэш в памяти живёт на сессию, повторные входы в каталог запросов не делают.
- **Минусы:** N дополнительных запросов при первой загрузке (параллельных); требует кэш-структуры в `state`; зависит от того, что `book_services?staff_id=` отдаёт `price_min` по каждой услуге мастера (подтверждено форматом в `masters.js:170-178`).
- **Последствия:** правка `app.js` (сбор после `book_staff`), новый кэш в `state`, новая логика агрегации (в `services.js` или отдельном хелпере), форматтер диапазона в `utils.js`.

#### Вариант A2 (отвергнут): запрос на каждую услугу — `book_staff?service_ids[]=` + цена по каждому мастеру

Для каждой услуги получить список её мастеров (`book_staff?service_ids[]=`, `api.js:132`), затем для каждого мастера — его цену. Это O(M услуг) запросов минимум, а цена мастера на конкретную услугу всё равно берётся из `book_services` → каскад запросов O(M × N) в худшем случае.

- **Минусы:** существенно больше запросов (O(M) или O(M×N)) при том же результате — прямо противоречит НФТ-2 («минимизировать число запросов», «один проход по мастерам, а не запрос на каждую услугу» — формулировка BA §8 №1); `book_staff?service_ids[]=` к тому же не возвращает цену мастера на услугу, потребовался бы всё равно `book_services`. При малом M выигрыша нет, проигрыш по числу запросов очевиден.
- **Последствия:** деградация загрузки, лишняя сложность. Отвергнут.

#### Вариант A3 (отвергнут): ленивый сбор «по требованию» — запрашивать цены только для видимой/тапнутой услуги

Не собирать ничего при загрузке; при показе/тапе услуги дёргать `book_staff?service_ids[]=` + `book_services`.

- **Минусы:** диапазон должен быть виден **в каталоге сразу** (ФТ-1, основной сценарий) — ленивый сбор «на тап» этого не даёт; «на скролл» (intersection) переусложняет vanilla-код, плодит запросы при прокрутке (деградация прокрутки — против НФТ-2) и непредсказуемо мигает ценой. При малом числе мастеров экономии нет — проще собрать всё разом фоном (A1).
- **Последствия:** мигание цен, сложность, противоречие НФТ-2. Отвергнут.

### Вопрос B. Когда выполнять сбор относительно первичной отрисовки каталога

НФТ-2 запрещает деградацию времени загрузки каталога.

#### Вариант B1 (выбран): первичная отрисовка — сразу из `/services/` (как сейчас), сбор цен мастеров — фоном, затем перерисовка

`initApp` отрисовывает каталог немедленно по `/services/` (текущее поведение `app.js:167-174`, `priceStr` через `_fmtPrice(s.price_min,s.price_max)` как фолбэк). Параллельно/после запускается сбор цен мастеров (Вопрос A1); по завершении — пересчёт `priceStr` (форматтер диапазона) и `renderServices()` (перерисовка с диапазонами). Сбор **не блокирует** первичную отрисовку.

- **Плюсы:** время до первого показа каталога **не деградирует** (НФТ-2) — клиент сразу видит услуги; диапазоны «доезжают» фоном и заменяют фолбэк-цену; естественно ложится на текущий `initApp` (там уже есть повторный `renderServices()`, `app.js:195`); при провале сбора каталог остаётся с фолбэк-ценой из `/services/` (graceful degradation, Вопрос E).
- **Минусы:** короткое «доуточнение» цены после загрузки (фолбэк → диапазон) — приемлемо и незаметно при малом N; требует, чтобы `priceStr` можно было пересчитать и перерисовать (он и так пересчитывается через `renderServices`).
- **Последствия:** сбор встраивается в `initApp` как фоновый шаг с финальным `renderServices()`; первичная отрисовка сохраняется.

#### Вариант B2 (отвергнут): блокировать отрисовку каталога до завершения сбора цен мастеров

Не показывать каталог, пока все `book_services` не собраны и диапазоны не посчитаны.

- **Минусы:** добавляет N запросов **в критический путь** первой отрисовки → деградация времени загрузки каталога (прямо против НФТ-2); при медленном/упавшем запросе мастера каталог «висит». Несоразмерно: диапазон — уточнение цены, не блокер показа.
- **Последствия:** деградация UX загрузки. Отвергнут.

### Вопрос C. Откуда брать конкретную цену выбранного мастера (подтверждение, сохранение, режим мастера)

BA №3/№4: после/при выборе мастера показывается и сохраняется **его** цена, не диапазон.

#### Вариант C1 (выбран): переиспользовать структуру `staffServicePrice[staffId][serviceId]`, собранную единым проходом (Вопрос A1)

Тот же проход A1 уже строит `staffServicePrice[staffId][serviceId] = price`. На подтверждении (`slots.js:85`) и при сохранении (`booking.js:30`): если выбран конкретный мастер (`getMaster()` не null) и есть `staffServicePrice[masterId][serviceId]` — показать/сохранить **эту** цену (форматом единичной суммы); иначе фолбэк на текущий `svc.priceStr`. В режиме предвыбранного мастера (`services.js:106-129`) — для каждой услуги показывать `staffServicePrice[masterId][serviceId]`, а не диапазон; там уже грузится `book_services?staff_id=` (`services.js:111`), значит цена мастера доступна прямо в этом ответе (`s.price_min`).

- **Плюсы:** **никаких дополнительных запросов** — данные уже собраны проходом A1 (и для режима мастера — прямо в локальном ответе `services.js:111`); единый источник цены устраняет рассинхрон между каталогом, подтверждением, сохранением; согласуется с тем, что подтверждение/сохранение и так читают контекст из `state`/`getService`/`getMaster`.
- **Минусы:** добавляет обращение к кэш-структуре в `slots.js`/`booking.js`/`services.js` — точечные правки; нужен фолбэк, если цены мастера в кэше нет (Вопрос E).
- **Последствия:** правки `slots.js` (строка «Стоимость»), `booking.js` (поле `price`), `services.js` (режим мастера) — берут цену мастера из кэша/локального ответа; форматирование — единичной суммой.

#### Вариант C2 (отвергнут): отдельный запрос цены мастера на услугу в момент подтверждения

На входе на подтверждение дёргать `book_services?staff_id=` ради одной цены.

- **Минусы:** лишний запрос там, где данные уже есть (A1 их собрал); задержка на экране подтверждения; дублирование логики. При уже собранном кэше — бессмысленно.
- **Последствия:** лишние запросы и задержка. Отвергнут.

### Вопрос D. Форматирование диапазона и единичной цены без поломки общей `_fmtPrice`

НФТ-3: `_fmtPrice` — общая утилита; нельзя менять её поведение в сценариях вне задачи.

#### Вариант D1 (выбран): новая функция `_fmtPriceRange(min, max)` в `utils.js`; `_fmtPrice` сохраняет текущую семантику

Добавить в `utils.js` отдельный форматтер диапазона `_fmtPriceRange(min, max)`:
- нет цены (`!min`) → «По запросу» (как `_fmtPrice`);
- `min === max` или нет `max` → единичная сумма `f(min)` (BA №6);
- `min < max` → «от {f(min)} до {f(max)}» (формат по умолчанию, BA №2; допускается «{f(min)} – {f(max)} ₽»);
- внутреннее форматирование суммы `f(n)=Math.round(n).toLocaleString('ru-RU')+' ₽'` — **то же**, что в `_fmtPrice` (`utils.js:19`), `ru-RU` + «₽» + округление (ФТ-4).

`_fmtPrice` остаётся как есть и продолжает использоваться для единичных цен (например, конкретная цена мастера). Каталог (`priceStr` при наличии собранного диапазона) формируется через `_fmtPriceRange`. Единичную цену мастера на подтверждении/сохранении — через `_fmtPrice` (или общий `f`).

- **Плюсы:** не меняет поведение `_fmtPrice` → нулевой риск регрессии в сценариях вне задачи (НФТ-3); единый источник форматирования сумм (`ru-RU`/«₽»/округление) переиспользуется; единый формат диапазона во всех местах показа (каталог + поиск на главной, BA №5) — оба зовут `_fmtPriceRange`.
- **Минусы:** дополнительная экспортируемая функция в `utils.js`. Минимально.
- **Последствия:** правка `utils.js` (новый `_fmtPriceRange`); вызовы из `services.js`/`app.js`/`search.js` (каталог/поиск) на диапазон, из `slots.js`/`booking.js` на единичную цену мастера.

#### Вариант D2 (отвергнут): изменить `_fmtPrice`, добавив режим диапазона

Расширить сигнатуру/поведение `_fmtPrice`, чтобы она возвращала диапазон.

- **Минусы:** меняет общую утилиту, вызываемую и в сценариях, где нужна единичная цена (риск незаметной регрессии — НФТ-3 прямо предостерегает); требует проверки и правки всех вызовов. Несоразмерно — проще новая функция.
- **Последствия:** риск регрессии общей утилиты. Отвергнут.

### Вопрос E. Поведение при недоступности данных мастера

BA §8 №1 требует явно учесть деградацию.

#### Вариант E1 (выбран): graceful degradation на доступные данные, без ошибок в консоли

- Сбор цен по мастерам: запросы независимы; провал/пустой ответ по мастеру **пропускается** (try/catch как в `services.js:121`, `api.js:32`), агрегация ведётся по успешно собранным.
- Агрегат услуги: если по услуге собрана **хотя бы одна** валидная цена — `min/max` по собранным; если различие — диапазон, иначе единичная.
- Если по услуге **не собрано ни одной** цены мастера (все запросы упали/пусты) — `priceStr` остаётся фолбэком из `/services/` (текущий `_fmtPrice(price_min,price_max)`), либо «По запросу» при отсутствии цены. Каталог не пустеет и не ломается.
- Конкретная цена мастера (Вопрос C): если в кэше нет `staffServicePrice[masterId][serviceId]` — фолбэк на текущий `svc.priceStr` (как сейчас).
- Во всех ветках — **ноль ошибок в консоли** (КП-8); существующие try/catch сохраняются.

- **Плюсы:** каталог всегда отрисовывается (устойчивость к частичным сбоям API); диапазон — улучшение поверх работающего фолбэка, а не точка отказа; согласуется с тем, как код уже глотает ошибки YCLIENTS (`api.js`, `services.js`).
- **Минусы:** при массовом сбое сбора каталог покажет старое «от {min}» (фолбэк) вместо диапазона — приемлемо (лучше, чем пустой/сломанный экран).
- **Последствия:** ветки фолбэка в агрегации и в местах конкретной цены; сохранение try/catch.

## Решение

- **Источник (BA №1):** диапазон вычисляется приложением как min/max по ценам мастеров; `price_min`/`price_max` из `/services/` используются **только как фолбэк** при недоступности собранных цен (Вопрос E).
- **Вопрос A — Вариант A1:** единый проход по мастерам — `GET /book_services/{company}?staff_id={ycId}` на каждого мастера (параллельно, после `book_staff`), агрегация в две кэш-структуры в памяти: `servicePriceRange[serviceId] = {min,max}` (диапазон по услуге) и `staffServicePrice[staffId][serviceId] = price` (цена мастера на услугу). Число запросов = число мастеров.
- **Вопрос B — Вариант B1:** первичная отрисовка каталога — немедленно по `/services/` (как сейчас, фолбэк-цена), сбор цен мастеров — **фоном**, по завершении пересчёт `priceStr` через форматтер диапазона и `renderServices()`. Сбор **не блокирует** первичную отрисовку (НФТ-2).
- **Вопрос C — Вариант C1:** конкретная цена выбранного мастера для подтверждения (`slots.js:85`), сохранения визита (`booking.js:30`) и режима предвыбранного мастера (`services.js:106-129`) берётся из `staffServicePrice` (или прямо из локального ответа `book_services?staff_id=` в режиме мастера, `services.js:111`) — **без дополнительных запросов**; форматируется как единичная сумма.
- **Вопрос D — Вариант D1:** новый `_fmtPriceRange(min,max)` в `utils.js` (формат «от {min} до {max} ₽» по умолчанию, BA №2; единичная сумма при равенстве/отсутствии max, BA №6; «По запросу» при отсутствии цены; форматирование сумм `ru-RU`/«₽»/округление — как в `_fmtPrice`). `_fmtPrice` **не меняется** (НФТ-3) и используется для единичных цен. Каталог и поиск на главной (BA №5) зовут `_fmtPriceRange` → единый формат диапазона (BA №2, ФТ-6).
- **Вопрос E — Вариант E1:** graceful degradation — частичные сбои сбора пропускаются, агрегация по доступным данным; при полном отсутствии собранных цен по услуге — фолбэк на цену из `/services/` (или «По запросу»); конкретная цена мастера при отсутствии в кэше — фолбэк на текущий `priceStr`. Во всех ветках — ноль ошибок в консоли.

Отвергнуты: A2/A3 (больше запросов / мигание / не дают диапазон в каталоге сразу), B2 (блокировка отрисовки — деградация загрузки), C2 (лишний запрос при уже собранном кэше), D2 (ломкая правка общей `_fmtPrice`).

### Границы решения

- Диапазон в **каталоге** (`_renderList`, `services.js:95`) и в **поиске на главной** (`search.js:27`) — через `_fmtPriceRange`, формат единый (BA №2, №5).
- **Конкретная цена выбранного мастера** — на подтверждении (`slots.js:85`), при сохранении визита (`booking.js:30`, поле `price`), в режиме предвыбранного мастера (`services.js:106-129`); не диапазон (BA №3, №4). Цена визита согласована с показанной на подтверждении (ФТ-8, КП-7).
- **Граница диапазон/единичная:** диапазон только при `min < max`; при равенстве/единственном мастере — единичная сумма; без цены — «По запросу» (BA №6).
- Источник цен мастеров — `book_services?staff_id=` (по каждому мастеру) + `book_staff` (список мастеров уже грузится). **Дополнительных эндпоинтов не вводится.**
- Кэш цен (`servicePriceRange`, `staffServicePrice`) живёт **в памяти на сессию** (`state`/кэш-модуль); persist в `localStorage` **не вводится** (вне объёма; цены могут меняться, сессионного кэша достаточно).
- **Не затрагиваются:** логика выбора мастера, расчёта слотов (`loadDates`/`loadSlots`), самого бронирования (`_bookWithSession`); вёрстка/дизайн карточки услуги и экрана подтверждения (меняется только текст строки цены); мок-данные `SERVICES_DATA` (`state.js:2-14`, fallback до прихода API — на диапазоны не переписываются); карточка мастера `openMasterCard` (`masters.js`, отдельный список услуг мастера со своей ценой — вне объёма, не регрессирует); захардкоженный YCLIENTS-токен.

## Последствия

- **Положительные:**
  - Клиент видит реальный диапазон стоимости услуги в каталоге (бизнес-цель BA); прозрачность ценообразования.
  - Минимум запросов — O(число мастеров); один проход даёт и диапазон по услуге, и цену мастера на услугу → подтверждение/сохранение/режим мастера обслуживаются **без дополнительных запросов** (НФТ-2).
  - Первичная отрисовка каталога **не деградирует** (сбор фоном, B1); диапазоны доезжают и заменяют фолбэк.
  - Единый источник форматирования сумм и единый формат диапазона во всех местах (каталог, поиск) — нет рассинхрона (BA №2, №5, ФТ-6).
  - Устойчивость к частичным сбоям API (E1) — каталог всегда отрисовывается; ноль ошибок в консоли.
  - `_fmtPrice` не тронута → нулевой риск регрессии вне задачи (НФТ-3).
- **Отрицательные / компромиссы:**
  - N дополнительных (параллельных, фоновых) запросов при первой загрузке — приемлемо при малом N (BA §6).
  - Кратковременное «доуточнение» цены (фолбэк → диапазон) после загрузки — незаметно при малом N.
  - Кэш цен — сессионный, в памяти; при долгой сессии не отражает изменения цен в YCLIENTS до перезагрузки — осознанный компромисс (persist вне объёма).
- **Технический долг:**
  - Persist кэша цен и его инвалидация — **не делается**; при необходимости — отдельным решением.
  - Унификация «цены мастера на услугу» между этим кэшем и списком услуг в `openMasterCard` (`masters.js:170-178`, своя `s.price_min`) — не консолидируется; к вопросу при следующем касании карточки мастера.

## Влияние на код

- **`docs/modules/utils.js`** — новый экспортируемый `_fmtPriceRange(min, max)` (Вопрос D1): «По запросу» при `!min`; единичная сумма при `min===max`/`!max`; «от {min} до {max} ₽» при `min<max` (формат по умолчанию BA №2, допускается «{min} – {max} ₽»); форматирование сумм идентично `_fmtPrice` (`utils.js:19`, `ru-RU`/«₽»/`Math.round`). `_fmtPrice` (`utils.js:17-21`) **не меняется**.
- **`docs/modules/state.js`** — кэш-структуры цен в памяти: `servicePriceRange` (`{ [serviceId]: { min, max } }`) и `staffServicePrice` (`{ [staffId]: { [serviceId]: price } }`) — как поля `state` или отдельный экспорт; сеттер(ы) для наполнения. Схема состояния по сути не меняется (аддитивно).
- **`docs/app.js`** — в `initApp` (`app.js:160-198`) после получения мастеров (`staffRes`, `app.js:163,176`) добавляется **фоновый** сбор цен (Вопрос A1, B1): `Promise.all` запросов `YC.get('/book_services/'+YC.company, { staff_id: ycId })` по каждому мастеру (использовать `ycId`/`id` мастера, как в `masters.js:146`); агрегация `price_min` по `service.id` в `servicePriceRange` (min/max) и `staffServicePrice`; затем пересчёт `priceStr` услуг через `_fmtPriceRange(range.min, range.max)` (при наличии собранного диапазона; иначе — текущий фолбэк `_fmtPrice(price_min,price_max)`) и `renderServices()`. Первичная отрисовка каталога (`app.js:167-174`) — **без изменений** (фолбэк-цена). Сбор не блокирует первичный показ.
- **`docs/modules/services.js`**:
  - Основной каталог: `priceStr` услуги уже формируется в `app.js`; `_renderList` (`services.js:95`) выводит `s.priceStr` — без структурных изменений (значение приходит из `_fmtPriceRange`).
  - Режим предвыбранного мастера (`renderServices`, `services.js:106-129`): для услуг показывать **конкретную цену мастера** из ответа `book_services?staff_id=` (`services.js:111`, поле `s.price_min` для нужной услуги) / из `staffServicePrice` — единичной суммой (через `_fmtPrice`/`f`), а не диапазон (BA №4). Текущее формирование «от {price_min} ₽» (`services.js:118`) заменить на единичную цену мастера согласованного формата.
- **`docs/modules/search.js`** — поиск на главной (`search.js:27`) выводит `s.priceStr`, который уже содержит диапазон (BA №5) — структурно без изменений, при условии что `priceStr` пересчитан через `_fmtPriceRange`. Проверить, что строка отображается корректно.
- **`docs/modules/slots.js`** — экран подтверждения «Стоимость» (`slots.js:85`): если выбран конкретный мастер (`getMaster()`), показать его цену из `staffServicePrice[masterId][serviceId]` (единичной суммой, BA №3); иначе фолбэк на `svc.priceStr`.
- **`docs/modules/booking.js`** — `_saveBookedRecord` (`booking.js:30`, поле `price`): сохранять цену, **согласованную с показанной на подтверждении** (BA №3, ФТ-8) — конкретную цену мастера при его выборе; иначе текущий `svc.priceStr`.
- **`docs/sw.js`** — состав `APP_SHELL` **не меняется** (правятся существующие файлы: `utils.js`, `state.js`, `app.js`, `services.js`, `search.js`, `slots.js`, `booking.js`); `CACHE_VERSION` **поднимается на одну ступень** (единое поднятие на релиз v1.10.0 совместно с ADR-011, см. спецификацию).
- **`docs/version.json`** — обновляется мейнтейнером при релизе (вне объёма реализации фичи).
- **Новые рантайм-зависимости:** нет. YCLIENTS-токен не трогается. Новых эндпоинтов не вводится. Операции с веткой `backup` запрещены.
- **Что ломается:** ничего из существующего. Каталог получает диапазоны (фоном, с фолбэком); подтверждение/сохранение/режим мастера переходят на конкретную цену мастера. `_fmtPrice`, `SERVICES_DATA`-fallback, логика слотов/бронирования, карточка мастера, YCLIENTS-интеграция — не ломаются.
