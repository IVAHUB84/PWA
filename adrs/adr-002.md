# ADR-002: Фильтрация мастеров по выбранной услуге — источник связи «мастер ↔ услуга»

- **Дата:** 2026-05-28
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-002.md`
- **Связанные ADR:** ADR-001 (не конфликтует; общий поток `услуга → мастер`)

## Контекст

Сценарий записи: клиент выбирает услугу (`s-services`) → попадает на экран мастеров (`s-masters`) → выбирает мастера или «Любой мастер» → выбирает слот. Сейчас на `s-masters` (в режиме перехода от услуги) показываются мастера, отфильтрованные по **статическому** полю `cats` (категории) из `docs/modules/state.js`.

Текущая фильтрация (`renderMasters` в `docs/modules/masters.js`):

```js
const svcCat = svc.cat;
if (svcCat) {
  const bycat = masters.filter(m => m.cats && m.cats.includes(svcCat));
  if (bycat.length) masters = bycat;   // если пусто — показываются все
}
```

Силы, действующие на решение:

- **`cats` не источник истины.** Поле `cats` в `MASTERS_DATA` заполняется при загрузке из YCLIENTS только по совпадению **имени** мастера со статической таблицей (`app.js`: `cats: sm?.cats || []`). Для мастеров, которых нет в статической таблице, `cats === []`, и фильтрация по категории для них не срабатывает. Рассинхрон статики и реального состава услуг мастера в YCLIENTS недопустим как источник истины (ФТ-2, НФТ-1).
- **Услуги уже грузятся из YCLIENTS с реальным ID.** `setServicesData` в `app.js` присваивает `id: String(s.id)` — то есть `state.serviceId` уже является **YCLIENTS-ID услуги**. Это снимает прежнее ограничение «ID услуг нестабильны» из ADR-001 (оно касалось привязки фото, не записи) и делает возможным запрос мастеров по `service_ids`.
- **Мастера грузятся одним общим запросом.** `app.js` зовёт `GET /book_staff/{company}` (без `service_ids`) при старте и кладёт результат в `MASTERS_DATA`. `id` мастера = YCLIENTS staff id.
- **Бэкенда нет, стек неизменен.** Vanilla JS/HTML/CSS, статический хостинг из `docs/`, единственный внешний источник — YCLIENTS API, без рантайм-npm-зависимостей (НФТ-4).
- **Отзывчивость на мобильном.** Фильтрация не должна заметно задерживать открытие `s-masters`; на время загрузки допустим индикатор (НФТ-2).
- **Деградация при ошибке API.** Принятое бизнес-решение — fail-open (BA, раздел 8, вопрос 1).
- **PWA/офлайн не должен регрессировать** (НФТ-5); влияние на `CACHE_VERSION` — предмет проектирования.

Решение охватывает два связанных вопроса: каким способом получить связь «мастер ↔ услуга» и как организовать поток данных/деградацию в `renderMasters`. Подробности вёрстки, имена функций и тексты сообщений — в спецификации релиза, не здесь.

YCLIENTS предоставляет два релевантных эндпоинта (BA, допущение 6):
- `GET /book_staff/{company}?service_ids[]={id}` — мастера, оказывающие услугу (фильтрация на стороне YCLIENTS).
- `GET /book_services/{company}?staff_id=N` — услуги конкретного мастера.

## Рассматриваемые варианты

### Вопрос 1. Как получить связь «мастер ↔ услуга»

#### Вариант A: «По услуге» — один запрос `GET /book_staff/{company}?service_ids[]={selectedServiceId}`

При открытии `s-masters` (режим от услуги) делается один запрос `book_staff`, параметризованный ID выбранной услуги. YCLIENTS возвращает список мастеров, оказывающих эту услугу. Состав списка `s-masters` строится из пересечения этого ответа с уже загруженным `MASTERS_DATA` (по `id`).

- **Плюсы:**
  - **Один сетевой запрос на открытие экрана** — минимальная задержка и трафик (НФТ-2).
  - Фильтрация выполняется **на стороне YCLIENTS** — это и есть «источник истины» (ФТ-2, НФТ-1), без клиентских эвристик и сопоставления по имени.
  - Тот же эндпоинт, что уже используется в `app.js` для общей загрузки мастеров (`/book_staff`) — переиспользуется существующая интеграция, никаких новых форматов данных.
  - `state.serviceId` уже = YCLIENTS-ID услуги, дополнительного маппинга не нужно.
  - Естественно реализует пустое состояние (ФТ-6) и fail-open (пустой/ошибочный ответ → решение по деградации в одном месте).
- **Минусы:**
  - Зависимость от поддержки `service_ids` в `/book_staff` (по BA-допущению поддерживается; проверяется на этапе реализации).
  - Возвращается отдельный (отфильтрованный) список мастеров, который нужно сопоставить с `MASTERS_DATA`, чтобы переиспользовать уже подготовленные карточки (аватар, избранное, градиент). Сопоставление по `id` — детерминированное и дешёвое.
- **Последствия:** в `renderMasters` появляется асинхронный шаг (запрос + индикатор загрузки); фильтрация по `svc.cat`/`cats` в режиме «от услуги» удаляется.

#### Вариант B: «По мастерам» — `GET /book_services/{company}?staff_id=N` для каждого мастера

Для каждого мастера из `MASTERS_DATA` запрашиваются его услуги (N параллельных запросов), затем клиентская фильтрация: оставить тех, у кого выбранная услуга присутствует.

- **Плюсы:**
  - Тоже опирается на YCLIENTS как источник истины.
  - Попутно даёт полный список услуг каждого мастера (может пригодиться в будущем).
- **Минусы:**
  - **N параллельных запросов** на каждое открытие `s-masters` — кратно больше трафика и нагрузки; на мобильном риск ощутимой задержки и частичных отказов (НФТ-2). Деградация при частичном отказе неоднозначна (часть мастеров «без данных»).
  - Клиентская фильтрация дублирует то, что YCLIENTS умеет сам (Вариант A).
  - Сложнее логика fail-open: что показывать, если упала часть запросов.
- **Последствия:** избыточная сложность и сетевые издержки без выигрыша для данной задачи.

#### Вариант C: «Предзагрузка» — при старте приложения грузить услуги каждого мастера в state

При `initApp` для каждого мастера загрузить `book_services?staff_id=N` и сохранить карту «мастер → услуги» в `state`. На `s-masters` фильтрация синхронная по этой карте.

- **Плюсы:**
  - На открытии `s-masters` нет сетевого запроса — фильтрация мгновенная.
  - Источник истины — YCLIENTS.
- **Минусы:**
  - **N запросов при старте приложения** — замедляет загрузку всего приложения ради экрана, на который пользователь может не зайти; противоречит духу НФТ-2 (издержки переносятся на старт, а не устраняются).
  - Данные могут устаревать в рамках сессии (расписание услуг изменилось у администратора) — рассинхрон, который мы как раз устраняем (НФТ-1).
  - Раздувает `state` и усложняет управление жизненным циклом данных без бэкенда.
- **Последствия:** преждевременная оптимизация; усложнение `state` и `initApp`.

### Вопрос 2. Деградация, кэширование и судьба поля `cats`

#### Вариант 2A (выбран): fail-open на уровне ответа `book_staff`, кэш ответа на время сессии, `cats` — удалить из критерия фильтрации

- При успешном непустом ответе — список = пересечение ответа с `MASTERS_DATA`.
- При ошибке/таймауте/«неуспехе» ответа — **fail-open**: показать всех мастеров из `MASTERS_DATA` (BA, вопрос 1).
- При успешном **пустом** ответе — пустое состояние с сообщением; кнопка «Любой мастер» **скрывается** (BA, вопрос 2).
- Результат запроса по конкретному `serviceId` кэшируется в памяти на время сессии (повторное открытие того же экрана не делает повторный запрос) — снимает задержку при возврате (НФТ-2).
- Поле `cats` **исключается из критерия фильтрации** на `s-masters`. Само поле в `MASTERS_DATA` может остаться (используется косвенно), но **не является источником фильтрации** (ФТ-2). Его роль legacy-фолбэка не нужна, т.к. fail-open показывает всех, а не «по категории».

- **Плюсы:** один источник истины (YCLIENTS), предсказуемая деградация, отзывчивость при повторных открытиях, отсутствие хрупкой привязки по имени/категории.
- **Минусы:** кэш в памяти теряется при перезагрузке (приемлемо — данные свежие на каждую сессию).

#### Вариант 2B (отвергнут): `cats` как клиентский фолбэк при ошибке API

При ошибке API фильтровать по статическому `cats` вместо показа всех.

- **Минусы:** возвращает рассинхрон, который мы устраняем (НФТ-1); для мастеров без статики `cats === []` → исчезли бы из списка при ошибке — это противоречит принятому fail-open (показывать всех). Сохраняет хрупкую привязку по имени.

## Решение

- **Вопрос 1 — выбран Вариант A** («по услуге», один запрос `GET /book_staff/{company}?service_ids[]={state.serviceId}`). Это единственный вариант, который при одном сетевом запросе на открытие экрана переносит фильтрацию на сторону YCLIENTS (источник истины, ФТ-2/НФТ-1), переиспользует уже существующую интеграцию `book_staff` и не вводит новых форматов данных. `state.serviceId` уже равен YCLIENTS-ID услуги, маппинг не нужен.
- **Вопрос 2 — выбран Вариант 2A**: fail-open при ошибке (показать всех), пустое состояние + скрытие «Любой мастер» при пустом ответе, кэш ответа на время сессии, `cats` исключается из критерия фильтрации `s-masters`.

Отвергнуты: B (N запросов на каждое открытие, избыточная сложность fail-open, дублирование серверной фильтрации), C (N запросов на старте, устаревание данных в сессии, раздувание `state`), 2B (возвращает рассинхрон и хрупкую привязку по имени, конфликтует с принятым fail-open).

### Границы решения

- Фильтрация применяется **только** в режиме «от услуги» на `s-masters`. Режимы `masterPreSelected` (ФТ-5) и `_browseAllMasters`/«Все мастера студии» (ФТ-8) — **без изменений**, без запроса `service_ids` и без фильтрации.
- Фильтрация — **по факту оказания услуги**, независимо от доступности слотов сегодня (BA, вопрос 3). Поле `avail`/`availText` и затенение недоступных карточек — текущее поведение, не меняется.
- Внешний вид карточек (`_masterCardHtml`) и логика бронирования после выбора — не меняются (ФТ-7).

## Последствия

- **Положительные:**
  - Состав `s-masters` соответствует реальным данным YCLIENTS об услугах мастера (ФТ-2, НФТ-1); устранён рассинхрон со статической таблицей.
  - Минимум сетевых издержек: один запрос на открытие, кэш в сессии (НФТ-2).
  - Переиспользована существующая интеграция `book_staff`; новых рантайм-зависимостей и форматов данных нет (НФТ-4).
  - Предсказуемая деградация: fail-open при ошибке, понятное пустое состояние (НФТ-3, ФТ-6).
- **Отрицательные / компромиссы:**
  - На открытие `s-masters` добавляется асинхронный шаг с индикатором загрузки (раньше рендер был синхронным).
  - Кэш фильтра — в памяти, сбрасывается при перезагрузке страницы (приемлемо: гарантирует свежесть данных на сессию).
- **Технический долг:**
  - Поле `cats` в `MASTERS_DATA` и его наполнение по совпадению имени в `app.js` становятся неиспользуемыми как критерий фильтрации. Полная очистка `cats` из модели данных — отдельная задача рефакторинга (вне обязательного объёма этого релиза, чтобы не задеть смежные использования). Возврат — при следующем касании `MASTERS_DATA`.
  - Зависимость от поддержки `service_ids` в `/book_staff` — если YCLIENTS изменит контракт, потребуется пересмотр (митигируется fail-open).

## Влияние на код

- **`docs/modules/api.js`** — точка получения мастеров по услуге. Либо новая функция-обёртка (например, `fetchStaffByService(serviceId)`) над `YC.get('/book_staff/...', { 'service_ids[]': serviceId })`, либо прямой вызов из места загрузки. Изоляция запроса упрощает кэширование и fail-open.
- **`docs/modules/masters.js`** — `renderMasters` в режиме «от услуги» становится асинхронным: показывает индикатор загрузки, запрашивает мастеров по услуге, строит список из пересечения с `MASTERS_DATA`, применяет fail-open и пустое состояние, управляет видимостью кнопки «Любой мастер». Прежняя фильтрация по `svc.cat`/`m.cats` удаляется. Ветки `browseAll` и `masterPreSelected` не затрагиваются. `_masterCardHtml`, сортировка по избранному, `selectMaster`/`selectAnyMaster` — без изменений.
- **`docs/modules/state.js`** — поле `cats` перестаёт быть критерием фильтрации. По решению архитектора само поле может остаться как есть (не удалять в этом релизе, чтобы не задеть смежный код), но новый критерий на него не опирается. При необходимости — место для кэша фильтра (в памяти).
- **`docs/app.js`** — общая загрузка мастеров `GET /book_staff` (без `service_ids`) при `initApp` сохраняется как источник полного `MASTERS_DATA` (нужно для режима «Все мастера» и для обогащения карточек). Изменения минимальны или отсутствуют.
- **`docs/sw.js`** — поднятие `CACHE_VERSION` только при фактическом изменении состава `APP_SHELL` (имена существующих модулей не меняются; если перечень кэшируемых файлов не изменился — `CACHE_VERSION` не трогать, см. спецификацию).
- **Новые рантайм-зависимости:** нет.
- **Что ломается:** меняется только **состав** списка на `s-masters` в режиме «от услуги» (намеренно, по требованию). Контракты навигации, бронирования, режимов `masterPreSelected` и «Все мастера», внешний вид карточек — не ломаются.
