# ADR-029: Ревизия контактного блока «О студии» — данные только из YCLIENTS `/companies?id=`, выбор карты

- **Дата:** 2026-05-30
- **Статус:** принято (вытесняет соответствующую часть ADR-025)
- **Связанные BA-требования:** `ba-req/ba-requirements-032.md` (источник `requirements/req-studio-contacts-data-031.md`)
- **Связанные ADR:** ADR-025 (первая версия контактного блока — частично **superseded**, см. ниже), ADR-024 (глобальный `user-select:none` — не отменяется, сохраняется локальное исключение)

## Контекст

Контактный блок «О студии» (`s-confirm`, `s-profile`) выпущен в v1.23.0 по ADR-025
на гибридной модели данных «свежий API `/company/{id}` → кэш `yc_company` →
плейсхолдеры `STUDIO`», с построением ссылок карт через `REVIEW_URLS`. Боевая
проверка 2026-05-30 (см. BA-032, разделы 1 и 6) выявила, что при пустых данных
API гибрид подставляет **недостоверные** значения:

1. Координат из `/company/{id}` нет (`0,0`) → фолбэк `STUDIO.mapYandex =
   REVIEW_URLS.yandex` → кнопка «Маршрут» ведёт на форму отзыва о **чужой**
   студии (Троицк), а не маршрут к нужной студии.
2. Partner-токен не отдаёт телефон (`phone:""`, `phones:[]`) → показывается
   плейсхолдер `+7 495 123-45-67` как кликабельный `tel:` → звонок в никуда.
3. Город теряется: показывается `address` без `city`.

Ключевой проверенный факт, меняющий архитектуру данных: endpoint
**`GET /companies?id={id}`** на ту же компанию `1940850` отдаёт **реальные
координаты** (`coordinate_lat=55.686006`, `coordinate_lon=37.85368`) и весь
нужный состав одним запросом (`title`, `public_title`, `address`, `city`,
`schedule`, `site`, `social`), тогда как `/company/{id}` возвращает `0,0` и
пустые поля. PUT настроек partner-токеном недоступен — редактирование вне scope.
Поля метро в API нет ни на одном проверенном endpoint.

Эта ревизия — целенаправленная смена принципа данных: «только YCLIENTS, честный
прочерк при отсутствии», а не гибрид с выдуманными значениями. Архитектура
проекта не меняется: vanilla JS, без рантайм-зависимостей, код в `docs/`.

### Что именно из ADR-025 вытесняется

| Решение ADR-025 | Статус | Замена в ADR-029 |
|---|---|---|
| Источник `GET /company/{id}` | **отменено** | `GET /companies?id={id}` |
| Гибрид «API → кэш → `STUDIO`», слияние с плейсхолдерами | **отменено** | «API → кэш → прочерк `—`»; `STUDIO` не источник |
| Плейсхолдер телефона `+7 495 123-45-67`, `tel:` всегда | **отменено** | телефон только из API; пусто → `—` без `tel:` |
| `REVIEW_URLS` как фолбэк ссылок карт в плашке | **отменено** | `REVIEW_URLS` только в `review.js`; карты строятся по координатам/адресу |
| «Маршрут» + «2ГИС» (две ссылки) | **отменено** | одна кнопка «Показать на карте» + bottom-sheet выбора |
| Состав: название/адрес/телефон/часы | **расширено** | + город в адресе, + Сайт, + Telegram |
| Единый рендер `renderStudioContacts({variant})` на два экрана | **сохранено** | сохраняется |
| Локальное исключение `user-select:text` для плашки | **сохранено** | сохраняется |
| Кэш `localStorage` `yc_company` | **сохранено, расширено** | + `city`, `site`, `social`, координаты |
| Загрузка в `Promise.all` `initApp()` | **сохранено** | сохраняется, меняется только endpoint |

Объект `STUDIO` в `constants.js` теряет роль источника данных плашки. Его можно
удалить целиком (после проверки, что других потребителей нет) — это решение
закреплено ниже как часть выбранного варианта.

## Рассматриваемые варианты

### ОВ-1. Компонент bottom-sheet выбора карты

#### Вариант A: переиспользовать существующий паттерн листа (выбран)
В проекте уже есть паттерн bottom-sheet (`#consentModal` в `prototype.html`:
backdrop `position:absolute;inset:0`, sheet `border-radius:20px 20px 0 0`,
`align-items:flex-end`, закрытие по клику на фон и `event.stopPropagation()` на
листе). Новый лист выбора карты строится по тому же паттерну: один контейнер в
`prototype.html`, наполнение пунктов — динамически из `studio.js`.
- Плюсы: визуальная и поведенческая консистентность; ноль новых абстракций;
  знакомый паттерн для ревьюера; минимум кода.
- Минусы: ещё один статический контейнер в `prototype.html`.

#### Вариант B: новый универсальный компонент-лист
- Плюсы: переиспользуемость в будущем.
- Минусы: преждевременная абстракция, больше кода, нарушает минимальность
  изменений; единственный потребитель сейчас — выбор карты.

### ОВ-2. Формат отображения телефона при появлении в API

#### Вариант A: телефон как текст, кликабельный `tel:` при непустом значении (выбран)
Сейчас телефон в API пуст → строка `Телефон: —`, без ссылки. Когда телефон
появится — показывать значение «как есть» из API (без переформатирования) и
делать его кликабельным `tel:` с нормализацией номера (`[^+\d]` → удаляется) в
href, текст — оригинальная строка из API.
- Плюсы: реальный телефон полезно набрать в одно нажатие; нормализация только в
  href исключает битый `tel:`; отображаемый текст не искажается.
- Минусы: правило «без `tel:`» из исходного требования относилось к
  плейсхолдеру; для реального номера `tel:` уместен. Зафиксировано явно.

#### Вариант B: телефон всегда только текст, без `tel:`
- Минусы: теряется полезное действие «позвонить»; UX хуже без выигрыша в
  достоверности (номер-то реальный).

### ОВ-3. Поведение Yandex Navigator без координат

#### Вариант A: маршрут по координатам, поиск по тексту без координат (выбран)
- С координатами: `yandexnavi://build_route_on_map?lat_to=<lat>&lon_to=<lon>`.
- Без координат, но с адресом: `yandexnavi://map_search?text=<urlencode(адрес)>`.
- Плюсы: единое поведение со всеми картами (фолбэк по адресу), маршрут там, где
  возможен; деградация управляемая.
- Минусы: на desktop app-схема может не сработать — ожидаемо и допустимо (BA,
  допущения).

#### Вариант B: скрывать Navigator без координат
- Минусы: непоследовательно (остальные карты умеют искать по адресу); лишняя
  условная логика в листе.

### ОВ-4. Прочие соцсети (`social.viber` заполнен)

#### Вариант A: только Сайт + Telegram (выбран — решение заказчика)
- Плюсы: соответствует BA (ФТ-9, out of scope); минимум строк, чистый блок.
- Минусы: `viber` не показывается, хотя заполнен — сознательно вне scope.

#### Вариант B: показывать все непустые соцсети
- Минусы: противоречит решению заказчика; раздувает блок.

### ОВ-5. Схема расширенного кэша `yc_company` и совместимость

#### Вариант A: расширить нормализованный объект, читать старый формат как частичный (выбран)
Кэшируется единый нормализованный объект новой схемы (см. «Решение»). При чтении
старого формата v1.23.0 (поля `title/address/phone/schedule/lat/lon`)
отсутствующие новые поля (`city`, `site`, `telegram`, `publicTitle`) трактуются
как пустые → прочерк/скрытие соответствующих строк. Перезапись кэша происходит
при первом успешном ответе нового endpoint.
- Плюсы: обратная совместимость без миграции и без сброса ключа; старый кэш
  деградирует мягко и быстро обновляется при первом онлайн-старте.
- Минусы: до первого онлайн-старта новые поля у старого кэша пустые — приемлемо.

#### Вариант B: сменить ключ кэша (`yc_company_v2`)
- Минусы: оставляет «мусорный» старый ключ; лишняя сложность; выгоды нет, формат
  совместим по чтению.

## Решение

Выбраны варианты A по всем пяти открытым вопросам. Принцип данных:
**свежий API `/companies?id=` → кэш `yc_company` → прочерк `—`**. Объект `STUDIO`
и `REVIEW_URLS` исключаются из плашки как источники.

### Нормализованная схема данных компании (память и кэш `yc_company`)

```
{
  title:        string,   // company.title (trim)
  publicTitle:  string,   // company.public_title (trim)
  address:      string,   // company.address (trim)
  city:         string,   // company.city (trim)
  phone:        string,   // первый непустой phones[] / phone (trim), может быть ''
  schedule:     string,   // company.schedule (trim)
  site:         string,   // company.site (trim)
  telegram:     string,   // social.telegram, handle без ведущего '@'
  lat:          number,   // coordinate_lat || 0
  lon:          number,   // coordinate_lon || 0
}
```

`setCompanyData(raw)` извлекает `raw.data || raw` (ответ `/companies?id=` —
объект компании или `{data:{...}}`/массив с одним элементом; нормализатор
обязан корректно достать единственную компанию). Кэш перезаписывается, если
ответ принёс хотя бы одно непустое поле или валидные координаты.

`getCompanyContacts()` отдаёт **отображаемые** значения: для каждого поля —
значение из памяти/кэша, иначе пустая строка; финальное превращение пустого в
`—` выполняет рендер (метка сохраняется). Слияния с `STUDIO` нет.

### Состав и правила рендера `renderStudioContacts({variant})`

| Строка | Значение | Пусто |
|---|---|---|
| Заголовок | `publicTitle` \|\| `title` | `—` |
| Адрес | `[city, address].filter(непустые).join(', ')` | `—` |
| Телефон | текст из API; `tel:` с нормализацией в href при непустом | `—`, без ссылки |
| Часы | `schedule` как есть | `—` |
| Сайт | ссылка, текст без `https?://` и завершающего `/` | строка **скрыта** |
| Telegram | ссылка `https://t.me/<telegram>`, текст `@<telegram>` | строка **скрыта** |

Метро не выводится. Адрес/телефон/часы — всегда присутствуют (с `—` при пустом).
Сайт и Telegram — присутствуют только при непустом значении (строка не
рендерится вовсе). Все динамические значения через `esc`; внешние http(s)-ссылки
— `target="_blank" rel="noopener"`.

### Кнопка карты и bottom-sheet

Одна кнопка «Показать на карте» вместо «Маршрут» + «2ГИС». Видимость: кнопка
показывается, если есть валидные координаты **или** непустой адрес; иначе
скрыта (ФТ-15). По нажатию открывается bottom-sheet (паттерн `#consentModal`) с
пунктами Apple Maps / Google Maps / Yandex Navigator / Yandex Maps + «Отмена».
Все четыре варианта показываются всегда.

Построение ссылок — функция в `studio.js`, возвращающая URL для каждого
провайдера по приоритету «координаты → адрес»:

При валидных координатах (оба ≠ 0):
- Apple Maps: `https://maps.apple.com/?ll=<lat>,<lon>&q=<urlencode(title)>`
- Google Maps: `https://www.google.com/maps/search/?api=1&query=<lat>,<lon>`
- Yandex Maps: `https://yandex.ru/maps/?pt=<lon>,<lat>&z=17&l=map`
- Yandex Navigator: `yandexnavi://build_route_on_map?lat_to=<lat>&lon_to=<lon>`

При отсутствии координат, но непустом адресе (`text = urlencode([city,address])`):
- Apple Maps: `https://maps.apple.com/?q=<text>`
- Google Maps: `https://www.google.com/maps/search/?api=1&query=<text>`
- Yandex Maps: `https://yandex.ru/maps/?text=<text>`
- Yandex Navigator: `yandexnavi://map_search?text=<text>`

Web-ссылки — `target="_blank" rel="noopener"`. App-схема `yandexnavi://` —
обычный `href` (без `target="_blank"`). Координаты считаются валидными, если оба
численно конечны и ≠ 0.

### Источники и оффлайн

Приоритет: свежий ответ `/companies?id=` (память) → кэш `yc_company` → пусто
(прочерк/скрытие). Отрисовка плашки не зависит от успеха запроса; ошибка/пустой
ответ не ломают `s-confirm`/`s-profile`. Кэш хранит только публичные поля.

### Загрузка endpoint

В `Promise.all` `initApp()` (`app.js` ~167-172) запрос `/company/${YC.company}`
заменяется на `/companies?id=${YC.company}` с тем же `.catch(() => ({}))`;
результат передаётся в `setCompanyData()`.

### `user-select`

Локальное исключение `.studio-contacts { user-select:text }` сохраняется
(ADR-024 не отменяется). Bottom-sheet наследует общие правила.

## Последствия

- **Положительные:** плашка показывает только достоверные данные студии; кнопка
  карты ведёт к нужной точке по реальным координатам с выбором приложения;
  город виден; нет фейкового телефона; «оживает» автоматически при заполнении
  админки; единый рендер; оффлайн-устойчивость через кэш; ноль новых
  рантайм-зависимостей.
- **Отрицательные / компромиссы:** на desktop app-схема Navigator может не
  открыться (ожидаемо); до первого онлайн-старта старый кэш v1.23.0 даёт пустые
  новые поля (прочерки) — самоисправляется; один дополнительный контейнер
  bottom-sheet в `prototype.html`.
- **Технический долг:** телефон остаётся `—` до появления в API (вне нашего
  контроля — partner-токен/настройки студии); запись настроек в YCLIENTS
  недоступна; `viber` и прочие соцсети сознательно не выводятся.
