# ADR-033: Персистентный кэш каталога и цен (stale-while-revalidate без TTL) + троттлинг фоновых обновлений главной

- **Дата:** 2026-05-31
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-036.md` (источник `requirements/req-perf-startup-optimization-035.md`)
- **Связанные ADR:** ADR-012 (диапазон цен — частично **вытесняется** в части хранения кэша цен, см. ниже), ADR-014 (SWR в Service Worker для фото — справочно, паттерн), ADR-025 (агрегированная загрузка справочников в `initApp` — контекст стартового цикла)

## Контекст

Холодный старт и навигация PWA воспринимаются медленными по трём причинам, подтверждённым кодом:

1. **N+1 запросов цен мастеров при старте.** `_collectStaffPrices` (`app.js:233-266`) шлёт `GET /book_services/{company}?staff_id=` на каждого мастера; результат (`servicePriceRange`, `staffServicePrice`) живёт **только в памяти** (`state.js:29-30`) и теряется между запусками — каждый старт повторяет весь сбор.
2. **Каталог не переживает запуск.** Услуги/мастера/категории грузятся из сети в `initApp` (`app.js:162-231`); SW намеренно не кэширует `api.yclients.com` (`NETWORK_ONLY_HOSTS`, `sw.js:47`). При повторном холодном старте каталог пуст до прихода ответа.
3. **Избыточные запросы на каждом входе на `s-home`.** Обработчик (`app.js:58-71`) безусловно идёт за записями (`_fetchAndMergeServerRecords`) и за GitHub-данными (`_ghRead`), причём `_ghRead` принудительно обходит кэш через `?_=Date.now()` (`github.js:11`).

Бизнес-цель — ускорить ощущаемый запуск и навигацию **без изменения видимого поведения, UI и логики бронирования** (ФТ-8, НФТ-2). Силы: vanilla JS без рантайм-зависимостей; код только в `docs/app.js` + `docs/modules/*.js` (+ `prototype.html`/`sw.js` при необходимости); запись в localStorage — в `try/catch`; кэш не должен содержать YCLIENTS-токен; данные каталога почти статичны.

Решение охватывает развилки BA §8: ОВ-1 (пересмотр ADR-012 о persist цен), ОВ-2 (хранилище кэша), ОВ-3 (TTL/инвалидация), ОВ-4 (устранение N+1), ОВ-5 (окно троттлинга `s-home`), ОВ-6 (`_ghRead` cache-bust), ОВ-7 (объём опциональных пунктов 4-5).

### Отношение к ADR-012 (ОВ-1)

ADR-012 (раздел «Границы решения», «Технический долг») **сознательно постановил**: кэш цен (`servicePriceRange`, `staffServicePrice`) живёт **в памяти на сессию**, а «persist кэша цен и его инвалидация — **не делается**; при необходимости — отдельным решением». Настоящее BA-требование (ФТ-4) именно вводит этот persist. Это **не противоречие, а целенаправленный пересмотр отложенного техдолга**: ADR-033 **вытесняет** ADR-012 **только в части хранения кэша цен** (memory-only → memory + localStorage-снимок). Всё остальное ADR-012 (схема сбора цен `book_services?staff_id=` по мастерам, агрегация min/max, фолбэки `_fmtPriceRange`/`getStaffPrice`, формат диапазона, graceful degradation) **остаётся в силе и не пересматривается**.

## Рассматриваемые варианты

### Вопрос 1 — Хранилище персистентного кэша (ОВ-2)

#### Вариант 1A (выбран): localStorage
- Плюсы: синхронное чтение → мгновенный первый кадр без `await`; данные малы (≈13 услуг, ≈4 мастера, цены); уже используется во всём коде, запись штатно в `try/catch` (`api.js:108`).
- Минусы: ~5 МБ лимит, синхронность блокирует поток на запись (объём ничтожен — некритично).
- Последствия: новый модуль чтения/записи снимка; ключи в неймспейсе `yc_*`.

#### Вариант 1B (отвергнут): IndexedDB
- Плюсы: больше места, асинхронность; уже используется для inbox.
- Минусы: асинхронное чтение **ломает мгновенный синхронный первый кадр** (нужен `await` до рендера) — прямо против ФТ-1/КП-1; избыточно для килобайт данных.

### Вопрос 2 — Модель инвалидации (ОВ-3)

#### Вариант 2A (выбран): чистый stale-while-revalidate без TTL
- Кэш используется **только для первого кадра**; при каждом старте безусловно запускается фоновое обновление из сети и заменяет данные по приходу ответа. Снимок перезаписывается после успешного сетевого ответа.
- Плюсы: нет «протухших» данных в видимом результате (сеть всегда догоняет); нет логики таймстемпов свежести/сравнения версий; согласуется с уже принятым SWR в SW (ADR-014).
- Минусы: при каждом старте всё равно идёт сетевой сбор (но **не в критическом пути** рендера) — приемлемо, цель в ощущаемой скорости, а не в экономии трафика.

#### Вариант 2B (отвергнут): TTL на снимок
- Плюсы: можно пропускать фоновое обновление, если кэш «свежий».
- Минусы: вводит риск показа устаревших цен/услуг между TTL-окнами; усложняет инвалидацию; BA §6 явно фиксирует «без TTL»; данные почти статичны — выигрыш от пропуска обновления не оправдывает риск рассинхрона.

### Вопрос 3 — Устранение N+1 цен (ОВ-4)

#### Вариант 3A (выбран): кэшировать результат сбора цен + фоновое неблокирующее обновление существующими пер-мастерными запросами
- Снимок `servicePriceRange`/`staffServicePrice` восстанавливается синхронно при старте → диапазоны видны сразу; `_collectStaffPrices` остаётся, но выполняется **в фоне** и по завершении перезаписывает снимок.
- Плюсы: ноль новых эндпоинтов; полностью наследует схему ADR-012 (A1/B1); минимальные изменения.
- Минусы: при первом запуске (нет снимка) поведение как сегодня — N запросов, но фоном (B1) — приемлемо.

#### Вариант 3B (отвергнут): batch-эндпоинт цен по всем мастерам
- Минусы: в YCLIENTS такого эндпоинта не подтверждено; ввод нового контракта вне объёма BA (§2); кэш + фон решает задачу без него.

### Вопрос 4 — Окно троттлинга `s-home` (ОВ-5)

#### Вариант 4A (выбран): 60 с, timestamp в памяти модуля; старт обновляет всегда
- В обработчике `s-home` фоновые запросы (`_fetchAndMergeServerRecords`, `_ghRead`) выполняются не чаще 1 раза в 60 с; первый вход и холодный старт (`checkSession`/`initApp`) — всегда.
- Плюсы: убирает повторные запросы при «прыжках» по вкладкам в одной сессии; timestamp в памяти — без записи в storage, обнуляется при перезапуске (старт всё равно обновит).
- Минусы: окно фиксировано (60 с) — данные записей могут «отставать» до минуты при активной навигации; приемлемо (записи и так фоновые, рендер идёт из локального состояния).

#### Вариант 4B (отвергнут): persist timestamp в localStorage
- Минусы: усложнение; между запусками троттлинг не нужен (старт обязан обновлять), а в рамках сессии достаточно памяти.

### Вопрос 5 — Cache-bust в `_ghRead` (ОВ-6)

#### Вариант 5A (выбран): убрать `?_=Date.now()`, положиться на HTTP-кэш + троттлинг (Вопрос 4)
- Плюсы: перестаёт принудительно обходить кэш браузера на каждом входе; актуальность регулируется троттлингом частоты обращений (ФТ-7).
- Минусы: возможен показ кэшированного браузером GitHub-ответа до истечения его собственного кэша — приемлемо (контент ленты почти статичен, обновляется при следующем не-троттленном входе/старте).

#### Вариант 5B (отвергнут): оставить cache-bust + троттлинг
- Минусы: cache-bust сводит на нет HTTP-кэш; BA §8 ОВ-6 предпочитает убрать.

### Вопрос 6 — Объём опциональных пунктов 4-5 (ОВ-7)

#### Вариант 6A (выбран): включить ФТ-11 (сокращение ре-рендеров `renderServices`), **исключить** ФТ-12 (ленивый EmailJS) из v1.31.0
- ФТ-11 — низкорисковая локальная правка `initApp` (устранение дублей `renderServices()`), органично укладывается в основную работу и не меняет итоговое отображение.
- ФТ-12 (ленивый EmailJS) затрагивает 5 мест (`emailjs.init` в `app.js:157` + 4 вызова `emailjs.send` в `auth.js`) + `prototype.html`, требует загрузчика-обёртки и аккуратной обработки ошибок загрузки CDN в каждом email-сценарии. Это отдельная единица риска поверх архитектурного изменения кэша.
- Плюсы: фокус релиза на основной архитектуре кэширования; меньше поверхность регрессий; EmailJS выносится отдельным требованием/релизом.
- Минусы: один сторонний скрипт продолжает грузиться на старте (`defer`, вне критического пути парсинга) — приемлемо до отдельного релиза.

#### Вариант 6B (отвергнут): включить оба опциональных пункта
- Минусы: совмещает два независимых риска (кэш + рефакторинг загрузки auth-сценария) в одном релизе; противоречит принципу минимальности; усложняет приёмку.

## Решение

- **Хранилище — localStorage** (1A): синхронный мгновенный первый кадр; данные малы; запись в `try/catch`.
- **Инвалидация — чистый SWR без TTL** (2A): снимок только для первого кадра, при каждом старте безусловно идёт фоновое обновление и перезаписывает снимок после успешного сетевого ответа.
- **N+1 — кэш + фоновое обновление** (3A): снимок цен восстанавливается синхронно; `_collectStaffPrices` остаётся фоновым и обновляет снимок.
- **Троттлинг `s-home` — 60 с, timestamp в памяти модуля** (4A); старт обновляет всегда.
- **`_ghRead` — убрать cache-bust** (5A); актуальность — через троттлинг.
- **Объём — ФТ-11 включён, ФТ-12 исключён** (6A): ленивый EmailJS выносится отдельным требованием.

### Модель кэша (нормативная)

- **Что кэшируется:** снимок каталога (нормализованные `SERVICES_DATA` — поля `id,name,cat,dur,price_min,priceStr,comment,photos`; нормализованные `MASTERS_DATA`), снимок цен (`servicePriceRange`, `staffServicePrice`). Категории отражены в `cat` услуг.
- **Что НЕ кэшируется:** YCLIENTS-токен и `userToken`, любые секреты, персональные данные клиента сверх уже хранимого сегодня (записи `yc_records`, OTP и т. п. остаются как есть и не расширяются).
- **Запись:** после **успешного** сетевого ответа `initApp` (каталог) и после завершения `_collectStaffPrices` (цены), всегда в `try/catch`; ошибка/переполнение → деградация до режима «без кэша» (ФТ-9).
- **Чтение:** синхронно при старте до сетевых запросов; при наличии снимка — заполнить `SERVICES_DATA`/`MASTERS_DATA`/`servicePriceRange`/`staffServicePrice` и отрисовать; затем сеть в фоне (SWR).
- **Версионирование снимка:** ключ снимка несёт версию схемы (например, суффикс `_v1`); при несовпадении версии снимок игнорируется (forward-compat без миграций).

### Границы решения

- Не меняется: видимый UI, состав экранов, контент, логика бронирования (логин → услуга → мастер → слот → бронь), экраны слотов, инбокс/push, стратегии SW для статики и фото (ADR-014), набор эндпоинтов YCLIENTS, способ хранения секретов/захардкоженный токен.
- ADR-012 пересматривается **только** в части persist кэша цен; схема сбора и форматирования цен сохраняется.
- ФТ-12 (ленивый EmailJS) — вне этого релиза.

## Последствия

- **Положительные:** мгновенный первый кадр каталога и цен при повторном старте (ФТ-1, ФТ-3, ФТ-4); работа каталога офлайн при наличии снимка (ФТ-5); меньше запросов при повторных входах на главную (ФТ-6) и без cache-bust для GitHub (ФТ-7); ноль новых зависимостей и эндпоинтов.
- **Отрицательные / компромиссы:** при каждом старте всё равно идёт фоновый сбор (не в критическом пути); GitHub-ответ может отдаваться из браузерного кэша до следующего не-троттленного обновления; запись снимка добавляет немного синхронной работы (объём ничтожен).
- **Технический долг:** ленивая загрузка EmailJS (ФТ-12) откладывается в отдельное требование/релиз; persist troттлинга и тонкая настройка окна (вне 60 с) — при необходимости отдельным решением; консолидация цены мастера с `openMasterCard` (наследие ADR-012) по-прежнему не делается.

## Влияние на код

- **`docs/modules/state.js`** — экспорт сеттеров для гидрации цен (`servicePriceRange`/`staffServicePrice` — уже module-level объекты, заполняются на месте через мутацию ключей; нужен способ их очистки/заполнения из снимка).
- **`docs/app.js`** — синхронная гидрация из снимка до сетевых запросов; запись снимка каталога после успешного `Promise.all`; запись снимка цен в конце `_collectStaffPrices`; устранение дублирующих `renderServices()` (ФТ-11); троттлинг в обработчике `s-home`.
- **`docs/modules/github.js`** — убрать `?_=Date.now()` в `_ghRead` (ФТ-7).
- **Новый модуль `docs/modules/dataCache.js`** (предпочтительно, минимальный): чтение/запись снимка каталога и цен в localStorage в `try/catch`, версионирование ключей. При изменении состава модулей в `APP_SHELL` (`sw.js:5-44`) — добавить файл и **поднять `CACHE_VERSION`** (`sw.js:1`, `v87`→`v88`).
- **`docs/sw.js`** — добавить `./modules/dataCache.js` в `APP_SHELL`; поднять `CACHE_VERSION`. Стратегии SW не меняются.
- **Не трогаются:** `prototype.html` (EmailJS остаётся — ФТ-12 вне релиза), `api.js` (контракт), логика слотов/бронирования, секреты/токен.
- **Новые рантайм-зависимости:** нет. Операции с веткой `backup` запрещены.
