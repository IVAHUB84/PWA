# ADR-011: Вёрстка экрана «Услуги» — карточки в один столбец с обложкой 16/9 и локальное выделение строки поиска

- **Дата:** 2026-05-29
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-012.md` (открытые вопросы №1–№4 закрыты решениями заказчика от 2026-05-29, §8)
- **Связанные ADR:** прямых пересечений с ADR-001…ADR-010 не выявлено. ADR-010 / ADR-009 (раздел «Лента» и его крупные карточки `.feed-card`) используются как **визуальный ориентир**, но не пересматриваются: настоящий ADR не меняет ни ленту, ни её стили.

## Контекст

Экран выбора услуги (`s-services`) — обязательный шаг основного сценария записи: клиент выбирает услугу и переходит к мастеру (`s-masters`) или слоту (`s-slots`). Сейчас каталог рендерится мелкими карточками в **два столбца**: `.svc-catalog { grid-template-columns: 1fr 1fr }` (`docs/style.css:261-265`), карточка `.svc-catalog-card` с обложкой `padding-top:75%` (`docs/style.css:267-304`), название 13px, мета 12px. Это плотно и расходится со сложившимся визуальным языком крупных карточек раздела «Лента» (`.feed-card`, один столбец, фото `aspect-ratio:16/9`, `docs/style.css:706-712`).

Строка поиска по услугам — общий класс `.search-bar` (`docs/style.css:252-257`, фон `var(--surface)`) — визуально сливается с подложкой экрана и плохо считывается как поле ввода. Тот же класс `.search-bar` переиспользуется на главной (`#homeSearchBar`, `docs/prototype.html:47`) поверх цветного хедера с инлайн-переопределениями фона/рамки и правилами `#homeSearchBar input` (`docs/style.css:258-259`).

**Бизнес-цель (BA, раздел 1):** повысить читаемость и привлекательность каталога услуг (крупные карточки в один столбец «как Лента») и сделать строку поиска по услугам заметной как точку входа в фильтрацию — при полном сохранении текущей функциональности экрана записи.

**Решения заказчика по открытым вопросам BA приняты 2026-05-29** (§8) и являются обязательными входными ограничениями (не пересматриваются):

1. **Формат карточки — обложка сверху** (как «Лента»): изображение во всю ширину карточки сверху, под ним название и мета «цена · длительность». Горизонтальная компоновка (миниатюра сбоку) отклонена.
2. **Пропорция обложки — `aspect-ratio:16/9`** (как `.feed-photo`), вместо текущего `padding-top:75%`.
3. **Выделение строки поиска локализуется под `s-services`** (отдельный селектор/идентификатор); **общий класс `.search-bar` и строка поиска на главной (`#homeSearchBar`) не трогаются**. Конкретный приём выделения (контрастный фон / рамка / тень) — на усмотрение архитектора/разработчика.
4. **Размеры текста в карточке** допустимо увеличить пропорционально; конкретные значения — деталь реализации.

Силы, действующие на решение:

- **Чисто презентационная задача (BA, scope §2).** Меняются только вёрстка и стили представления уже имеющихся данных каталога. Логика выбора услуги (`selectService`), поиска (`filterSearch`), фильтра по категории (`filterCategory`/`_openCatFilter`), баннер предвыбранного мастера (`_updateMasterBanner`) и ветка «услуги мастера» — переиспользуются как есть. Состав данных карточки (`name`, `priceStr`, `dur`, изображение из `resolveServiceImage`) не расширяется. **Архитектор подтверждает:** существующего состава данных карточки достаточно для крупной одностолбцовой карточки — дополнительные поля и запросы не требуются (BA §6, допущение, требующее подтверждения архитектора).
- **Единая точка рендера каталога — `_renderList`.** Разметку карточки задаёт `_renderList` (`docs/modules/services.js:86-99`), общая для основного каталога и режима предвыбранного мастера (оба пути сходятся в `_renderList`, `services.js:127` и `:137`). Правка вёрстки в одном месте автоматически охватывает обе ветки (ФТ-7).
- **Риск регрессии главной через общий `.search-bar`.** Глобальная правка `.search-bar` затронула бы `#homeSearchBar`. Решение №3 заказчика снимает риск, требуя **локализации** выделения под экран услуг без касания общего класса.
- **Сохранение `loading="lazy"` и плавности прокрутки (НФТ-5).** Переход на один столбец и крупные обложки не должен деградировать отрисовку; ленивая загрузка изображений сохраняется (она уже в `_renderList`, `services.js:89`).
- **Vanilla JS/CSS, минимальность (НФТ-1, НФТ-2), без рантайм-зависимостей.** Правки — в существующих файлах `APP_SHELL` (`style.css`, при необходимости `prototype.html` и точечно `services.js`). При изменении кэшируемых файлов — поднятие `CACHE_VERSION` в `sw.js` (НФТ-4).

Решение охватывает два инженерных вопроса в рамках принятой заказчиком формы: (A) как реализовать одностолбцовую карточку с обложкой 16/9 — переписать существующие стили `.svc-catalog`/`.svc-catalog-card` или переиспользовать классы `.feed-card`; (B) как локализовать выделение строки поиска под `s-services` без касания общего `.search-bar`. Конкретные значения отступов/размеров текста, цвет/тень выделения — детали реализации, фиксируются в спецификации релиза.

## Рассматриваемые варианты

> Вопросы «обложка сверху vs миниатюра сбоку», «пропорция 16/9», «локализовать выделение поиска под экран услуг», «размеры текста» — **закрыты решениями заказчика** (BA §8). Архитектор их не пересматривает; ниже — только инженерные развилки реализации в рамках принятой формы.

### Вопрос A. Как реализовать одностолбцовую карточку с обложкой 16/9

Цель — крупная карточка в один столбец «как Лента», без изменения данных и логики каталога.

#### Вариант A1 (выбран): переписать стили `.svc-catalog` / `.svc-catalog-card` / `.svc-cover`, сохранив текущие классы и HTML `_renderList`

Изменить CSS: `.svc-catalog` — один столбец (убрать `grid-template-columns: 1fr 1fr`, перейти на одну колонку), `.svc-cover` — `aspect-ratio:16/9` вместо `padding-top:75%`, увеличить `.svc-name`/`.svc-meta` пропорционально (BA №4). HTML в `_renderList` (`services.js:86-99`) остаётся прежним (те же классы `svc-catalog-card`/`svc-cover`/`svc-info`/`svc-name`/`svc-meta`, тот же `data-sid`/`onclick="selectService(...)"`, тот же `coverHtml` с `loading="lazy"`).

- **Плюсы:** минимальная площадь — правка преимущественно в `style.css`; JS-рендер `_renderList` не трогается (или трогается минимально), поэтому логика выбора/поиска/фильтра/баннера/ветки мастера гарантированно сохраняется; обе ветки каталога (основная и режим мастера) обновляются автоматически, т.к. сходятся в `_renderList`; классы остаются доменно-осмысленными (`svc-*` = услуги, не лента).
- **Минусы:** некоторое дублирование визуальных приёмов с `.feed-*` (обложка 16/9, скругление, тень) — приемлемо: это разные домены (услуги vs публикации) с разным составом и поведением (тап → запись vs тап → пост).
- **Последствия:** правка `style.css` (сетка, обложка, типографика карточки); `services.js`/`prototype.html` — без изменения логики (HTML карточки в `_renderList` сохраняет классы).

#### Вариант A2 (отвергнут): переиспользовать классы `.feed-card` / `.feed-photo` для карточек услуг

Рендерить карточку услуги теми же классами, что карточку ленты, ради «побайтового» сходства.

- **Минусы:** связывает два независимых домена (услуги и лента) общими классами — любая будущая правка ленты непреднамеренно меняет каталог услуг и наоборот (хрупкая связь, нарушение разделения ответственности). BA прямо трактует «как Лента» как **ориентир**, а не обязательное переиспользование классов (BA §6, допущение). Карточка услуги имеет иной состав (нет тега/кнопки «Записаться →»/полного текста) — переиспользование `.feed-body`/`.feed-tag`/`.feed-book` всё равно потребовало бы расхождений.
- **Последствия:** межоменная связанность и риск перекрёстных регрессий. Отвергнут.

#### Вариант A3 (отвергнут): новый отдельный набор классов `.svc-card-lg` рядом со старыми

Ввести новые классы для крупной карточки, оставив старые `.svc-catalog-card` мёртвым кодом.

- **Минусы:** мёртвый CSS (старые двухколоночные стили больше не используются), рост поверхности стилей без выгоды; требует синхронной правки имён классов и в `_renderList`. Старые классы и так используются только в `_renderList` — проще переопределить их, чем плодить новые (НФТ-2, минимальность).
- **Последствия:** мёртвый код и лишние касания JS. Отвергнут.

### Вопрос B. Как локализовать выделение строки поиска под `s-services` без касания общего `.search-bar`

Заказчик (№3) требует: выделение поиска **только** на экране услуг; общий `.search-bar` и `#homeSearchBar` — не трогать.

#### Вариант B1 (выбран): добавить идентификатор/локальный класс на контейнер строки поиска `s-services` и стилизовать через него

На блоке строки поиска экрана услуг (`docs/prototype.html:110`, `<div class="search-bar">`) добавить устойчивый якорь — отдельный `id` (например `serviceSearchBar`) **или** дополнительный класс-модификатор. Выделение (контрастный фон / заметная рамка / тень — деталь реализации, BA №3) задаётся новым селектором (`#serviceSearchBar` или `.search-bar--services`), который **дополняет**, а не изменяет общий `.search-bar`. Общий класс `.search-bar` и правила `#homeSearchBar` (`style.css:252-259`) остаются без изменений.

- **Плюсы:** нулевой риск регрессии главной — общий `.search-bar` и `#homeSearchBar` физически не редактируются (ФТ-9/КП-9 сводятся к «общий класс не изменён»); локальная специфичность нового селектора перекрывает базовый `.search-bar` только на экране услуг; согласуется с уже применённым в проекте приёмом точечного переопределения через `#homeSearchBar` (`style.css:258-259`).
- **Минусы:** дополнительный селектор в `style.css` и (при варианте с `id`) точечная правка одной строки разметки в `prototype.html`. Минимально.
- **Последствия:** правка `style.css` (новый селектор выделения) + при необходимости одна строка `prototype.html` (добавление `id`/класса на контейнер поиска `s-services`).

#### Вариант B2 (отвергнут): глобально изменить `.search-bar` и компенсировать вид главной отдельными правилами

Усилить общий `.search-bar`, затем «починить» `#homeSearchBar` дополнительными переопределениями.

- **Минусы:** прямо противоречит решению заказчика №3 («общий `.search-bar` и `#homeSearchBar` не трогать»); вносит риск визуальной регрессии главной (НФТ-3, КП-9) и связывает два экрана через общий класс. Несоразмерно простой локальной задаче.
- **Последствия:** риск регрессии и нарушение решения заказчика. Отвергнут.

#### Вариант B3 (отвергнут): инлайн-стили выделения прямо на `<div class="search-bar">` в `prototype.html`

Прописать фон/рамку инлайн на контейнере строки поиска экрана услуг (как сделано для `#homeSearchBar`, `prototype.html:47`).

- **Минусы:** смешивает стили с разметкой и плодит инлайн-CSS; хуже для поддержки, чем именованный селектор; для разных состояний (`:focus-within`, placeholder) инлайн неудобен. Прецедент `#homeSearchBar` — вынужденное наследие, не образец для нового кода.
- **Последствия:** ухудшение сопровождаемости. Отвергнут (предпочтён именованный селектор B1).

## Решение

- **Форма (BA №1, №2):** каталог услуг на `s-services` — **один столбец**, одна карточка на строку во всю ширину контентной области; карточка крупная, с обложкой сверху `aspect-ratio:16/9` (как `.feed-photo`), под обложкой — название и мета «цена · длительность». Состав карточки не расширяется (обложка/плейсхолдер из `resolveServiceImage`, название, мета).
- **Вопрос A — Вариант A1:** реализовать через **переопределение существующих стилей** `.svc-catalog` (один столбец), `.svc-cover` (`aspect-ratio:16/9`), `.svc-name`/`.svc-meta` (увеличенная типографика, BA №4) в `style.css`. HTML-разметка `_renderList` (`services.js:86-99`) сохраняет текущие классы, `data-sid`, `onclick="selectService(...)"`, `loading="lazy"` и структуру `coverHtml` — логика и состав данных не меняются. Обе ветки каталога (основная и режим предвыбранного мастера) обновляются автоматически через общий `_renderList`.
- **Вопрос B — Вариант B1:** выделение строки поиска **локализуется** под `s-services` отдельным якорем (новый `id`, например `serviceSearchBar`, или класс-модификатор на контейнере `prototype.html:110`) и новым селектором в `style.css`. **Общий класс `.search-bar` и правила `#homeSearchBar` не изменяются.** Конкретный приём выделения (контрастный фон / рамка / тень) — деталь реализации в спецификации; цель — поле явно различимо на фоне экрана, плейсхолдер «Поиск услуг…» и иконка читаемы.

Отвергнуты: A2 (межоменная связанность с `.feed-*`), A3 (мёртвый CSS и лишние касания JS), B2 (нарушение решения №3 и риск регрессии главной), B3 (инлайн-стили — хуже сопровождаемость).

### Границы решения

- Меняются **только** вёрстка и стили каталога услуг (`s-services`) и выделение его строки поиска. Логика выбора услуги (`selectService`), поиска (`filterSearch`/`#serviceSearch`), фильтра по категории (`filterCategory`/`_openCatFilter`/`#catFilterBtn`), баннер предвыбранного мастера (`_updateMasterBanner`/`masterBanner`), ветка «услуги мастера» (`/book_services/...`, `renderServices`, `services.js:106-129`) и состояние «Ничего не найдено» (`services.js:82-84`) — **не меняются**.
- Состав данных карточки (`name`, `priceStr`, `dur`, изображение из `resolveServiceImage`) **не расширяется**; новых полей и запросов нет.
- **Общий класс `.search-bar` и строка поиска на главной `#homeSearchBar` (`prototype.html:47`, `style.css:252-259`) не изменяются** (BA №3). Риск регрессии главной снят на уровне решения.
- Раздел «Лента» (`.feed-card`, `_paintFeed`) — только визуальный ориентир, **не дорабатывается**.
- Кнопка фильтра категорий (`#catFilterBtn`) и модалка выбора категории (`_openCatFilter`) — без изменений, кроме вынужденного согласования отступов при переходе на один столбец.
- Захардкоженный YCLIENTS-токен — **не трогается**.

## Последствия

- **Положительные:**
  - Каталог услуг — крупные читаемые карточки в один столбец в духе «Ленты» (бизнес-цель BA); единый визуальный язык приложения.
  - Строка поиска по услугам заметна как поле ввода (ФТ-4), без риска для главной (выделение локализовано, общий класс не тронут).
  - Минимальная площадь правок: преимущественно `style.css` + при необходимости одна строка `prototype.html`; JS-логика каталога не затрагивается (надёжное сохранение функциональности — ФТ-5…ФТ-8).
  - Обе ветки каталога (основная и режим предвыбранного мастера) получают новую вёрстку автоматически через общий `_renderList` (ФТ-7).
- **Отрицательные / компромиссы:**
  - Старые двухколоночные стили карточки полностью замещаются — это намеренно (возврата к 2 колонкам не предполагается).
  - Некоторое визуальное дублирование приёмов с `.feed-*` (обложка 16/9, скругление, тень) при раздельных классах — осознанный компромисс ради развязки доменов (услуги vs лента).
- **Технический долг:**
  - Унификация «крупной карточки с обложкой» между услугами и лентой в единый параметризуемый компонент **не делается** (разные домены/состав/поведение) — к вопросу можно вернуться при следующем существенном касании обоих разделов.

## Влияние на код

- **`docs/style.css`** — основной объём правки:
  - `.svc-catalog` (стр. 261-265): один столбец вместо `grid-template-columns: 1fr 1fr` (одна колонка), согласование `gap`/`padding`.
  - `.svc-cover` (стр. 277-283): `aspect-ratio:16/9` вместо `padding-top:75%` (с соответствующей корректировкой позиционирования `.svc-cover-img`/`.svc-cover-placeholder`, чтобы обложка корректно заполняла блок 16/9).
  - `.svc-name`/`.svc-meta` (стр. 303-304): увеличенная типографика пропорционально крупной карточке (BA №4) — конкретные значения в спецификации.
  - Новый селектор выделения строки поиска экрана услуг (`#serviceSearchBar` или `.search-bar--services`): контрастный фон / рамка / тень — **дополняет** `.search-bar`, не изменяет его. Общий `.search-bar` (стр. 252-257) и `#homeSearchBar`-правила (стр. 258-259) — **без изменений**.
- **`docs/prototype.html`** — при выборе якоря через `id`/класс: добавить `id="serviceSearchBar"` (или класс-модификатор) на контейнер строки поиска экрана услуг (стр. 110, `<div class="search-bar">`). Строка поиска на главной (`#homeSearchBar`, стр. 47) — без изменений. Прочая разметка `s-services` (баннер мастера, кнопка фильтра, `#serviceList`, tab-bar) — без изменений.
- **`docs/modules/services.js`** — HTML-разметка `_renderList` (стр. 86-99) сохраняет текущие классы и поведение; правка логики **не требуется**. (Если для типографики/структуры обложки потребуется минимальная корректировка разметки карточки — она допустима строго в пределах сохранения классов, `data-sid`, `onclick`, `loading="lazy"` и состава данных.)
- **`docs/sw.js`** — состав `APP_SHELL` **не меняется** (правятся существующие файлы: `style.css`, при необходимости `prototype.html`/`services.js`); `CACHE_VERSION` **поднимается на одну ступень** (общая координация с ADR-012 — в рамках единого релиза v1.10.0 поднятие выполняется один раз, см. спецификацию).
- **`docs/version.json`** — обновляется мейнтейнером при релизе (вне объёма реализации фичи).
- **Новые рантайм-зависимости:** нет. YCLIENTS-токен не трогается. Операции с веткой `backup` запрещены.
- **Что ломается:** ничего из поведения. Меняется только визуальное представление каталога услуг и вид строки поиска на `s-services`. Главная (`#homeSearchBar`), лента, логика записи — не затрагиваются.
