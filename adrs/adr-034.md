# ADR-034: In-app обратная связь — переиспользуемые примитивы toast и bottom-sheet, восстановление после сетевых ошибок, индикатор офлайна/устаревших данных

- **Дата:** 2026-05-31
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-037.md` (источник `requirements/req-inapp-feedback-036.md`)
- **Связанные ADR:** ADR-033 (персистентный кэш каталога, SWR без TTL — даёт точку определения «провала revalidate»), ADR-014 (SWR в Service Worker — справочно). Стилевой паттерн оверлея — `services.js:_openCatFilter`.

## Контекст

Приложение использует браузерные `alert()/confirm()/prompt()` (всего 23 места). В клиентских сценариях (создание, перенос, отмена записи; отзыв) нативные модалки блокируют интерфейс, рвут анимации, игнорируют тему, а при сетевой ошибке оставляют пользователя в тупике без выхода. После ADR-033 каталог гидрируется из снимка localStorage, а фоновый revalidate (`initApp`) может молча упасть — пользователь не знает, что видит устаревшие данные, и нет сигнала об офлайне.

Силы:
- Vanilla JS без рантайм-зависимостей; код в `docs/modules/*.js`, `docs/prototype.html`, `docs/style.css`; статический хостинг GitHub Pages.
- Существует готовый паттерн bottom-sheet (`services.js:_openCatFilter`, 31-54): `position:fixed;inset:0`, фон `rgba(0,0,0,.45)`, контейнер `border-radius:24px 24px 0 0`, drag-handle, клик по фону закрывает.
- Весь вставляемый текст (включая `meta.message` от YCLIENTS) обязан экранироваться `esc()` (`utils.js`).
- Нельзя ломать защиту от двойной отправки (`booking.js:_bookInProgress`, `_rescheduleInProgress`, `_cancelInProgress`) и переходы на экраны успеха (`s-confirm`, `go('s-home')`).
- Телефон филиала уже доступен: `studio.js:getCompanyContacts().phone` + готовый `studio.js:_telHref(phone)`.
- Эти примитивы будут переиспользованы в v1.33.0 (a11y, Esc-закрытие) и v1.34.0 — закладывается расширяемость API без переписывания вызовов.

Решение охватывает развилки BA §8: ОВ-1 (граница клиентское/админское), ОВ-2 (UX-токены тайминга/позиции/очереди), ОВ-3 (определение «провала revalidate»).

## Рассматриваемые варианты

### Вопрос 1 — Размещение примитивов (новый модуль vs расширение существующего)

#### Вариант 1A (выбран): новый модуль `docs/modules/ui.js`
- Экспортирует `toast(message, type, opts)`, `confirmSheet(opts)`, `errorSheet(opts)`, `setOfflineBanner(state)`. Чистые хелперы тайминга/очереди выносятся в `docs/modules/uiQueue.js` (тестируемый, без DOM).
- Плюсы: единая точка для всех будущих UI-примитивов (v1.33/v1.34); явная граница ответственности; импортируется из `booking.js`, `review.js`, `app.js` без циклов; имя `ui.js` шире, чем `toast.js`, и не придётся переименовывать при добавлении sheet/banner.
- Минусы: +1 файл в `APP_SHELL` → подъём `CACHE_VERSION`.
- Последствия: новый модуль регистрирует глобальные функции через `window` только там, где они вызываются из инлайн-`onclick` (как существующие модули); прямые импорты — предпочтительны.

#### Вариант 1B (отвергнут): `docs/modules/toast.js` только под toast, sheet inline в каждом модуле
- Минусы: bottom-sheet нужен в нескольких модулях (booking, review) → дублирование разметки; противоречит НФТ-2 (переиспользуемость); при v1.33 (Esc, a11y) правки разъедутся по копиям. Имя `toast.js` сузит модуль и потребует переименования.

#### Вариант 1C (отвергнут): расширить `utils.js`
- Минусы: `utils.js` — чистые функции без DOM/таймеров; внесение оверлеев и таймеров туда смешивает слои и ломает тестируемость существующих утилит.

### Вопрос 2 — Модель toast: одиночный slot vs очередь

#### Вариант 2A (выбран): одиночный активный toast с вытеснением + чистый модуль тайминга
- В каждый момент виден один toast; новый вызов заменяет текущий (сбрасывает таймер). Логика «какой toast показать сейчас, когда скрыть» вынесена в `uiQueue.js` как чистые функции (без DOM, без реальных таймеров — таймер инжектируется), что покрывается unit-тестами (КП-10).
- Тайминги (UX-токены, ОВ-2): success/info — **3000 мс**, error — **5000 мс**; одна строка-контейнер сверху (`position:fixed;top`), `z-index` ниже sheet, но выше контента.
- Плюсы: предсказуемый UX для коротких подтверждений; простая тестируемая логика; нет накопления тостов при шторме событий.
- Минусы: при двух быстрых событиях второе вытеснит первое — приемлемо для клиентских сценариев (подтверждения единичны).

#### Вариант 2B (отвергнут): полноценная FIFO-очередь с последовательным показом
- Минусы: усложнение (состояние очереди, тайминги перехода) без потребности в текущих сценариях; больше поверхность багов; BA §8 ОВ-2 отдаёт параметры на усмотрение архитектора — выбираем минимум.

### Вопрос 3 — Диалог сетевой ошибки: универсальный `errorSheet` vs частные

#### Вариант 3A (выбран): один `errorSheet({ title, message, onRetry, phone })` поверх bottom-sheet
- Кнопки: «Повторить» (вызывает `onRetry`), «Позвонить в студию» (`tel:` через `_telHref`, показывается только если `phone` непустой), «Закрыть». Текст `message` всегда через `esc()`.
- «Повторить» закрывает sheet и повторно вызывает то же действие (`startBooking()` / `confirmCancel()` / submit отзыва) — контекст в `state` (`_rescheduleId`, `_cancelId`, поля брони) **не сбрасывается до успеха**, поэтому повтор работает без потери ввода (ФТ-6).
- Плюсы: единая реализация для брони/отмены/переноса/отзыва; наследует стиль `_openCatFilter`; фолбэк по телефону встроен (ФТ-7).
- Минусы: вызывающий код обязан передать корректный `onRetry` (замыкание на текущее действие) — фиксируется в спецификации.

#### Вариант 3B (отвергнут): отдельные диалоги на каждый сценарий
- Минусы: дублирование; расхождение поведения; против НФТ-2.

### Вопрос 4 — Определение «провала revalidate каталога» (ОВ-3)

#### Вариант 4A (выбран): провал = исключение/неуспех сетевого обновления `initApp` **при наличии гидрированного снимка**
- Триггер индикатора «Данные могли устареть» взводится, когда: на старте была успешная синхронная гидрация из снимка (`readCatalogSnapshot()` вернул данные) **и** последующий фоновый сбор в `initApp` завершился ошибкой — `catch(e)` в `initApp` (`app.js:238`) **или** ответы каталога неуспешны (`!svcRes.success`), то есть `writeCatalogSnapshot` не был вызван. При успешном revalidate индикатор не показывается (свежие данные дописаны).
- Без отдельного таймаута сверх существующего: сетевые вызовы YCLIENTS уже идут через `YC.get`; добавлять новый таймаут не требуется — нужно различать «успех/неуспех», а не «медленно».
- Плюсы: использует уже существующие точки ADR-033 (успех = `writeCatalogSnapshot`, провал = `catch`/`!success`); не вводит новых таймеров; детерминированно.
- Минусы: «медленный, но успешный» ответ не считается устаревшим (и не должен) — корректно по семантике SWR.

#### Вариант 4B (отвергнут): таймаут N секунд как признак устаревания
- Минусы: медленная сеть ≠ устаревшие данные; ложные срабатывания; усложнение; BA ОВ-3 допускает выбор «ошибка ответа».

### Вопрос 5 — Граница «клиентское/админское» (ОВ-1)

#### Вариант 5A (выбран): весь админский слой вне объёма
- `prompt()` GH-токена (`github.js:61,76`) и 6 диалогов `admin.js` остаются на нативных модалках — это не клиентский UX, у админа другие ожидания, риск регрессий в чувствительном слое не оправдан.
- Плюсы: фокус релиза на клиентском пути; меньше поверхность регрессий; совпадает с рабочим допущением BA.
- Минусы: в коде остаётся смешение паттернов (часть native, часть in-app) — приемлемый техдолг, помечен.

#### Вариант 5B (отвергнут): заменить и админские диалоги
- Минусы: расширяет объём за пределы BA §2; чувствительный admin-слой; против принципа минимальности.

## Решение

- **Примитивы — новый модуль `docs/modules/ui.js`** (1A): `toast`, `confirmSheet`, `errorSheet`, `setOfflineBanner`; чистая логика тайминга/выбора активного toast — в `docs/modules/uiQueue.js` (тестируемо).
- **Toast — одиночный с вытеснением** (2A): success/info 3000 мс, error 5000 мс; `aria-live="polite"` контейнер; текст через `esc()`.
- **Диалог ошибки — универсальный `errorSheet`** (3A): «Повторить» / «Позвонить» (фолбэк при пустом телефоне) / «Закрыть»; повтор без сброса контекста.
- **Провал revalidate** (4A): индикатор «Данные могли устареть» при гидрации из снимка + неуспехе фонового обновления каталога; офлайн-баннер — по `navigator.onLine` + `online`/`offline` событиям.
- **Граница** (5A): весь админский слой (`admin.js`, `github.js` prompt) вне объёма.

### Нормативный контракт примитивов (стабилен для v1.33/v1.34)

- `toast(message, type = 'info', opts = {})` — `type ∈ {'success','error','info'}`; `opts.duration?` (по умолчанию из правил выше). Текст экранируется внутри. Возвращает `void`.
- `confirmSheet({ title, message?, confirmText, cancelText = 'Отмена', danger = false }) → Promise<boolean>` — резолвится `true` (подтверждение) / `false` (отмена, клик по фону, кнопка отмены). Замена клиентских `confirm()`.
- `errorSheet({ title, message, onRetry, phone? }) → void` — рендерит sheet ошибки; «Повторить» закрывает sheet и зовёт `onRetry()`; «Позвонить» рендерится только при непустом `phone`.
- `setOfflineBanner(mode)` — `mode ∈ {'offline','stale',null}`; `null` скрывает баннер. Приоритет: `offline` важнее `stale` (если нет сети, показываем «Нет соединения»).
- Расширяемость (закладка под v1.33): сигнатуры принимают `opts`-объект; реализация sheet строится так, чтобы добавление Esc-закрытия, фокус-ловушки и ARIA-ролей не меняло публичные сигнатуры — только внутреннюю реализацию `ui.js`.

### Точки замены нативных диалогов (нормативно)

- `booking.js:53` — `alert` «Услуги ещё загружаются» → `toast(..., 'info')`.
- `booking.js:84` — ошибка создания брони → `errorSheet({ onRetry: () => startBooking(), phone })`.
- `booking.js:125` — ошибка переноса → `errorSheet({ onRetry: () => startBooking(), phone })` (контекст `_rescheduleId` сохранён).
- `booking.js:226` — ошибка отмены (HTTP ≥ 400) → `errorSheet({ onRetry: () => confirmCancel(), phone })`.
- `booking.js:234` — нет соединения при отмене → `errorSheet({ onRetry: () => confirmCancel(), phone })`.
- `review.js:132` — нет данных визита → `toast(..., 'error')` (повтор бессмыслен — это валидация контекста).
- `review.js:150` — успех отзыва → `toast('Спасибо! Ваш отзыв отправлен.', 'success')` затем `go('s-home')`.
- `review.js:154` — ошибка отправки отзыва → `errorSheet({ onRetry: () => submitReview(), phone })`.
- Отмена записи (успех, `confirmCancel` после `go('s-home')`) → `toast('Запись отменена', 'success')` (ФТ-8: у отмены нет своего экрана успеха).
- Создание брони (успех) → **без тоста** (есть экран `s-confirm`, ФТ-8).

### Границы решения

- Не меняется: логика бронирования/отмены/переноса, защита от двойной отправки, переходы на экраны успеха, контракт YCLIENTS API, инлайн-валидация форм (`auth.js`/`pin.js`/`slots.js`), стратегии SW, секреты/токен.
- Весь админский слой (`admin.js`, `github.js` prompt) — вне объёма.
- Источник телефона — только `getCompanyContacts().phone`; новых запросов не вводится.

## Последствия

- **Положительные:** согласованный in-app UX в обеих темах; пользователь может завершить запись при сбое (повтор/звонок) → меньше потерянных записей; переиспользуемые примитивы под v1.33/v1.34; сигнал об офлайне и устаревших данных; ноль рантайм-зависимостей и новых эндпоинтов.
- **Отрицательные / компромиссы:** в коде сосуществуют in-app (клиент) и native (админ) диалоги; одиночный toast вытесняет предыдущий при шторме событий (приемлемо); +2 модуля → подъём `CACHE_VERSION`.
- **Технический долг:** Esc-закрытие, фокус-ловушка и полные ARIA-роли sheet откладываются в v1.33.0 (a11y); миграция админских диалогов на in-app — отдельное требование при необходимости.

## Влияние на код

- **Новый модуль `docs/modules/ui.js`** — `toast`, `confirmSheet`, `errorSheet`, `setOfflineBanner`; рендер оверлеев по паттерну `_openCatFilter`; весь текст через `esc()`; регистрация в `window` только для вызовов из инлайн-`onclick`.
- **Новый модуль `docs/modules/uiQueue.js`** — чистая логика тайминга/выбора активного toast (без DOM и реальных таймеров), покрыта unit-тестами.
- **`docs/modules/booking.js`** — замена `alert` в строках 53, 84, 125, 226, 234 на `toast`/`errorSheet`; `confirmCancel` успех → `toast`; импорт `getCompanyContacts`/`_telHref` из `studio.js` (через `ui.js` или напрямую). Защита от двойной отправки не трогается.
- **`docs/modules/review.js`** — замена `alert` в строках 132, 150, 154 на `toast`/`errorSheet`.
- **`docs/app.js`** — взвод индикатора «устарело» в `initApp`: при гидрации из снимка + неуспехе обновления → `setOfflineBanner('stale')`; при успехе — `setOfflineBanner(null)` (если не offline); регистрация `online`/`offline` слушателей и старт по `navigator.onLine`.
- **`docs/prototype.html`** — DOM-контейнеры toast (`aria-live`), хост sheet, узел офлайн-баннера; обе темы через `var(--…)`.
- **`docs/style.css`** — стили toast (3 типа), bottom-sheet ошибки/подтверждения, офлайн-баннер; токены `var(--bg/--surface/--border/--accent/--text/--text-2/--danger)`; анимации без разрыва существующих экранов.
- **`docs/sw.js`** — добавить `./modules/ui.js` и `./modules/uiQueue.js` в `APP_SHELL`; поднять `CACHE_VERSION` `v88`→`v89`. Стратегии SW не меняются.
- **Не трогаются:** `admin.js`, `github.js`, `auth.js`/`pin.js`/`slots.js` (инлайн-валидация), `api.js` (контракт), `state.js`, секреты/токен. Операции с веткой `backup` запрещены.
- **Новые рантайм-зависимости:** нет.
