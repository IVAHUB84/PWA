# ADR-035: Доступность (a11y) и клавиатурная навигация клиентского потока

- **Дата:** 2026-05-31
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-038.md` (источник `requirements/req-accessibility-pass-037.md`)
- **Связанные ADR:** ADR-034 (примитивы `ui.js`/`uiQueue.js`: toast/`aria-live`, `confirmSheet`, `errorSheet`, sheet-хост — техдолг a11y явно вынесен в v1.33.0), ADR-014 (SWR в SW — справочно). Стилевой паттерн оверлея — `services.js:_openCatFilter`.

## Контекст

UX-ревью от 2026-05-31 показало, что приложение недоступно для пользователей клавиатуры и скринридеров: ~76 интерактивных элементов — это `div[onclick]` (недостижимы по Tab/Enter, невидимы скринридеру); всего 4 `aria-label`; перенос фокуса при смене экрана почти отсутствует; Esc закрывает лишь 2 оверлея; `prefers-reduced-motion` учтён минимально (один блок `style.css:1169`), слайды экранов (`.screen translateX`, `style.css:125-139`), карусель фото (`services.js`) и scale-анимации не заглушены; нет видимого focus-ring для клавиатуры.

Силы:
- Vanilla JS без рантайм-зависимостей; код в `docs/prototype.html`, `docs/modules/*.js`, `docs/style.css`; статический хостинг GitHub Pages.
- ADR-034 уже дал инфраструктуру: модуль `ui.js` с sheet-хостом (`#uiSheetHost`), `confirmSheet`/`errorSheet` поверх единого оверлея (`.ui-sheet-overlay` → `.ui-sheet[role=dialog][aria-modal=true]`), контейнер тостов с `aria-live="polite"` (`#uiToastHost`). Публичные сигнатуры (`toast`, `confirmSheet`, `errorSheet`, `setOfflineBanner`) объявлены **стабильными**; ADR-034 §«Расширяемость» прямо разрешил добавлять Esc/фокус-ловушку/ARIA, **не меняя сигнатуры** — только внутреннюю реализацию `ui.js`.
- Оверлей выбора категории (`services.js:31-54`, `_openCatFilter`) — отдельная разметка вне `ui.js`, закрывается кликом по фону, но не по Esc.
- `navigation.js:go()` — единая точка смены экранов; экраны имеют контейнер `div.screen#s-*` и (у большинства) заголовок `.page-title` / `.section-title`; точечные `.focus()` есть в `auth.js`/`review.js` — их нельзя ломать.
- Существующие стили опираются на `:active` (тач/мышь) и токены тем `var(--…)`, есть `--focus-ring` в обеих темах (`style.css:40,69`).
- `tab`-навигация и карточки/строки реализованы как `div`, стилизованы по классам (не по тегу) — полная замена тега на `<button>` рискует задеть верстку (flex/grid, сброс стилей кнопки).

Решение охватывает развилки BA §6/§8: Допущение-1 (граница клиент/админ), Допущение-2 (button vs role), Допущение-3 (цель переноса фокуса), ОВ-3 (параметры focus-ring/заглушения — поведенческие критерии).

## Рассматриваемые варианты

### Вопрос 1 — Esc-закрытие и фокус-ловушка sheet: где реализовать

#### Вариант 1A (выбран): внутри `ui.js`, в общем хелпере открытия оверлея
- `confirmSheet`/`errorSheet` уже строят одинаковый `.ui-sheet-overlay`. Вводится приватный хелпер `_mountSheet(overlay, { onClose })`, который: вешает `keydown`-слушатель на `document` (Esc → `onClose`), реализует фокус-ловушку (Tab/Shift+Tab циклит внутри `.ui-sheet`), переносит фокус на первую интерактивную кнопку при открытии, возвращает фокус на ранее активный элемент при закрытии, снимает слушатель при `remove()`. Публичные сигнатуры не меняются.
- Плюсы: ноль дублирования (НФТ-2, КП-10); один источник правды для всех sheet текущих и будущих (v1.34); сигнатуры стабильны (соответствует закладке ADR-034); тестируема чистая функция выбора фокус-цели/обработки Esc.
- Минусы: `ui.js` растёт; нужно аккуратно снимать `document`-слушатель, чтобы не копить хендлеры.
- Последствия: при наличии нескольких sheet одновременно (не штатный сценарий) Esc закрывает верхний — хелпер хранит стек активных оверлеев.

#### Вариант 1B (отвергнут): Esc-слушатель в каждом вызывающем модуле
- Минусы: дублирование логики по `booking.js`/`review.js`; прямо нарушает НФТ-2 и КП-10; расхождение поведения; ADR-034 уже централизовал sheet в `ui.js`.

#### Вариант 1C (отвергнут): глобальный делегированный Esc-обработчик на уровне `app.js`
- Минусы: `app.js` не владеет жизненным циклом оверлеев `ui.js`; пришлось бы экспортировать реестр оверлеев наружу, размывая инкапсуляцию; сложнее корректно вернуть фокус.

### Вопрос 2 — Конвертация `div[onclick]`: `<button>` vs `role="button"`+tabindex (Допущение-2)

#### Вариант 2A (выбран): по контексту — `role="button"`+`tabindex="0"`+Enter/Space для карточек/строк, `<button>` для смысловых кнопок
- Визуальные строки списка/карточки (`.svc-catalog-card`, `.master-card`, `.s-row`, `.slot`, `.tab`, `.any-master`, `.hero-card`) стилизованы по классам и встроены в flex/grid-раскладку — для них добавляются `role="button"` + `tabindex="0"` + `aria-label`/доступное имя + обработка Enter и Space, стили не трогаются. Элементы, по смыслу являющиеся кнопкой действия и не несущие сложной верстки (иконочные кнопки навигации/действий), переводятся в `<button>` с классом, сбрасывающим дефолтные стили.
- Активация с клавиатуры (Enter и Space) выполняет то же действие, что клик. Чтобы не дублировать логику и не плодить инлайн-хендлеры, вводится **единый делегированный обработчик** keydown на корне приложения: для элементов с `role="button"` (или `[data-key-activate]`) Enter/Space вызывают `el.click()` (с `preventDefault` для Space, чтобы не скроллить). Это исключает разрастание инлайна и работает для динамически отрендеренных карточек.
- Плюсы: минимальные изменения верстки (КП-8 — без регрессий для мыши/тача); один обработчик активации вместо десятков инлайнов; существующие `onclick` остаются рабочими (клик мышью и `.click()` идут через тот же путь); скринридер объявляет элементы как кнопки.
- Минусы: `role="button"` менее «родной», чем `<button>` (нет автоматической семантики disabled/формы) — приемлемо, элементы не являются формами; нужно следить за доступным именем каждой такой роли.
- Последствия: для динамически рендерящихся списков (services/masters/slots) роль/`tabindex` добавляются в шаблоны рендера соответствующих модулей; делегированный keydown покрывает их без правки шаблонов под каждый элемент.

#### Вариант 2B (отвергнут): всё в `<button>`
- Минусы: `<button>` сбрасывает наследование, ломает flex-выравнивание карточек/строк, требует массовых правок CSS (риск регрессий КП-8); больше дифф; противоречит минимальности.

#### Вариант 2C (отвергнут): всё в `role="button"`+tabindex
- Минусы: смысловые кнопки действий теряют родную семантику без выгоды; контекст BA (ФТ-1) прямо требует дифференциации.

### Вопрос 3 — Цель переноса фокуса при `go()` (Допущение-3)

#### Вариант 3A (выбран): заголовок экрана, фолбэк — контейнер с `tabindex="-1"`
- В `go()` после завершения перехода фокус переносится на заголовок нового экрана (`.page-title`/`.section-title` внутри `#<id>`), которому при необходимости проставляется `tabindex="-1"`; если заголовка нет — на сам контейнер `#<id>` (`tabindex="-1"`). Логика выбора цели выносится в чистую функцию `focusTargetSelector(screenEl)` (тестируема, без `.focus()`), вызывающий код делает `target.focus({ preventScroll: true })`.
- Перенос выполняется после анимации (в том же `setTimeout`, что и `onEnterScreen`) и **только** если фокус не был уже перемещён точечным `.focus()` экрана (проверка `document.activeElement` внутри контейнера — если активный элемент уже внутри нового экрана, повторно не двигаем). Это сохраняет `auth.js`/`review.js` (ФТ-3, КП-3).
- Плюсы: скринридер озвучивает заголовок = «куда я попал»; чистая тестируемая функция выбора цели (КП-3, автотест); не конфликтует с точечными `.focus()`.
- Минусы: `tabindex="-1"` на заголовке добавляет узлы в программный фокус (не в Tab-порядок) — нейтрально для мыши.

#### Вариант 3B (отвергнут): фокус всегда на первый интерактивный элемент экрана
- Минусы: скринридер не объявляет контекст экрана; на разных экранах первый интерактив разный (нестабильно); может «проскочить» заголовок.

#### Вопрос 4 — Заглушение анимаций при `prefers-reduced-motion` (ФТ-6)

#### Вариант 4A (выбран): CSS-медиазапрос + JS-гейт для inline-transform
- Декларативные анимации (`.screen` translateX `style.css:125-139`, scale-эффекты `:active`, `@keyframes pop/tapPulse/inboxBadgePulse`, карусельная `transition`) заглушаются расширением существующего блока `@media (prefers-reduced-motion: reduce)` (`style.css:1169`): для `.screen` — `transition: none`; scale/transform-`@keyframes` → `animation: none`; карусельный трек — `transition: none !important` (перебивает inline `track.style.transition`).
- Для анимаций, управляемых из JS через inline-стиль (переход экрана в `navigation.js:setTx`, карусель `services.js`), вводится единый чистый предикат `prefersReducedMotion()` (обёртка над `matchMedia('(prefers-reduced-motion: reduce)').matches`): при `true` `setTx` ставит переход мгновенно (`animated=false` независимо от аргумента), карусель снапается без `transition`. Функциональность (смена слайда/экрана) сохраняется — меняется только наличие движения.
- Плюсы: переиспользует существующий блок (минимальность); CSS покрывает декларативное, JS-гейт — то, что задаётся inline и CSS перебить не может надёжно; предикат тестируем.
- Минусы: дублирование намерения в CSS и JS для карусели — оправдано, т.к. inline-`transition` иначе побеждает каскад без `!important`, а отключать движение надёжнее в источнике.

#### Вариант 4B (отвергнут): только CSS
- Минусы: inline `style.transition`/`style.transform`, выставляемые JS (`navigation.js`, `services.js`), CSS-медиазапросом надёжно не отменить без россыпи `!important`; движение всё равно проиграется при программной установке.

### Вопрос 5 — focus-ring: глобально `:focus` vs `:focus-visible` (ФТ-7)

#### Вариант 5A (выбран): `:focus-visible` + `outline` через токен `--focus-ring`
- Глобальное правило `:focus-visible { outline: 2px solid var(--focus-ring); outline-offset: 2px; border-radius наследуется }` для интерактивных элементов (`button, a, [role="button"], [tabindex], input, textarea, select`). `:focus:not(:focus-visible) { outline: none }` гарантирует отсутствие кольца при клике мышью. Токен `--focus-ring` уже определён в обеих темах (`style.css:40,69`).
- Плюсы: ring при Tab, нет при клике (КП-7); одинаково корректен в светлой/тёмной теме (НФТ-3); минимум кода.
- Минусы: на элементах со скруглением outline может не повторять радиус идеально — допустимо (offset нивелирует).

#### Вариант 5B (отвергнут): `:focus` глобально
- Минусы: кольцо «висит» после клика мышью/тачем — прямое нарушение ФТ-7/КП-7/КП-8.

### Вопрос 6 — Граница клиент/админ (Допущение-1)

#### Вариант 6A (выбран): только критичный клиентский поток
- Конвертация `div[onclick]`, Esc, фокус, ARIA — для потока «логин → услуга → мастер → слот → бронь», навигации (tab-bar) и клиентских оверлеев/sheet. Весь админский слой (`admin.js`, `github.js`) — вне объёма (согласовано с ADR-034 §5A).
- Плюсы: фокус на ценности, меньше поверхность регрессий, совпадает с BA.
- Минусы: остаётся техдолг по a11y админки — помечен.

#### Вариант 6B (отвергнут): сплошной a11y-аудит всего приложения
- Минусы: выходит за BA §2; чувствительный админ-слой; против минимальности.

## Решение

- **Esc/фокус-ловушка sheet** (1A): внутри `ui.js`, общий приватный `_mountSheet`/`_dismissSheet`; публичные сигнатуры ADR-034 не меняются. Оверлей выбора категории `services.js:_openCatFilter` получает Esc-закрытие тем же приёмом (свой `keydown`+возврат фокуса, либо переиспользование хелпера, если он экспортируется как внутренний — но без слома стиля оверлея).
- **Конвертация `div[onclick]`** (2A): по контексту `<button>` (смысловые кнопки) / `role="button"`+`tabindex="0"`+`aria-label` (карточки/строки); активация Enter/Space через **единый делегированный keydown-обработчик** (`el.click()`, `preventDefault` для Space). Существующие `onclick` сохраняются.
- **Перенос фокуса** (3A): в `go()` после перехода — на заголовок нового экрана, фолбэк контейнер `tabindex="-1"`; чистая функция `focusTargetSelector`; не перебивает уже выполненный точечный `.focus()`.
- **Reduced-motion** (4A): расширение CSS-блока `style.css:1169` (`.screen`, scale-keyframes, карусель) + чистый предикат `prefersReducedMotion()`, которым гейтятся inline-анимации `navigation.js:setTx` и карусель `services.js`.
- **focus-ring** (5A): `:focus-visible` + `--focus-ring`, гашение при `:focus:not(:focus-visible)`; обе темы.
- **Граница** (6A): только клиентский поток; админ — вне объёма.

### Нормативный контракт изменений (стабильный)

- `ui.js` — публичные сигнатуры `toast`/`confirmSheet`/`errorSheet`/`setOfflineBanner` **не меняются**. Добавляются только приватные хелперы (`_mountSheet(overlay, { onClose, returnFocusTo? })`, `_dismissSheet`). Поведение: при открытии sheet фокус на первую кнопку `.ui-sheet`; Tab/Shift+Tab циклит внутри `.ui-sheet`; Esc вызывает тот же путь закрытия, что клик по фону/кнопке «Отмена»/«Закрыть» (для `confirmSheet` Esc резолвит `false`); при закрытии фокус возвращается на элемент, активный до открытия.
- `navigation.js` — экспортируется чистая функция `focusTargetSelector(screenEl) → HTMLElement | null` (возвращает заголовок `.page-title`/`.section-title` или сам контейнер; не вызывает `.focus()`). `go()` переносит фокус после анимации, не перебивая точечный `.focus()`.
- `utils.js` (или `constants.js`) — экспортируется чистый предикат `prefersReducedMotion() → boolean`. (Размещение в `utils.js` — оно уже импортируется широко и держит чистые хелперы.)
- Делегированный keydown-обработчик активации `role="button"` регистрируется один раз (в `app.js` или `navigation.js` на корне `#phone`); Enter/Space → `e.target.closest('[role="button"]').click()`.

### Границы решения

- Не меняется: видимое поведение для мыши/тача (вид, layout, тексты, потоки — КП-8); бизнес-логика брони/отмены/переноса, защита от двойной отправки, переходы на экраны успеха; контракт YCLIENTS API; стратегии SW; секреты/токен; инлайн-валидация `auth.js`/`pin.js`/`slots.js` (только добавление a11y-атрибутов, не логики).
- Весь админский слой (`admin.js`, `github.js`) — вне объёма.
- Ветка `backup` — никаких операций.

## Последствия

- **Положительные:** основной клиентский поток проходим с клавиатуры; скринридер понимает экран (фокус на заголовок), роли элементов и динамические сообщения (`aria-live` из ADR-034); комфорт для вестибулярной чувствительности (reduced-motion); ноль рантайм-зависимостей; sheet-примитивы остаются стабильными для v1.34.
- **Отрицательные / компромиссы:** часть элементов остаётся `role="button"`, а не нативной кнопкой (приемлемо); reduced-motion-намерение продублировано в CSS и JS для карусели (оправдано каскадом); `ui.js` растёт; админский слой остаётся недоступным (помеченный техдолг).
- **Технический долг:** a11y админского слоя; полный аудит WCAG AA с сертификацией; конвертация некритичных (не входящих в основной поток) `div[onclick]` — отложены за пределы релиза.

## Влияние на код

- **`docs/modules/ui.js`** — приватные `_mountSheet`/`_dismissSheet`: Esc-закрытие (стек активных оверлеев, снятие `document`-слушателя при закрытии), фокус-ловушка Tab/Shift+Tab внутри `.ui-sheet`, перенос фокуса на первую кнопку и возврат при закрытии. Подключаются в `confirmSheet`/`errorSheet`. Публичные сигнатуры неизменны.
- **`docs/modules/navigation.js`** — экспорт `focusTargetSelector(screenEl)`; в `go()` после анимации `focusTargetSelector(toEl)?.focus({preventScroll:true})` при условии, что фокус ещё не внутри `toEl`; `setTx` уважает `prefersReducedMotion()` (мгновенно). Возможна регистрация делегированного keydown активации `role="button"`.
- **`docs/modules/services.js`** — `_openCatFilter`: добавить Esc-закрытие и возврат фокуса оверлею выбора категории (без смены стиля); строки `.s-row` категорий получают `role="button"`+`tabindex="0"`+доступное имя; карусель уважает `prefersReducedMotion()` (снап без `transition`); шаблоны карточек услуг получают `role="button"`+`tabindex`+`aria-label`.
- **`docs/modules/utils.js`** — экспорт `prefersReducedMotion()`.
- **`docs/prototype.html`** — критичным клиентским `div[onclick]` (tab-bar, карточки, строки) добавить `role="button"`+`tabindex="0"`+`aria-label`/доступное имя; иконочным кнопкам — `aria-label`; смысловые кнопки действий перевести в `<button>` там, где это безопасно для стилей; заголовкам экранов — при необходимости `tabindex="-1"`. Тексты и layout не меняются.
- **`docs/style.css`** — глобальный `:focus-visible` + `:focus:not(:focus-visible)` через `var(--focus-ring)` (обе темы); расширение блока `@media (prefers-reduced-motion: reduce)` (`.screen` transition none, scale-keyframes, карусельный трек). Класс-сброс для новых `<button>`, копирующий вид существующего `div`, без изменения визуала.
- **`docs/sw.js`** — изменяются файлы из `APP_SHELL` → поднять `CACHE_VERSION` `v89`→`v90`. Стратегии SW не меняются. (Если меняются только уже перечисленные в `APP_SHELL` файлы — состав `APP_SHELL` не расширяется, поднимается только версия.)
- **Не трогаются:** `admin.js`, `github.js`, `api.js` (контракт), `state.js`, секреты/токен, бизнес-логика брони. Ветка `backup` — запрещена.
- **Новые рантайм-зависимости:** нет.
