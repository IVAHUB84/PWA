# ADR-005: Предложение установки PWA на экран «Домой» — единый механизм install prompt

- **Дата:** 2026-05-28
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-005.md`
- **Связанные ADR:** ADR-001 (не конфликтует), ADR-002 (не конфликтует), ADR-003 (не конфликтует), ADR-004 (не конфликтует; install — основа для уже существующей push-инфраструктуры)

## Контекст

Studio PWA — устанавливаемое приложение онлайн-записи, отдаётся статически из `docs/` на GitHub Pages. Установка на экран «Домой» — главный канал удержания (standalone-запуск, иконка, фундамент для push). Манифест (`docs/manifest.json`: `display: standalone`, иконки 192/512/180) и Service Worker (`docs/sw.js`) уже обеспечивают базовую устанавливаемость — задача добавляет **предложение** установки, а не вводит устанавливаемость с нуля.

Текущее состояние (из кода):

- Экран-инструкция `s-install` (`docs/prototype.html:245`) — ручные шаги для iOS/Safari («Поделиться → На экран «Домой» → Добавить»). Кнопка ставит `yc_install_seen='1'` и уходит на `s-home`.
- Пассивный баннер `#installBanner` на `s-home` (`docs/prototype.html:64`) ведёт на `s-install`; его видимость управляется в `renderHomeHero` (`docs/modules/profile.js:65-66`) по флагу `localStorage` `yc_install_seen`.
- Программного предложения для Android/Chrome через `beforeinstallprompt` **нет**; событие не перехватывается.
- Нет детекта standalone (уже установлено) — предложение не подавляется для установивших.
- Нет автоматического показа при открытии, нет различения платформ, нет правила повторного показа.

Силы, действующие на решение:

- **Стек неизменен.** Vanilla JS/HTML/CSS, статический хостинг из `docs/`, без рантайм-npm-зависимостей (НФТ-1).
- **Политики браузеров.** `beforeinstallprompt` генерируется браузером по своим критериям вовлечённости и может прийти не мгновенно; `event.prompt()` разрешено вызывать **только** в ответ на жест пользователя. Событие нужно перехватить (`preventDefault`), сохранить и вызвать позже по кнопке. iOS/Safari программного API установки не имеет — только ручная инструкция.
- **Уже установлено определимо на клиенте.** Standalone-режим выявляется через `matchMedia('(display-mode: standalone)')` и/или `navigator.standalone === true` (iOS). При запуске как PWA предложение подавляется (ФТ-2).
- **Два конкурирующих механизма недопустимы.** Существующие `#installBanner`, `s-install` и флаг `yc_install_seen` должны быть унифицированы под один источник состояния, иначе пользователь видит два предложения, а состояния «показано/отклонено/установлено» противоречат друг другу (ФТ-9).
- **Ненавязчивость.** Предложение не мешает первому действию, не появляется после отказа/установки; частота показов контролируема (НФТ-3).
- **Деградация.** В браузере без поддержки установки — нет «битых» элементов и ошибок в консоли; основной функционал работает без установки (НФТ-5, ФТ-10).
- **PWA/офлайн не должен регрессировать** (НФТ-2); влияние на `CACHE_VERSION` (`docs/sw.js`, текущий `v37`) — предмет проектирования.

Решение охватывает связанные вопросы: где разместить логику; как перехватывать и вызывать `beforeinstallprompt`; как определять standalone и платформу; судьбу существующих механизмов; правило повторного показа; момент показа; аудиторию; влияние на `CACHE_VERSION`. Конкретные имена функций, тексты и вёрстка — в спецификации релиза, не здесь.

## Рассматриваемые варианты

### Вопрос 1. Где разместить логику предложения установки

#### Вариант 1A (выбран): новый модуль `docs/modules/install.js` — единая точка состояния и поведения

Один ES-модуль инкапсулирует: перехват `beforeinstallprompt` (с `preventDefault` и сохранением отложенного события), детект standalone/платформы, правило повторного показа, показ/скрытие фирменного оверлея, управление видимостью `#installBanner`, переиспользование `s-install` как iOS-контента, обработку `appinstalled`. Экспортирует функции, регистрируемые на `window` (как остальные модули), и инициализируется из `app.js`.

- **Плюсы:**
  - Единый источник истины о состоянии «показано/отклонено/установлено» — исключает два конкурирующих предложения (ФТ-9, КП-8).
  - Перехват `beforeinstallprompt` должен стоять как можно раньше (событие может прийти до первого рендера) — отдельный модуль с ранней инициализацией это обеспечивает.
  - Логика изолирована и тестируема (детект платформы, правило повторного показа — чистые функции).
  - Согласуется с модульной структурой проекта (`docs/modules/*.js`, bindings через `Object.assign(window, …)`).
- **Минусы:**
  - Новый файл → правка `APP_SHELL` и подъём `CACHE_VERSION` (приемлемо, разовая операция).
- **Последствия:** новый модуль добавляется в `APP_SHELL`; `app.js` инициализирует модуль; `profile.js` перестаёт сам управлять видимостью `#installBanner`.

#### Вариант 1B (отвергнут): расширить `profile.js` (там уже живёт управление `#installBanner`)

- **Минусы:** `profile.js` — рендер главной/профиля; перехват `beforeinstallprompt` должен инициализироваться раньше и независимо от рендера home. Смешение ответственности (рендер экрана + жизненный цикл установки), труднее тестировать, риск, что событие не перехвачено к моменту первого `renderHomeHero`. Раздувает и без того крупный модуль.

#### Вариант 1C (отвергнут): инлайн-скрипт в `prototype.html` / прямо в `app.js`

- **Минусы:** логика нетривиальна (детект, состояние, оверлей, события платформы) — инлайн усложняет тестирование и нарушает принятую модульную организацию. `app.js` остаётся точкой инициализации, но сама логика должна жить в модуле.

### Вопрос 2. Перехват и вызов `beforeinstallprompt` (Android/Chrome)

#### Вариант 2A (выбран): перехватить событие, сохранить отложенным, вызвать `prompt()` по кнопке

- На `window` слушается `beforeinstallprompt`; обработчик зовёт `event.preventDefault()` (подавляет нативный мини-инфобар) и **сохраняет** событие в переменную модуля (`deferredPrompt`).
- Фирменный оверлей показывается по правилам Вопроса 4/6. Кнопка «Установить» внутри оверлея вызывает `deferredPrompt.prompt()` (это и есть «жест пользователя», требуемый браузером), затем читает `deferredPrompt.userChoice`.
- После использования событие обнуляется (повторно `prompt()` вызвать нельзя). Подписка на `appinstalled` фиксирует факт установки (Вопрос 5).

- **Плюсы:** совместимо с политиками браузеров (prompt только по жесту); фирменный оверлей показывается заранее, платформенный prompt — по явному действию; нативный мини-инфобар подавлён, конкурирующего предложения нет.
- **Минусы:** браузер сам решает, когда (и решит ли) сгенерировать событие — на части открытий Android-prompt недоступен. Закрыто деградацией: если событие не пришло и платформа не iOS — оверлей не показываем (или показываем только если поддержка очевидна), UI не ломается.
- **Последствия:** появляется состояние `deferredPrompt` в модуле; кнопка оверлея — асинхронный обработчик (`prompt()` + `userChoice`).

#### Вариант 2B (отвергнут): вызывать `prompt()` автоматически при загрузке

- **Минусы:** браузеры запрещают вызов `prompt()` без пользовательского жеста — вызов будет проигнорирован/выбросит ошибку. Нарушает политики, не работает.

### Вопрос 3. Детект «уже установлено» (standalone) и платформы

#### Вариант 3A (выбран): standalone через `display-mode` + `navigator.standalone`; iOS — через UA + отсутствие `beforeinstallprompt`

- **Уже установлено (любая платформа):** `window.matchMedia('(display-mode: standalone)').matches === true` **или** `navigator.standalone === true` (iOS Safari). При истинности — предложение **не показывается** ни в каком виде (ФТ-2, КП-2).
- **iOS/Safari:** определяется по User-Agent (признаки iPhone/iPad/iPod, исключая «настоящий» Android-Chrome) — для показа ручной инструкции, т.к. `beforeinstallprompt` на iOS не приходит (ФТ-4).
- **Android/Chrome и совместимые:** факт поддержки программной установки определяется приходом события `beforeinstallprompt` (Вопрос 2), а не парсингом UA.

- **Плюсы:** стандартные, проверенные сигналы; standalone-детект надёжен; iOS-детект достаточен для выбора «инструкция vs prompt»; не требует библиотек.
- **Минусы:** UA-детект iOS хрупок к будущим изменениям UA — но iOS не предоставляет иного способа; митигируется тем, что ошибка детекта приводит максимум к показу/непоказу инструкции, не к поломке.
- **Последствия:** в модуле — чистые функции `isStandalone()` и `isIOS()`; решение о ветке (prompt vs инструкция vs ничего) принимается на их основе.

#### Вариант 3B (отвергнут): полагаться только на `beforeinstallprompt` без детекта standalone

- **Минусы:** не подавляет предложение для уже установивших на iOS (где события нет вовсе) и не различает «не установлено, но браузер не сгенерировал событие». Нарушает ФТ-2.

### Вопрос 4. Судьба `#installBanner`, `s-install` и флага `yc_install_seen`

#### Вариант 4A (выбран): единый модуль управляет всем; `s-install` — переиспользуемый iOS-контент; новый ключ состояния вместо `yc_install_seen`

- **Фирменный оверлей** — новый элемент (баннер/модал) в стиле приложения, показывается автоматически по правилам Вопроса 6. Это основное предложение.
- **`#installBanner`** на `s-home` сохраняется как пассивная точка входа, но его видимостью управляет **тот же модуль** (а не `profile.js` напрямую): он скрыт, если установлено / отклонено в пределах окна / нет поддержки; иначе доступен. Пользователь не видит одновременно оверлей и баннер как два конкурирующих предложения (модуль гарантирует один активный канал за раз).
- **`s-install`** переиспользуется как **iOS-инструкция** (контент «Поделиться → На экран «Домой»»). На iOS «принять» в оверлее = переход/показ `s-install`. Кнопки `s-install` («Понятно, добавлю» / «Позже») переводятся на новый ключ состояния (не на legacy `yc_install_seen`).
- **`yc_install_seen`** (булев флаг «баннер показан») заменяется новым ключом состояния (Вопрос 5), хранящим время отказа и факт установки. Запись в `yc_install_seen` прекращается; legacy-чтение в `profile.js` удаляется (видимость баннера считает модуль).

- **Плюсы:** один источник истины о состоянии и видимости (ФТ-9, КП-8); переиспользование готового iOS-контента `s-install` (без редизайна); устранён конфликт двух механизмов.
- **Минусы:** нужно аккуратно перевести три места (`profile.js`, кнопки `s-install`, новый оверлей) на единый ключ — разово, в пределах модуля и точечных правок разметки.
- **Последствия:** `profile.js` теряет прямое управление `#installBanner` (делегирует модулю); разметка `s-install` получает новые обработчики кнопок; вводится новый ключ `localStorage`.

#### Вариант 4B (отвергнут): оставить `yc_install_seen` и добавить оверлей рядом

- **Минусы:** два независимых состояния («баннер показан» vs «оверлей отклонён/установлено») рассинхронизируются → два конкурирующих предложения и противоречивые состояния. Прямо нарушает ФТ-9.

### Вопрос 5. Правило повторного показа и хранение состояния (закрывает открытый вопрос BA №1)

#### Вариант 5A (выбран): «после отказа — не показывать 30 дней; после установки — никогда»; единый ключ в `localStorage`

- **После отказа** (нажал «Позже»/закрыл оверлей, либо отклонил платформенный prompt) — предложение **не показывается 30 дней**. По истечении 30 дней — может показаться снова (ненавязчивое напоминание).
- **После установки** (событие `appinstalled` либо `userChoice.outcome === 'accepted'`) — предложение **не показывается никогда** на этом устройстве.
- Состояние хранится в одном ключе `localStorage` (например `yc_install_state`) с полями: метка времени последнего отказа и флаг «установлено». Различение «отклонил» и «установил» — обязательно. Состояние сохраняется между сессиями (ФТ-6, КП-6).
- На новом устройстве предложение может появиться снова — приемлемо (BA, допущение).

- **Плюсы:** прямо закрывает открытый вопрос BA №1 разумным дефолтом; различает отказ и установку; контролируемая частота (НФТ-3); один ключ — одно состояние.
- **Минусы:** «30 дней» — продуктовое значение по умолчанию; при необходимости заказчик скорректирует (значение вынесено в одну константу модуля).
- **Последствия:** в модуле — функция `shouldOffer()`, читающая ключ и применяющая правило 30 дней + «установлено никогда».

#### Вариант 5B (отвергнут): «после отказа не показывать совсем»

- **Минусы:** теряет возможность мягкого повторного предложения для тех, кто отказался импульсивно; BA допускает оба варианта, но «раз в 30 дней» лучше отвечает бизнес-цели удержания без раздражения. Отвергнут как менее гибкий.

### Вопрос 6. Момент показа (закрывает открытый вопрос BA №2)

#### Вариант 6A (выбран): фирменный оверлей — при загрузке (если условия выполнены); платформенный prompt — по кнопке

- **Фирменный оверлей** показывается **при открытии приложения** (после инициализации, не блокируя первый рендер), если: не standalone **и** `shouldOffer()` истинно (Вопрос 5) **и** аудитория подходит (Вопрос 7) **и** есть, что предложить (пришёл `beforeinstallprompt` **или** платформа iOS). Это считается «сразу при открытии».
- **Платформенный `prompt()`** — **только** по нажатию кнопки «Установить» внутри оверлея (Вопрос 2), что совместимо с политиками браузеров.
- Если `beforeinstallprompt` приходит позже первого рендера — модуль при его приходе пересматривает условия и может показать оверлей (в рамках того же открытия), не дожидаясь следующего запуска.

- **Плюсы:** удовлетворяет требованию «показывать сразу при открытии» в части **нашего** предложения, при этом не нарушает политику «prompt только по жесту»; не задерживает первичную отрисовку (НФТ-6).
- **Минусы:** на части открытий Android-событие не приходит — тогда на не-iOS оверлей не показывается (приемлемая деградация, ФТ-10).
- **Последствия:** инициализация модуля в `app.js` после boot; показ оверлея отложен/реактивен на приход события.

#### Вариант 6B (отвергнут): показывать оверлей только по действию пользователя (например, по тапу баннера)

- **Минусы:** противоречит бизнес-намерению «автоматически при открытии» (ФТ-1) — сводит к текущему пассивному баннеру, не доставляя новой ценности.

### Вопрос 7. Аудитория (закрывает открытый вопрос BA №4)

#### Вариант 7A (выбран): только мобильные; десктоп — без предложения

- Предложение показывается только на мобильных. Признак «мобильный»: приход `beforeinstallprompt` на узком экране (`max-width: 767px`) / мобильный контекст, **или** iOS (по Вопросу 3). На десктопе оверлей **не показывается** (десктоп — вне фокуса BA), но десктоп не ломается (ФТ-10).
- Практическое правило: показывать, если (`isIOS()` **или** есть `deferredPrompt`) **и** окружение мобильное (узкий экран / отсутствие у браузера признаков десктопа). Конкретный порог и сигналы — в спецификации.

- **Плюсы:** соответствует акценту BA «на мобильном устройстве»; не навязывает установку десктопным пользователям.
- **Минусы:** граница «мобильный/десктоп» по ширине экрана/UA приблизительна — но ошибка приводит максимум к показу/непоказу, не к поломке.
- **Последствия:** в правиле показа (Вопрос 6) добавляется проверка мобильной аудитории.

#### Вариант 7B (отвергнут): показывать и на десктопе (Chrome тоже поддерживает установку)

- **Минусы:** выходит за фокус BA; тексты («напоминания о визите») ориентированы на мобильного клиента; расширение можно сделать отдельной задачей при запросе заказчика.

### Вопрос 8. Влияние на `CACHE_VERSION` и тексты

- **`CACHE_VERSION`** поднимается на одну ступень (`v37` → `v38`): добавляется новый модуль `install.js` в `APP_SHELL` и правятся существующие файлы в `APP_SHELL` (`app.js`, `profile.js`, `prototype.html`, `style.css`). Состав `APP_SHELL` расширяется одной строкой (`./modules/install.js`).
- **Тексты** (закрывает открытый вопрос BA №5) — разумные плейсхолдеры на русском, в едином тоне: заголовок «Установите приложение», подзаголовок «Добавьте приложение на экран «Домой» и получайте напоминания о ваших записях», действия «Установить» / «Позже». iOS-инструкция переиспользует существующий контент `s-install`. Финальные формулировки заказчик может скорректировать.
- **Метрики** (открытый вопрос BA №6) — **вне рамок**, не учитываются.
- **Манифест и иконки** — без изменений (устанавливаемость уже обеспечена); правки манифеста в этот релиз не входят.

## Решение

- **Вопрос 1 — Вариант 1A:** новый модуль `docs/modules/install.js` как единая точка состояния и поведения; инициализация из `app.js`.
- **Вопрос 2 — Вариант 2A:** перехватить `beforeinstallprompt` (`preventDefault` + сохранить), вызывать `prompt()` только по кнопке «Установить».
- **Вопрос 3 — Вариант 3A:** standalone через `display-mode: standalone` + `navigator.standalone`; iOS — по UA + отсутствие `beforeinstallprompt`.
- **Вопрос 4 — Вариант 4A:** единый модуль управляет фирменным оверлеем и видимостью `#installBanner`; `s-install` переиспользуется как iOS-инструкция; `yc_install_seen` заменён новым ключом состояния.
- **Вопрос 5 — Вариант 5A:** не показывать 30 дней после отказа; после установки — никогда; единый ключ `localStorage`; отказ и установка различаются.
- **Вопрос 6 — Вариант 6A:** фирменный оверлей — при загрузке (если условия выполнены), платформенный prompt — по кнопке; реактивный показ при позднем приходе события.
- **Вопрос 7 — Вариант 7A:** только мобильные; десктоп — без предложения, но без поломки.
- **Вопрос 8:** `CACHE_VERSION` `v37` → `v38`; тексты-плейсхолдеры; метрики и манифест — вне рамок.

Отвергнуты: 1B (смешение ответственности, поздняя инициализация), 1C (нетестируемый инлайн), 2B (запрещён политикой браузера), 3B (не подавляет для установивших на iOS), 4B (рассинхрон двух состояний), 5B (менее гибко для удержания), 6B (сводит к текущему пассивному баннеру), 7B (вне фокуса BA).

### Границы решения

- Изменяется **только** механизм предложения установки: новый модуль, фирменный оверлей, управление `#installBanner`, переиспользование `s-install`, замена `yc_install_seen` единым ключом состояния.
- **Не затрагиваются:** сценарий записи (`s-services`/`s-masters`/`s-slots`/`s-confirm`), отзывы (`s-review`, `/comments`, Worker `/review`), push-логика (`docs/modules/push.js`, Worker `/subscribe`/`/send`/cron), содержимое манифеста и иконки, состав запросов к YCLIENTS. Установка лишь создаёт почву для push — сама push-логика вне рамок (BA «Вне рамок»).
- Принудительной установки/блокировки функционала нет: приложение полностью работает без установки (ФТ-10).

## Последствия

- **Положительные:**
  - Единый источник истины об установке — нет двух конкурирующих предложений; состояния «показано/отклонено/установлено» непротиворечивы (ФТ-9, КП-7, КП-8).
  - Автоматическое фирменное предложение при открытии на мобильном (ФТ-1, ФТ-8); программная установка на Android/Chrome (ФТ-3), ручная инструкция на iOS (ФТ-4).
  - Подавление для установивших (ФТ-2) и контролируемая частота повторов (ФТ-6, НФТ-3).
  - Предсказуемая деградация без ошибок в консоли в неподдерживаемых браузерах (ФТ-10, НФТ-5); офлайн/устанавливаемость не регрессируют (НФТ-2); без новых рантайм-зависимостей (НФТ-1).
- **Отрицательные / компромиссы:**
  - Android-prompt доступен только когда браузер сгенерировал `beforeinstallprompt` — на части открытий программная установка недоступна (на не-iOS оверлей тогда не показывается). Принято: ограничение платформы, а не дефект.
  - UA-детект iOS приблизителен — но это единственный путь на iOS; ошибка не ломает UI.
  - «30 дней» и порог мобильной аудитории — продуктовые значения по умолчанию (вынесены в константы, корректируются без переписывания логики).
  - Состояние локально на устройстве: на новом устройстве предложение появится снова (приемлемо, BA-допущение).
- **Технический долг:**
  - Метрики установок (показы/принятия/отказы) — отложены (BA «Вне рамок»); модуль не закрывает возможность добавить их точечно.
  - Полная зачистка legacy-ключа `yc_install_seen` у существующих пользователей — прекращение записи/чтения в этом релизе; физическое удаление ключа из `localStorage` пользователей — при следующем касании.
  - Десктопная установка как отдельная оптимизация — отложена (Вопрос 7B).

## Влияние на код

- **Новый модуль `docs/modules/install.js`** — единая логика: перехват `beforeinstallprompt` (`preventDefault` + хранение `deferredPrompt`), подписка на `appinstalled`, чистые функции `isStandalone()`/`isIOS()`/`shouldOffer()`, правило повторного показа (константа дней по умолчанию 30), показ/скрытие фирменного оверлея, управление видимостью `#installBanner`, ветка iOS (переход/показ `s-install`), запись состояния в новый ключ `localStorage`. Bindings на `window` (как другие модули). Инициализатор (например `initInstall()`).
- **`docs/app.js`** — импорт и вызов инициализатора модуля `install.js` в boot (рано, чтобы перехватить событие); регистрация bindings. Существующая регистрация SW и `initApp` — без изменений по смыслу.
- **`docs/modules/profile.js`** — `renderHomeHero` перестаёт сам управлять `#installBanner` по `yc_install_seen`; видимость баннера делегируется модулю `install.js` (вызов его функции либо удаление прямой строки `installBanner.style.display = …`). Прочая логика `renderHomeHero` — без изменений.
- **`docs/prototype.html`** — добавляется разметка фирменного оверлея (баннер/модал) в стиле приложения; кнопки `s-install` («Понятно, добавлю»/«Позже») переводятся с `yc_install_seen` на функции модуля; `#installBanner` сохраняется как точка входа (обработчик может остаться `go('s-install')` для iOS или вызывать модуль).
- **`docs/style.css`** — стили фирменного оверлея/баннера (светлая/тёмная тема), без поломки существующих классов.
- **`docs/modules/constants.js`** *(опционально)* — если константы (дней до повтора, ключ состояния, порог мобильной ширины) выносятся сюда; иначе — внутри `install.js`.
- **`docs/sw.js`** — в `APP_SHELL` добавляется `./modules/install.js`; `CACHE_VERSION` поднимается `v37` → `v38` (изменён состав `APP_SHELL` и код модулей в нём).
- **`docs/version.json`** — `build`/`version` обновляются при релизе (мейнтейнер).
- **Новые рантайм-зависимости:** нет. Манифест и иконки не трогаются.
- **Что ломается:** заменяется механизм показа предложения установки (пассивный баннер по `yc_install_seen` → единый модуль с оверлеем; намеренно). Сценарий записи, отзывы, push, навигация, внешний вид прочих экранов — не ломаются.
