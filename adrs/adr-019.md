# ADR-019: Лента в разрезе реальных категорий/услуг YCLIENTS — типизированный пост (free/category/service) с маршрутизацией CTA

- **Дата:** 2026-05-29
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-022.md` (открытые вопросы ОВ-1…ОВ-8 решены заказчиком 2026-05-29; см. §«Принятые решения по ОВ»)
- **Связанные ADR:** `adrs/adr-010.md` (экран одной публикации `s-post`, общий рендер карточки `_feedCardHtml` для `s-feed` и `s-post`, мягкий фолбэк) — переиспользуется, единообразие CTA обязано работать в обоих местах; `adrs/adr-013.md` (детальная карточка услуги `s-service`, `openServiceCard(serviceId)`) — целевой экран CTA поста по услуге; `adrs/adr-001.md`, `adrs/adr-002.md` (источник данных каталога услуг/категорий). Конфликтов не выявлено.

## Контекст

Лента студии (`s-feed`, `_paintFeed`, `docs/modules/feed.js`) живёт отдельно от реального каталога. Пост привязан к произвольной строке-категории из хардкод-чипов админ-формы (`Брови / Ногти / Лицо / Волосы / Тело / Акции`, fallback `Другое`; поле `cat` в модели поста, `admin.js:109-115`). Эти строки не совпадают с реальными категориями YCLIENTS (`catMap` из `GET /service_categories/{company}`, `SERVICES_DATA` из `GET /services/{company}`, `app.js:186-200`; реальные категории, например `Брови, Губы, Глаза, Эпиляция`). Из-за рассинхрона строк CTA «Записаться →» (`_feedGoBook(p.cat)` → `filterCategory(cat)` + `go('s-services')`, `feed.js:68,111-114`) часто приводит к «Ничего не найдено». Перехода на конкретную услугу из поста нет.

Бизнес-цель: перевести ленту в разрез реальных категорий/услуг, чтобы CTA доводил клиента до записи. В частности — пост «по услуге» по кнопке «Записаться» открывает карточку именно этой услуги (`s-service`, ADR-013). Эталон: «Акция на стрижку 20%» → «Записаться» → карточка услуги «Стрижка».

Силы, действующие на решение:

- **Источник данных уже загружен.** `SERVICES_DATA` (`{id:String, name, cat, ...}`) и `catMap` грузятся при старте (`app.js:186-200`). Услуга имеет стабильный `id` (используется в `openServiceCard`, ADR-013), реальная категория услуги — строка `cat`. `filterCategory(cat)` фильтрует каталог по названию категории-строки (`services.js:61-68`).
- **Общий рендер карточки.** `_feedCardHtml(p)` (`feed.js:54-74`) используется и в `s-feed` (`_paintFeed`), и в `s-post` (`renderPostScreen`), и частично дублируется в превью на главной (`_renderHomeFeedPreview`, `profile.js:135-163`). Любое изменение полей/CTA затрагивает все три места.
- **Визуал завязан на `cat`.** Градиент/иконка карточки берутся из `_CAT_GRADS`/`_CAT_ICONS2` по `p.cat` (`feed.js:5-13`, дубль в `profile.js:147` и `admin.js:160`). Реальные категории (`Губы`, `Глаза`, `Эпиляция`) в этих словарях отсутствуют → плейсхолдер `📝` / серый градиент.
- **Синхронизация постов.** Посты в `localStorage` `yc_feed_posts` и в GitHub `posts.json` (`_ghSyncPosts`/`_ghPullToLocal`, `github.js`). Критично: `_ghRead` фильтрует посты валидатором `_VALID_POST` (`github.js:4-7`), который **требует** `typeof p.cat === 'string'`. Пост без строкового `cat` будет молча отброшен при чтении из GitHub — это блокирует кросс-девайс синхронизацию новых постов, если поле `cat` убрать.
- **Старые посты.** Существующие посты имеют только `cat` (хардкод-строка), без `type`. Их нельзя ломать (`s-feed`, `s-post`, превью на главной).
- **Ограничения проекта.** Vanilla JS/HTML/CSS, без рантайм-npm-зависимостей; код в `docs/`; новые/тяжёлые запросы к YCLIENTS запрещены; при изменении файлов `APP_SHELL` — поднятие `CACHE_VERSION` в `sw.js`.

## Принятые решения по открытым вопросам (зафиксировано заказчиком)

- **ОВ-1.** Категории/услуги для админ-формы берутся из уже загруженных `SERVICES_DATA`/`catMap`; новые запросы не вводятся.
- **ОВ-2.** Старые посты (поле `cat` без `type`) трактуются как «свободные» (`free`) до перепубликации. Без миграции данных, без угадывания категории маппингом строк.
- **ОВ-3.** У свободного поста CTA «Записаться» скрывается.
- **ОВ-4.** Фильтр ленты заменяется на фильтр по типу поста (а не по строке `cat`).
- **ОВ-5.** Пост с привязкой к мастеру — вне scope v1.17.0.
- **ОВ-6.** Одна услуга на пост (мульти-услуги вне scope).
- **ОВ-7.** Промо-бейдж/скидка в v1.17.0 не вводится (кейс акции покрыт типом «по услуге» + текстом поста).
- **ОВ-8.** Если `serviceId`/категория из поста больше не существует в `SERVICES_DATA` — graceful fallback (не падать, не вести в тупик).

## Рассматриваемые варианты

### Решение 1 — модель поста: как кодировать тип и привязку

#### Вариант A: новые поля `type` + `serviceId`/`serviceName`, поле `cat` оставить как «отображаемую категорию» (denormalized)
- Модель: `{ id, type:'free'|'category'|'service', cat, serviceId?, serviceName?, text, image, date, draft }`.
- Для `category` `cat` = реальное название категории YCLIENTS (для `filterCategory` и для визуала/тега).
- Для `service` `cat` = категория услуги (`SERVICES_DATA.cat`) для визуала/тега, `serviceId`+`serviceName` — привязка.
- Для `free` `cat` = пустая строка `''` (или старая строка у унаследованных постов).
- **Плюсы:** `cat` остаётся строкой всегда → `_VALID_POST` (`github.js:4-7`) **не требует изменений**, кросс-девайс синхронизация новых постов работает без правки валидатора; визуальный градиент/иконка (`feed.js:5-13`) и тег карточки переиспользуются как есть; старые посты (`type` отсутствует) естественно читаются — отсутствие `type` трактуется как `free` (ОВ-2); `filterCategory(cat)` вызывается без преобразований.
- **Минусы:** дублирование (`cat` хранит производное от привязки); словари градиентов/иконок не покрывают новые реальные категории (`Губы`, `Глаза`…) → плейсхолдер — приемлемо (визуал не регрессирует, просто generic для незнакомых категорий).
- **Последствия:** минимальная правка `github.js` (можно вовсе без правки валидатора); новые поля `type`/`serviceId`/`serviceName` проходят через `posts.json` как дополнительные ключи (JSON-friendly, `_VALID_POST` их не отбрасывает, т.к. проверяет только обязательные).

#### Вариант B: заменить `cat` на структурированную ссылку (`{type, ref:{categoryId|serviceId}}`), убрать строковый `cat`
- **Плюсы:** «чистая» модель без денормализации.
- **Минусы:** ломает `_VALID_POST` (требует `typeof p.cat === 'string'`) → правка валидатора обязательна, риск отбросить уже синхронизированные старые посты; ломает визуал по `cat` в трёх местах; требует резолва названия категории по id при каждом рендере; больше изменений вопреки НФТ-4 (минимальность).
- **Последствия:** заметно больший объём, выше регрессионный риск синхронизации.

**Выбран Вариант A.** Денормализованное хранение `cat` как «отображаемой/фильтровальной строки» сохраняет обратную совместимость валидатора синхронизации и визуала при минимуме изменений, что прямо соответствует ОВ-2 и НФТ-3/НФТ-4. Поле `type` — единственный новый управляющий признак; отсутствие `type` = `free` даёт graceful-трактовку старых постов без миграции.

### Решение 2 — маршрутизация CTA по типу

#### Вариант A: единый обработчик `_feedGoBook(type, ref)` с ветвлением по типу и graceful-фолбэком
- `service` → если услуга существует в `SERVICES_DATA` → `openServiceCard(serviceId)` (ADR-013, сам делает `go('s-service')`); иначе фолбэк.
- `category` → если категория существует среди `SERVICES_DATA.cat` → `filterCategory(cat)` + `go('s-services')`; иначе фолбэк.
- `free` → CTA не рендерится (ОВ-3).
- Фолбэк при удалённой привязке (ОВ-8): открыть общий каталог `s-services` без фильтра (`filterCategory('Все')` + `go('s-services')`), без ошибок в консоли.
- **Плюсы:** одна точка истины для CTA, единообразно для `s-feed` и `s-post` (ADR-010); фолбэк локализован.
- **Минусы:** требует передавать в кнопку и тип, и ссылку (через `data-*`).

#### Вариант B: оставить `_feedGoBook(cat)` и доклеить отдельный обработчик для услуг
- **Минусы:** две точки маршрутизации, риск рассинхрона поведения между `s-feed` и `s-post`, дублирование фолбэка.

**Выбран Вариант A.** Единый обработчик с ветвлением и data-атрибутами кнопки (`data-type`, `data-ref`) — минимально и единообразно.

### Решение 3 — фильтр ленты (ОВ-4)

#### Вариант A: фильтр по типу поста, три чипа «Все» / «Акции и услуги» / «Новости»
- «Акции и услуги» = посты `type ∈ {service, category}`; «Новости» = `type === 'free'` (включая унаследованные посты без `type`).
- Заменяет текущий фильтр по `cat` (`_renderFeedCats`/`_feedFilterCat`, `feed.js:87-109`).
- **Плюсы:** соответствует ОВ-4; стабильный фиксированный набор чипов (не зависит от данных); понятная клиенту семантика.
- **Минусы:** теряется фильтрация по конкретной категории — осознанно отложено (ОВ-4, v1).

#### Вариант B: комбинированный фильтр (тип + категория)
- **Минусы:** раздувает объём, вне решения ОВ-4 для v1.

**Выбран Вариант A.**

## Решение

1. **Модель поста** расширяется полем `type:'free'|'category'|'service'` и для `service` — `serviceId:String`, `serviceName:String`. Поле `cat` сохраняется как строка отображаемой/фильтровальной категории (для `category`/`service`) либо `''` (для `free`). Отсутствие `type` у поста = `free` (ОВ-2). Валидатор `_VALID_POST` не меняется по существу (см. §«Влияние на код» — допустимо добавить терпимость к `cat === ''`).
2. **Админ-форма** `#s-admin-post`: хардкод-чипы заменяются на переключатель типа (3 чипа: Свободный / По категории / По услуге) + условные контролы — селект реальной категории (из `catMap`/`new Set(SERVICES_DATA.map(s=>s.cat))`) и поиск/селект реальной услуги (из `SERVICES_DATA`, сохраняется `serviceId`+`serviceName`+`cat` услуги).
3. **CTA** маршрутизируется единым обработчиком по `type` с graceful-фолбэком на общий каталог при удалённой привязке (ОВ-8); для `free` CTA скрыт (ОВ-3). Поведение единообразно в `s-feed` и `s-post` (ADR-010); превью на главной не ломается.
4. **Фильтр ленты** заменяется на фильтр по типу поста (три фиксированных чипа).
5. Экран `s-service` (ADR-013), `openServiceCard`, `filterCategory` — переиспользуются без изменений.

## Последствия

- **Положительные:** CTA доводит до записи предсказуемо (на услугу/в релевантный каталог); контент ленты опирается на реальный каталог; обратная совместимость старых постов и кросс-девайс синхронизации сохранена; минимум изменений.
- **Отрицательные / компромиссы:** денормализация `cat` (дублирует производное от привязки) — осознанно ради совместимости валидатора и визуала; словари градиентов/иконок (`feed.js:5-13`) не покрывают все реальные категории → для незнакомых категорий generic-плейсхолдер (приемлемо, не регрессия).
- **Технический долг:** (а) расширение словарей градиентов/иконок под реальные категории — отдельная UI-задача; (б) промо-бейдж/скидка (ОВ-7), пост по мастеру (ОВ-5), мульти-услуги (ОВ-6), комбинированный фильтр тип+категория (ОВ-4) — отдельные требования; (в) дубль рендера превью в `profile.js` vs `_feedCardHtml` — не объединяется в этом релизе.

## Влияние на код

- `docs/modules/admin.js` — `publishPost`: читать тип поста и привязку из новой формы; собирать модель с `type`/`serviceId`/`serviceName`/`cat`. `_renderAdminFeedFromCache` — корректная подпись строки (тип/категория) без падений на постах без `cat`/`type`.
- `docs/modules/feed.js` — `_feedCardHtml`: рендер CTA по `type` (скрыт для `free`), data-атрибуты для маршрутизации; `_feedGoBook`: ветвление по типу + фолбэк ОВ-8; `_renderFeedCats`/`_feedFilterCat`: фильтр по типу поста; нормализация старого поста (нет `type` → `free`).
- `docs/modules/profile.js` — `_renderHomeFeedPreview`: устойчивость к постам без `cat` (тег/иконка), без ошибок.
- `docs/modules/github.js` — `_VALID_POST`: сохранить приём старых постов; убедиться, что `cat === ''` (свободный пост) проходит валидацию (сейчас требует непустую строку? — `typeof p.cat === 'string'` пропускает `''`, дополнительной правки может не потребоваться; новые поля `type`/`serviceId` валидатор не отбрасывает). Механизм синхронизации не меняется.
- `docs/prototype.html` — экран `#s-admin-post`: переключатель типа + условные контролы (селект категории, поиск услуги); удаление хардкод-чипов.
- `docs/app.js` — при необходимости хелпер заполнения списков категорий/услуг в форме из `SERVICES_DATA`/`catMap` (без новых запросов). `adminCat` (`app.js:304`) — переосмыслить/заменить.
- `docs/style.css` — стили переключателя типа/контролов (при необходимости).
- `docs/sw.js` — поднять `CACHE_VERSION` (затрагиваются файлы `APP_SHELL`).
- Новые зависимости: нет. Что ломается: ничего при соблюдении совместимости `cat`/`type` выше.
