# ADR-008: Контекстно-зависимый тап по мастеру в списке — прямой переход на выбор времени в потоке записи

- **Дата:** 2026-05-28
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-008.md`
- **Связанные ADR:** ADR-002 (фильтрация состава `s-masters` по услуге — не пересматривается), ADR-003 (чтение отзывов на `s-master` — не затрагивается)

## Контекст

Сценарий онлайн-записи: услуга (`s-services`) → список мастеров (`s-masters`) → мастер → время (`s-slots`) → подтверждение (`s-confirm`). Сейчас тап по карточке мастера в списке вызывает `openMasterCard(this.dataset.mid)` (`docs/modules/masters.js:16`) и открывает промежуточный экран-профиль `s-master`; только оттуда по кнопке «Записаться к этому мастеру» (`bookFromMaster` → `bookWithMaster`, `booking.js:266`) клиент идёт дальше. Между «выбрал мастера» и «выбираю время» вставлен лишний экран.

Бизнес-цель (BA, раздел 1): в **потоке записи** тап по мастеру должен вести **сразу** на `s-slots`, приводя поведение «конкретный мастер» в соответствие с уже работающим «Любой мастер» (`selectAnyMaster`, `masters.js:236`), который и сейчас ведёт прямо на `s-slots`.

**Решения по открытым вопросам BA приняты 2026-05-28** и являются обязательными входными ограничениями:
1. Экран `s-master` **сохраняется** (код `openMasterCard`/`bookFromMaster`/регистрация `s-master` в `SCREENS` не удаляются). Профиль + отзывы (ADR-003) остаётся доступным для **просмотра вне записи**.
2. Поведение **контекстно-зависимое**: в потоке записи (услуга выбрана) тап → прямой переход на время; в режиме просмотра (услуга не выбрана: «Все мастера», главный экран, поиск) тап → по-прежнему профиль `s-master`.
3. Архитектор подтверждает достаточность `state.masterId` для отображения мастера на целевых экранах прямого перехода.

Силы, действующие на решение:

- **Карточка `_masterCardHtml` — общий компонент.** Используется в трёх местах с разным контекстом: поток записи (`renderMasters`, ветка «от услуги»), режим «Все мастера студии» (`renderMasters` с `browseAll`), а похожая разметка с тем же `onclick="openMasterCard(...)"` — на главном экране (`profile.js:73`) и в поиске (`search.js:19`). Поведение тапа должно различаться **только** для потока записи.
- **Различающий признак уже в состоянии.** Поток записи однозначно опознаётся: услуга выбрана (`state.serviceId` установлен) И список открыт не в режиме просмотра «Все мастера» (`browseAll`). `services.js:140-148`: при входе в `s-masters` из услуги `selectService` ставит `state.serviceId` и обнуляет `state.masterId = null`. Режим «Все мастера» помечается флагом `state._mastersAll` (`_browseAllMasters`, `masters.js:41`), который `renderMasters` считывает и **сбрасывает** в начале рендера (`masters.js:47-48`) — то есть на момент тапа флаг уже не доступен в `state`.
- **Минимальность изменений (НФТ-6).** Желательно тронуть один модуль (`masters.js`, входит в `APP_SHELL`) и не менять контракты `selectMaster`/`openMasterCard`/`selectAnyMaster`, разметку `prototype.html`, точки входа `profile.js`/`search.js`.
- **Vanilla JS, без рантайм-зависимостей (НФТ-1);** при изменении файла из `APP_SHELL` — поднятие `CACHE_VERSION` в `sw.js` (правило проекта, НФТ-4).
- **Достоверность отображения мастера (НФТ-3).** Имя/роль/аватар на `s-slots`/`s-confirm` и `staff_id` в брони должны соответствовать выбранному мастеру без рассинхрона с UI.

Решение охватывает два связанных вопроса: (1) как технически сделать тап контекстно-зависимым; (2) достаточно ли `state.masterId` для корректного отображения и брони при прямом переходе. Конкретные имена функций сверх ключевого диспетчера, тексты и порядок правок — в спецификации релиза, не здесь.

## Рассматриваемые варианты

### Вопрос 1. Как сделать тап по карточке контекстно-зависимым

#### Вариант A: флаг режима в `_masterCardHtml` → разный `onclick` в разметке

`_masterCardHtml(m, i, total, mode)` принимает признак режима и подставляет либо `onclick="selectMaster(...)"` (поток записи), либо `onclick="openMasterCard(...)"` (просмотр). `renderMasters` передаёт режим, зная про `browseAll` и `state.serviceId`.

- **Плюсы:**
  - Решение принимается в момент рендера, где контекст (`browseAll`, наличие услуги) уже известен.
  - В DOM сразу «правильный» обработчик — нет дополнительной диспетчеризации в рантайме.
- **Минусы:**
  - `_masterCardHtml` используется **минимум в двух местах внутри `renderMasters`** (ветка `browseAll` и ветка «от услуги»), а похожая разметка дублируется в `profile.js`/`search.js` с захардкоженным `openMasterCard`. Чтобы решение было консистентным, пришлось бы протаскивать флаг через все вызовы карточки и держать в голове, что в `profile.js`/`search.js` он всегда «просмотр». Расширение сигнатуры компонента ради одного бита контекста.
  - Логика «что есть поток записи» размазывается по нескольким точкам вызова (каждая обязана корректно вычислить флаг), повышая риск рассинхрона при будущих правках.
- **Последствия:** изменение сигнатуры общего компонента + правки во всех его вызовах; больше площадь изменений вопреки НФТ-6.

#### Вариант B (выбран): единый диспетчер `tapMaster(id)` в `onclick` карточки

`_masterCardHtml` сохраняет единый `onclick="tapMaster(this.dataset.mid)"`. Новая функция-диспетчер `tapMaster(id)` решает в рантайме по состоянию: если это поток записи (услуга выбрана и список не в режиме «Все мастера») — вызывает `selectMaster(id)` (прямой переход на `s-slots`); иначе — `openMasterCard(id)` (профиль). Признак «поток записи vs просмотр» фиксируется при рендере `renderMasters` (где `browseAll` и `state.serviceId` уже известны) и считывается диспетчером.

- **Плюсы:**
  - **Одна точка принятия решения** — вся логика контекста в `tapMaster`, не размазана по местам вызова карточки.
  - `_masterCardHtml` остаётся неизменным по сигнатуре; единый `onclick` упрощает поддержку и согласуется с НФТ-6 (минимум изменений: один модуль `masters.js`).
  - Контракты `selectMaster`/`openMasterCard`/`selectAnyMaster` не меняются — диспетчер лишь выбирает между ними.
  - Поведение `profile.js`/`search.js` (вне записи) не требует правок: даже если они продолжат звать `openMasterCard` напрямую — это корректный «просмотр»; если в будущем перейдут на `tapMaster`, диспетчер сам распознает отсутствие услуги как «просмотр».
- **Минусы:**
  - Решение принимается в рантайме при тапе, а не «запекается» в разметку — нужен надёжный признак контекста. Закрывается явной установкой признака в `renderMasters` (а не косвенным выводом из `state.serviceId`, который в `state` всегда непустой по умолчанию — см. ниже).
- **Последствия:** добавляется функция `tapMaster` и модульный признак «список открыт в потоке записи»; `_masterCardHtml.onclick` меняется с `openMasterCard` на `tapMaster`.

**О признаке контекста.** `state.serviceId` по умолчанию непуст (`state.js:31` = `'svc1'`), поэтому «услуга выбрана» нельзя надёжно вывести из одного `state.serviceId`. Надёжный признак — то, в каком режиме `renderMasters` фактически отрисовал список: «от услуги» (поток записи) либо `browseAll` (просмотр). Этот режим однозначно известен внутри `renderMasters` (ветвление по `state._mastersAll`). Поэтому `renderMasters` при отрисовке выставляет модульный флаг (например, `_masterTapMode = 'book' | 'browse'`), а `tapMaster` читает его. Это устраняет зависимость от значения `state.serviceId` по умолчанию и от того, что флаг `_mastersAll` уже сброшен к моменту тапа.

### Вопрос 2. Достаточность `state.masterId` для отображения мастера и брони при прямом переходе

#### Вариант 2A (выбран): `state.masterId` достаточно; дозаполнение `masterName/masterAvatar/masterGrad` не требуется

Анализ кода целевых экранов прямого перехода:
- **`s-slots`** — `updateSlotsScreen` (`slots.js:14-31`) берёт мастера через `getMaster()` (`state.js:56`), который возвращает объект из `MASTERS_DATA` по `state.masterId`. Имя, роль, аватар (`avatar_big`/`avatar`), градиент, признак избранного читаются из этого объекта, **не** из `state.masterName/masterAvatar/masterGrad`.
- **`s-confirm`** — `updateConfirmScreen` (`slots.js:65-90`) аналогично через `getMaster()`; при `null` показывает «Любой свободный мастер».
- **`s-cancel`** — `cancelRecord` (`booking.js:175`) читает `rec.masterName` из сохранённой записи; запись пишет `_saveBookedRecord` (`booking.js:19,28`) тоже через `getMaster()` → `m.name`/`m.id`. То есть для отмены достаточно корректного `masterId` на момент брони.
- **Бронь (`staff_id`)** — `_bookWithSession` (`booking.js:45`) использует `getMaster()` → `m.id`. Состав списка `s-masters` в потоке записи — пересечение ответа YCLIENTS с `MASTERS_DATA` (ADR-002), поэтому выбранный мастер **гарантированно** присутствует в `MASTERS_DATA` и `getMaster()` его найдёт.

Существующий `selectMaster(id)` (`masters.js:112`) ставит только `state.masterId` и делает `go('s-slots')` — этого достаточно для всех перечисленных экранов и для брони.

- **Плюсы:** минимальные изменения; `selectMaster` используется как есть, без расширения; нет дублирования данных мастера в `state` (единый источник — `MASTERS_DATA` через `getMaster()`), что исключает рассинхрон UI и `masterId` (НФТ-3).
- **Минусы:** поля `state.masterName/masterAvatar/masterGrad` остаются незаполненными при прямом переходе. Это **безопасно**: их единственный потребитель — баннер «запись от мастера» на `s-services` (`services.js:15`, через `state.masterPreSelected && state.masterId`), который в данном потоке (услуга уже выбрана, на `s-services` мы не возвращаемся «от мастера») не активируется. `selectMaster` не ставит `masterPreSelected`, поэтому баннер не покажет пустых данных.

#### Вариант 2B (отвергнут): дозаполнять `masterName/masterAvatar/masterGrad` в `selectMaster` «для единообразия»

Скопировать поведение `bookWithMaster`/`bookServiceFromMaster`, заполнив поля мастера в `state`.

- **Минусы:** вводит второй источник данных мастера (поля `state.*` vs `getMaster()`), который может разойтись с `MASTERS_DATA`; целевые экраны эти поля всё равно не читают — заполнение «мёртвое». Если попутно выставить `masterPreSelected`, это изменит поведение `s-services` (баннер) — выход за рамки задачи. Лишний код вопреки НФТ-6 без функциональной пользы.

## Решение

- **Вопрос 1 — Вариант B:** единый диспетчер `tapMaster(id)` в `onclick` карточки. `renderMasters` при отрисовке списка выставляет модульный признак режима (поток записи vs просмотр); `tapMaster` по этому признаку вызывает `selectMaster(id)` (прямой переход на `s-slots`) либо `openMasterCard(id)` (профиль). `_masterCardHtml` меняет только `onclick` (`openMasterCard` → `tapMaster`); сигнатура компонента и контракты `selectMaster`/`openMasterCard`/`selectAnyMaster` не меняются.
- **Вопрос 2 — Вариант 2A:** `state.masterId` достаточно для корректного отображения мастера на `s-slots`/`s-confirm`/`s-cancel` и для привязки брони (`staff_id`). Дозаполнение `state.masterName/masterAvatar/masterGrad` при прямом переходе **не требуется и не выполняется**. Это формальное подтверждение по открытому вопросу BA №3.

Отвергнуты: A (расширение сигнатуры общего компонента и размазывание логики контекста по местам вызова), 2B (мёртвое дублирование данных мастера и риск рассинхрона/смены поведения `s-services`).

### Границы решения

- Изменяется **только** действие при тапе по карточке мастера: в потоке записи (`s-masters` открыт от услуги) — прямой переход на `s-slots`; вне записи — профиль `s-master` как прежде.
- Экран `s-master`, функции `openMasterCard`/`bookFromMaster`/`bookServiceFromMaster`, регистрация `s-master` в `SCREENS` — **сохраняются** (решение BA №1).
- **Не затрагиваются:** состав списка мастеров и фильтрация по услуге (ADR-002), чтение и рендер отзывов на `s-master` (ADR-003), режим «Все мастера студии» (`browseAll`), точки входа в профиль с главного экрана (`profile.js:73`) и из поиска (`search.js:19`), вариант «Любой мастер» (`selectAnyMaster`), содержимое `s-slots`/`s-confirm`, загрузка дат/слотов и создание записи, YCLIENTS-интеграция, захардкоженный токен.
- Недоступные мастера (`avail:false`) — без `onclick` в карточке (`masters.js:14,16`), поведение тапа для них не меняется (тапа нет).

## Последствия

- **Положительные:**
  - Путь записи короче на один экран в потоке «услуга → мастер → время» (бизнес-цель BA).
  - Поведение «конкретный мастер» согласовано с «Любой мастер» (оба ведут прямо на `s-slots`).
  - Профиль мастера с отзывами (ADR-003) сохранён для просмотра вне записи.
  - Минимальная площадь изменений: один модуль `masters.js`; контракты и разметка экранов не меняются (НФТ-6).
  - Единый источник данных мастера (`getMaster()` из `MASTERS_DATA`) — нет рассинхрона UI и `masterId` (НФТ-3).
- **Отрицательные / компромиссы:**
  - Решение о действии тапа принимается в рантайме через модульный признак режима, а не «запекается» в разметку. Закрыто явной установкой признака в `renderMasters`.
  - `profile.js`/`search.js` продолжают звать `openMasterCard` напрямую (это корректно — там услуга не выбрана). Возможный будущий переход этих точек на `tapMaster` — отдельная косметическая унификация, в этом релизе не делается.
- **Технический долг:**
  - Унификация точек входа в карточку (главный экран, поиск) на единый `tapMaster` — отложена (не нужна для цели релиза, поведение там и так корректное). Возврат — при следующем касании `profile.js`/`search.js`.
  - Поля `state.masterName/masterAvatar/masterGrad` остаются «полуживыми» (используются только баннером `s-services` через `bookWithMaster`/`bookServiceFromMaster`). Их консолидация вокруг `getMaster()` — вне рамок.

## Влияние на код

- **`docs/modules/masters.js`** — основной и (предположительно) единственный изменяемый модуль:
  - `_masterCardHtml` (стр. 16): `onclick="openMasterCard(this.dataset.mid)"` → `onclick="tapMaster(this.dataset.mid)"`. Прочая разметка карточки, `data-mid`, кнопка избранного, обработка недоступных мастеров — без изменений.
  - `renderMasters`: при отрисовке выставляет модульный признак режима списка (поток записи vs просмотр) — в ветке `browseAll` это «просмотр», в ветке «от услуги» — «поток записи». Прочая логика (ADR-002: запрос по услуге, пересечение, fail-open, пустое состояние, кэш, сортировка по избранному) — без изменений.
  - Новая функция `tapMaster(id)`: читает признак режима; в потоке записи вызывает `selectMaster(id)`, иначе `openMasterCard(id)`. Экспортируется в `window` (как прочие обработчики `onclick`).
  - `selectMaster`/`openMasterCard`/`selectAnyMaster`/`bookFromMaster`/`bookServiceFromMaster` — **без изменений** контракта.
- **`docs/modules/state.js`** — изменения, как правило, не требуются (признак режима — модульная переменная в `masters.js`, не глобальное состояние). Если по решению разработчика признак удобнее держать в `state` — допустимо добавить одно поле, но без изменения семантики существующих полей.
- **`docs/modules/profile.js`, `docs/modules/search.js`** — **без изменений** (продолжают открывать профиль вне записи).
- **`docs/prototype.html`** — **без изменений** (разметка экранов и `s-master` не трогаются).
- **`docs/sw.js`** — состав `APP_SHELL` не меняется (правится существующий модуль `masters.js`); `CACHE_VERSION` **поднимается на одну ступень** (v41 → v42), чтобы клиенты получили новый `masters.js`.
- **`docs/version.json`** — обновляется мейнтейнером при релизе (вне объёма реализации фичи).
- **Новые рантайм-зависимости:** нет. Токен не трогается.
- **Что ломается:** меняется только действие тапа по карточке мастера в потоке записи (намеренно, по требованию). Навигация вне записи, профиль/отзывы, «Любой мастер», бронирование, YCLIENTS-интеграция — не ломаются.
