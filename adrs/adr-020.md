# ADR-020: Тактильная отзывчивость интерфейса — единый CSS press feedback, централизованная haptic-утилита и CSS pulse бейджа уведомлений

- **Дата:** 2026-05-29
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-023.md` (закрывает ФТ-1…ФТ-11, НФТ-1…НФТ-7, КП-1…КП-10; фиксирует ОВ-1/ОВ-2/ОВ-3/ОВ-4/ОВ-5).
- **Связанные ADR:** прямых архитектурных конфликтов не выявлено. Затрагиваемые поверхности (`docs/style.css`, `docs/modules/inbox.js`, `docs/modules/booking.js|services.js|masters.js|slots.js|review.js|admin.js`, `docs/prototype.html`, `docs/sw.js`) не пересекаются с принятыми решениями ADR-001…ADR-019. ADR-018 трогает `_masterCardHtml`/`.fav-badge` — не пересекается. ADR-019 трогает `feed.js` (модель типизированных постов) — точечно интегрируется с press feedback на `.feed-cat-chip`, `.feed-card`, `.feed-book` без изменения их поведения. ADR-013 (`openServiceCard`) и ADR-010 (`_feedCardHtml`) — без изменений; haptic вызывается из обработчиков-входных точек, а не из самой логики.

## Контекст

Studio PWA — устанавливаемое статическое приложение, vanilla JS/HTML/CSS, отдаётся из `docs/` на GitHub Pages, без рантайм-npm-зависимостей. Текущий `CACHE_VERSION = 'v66'` (`docs/sw.js:1`).

BA-023 фиксирует три направления UX-полировки:

1. **Унифицированный press feedback (CSS, `:active`).** Сейчас в `docs/style.css` `:active`-эффекты есть только у `.tab-center-btn` (`:156`), `.btn-primary` (`:186`), `.svc-catalog-card` (`:282`), `.master-card` (`:317`), `.review-ext-platform` (`:723`), `.pin-key` (`:804`), `.inbox-bell-btn` (`:887`), `.inbox-clear-btn` (`:912`), `.inbox-dots` (`:940`), `.inbox-menu button` (`:960`) — реализованы разными примитивами (`transform: scale(...)` или `opacity: ...`), без единой системы. Многие интерактивные элементы — `.feed-cat-chip`, `.feed-book`, `.seg-btn`, `.audience-row`, `.push-template`, `.consent-check`, `.consent-item`, `.cs-add`, `.tip-btn`, `.chip`, `.tab` (кроме центральной), `.slot`, `.btn-ghost`, `.btn-secondary`, `.btn-danger`, `.btn-yandex`, `.hero-card`, `.review-ext-btn` — не имеют визуального отклика на нажатие.
2. **Haptic feedback (`navigator.vibrate`).** В кодовой базе не используется (проверено `grep navigator.vibrate|hapticTap|tap\(` по `docs/` — 0 совпадений). На Android-устройствах с поддержкой Vibration API клиент не получает тактильного подтверждения ключевых действий.
3. **Pulse-анимация бейджа `inbox-badge`.** Логика показа сохраняется (`updateInboxBadge`, `inbox.js:12-26`: `display:flex` при `count > 0`, `display:none` иначе; текст `count > 9 ? '9+' : count`). Стиль `.inbox-badge` (`style.css:890-902`) статичен.

BA в открытых вопросах ОВ-1…ОВ-5 дал явные рекомендации; пользователь подтвердил их во входных параметрах задачи. Архитектурно ADR закрепляет: единую систему CSS-токенов длительностей и амплитуд press feedback, единую утилиту-обёртку `hapticTap` в существующем модуле (без появления нового файла, чтобы не трогать `APP_SHELL` шире необходимого), CSS-механизм pulse через переключаемый класс `inbox-badge--pulse` (а не через CSS-селекторы по inline-стилю), и единый media-блок `prefers-reduced-motion: reduce` для отключения motion-эффектов и haptic.

Развилок несколько; их и фиксируем.

## Рассматриваемые варианты

### Вопрос 1. Структура CSS press feedback

**Q1.A (выбран).** Один общий блок `:active`-правил в `docs/style.css` (новая секция `/* ── PRESS FEEDBACK ── */` после `/* ── TAP INDICATOR ── */`, перед `/* ── INSTALL OVERLAY ── */`, ~строки 461). Внутри — 4 группы по типу элемента: «кнопки/действия» (`.btn-primary`, `.btn-secondary`, `.btn-danger`, `.btn-ghost`, `.btn-yandex`, `.feed-book`, `.tip-btn`, `.review-ext-btn`, `.cs-add`), «карточки/строки» (`.svc-catalog-card`, `.master-card`, `.hero-card`, `.feed-card`, `.audience-row`, `.push-template`, `.consent-item`, `.slot`), «чипы/переключатели» (`.chip`, `.feed-cat-chip`, `.seg-btn`, `.tab` — кроме центральной, `.consent-check`), «клавиши» (`.pin-key` — уже есть, унифицируется в группе). Уже существующие `:active`-правила консолидируются в группах со значениями этой группы; явные несоответствия (`.btn-primary:active { opacity: 0.85 }` vs `.btn-ghost` без эффекта) приводятся к одному стандарту. Параметры — фиксированные CSS-переменные `--press-scale-card`, `--press-scale-button`, `--press-scale-chip`, `--press-scale-key`, `--press-opacity-button`, `--press-duration`, объявляются в `:root` рядом с прочими токенами темы (`style.css:10-27`).

- **Плюсы:** единая точка обзора и правки, токены позволяют дальнейшую подкрутку без рассинхрона; нет дублирования; легко проверяется ревьюером (одна секция).
- **Минусы:** консолидация затрагивает уже существующие правила — есть риск визуального регресса там, где сейчас «работает». Митигация: для каждого существующего правила выбираются значения той же группы (карточки ~0.97-0.99, кнопки opacity ~0.85, клавиши ~0.91 — в пределах текущих значений), что не даёт регресса (КП-2/ФТ-2).
- **Последствия:** одна правка `style.css` затрагивает ~10-15 правил, новый блок объёмом ~40 строк. Файл в `APP_SHELL` → `CACHE_VERSION` поднимается (см. вопрос 5).

**Q1.B (отвергнут).** Раскидать `:active`-правила по существующим секциям рядом с базовыми стилями каждого элемента (как сейчас).

- **Минусы:** теряется единая точка обзора; легко рассинхронизировать значения для одной группы; дублирование `transition`/`transform`; новые элементы будут забывать про press feedback. Прямо противоречит ФТ-3 (единообразие по типу элемента).
- **Отвергнут.**

**Q1.C (отвергнут).** Один универсальный селектор `a, button, [onclick], .chip, .svc-catalog-card, .master-card, .slot, .hero-card { ... :active ... }` поверх существующего общего правила (`style.css:2`).

- **Минусы:** ломает `.tab-center-btn` (там `:active` применяется к вложенному `.tab-fab`, а не к самой кнопке — внешнее правило конфликтует с уже специфичным); затрагивает невидимые/служебные кнопки (`.inbox-dots`, `.push-notification`, ссылки в `.consent-link`), где press feedback не нужен или мешает; нарушает «не переделывать ради переделывания».
- **Отвергнут.**

### Вопрос 2. Реализация и расположение haptic-утилиты

**Q2.A (выбран).** Добавить функцию `hapticTap(kind)` в существующий `docs/modules/utils.js`. Файл уже в `APP_SHELL` (`sw.js:10`), импортируется из большинства модулей. Не появляется новый файл — не расширяется `APP_SHELL`. Подпись:

```
export function hapticTap(kind = 'tap') { ... }
```

`kind` — короткая строка-ярлык; во внутреннем словаре `_PATTERNS` соответствует длительности в мс. Категории и длительности (по ОВ-1):

| kind | длительность | контекст вызова |
|---|---|---|
| `'select'` | 10 мс | выбор услуги/мастера/слота (тап по карточке-варианту) |
| `'submit'` | 12 мс | подтверждение записи (финальная кнопка), отправка формы (отзыв, публикация поста, отправка push) |
| `'fab'` | 12 мс | тап по центральной FAB-кнопке таб-бара (`.tab-center-btn`) |
| `'tap'` | 10 мс | дефолт-фолбэк (зарезервирован, не вызывается явно в v1.18.0) |

Поведение:
- `if (!('vibrate' in navigator)) return;` — graceful no-op без записи в консоль (КП-5).
- `const mql = window.matchMedia && window.matchMedia('(prefers-reduced-motion: reduce)'); if (mql && mql.matches) return;` — уважаем `prefers-reduced-motion` (ОВ-4, ФТ-10/КП-8).
- `try { navigator.vibrate(_PATTERNS[kind] || _PATTERNS.tap); } catch { /* iOS-устройства, где `vibrate` есть в прототипе, но игнорируется — no-op */ }`.
- Возвращает `undefined`. Без побочных эффектов, кроме самого вызова `vibrate`.

Точки вызова в коде (полный перечень — ФТ-4/КП-3):

- `docs/modules/services.js` — `selectService(id)` (`:147`): `hapticTap('select')` в начале.
- `docs/modules/masters.js` — `tapMaster(id)` (`:251`) и `selectAnyMaster()` (`:246`): `hapticTap('select')`. **Не вызывается** в `selectMaster(id)` (`:122`), поскольку это вызывается и из `tapMaster`, и программно из других сценариев — двойной haptic не нужен. Также `bookFromMaster(id)` (`:231`) и `bookServiceFromMaster(masterId, serviceId)` (`:235`) — `hapticTap('submit')` (это ключевые «записаться»-точки в детальной карточке).
- `docs/modules/slots.js` — `selectSlot(el, time)` (`:211`): `hapticTap('select')` в начале.
- `docs/modules/booking.js` — `startBooking()` (`:138`): `hapticTap('submit')` в самом начале (до проверки сессии). Это финальный тап «Подтвердить запись» (`#s-confirm-pre .btn-primary` / `#s-slots .btn-primary`). Покрывает оба сценария бронирования (новая запись и rescheduling).
- `docs/modules/review.js` — `submitReview()` (`:105`): `hapticTap('submit')` в начале.
- `docs/modules/admin.js` — `publishPost(draft)` (`:103`): `hapticTap('submit')` при `draft === false` (публикация); при сохранении черновика — не нужно. `sendNewPush()` (`:480`) и `sendPush()` (`:513`): `hapticTap('submit')` в начале.
- `docs/prototype.html` — центральная FAB-кнопка (`.tab-center-btn`, шапка `<div class="tab tab-center" onclick="go('s-services','tab')">`, `prototype.html:95`): добавить `hapticTap('fab')` через минимально инвазивную точку — атрибут `onclick="hapticTap('fab');go('s-services','tab')"`. Для этого `hapticTap` экспортируется в `window` (см. ниже). Прочие тапы (тапы по табам, по кнопкам шапки, по `.inbox-bell-btn`, по кнопкам ленты «Записаться» из ленты — `_feedGoBook`) **не получают haptic** в v1.18.0 (намеренно минимальная подборка по ОВ-1; не вибрировать на каждый тап подряд, ФТ-4).

`window.hapticTap` экспортируется одной строкой в конце `utils.js`: `if (typeof window !== 'undefined') window.hapticTap = hapticTap;` — это нужно для вызова из inline-`onclick` в `prototype.html`. Прочие модули импортируют `hapticTap` обычным `import { hapticTap } from './utils.js';`.

- **Плюсы:** одно место для значений длительностей и проверки поддержки/`prefers-reduced-motion`; нет нового модуля и нет расширения `APP_SHELL`; обновление SW не требуется сверх обязательного (см. Q5); легко искать вызовы (`grep hapticTap`); legitimate-кросс-импорты не плодятся.
- **Минусы:** `utils.js` слегка вырастает в роли (помимо форматтеров теперь живёт UX-утилита). Митигация: новый блок выделен комментарием `/* ── HAPTIC ── */`, не пересекается с остальной логикой; будущее выделение в отдельный модуль — отдельный ADR.
- **Последствия:** `utils.js` (`APP_SHELL`) меняется → `CACHE_VERSION` поднимается. Точки вызова — 6 модулей (`services.js`, `masters.js`, `slots.js`, `booking.js`, `review.js`, `admin.js`) — все в `APP_SHELL`, всё равно бы потребовали `CACHE_VERSION`. `prototype.html` (`APP_SHELL`) — затрагивается только одной строкой шапки.

**Q2.B (отвергнут).** Новый модуль `docs/modules/haptic.js`, добавление в `APP_SHELL` (`sw.js`), импорт в каждом из 6 потребителей.

- **Минусы:** расширение `APP_SHELL` без функциональной выгоды; больше движущихся частей; новый файл нужно сопровождать комментарием/документацией. Никаких архитектурных преимуществ перед Q2.A.
- **Отвергнут.**

**Q2.C (отвергнут).** Прямые вызовы `navigator.vibrate(12)` в каждой точке без обёртки.

- **Минусы:** прямо нарушает ФТ-5/КП-4 (единая утилита-обёртка, ноль прямых вызовов `navigator.vibrate` вне обёртки). Невозможно централизованно обработать `prefers-reduced-motion` без размножения проверок.
- **Отвергнут.**

**Q2.D (отвергнут).** Делегирование через document-level listener (`document.addEventListener('click', e => { if (e.target.matches('[data-haptic]')) ... })`) — атрибуты `data-haptic` на элементах в HTML.

- **Минусы:** требует ревизии всех точек разметки (включая dynamically-rendered), привязка к DOM-структуре теряет связь с бизнес-логикой («подтверждение записи» — не разметка, а вызов API). Перенос ответственности из доменного слоя в presentation-слой ухудшает читаемость. Кроме того, асинхронное подтверждение записи (`startBooking` → запрос → успех) лучше haptic-сигналом отбить _в момент действия пользователя_, а не где-то в обработчике.
- **Отвергнут.**

### Вопрос 3. Реализация pulse-анимации бейджа

**Q3.A (выбран).** CSS-`@keyframes inboxBadgePulse` на `transform: scale(1) → scale(1.10) → scale(1)`, длительность 1.6s, `ease-in-out`, `infinite`. Применяется к классу `.inbox-badge.inbox-badge--pulse`. Класс переключается в `updateInboxBadge` (`docs/modules/inbox.js:12-26`) явно: при `count > 0` — `badge.classList.add('inbox-badge--pulse')`, при `count === 0` (включая ветку `catch`) — `badge.classList.remove('inbox-badge--pulse')`. Логика порога (`count > 0`) и текст (`count > 9 ? '9+' : count`) — без изменений (ФТ-11/КП-9).

Учёт `prefers-reduced-motion: reduce` — единый media-блок в `style.css`:

```
@media (prefers-reduced-motion: reduce) {
  .inbox-badge--pulse { animation: none; }
  /* press feedback тоже глушим — см. вопрос 4 */
}
```

`transform-origin` для бейджа — центр (`50% 50%`); `position: absolute; bottom: 4px; right: 4px;` сохраняется → scale 1.10 не выйдет за пределы кнопки-колокольчика (bell-btn 44×44, бейдж 18×18 в углу: при scale 1.10 размер 19.8×19.8, смещение центра остаётся в углу).

- **Плюсы:**
  - Чистая CSS-анимация на `transform` — без layout-thrash, не задействует CPU-перерисовку (ФТ-9/НФТ-2).
  - Управление через явный класс — надёжно, не зависит от inline-стиля `display`, не нарушается при перерисовке `updateInboxBadge` или при `PUSH_RECEIVED` (`inbox.js:106-115`).
  - Постоянный мягкий пульс (ОВ-3 рекомендация BA) — простая логика без JS-таймеров и без отслеживания «момента появления нового непрочитанного».
  - При `count === 0` класс снимается → анимация останавливается → нулевая нагрузка (ФТ-8/КП-7).
- **Минусы:** при первом `updateInboxBadge` после загрузки страницы пульс стартует с фазы 0% (scale 1.0), что не даёт «первого акцентного бампа». Это сознательный компромисс (ОВ-3 рекомендация BA — постоянный мягкий пульс предпочтительнее однократного бампа).
- **Последствия:** новые `@keyframes inboxBadgePulse` и правило `.inbox-badge--pulse` в `style.css` (~10 строк); в `inbox.js` — две строки (`classList.add` / `classList.remove`) в `updateInboxBadge`. Оба файла в `APP_SHELL`.

**Q3.B (отвергнут).** Применять анимацию напрямую к `.inbox-badge` через CSS-селектор атрибута: `.inbox-badge:not([style*="display: none"]) { animation: ... }`.

- **Минусы:** хрупко относительно изменений inline-стиля (если позже добавится `style="display: flex; ..."` с дополнительными свойствами, селектор может ломаться); сложнее читать и тестировать; «правильное» решение — отдельный класс.
- **Отвергнут.**

**Q3.C (отвергнут).** JS-управление анимацией через `Element.animate(...)` (Web Animations API) внутри `updateInboxBadge`.

- **Минусы:** ничем не лучше CSS-анимации в данном кейсе (одна циклическая ключевая последовательность), но добавляет JS-логику cleanup-пути (отмена анимации при `count === 0`, обработка PUSH_RECEIVED). Прямо противоречит ограничению BA-023 §6 («Реализация pulse — только CSS, без JS-таймеров и без `requestAnimationFrame`»).
- **Отвергнут.**

**Q3.D (отвергнут).** Один начальный «бамп» при появлении бейджа без постоянного пульса (через `animation-iteration-count: 1`).

- **Минусы:** прямо отклонено в ОВ-3 рекомендации BA (постоянный мягкий пульс, чтобы бейдж не «терялся»). Эффект «бамп один раз и забыли» не решает заявленную проблему «клиент пропускает непрочитанное».
- **Отвергнут.**

### Вопрос 4. Учёт `prefers-reduced-motion`

**Q4.A (выбран).** Один media-блок `@media (prefers-reduced-motion: reduce)` в конце секции `/* ── PRESS FEEDBACK ── */` в `style.css`. Внутри:

```
@media (prefers-reduced-motion: reduce) {
  .inbox-badge--pulse { animation: none; }
  /* Группа кнопок/действий: сводим press feedback к мгновенной смене opacity — без перехода. */
  .btn-primary:active, .btn-secondary:active, .btn-danger:active,
  .btn-ghost:active, .btn-yandex:active, .feed-book:active,
  .tip-btn:active, .review-ext-btn:active, .review-ext-platform:active,
  .cs-add:active, .inbox-bell-btn:active, .inbox-clear-btn:active { transition: none; }
  /* Группа карточек/чипов/клавиш: убираем масштабирование. */
  .svc-catalog-card:active, .master-card:active, .hero-card:active,
  .feed-card:active, .audience-row:active, .push-template:active,
  .consent-item:active, .slot:active,
  .chip:active, .feed-cat-chip:active, .seg-btn:active, .tab:not(.tab-center):active,
  .consent-check:active, .pin-key:active,
  .tab-center-btn:active .tab-fab { transform: none; transition: none; }
}
```

Haptic — отключается **внутри `hapticTap`** (Q2.A), не в CSS. Уважение `prefers-reduced-motion` для haptic — по ОВ-4 рекомендации BA (вибрация — motion-стимул, отключаем при `reduce`).

- **Плюсы:** один media-блок — одна точка чтения для ревьюера; полное соответствие ФТ-10/КП-8; для haptic проверка в JS через `matchMedia` живёт в одной обёртке; ноль дублирований.
- **Минусы:** список селекторов в media-блоке нужно поддерживать синхронным с основным списком press feedback. Митигация: оба списка расположены в одной секции `/* ── PRESS FEEDBACK ── */` — визуально близко друг к другу.
- **Последствия:** +~15 строк в `style.css` в той же секции.

**Q4.B (отвергнут).** Игнорировать `prefers-reduced-motion`, мотивируя «эффекты и так мягкие».

- **Минусы:** прямо нарушает ФТ-10/НФТ-3/КП-8 и общепринятые a11y-практики. Эмуляция в DevTools покажет неработающий КП-8.
- **Отвергнут.**

## Последствия

- **Положительные:**
  - Интерфейс перестаёт казаться «мёртвым»: каждый клик по интерактивному элементу даёт визуальную обратную связь, ключевые действия — тактильную; бейдж непрочитанных привлекает внимание мягким пульсом.
  - Единая утилита `hapticTap` и единый CSS-блок press feedback упрощают будущее расширение (новые экраны/элементы — добавление селектора в один блок и одной строки `hapticTap` в обработчик).

  - Не возникает.

## Риски

- **Риск (СРЕДНИЙ): визуальный регресс уже работающих `:active`-эффектов при консолидации.** → **Митигация:** значения токенов выбираются в пределах уже используемых (`--press-scale-card: 0.97` для `.master-card` — текущее значение `0.99`, для `.svc-catalog-card` — текущее `0.97`; компромисс `0.97` приемлем; альтернатива — два токена `--press-scale-card-tight: 0.99` и `--press-scale-card: 0.97`); `--press-opacity-button: 0.85` — текущее `.btn-primary` и `.review-ext-platform`; `--press-scale-key: 0.91` — текущее `.pin-key`; `--press-duration: 120ms` — в пределах ФТ-1 «80–120 мс». Ручная проверка по уже существующим элементам обязательна (тестер).
- **Риск (НИЗКИЙ): двойной haptic при тапе по карточке мастера.** → **Митигация:** `hapticTap('select')` вставляется только в `tapMaster`, не в `selectMaster`; внутри `selectMaster` нет своего обработчика DOM.
- **Риск (НИЗКИЙ): haptic срабатывает при асинхронной ошибке `startBooking` (когда запись не создалась).** → **Принято:** `hapticTap('submit')` сигнализирует _намерение пользователя_, не успех. Подтверждение факта — отдельный визуальный шаг (`go('s-confirm')`). Дополнительный haptic на успех в v1.18.0 не вводим (минимальность).
- **Риск (НИЗКИЙ): `navigator.vibrate` существует в прототипе на iOS Safari, но игнорируется.** → **Митигация:** `try/catch` в `hapticTap`; в консоль ничего не пишем (КП-5).
- **Риск (НИЗКИЙ): pulse бейджа выходит за границы кнопки-колокольчика при scale 1.10.** → **Митигация:** бейдж 18×18 в правом нижнем углу bell-btn 44×44; scale 1.10 → 19.8×19.8; центр остаётся в углу. Визуально проверяется тестером.
- **Риск (НИЗКИЙ): старый кэш SW отдаёт прежние файлы → клиенты не видят правок.** → **Митигация:** обязательное поднятие `CACHE_VERSION` `v66 → v67` (Q5).
- **Риск (НИЗКИЙ): новый класс `inbox-badge--pulse` не снимается при `count === 0` в ветке `catch` `updateInboxBadge`.** → **Митигация:** снятие класса — в обеих ветках (`else` и `catch`) `updateInboxBadge`, явно в спецификации.
- **Риск (НИЗКИЙ): inline-`onclick` в `prototype.html:95` ломается из-за того, что `hapticTap` ещё не загружен.** → **Митигация:** `utils.js` загружается ранним импортом в `app.js`/модулях, до пользовательских кликов; экспорт `window.hapticTap` выполняется при первом импорте `utils.js`; до этого тап по FAB просто не вызовет haptic (но `go('s-services','tab')` отработает, если `go` в `window`). Дополнительная защита — обёртка в inline: `onclick="(window.hapticTap||function(){})( 'fab'); go('s-services','tab')"` — **отвергнута** в пользу простоты; реальная задержка между загрузкой `utils.js` и первым тапом пренебрежима.
