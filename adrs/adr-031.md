# ADR-031: Месячный календарь и группировка слотов на экране «Выберите время»

- **Дата:** 2026-05-30
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-034.md` (источник `requirements/req-slots-time-picker-yclients-033.md`)
- **Связанные ADR:** —

## Контекст

Экран `s-slots` — ключевой шаг воронки онлайн-записи. Сейчас дата выбирается
горизонтальной лентой ближайших доступных дней (`_renderDates`: проход 21 дня
вперёд с фильтрацией по `booking_dates`), а слоты выводятся плоской сеткой 3×N
без группировки (`loadTimes`). Пользователь не видит весь месяц, навигация по
большому числу слотов затруднена, на экране присутствует карточка мастера
(`#masterStrip`), которой нет в референсном виде мобильного приложения YCLIENTS.

Требуется привести экран к привычному виду мобильного YCLIENTS (скриншот-референс
`IMG_20260530_230100.png`): полноценный месячный календарь-карточка с навигацией
по месяцам и реальной подгрузкой `book_dates`, слоты сгруппированы по периодам
суток со сворачиваемыми секциями. Сохраняется наша подпись кнопки («Записаться»),
убирается брендирование «Работает на yclients» и карточка мастера.

Действующие силы и ограничения:
- Только vanilla JS/HTML/CSS, без рантайм-зависимостей; код в `docs/`.
- Минимальные изменения: поток бронирования (`startBooking`, `s-confirm`,
  `state.slot/dateISO/dateFull`, перенос `_rescheduleId`) не трогать сверх
  необходимого.
- Сохранить санитизацию времени до `HH:MM` перед вставкой в `onclick`.
- Корректность тёмной темы (токены `var(--…)`), тактильный отклик (`hapticTap`).
- Эндпоинты неизменны: `GET /book_dates/{company}` (ответ `booking_dates` —
  массив ISO-дат), `GET /book_times/{company}/{staffId}/{iso}` (ответ `r.data` —
  массив `{time|datetime}`).
- Горизонт доступных дат, отдаваемый API, ограничен (≈ месяц вперёд).

## Рассматриваемые варианты

Решения, уже зафиксированные пользователем (полноценный месячный грид; реальная
подгрузка соседних месяцев; кнопка «Записаться»; удаление `#masterStrip`; без
«Работает на yclients»; сохранение `s-confirm`), пересмотру не подлежат. Ниже —
варианты реализации в рамках этих рамок.

### Решение по календарю

#### Вариант А: Месячный грид с подгрузкой `book_dates` на каждый посещённый месяц, кэш ответов в памяти сессии
- Рисуется полная сетка месяца (6 строк × 7 дней, ведущие/хвостовые ячейки
  пустые). Для каждого отображаемого месяца отдельный запрос `book_dates`,
  параметризованный диапазоном, либо берётся уже полученный набор и
  фильтруется. Ответы кэшируются в Map по ключу `YYYY-MM` на время жизни экрана.
- **Плюсы:** соответствует решению пользователя (реальная подгрузка); кэш
  исключает повторные запросы при листании туда-обратно; естественно ложится на
  существующий формат `booking_dates`.
- **Минусы:** усложнение `slots.js` (состояние текущего месяца, кэш, навигация);
  больше DOM-узлов, чем у ленты.
- **Последствия:** `_renderDates` заменяется на рендер месячного грида;
  появляется модульное состояние навигации; добавляются CSS-правила для грида.

#### Вариант Б: Один запрос всех дат + клиентская навигация по месяцам без подгрузки
- Загрузить `book_dates` один раз, разложить даты по месяцам, листать локально
  без новых запросов.
- **Плюсы:** проще, меньше запросов.
- **Минусы:** **противоречит зафиксированному решению пользователя** (реальная
  подгрузка соседних месяцев стрелками); горизонт API ограничен, реальной
  «навигации по месяцам» не получится.
- **Последствия:** отвергнут как нарушающий согласованные требования.

### Решение по группировке слотов

#### Вариант В: Группировка на клиенте по `HH:MM`, секции с шевроном, состояние раскрытия в DOM/памяти сессии
- В `loadTimes` распарсить санитизированное время в минуты, разложить по
  трём периодам (Утро `< 12:00`, День `12:00–16:59`, Вечер `≥ 17:00`), отрисовать
  непустые секции с заголовком и шевроном. Сворачивание — переключение класса на
  секции; состояние живёт только в текущем DOM экрана.
- **Плюсы:** не требует изменения API/формата; сохраняется санитизация и
  авто-выбор первого слота; минимальные изменения вне `loadTimes`.
- **Минусы:** разметка слотов усложняется (обёртки-секции вместо одного грида).
- **Последствия:** `#slotsGrid` превращается в контейнер секций; CSS добавляет
  стили секции/шеврона/анимации.

### Решение по удалению карточки мастера

#### Вариант Г: Удалить `#masterStrip` из разметки, ветку рендера в `updateSlotsScreen` сделать no-op-безопасной
- Удалить блок `#masterStrip` из `#s-slots`. В `updateSlotsScreen` обращение к
  `masterStrip` уже защищено `if (strip)`, но баннер переноса (`rescheduleBanner`)
  вставляется через `strip.insertAdjacentElement`. Нужно перепривязать вставку
  баннера к существующему якорю (`.page-title` экрана `s-slots`).
- **Плюсы:** карточка убрана согласно референсу; сценарий переноса сохраняет
  контекстный баннер.
- **Минусы:** требуется аккуратно сохранить логику баннера переноса при удалении
  его прежнего якоря.
- **Последствия:** правка `updateSlotsScreen` (якорь баннера) и разметки.

## Решение

Выбраны **Вариант А** (календарь), **Вариант В** (группировка слотов),
**Вариант Г** (удаление карточки мастера).

Обоснование:
- Вариант А — единственный, не противоречащий зафиксированному решению о реальной
  подгрузке соседних месяцев; кэш в памяти сессии устраняет лишние запросы и
  держит навигацию отзывчивой.
- Вариант В минимально вторгается в поток: меняется только формирование разметки
  внутри `loadTimes`, санитизация и авто-выбор первого слота сохраняются.
- Вариант Г убирает карточку мастера, сохраняя баннер переноса через
  перепривязку якоря.

Решения по открытым вопросам BA (подтверждены пользователем):
- **ОВ-1:** в сценарии переноса (`state._rescheduleId`) подпись кнопки остаётся
  «Перенести запись»; на «Записаться» меняется только обычный сценарий записи.
- **ОВ-2:** стрелка «назад» неактивна на текущем месяце (не уходим в прошлое);
  стрелка «вперёд» неактивна, когда соседний месяц выходит за горизонт доступных
  дат. Месяц без доступных дат отображается с приглушёнными числами и нейтральной
  подсказкой об отсутствии свободных дат.
- **ОВ-3:** лёгкая анимация раскрытия/сворачивания секций (и календаря) средствами
  CSS-перехода; при технических сложностях допустимо мгновенное переключение.

## Последствия

- **Положительные:** экран соответствует привычному виду YCLIENTS; обзор всего
  месяца; удобная навигация по слотам; убрано стороннее брендирование и лишняя
  карточка мастера.
- **Отрицательные / компромиссы:** рост сложности `slots.js` (состояние навигации,
  кэш месяцев) и числа DOM-узлов; больше CSS. Горизонт навигации ограничен тем,
  что отдаёт API — за горизонтом стрелка «вперёд» блокируется.
- **Технический долг:** drag-handle «раскрытия на полный месяц» (как «ручка» на
  референсе) не реализуется как отдельный жест — календарь сразу показывает полный
  месяц. Анимация может быть упрощена до мгновенной при сложностях. Возврат к
  этому — при отдельном требовании на жесты/полировку.

## Влияние на код

- `docs/prototype.html` (`#s-slots`): удаление `#masterStrip`; замена `.date-picker`
  (лента) на разметку календаря-карточки (заголовок месяца, стрелки, дни недели,
  грид); замена `#slotsGrid` на контейнер секций; подпись кнопки «Записаться».
  Проверить отсутствие «Работает на yclients» (в текущей разметке `#s-slots` её нет).
- `docs/modules/slots.js`: модульное состояние навигации месяца + кэш `book_dates`
  по `YYYY-MM`; рендер месячного грида с состояниями дат; навигация стрелками с
  блокировкой на границах; группировка слотов по периодам и сворачивание секций;
  перепривязка якоря баннера переноса; санитизация времени и авто-выбор сохранены.
- `docs/modules/booking.js`: текст «сброса» кнопки после успешной/неуспешной
  записи — на «Записаться» (строка ~88), не ломая ветку переноса.
- `docs/style.css`: стили календаря-карточки, дней недели, ячеек дат (состояния
  доступная/недоступная/выбранная/сегодня), секций слотов с шевроном и анимацией;
  тёмная тема через токены.
- `docs/sw.js`: при изменении содержимого `APP_SHELL`-файлов поднять
  `CACHE_VERSION` (текущая `v85`).
- `docs/version.json`: инкремент `build`, версия `v1.29.0`.
- Без новых зависимостей. Эндпоинты YCLIENTS неизменны. Поток `startBooking` /
  `s-confirm` / структура `state` не меняются.
