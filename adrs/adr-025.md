# ADR-025: Контактный блок студии (гибрид YCLIENTS /company + плейсхолдеры STUDIO)

- **Дата:** 2026-05-30
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-028.md`
- **Связанные ADR:** ADR-024 (токены/`user-select` тёмной темы — контактный блок не должен нарушать выделение текста; механика `theme-color` не затрагивается)

## Контекст

Studio PWA нигде не показывает контакты студии. После брони (`s-confirm`) клиент
не знает адреса и маршрута, в профиле (`s-profile`) нет блока «О студии». BA-028
требует компактный контактный блок на обоих экранах: название, адрес, маршрут на
карте, кликабельный телефон, часы работы.

Действующие силы и проверенные факты:

- `GET /company/1940850` → `success:true`, но **все** контактные поля пусты:
  `phone:""`, `phones:[]`, `address:""`, `schedule:""`, `coordinate_lat:0`,
  `coordinate_lon:0`, `social` пусты, `city:"Люберцы"`.
- `PUT /company/{id}` partner-токеном → `success:false` («Недопустимый формат
  запроса»). Запись настроек студии из приложения **невозможна**. Следовательно
  плейсхолдеры обязательны — без них блок будет пустым.
- Источник истины по адресу/координатам/ссылкам карт уже есть в репозитории:
  `docs/modules/constants.js` → `REVIEW_URLS` (Я.Карты с `ll=37.289422,55.466536`,
  2ГИС firm id `70000001042765322`), консистентно с Троицком, а не с `city`
  из API.
- Глобальный `user-select:none` (`docs/style.css:11-17`, введён в v1.20.0)
  запрещает выделение текста — телефон/адрес скопировать нельзя без локального
  исключения.
- Ограничения проекта (CLAUDE.md): vanilla HTML/CSS/JS, без рантайм-зависимостей,
  код в `docs/`; внешние ссылки `rel="noopener"`; строки экранируются через `esc`;
  при изменении файлов из `APP_SHELL` поднимать `CACHE_VERSION` в `docs/sw.js`.

ADR закрывает открытые вопросы BA ОВ-1…ОВ-6. Конфликтов с ADR-001…024 не
выявлено: затрагиваемые экраны и модули в существующих ADR не фиксируют
контактный блок; ADR-024 фиксирует токены и общий `user-select:none` — данное
решение добавляет **локальное** исключение через выделенный класс, не отменяя
глобальное правило.

## Рассматриваемые варианты

### Где держать дефолты STUDIO (ОВ-1)

#### Вариант A: объект `STUDIO` в `docs/modules/constants.js` (выбран)
Рядом с `REVIEW_URLS`, где уже лежат координаты и ссылки карт.
- Плюсы: ноль дополнительных сетевых запросов; источник карт и дефолтов в одном
  месте; статический модуль уже в `APP_SHELL` (доступен оффлайн); консистентно
  со стилем проекта.
- Минусы: правка дефолтов требует релиза (но они и должны меняться редко —
  приоритет всегда у непустых данных API).

#### Вариант B: отдельный `docs/studio.json`
- Плюсы: можно править без пересборки модулей.
- Минусы: лишний сетевой запрос/кэш-инвалидация, новый файл в `APP_SHELL`, две
  точки истины (json + constants), усложнение без выгоды — данные всё равно
  захардкожены в репозитории.

### Когда грузить /company (ОВ-2)

#### Вариант A: в `initApp()` в общий `Promise.all` на старте (выбран)
`app.js:166` уже агрегирует загрузку справочников (`/service_categories`,
`/services`, `/book_staff`, `/book_services`).
- Плюсы: один запрос на старте, данные готовы к моменту захода на `s-confirm`/
  `s-profile`; легко переиспользовать; естественная точка хранения результата.
- Минусы: один лишний параллельный запрос на старте (без влияния на критический
  путь — отрисовка блока не ждёт его, есть плейсхолдеры).

#### Вариант B: ленивая загрузка при заходе на экран
- Плюсы: запрос не делается, если экраны не открыты.
- Минусы: возможна задержка/мигание блока при первом заходе; дублирование логики
  загрузки на двух экранах; запрос всё равно почти всегда выполнится.

### Рендер блока (ОВ-3)

#### Вариант A: одна функция-рендер, переиспользуемая на обоих экранах (выбран)
- Плюсы: единый источник разметки и форматирования, нет расхождений между
  экранами, меньше кода.
- Минусы: функция параметризуется контекстом (компактный вариант для
  `s-confirm` vs секция профиля) — небольшая дополнительная сложность.

#### Вариант B: две независимые копии разметки
- Минусы: дублирование, риск рассинхронизации, нарушает минимальность изменений.

### Карты (ОВ-4)

#### Вариант A: «Маршрут» → Я.Карты по координатам, 2ГИС опционально (выбран)
- Плюсы: координаты уже есть, одна основная кнопка не загромождает блок; 2ГИС —
  вторая ссылка для тех, кто им пользуется.
- Минусы: две внешние ссылки. Решается визуальной иерархией (primary + ghost).

#### Вариант B: только одна карта
- Минусы: часть пользователей предпочитает 2ГИС; ссылка уже есть в `REVIEW_URLS`.

## Решение

Приняты рекомендации BA по всем открытым вопросам.

### ОВ-1. Дефолты STUDIO
Новый экспортируемый объект `STUDIO` в `docs/modules/constants.js` рядом с
`REVIEW_URLS`:

```
export const STUDIO = {
  title:    'Реснички, сестрички',
  address:  'Троицкий бульвар, 4, Троицк, Москва',
  lat:      55.466536,
  lon:      37.289422,
  phone:    '+7 495 123-45-67', // ПЛЕЙСХОЛДЕР — заменить на реальный телефон
  schedule: 'Ежедневно 10:00–21:00',
  mapYandex: REVIEW_URLS.yandex,           // ll=37.289422,55.466536
  map2gis:   REVIEW_URLS['2gis'],          // firm 70000001042765322
};
```

Телефон обязательно сопровождается комментарием-меткой (ФТ-10, КП-8).

### ОВ-2. Загрузка /company
`GET /company/{YC.company}` добавляется в `Promise.all` в `initApp()`
(`app.js:166`) с `.catch(() => ({}))`, как уже сделано для `book_services`.
Результат нормализуется и кладётся в память — поле `state.company` (см. ОВ-6 про
`localStorage`). Отрисовка контактного блока **не зависит** от успеха запроса:
функция-рендер всегда строит данные через слияние «API-поле, если непустое,
иначе плейсхолдер».

### ОВ-3. Общий рендер
Одна функция `renderStudioContacts(opts)` в новом модуле
`docs/modules/studio.js`. Она:
1. читает нормализованные данные компании из памяти (`getCompanyContacts()`),
2. сливает каждое поле с `STUDIO` (приоритет — непустое значение API),
3. возвращает HTML-строку контактного блока, все динамические значения через
   `esc`,
4. вызывается из `updateConfirmScreen()` (`modules/slots.js`) и
   `renderProfileScreen()` (`modules/profile.js`).

Параметр `opts.variant` (`'confirm' | 'profile'`) переключает обёртку:
компактная карточка под `#confirmRows` vs секция `settings-group` с заголовком
«О студии». Внутренности (строки контактов) общие.

Слияние полей (ФТ-6, приоритет API над плейсхолдером):

| Поле блока | Источник API | Плейсхолдер |
|---|---|---|
| Название | `company.title` (если непуст) | `STUDIO.title` |
| Адрес | `company.address` (если непуст) | `STUDIO.address` |
| Телефон | первый непустой из `company.phones[]` / `company.phone` | `STUDIO.phone` |
| Координаты карты | `coordinate_lat`/`coordinate_lon`, если оба ≠ 0 | `STUDIO.lat/lon` + `STUDIO.mapYandex` |
| Часы | `company.schedule` (если непуст) | `STUDIO.schedule` |

«Непустое» = строка после `trim()` не пустая; для координат — оба значения
численно ≠ 0.

### ОВ-4. Карты
Кнопка «Маршрут» (primary-стиль) → Я.Карты. Когда координаты пришли из API —
ссылка строится как `https://yandex.ru/maps/?pt=<lon>,<lat>&z=17&l=map`
(или `?ll=<lon>,<lat>`); при плейсхолдере используется `STUDIO.mapYandex`
(`REVIEW_URLS.yandex`, уже содержит нужные `ll`). Вторая, опциональная ссылка
2ГИС (ghost/мелкая) → `STUDIO.map2gis`. Обе ссылки — `target="_blank"
rel="noopener"` (ФТ-4, КП-4, КП-9).

### ОВ-5. Формат часов
`schedule` из API показывается «как есть» (после `esc`), если строка непуста;
иначе — `STUDIO.schedule` («Ежедневно 10:00–21:00»). Преобразование формата не
выполняется — поле свободного формата, владелец сам задаёт читаемый вид.

### ОВ-6. Оффлайн и localStorage-кэш
При недоступном/пустом `/company` блок показывает плейсхолдеры; экран не
ломается (ФТ-8, НФТ-3, КП-7). **Решение по кэшу:** да, кэшировать последний
успешный нормализованный ответ в `localStorage` под ключом `yc_company`.
Обоснование: при оффлайн-старте (PWA из кэша) `Promise.all` вернёт пусто, и без
кэша заполненные владельцем данные не показались бы. На старте: если запрос
успешен и вернул хотя бы одно непустое контактное поле — перезаписать
`yc_company`; иначе читать из `yc_company` как промежуточный источник, и лишь
затем падать на `STUDIO`. Приоритет источников: свежий API → `localStorage` →
`STUDIO`. Кэш хранит только публичные контактные поля, секретов в нём нет.

### Локальное исключение из user-select (ФТ-5, КП-5)
Контактный блок получает CSS-класс (напр. `.studio-contacts`), для которого
объявляется `user-select:text` (+ вендорные префиксы) и
`-webkit-touch-callout:default`, по образцу существующего исключения для
`input, textarea` (`docs/style.css:18-23`). Глобальное правило `body {
user-select:none }` (ADR-024 / v1.20.0) не меняется.

## Последствия

- **Положительные:** клиент после брони видит адрес/маршрут/телефон/часы и в
  профиле; данные автоматически «оживают» при заполнении админки без релиза
  (ФТ-7); единый рендер исключает расхождение экранов; оффлайн-устойчивость за
  счёт плейсхолдеров и кэша; ноль новых рантайм-зависимостей.
- **Отрицательные / компромиссы:** дефолты захардкожены — их правка требует
  релиза (приемлемо, меняются редко); один дополнительный запрос на старте
  (некритичный путь); `localStorage`-ключ `yc_company` добавляет небольшую
  поверхность состояния.
- **Технический долг:** телефон остаётся плейсхолдером до получения реального
  номера (помечен комментарием); запись настроек в YCLIENTS недоступна — вне
  scope; социальные сети/мессенджеры/email не выводятся (вне BA).

## Влияние на код

- `docs/modules/constants.js`: новый экспорт `STUDIO` (после `REVIEW_URLS`).
- `docs/modules/studio.js` (**новый** модуль): нормализация ответа `/company`,
  кэш `yc_company`, `getCompanyContacts()`, `renderStudioContacts(opts)`,
  построение ссылок карт. Регистрируется в `APP_SHELL` (`docs/sw.js`).
- `docs/app.js`: добавить `GET /company` в `Promise.all` `initApp()` (~166),
  сохранить нормализованный результат; импорт из `studio.js`.
- `docs/modules/slots.js`: `updateConfirmScreen()` дописывает блок
  `renderStudioContacts({variant:'confirm'})` после `#confirmRows`.
- `docs/modules/profile.js`: `renderProfileScreen()` рендерит секцию «О студии»
  через `renderStudioContacts({variant:'profile'})`.
- `docs/prototype.html`: контейнеры под блок на `s-confirm` (после `#confirmRows`,
  ~226) и в `s-profile` (новая `settings-group`, ~324). Возможна правка
  `s-confirm` HTML → файл в `APP_SHELL`.
- `docs/style.css`: класс `.studio-contacts` с локальным `user-select:text`;
  стили строк/кнопок блока (через существующие токены, без хардкода цвета).
- `docs/sw.js`: добавить `./modules/studio.js` в `APP_SHELL`, поднять
  `CACHE_VERSION` (затронуты файлы APP_SHELL).
- Новых рантайм-зависимостей нет. Направление зависимостей: `studio.js` зависит
  от `constants.js` и `utils.js` (`esc`); `slots.js`/`profile.js` зависят от
  `studio.js` — без циклов.
