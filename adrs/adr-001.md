# ADR-001: Визуальный каталог услуг — хранение фото и механизм привязки

- **Дата:** 2026-05-28
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-001.md`
- **Связанные ADR:** —

## Контекст

Экран выбора услуги `s-services` сейчас — текстовый список карточек (`docs/modules/services.js`, классы `.svc-card`/`.svc-line`/`.svc-body`/`.svc-right` в `docs/style.css`). Заказчик хочет визуальный премиум-каталог: карточка с крупной фотографией результата процедуры, названием, ценой и длительностью.

Силы, действующие на решение:

- **Стек неизменен.** Vanilla JS/HTML/CSS, статический хостинг из `docs/` на GitHub Pages, без рантайм-npm-зависимостей (НФТ-4).
- **Бэкенда нет.** Единственный внешний источник — YCLIENTS API (название, цена, длительность, категория). Поля услуги и источник не меняются (ФТ-9).
- **ID услуг нестабильны.** Привязка фото не может опираться на YCLIENTS-ID (ФТ-4). Заказчик зафиксировал: приоритет — точное название услуги, фолбэк — категория, иначе плейсхолдер.
- **Будущее — S3.** Заказчик уже принял целевое решение: тенантно-изолированное S3-совместимое хранилище, бэкенд возвращает PWA готовые URL. Но это будущее; для текущего MVP бэкенда нет. Архитектура MVP не должна противоречить этой миграции.
- **PWA/офлайн не должен регрессировать** (НФТ-5); влияние на `CACHE_VERSION` (`docs/sw.js`, текущий `v33`) — предмет проектирования.
- **Редактируемость данных** (ФТ-10): добавление услуги/фото не должно требовать переписывания логики рендера.

Решение охватывает три связанных вопроса: где физически лежат фото в MVP, как фото сопоставляется услуге, как карточка деградирует при отсутствии/ошибке фото. Подробности вёрстки и имена функций — в спецификации релиза, не здесь.

## Рассматриваемые варианты

### Вопрос 1. Где хранить фото в MVP (бэкенда нет)

#### Вариант 1A: Фото в репозитории `docs/images/`, отдаются GitHub Pages

Изображения коммитятся в `docs/images/services/`, ссылаются относительными путями.

- **Плюсы:**
  - Нулевая инфраструктура: тот же статический хостинг, тот же CD (`git push`).
  - Работает офлайн и под Service Worker без отдельной CORS-настройки (same-origin).
  - Версионируется вместе с кодом, откат через `git revert`.
  - Полностью соответствует НФТ-4 (без новых зависимостей и сервисов).
- **Минусы:**
  - Бинарники в git раздувают репозиторий; неудобно для частой смены контента силами не-разработчика.
  - Заказчик не может менять фото без участия разработки (но админ-CMS и так вне scope, см. BA «Вне рамок»).
- **Последствия:** требуется оптимизация изображений на этапе сборки контента (вес/размер, НФТ-2); состав APP_SHELL и стратегия кэша в `sw.js` пересматриваются.

#### Вариант 1B: Фото на внешнем CDN / стороннем object storage уже сейчас

Залить фото на внешний хостинг (Cloudinary free, imgix, публичный bucket) и ссылаться абсолютными URL.

- **Плюсы:**
  - Не раздувает git; ближе по форме к целевому S3 (абсолютные URL).
  - Возможны трансформации/ресайз на лету у части CDN.
- **Минусы:**
  - Новая внешняя зависимость и точка отказа вне нашего контроля; для премиум-бренда — риск недоступности.
  - Кросс-оригин: усложняется кэширование в Service Worker, нужны корректные CORS-заголовки.
  - Для MVP это инфраструктура, которой у проекта нет; противоречит духу НФТ-4 «в рамках стека».
  - Это «половинчатый S3»: всё равно придётся мигрировать на целевое тенантное хранилище.

### Вопрос 2. Механизм привязки фото к услуге

#### Вариант 2A: Карта сопоставления в JS-данных, разделённая на «по названию» и «по категории»

Отдельный модуль-данные (например `docs/modules/serviceImages.js`), экспортирующий две структуры: точные сопоставления «название → ресурс фото» и дефолты «категория → ресурс фото + эмодзи + градиент». Чистая функция резолвинга возвращает фото для услуги по приоритету: точное название (после нормализации) → категория → плейсхолдер.

- **Плюсы:**
  - Данные отделены от логики рендера: добавить услугу/фото = правка одного объекта (ФТ-10, КП-9).
  - Не зависит от YCLIENTS-ID (ФТ-4).
  - Резолвер — чистая функция, тестируема изолированно.
  - Форма «ключ → URL» совместима с будущим S3: при появлении бэкенда меняется только источник URL внутри резолвера, не его контракт.
- **Минусы:**
  - Привязка по названию хрупка к расхождениям написания в YCLIENTS; смягчается нормализацией (регистр, пробелы, типографика) и фолбэком на категорию.

#### Вариант 2B: Хранить путь к фото прямо в `SERVICES_DATA`

Добавить поле `img` в каждый объект услуги в `docs/modules/state.js`.

- **Плюсы:** проще на первый взгляд — нет отдельного модуля.
- **Минусы:**
  - `SERVICES_DATA` — кандидат на замену данными из YCLIENTS; смешивать визуальный слой с моделью данных услуги нельзя.
  - Не покрывает услуги мастера, приходящие из YCLIENTS в обход `SERVICES_DATA` (ветка `masterPreSelected` в `renderServices`), — там фото нужно резолвить так же.
  - Привязка «по категории» как фолбэк выражается неестественно.

### Вопрос 3. Деградация и производительность изображений

#### Вариант 3A: `loading="lazy"` + `onerror`-фолбэк на плейсхолдер, фото не в APP_SHELL

Карточка всегда рендерит контейнер-обложку фиксированного соотношения сторон. Если фото есть — `<img loading="lazy">` с `onerror`, переключающим на плейсхолдер (градиент + эмодзи категории). Фото кэшируются Service Worker лениво в RUNTIME_CACHE при первом запросе, в APP_SHELL не добавляются (чтобы не блокировать установку SW и не раздувать обязательный кэш).

- **Плюсы:**
  - Соответствует НФТ-2 (ленивая подгрузка, без блокировки) и НФТ-3 (фолбэк без поломки вёрстки).
  - Не ломает офлайн: APP_SHELL не растёт фото; уже просмотренные фото остаются в RUNTIME_CACHE.
  - `CACHE_VERSION` поднимается один раз из-за правок APP_SHELL (новый модуль данных + изменённый `services.js`/`style.css`), а не из-за каждого фото.
- **Минусы:** при первом офлайн-визите непросмотренные фото покажут плейсхолдер — приемлемо и согласуется с НФТ-3.

#### Вариант 3B: Добавить все фото в APP_SHELL для офлайн-доступности

- **Плюсы:** все фото доступны офлайн сразу.
- **Минусы:** раздувает обязательный кэш, замедляет установку SW, каждое новое фото требует подъёма `CACHE_VERSION`; противоречит НФТ-2.

## Решение

- **Вопрос 1 — выбран Вариант 1A** (фото в `docs/images/services/`, отдаются GitHub Pages). Для MVP без бэкенда это единственный путь, не вводящий новую инфраструктуру/зависимости и не нарушающий НФТ-4. Совместимость с будущим S3 обеспечивается не местом хранения, а контрактом резолвера (см. Вопрос 2): сегодня резолвер отдаёт относительный путь в `docs/images/`, завтра — абсолютный URL от бэкенда; вызывающий код не меняется.
- **Вопрос 2 — выбран Вариант 2A** (отдельный модуль-данные + чистая функция-резолвер, приоритет: точное название → категория → плейсхолдер). Это прямо удовлетворяет ФТ-4 и ФТ-10 и изолирует визуальный слой от модели данных услуги и от источника URL.
- **Вопрос 3 — выбран Вариант 3A** (`loading="lazy"` + `onerror`-фолбэк, фото в RUNTIME_CACHE, не в APP_SHELL).

Отвергнуты: 1B (преждевременная внешняя инфраструктура, «половинчатый S3», CORS-риски), 2B (смешение визуального слоя с моделью данных, не покрывает услуги мастера из YCLIENTS), 3B (нарушает НФТ-2, привязывает каждое фото к `CACHE_VERSION`).

### Путь миграции на целевой S3 (будущее, вне данного релиза)

Резолвер фото — единственная точка, знающая, как из «ключа услуги/категории» получить URL картинки. Контракт резолвера (вход: услуга; выход: либо URL/путь картинки, либо дескриптор плейсхолдера) при миграции сохраняется. Меняется лишь внутренняя реализация: вместо относительного пути в `docs/images/` резолвер берёт URL из данных, отданных бэкендом (например, поле, пришедшее вместе с услугой, или отдельный справочник с бэкенда). Слой рендера карточки, lazy-loading и `onerror`-деградация остаются без изменений.

## Последствия

- **Положительные:**
  - Нулевая новая инфраструктура; релиз доставляется тем же CD.
  - Визуальный слой полностью изолирован от модели данных и от способа доставки URL — миграция на S3 локализована в резолвере.
  - Привязка устойчива к смене YCLIENTS-ID; добавление контента — правка данных, не кода (ФТ-10).
  - Офлайн и PWA-устанавливаемость не регрессируют (НФТ-5).
- **Отрицательные / компромиссы:**
  - Бинарные фото в git увеличивают размер репозитория и требуют ручной оптимизации контента перед коммитом.
  - Смена фото силами заказчика без разработчика невозможна (админ-CMS вне scope по BA).
  - Привязка по названию чувствительна к расхождениям написания в YCLIENTS; смягчена нормализацией и фолбэком на категорию.
- **Технический долг:**
  - Миграция хранилища `docs/images/` → тенантный S3 + получение URL с бэкенда. Возврат — когда появится бэкенд (отдельное BA-требование/релиз). Долг локализован в резолвере фото и в составе кэшируемых ресурсов `sw.js`.
  - Возможная будущая автоматизация оптимизации изображений в сборке (сейчас — ручная).

## Влияние на код

- **Новый модуль данных+резолвер** для сопоставления «услуга/категория → фото/плейсхолдер» (`docs/modules/`). Содержит карту точных названий, карту категорий (фото + эмодзи + градиент), нормализацию названия и чистую функцию резолвинга. Не зависит от YCLIENTS-ID.
- **`docs/modules/services.js`** — рендер карточки переходит с текстовой строки на карточку-обложку; обе ветки `renderServices` (обычная и `masterPreSelected`) используют резолвер. Логика выбора/перехода (`selectService`), поиска и фильтра категорий — без изменений (ФТ-3, ФТ-6, ФТ-7).
- **`docs/style.css`** — новые/переработанные классы карточки с областью изображения фиксированного соотношения сторон и стилем плейсхолдера; согласованность тем (светлая/тёмная).
- **`docs/prototype.html`** — разметка экрана `s-services` (контейнер `#serviceList`, баннер мастера, поиск, фильтр) сохраняется; меняется только внутренний HTML карточек, генерируемый в `services.js`.
- **Новый каталог `docs/images/services/`** — оптимизированные фото.
- **`docs/sw.js`** — новый модуль данных добавляется в APP_SHELL; фото обслуживаются как same-origin (RUNTIME_CACHE, lazy), в APP_SHELL не добавляются. `CACHE_VERSION` поднимается (правка APP_SHELL).
- **Новые рантайм-зависимости:** нет.
- **Что ломается:** внешний вид экрана `s-services` (намеренно). Контракты навигации, поиска, фильтра, источника данных услуг — не ломаются.
