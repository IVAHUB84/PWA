# ADR-013: Детальная карточка услуги — отдельный экран `s-service`, рендеринг при тапе, маппинг `comment`/`image_group`, карусель фото на CSS scroll-snap

- **Дата:** 2026-05-29
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-014.md` (открытые вопросы №1, №3, №4, №5 РЕШЕНЫ заказчиком 2026-05-29; №2, №6, №7 — значения по умолчанию зафиксированы; остаётся один технический риск к проверке: хранит ли YCLIENTS >1 фото на услугу — см. §8 №1)
- **Связанные ADR:** `adrs/adr-012.md` (диапазон цен — переиспользуется как есть для строки цены на детальной карточке, не пересматривается); `adrs/adr-001.md` (визуальный каталог, плейсхолдеры — резолв обложки расширяется). Прямых пересечений/конфликтов с ADR-002…ADR-011 не выявлено. Затрагивает схему загрузки/маппинга услуг (`app.js`, `initApp`) и резолв обложки услуги (`serviceImages.js`).

## Контекст

В каталоге услуг (`s-services`) тап по карточке вызывает `selectService(id)` (`docs/modules/services.js:91,147-156`), который ставит `state.serviceId` и **немедленно** переходит к выбору мастера (`go('s-masters')`) либо, если мастер предвыбран, к выбору времени (`go('s-slots')`). Промежуточного экрана, на котором клиент мог бы изучить услугу (описание, фото работ), нет.

**Бизнес-цель (BA, раздел 1):** дать клиенту изучить услугу перед записью — описание и фотографии работ на отдельном детальном экране — и только осознанно перейти к выбору мастера. Дополнительно — показать в каталоге реальное фото услуги из YCLIENTS вместо текущего категорийного плейсхолдера, когда фото есть.

Силы, действующие на решение:

- **Источник данных уже загружен.** Услуги грузятся из `GET /services/{company}` в `app.js:167-176` и маппятся в `SERVICES_DATA` с полями `id, name, cat, dur, price_min, priceStr`. Поля `comment` (описание) и `image_group` (фото) приходят в ответе, но **отбрасываются** маппингом. BA ФТ-11/НФТ-2/КП-10 требуют: описание и фото берутся **из уже загруженного объекта**, без дополнительных запросов на каждую услугу.
- **Цена/длительность — уже агрегированы (ADR-012).** Для услуги `s.priceStr` уже содержит диапазон цен (агрегированный фоном проходом по мастерам, ADR-012, вариант A1/B1), `s.dur` — длительность. Детальная карточка (BA §8 №3, ФТ-12, КП-15) показывает цену (диапазон) + длительность — это **переиспользование** уже посчитанного `priceStr`, без обращения к логике ADR-012 и без её изменения.
- **Готовый образец детального экрана.** Профиль мастера `s-master` (`docs/prototype.html:160-164`) — это `<div class="screen">` со `status-bar`, `nav-bar` (кнопка «Назад» → `back()`) и пустым `scroll`-контейнером `#masterCardContent`, который заполняется императивно функцией `openMasterCard(id)` при тапе (`masters.js:128-222`). Экран `s-master` **не** регистрируется в `registerOnEnter` (`app.js:34-107`) — его контент зависит от выбранного объекта и рендерится в момент перехода. Стек навигации (`go`/`back`, `navHistory`, `navigation.js`) уже поддерживает такие экраны: `back()` возвращает на предыдущий экран и сохраняет его DOM-состояние (фильтр/поиск/баннер каталога не сбрасываются).
- **Резолв обложки игнорирует API.** `resolveServiceImage(service)` (`serviceImages.js:21-31`) сейчас резолвит обложку через пустой `SERVICE_PHOTO_MAP` (`{}`) с фолбэком на категорийный градиент+эмодзи (`CATEGORY_MAP`); `image_group` из API не используется. BA ФТ-9/КП-8 требуют: обложка каталога — первое фото из `image_group`; нет фото — категорийный плейсхолдер (как сейчас).
- **Неоднородность `image_group` в YCLIENTS (BA §6, явно отмечено).** У мастеров `image_group` — это **объект размеров** (`image_group.images.norm.path`, см. `masters.js:168`), а не массив. Для услуг ожидается **массив** фото (`image_group.images[]`, каждый элемент — объект с полями вроде `path`/`id`). Фактический формат фото услуги **не подтверждён на реальных данных** и должен быть проверен разработчиком; маппинг и резолв обязаны защищённо обрабатывать оба варианта и любые отсутствующие поля.
- **Технический риск >1 фото (BA §8 №1).** `POST /api/v1/images/service` по тому же `service_id` **перезаписывает** фото — не подтверждено, что YCLIENTS вообще хранит и отдаёт >1 фото на услугу. Дозагрузка нескольких фото — отдельная контентная задача, **вне объёма этого релиза**. Решение должно работать на **массиве** фото произвольной длины: 0 → плейсхолдер; 1 → статичное фото без листалки; N>1 → карусель. Карусель «включается сама», когда фото станет несколько.
- **Безопасность (НФТ-1, КП-12).** Описание (`comment`), URL фото, название услуги — внешние данные; выводятся только через `esc()`. Вставка непроверенного HTML недопустима.
- **Режим предвыбранного мастера (BA §8 №5, ФТ-4, КП-4).** Когда `state.masterPreSelected` — детальная карточка **пропускается**: тап по услуге ведёт сразу к выбору времени (`s-slots`), как сейчас. Детальная карточка — только в обычном потоке.
- **Стек, минимальность, PWA/офлайн (НФТ-4, НФТ-5, НФТ-6).** Vanilla JS/HTML/CSS, код в `docs/`, без рантайм-npm-зависимостей; правки строго в объёме; при изменении файлов `APP_SHELL` — поднятие `CACHE_VERSION` в `docs/sw.js`.

Решение охватывает: (A) как реализован сам экран детальной карточки (отдельный экран vs оверлей/инлайн); (B) когда и как рендерится контент карточки; (C) где «разрезается» тап по услуге (обычный поток vs режим предвыбора); (D) расширение маппинга услуги (`comment`/`image_group`) и подключение фото к резолву обложки; (E) реализация карусели нескольких фото; (F) нормализация неоднородной/неполной структуры `image_group` и деградация.

## Рассматриваемые варианты

> Состав карточки (название/описание/цена-диапазон+длительность/фото/кнопка; без категории и мастеров — §8 №3), формат галереи (горизонтальная карусель со свайпом и точками — §8 №4), поведение в режиме предвыбора (пропуск — §8 №5), поведение «Назад» (в каталог стандартным стеком — §8 №6), дефолты пустого описания/нет фото/качества фото (§8 №2, №7) — **закрыты заказчиком/значениями по умолчанию**. Архитектор их не пересматривает; ниже — инженерные развилки реализации.

### Вопрос A. Способ реализации экрана детальной карточки

#### Вариант A1 (выбран): отдельный экран `<div class="screen" id="s-service">` по образцу `s-master`

Добавить в `docs/prototype.html` новый экран `s-service` ровно по шаблону `s-master` (`prototype.html:160-164`): `status-bar` + `nav-bar` с кнопкой `‹ Назад` (`onclick="back()"`) + пустой `scroll`-контейнер `#serviceCardContent`. Контент заполняется императивно при тапе по услуге (Вопрос B). Навигация — через существующие `go('s-service')` / `back()`.

- **Плюсы:** прямое переиспользование готового, проверенного паттерна (`s-master`); естественно ложится на стек `go`/`back` — «Назад» возвращает в каталог, состояние каталога (фильтр/поиск/баннер) сохраняется само (BA §8 №6, ФТ-5, КП-5); минимальные изменения — один экран в разметке + одна функция рендера; не трогает соседние экраны.
- **Минусы:** ещё один `screen`-блок в `prototype.html` (файл `APP_SHELL` → поднятие `CACHE_VERSION`). Приемлемо — так устроены все экраны.
- **Последствия:** правка `prototype.html` (новый экран), новая функция рендера в `services.js`, поднятие `CACHE_VERSION`.

#### Вариант A2 (отвергнут): модальный оверлей поверх каталога (как фильтр категории `_openCatFilter`)

Рисовать карточку как полноэкранный/нижний оверлей `document.createElement('div')` поверх `s-services`, без отдельного экрана.

- **Минусы:** ломает единообразие — все детальные экраны проекта (мастер, слоты, подтверждение) — это `screen` со стеком `go`/`back`; оверлей не попадает в `navHistory`, аппаратная/жестовая «назад» и анимация перехода работали бы иначе; дублирование логики анимации. Оверлеи в проекте — для коротких пикеров (категория), не для полноценного экрана с галереей.
- **Последствия:** рассинхрон с навигационной моделью, лишняя сложность. Отвергнут.

### Вопрос B. Когда и как рендерится контент карточки

#### Вариант B1 (выбран): императивный рендер при тапе (`openServiceCard(id)`), как `openMasterCard`

Тап по услуге в обычном потоке вызывает `openServiceCard(id)`: ставит `state.serviceId = id`, заполняет `#serviceCardContent` через `innerHTML` (данные из `getService()` — уже в `SERVICES_DATA`), затем `go('s-service')`. Без `registerOnEnter` для `s-service` (контент зависит от конкретной услуги; при возврате `back()` DOM остаётся валидным до следующего тапа).

- **Плюсы:** один источник истины — функция рендера зовётся ровно при выборе услуги; данные уже в памяти (никаких запросов — ФТ-11, КП-10); полностью повторяет работающий паттерн `openMasterCard`; не нужно трогать `app.js`-регистрацию onEnter.
- **Минусы:** при прямом `go('s-service')` без предварительного `openServiceCard` экран был бы пуст — но такого пути в коде нет (вход только через тап по услуге).
- **Последствия:** новая функция `openServiceCard(id)` в `services.js`; экспорт в `window`.

#### Вариант B2 (отвергнут): рендер через `registerOnEnter('s-service', …)`

Регистрировать рендер по входу на экран, читая `state.serviceId`.

- **Минусы:** дублирует механику — `state.serviceId` всё равно ставится в момент тапа; для экрана, в который входят только из одной точки, onEnter-обвязка избыточна; `openMasterCard` (аналогичный экран) сознательно её не использует. Лишняя несимметрия с образцом.
- **Последствия:** избыточность. Отвергнут.

### Вопрос C. Где «разрезается» тап по услуге

#### Вариант C1 (выбран): развилка внутри `selectService(id)` по `state.masterPreSelected`

`selectService(id)` (`services.js:147-156`) перестраивается:
- **режим предвыбранного мастера** (`state.masterPreSelected`) — **текущее** поведение без изменений: `state.serviceId = id; go('s-slots')` (BA §8 №5, ФТ-4, КП-4);
- **обычный поток** — вместо немедленного `go('s-masters')` вызывается `openServiceCard(id)` (Вопрос B1), который ведёт на `s-service`.

Переход с детальной карточки к мастерам выполняет отдельный обработчик кнопки «Выбрать мастера» (`chooseMaster()` / inline): сбрасывает `state.masterId`/`state.masterName` (как сейчас в ветке обычного потока `selectService`) и `go('s-masters')`.

- **Плюсы:** одна точка ветвления — `selectService` остаётся единым входом тапа по каталогу; режим предвыбора не трогается (нулевой риск регрессии потока «пришёл от мастера»); логика сброса мастера сохраняется, просто переносится на кнопку «Выбрать мастера».
- **Минусы:** появляется маленькая новая функция перехода к мастерам. Минимально.
- **Последствия:** правка `selectService`; новая функция перехода к мастерам (или inline `onclick` на кнопке); `renderMasters` (через `registerOnEnter('s-masters')`) отрабатывает как сейчас — `state.serviceId` уже установлен.

#### Вариант C2 (отвергнут): новый отдельный обработчик тапа в `_renderList`, `selectService` не трогать

Завести `openServiceCard` как onclick карточки каталога, оставив `selectService` для кнопки на детальной карточке.

- **Минусы:** `selectService` уже разветвлён по `masterPreSelected` и должен пропускать детальную карточку в режиме предвыбора — значит ветвление по режиму всё равно нужно в обработчике тапа каталога; вынеся его из `selectService`, мы дублируем условие в двух местах. Единая точка (C1) проще и безопаснее.
- **Последствия:** дублирование условия предвыбора. Отвергнут.

### Вопрос D. Маппинг `comment`/`image_group` и подключение фото к резолву обложки

#### Вариант D1 (выбран): расширить маппинг в `app.js`, нормализовать фото в массив URL; `resolveServiceImage` использует первое фото

В маппинге услуг (`app.js:168-174`) добавить два поля:
- `comment: s.comment || ''` — описание (строка, может быть пустой);
- `photos: <нормализованный массив URL фото из s.image_group>` — массив строк-URL (см. Вопрос F для нормализации); пустой массив, если фото нет.

`resolveServiceImage(service)` (`serviceImages.js`) дополняется: если `service.photos` непуст — вернуть `{ type:'photo', src: service.photos[0], … }` (первое фото — обложка каталога, BA ФТ-9/КП-8); иначе — существующая логика (`SERVICE_PHOTO_MAP` → категорийный плейсхолдер). Приоритет: реальное фото услуги > `SERVICE_PHOTO_MAP` (ручной маппинг, сейчас пустой) > категорийный плейсхолдер.

- **Плюсы:** данные сохраняются один раз при загрузке, без запросов на тап (ФТ-11, НФТ-2, КП-10); нормализованный `photos` (массив URL) — единый формат и для обложки каталога (первый элемент), и для галереи на детальной карточке (весь массив) → одна структура обслуживает оба сценария; `resolveServiceImage` сохраняет текущую сигнатуру и фолбэки (минимальность, обратная совместимость с каталогом и поиском).
- **Минусы:** маппинг должен знать форму `image_group` услуги (Вопрос F); нормализация выносится в общий хелпер во избежание дублирования между `app.js` и резолвом.
- **Последствия:** правка маппинга `app.js`; правка `resolveServiceImage` (`serviceImages.js`); общий хелпер нормализации фото (в `serviceImages.js`, экспортируется и зовётся из `app.js`).

#### Вариант D2 (отвергнут): хранить сырой `image_group` в услуге, разбирать в каждом потребителе

Сохранять `s.image_group` как есть, парсить отдельно в резолве каталога и в рендере галереи.

- **Минусы:** дублирование логики разбора неоднородной структуры в двух местах (риск рассинхрона при правках); сырой объект тяжелее и провоцирует разные интерпретации формата. Нормализация при маппинге (D1) проще и безопаснее.
- **Последствия:** дублирование, рассинхрон. Отвергнут.

### Вопрос E. Реализация горизонтальной карусели нескольких фото

#### Вариант E1 (выбран): нативный CSS scroll-snap для свайпа + минимальный JS для точек-индикаторов

Контейнер галереи — горизонтальный flex/scroll-контейнер с `overflow-x:auto`, `scroll-snap-type:x mandatory`; каждое фото — слайд с `scroll-snap-align:center` фиксированной ширины (ширина макета). Свайп обеспечивается **нативной** прокруткой (тач/трекпад) — без рукописных touch-хендлеров. Точки-индикаторы (по числу фото) рендерятся под каруселью; активная точка подсвечивается по событию `scroll` контейнера (вычисление индекса по `scrollLeft / slideWidth`, throttle не обязателен при малом числе фото). При **одном** фото — статичное изображение без контейнера-карусели и без точек (BA ФТ-6). При **нуле** — категорийный плейсхолдер (Вопрос F, BA ФТ-7).

- **Плюсы:** без рантайм-зависимостей (НФТ-4) — карусель целиком на CSS + ~10 строк JS на индикаторы; нативный свайп = плавность и инерция «из коробки», корректно на iOS/Android; деградирует сама (1 фото → нет листалки; 0 → плейсхолдер) — границы по длине массива; CSS — в `style.css` (`APP_SHELL`).
- **Минусы:** подсветка активной точки требует слушателя `scroll` (снимается при ре-рендере экрана; экран рендерится заново при каждом тапе по услуге — старый DOM заменяется); зум/лайтбокс не входит (явно вне объёма, §8 №4).
- **Последствия:** CSS-классы карусели в `style.css`; JS рендера слайдов + точек + слушателя scroll в функции рендера карточки (`services.js`).

#### Вариант E2 (отвергнут): рукописный свайп на touch-событиях + transform-перелистывание

Самостоятельно обрабатывать `touchstart`/`touchmove`/`touchend`, двигать ленту через `translateX`.

- **Минусы:** заметно больше кода (порог свайпа, инерция, границы, отмена) при vanilla-стеке (против НФТ-5); риск багов на разных устройствах; дублирует то, что CSS scroll-snap даёт нативно. Несоразмерно для галереи из нескольких фото.
- **Последствия:** лишняя сложность, риск регрессий. Отвергнут.

### Вопрос F. Нормализация неоднородной/неполной `image_group` и деградация

#### Вариант F1 (выбран): защищённый хелпер `serviceImageUrls(image_group)` → массив URL, любые отсутствия → пустой массив; деградация фото на категорийный плейсхолдер

Единый хелпер (в `serviceImages.js`) принимает `image_group` услуги и возвращает массив URL-строк, защищённо обрабатывая форматы:
- **массив фото** `image_group.images` (ожидаемый формат услуги): для каждого элемента взять доступный URL-размер (например `el.path` или вложенный размер) с защитой от отсутствия поля;
- **объект размеров** (как у мастеров, `image_group.images.norm.path`) — если у услуги встретится такая форма: взять доступный путь как единственный URL;
- **отсутствие/`null`/неожиданная форма** → пустой массив `[]`.

Конкретную форму элемента массива (`images[].path` vs вложенный объект размеров) **разработчик подтверждает на реальных данных** (см. риск ниже) и реализует извлечение URL соответственно; хелпер обязан не падать на любом из вариантов и логировать ничего в консоль не должен. Деградация отображения:
- **нет фото** (`photos.length === 0`) — категорийный плейсхолдер (`CATEGORY_MAP`, градиент+эмодзи) и в каталоге, и на детальной карточке (BA §8 №2, ФТ-7, КП-6/КП-8);
- **битый URL фото** (`<img onerror>`) — переключение на категорийный плейсхолдер без поломки вёрстки (как уже сделано в каталоге, `services.js:89`), и в каталоге, и в галерее (BA ФТ-10, КП-9, НФТ-3).

- **Плюсы:** один источник разбора неоднородной структуры → нет рассинхрона (закрывает риск неоднородности `image_group` из BA §6); устойчивость к неполным/неожиданным данным (ноль ошибок в консоли — КП-11); деградация фото единообразна с уже работающим каталогом (`onerror` → плейсхолдер).
- **Минусы:** точная форма элемента не зафиксирована в ADR (зависит от реальных данных) — компенсируется требованием подтвердить на данных и защищённым разбором.
- **Последствия:** хелпер нормализации в `serviceImages.js`; `<img onerror>` на плейсхолдер в галерее по образцу каталога; разработчик подтверждает форму `image_group` услуги на реальном ответе `/services/`.

#### Вариант F2 (отвергнут): жёстко предположить `image_group.images[].path`, без защиты от иной формы

- **Минусы:** при неоднородности (BA прямо предупреждает) — `undefined`/ошибки в консоли или пустая галерея на валидных данных; хрупко относительно реального формата, который ещё не подтверждён.
- **Последствия:** риск ошибок и пустых галерей. Отвергнут.

## Решение

- **Вопрос A — A1:** детальная карточка — отдельный экран `<div class="screen" id="s-service">` в `docs/prototype.html` по образцу `s-master` (`status-bar` + `nav-bar` с `‹ Назад`→`back()` + пустой `scroll`-контейнер `#serviceCardContent`); навигация — существующие `go('s-service')`/`back()`; «Назад» возвращает в каталог, состояние каталога сохраняется стеком (BA §8 №6, ФТ-5).
- **Вопрос B — B1:** контент карточки рендерится императивно функцией `openServiceCard(id)` (по образцу `openMasterCard`): ставит `state.serviceId`, заполняет `#serviceCardContent` через `innerHTML` из `getService()` (данные уже в `SERVICES_DATA`, без запросов — ФТ-11, КП-10), затем `go('s-service')`. Без `registerOnEnter` для `s-service`.
- **Вопрос C — C1:** развилка в `selectService(id)`: при `state.masterPreSelected` — **без изменений** `go('s-slots')` (ФТ-4, КП-4); иначе — `openServiceCard(id)`. Переход к мастерам — отдельный обработчик кнопки «Выбрать мастера»: сброс `state.masterId`/`masterName` + `go('s-masters')`.
- **Вопрос D — D1:** маппинг услуги в `app.js` расширяется полями `comment` (строка) и `photos` (нормализованный массив URL из `image_group`); `resolveServiceImage` использует `photos[0]` как обложку каталога при наличии, иначе текущие фолбэки. Приоритет: фото услуги > `SERVICE_PHOTO_MAP` > категорийный плейсхолдер.
- **Вопрос E — E1:** галерея — нативный CSS scroll-snap (горизонтальный свайп) + минимальный JS для точек-индикаторов; при 1 фото — статичное изображение без листалки; при 0 — категорийный плейсхолдер. Зум/лайтбокс — вне объёма (§8 №4).
- **Вопрос F — F1:** защищённый хелпер `serviceImageUrls(image_group)` нормализует неоднородную/неполную структуру в массив URL (любые отсутствия → `[]`); форму элемента массива разработчик подтверждает на реальных данных; деградация: нет фото → категорийный плейсхолдер; битый URL → `<img onerror>` на плейсхолдер; ноль ошибок в консоли.
- **Цена/длительность (BA §8 №3, ФТ-12):** на детальной карточке показываются `s.priceStr` (диапазон, уже агрегирован ADR-012) и `s.dur` — **переиспользование** уже посчитанных значений; логика ADR-012 не вызывается и не меняется. Категория и список мастеров не показываются.
- **Безопасность (НФТ-1, КП-12):** `comment`, URL фото, `name` выводятся только через `esc()`.

Отвергнуты: A2 (оверлей вместо экрана — рассинхрон с навигационной моделью), B2 (избыточный onEnter), C2 (дублирование условия предвыбора), D2 (сырой `image_group` у потребителей — дублирование разбора), E2 (рукописный touch-свайп — несоразмерная сложность), F2 (жёсткое предположение формата — хрупко).

### Границы решения

- **Детальная карточка** показывает: название, описание (`comment`, скрыто при пустом — §8 №2), цена-диапазон + длительность, фото (галерея/одно/плейсхолдер), кнопку «Выбрать мастера». **Не** показывает категорию и список мастеров (§8 №3).
- **Каталог:** меняется только источник обложки (`resolveServiceImage` → первое фото из `image_group`); вёрстка, поиск, фильтр категории, баннер предвыбранного мастера, состояние «Ничего не найдено», строка цены/длительности — **без изменений** (BA scope, ФТ-14, КП-11).
- **Режим предвыбранного мастера** не затрагивается: тап ведёт сразу на `s-slots` (§8 №5, ФТ-4).
- **Не затрагиваются:** логика выбора мастера (`renderMasters`, `selectMaster`), расчёта слотов, бронирования и подтверждения; логика цен ADR-012 (`_fmtPriceRange`, сбор по мастерам) — только переиспользуется `priceStr`; карточка мастера `s-master`/`openMasterCard`; набор эндпоинтов YCLIENTS (никаких новых запросов); захардкоженный YCLIENTS-токен (известный техдолг — не усугублять).
- **Дозагрузка нескольких фото работ в YCLIENTS** — отдельная контентная задача, вне объёма этого релиза (§8 №1). PWA отображает то, что фактически есть в `image_group`.

## Последствия

- **Положительные:**
  - Клиент изучает услугу (описание + фото работ) на отдельном экране перед записью (бизнес-цель BA); повышение информированности выбора и воспринимаемой ценности.
  - Переиспользование проверенного паттерна `s-master`/`openMasterCard` и стека `go`/`back` → минимум нового кода, «Назад» и сохранение состояния каталога работают «бесплатно».
  - Описание и фото — из уже загруженного объекта услуги, без дополнительных запросов (НФТ-2, ФТ-11, КП-10).
  - Цена-диапазон + длительность переиспользуют результат ADR-012 без его изменения — нет рассинхрона цен.
  - Карусель на CSS scroll-snap — без рантайм-зависимостей, нативный свайп, автоматическая деградация (0/1/N фото).
  - Каталог получает реальное фото услуги из YCLIENTS с устойчивым фолбэком на категорийный плейсхолдер (ФТ-9, ФТ-10, КП-8, КП-9).
  - Решение робастно к неоднородной/неполной `image_group` (единый защищённый хелпер) — ноль ошибок в консоли.
- **Отрицательные / компромиссы:**
  - Дополнительный шаг в потоке записи (каталог → детальная карточка → мастер) — осознанный бизнес-выбор заказчика ради информированности; в режиме предвыбора шаг пропускается.
  - Пока YCLIENTS фактически отдаёт одно фото на услугу, галерея деградирует до одного изображения; карусель «включится сама» при появлении нескольких фото (зависит от внешней контентной задачи и технического подтверждения, см. риски).
  - Качество фото ≈420×280 из `image_group` — принято как приемлемое (§8 №7); более качественный источник — по отдельному запросу заказчика (техдолг).
- **Технический долг:**
  - Зум/полноэкранный просмотр (лайтбокс) — не делается (§8 №4); при необходимости — отдельным решением.
  - Источник более качественных фото — при запросе заказчика.
  - Унификация разбора `image_group` между услугами и мастерами (`masters.js:168`) — не консолидируется в этом релизе; к вопросу при следующем касании.

## Риски

- **Риск (ВЫСОКИЙ, к проверке разработчиком): YCLIENTS не хранит >1 фото на услугу.** `POST /api/v1/images/service` перезаписывает фото по `service_id`; не подтверждено, что API отдаёт массив из нескольких фото (§8 №1). → **Митигация:** решение спроектировано на **массив** фото произвольной длины (карусель «включается сама» при N>1); при одном/нуле фото деградирует корректно. Разработчик подтверждает на реальном ответе `/services/`, действительно ли `image_group.images` содержит >1 элемента; если нет — фича функциональна на одном фото, вопрос об альтернативном источнике эскалируется заказчику (вне объёма реализации).
- **Риск (СРЕДНИЙ, к проверке разработчиком): фактическая форма `image_group` услуги не подтверждена и неоднородна.** У мастеров — объект размеров (`image_group.images.norm.path`), у услуг ожидается массив `images[]`. → **Митигация:** защищённый хелпер `serviceImageUrls` (Вопрос F1) обрабатывает оба варианта и любые отсутствия (→ `[]`); разработчик подтверждает форму элемента на реальных данных и извлекает корректный URL-размер; ноль ошибок в консоли при любой форме.
- **Риск (НИЗКИЙ): кэш Service Worker отдаёт старую разметку/CSS/JS после релиза.** → **Митигация:** при изменении файлов `APP_SHELL` (`prototype.html`, `style.css`, `services.js`, `serviceImages.js`, `app.js`) **поднять `CACHE_VERSION`** в `docs/sw.js` (текущее значение `v56` → `v57`) — правило проекта (НФТ-6, КП-13).
- **Риск (НИЗКИЙ): XSS через `comment`/URL фото.** → **Митигация:** все внешние данные — только через `esc()` (НФТ-1, КП-12); вставка непроверенного HTML недопустима.
