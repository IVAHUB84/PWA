# ADR-030: Кликабельные уведомления (диплинки) — единый маршрутизатор цели для инбокса и `notificationclick`, точечная отметка о прочтении, поле `target` в payload/IDB, цель в композере и в напоминаниях

- **Дата:** 2026-05-30
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-033.md` (источник `requirements/req-inbox-clickable-notifications-032.md`; блокирующих открытых вопросов нет, §8 — некритичные развилки реализации, закрываются этим ADR)
- **Связанные ADR:**
  - `adrs/adr-015.md` — инбокс/IndexedDB, схема записи `{id,title,body,icon?,ts,read}`, поведение «заход в инбокс помечает всё прочитанным» (`markAllRead` при входе). **Частично пересматривается** этим ADR: поведение массовой отметки при входе вытесняется точечной отметкой при клике; схема записи расширяется полем `target`. См. раздел «Связь с ADR-015 / ADR-026».
  - `adrs/adr-026.md` — «честный» композер рассылок (текст + иконка, без декоративной аудитории/расписания). **Расширяется** этим ADR: в композер добавляется выбор цели.
  - `adrs/adr-006.md` — доставка push, схема подписок `sub:${clientId}:${endpointHash}`, `handleSend`/`handleCron`. Доставка/криптография/схема подписок **не пересматриваются**; в payload лишь добавляется необязательное поле `target`.
  - `adrs/adr-007.md` — per-client предпочтения `promo`/`remind`; не затрагивается.

## Контекст

Инбокс (ADR-015) сохраняет историю push в IndexedDB и показывает её на экране `s-inbox` (колокольчик на `s-home`), есть бейдж непрочитанных, дедуп, лимит 50, очистка. Композер админских рассылок приведён к «честному» виду (ADR-026): текст + иконка, отправка через `sendAdminPush` → Worker `/send`.

Подтверждено кодом:

- **Уведомления некликабельны.** `renderInbox()` (`docs/modules/inbox.js:44-57`) рисует `.inbox-item` только с меню «три точки» (`inboxToggleMenu`/`inboxMarkUnreadAt`/`inboxDeleteAt`); обработчика перехода по телу карточки нет.
- **Заход в инбокс помечает всё прочитанным.** `enterInbox()` (`inbox.js:96-100`) вызывает `markAllRead()` сразу при входе. Точечной отметки «прочитал именно это» нет.
- **Кнопка в шапке = «метёлка», действие «очистить всю историю».** `prototype.html:874-876` — SVG метёлки, `onclick="clearInboxHistory()"` (`inbox.js:102-107`, с `confirm()`).
- **В записи и payload нет цели.** Схема IDB-записи `{id,title,body,icon?,ts,read}` (`inboxStore.js:31-35`, инлайн-копия `_IDB_*` в `sw.js:163-228`). Payload рассылок `{title,body,icon}` (`sw.js`, `cf-worker/src/index.js:229`); payload напоминаний `{title,body,icon}` (`index.js:363-367`), хотя `rec.id`/`clientId` в обработчике доступны.
- **`notificationclick` всегда открывает корень.** `sw.js:262-272`: `url = event.notification.data?.url || './'`, `data` = `{url:'./'}` (`sw.js:240`). Диплинка нет.
- **Готовые целевые переходы существуют.** `openServiceCard(id)` (`docs/modules/services.js:353`, по числовому `id` из `SERVICES_DATA`). Раздел «Записи» — это таб-экран `s-history` (`SCREENS`, `docs/modules/constants.js:4`), уже открываемый через `go('s-history','tab')` (`docs/modules/notifications.js:13`). Категории каталога ключуются **по имени** (строка): `filterCategory(name)` (`services.js:61`) фильтрует `SERVICES_DATA` по `s.cat`; числового id у категории **нет**.
- **Текущая версия:** `docs/version.json` → build 47, v1.27.0; `CACHE_VERSION='v83'` (`sw.js:1`).

Силы, действующие на решение:

- Цель уведомления приходит из двух разных мест и должна давать **одинаковый** переход: (1) клик по телу карточки в инбоксе (цель берётся из IDB-записи); (2) тап по системному уведомлению ОС — `notificationclick` (цель берётся из `event.notification.data`), в т. ч. при **закрытом** приложении (окна может не быть).
- Доставка push (ADR-006) и формат записи (ADR-015) не должны ломаться для старых клиентов и старых записей: новое поле — **необязательное**, его отсутствие = «без цели».
- Контракт IDB продублирован в `inboxStore.js` и `sw.js` (`_IDB_*`) — изменения схемы синхронизируются в обоих местах.
- При изменении `APP_SHELL`/`sw.js` поднимается `CACHE_VERSION`. Worker деплоится отдельно.
- Внешний ввод (payload, IDB-запись) не доверенный: цель/`id` валидируются, строки экранируются `esc()`.

Решение охватывает развилки: **(A)** формат поля цели; **(B)** механизм навигации `notificationclick` при закрытом приложении и единый маршрутизатор; **(C)** объём цели «категория»; **(D)** перенос логики «прочитано» (точечная отметка) и поведение `enterInbox`/кнопок шапки; **(E)** bump `_IDB_VERSION`; **(F)** цель в напоминаниях (cron) и порядок выката.

## Рассматриваемые варианты

### Вопрос A. Формат поля цели в payload и IDB-записи

#### Вариант A1 (выбран): объект `target: {type, id?}`, типы из закрытого множества; отсутствие/`null`/неизвестный тип = «без цели»

Поле `target` — объект `{type, id?}`:
- `{type:'service', id:<number>}` — карточка услуги;
- `{type:'records'}` — раздел «Записи» (`s-history`), без `id`;
- `{type:'category', id:<string>}` — категория каталога (`id` = имя категории, см. Вопрос C);
- отсутствие поля, `null`, пустой объект, неизвестный `type` или невалидный `id` → трактуется как «без цели» (ФТ-5, ФТ-17, НФТ-2).

`target` кладётся: в payload push (рассылки и напоминания), в `event.notification.data.target` (через `showNotification(..., {data})`), в IDB-запись (новое поле `target`). Маршрутизатор принимает один и тот же объект из обеих веток.

- **Плюсы:** структурированно и расширяемо (новый тип = новая ветка в маршрутизаторе); закрытое множество типов легко валидировать; не привязывает клиент к строковому URL-формату; `records` без `id` естественно ложится в объект.
- **Минусы:** чуть многословнее строкового маршрута; требует валидатора. Приемлемо.
- **Последствия:** валидатор/нормализатор цели и маршрутизатор — в одном новом модуле (`docs/modules/inboxTarget.js`); поле `target` добавляется в запись в `inboxStore.js` и `_IDB_*` в `sw.js`.

#### Вариант A2 (отвергнут): строковый `url`/маршрут (`data.url='./?n=service:123'`)

- **Минусы:** парсинг строкового маршрута на клиенте и в SW дублирует логику; легче ошибиться с экранированием/валидацией; «без цели» и «records» выражаются неуклюже; сложнее расширять типобезопасно. Объект (A1) чище и валидируется проще.
- **Последствия:** хрупкий парсинг в двух местах. Отвергнут.

### Вопрос B. Механизм навигации `notificationclick` (в т. ч. при закрытом приложении) и единый маршрутизатор

#### Вариант B1 (выбран): единый клиентский маршрутизатор `routeToTarget(target)`; `notificationclick` доставляет цель через `postMessage` в открытое окно, а при отсутствии окна — через query-параметр в `openWindow('./?n=<encoded>')`, разбираемый при старте приложения

Вводится **один** клиентский хелпер `routeToTarget(target)` (модуль `docs/modules/inboxTarget.js`), который по нормализованной цели выполняет переход (`go('s-history','tab')` для `records`, `openServiceCard(id)` для `service`, `filterCategory(name)+go('s-services','tab')` для `category`, ничего — для «без цели»). Этот хелпер вызывают:
- **клик в инбоксе** — напрямую `routeToTarget(item.target)`;
- **`notificationclick`** в `sw.js` — не навигирует сам (в SW нет DOM), а доставляет цель в приложение:
  - если есть открытое окно (`clients.matchAll`) — `focus()` + `postMessage({type:'NOTIFICATION_TARGET', target})`; приложение слушает сообщение и вызывает `routeToTarget`;
  - если окна нет — `clients.openWindow('./?n=' + encodeURIComponent(JSON.stringify(target)))`; при старте приложение читает `?n=`, парсит/валидирует цель, вызывает `routeToTarget`, очищает параметр из URL (`history.replaceState`).

Так обе ветки (инбокс и системный клик) сходятся в **одном** `routeToTarget` (ФТ-15, КП-8).

- **Плюсы:** единая точка маршрутизации — одинаковый результат для одинаковой цели из любой ветки; работает и при открытом, и при закрытом приложении; не требует серверных правок; query-параметр самодостаточен (цель целиком в URL), не зависит от гонок `postMessage`.
- **Минусы:** две доставки в `notificationclick` (postMessage vs openWindow) — но обе ведут в один `routeToTarget`; нужен разбор `?n=` при старте (валидация обязательна — внешний ввод). Приемлемо.
- **Последствия:** `routeToTarget` + парсер `?n=` в `inboxTarget.js`; слушатель `NOTIFICATION_TARGET` в `app.js`; правка `notificationclick` в `sw.js`; вызов парсера `?n=` при инициализации приложения.

#### Вариант B2 (отвергнут): только `openWindow` с hash, без `postMessage` для открытого окна

- **Минусы:** при уже открытом приложении `openWindow`/повторный фокус не доставляет цель надёжно (часто только фокусирует существующее окно без перезагрузки и без передачи параметра); пришлось бы перезагружать окно ради навигации. `postMessage` для открытого окна — корректнее и без перезагрузки.
- **Последствия:** ненадёжный диплинк при открытом приложении. Отвергнут.

#### Вариант B3 (отвергнут): дублировать навигацию отдельно в клике инбокса и отдельно в обработке системного клика

- **Минусы:** прямое нарушение ФТ-15 (единый механизм) — два кодовых пути разойдутся по поведению. Отвергнут.

### Вопрос C. Объём цели «категория»

#### Вариант C1 (выбран): включить «категорию» минимально, используя имя категории как `id`

Готовый переход «открыть категорию» **есть**, но ключ категории — **имя** (строка), не число: `filterCategory(name)` (`services.js:61`) + переход на каталог `s-services`. Поэтому `{type:'category', id:<имя категории>}` маршрутизируется как `filterCategory(id); go('s-services','tab')`. Имя сверяется со списком существующих категорий (`[...new Set(SERVICES_DATA.map(s=>s.cat))]`); если имени нет — деградация в «без цели» (ФТ-16/ФТ-5). В композере опция «категория» предлагает выбор из существующих категорий по имени.

- **Плюсы:** переиспользует существующий `filterCategory`; объём минимальный (нет нового экрана/хелпера навигации по категории); закрывает ФТ-16 без раздувания; промо «на категорию» работает.
- **Минусы:** `id` категории — имя-строка, а не число (несимметрично с `service`); при переименовании категории старые уведомления деградируют в «без цели» — приемлемо (валидация по списку, без ошибок).
- **Последствия:** ветка `category` в `routeToTarget`; в композере — селектор категорий по имени.

#### Вариант C2 (отвергнут): исключить «категорию» из релиза

- **Минусы:** теряет ФТ-16, хотя готовый переход по сути существует (`filterCategory`). Исключение оправдано, только если хелпера нет; здесь он есть, а стоимость ветки мала. Отвергнут в пользу C1.

> Примечание: если на этапе реализации выяснится, что увязка имени категории с композером требует непропорциональных усилий, допустима деградация к C2 (опция «категория» скрыта/трактуется как «без цели»). База ADR — C1.

### Вопрос D. Логика «прочитано», `enterInbox` и кнопки шапки

#### Вариант D1 (выбран): точечная отметка при клике; вход в инбокс **не** помечает всё; «корзина» = удалить всё; текстовая кнопка «Прочитать всё» = `markAllRead` без удаления

- Клик по телу карточки: `setRead(item.id, true)` именно для этой записи, затем переход по цели (если есть), затем пересчёт бейджа (ФТ-1, ФТ-6).
- `enterInbox()` больше **не** вызывает `markAllRead()` — только `renderInbox()` + `updateInboxBadge()` (ФТ-7). Это вытеснение поведения ADR-015.
- Шапка `s-inbox`: иконка метёлки заменяется на **корзину**; действие — прежний `clearInboxHistory()` (`clearAll` + `confirm`) (ФТ-9).
- Над списком — текстовая кнопка **«Прочитать всё»**: вызывает существующий `markAllRead()` (пометить все прочитанными **без** удаления) + перерисовка + бейдж (ФТ-8). Видимость: показывать всегда, но **disabled/скрыта при отсутствии непрочитанных и при пустом списке** (ОВ-7).
- Существующее меню «три точки» (сделать непрочитанным / удалить) и `PUSH_RECEIVED` сохраняются; чтобы клик по телу не конфликтовал с меню — обработчик тела и кнопка «три точки» разведены (`event.stopPropagation` в `inboxToggleMenu` уже есть).

- **Плюсы:** «прочитано = открыто» — честный статус (бизнес-цель); два намерения (убрать бейдж vs очистить) разведены в два явных элемента; переиспользует уже существующие `setRead`/`markAllRead`/`clearAll` (минимум нового кода).
- **Минусы:** ревизия UX относительно ADR-015 — осознанная, зафиксирована ниже.
- **Последствия:** правки `inbox.js` (`enterInbox`, рендер карточки + обработчик клика, новый обработчик «Прочитать всё»); `prototype.html` (иконка корзины + кнопка «Прочитать всё»); `style.css`.

#### Вариант D2 (отвергнут): сохранить `markAllRead` при входе, добавить лишь клики

- **Минусы:** вход в инбокс продолжил бы обнулять бейдж, делая «прочитано» неинформативным — прямо против ФТ-7 и бизнес-цели. Отвергнут.

### Вопрос E. Bump `_IDB_VERSION`

#### Вариант E1 (выбран): **не** поднимать `_IDB_VERSION` (остаётся `1`)

Store schemaless (`keyPath:'id'`, без индексов по `target`). Добавление необязательного поля `target` в запись **не требует** `onupgradeneeded`/миграции: старые записи без `target` валидны и читаются как «без цели». Версия БД остаётся `1` синхронно в `inboxStore.js` и `_IDB_*` в `sw.js`; обновляется лишь комментарий-контракт и форма записи в `addNotification`/`_idbAddNotification`.

- **Плюсы:** нет риска `versionchange`-блокировок и миграций; обратная совместимость данных бесплатна (НФТ-2).
- **Минусы:** нет (индексы по `target` не нужны — выборка по id).
- **Последствия:** синхронно обновить комментарий схемы и форму записи в обоих файлах; `DB_VERSION`/`_IDB_VERSION` = `1`.

#### Вариант E2 (отвергнут): поднять `_IDB_VERSION` до 2 с `onupgradeneeded`

- **Минусы:** избыточно для добавления необязательного поля в schemaless store; вводит риск миграции и рассинхрона версий между двумя файлами без выгоды. Отвергнут.

### Вопрос F. Цель напоминаний (cron) и порядок выката Worker

#### Вариант F1 (выбран): `processReminder` кладёт `target:{type:'records'}` в payload; Worker деплоится отдельно; порядок выката безопасен в любую сторону

`cf-worker/src/index.js` (`processReminder`) добавляет в payload напоминания `target:{type:'records'}` (без `record_id` — детальной карточки визита нет; задел на будущее не вводим, чтобы не раздувать объём — ОВ-4). `handleSend` принимает `target` из тела запроса и кладёт в payload как есть (валидирует наличие/тип на клиенте; Worker — транспорт). Worker деплоится `cd cf-worker && npx wrangler deploy` **отдельным шагом**.

Совместимость выката (НФТ-2/НФТ-6):
- **Старый клиент + новый Worker:** старый SW игнорирует незнакомое поле `target` в payload (читает только `title/body/icon`), `notificationclick` ведёт в корень — без ошибок.
- **Новый клиент + старый Worker:** напоминания приходят без `target` → «без цели», клик помечает прочитанным — без ошибок.
Порядок выката не критичен; рекомендуется сначала PWA, затем Worker (чтобы новые клиенты раньше понимали `target`).

- **Плюсы:** напоминания становятся диплинком в «Записи»; обратная совместимость в обе стороны; без пересмотра доставки/подписок ADR-006.
- **Минусы:** Worker — отдельный деплой (организационный шаг, отражён в спецификации).
- **Последствия:** правка `processReminder`/`handleSend` в `index.js`; отдельный шаг деплоя.

## Решение

- **A — A1:** цель = объект `target:{type,id?}`, `type ∈ {service, records, category}`; отсутствие/`null`/неизвестный тип/невалидный `id` = «без цели». Единый валидатор/нормализатор.
- **B — B1:** единый клиентский маршрутизатор `routeToTarget(target)` (`docs/modules/inboxTarget.js`), общий для клика в инбоксе и для `notificationclick`. `notificationclick`: открытое окно → `focus()`+`postMessage({type:'NOTIFICATION_TARGET',target})`; нет окна → `openWindow('./?n='+encodeURIComponent(JSON.stringify(target)))`, разбор `?n=` при старте с валидацией и очисткой URL.
- **C — C1:** «категория» **входит** в релиз: `{type:'category', id:<имя>}` → `filterCategory(имя)`+`go('s-services','tab')`, имя сверяется со списком категорий; нет имени → «без цели». (Допустимая деградация к C2, если увязка с композером окажется непропорциональной.)
- **D — D1:** клик помечает прочитанным **именно эту** запись и переходит по цели; `enterInbox` **не** помечает всё; метёлка → корзина (`clearInboxHistory`); текстовая кнопка «Прочитать всё» = `markAllRead` без удаления, скрыта/disabled при отсутствии непрочитанных.
- **E — E1:** `_IDB_VERSION` остаётся `1`; поле `target` добавляется без миграции; контракт синхронизируется в `inboxStore.js` и `sw.js`.
- **F — F1:** напоминания несут `target:{type:'records'}`; `handleSend` пробрасывает `target` из тела; Worker деплоится отдельно; выкат совместим в обе стороны.
- **`CACHE_VERSION`:** правятся `sw.js` + файлы `APP_SHELL` (`prototype.html`, `style.css`, JS-модули, новый `inboxTarget.js`) → поднять `v83 → v84`; `inboxTarget.js` добавить в `APP_SHELL`.
- **Безопасность:** `target`/`id` из payload и IDB валидируются перед использованием; тексты рендерятся через `esc()`; невалидная цель = «без цели» без ошибок в консоли (ФТ-17, НФТ-4).
- **Целостность потока push:** показ системного уведомления, подписка (`push.js`), бейдж, тост, дедуп, лимит, меню «три точки», `PUSH_RECEIVED` — не переписываются; запись `target` — дополнительный шаг, его сбой не ломает показ (НФТ-3).

Отвергнуты: A2 (строковый URL — хрупкий парсинг в двух местах), B2 (только openWindow — ненадёжно при открытом окне), B3 (дублировать навигацию — против ФТ-15), C2 (исключить категорию — хелпер существует), D2 (сохранить markAllRead при входе — против ФТ-7), E2 (bump версии — избыточно).

### Связь с ADR-015 / ADR-026 (суперседы и расширения)

- **ADR-015 — частичная ревизия.** Решение «при входе в `s-inbox` все непрочитанные помечаются прочитанными» (ADR-015, Вопрос D/Решение) **вытесняется**: вход больше не помечает всё; отметка точечная (при клике) + явная кнопка «Прочитать всё». Остальные решения ADR-015 (IndexedDB как хранилище, дедуп по `id`, лимит 50, отдельный экран `s-inbox`, бейдж из IndexedDB + `postMessage`) **остаются в силе**. Схема записи ADR-015 `{id,title,body,icon?,ts,read}` **расширяется** необязательным полем `target` (без bump версии).
- **ADR-026 — расширение.** «Честный» композер (текст + иконка) дополняется выбором цели (услуга / категория / без цели, дефолт «без цели»); цель пробрасывается в payload (`sendAdminPush` → `/send`). Решения ADR-026 об отсутствии декоративной аудитории/расписания не затрагиваются.
- **ADR-006 — не пересматривается.** Доставка, подписки, криптография неизменны; `target` — необязательное поле payload, игнорируемое старыми клиентами.

ADR-015 и ADR-026 остаются в статусе «принято» (точечные ревизии/расширения, не полная замена); формальный `Superseded` не выставляется.

## Последствия

- **Положительные:**
  - Уведомление становится точкой входа к сущности: напоминание → «Записи», промо об услуге → карточка услуги, промо на категорию → каталог с фильтром.
  - Единый `routeToTarget` гарантирует одинаковый переход из инбокса и из системного клика, в т. ч. при закрытом приложении.
  - «Прочитано = открыто» — честный статус; управление историей разведено на «удалить всё» (корзина) и «прочитать всё» (кнопка).
  - Обратная совместимость данных и доставки: старые записи и старые клиенты не падают; `target` необязателен.
  - Минимум нового кода: переиспользуются `openServiceCard`, `filterCategory`, `go('s-history','tab')`, `setRead`, `markAllRead`, `clearAll`.
- **Отрицательные / компромиссы:**
  - Цель «категория» ключуется по имени-строке (несимметрично с числовым `id` услуги); переименование категории деградирует старые уведомления в «без цели».
  - `notificationclick` усложняется (две доставки цели — postMessage/openWindow), оба пути сходятся в `routeToTarget`.
  - Разбор `?n=` при старте — внешний ввод, обязательная валидация и очистка URL.
- **Технический долг:**
  - Детальная карточка визита и `record_id` в цели напоминания — отложены (целью визита остаётся раздел «Записи»).
  - Числовой id категории — не вводится; при будущем рефакторинге каталога цель `category` можно перевести на id.
  - Захардкоженный YCLIENTS-токен — известный долг, не усугубляется.
