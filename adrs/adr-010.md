# ADR-010: Отдельный экран одной публикации ленты (`s-post`) с переиспользованием рендера карточки и мягким фолбэком на `s-feed`

- **Дата:** 2026-05-29
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-011.md` (открытые вопросы №1–№5 закрыты решениями заказчика от 2026-05-29)
- **Связанные ADR:** ADR-009 (отдельный экран `s-upcoming` с переиспользованием рендера предстоящих — задаёт паттерн «список → отдельный экран одного элемента, зарегистрированный в `SCREENS`», прямо взятый за образец; не пересматривается). ADR-008 (контекстный тап по мастеру — родственный паттерн «тап по конкретному элементу → его представление»; не затрагивается). Прямых пересечений с ADR-001…ADR-007 не выявлено.

## Контекст

На главном экране блок «Лента студии» (`section-header` + `#homeFeedPreview`, `docs/prototype.html:70-76`) рендерит до 6 превью последних публикаций (`_renderHomeFeedPreview`, `docs/modules/profile.js:132-160`). Каждая плашка превью имеет `onclick="go('s-feed')"` (`profile.js:154`) — **тот же самый переход**, что и у ссылки «Все →» рядом с заголовком секции (`prototype.html:72`). То есть тап по конкретной плашке открывает **общий список всех публикаций** (`s-feed` → `_paintFeed`, `feed.js:53-80`), а не выбранную публикацию. Это расходится с пользовательским ожиданием «тап по конкретной плашке → именно эта публикация» и с уже сложившимся в приложении паттерном (ADR-008 для мастеров, ADR-009 для визитов).

Источник публикаций уже существует: `yc_feed_posts` в `localStorage`, обновляемый из GitHub через `_ghReadFn` (`feed.js:14,38`); видимыми считаются только не-черновики (`!p.draft`), как в текущем коде превью и списка. У каждой публикации есть устойчивый идентификатор `p.id` = `Date.now()` (`admin.js:111`), доступный и в превью, и в списке — единственный стабильный адресный признак (надёжнее позиции в массиве, который меняется при публикации/удалении и обрезается до `slice(0,6)` в превью).

**Бизнес-цель (BA, раздел 1):** тап по конкретной плашке превью ленты на главной открывает **именно выбранную** публикацию; ссылка «Все →» сохраняет текущее поведение (полный список `s-feed`). Это устраняет вводящее в заблуждение поведение и приводит превью к паттерну «конкретный элемент → его представление».

**Решения заказчика по открытым вопросам BA приняты 2026-05-29** и являются обязательными входными ограничениями (не пересматриваются):

1. **Форма представления — отдельный экран `s-post`** (не модалка, не прокрутка с подсветкой в `s-feed`), зарегистрированный в `SCREENS` (`constants.js`) — по аналогии с `s-upcoming` (ADR-009).
2. **Бара фильтра категорий в `s-post` нет** (следствие выбора отдельного экрана).
3. **«Назад» из `s-post` возвращает на главную** (`s-home`).
4. **Состав `s-post` — как в карточке списка** (`_paintFeed`, `feed.js:53-80`): фото/иконка категории, полный текст, категория, дата, кнопка «Записаться →» с тем же поведением (`_feedGoBook`, `feed.js:106-109`). Переиспользовать существующий рендер карточки, не вводя новых полей/запросов.
5. **При отсутствующем/удалённом посте по `id`** (нет среди `!p.draft`) — **открыть общий список ленты `s-feed`** (мягкий фолбэк, без ошибки в консоли).

Силы, действующие на решение:

- **Единый источник данных и логики отбора (НФТ-2, НФТ-3).** Представление одной публикации должно строиться из тех же данных (`yc_feed_posts` + обновление через `_ghReadFn`) и той же фильтрации `!p.draft`, что превью и список. Любая параллельная логика отбора — источник рассинхрона между тремя представлениями.
- **Запрет дублирования разметки (BA решение №4).** Разметка карточки публикации уже есть в `_paintFeed` (`feed.js:58-78`): фото/иконка-градиент, тег категории, кнопка «Записаться →», текст, дата. Копировать её в новый экран — техдолг и риск расхождения. Требуется переиспользование рендера одной карточки.
- **Регистрация экрана (ограничение из ADR-009).** Любой экран обязан быть в `SCREENS` (`constants.js:3-9`) — иначе анимация навигации (`navigation.js`) и сброс сценариев (`scenarios.js`) его не учтут. onEnter регистрируется в `app.js` через `registerOnEnter`.
- **Передача `id` в onEnter.** `registerOnEnter(screenId, fn)` вызывает `fn()` без аргументов (`navigation.js:7-8,42-45`). Значит идентификатор открываемой публикации нельзя передать как параметр `go(...)` — его нужно положить в общее состояние (`state`) перед переходом и прочитать в onEnter `s-post`. Это согласуется с тем, как `s-slots`/`s-confirm`/`s-master` уже читают контекст из `state`/глобалей при входе.
- **Навигация «Назад» бесплатна при push-переходе.** `go('s-post')` из `s-home` — push-режим (`navigation.js:77-93`): `s-post` кладётся в `navHistory`. Кнопка «Назад» (`back()`, `navigation.js:98-125`) автоматически вернёт на предыдущий экран (`s-home`). Отдельная логика возврата не нужна — достаточно стандартной `back-btn` `onclick="back()"`, как на прочих push-экранах. Это удовлетворяет решению заказчика №3 без специального кода. **Важно (учтено в Вопросе C):** в push-ветке `go` (`navigation.js:84-92`) `navHistory.push(id)` выполняется **синхронно**, а `onEnterScreen` — **отложенно** (`setTimeout(DUR+20)`); значит к моменту срабатывания onEnter `s-post` уже на вершине истории. Поэтому фолбэк, вызываемый из onEnter, не должен делать обычный `go('s-feed')` (это был бы push поверх `s-post` → петля «Назад»), а должен **замещать** текущий экран — см. Вопрос C.
- **Кнопка «Записаться →» уводит с экрана.** `_feedGoBook` (`feed.js:106-109`) делает `go('s-services')`. То есть после записи пользователь покидает `s-post` — синхронная перерисовка не нужна; актуализация (если потребуется) произойдёт при повторном входе.
- **Vanilla JS, без рантайм-зависимостей (НФТ-1); минимальность (НФТ-3).** Новый экран — это правка существующих файлов `APP_SHELL` (`prototype.html`, `constants.js`, `feed.js`, `profile.js`, `app.js`), без добавления новых файлов в `APP_SHELL`. При изменении содержимого кэшируемых файлов — поднятие `CACHE_VERSION` в `sw.js` (правило проекта, НФТ-4).
- **Не ломать `s-feed`, бар фильтра, «Записаться →», админку ленты (НФТ-3, КП-8).** Список `s-feed` (`_paintFeed`, `_renderFeedCats`/`_feedFilterCat`), кнопка записи (`_feedGoBook`) и админка (`admin.js`) должны работать как прежде.

Решение охватывает четыре связанных вопроса: (1) форма представления — определена решением заказчика (отдельный экран `s-post`), архитектор фиксирует механику; (2) как переиспользовать рендер карточки и где разместить функцию рендера одной публикации; (3) как доставить `id` в onEnter экрана без параметра; (4) как реализовать мягкий фолбэк при отсутствующей публикации. Конкретные имена функций сверх ключевых, тексты и порядок правок — в спецификации релиза, не здесь.

## Рассматриваемые варианты

> Вопрос «отдельный экран vs модалка vs прокрутка с подсветкой в `s-feed`», а также «нужен ли бар фильтра», «куда ведёт Назад», «какой состав» — **закрыты решениями заказчика** (BA раздел 8, блок «РЕШЕНО заказчиком 2026-05-29»). Архитектор не пересматривает их; ниже рассматриваются только инженерные развилки в рамках принятой формы.

### Вопрос A. Где разместить рендер одной публикации и как переиспользовать карточку списка

Цель — единый источник разметки карточки публикации, используемый и в `s-feed` (`_paintFeed`), и в `s-post`, без дублирования (BA решение №4, НФТ-2).

#### Вариант A1 (выбран): вынести рендер одной карточки в хелпер `feed.js` и переиспользовать в `_paintFeed` и в рендере `s-post`

Извлечь разметку одной карточки из `_paintFeed` (`feed.js:58-78`) в экспортируемый хелпер модуля `feed.js`, например `_feedCardHtml(p)` (HTML одной публикации: фото/иконка-градиент, тег категории, кнопка «Записаться →» с тем же `onclick="_feedGoBook(this.dataset.cat)"`, текст, дата). `_paintFeed` собирает список из `_feedCardHtml(p)` (специфика последнего элемента — `margin-bottom` — остаётся параметром/обёрткой в `_paintFeed`). Новая функция рендера экрана (например `renderPostScreen()`) читает `id` из `state`, ищет публикацию среди `!p.draft` и рисует её через тот же `_feedCardHtml(p)` в контейнер `s-post`.

- **Плюсы:** хелпер живёт рядом с данными ленты и существующей разметкой карточки; переиспользует уже импортированные `esc`, `_CAT_GRADS`, `_CAT_ICONS2`; единый источник разметки карточки (нет расхождения между `s-feed` и `s-post` при будущих правках). Чистая строковая сигнатура согласуется с тем, как весь проект строит UI (innerHTML из шаблонных строк). Прямой аналог решения ADR-009 (Вариант 2A: хелпер рендера в доменном модуле).
- **Минусы:** `feed.js` получает дополнительную экспортируемую поверхность; минимальный рефактор `_paintFeed` (с сохранением внешнего поведения, включая `margin-bottom` последнего элемента).
- **Последствия:** правка `feed.js` (извлечение хелпера + новая функция рендера экрана), без смены поведения `s-feed`.

#### Вариант A2 (отвергнут): новая функция рендера `s-post` дублирует разметку карточки

Скопировать разметку карточки из `_paintFeed` в отдельную функцию рендера `s-post`, не трогая `_paintFeed`.

- **Минусы:** дублирование разметки карточки в двух местах → техдолг и риск расхождения при будущих правках вёрстки/полей (нарушает BA решение №4 «переиспользуй существующий рендер» и НФТ-3). Любая правка карточки потребует синхронных изменений в двух функциях.
- **Последствия:** меньше касаний `_paintFeed`, но устойчивый источник рассинхрона. Отвергнут.

#### Вариант A3 (отвергнут): рендер одной публикации в `profile.js` или `utils.js`

Положить рендер карточки в `profile.js` (где живёт превью) или в `utils.js` как «общий компонент».

- **Минусы:** карточка ссылается на доменные данные ленты (`_CAT_GRADS`, `_CAT_ICONS2`) и доменное действие `_feedGoBook` — её место в `feed.js`, а не в `profile.js` (превью — это другой формат плашки) и тем более не в `utils.js` (низкоуровневые утилиты без доменной логики; внесение туда доменной разметки нарушает уровень абстракции и направление зависимостей — тот же аргумент, что в ADR-009, Вариант 2B). 
- **Последствия:** размытие ответственности модулей, нежелательная связанность. Отвергнут.

### Вопрос B. Как доставить `id` выбранной публикации в onEnter `s-post`

`registerOnEnter` вызывает обработчик без аргументов (`navigation.js:7-8,42-45`), поэтому `go('s-post')` не может передать `id` как параметр.

#### Вариант B1 (выбран): записать `id` в общее состояние перед переходом, прочитать в onEnter

Перед `go('s-post')` положить выбранный `id` в `state` (например `state._openPostId`); onEnter `s-post` (`renderPostScreen`) читает его, ищет публикацию и рисует. Плашка превью передаёт `id` через `data`-атрибут и тонкий глобальный диспетчер (например `_openFeedPost(id)`), который и пишет `state`, и вызывает `go(...)` либо фолбэк — по аналогии с тем, как кнопки в проекте используют `data-*` + `onclick` (ср. `_feedGoBook`, `rescheduleRecord`).

- **Плюсы:** не требует менять сигнатуру `go`/`registerOnEnter` (минимальность, не затрагивает навигацию глобально); согласуется с тем, как экраны уже читают контекст из `state`/глобалей при входе (`s-slots`, `s-confirm`, `s-master`). Диспетчер инкапсулирует и фолбэк (Вопрос C).
- **Минусы:** контекст хранится в общем `state`, а не в URL/параметре — приемлемо для данного приложения (нет роутинга по URL, экраны и так контекст-зависимы).
- **Последствия:** добавляется поле в `state` и тонкий диспетчер; `go`/`registerOnEnter` не меняются.

#### Вариант B2 (отвергнут): расширить `go(id, mode, payload)` и пробросить payload в onEnter

Изменить сигнатуру `go` и `registerOnEnter`, чтобы передавать произвольные данные в обработчик входа.

- **Минусы:** меняет общий контракт навигации ради одного экрана — ломающее по духу изменение базового модуля, затрагивающее все вызовы `go`/`onEnter` (риск регрессии всех экранов, противоречит минимальности НФТ-3). Несоразмерно задаче.
- **Последствия:** широкая площадь изменений и риск. Отвергнут.

### Вопрос C. Мягкий фолбэк при отсутствующей/удалённой публикации (BA решение №5)

> **Правка от 2026-05-29 (по итогам код-ревью v1.9.0).** Первоначальная редакция C1 предписывала и диспетчеру, и страховочному onEnter `renderPostScreen` вызывать `go('s-feed')`. Код-ревью выявило, что для onEnter этот вариант некорректен из-за механики `navigation.js`: в push-ветке `go` (строки 92, 85-91) `navHistory.push(id)` выполняется синхронно, а `onEnterScreen` — отложенно (`setTimeout(DUR+20)`). Поэтому к моменту срабатывания onEnter `s-post` уже на вершине `navHistory`, и `go('s-feed')` из onEnter трактуется как **push** (`current='s-post' !== 's-feed'`), давая историю `[…, 's-home', 's-post', 's-feed']`. Тогда «Назад» из `s-feed` возвращает на `s-post`, его onEnter снова не находит пост → снова `go('s-feed')` → **петля «Назад»**, возврат на `s-home` недостижим (ошибок в консоли нет — КП-7 формально выполнен, но навигация деградирует). Диспетчерный путь `_openFeedPost` этим не страдает: там фолбэк срабатывает из `s-home` (пост не найден **до** навигации), `go('s-feed')` — корректный push поверх `s-home`. Ниже C1 переформулирован: диспетчерный фолбэк оставлен как есть, страховочный onEnter-фолбэк переведён на **замещение** текущего экрана в истории, чтобы сохранить единый пользовательский исход «оказаться на `s-feed`» (BA №5) и не разрушить «Назад».

#### Вариант C1 (выбран, в редакции 2026-05-29): диспетчерный фолбэк `go('s-feed')` + страховочный onEnter-фолбэк через замещение экрана в истории

Диспетчер `_openFeedPost(id)` ищет публикацию в `yc_feed_posts` среди `!p.draft`. Если найдена — пишет `state._openPostId` и `go('s-post')`. Если **не найдена** (пост недоступен до навигации, путь из `s-home`) — `go('s-feed')` (push поверх `s-home`, корректный фолбэк на общий список), без ошибки в консоли.

Дополнительно onEnter `s-post` (`renderPostScreen`) страхует **гонку** «пост исчез/стал черновиком между тапом и onEnter». На момент входа `s-post` уже на вершине `navHistory` (см. правку выше), поэтому страховочный фолбэк не должен **добавлять** `s-feed` поверх `s-post`, а должен **заместить** `s-post` на `s-feed`. Для этого в модуль навигации добавляется аддитивный режим `go(id, 'replace')`: он заменяет верхний элемент `navHistory` (`navHistory[len-1] = id`) вместо `push`, анимация — как у обычного push. `renderPostScreen` при отсутствии публикации вызывает `go('s-feed', 'replace')`: история становится `[…, 's-home', 's-feed']`, пользовательский исход — тот же «открыть `s-feed`» (BA №5), а «Назад» из `s-feed` корректно ведёт на `s-home`, петли нет.

- **Плюсы:** удовлетворяет КП-7 («без ошибки и без пустого экрана») и BA №5 (исход — `s-feed`) **в обоих путях** — единый пользовательский результат независимо от того, где обнаружено отсутствие; устраняет петлю «Назад»; логика истории остаётся **внутри модуля навигации** (правильное направление зависимостей — `feed.js` не мутирует `navHistory` напрямую); режим `replace` — обобщаемая навигационная примитивная операция, переиспользуемая будущими фолбэками.
- **Минусы:** расширяется контракт `go` третьим режимом (`'replace'`) — аддитивно, существующие вызовы `go(id)`/`go(id,'tab')` не затрагиваются; двойная проверка фолбэка (диспетчер + onEnter) сохраняется как защита от гонки.
- **Последствия:** правка локализована в `navigation.js` (новый режим) и в страховочном фолбэке `renderPostScreen`; диспетчерный фолбэк и общий список `s-feed` не меняются.

> Почему не `back()` в onEnter (рассмотрено и отвергнуто): `back()` из `s-post` вернул бы на `s-home`, что технически убирает петлю, но меняет **пользовательский исход** страховочного фолбэка с «открыть `s-feed`» на «вернуться на главную». Это расходится с BA №5 (исход — `s-feed`) и делает поведение фолбэка **непоследовательным** между путями: диспетчер → `s-feed`, onEnter → `s-home`. Замещение через `replace` сохраняет единый исход.
> Почему не прямая мутация `navHistory` из `feed.js`: вынудила бы `feed.js` знать внутреннее устройство истории навигации и опираться на деталь «push сделан синхронно» — нарушение границ модулей и хрупкая связь с реализацией `go`. Логика истории должна жить в `navigation.js`.

#### Вариант C2 (отвергнут): показать в `s-post` сообщение «Публикация недоступна»

Открыть `s-post` и при отсутствии публикации показать заглушку с текстом.

- **Минусы:** прямо противоречит решению заказчика №5 (выбран фолбэк на `s-feed`, не сообщение); добавляет новый пользовательский текст и пустое состояние, которых заказчик не просил. Отвергнут.

## Решение

- **Форма (BA решение №1):** отдельный экран `s-post` «Публикация», зарегистрированный в `SCREENS` (`constants.js`) и через `registerOnEnter` (`app.js`), показывающий **одну** выбранную публикацию. Без бара фильтра категорий (BA решение №2). «Назад» — стандартный `back()` → возврат на `s-home` за счёт push-перехода (BA решение №3).
- **Вопрос A — Вариант A1:** рендер одной карточки публикации выносится в экспортируемый хелпер `feed.js` (например `_feedCardHtml(p)`); `_paintFeed` переводится на него без изменения внешнего поведения списка; рендер `s-post` использует тот же хелпер. Дублирования разметки нет. Состав карточки — как в списке (BA решение №4): фото/иконка категории, тег категории, текст, дата, кнопка «Записаться →» с прежним `_feedGoBook`.
- **Вопрос B — Вариант B1:** выбранный `id` записывается в `state` перед `go('s-post')` тонким диспетчером (например `_openFeedPost(id)`); onEnter `s-post` (`renderPostScreen`) читает его из `state`. Сигнатуры `go`/`registerOnEnter` не меняются.
- **Вопрос C — Вариант C1 (в редакции 2026-05-29):** при отсутствии публикации по `id` среди `!p.draft` — мягкий фолбэк на `s-feed` (BA решение №5), без ошибки в консоли. Диспетчерный путь `_openFeedPost` (отсутствие обнаружено до навигации, из `s-home`) — `go('s-feed')` (push). Страховочный onEnter-путь `renderPostScreen` (гонка: пост исчез между тапом и onEnter, `s-post` уже в истории) — `go('s-feed', 'replace')`, который **замещает** `s-post` на `s-feed` в `navHistory`, сохраняя исход «открыть `s-feed`» и не зацикливая «Назад». Для этого в `navigation.js` вводится аддитивный режим `replace` (см. правку Вопроса C по итогам код-ревью).

Отвергнуты: A2 (дублирование разметки карточки — техдолг и рассинхрон), A3 (размытие ответственности модулей), B2 (ломающее изменение контракта навигации ради одного экрана), C2 (противоречит решению заказчика №5).

### Границы решения

- Экран `s-post` показывает **одну** публикацию, найденную по `state._openPostId` среди `!p.draft` в текущем `yc_feed_posts` — **той же фильтрацией**, что превью (`profile.js:136`) и список (`feed.js:23,40`). Параллельная/иная логика отбора не вводится.
- Состав `s-post` — **тот же**, что карточка списка (`_paintFeed`): фото/иконка-градиент категории, тег категории, полный текст, дата, кнопка «Записаться →». Кнопка «Записаться →» переиспользует `_feedGoBook` без изменения поведения (`go('s-services')` с предварительной фильтрацией категории).
- Бара фильтра категорий (`feedCatBar`, `_renderFeedCats`/`_feedFilterCat`) в `s-post` **нет** (BA решение №2).
- «Назад» из `s-post` ведёт на `s-home` за счёт стандартного push/`back()`; **специальная логика возврата не вводится**. Страховочный onEnter-фолбэк использует `go('s-feed', 'replace')`, что также сохраняет корректный возврат «Назад» на `s-home` (история `[…, 's-home', 's-feed']`).
- Ссылка «Все →» (`prototype.html:72`) **сохраняет** `go('s-feed')` — не меняется.
- При отсутствующей публикации по `id` — мягкий фолбэк на `s-feed`, без ошибки и без пустого экрана: диспетчер → `go('s-feed')` (из `s-home`), страховочный onEnter → `go('s-feed', 'replace')` (замещает `s-post` в истории).
- **Не затрагиваются:** общий список `s-feed` по сути (`_paintFeed` — только извлечение хелпера без смены поведения; `_renderFeedCats`/`_feedFilterCat` — без изменений); кнопка «Записаться →» (`_feedGoBook`); админка ленты (`admin.js`), формирование `id`, синхронизация с GitHub (`_ghReadFn`); состав полей публикации и формат `yc_feed_posts`; превью на главной по сути (количество плашек `slice(0,6)`, вёрстка плашки, условие показа секции) — меняется только целевое действие тапа по плашке; захардкоженный YCLIENTS-токен (не трогать).
- Источник данных `s-post` — кэш `yc_feed_posts`, наполняемый на входе в `s-home` (`app.js:58-63`) и `s-feed` (`renderClientFeed`, `feed.js:38-43`). `s-post` **не** инициирует дополнительный запрос к GitHub/YCLIENTS сверх существующих (НФТ-2): тап по плашке возможен только когда превью уже отрисовано, то есть кэш уже есть.

## Последствия

- **Положительные:**
  - Тап по конкретной плашке открывает именно выбранную публикацию (бизнес-цель BA); поведение приведено к паттерну ADR-008/ADR-009.
  - Единый источник разметки карточки (общий хелпер `_feedCardHtml`) — нет дублирования, нет расхождения между `s-feed` и `s-post` (BA решение №4, НФТ-3).
  - Единый источник данных и фильтрации (`yc_feed_posts` + `!p.draft`) — состав согласован с превью и списком (НФТ-2).
  - Адресация по устойчивому `p.id`, а не по индексу — при изменении состава/порядка списка открывается именно выбранный пост (КП-2).
  - «Назад» на `s-home` реализуется штатным push/`back()` без специального кода (BA решение №3).
  - Минимальная площадь: правки в существующих файлах `APP_SHELL`, без новых файлов и рантайм-зависимостей (НФТ-1, НФТ-3).
- **Отрицательные / компромиссы:**
  - `feed.js` получает дополнительную экспортируемую поверхность (хелпер + функция рендера экрана) — приемлемо, логически принадлежит модулю ленты.
  - Контекст открываемой публикации хранится в общем `state`, а не передаётся параметром перехода — осознанный компромисс ради сохранения контракта навигации (`go`/`registerOnEnter` не меняются).
  - Мягкий фолбэк проверяется в двух точках (диспетчер + onEnter) — небольшая избыточность ради защиты от гонки обновления данных.
- **Технический долг:**
  - Дальнейшая унификация форматов «плашка превью» (`profile.js`) и «карточка списка» (`feed.js`) в единый параметризуемый компонент **не делается** — это разные визуальные форматы; возврат к вопросу — при следующем существенном касании ленты.
  - Хранение навигационного контекста в `state` вместо параметра перехода — общий паттерн приложения; консолидация (payload в навигации) вне рамок релиза.

## Влияние на код

- **`docs/modules/constants.js`** — в массив `SCREENS` (стр. 3-9) добавляется `'s-post'`. Семантика прочих экранов не меняется.
- **`docs/modules/navigation.js`** — в `go(id, mode)` добавляется аддитивный режим `mode === 'replace'` (правка по итогам код-ревью, Вопрос C): анимация перехода — как у обычного push (slide-in нового экрана), но вместо `navHistory.push(id)` верхний элемент истории **замещается** (`navHistory[navHistory.length - 1] = id`). Существующие режимы (`push` по умолчанию, `'tab'`) и сигнатуры `go`/`back`/`registerOnEnter` **не меняются** — расширение строго аддитивно. Используется страховочным onEnter-фолбэком `renderPostScreen`, чтобы фолбэк на `s-feed` не добавлял третий элемент поверх `s-post` (иначе «Назад» зацикливается, см. Вопрос C). Прочая логика навигации (`back`, guard `navHistory[len-1] !== targetId` в отложенных onEnter, `_clearAnimTimers`) не затрагивается.
- **`docs/modules/feed.js`** — извлечение общего хелпера и новая логика:
  - новый экспортируемый `_feedCardHtml(p)` (имя ориентировочное) — HTML одной карточки публикации, идентичный текущей разметке `_paintFeed` (`feed.js:58-78`): фото/иконка-градиент по категории (`_CAT_GRADS`/`_CAT_ICONS2`), тег категории, кнопка «Записаться →» с тем же `data-cat`/`onclick="_feedGoBook(this.dataset.cat)"`, полный текст (`esc(p.text)`), дата;
  - `_paintFeed` переводится на сборку списка из `_feedCardHtml(p)` — **внешнее поведение списка `s-feed` не меняется** (та же разметка карточек, поведение `margin-bottom` последнего элемента сохраняется в обёртке `_paintFeed`); `_renderFeedCats`/`_feedFilterCat` — без изменений;
  - новая функция рендера экрана `s-post` (например `renderPostScreen`): читает `id` из `state` (`state._openPostId`), ищет публикацию среди `!p.draft` в `yc_feed_posts`; при наличии — рисует `_feedCardHtml(p)` в контейнер экрана; при отсутствии — `go('s-feed', 'replace')` (страховочный фолбэк через замещение `s-post` в истории — см. Вопрос C, редакция 2026-05-29);
  - тонкий диспетчер открытия (например `_openFeedPost(id)`): находит публикацию среди `!p.draft`; если найдена — пишет `state._openPostId` и `go('s-post')`; иначе `go('s-feed')` (push из `s-home`); экспортируется в `window` (как `_feedGoBook`, `feed.js:111`) для вызова из `onclick` плашки.
- **`docs/modules/profile.js`** — `_renderHomeFeedPreview`: у плашки превью `onclick="go('s-feed')"` (`profile.js:154`) заменяется на вызов диспетчера с передачей `id` через `data`-атрибут, например `data-pid="${esc(String(p.id))}"` + `onclick="_openFeedPost(this.dataset.pid)"`. Остальное (фото/иконка, обрезка текста `slice(0,45)`, `slice(0,6)`, условие показа секции) — без изменений.
- **`docs/app.js`** — `registerOnEnter('s-post', () => renderPostScreen())` (рядом с `s-feed`, `app.js:66`); импорт `renderPostScreen` из `feed.js`.
- **`docs/prototype.html`** — добавляется новый блок `<div class="screen" id="s-post">` (по образцу `s-feed`/`s-upcoming`): status-bar, nav-bar с заголовком «Публикация» и `back-btn` `onclick="back()"`, контейнер-список (`<div class="scroll">` с узлом под `_feedCardHtml`). **Бара фильтра категорий (`feed-cats`/`feedCatBar`) в `s-post` нет** (BA решение №2). Tab-bar — по аналогии с другими экранами (опционально). Ссылка «Все →» (`prototype.html:72`) и секция «Лента студии» по сути не меняются.
- **`docs/modules/state.js`** — при необходимости добавляется поле для `_openPostId` (либо используется как свободное поле объекта `state`); схема состояния по сути не меняется.
- **`docs/sw.js`** — состав `APP_SHELL` **не меняется** (правятся существующие файлы, уже входящие в `APP_SHELL`: `prototype.html`, `constants.js`, `feed.js`, `profile.js`, `app.js`, при необходимости `state.js`); `CACHE_VERSION` **поднимается на одну ступень** (`v53` → `v54`), чтобы клиенты получили обновлённые файлы.
- **`docs/version.json`** — обновляется мейнтейнером при релизе (вне объёма реализации фичи).
- **Новые рантайм-зависимости:** нет. YCLIENTS-токен не трогается. Операции с веткой `backup` запрещены.
- **Что ломается:** меняется целевое действие тапа по плашке превью (`go('s-feed')` → открытие конкретной публикации) — **намеренно**. `s-feed` (список, бар фильтра), кнопка «Записаться →» (`_feedGoBook`), ссылка «Все →», админка ленты, синхронизация с GitHub, YCLIENTS-интеграция — **не ломаются**.
