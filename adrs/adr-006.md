# ADR-006: Мультидевайс-подписки на push — устранение коллизии подписок по `clientId`

- **Дата:** 2026-05-28
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-006.md` (объём сужен: см. раздел «Контекст»; реализуется только починка доставки — корень проблемы), `requirements/req-admin-push-006.md` (раздел «Диагностика — подтверждено в ходе разбора»)
- **Связанные ADR:** ADR-005 (push-логика явно вне рамок того релиза; настоящее решение дорабатывает именно push-инфраструктуру, конфликта нет), ADR-001…ADR-004 (темы не пересекаются)

## Контекст

Studio PWA — устанавливаемое приложение онлайн-записи, отдаётся статически из `docs/` на GitHub Pages. Push-инфраструктура существует: клиентский модуль `docs/modules/push.js`, бэкенд — Cloudflare Worker `cf-worker/src/index.js` (`/subscribe`, `/unsubscribe`, `/send` с `X-Admin-Secret`, cron-напоминания), KV `SUBSCRIPTIONS`. Worker задеплоен на https://studio-push.ivahub84.workers.dev.

BA-требование 006 описывает широкий объём (реальные сегменты аудитории, серверное планирование, история рассылок, push конкретному клиенту, достоверная обратная связь). **Однако этот объём был сформулирован до того, как был подтверждён реальный корень проблемы и до решения пользователя об объёме.** Пользователь явно сузил настоящий релиз (v1.5.1, patch) **только до починки доставки**. Поэтому открытые вопросы BA №1, №2, №3, №5, №6, №7 (сегменты, серверная сегментация, планирование, единый журнал истории, подтверждение перед отправкой, поведение «нет подписчиков») и ФТ-2…ФТ-9 в части новой функциональности — **вне рамок настоящего ADR и не блокируют его**. Они остаются как есть и адресуются отдельным релизом при необходимости.

**Подтверждённый корень проблемы** (диагностирован совместно с пользователем). Симптом: push приходит в браузере на десктопе, но не приходит на iPhone. Предпосылки iOS соблюдены (standalone-установка на «Домой», iOS 16.4+, разрешение на уведомления выдано). На обоих устройствах выполнен вход под **одним** клиентом. Тост показывает «Отправлено 1», хотя подписаны два устройства.

Причина — **коллизия подписок по `clientId`**. Worker хранит ровно одну подписку на клиента под ключом `sub:${clientId}` (`handleSubscribe`, `cf-worker/src/index.js:142`). Два устройства одного клиента пишут в один и тот же ключ — подписавшееся последним **затирает** запись предыдущего. В KV остаётся единственный endpoint → `/send` доставляет только на него. iPhone push не получает, потому что его подписки в KV больше нет.

Тот же дефект ключа `sub:${clientId}` проявляется во всех местах, работающих с подписками:

- **`handleCron`** (напоминания, `index.js:247`) читает одну подписку на клиента → мультидевайс-клиент получает напоминание лишь на одно устройство.
- **`handleUnsubscribe`** (`index.js:154`) удаляет по `clientId` → отписка на одном устройстве сносит подписку другого устройства того же клиента.
- **`handleSend`** не логирует статус ответа push-сервиса по каждой подписке → отладка вслепую (нельзя понять, на какой endpoint и с каким кодом ушла доставка).

Формат отправки (VAPID JWT ES256 raw, `aud`=origin, `exp` 12 ч, aes128gcm; `encryptPayload`/`vapidToken`/`sendOne`) валиден и для Apple — **это не источник проблемы** и не пересматривается.

Силы, действующие на решение:

- **Стек неизменен.** Vanilla JS, без рантайм-npm-зависимостей в приложении; секреты только в Worker; единственный управляемый бэкенд — Worker, деплоится отдельно (`npx wrangler deploy`).
- **Минимальные изменения строго по объёму.** Меняется только маршрутизация/хранение подписок; криптография, сегментация, планирование, история — не трогаются.
- **Идемпотентность переподписки.** Повторная подписка того же устройства (тот же endpoint) не должна плодить дубли в KV.
- **Обратная совместимость хранилища.** В KV уже есть записи `sub:${clientId}` от текущих клиентов; нужна стратегия их судьбы.
- **Совместимость клиента и Worker.** Клиентский `unsubscribePush` сейчас шлёт `{clientId}`; для пер-устройственной отписки Worker нужен endpoint конкретного устройства — контракт клиент↔Worker меняется и должен оставаться согласованным.

Решение охватывает связанные вопросы: схема ключей KV для мультидевайс-подписок; идемпотентность переподписки; маршрутизация `/send` (target и broadcast); `handleCron`; контракт `/unsubscribe` и клиентский `unsubscribePush`; логирование и очистка по 410 в `handleSend`; миграция существующих записей; влияние на `CACHE_VERSION`. Конкретные имена полей, тексты и точная разметка — в спецификации релиза.

## Рассматриваемые варианты

### Вопрос 1. Схема ключей KV для нескольких устройств одного клиента

#### Вариант 1A (выбран): ключ по хешу endpoint, `clientId` хранится внутри значения

- Ключ записи подписки — `sub:${clientId}:${endpointHash}`, где `endpointHash` — стабильный детерминированный хеш `subscription.endpoint` (например SHA-256 от endpoint в base64url, доступен через Web Crypto в Worker без зависимостей). `clientId` хранится **внутри значения** (как сейчас), плюс сам `endpoint` для логирования/диагностики.
- Один endpoint (одно устройство) → один ключ. Несколько устройств одного клиента сосуществуют под общим префиксом `sub:${clientId}:`.
- Все подписки клиента перечисляются через `SUBSCRIPTIONS.list({ prefix: 'sub:${clientId}:' })`. Все подписки вообще (broadcast) — через `list({ prefix: 'sub:' })` (как сейчас, но теперь это все устройства).
- Переподписка того же устройства даёт тот же `endpointHash` → тот же ключ → запись **обновляется идемпотентно**, дубль не возникает.

- **Плюсы:**
  - Несколько устройств одного клиента сосуществуют (прямо устраняет корень проблемы).
  - Перечисление подписок клиента — дешёвый `list` по префиксу `sub:${clientId}:` (не требует индекса).
  - Broadcast `list({ prefix: 'sub:' })` продолжает работать без изменений по смыслу.
  - Идемпотентность переподписки бесплатна (детерминированный ключ от endpoint).
  - Удаление конкретного устройства по endpoint — вычислил хеш, удалил один ключ.
- **Минусы:**
  - Нужен хеш endpoint в Worker (Web Crypto `crypto.subtle.digest` — без зависимостей, есть в среде Worker).
  - Длина ключа растёт (clientId + хеш) — некритично для KV.
- **Последствия:** `handleSubscribe`, `handleUnsubscribe`, `handleSend` (target-ветка), `handleCron` переписываются на `list`-по-префиксу и хеш-ключи. Клиент должен передавать endpoint в `/unsubscribe`.

#### Вариант 1B (отвергнут): ключ `sub:${endpointHash}` без `clientId` в ключе

- Глобальный ключ только по хешу endpoint; `clientId` — только в значении.
- **Минусы:** отправка всем подпискам **конкретного клиента** (`targetClientId`, `handleCron`) потребовала бы перебора **всех** ключей `sub:` и фильтрации по `clientId` из значения — дороже и медленнее, особенно в cron при росте базы. Префикс `sub:${clientId}:` (1A) делает выборку по клиенту прямой. Отвергнут как менее эффективный для основных операций (target-send и cron).

#### Вариант 1C (отвергнут): хранить массив подписок под одним ключом `sub:${clientId}`

- Значение `sub:${clientId}` становится массивом `[{subscription,…}, …]`; устройства добавляются/удаляются внутри массива.
- **Минусы:** конкурентные подписки/отписки разных устройств в одно значение → гонка read-modify-write (KV не транзакционен), возможна потеря записи — тот же класс дефекта, что чинит этот релиз. Удаление одного устройства требует перезаписи всего значения. Отвергнут как ненадёжный при параллельных операциях.

### Вопрос 2. Маршрутизация `/send`

#### Вариант 2A (выбран): target → все подписки клиента; broadcast → все подписки

- При `targetClientId`: `list({ prefix: 'sub:${targetClientId}:' })`, отправить на **каждую** подписку клиента. Раньше отправлялась ровно одна (последняя записанная) — это и есть исправляемый дефект для адресной отправки.
- Без `targetClientId` (broadcast): `list({ prefix: 'sub:' })`, отправить на все подписки всех устройств (как сейчас по смыслу; теперь покрывает все устройства, а не по одному на клиента).
- Возвращаемый счётчик `sent`/`failed` теперь считает по устройствам — поэтому «Отправлено N» становится достоверным для мультидевайс-клиента.

- **Плюсы:** адресная и широковещательная доставка достигают всех устройств; счётчик достоверен; broadcast-контракт `/send` (тело, заголовок, ответ) внешне не меняется.
- **Минусы:** при target с несколькими устройствами `sent` может быть > 1 для одного клиента — это корректно (несколько устройств), но администратору в текущем UI этого релиза не объясняется (UI не в объёме). Приемлемо: число честно отражает доставленные уведомления.
- **Последствия:** target-ветка `handleSend` переписывается с одиночного `get` на `list`-по-префиксу клиента.

#### Вариант 2B (отвергнут): оставить target на одну «основную» подписку клиента

- **Минусы:** не доставляет на все устройства — прямо не устраняет корень проблемы для адресной отправки. Противоречит критерию готовности требования («на всех своих устройствах»). Отвергнут.

### Вопрос 3. `handleCron` (напоминания)

#### Вариант 3A (выбран): напоминание на все подписки клиента

- Для каждого клиента с записью на завтра: `list({ prefix: 'sub:${clientId}:' })`, отправить напоминание на **каждую** его подписку. Тот же дефект, что в `/send` target.
- Очистка по 410 — удалять конкретный ключ устройства (см. Вопрос 5), а не все подписки клиента.

- **Плюсы:** мультидевайс-клиент получает напоминание на всех устройствах; согласовано с Вопросом 2A; чинит тот же корень.
- **Минусы:** немного больше отправок (по числу устройств) — это и есть желаемое поведение.
- **Последствия:** цикл `handleCron` переходит с одиночного `get` на `list`-по-префиксу клиента.

#### Вариант 3B (отвергнут): не трогать cron в этом релизе

- BA относит изменение механизма cron к «вне рамок», **но** пользователь явно включил `handleCron` в объём v1.5.1 как проявление того же дефекта. Механизм напоминаний (когда и о чём) не меняется — меняется только перебор подписок клиента. Отвергнут: оставлять заведомо тот же баг недопустимо при явном включении в объём.

### Вопрос 4. Контракт `/unsubscribe` и клиентский `unsubscribePush`

#### Вариант 4A (выбран): отписка по endpoint конкретного устройства

- `/unsubscribe` принимает `{ clientId, endpoint }`; Worker вычисляет `endpointHash` и удаляет **один** ключ `sub:${clientId}:${endpointHash}` — подписку только этого устройства. Подписки других устройств клиента сохраняются.
- Клиентский `unsubscribePush` (`docs/modules/push.js`) получает текущий endpoint из `reg.pushManager.getSubscription()` **до** локального `sub.unsubscribe()` и передаёт его в теле запроса вместе с `clientId`.
- Совместимость: если `endpoint` не передан (старый клиент из кэша) — Worker возвращает понятную ошибку `missing_fields` (а не молча удаляет всё). Это безопаснее прежнего поведения «снести по clientId».

- **Плюсы:** отписка устройства не затрагивает другие устройства клиента (устраняет следствие дефекта); endpoint доступен на клиенте до локальной отписки; контракт явный.
- **Минусы:** меняется тело запроса `/unsubscribe` (добавлен `endpoint`) — нужно синхронно обновить клиент и Worker. Поскольку оба в этом релизе — управляемо.
- **Последствия:** правки `handleUnsubscribe` (Worker) и `unsubscribePush` (`push.js`). `push.js` входит в `APP_SHELL` → подъём `CACHE_VERSION` (Вопрос 7).

#### Вариант 4B (отвергнут): оставить `/unsubscribe` по `clientId` (удаляет все устройства)

- **Минусы:** отписка на одном устройстве сносит подписки остальных устройств клиента — прямо воспроизводит следствие исправляемого дефекта. Отвергнут.

### Вопрос 5. Логирование и очистка по 410 в `handleSend`

#### Вариант 5A (выбран): логировать хост endpoint + статус по каждой подписке; 410 удаляет конкретную запись

- В `handleSend` (и при желании в `handleCron`) для каждой отправки логировать диагностируемую строку: хост endpoint (`new URL(endpoint).host`, без полного endpoint — он чувствителен) и статус ответа push-сервиса. Полный endpoint в лог не пишется.
- При статусе 410 (Gone) — удалять **конкретный** ключ устройства `sub:${clientId}:${endpointHash}`, а не подписку клиента целиком. Это даёт самоочистку устаревших устройств без задевания живых.

- **Плюсы:** доставка становится диагностируемой (видно, на какой push-сервис и с каким кодом ушло); очистка по 410 точечная; не логируется чувствительный полный endpoint.
- **Минусы:** немного больше строк в логах Worker — приемлемо.
- **Последствия:** в циклах `handleSend`/`handleCron` добавляется `console.log`/`console.warn` с хостом и статусом; удаление по 410 — по точному ключу.

#### Вариант 5B (отвергнут): не логировать, оставить как есть

- **Минусы:** диагностика остаётся «вслепую» (одно из следствий дефекта, явно включённое в объём). Отвергнут.

### Вопрос 6. Миграция существующих записей `sub:${clientId}` в KV

#### Вариант 6A (выбран): полагаться на переподписку клиентов; старые ключи не мигрировать, дать им самоистечь

- Существующие записи формата `sub:${clientId}` (без суффикса хеша) не мигрируются принудительно.
- При следующем запуске клиента `subscribePush` (вызывается при логине из `app.js`/`auth.js`) запишет подписку в **новый** ключ `sub:${clientId}:${endpointHash}` → устройство сразу попадает в новую схему.
- Чтобы старые «осиротевшие» ключи `sub:${clientId}` не порождали мусорную доставку и не путали логику: broadcast `list({ prefix: 'sub:' })` подберёт и их; рекомендуется при чтении значения толерантно обрабатывать обе формы (старая запись валидна как одна подписка), а 410-очистка постепенно удалит мёртвые. Дополнительно допустим разовый ручной сброс KV-неймспейса администратором (вне кода).

- **Плюсы:**
  - Ноль кода миграции — минимальные изменения строго по объёму (НФТ-1, принцип «минимальные изменения»).
  - Клиенты переподписываются при первом же логине после деплоя — естественная и быстрая конвергенция (подписка идёт на каждом входе).
  - Нет риска некорректной миграции (старый формат содержит достаточно данных, но миграционный скрипт — лишний код и риск).
- **Минусы:**
  - До переподписки клиента его доставка идёт по старому одиночному ключу (т.е. сохраняется старое поведение для этого клиента до первого нового логина) — приемлемо: дефект исчезает по мере естественной переподписки, регресса нет.
  - Возможны временные «осиротевшие» старые ключи — самоочистятся по 410 либо разовым ручным сбросом.
- **Последствия:** код чтения подписки должен толерантно принимать и старый формат значения (структура значения совместима — `{subscription, clientId, …}`).

#### Вариант 6B (отвергнут): написать миграционный проход по KV (переложить `sub:${clientId}` → `sub:${clientId}:${hash}`)

- **Минусы:** дополнительный одноразовый код (эндпоинт/скрипт миграции), который после применения становится мёртвым; риск ошибок при массовом переписывании; не оправдан при малой базе и при том, что переподписка происходит на каждом логине. Противоречит принципу минимальных изменений. Отвергнут.

### Вопрос 7. Влияние на `CACHE_VERSION` и версию

#### Вариант 7A (выбран): поднять `CACHE_VERSION` v38 → v39 (правится `push.js` в `APP_SHELL`); состав `APP_SHELL` не меняется

- Изменяется клиентский файл `docs/modules/push.js`, который **уже входит** в `APP_SHELL` (`docs/sw.js:32`). По правилу проекта при изменении файла в `APP_SHELL` поднимается `CACHE_VERSION`.
- Состав `APP_SHELL` **не меняется** (новых файлов нет). Поднимается только номер версии: `v38` → `v39`.
- `version.json`: `build` 6 → 7, `version` → `v1.5.1`.

- **Плюсы:** соблюдено правило проекта (кэш SW обновится, клиенты получат новый `push.js` с endpoint в `/unsubscribe`); минимальная правка SW.
- **Минусы:** нет (разовая операция).
- **Последствия:** правка `docs/sw.js` (только номер) и `docs/version.json`.

#### Вариант 7B (отвергнут): не поднимать `CACHE_VERSION`

- **Минусы:** старый `push.js` остался бы в кэше у установленных клиентов → клиент слал бы `/unsubscribe` без endpoint, рассинхрон с новым Worker. Нарушает правило проекта. Отвергнут.

## Решение

- **Вопрос 1 — Вариант 1A:** ключ `sub:${clientId}:${endpointHash}` (хеш endpoint через Web Crypto), `clientId` и `endpoint` — внутри значения; выборка подписок клиента по префиксу `sub:${clientId}:`, broadcast по `sub:`.
- **Вопрос 2 — Вариант 2A:** `/send` с `targetClientId` → все подписки клиента; broadcast → все подписки.
- **Вопрос 3 — Вариант 3A:** `handleCron` шлёт напоминание на все подписки клиента.
- **Вопрос 4 — Вариант 4A:** `/unsubscribe` удаляет подписку конкретного устройства по `endpoint`; клиентский `unsubscribePush` передаёт endpoint.
- **Вопрос 5 — Вариант 5A:** логировать хост endpoint + статус по каждой подписке; 410 удаляет конкретный ключ устройства.
- **Вопрос 6 — Вариант 6A:** миграцию не делать, полагаться на переподписку при логине; толерантное чтение старого формата; самоочистка по 410.
- **Вопрос 7 — Вариант 7A:** `CACHE_VERSION` v38 → v39 (правится `push.js`); состав `APP_SHELL` без изменений; `version.json` → `v1.5.1`.

Отвергнуты: 1B (дорогой target/cron-перебор), 1C (гонка read-modify-write), 2B (не доставляет на все устройства), 3B (оставляет тот же баг в cron), 4B (отписка сносит чужие устройства), 5B (диагностика вслепую), 6B (лишний миграционный код), 7B (рассинхрон клиента и Worker).

### Границы решения

- Изменяется **только** хранение и маршрутизация push-подписок (мультидевайс) и связанная диагностика. Затрагиваются: `cf-worker/src/index.js` (`handleSubscribe`, `handleUnsubscribe`, `handleSend`, `handleCron`, плюс хелпер хеша endpoint), `docs/modules/push.js` (`unsubscribePush`; `subscribePush` — без изменения контракта: endpoint уже передаётся внутри `subscription`), `docs/sw.js` (`CACHE_VERSION`), `docs/version.json`.
- **Не затрагиваются (остаются как есть, вне объёма):** сегменты аудитории и `updatePushAudience` (`admin.js`), серверное планирование рассылок, история рассылок (`localStorage` `yc_push_campaigns`), вывод «push конкретному клиенту» в UI списка клиентов, достоверность обратной связи сверх корректного `sent`/`failed` от Worker, криптография Web Push (`encryptPayload`/`vapidToken`/`sendOne`), формат VAPID, `/review`-проксирование, манифест, иконки, состав `APP_SHELL`, сценарий записи, отзывы.
- **Контракт `/subscribe` внешне не меняется** (тело `{subscription, clientId, phone}` то же) — меняется только ключ, под которым Worker сохраняет запись.

## Последствия

- **Положительные:**
  - Несколько устройств одного клиента сосуществуют в KV; push доставляется на **все** устройства (устранён корень: iPhone снова получает push при общем входе с десктопом).
  - `handleCron` доставляет напоминания на все устройства клиента; адресная и широковещательная отправка достигают всех устройств; счётчик `sent` достоверен для мультидевайса.
  - Отписка устройства не задевает другие устройства клиента; переподписка того же устройства идемпотентна (без дублей).
  - Доставка диагностируема (хост endpoint + статус в логах Worker); устаревшие подписки самоочищаются по 410 точечно.
  - Без новых рантайм-зависимостей (НФТ-1); секреты не затрагиваются и остаются в Worker (НФТ-2); `/send` остаётся под `X-Admin-Secret`.
- **Отрицательные / компромиссы:**
  - До первой переподписки клиента (первый логин после деплоя) его доставка идёт по старому одиночному ключу — временно сохраняется старое поведение для такого клиента (без регресса). Конвергенция быстрая (подписка на каждом логине).
  - Возможны временные «осиротевшие» ключи `sub:${clientId}` — самоочистка по 410 или разовый ручной сброс KV администратором.
  - `/unsubscribe` теперь требует endpoint; старый кэшированный клиент без endpoint получит `missing_fields` (безопасный отказ вместо удаления всего) — устраняется обновлением SW (`CACHE_VERSION` v39).
- **Технический долг:**
  - Реальные сегменты аудитории, серверное планирование, единый журнал истории, push-в-UI-списке-клиентов, явная защита от ложного «успеха» при отсутствии конфигурации — **отложены** (вне объёма v1.5.1), адресуются отдельным релизом из BA-006.
  - Захардкоженный YCLIENTS-токен в `prototype.html` — известный долг (CLAUDE.md), не усугубляется и не трогается.

## Влияние на код

- **`cf-worker/src/index.js`:**
  - Новый хелпер `endpointHash(endpoint)` — детерминированный base64url-хеш endpoint через `crypto.subtle.digest('SHA-256', …)` (без зависимостей).
  - `handleSubscribe` — писать под ключ `sub:${clientId}:${await endpointHash(subscription.endpoint)}`; значение хранит `subscription`, `clientId`, `phone`, `endpoint`, `ts`. Идемпотентно (тот же endpoint → тот же ключ).
  - `handleUnsubscribe` — принимать `{ clientId, endpoint }`; при отсутствии `endpoint` → `missing_fields`; удалять один ключ `sub:${clientId}:${hash}`.
  - `handleSend` — target-ветка: `list({ prefix: 'sub:${targetClientId}:' })`, отправка на все; broadcast: `list({ prefix: 'sub:' })` (как есть). Логировать хост+статус по каждой подписке; 410 → удалить конкретный ключ. Толерантно читать старый формат значения.
  - `handleCron` — для каждого клиента с записью: `list({ prefix: 'sub:${clientId}:' })`, отправка на все; 410 → удалить конкретный ключ; логировать хост+статус.
- **`docs/modules/push.js`:**
  - `unsubscribePush` — получить endpoint текущей подписки из `reg.pushManager.getSubscription()` **до** локального `sub.unsubscribe()`; передать `{ clientId, endpoint }` в `/unsubscribe`.
  - `subscribePush` — контракт без изменений (endpoint уже внутри `subscription.toJSON()`); правок не требует, если не считать опциональной согласованности.
- **`docs/sw.js`:** `CACHE_VERSION` `v38` → `v39` (изменён `push.js` в `APP_SHELL`). Состав `APP_SHELL` не меняется.
- **`docs/version.json`:** `build` 6 → 7, `version` → `v1.5.1` (мейнтейнер обновляет `date`/`time`/`sha` при релизе).
- **Деплой:** Worker деплоится отдельно (`cd cf-worker && npx wrangler deploy`); PWA — `git push origin main`. Оба должны выкатываться согласованно (новый контракт `/unsubscribe`).
- **Новые рантайк-зависимости:** нет. Манифест/иконки/`/review`/криптография — не трогаются.
- **Что ломается:** тело запроса `/unsubscribe` (добавлен `endpoint`) — синхронно обновляются клиент и Worker. Старый кэшированный клиент без endpoint получает безопасный отказ до обновления SW. Сценарий записи, отзывы, broadcast `/send`, навигация — не ломаются.
