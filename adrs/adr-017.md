# ADR-017: Скрытие плашки «Оцените прошлый визит» на `s-home` без оценки — пер-`recordId` дисмисс в новом ключе `localStorage` `yc_dismissed_rate_ids`, крестик «×» на карточке, перерисовка через `window.renderHomeHero()`

- **Дата:** 2026-05-29
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-020.md` (возможность скрыть плашку «Оцените прошлый визит» без выставления оценки; ОВ-1 РЕШЕН заказчиком — вариант «а»; ОВ-2/ОВ-3 — за архитектором; ОВ-4 подтверждён; ОВ-5 закрыт анализом)
- **Связанные ADR:** ADR-016 (`adrs/adr-016.md`) — серия UI-polish раздела услуг, не пересекается с правкой `s-home`; паттерн `registerOnEnter('s-home')` (`docs/app.js:56-69`) и условный рендер блока главного экрана переиспользуются. Прямых конфликтов с ADR-001…ADR-016 не выявлено: ни один ADR не описывает логику плашки «Оцените прошлый визит» как принятое решение — это поведение появилось в коде `profile.js:90-114` без отдельного ADR; настоящий ADR его уточняет, не пересматривает.

## Контекст

Studio PWA — устанавливаемое статическое приложение онлайн-записи (студия красоты), отдаётся из `docs/` на GitHub Pages, vanilla JS/HTML/CSS без рантайм-зависимостей.

На главном экране `s-home` (блок `#homeHero`, рендер `renderHomeHero` — `docs/modules/profile.js:63-130`), когда у клиента **нет предстоящих записей** (`!next`, `profile.js:92`), показывается плашка **«Оцените прошлый визит»** для «последнего неоценённого визита» `lastUnreviewed`: запись за последние 7 дней, не отменённая, уже прошедшая, чьего `String(r.id)` нет в множестве `reviewedIds = new Set(_loadReviewedIds())` (`profile.js:93-99`). В плашке **единственное** действие — кнопка «Оставить отзыв →» (`btn-ghost`, `profile.js:105`), ведущая на `s-review` через `openRateVisit(...)`. Если `lastUnreviewed` нет — выводится штатный фолбэк «Нет предстоящих записей» (`profile.js:109-112`).

Проблема (BA-020 §1): убрать плашку можно **только оставив отзыв**. Клиент, не желающий оценивать визит, видит её при каждом заходе на главный, пока запись не выпадет из 7-дневного окна — воспринимается как навязчивость. Нужно дать **отдельное явное действие «скрыть без оценки»**, после которого плашка исчезает (на её место — «Нет предстоящих записей»), а при повторных заходах для того же визита больше не появляется.

Подтверждено кодом на момент проектирования:

- **Паттерн персистентности «оценено».** `yc_reviewed_ids` (массив строковых id) + `_loadReviewedIds()`/`_saveReviewedId(recordId)` с `try/catch` (`docs/modules/review.js:8-21`); добавление id после успешной отправки отзыва — `review.js:133`. Чтение возвращает `[]` при любой ошибке `JSON.parse`/недоступности `localStorage` (graceful degradation).
- **Условие показа и рендер плашки.** `profile.js:92-114`: проверка `!next`, формирование `lastUnreviewed` через цепочку `.filter`, рендер карточки с экранированием `esc()` (`svcName`, `masterName`, `data-*`-атрибуты), `return` после плашки; иначе фолбэк «Нет предстоящих записей».
- **Перерисовка главного экрана.** `renderHomeHero` экспортирован в `window` (`profile.js:318`) и регистрируется как `registerOnEnter('s-home', …)` (`docs/app.js:56-69`); также вызывается из `auth.js:286,363`, `booking.js:126,238`, `app.js:221,283`. Повторный вызов `renderHomeHero()` полностью перестраивает `innerHTML` блока `#homeHero` — это штатный путь обновления.
- **История визитов (вне scope).** Кнопка «Оценить» в `_renderHistoryFromCache` (`docs/modules/history.js:206`) завязана **только** на `reviewedIds` (`history.js:157`), не на скрытие; ОВ-4 — не затрагивается.
- **Счётчики/уведомления.** Поиском по `docs/modules/` логики бейджей/счётчиков на «неоценённые визиты» не обнаружено (BA-020 Д-3, ОВ-5) — скрытие ни на что, кроме показа плашки, не влияет.

ADR охватывает три развилки: **(N)** семантика и место хранения дисмисса; **(O)** UI-элемент действия скрытия и стабильность вёрстки; **(P)** механизм немедленной перерисовки блока после скрытия.

## Рассматриваемые варианты

### Вопрос N (ОВ-1/ОВ-2). Семантика «скрыть» и место хранения

> ОВ-1 РЕШЕН заказчиком — **вариант «а»**: скрытие помечает конкретный `recordId` как «не показывать подсказку об оценке». «Оценено» (`yc_reviewed_ids`) и «скрыто» — **разные** состояния (Д-1). Для нового неоценённого визита (другой `recordId`) плашка появляется снова (Д-5). Архитектор фиксирует имя ключа/формат (ОВ-2).

#### Вариант N1 (выбран): новый отдельный ключ `localStorage` `yc_dismissed_rate_ids` (массив строковых id), с помощниками-аналогами `_loadDismissedRateIds()`/`_saveDismissedRateId()` в `docs/modules/review.js`

Полностью переиспользуем проверенный паттерн `yc_reviewed_ids`:

- Ключ: `yc_dismissed_rate_ids` (рекомендация BA приемлема — отдельный ключ, не смешивать с `yc_reviewed_ids`, Д-1).
- Формат: массив строковых id (`String(recordId)`), как в `yc_reviewed_ids`.
- Помощники: `_loadDismissedRateIds()` (читает массив, `[]` при ошибке/недоступности) и `_saveDismissedRateId(recordId)` (добавляет `String(recordId)` без дублей, запись в `try/catch`) — структурно идентичны `_loadReviewedIds()`/`_saveReviewedId()` (`review.js:8-21`), размещаются рядом с ними в том же модуле.
- В условии показа плашки (`profile.js:93-99`) к существующему `.filter(r => !reviewedIds.has(String(r.id)))` добавляется параллельный фильтр по множеству `dismissedIds = new Set(_loadDismissedRateIds())`: `.filter(r => !dismissedIds.has(String(r.id)))`.

- **Плюсы:** прямое переиспользование паттерна (НФТ-1, минимальность НФТ-3); «оценено» и «скрыто» строго разделены (Д-1) — кнопка «Оценить» в истории по-прежнему завязана только на `reviewedIds` и не затрагивается (ФТ-5, ОВ-4); graceful degradation наследуется из `try/catch` (ФТ-7, КП-7); пер-`recordId` гранулярность даёт предсказуемое поведение С2/С3 (Д-5, КП-3, КП-4); никаких секретов в хранилище.
- **Минусы:** ещё один ключ `localStorage` (но согласуется с явным требованием не смешивать состояния, Д-1).
- **Последствия:** правка `docs/modules/review.js` (2 новых экспортируемых помощника рядом с `review.js:8-21`) и `docs/modules/profile.js` (импорт + новый фильтр в `profile.js:93-99` + обработчик скрытия). Файлы `APP_SHELL` → поднятие `CACHE_VERSION`.

#### Вариант N2 (отвергнут): хранить дисмисс в том же `yc_reviewed_ids`

- **Минусы:** прямо нарушает Д-1 — скрытый визит стал бы «оценённым», и кнопка «Оценить» в истории (`history.js:206`, завязана на `reviewedIds`) исчезла бы для него (нарушение ФТ-5, ОВ-4). Смешение двух разных состояний. Отвергнут.

#### Вариант N3 (отвергнут): временный/глобальный дисмисс (ОВ-1 «б»/«в») — флаг «не показывать N дней» или «навсегда»

- **Минусы:** заказчик выбрал вариант «а» (пер-`recordId`). «б»/«в» подавляли бы плашку и для **будущих** новых неоценённых визитов (другой `recordId`), что противоречит Д-5/КП-4 и бизнес-цели «сохранить возможность собирать отзывы». Отвергнут как противоречащий принятому решению ОВ-1.

### Вопрос O (ОВ-3). UI-элемент действия скрытия и стабильность вёрстки

> ОВ-3 — на проектное усмотрение; без подтверждающей модалки; учесть тач/UX (НФТ-4) и стабильность вёрстки. Выбор зафиксировать в ADR.

#### Вариант O1 (выбран): крестик «×» в правом верхнем углу карточки плашки, абсолютным позиционированием поверх контента, кнопка «Оставить отзыв →» сохраняется как основное действие

- В карточке плашки (`profile.js:101-106`) добавляется кнопка-крестик «×» в правом верхнем углу: `position:absolute` относительно карточки (`position:relative`), достаточная тач-зона (рекомендуемый размер ≥ 32×32px, визуальный глиф меньше), `aria-label="Скрыть"`, `type="button"`. Кнопка «Оставить отзыв →» остаётся внизу как основное действие — действия визуально и по приоритету разделены: основное (отзыв) — крупная кнопка, второстепенное (скрыть) — ненавязчивый крестик.
- Без подтверждающей модалки (действие обратимо через историю визитов — ОВ-4).
- Стабильность вёрстки: крестик позиционируется абсолютно поверх карточки и **не влияет** на поток (заголовок/название услуги/дата не сдвигаются); карточка получает `position:relative` и при необходимости небольшой `padding-right`, чтобы текст заголовка не подползал под крестик.
- Обработчик: `onclick` крестика вызывает функцию-обработчик скрытия с `recordId` визита (передаётся через `data-rid`, как у существующей кнопки отзыва — `profile.js:105`), без `stopPropagation`-конфликтов (карточка некликабельна целиком, в отличие от истории).

- **Плюсы:** общепринятый и узнаваемый паттерн «закрыть карточку»; крестик не конкурирует с основным действием «Оставить отзыв →» (сохраняется приоритет сбора отзывов — бизнес-цель); абсолютное позиционирование не двигает контент (стабильность вёрстки, НФТ-4); компактно, без новой строки в карточке.
- **Минусы:** на узких экранах нужно зарезервировать место справа у заголовка, чтобы текст не уходил под крестик (решается `padding-right`/`max-width` заголовка).
- **Последствия:** правка разметки карточки в `profile.js:101-106` (добавить крестик + `position:relative` карточке) и при необходимости стили в `docs/style.css` (класс крестика/тач-зона). Файлы `APP_SHELL` → `CACHE_VERSION`.

#### Вариант O2 (отвергнут): текстовая кнопка «Позже»/«Скрыть» рядом/под «Оставить отзыв →»

- **Минусы:** две равновесные текстовые кнопки в одной маленькой карточке усиливают визуальный шум, который требование как раз и стремится снизить; «Позже» неоднозначна (наводит на «временно», что соответствует отвергнутому N3, а не пер-`recordId`-дисмиссу). Крестик семантически чище для «закрыть навсегда этот элемент». Допустим как запасной, но проигрывает O1 по приоритизации основного действия. Отвергнут.

#### Вариант O3 (отвергнут): свайп карточки для скрытия

- **Минусы:** конфликтует с вертикальным скроллом главного экрана; невидимый аффорданс (нет визуального намёка), риск случайных срабатываний при скролле (прямо против НФТ-4); сложнее реализовать в vanilla без жест-движка. Отвергнут.

### Вопрос P. Механизм немедленной перерисовки блока после скрытия

> ФТ-2/КП-2: после нажатия скрытия плашка немедленно исчезает, на её месте — «Нет предстоящих записей».

#### Вариант P1 (выбран): обработчик сохраняет id через `_saveDismissedRateId(recordId)`, затем вызывает `window.renderHomeHero()` для полной перерисовки блока `#homeHero`

Обработчик скрытия (новая функция в `docs/modules/profile.js`, экспортируемая в `window`, по образцу прочих `window`-функций модуля — `profile.js:318`):

1. `_saveDismissedRateId(recordId)` — персистит дисмисс (переживает перезагрузку, ФТ-3).
2. `renderHomeHero()` — повторно строит `#homeHero`; на втором проходе условия `lastUnreviewed` теперь добавлен фильтр `!dismissedIds.has(...)`, скрытый визит отсеивается → выводится фолбэк «Нет предстоящих записей» (ФТ-2). Если есть другой неоценённый/нескрытый визит — покажется он (корректно по Д-5).

- **Плюсы:** переиспользует штатный и многократно вызываемый путь обновления (`renderHomeHero` уже в `window` и в `registerOnEnter('s-home')`) — никакой новой инфраструктуры (НФТ-1, минимальность); единый источник правды о состоянии блока (нет рассинхрона между ручным удалением DOM и условием показа); персист гарантирует поведение при перезагрузке (ФТ-3/ФТ-4).
- **Минусы:** перерисовывает весь `#homeHero` (включая ветку мастеров/предстоящих), а не только плашку — но это дёшево и уже происходит при каждом входе на экран; визуально незаметно.
- **Последствия:** обработчик в `profile.js` вызывает существующий `renderHomeHero`; новых точек входа не вводится.

#### Вариант P2 (отвергнут): вручную удалить/спрятать DOM-узел плашки без перерисовки

- **Минусы:** ручное удаление узла оставляет блок пустым — не покажет фолбэк «Нет предстоящих записей» (нарушение ФТ-2/КП-2) без дублирования логики фолбэка в обработчике; создаёт второй источник правды о содержимом блока, рассинхрон с условием показа. Отвергнут в пользу единого пути перерисовки P1.

## Решение

- **Вопрос N — N1:** новый ключ `localStorage` `yc_dismissed_rate_ids` (массив строковых id) с помощниками `_loadDismissedRateIds()`/`_saveDismissedRateId(recordId)` в `docs/modules/review.js`, структурно идентичными `_loadReviewedIds()`/`_saveReviewedId()` (`review.js:8-21`), включая `try/catch` graceful degradation. В условии показа плашки (`profile.js:93-99`) добавляется параллельный фильтр `.filter(r => !dismissedIds.has(String(r.id)))` к множеству `dismissedIds = new Set(_loadDismissedRateIds())`. «Оценено» и «скрыто» — разные состояния (Д-1); кнопка «Оценить» в истории (`history.js:206`) не затрагивается (ОВ-4). *Закрывает ФТ-3, ФТ-4, ФТ-5, ФТ-7; КП-3, КП-4, КП-5, КП-7.*
- **Вопрос O — O1:** действие скрытия — крестик «×» в правом верхнем углу карточки плашки, абсолютным позиционированием поверх контента (карточке — `position:relative`), тач-зона ≥ 32×32px, `aria-label="Скрыть"`, `type="button"`, без подтверждающей модалки. Кнопка «Оставить отзыв →» сохраняется как основное действие и работает как прежде. Контент карточки (заголовок/название/дата) не сдвигается. *Закрывает ФТ-1, ФТ-6; КП-1, КП-6; НФТ-2, НФТ-4.*
- **Вопрос P — P1:** обработчик скрытия (новая `window`-функция в `profile.js`, получает `recordId` через `data-rid`) вызывает `_saveDismissedRateId(recordId)`, затем `renderHomeHero()` — блок немедленно перерисовывается в фолбэк «Нет предстоящих записей» (или в другой нескрытый неоценённый визит, если такой есть). *Закрывает ФТ-2; КП-2, КП-8.*

Отвергнуты: N2 (общий ключ с `yc_reviewed_ids` — ломает Д-1/ФТ-5/ОВ-4), N3 (временный/глобальный дисмисс — противоречит ОВ-1«а»/Д-5/КП-4), O2 (текстовая кнопка — визуальный шум, неоднозначность «Позже»), O3 (свайп — конфликт со скроллом, против НФТ-4), P2 (ручное удаление DOM — не показывает фолбэк, второй источник правды).

### Границы решения

- Изменения **только** в блоке плашки «Оцените прошлый визит» на `s-home` (`profile.js:100-106`), условии её показа (`profile.js:93-99`), новом механизме хранения дисмисса (`review.js`) и связанном обработчике/стилях. **Не трогаются:**
  - Поток «Оставить отзыв» (`openRateVisit` → `s-review` → `submitReview`, `review.js`) — логика отправки отзыва и `yc_reviewed_ids` без изменений (ФТ-6).
  - Кнопка «Оценить» в истории визитов (`history.js:206`) и её зависимость от `reviewedIds` (`history.js:157`) — без изменений (ОВ-4, ФТ-5).
  - Логика предстоящих записей (`next`, `upcoming`, `profile.js:85-88,116-129`) и сам фолбэк «Нет предстоящих записей» (`profile.js:109-112`) — без изменений (плашка лишь чаще уступает место фолбэку).
  - Правила формирования `lastUnreviewed` (окно 7 дней, фильтры по статусу/дате) — без изменений, кроме добавления параллельного фильтра по `dismissedIds`.
- Серверная/YCLIENTS-сторона: при скрытии **ничего не отправляется на бэкенд** — состояние только локальное (`localStorage`).
- Без новых рантайм-зависимостей. Внешние данные визита в плашке — только через `esc()` (НФТ-2). YCLIENTS-токен не трогается (техдолг — не усугублять). Операции с веткой `backup` запрещены.

## Последствия

- **Положительные:**
  - Клиент может отклонить подсказку об оценке одним тапом по крестику; навязчивость устранена при сохранении возможности оставить отзыв (основная кнопка на месте).
  - Дисмисс переживает перезагрузку/переустановку сессии и не повторяется для того же визита (ФТ-3/ФТ-4).
  - «Оценено» и «скрыто» строго разделены: визит остаётся неоценённым, доступен для оценки из истории (ФТ-5).
  - Решение локально, переиспользует проверенные паттерны (`yc_reviewed_ids`-помощники, `renderHomeHero`), не задевает поток отзывов, историю и предстоящие записи.
  - Для будущего нового неоценённого визита (другой `recordId`) плашка появляется снова (Д-5) — сбор отзывов не подавляется глобально.
- **Отрицательные / компромиссы:**
  - Появляется второй параллельный список id в `localStorage` (`yc_dismissed_rate_ids`) рядом с `yc_reviewed_ids` — осознанный компромисс ради разделения состояний (Д-1).
  - Крестик требует резервирования места справа у заголовка карточки на узких экранах (решается `padding-right`/`max-width`).
- **Технический долг:**
  - Очистка `yc_dismissed_rate_ids` от id визитов, выпавших из 7-дневного окна, не делается (список монотонно растёт) — объём незначителен (строковые id), вне scope BA-020; при необходимости — отдельным требованием.
  - Синхронизация дисмисса между устройствами/на бэкенд не предусмотрена (локальное состояние) — вне scope.

## Риски

- **Риск (НИЗКИЙ): скрытие делает визит «оценённым» (попадает в `yc_reviewed_ids`) → исчезает «Оценить» в истории.** → **Митигация:** дисмисс хранится строго в отдельном ключе `yc_dismissed_rate_ids`; `yc_reviewed_ids` и `_saveReviewedId` не трогаются; фильтр по `dismissedIds` параллелен фильтру по `reviewedIds`, не заменяет его. Проверить, что «Оценить» в истории для скрытого визита доступна (КП-5).
- **Риск (НИЗКИЙ): после скрытия блок пуст вместо фолбэка «Нет предстоящих записей».** → **Митигация:** перерисовка через `renderHomeHero()` (P1), а не ручное удаление узла; на втором проходе условие показа само выводит фолбэк (КП-2).
- **Риск (НИЗКИЙ): глобальное подавление плашки для будущих визитов.** → **Митигация:** дисмисс пер-`recordId`; фильтр исключает только конкретный `String(r.id)`; для другого `recordId` плашка появляется снова (КП-4, Д-5).
- **Риск (НИЗКИЙ): недоступность `localStorage` ломает рендер главного экрана.** → **Митигация:** `_loadDismissedRateIds()` возвращает `[]` при любой ошибке, `_saveDismissedRateId` пишет в `try/catch` (как `review.js:19`); рендер не падает, в консоли нет ошибок (КП-7).
- **Риск (НИЗКИЙ): крестик сдвигает контент карточки / мелкая тач-зона.** → **Митигация:** абсолютное позиционирование поверх карточки (не в потоке), тач-зона ≥ 32×32px, `padding-right` заголовка; визуальная проверка на узком экране (КП-8, НФТ-4).
- **Риск (НИЗКИЙ): XSS через данные визита в плашке.** → **Митигация:** `svcName`, `masterName`, `data-*` — только через `esc()`, как в текущем рендере; крестик не выводит пользовательских данных (КП-8, НФТ-2).
- **Риск (НИЗКИЙ): старый кэш SW отдаёт прежние `review.js`/`profile.js`/`style.css`.** → **Митигация:** при изменении файлов `APP_SHELL` **поднять `CACHE_VERSION`** в `docs/sw.js` (`v63 → v64`); мейнтейнер обновляет `docs/version.json` (НФТ-6).
