# ADR-007: Рабочие per-client предпочтения push (рассылки / напоминания) и второе напоминание перед визитом

- **Дата:** 2026-05-28
- **Статус:** принято
- **Связанные BA-требования:** `ba-req/ba-requirements-007.md` (все 6 открытых вопросов закрыты решениями от 2026-05-28)
- **Связанные ADR:** ADR-006 (мультидевайс-подписки `sub:${clientId}:${endpointHash}` — фундамент, не пересматривается), ADR-005 (install prompt; push вне рамок того релиза), ADR-001…ADR-004 (темы не пересекаются)

## Контекст

Studio PWA — устанавливаемое приложение онлайн-записи, отдаётся статически из `docs/` на GitHub Pages; бэкенд push — Cloudflare Worker `cf-worker/src/index.js` (KV `SUBSCRIPTIONS`, ключ устройства `sub:${clientId}:${endpointHash}`, ADR-006). Worker задеплоен на https://studio-push.ivahub84.workers.dev.

В профиле клиента есть раздел «Настройки уведомлений» с тумблерами. **Подтверждено кодом:** тумблеры косметические — `consentToggle` (`docs/modules/consent.js`) лишь переключает CSS-класс `on` и показывает модалку. Они не подписывают/отписывают push, не сохраняют серверную настройку, не влияют на `handleSend` (админские рассылки) и `handleCron` (напоминания). Подписка работает «всё или ничего» (`subscribePush`/`unsubscribePush` в `docs/modules/push.js`).

BA-007 требует дать клиенту **два реально работающих независимых** тумблера и убрать два нерелевантных пункта. Принятые продуктовые решения (BA, раздел 8, от 2026-05-28), обязательные к учёту:

1. Объём — только настройки профиля; onboarding-экран `s-consent` не трогать.
2. Два независимых типа push: «Push-уведомления» (админские рассылки, `handleSend`) и «Напоминания о визитах» (`handleCron`).
3. «Оба выключены» = подписка **сохраняется** в KV, оба типа подавляются предпочтениями (без `unsubscribe`). Повторное включение мгновенно, без повторного запроса разрешения.
4. Гранулярность — **на уровне клиента**: настройка применяется ко всем устройствам клиента (`sub:${clientId}:*`).
5. Тайминг напоминаний: существующее «накануне» (cron `0 15 * * *`) сохранить + добавить второе напоминание примерно за 2 часа до визита в день записи. Механику (частота cron, окно выборки, анти-дубли, часовой пояс) финализирует архитектор.
6. Дефолт для существующих и новых подписчиков — оба типа включены (отсутствие сохранённого предпочтения трактуется как «включено»).
7. Пункты «Публикация фото и видео» и «Публикация отзывов» удалить вместе с кнопкой «Запросить удаление фото» (`#photoDeleteBtn`); связанный мёртвый код в `consent.js` оценить и убрать.

Силы, действующие на решение:

- **Стек неизменен.** Vanilla JS, без рантайм-npm-зависимостей; секреты только в Worker; единственный управляемый бэкенд — Worker, деплоится отдельно.
- **Минимальные изменения строго по объёму.** Криптография Web Push, формат VAPID, схема ключей `sub:${clientId}:${endpointHash}` (ADR-006) не пересматриваются.
- **Достоверность (НФТ-3).** Отображаемое состояние тумблера обязано совпадать с фактически применяемым предпочтением доставки.
- **Согласованность с мультидевайс-схемой (НФТ-4).** Предпочтения не должны ломать мультидевайс-доставку и идемпотентность переподписки.
- **Cloudflare free-tier cron.** Допускается несколько cron-триггеров; нет необходимости в очередях/Durable Objects. Частый cron должен быть дешёвым и не плодить дубли.
- **Часовой пояс.** Cron срабатывает в UTC; время визита в YCLIENTS — локальное время студии (Москва, UTC+3). Без корректного смещения «за 2 часа до визита» будет промахиваться на 3 часа.

Решение охватывает связанные вопросы: где и как хранить предпочтения; контракт чтения/записи предпочтений (клиент↔Worker); как `handleSend` и `handleCron` учитывают предпочтения; механика второго напоминания и защита от дублей; изменение cron-расписания; часовой пояс; дефолты/миграция; удаление мёртвого кода согласий; влияние на `CACHE_VERSION`. Точные имена полей, тексты и разметка — в спецификации релиза.

## Рассматриваемые варианты

### Вопрос 1. Где хранить предпочтения клиента

#### Вариант 1A (выбран): отдельный ключ `prefs:${clientId}` в KV `SUBSCRIPTIONS`

- Значение — JSON `{ promo: boolean, remind: boolean, ts: number }`, по одному ключу на клиента.
- Чтение: `handleSend`/`handleCron` берут `prefs:${clientId}` один раз на клиента.
- Запись: клиент шлёт обе булевы при изменении любого тумблера (полное состояние, идемпотентная перезапись).
- **Плюсы:** гранулярность «на уровне клиента» (решение №4) выражается естественно — один ключ на клиента, не зависит от числа устройств; не нужно синхронизировать N подписок при изменении настройки; не трогает значение подписки (схема ADR-006 неизменна); один `get` на клиента в `handleSend`/`handleCron` (broadcast группирует по clientId).
- **Минусы:** в broadcast `handleSend` (`list({prefix:'sub:'})`) ключи идут вперемешку по клиентам — нужен кэш прочитанных `prefs` в пределах одного вызова, чтобы не читать `prefs:${clientId}` повторно для каждого устройства того же клиента.
- **Последствия:** новый префикс ключей `prefs:`; новый эндпоинт `/preferences`; `handleSend`/`handleCron` читают и кэшируют prefs.

#### Вариант 1B (отвергнут): хранить флаги внутри каждой подписки `sub:${clientId}:${endpointHash}`

- **Минусы:** гранулярность по решению №4 — на уровне клиента, а не устройства. Хранение в каждой подписке означало бы при изменении настройки переписать **все** ключи `sub:${clientId}:*` (read-modify-write по N записям, гонки при параллельной переподписке — тот же класс дефекта, что чинил ADR-006). Новые подписки устройств наследовали бы устаревшие флаги. Дороже и хрупче. Отвергнут.

#### Вариант 1C (отвергнут): хранить предпочтения на клиенте (localStorage) и фильтровать на стороне Service Worker при `push`-событии

- **Минусы:** доставка всё равно происходит (Worker шлёт push), SW лишь подавляет показ — это не «не приходит» (НФТ-3 нарушен: push фактически доставлен, расход на отправку, на iOS возможен системный показ). Настройка не переносится между устройствами клиента (решение №4 невыполнимо). Отвергнут.

### Вопрос 2. Контракт чтения/записи предпочтений (клиент↔Worker)

#### Вариант 2A (выбран): новый эндпоинт `POST /preferences` (запись) + `GET /preferences?clientId=` (чтение), без `X-Admin-Secret`

- `POST /preferences` принимает `{ clientId, promo, remind }`, пишет `prefs:${clientId}`. `GET /preferences?clientId=…` возвращает `{ promo, remind }` (с дефолтами при отсутствии ключа).
- Авторизация — как у существующего `/subscribe`: без админ-секрета (клиентский эндпоинт; в значении только две булевы, привязанные к `clientId`; чувствительных данных нет). `/send` остаётся под `X-Admin-Secret`.
- **Плюсы:** явный, отдельный контракт; чтение при открытии профиля даёт достоверное состояние тумблеров (решение №6, НФТ-3) — UI не угадывает, а узнаёт фактическое значение с сервера; не перегружает `/subscribe` (подписка и предпочтения — разные жизненные циклы: подписка при логине, предпочтения по требованию).
- **Минусы:** один новый маршрут в Worker и две клиентские функции (`getPreferences`/`savePreferences`).
- **Последствия:** правка `cf-worker/src/index.js` (роутинг + 2 хелпера), `docs/modules/push.js` (2 функции), `docs/version.json`.

#### Вариант 2B (отвергнут): расширить `/subscribe` полями `promo`/`remind`

- **Минусы:** `subscribePush` вызывается при логине и не должен затирать ранее заданные пользователем предпочтения дефолтами; пришлось бы либо читать-сливать (сложнее), либо рисковать сбросом настроек при каждом входе. Подписка устройства и клиентские предпочтения — разные сущности (устройство vs клиент, решение №4). Смешение контрактов хрупко. Отвергнут.

#### Вариант 2C (отвергнут): только запись, без отдельного чтения; UI восстанавливает из localStorage

- **Минусы:** localStorage привязан к устройству → на втором устройстве клиента тумблеры показали бы не реальное (общее, клиентское) состояние (решение №4, НФТ-3 — рассинхрон). Отвергнут. (localStorage допустим лишь как необязательный кэш для мгновенной отрисовки до ответа `GET`, но источник истины — сервер.)

### Вопрос 3. Семантика «оба выключены» и повторное включение

#### Вариант 3A (выбран): подписка всегда сохраняется; типы подавляются только предпочтениями

- `handleSend` пропускает клиента, если `promo === false`; `handleCron` пропускает, если `remind === false`. Сама подписка в KV не удаляется ни при каких комбинациях тумблеров.
- Повторное включение — обычный `POST /preferences` с `true`; разрешение на уведомления и подписка устройства уже есть → доставка возобновляется немедленно (решение №3).
- **Плюсы:** прямо реализует решение №3; нет повторного запроса разрешения (особенно ценно на iOS, где повторный запрос проблематичен); согласовано с ADR-006 (подписки живут независимо). 
- **Минусы:** нет (отписку выполняет только явный `unsubscribePush`, который в объём этой задачи не входит).
- **Последствия:** тумблеры профиля **не** вызывают `subscribePush`/`unsubscribePush`; они вызывают только `savePreferences`.

#### Вариант 3B (отвергнут): «оба выключены» → `unsubscribe` устройства

- **Минусы:** противоречит решению №3 (повторное включение потребовало бы переподписки и, возможно, повторного разрешения); не соответствует гранулярности «клиент» (unsubscribe — по устройству); сложнее и менее предсказуемо. Отвергнут.

### Вопрос 4. Учёт предпочтений в `handleSend` (broadcast и target)

#### Вариант 4A (выбран): фильтр по `promo` с кэшем prefs в пределах вызова

- В broadcast (`list({prefix:'sub:'})`) и target (`list({prefix:'sub:${targetClientId}:'})`) для каждой подписки определить `clientId` (из значения), один раз прочитать `prefs:${clientId}` (с кэшированием в `Map` на время вызова), и при `promo === false` пропустить отправку на устройства этого клиента.
- Дефолт: отсутствие ключа `prefs` или отсутствие поля `promo` → `true` (решение №6, толерантное чтение в духе ADR-006 Вопрос 6).
- **Плюсы:** один тип push (рассылки) управляется одним флагом; кэш исключает повторные чтения для мультидевайс-клиента; достоверность `sent`/`failed` сохраняется (подавленные не считаются отправленными).
- **Минусы:** дополнительные чтения KV (по одному на уникального клиента в рассылке) — приемлемо при текущей базе.
- **Последствия:** target и broadcast ветки `handleSend` получают предварительную проверку prefs.

#### Вариант 4B (отвергнут): фильтровать только broadcast, target всегда доставлять

- **Минусы:** target-рассылка администратора конкретному клиенту — это тоже «Push-уведомления» в смысле тумблера; игнорировать его при адресной отправке нарушило бы ФТ-3/КП-2. Отвергнут. (Примечание: target-UI вне объёма по ADR-006, но семантика флага должна соблюдаться единообразно.)

### Вопрос 5. Механика второго напоминания и защита от дублей

#### Вариант 5A (выбран): два вида напоминаний (`day`, `soon`), частый cron, окно выборки, анти-дубль через `reminded:${recordId}:${kind}` с TTL, смещение часового пояса студии

- **Расписание cron:** заменить `0 15 * * *` на частый запуск `*/30 * * * *` (каждые 30 минут). На каждом запуске Worker определяет «сегодня» и «завтра» в локальном времени студии и обрабатывает оба вида.
- **Вид `day` (накануне):** сохраняет текущее поведение. Запускается на тех тиках cron, что попадают в локальные ~18:00 студии (эквивалент прежнего `0 15 * * *` = 18:00 МСК), выбирает записи на завтра, шлёт «Завтра в HH:MM…». Анти-дубль исключает повторную отправку при нескольких тиках в этот час.
- **Вид `soon` (≈ за 2 часа до визита):** на каждом запуске выбирает записи на сегодня, время начала которых попадает в окно `[now+90мин, now+150мин)` (целевое смещение 2 часа, ширина окна 60 минут покрывает дискретность cron 30 минут), шлёт «Сегодня в HH:MM…».
- **Защита от дублей:** перед отправкой проверять ключ `reminded:${recordId}:${kind}` в KV `SUBSCRIPTIONS`; если есть — пропустить; после успешной отправки записать его с `expirationTtl` (например 2 суток), чтобы при частом cron один и тот же `(record, kind)` не отправлялся повторно. Маркер ставится **на запись+вид** (один на всю запись), а не на устройство — клиент с несколькими устройствами получает напоминание на все устройства один раз за вид (согласовано с ADR-006 Вопрос 3).
- **Часовой пояс:** ввести константу смещения студии (Москва, UTC+3) и считать «локальное сейчас» = `now(UTC) + 3ч`. Дата для выборки YCLIENTS и сравнение времени визита (`rec.date`, локальное) — в локальном времени студии. Окно `soon` сравнивается в локальном времени.
- **Плюсы:** реализует решение №5 (накануне + за ~2 часа) без очередей/Durable Objects; анти-дубль устойчив к частому cron и к нескольким тикам в окне; смещение TZ устраняет промах на 3 часа; маркер на запись (а не устройство) корректен для мультидевайса; TTL самоочищает маркеры.
- **Минусы:** cron каждые 30 минут = больше запусков (48/сутки) и больше чтений KV — в пределах free-tier и текущей базы приемлемо; небольшая погрешность срабатывания (±30 мин) допустима по решению №5 («примерно за 2 часа»).
- **Последствия:** `handleCron` переписывается: цикл по двум видам, вычисление локального времени, окна, проверка/запись маркеров; `wrangler.toml` cron → `*/30 * * * *`.

#### Вариант 5B (отвергнут): дублировать cron-триггеры (один для `day`, отдельный для `soon`) без маркеров, рассчитывая на точность времени

- **Минусы:** Cloudflare не гарантирует минутную точность запуска scheduled; без маркеров повторный тик в окне или дрожание расписания → повторные напоминания (спам). Несколько cron-выражений усложняют логику маршрутизации внутри одного `scheduled`. Анти-дубль всё равно необходим. Отвергнут в пользу единого частого cron + маркеры (5A).

#### Вариант 5C (отвергнут): хранить «последнее отправленное напоминание» внутри подписки/в отдельной структуре по клиенту

- **Минусы:** напоминание привязано к **записи** (`recordId`), а не к клиенту; у клиента может быть несколько записей в день. Маркер по клиенту слипал бы разные записи. Маркер по `recordId` (5A) точен. Отвергнут.

### Вопрос 6. Дефолты и миграция

#### Вариант 6A (выбран): дефолт «включено», миграцию не делать, толерантное чтение

- Отсутствие ключа `prefs:${clientId}` или отсутствие конкретного поля → трактуется как `true` (решение №6). Существующим подписчикам ничего создавать не нужно — доставка не отключается.
- Согласовано с ADR-006 Вопрос 6 (толерантное чтение, ноль кода миграции, естественная конвергенция).
- **Плюсы:** ноль кода миграции; нулевой риск случайно отключить доставку действующим клиентам; запись `prefs` появляется только когда пользователь впервые меняет настройку.
- **Минусы:** до первого открытия профиля у клиента нет записи `prefs` — это и есть «оба включено», корректно.
- **Последствия:** хелпер чтения prefs возвращает `{promo:true, remind:true}` при отсутствии ключа/полей.

#### Вариант 6B (отвергнут): при деплое создать `prefs` для всех существующих подписчиков

- **Минусы:** лишний одноразовый проход по KV (как ADR-006 Вопрос 6B), риск ошибок, не нужен при дефолте «включено». Отвергнут.

### Вопрос 7. Удаление пунктов «фото/видео» и «отзывы» и мёртвого кода

#### Вариант 7A (выбран): убрать строки из настроек профиля + удалить ставший мёртвым код согласий

- Из `prototype.html` (раздел «Настройки уведомлений») удалить строку `#photoConsentRow` (фото/видео), строку «Публикация отзывов» и блок `#photoDeleteBtn`.
- В `consent.js` после удаления становятся мёртвыми: `CONSENT_INFO.photo` и `CONSENT_INFO.reviews`, ветка `type === 'photo'` в `consentToggle`/`confirmConsentOff`, функции `showDeleteRequest`/`confirmDeleteRequest` и хелпер `closeDeleteModal` (модалка `#deleteModal`/`#deleteToast` удаления материалов). Их удалить, как и связанную разметку модалки удаления в `prototype.html`, если она больше нигде не используется.
- `toggleConsent` (onboarding `s-consent`) и `acceptConsent` — **не трогать** (решение №1, onboarding вне объёма).
- **Плюсы:** чистое удаление по объёму, без осиротевшего кода; меньше путаницы.
- **Минусы:** требует аккуратной проверки, что удаляемые функции не вызываются из onboarding и других экранов (проверяется grep'ом перед удалением).
- **Последствия:** правки `prototype.html` и `consent.js`; обновить `docs/reference.html`, где перечислены публичные функции `consent.js`, если оно служит документацией (косметически).

#### Вариант 7B (отвергнут): только скрыть строки в UI, код согласий оставить

- **Минусы:** оставляет мёртвый код (BA решение №7 явно требует оценить и убрать мёртвый код); противоречит принципу минимальных, но чистых изменений. Отвергнут.

### Вопрос 8. Влияние на `CACHE_VERSION` и версию

#### Вариант 8A (выбран): поднять `CACHE_VERSION` v40 → v41; состав `APP_SHELL` не меняется

- Изменяются клиентские файлы `docs/modules/push.js`, `docs/modules/consent.js`, `docs/modules/profile.js`, `docs/prototype.html` — все уже входят в `APP_SHELL`. По правилу проекта при изменении файла в `APP_SHELL` поднимается `CACHE_VERSION`.
- Новых файлов нет → состав `APP_SHELL` не меняется. Поднимается только номер: `v40` → `v41`.
- `version.json`: `build` 8 → 9, `version` → `v1.6.0`.
- **Плюсы:** соблюдено правило проекта; установленные клиенты получают новый UI и логику предпочтений.
- **Минусы:** нет.
- **Последствия:** правка `docs/sw.js` (только номер) и `docs/version.json`.

#### Вариант 8B (отвергнут): не поднимать `CACHE_VERSION`

- **Минусы:** установленные клиенты остались бы со старым косметическим UI и без `getPreferences`/`savePreferences` → рассинхрон с новым Worker и нерабочие тумблеры. Нарушает правило проекта. Отвергнут.

## Решение

- **Вопрос 1 — 1A:** предпочтения в отдельном ключе `prefs:${clientId}` (`{promo, remind, ts}`), per-client; не в подписке.
- **Вопрос 2 — 2A:** новый эндпоинт `POST /preferences` (запись) + `GET /preferences?clientId=` (чтение), без `X-Admin-Secret`.
- **Вопрос 3 — 3A:** подписка всегда сохраняется; типы подавляются только предпочтениями; повторное включение мгновенно (тумблеры не трогают подписку).
- **Вопрос 4 — 4A:** `handleSend` (broadcast и target) фильтрует по `promo` с кэшем prefs в пределах вызова.
- **Вопрос 5 — 5A:** два вида напоминаний (`day`, `soon`), cron `*/30 * * * *`, окно `soon` `[now+90мин, now+150мин)` в локальном времени студии (UTC+3), анти-дубль `reminded:${recordId}:${kind}` с TTL; `handleCron` фильтрует по `remind`.
- **Вопрос 6 — 6A:** дефолт «оба включены»; миграции нет; толерантное чтение prefs.
- **Вопрос 7 — 7A:** убрать строки «фото/видео», «отзывы», `#photoDeleteBtn` из профиля; удалить ставший мёртвым код согласий в `consent.js`; onboarding `s-consent` не трогать.
- **Вопрос 8 — 8A:** `CACHE_VERSION` v40 → v41; `APP_SHELL` без изменений; `version.json` → `v1.6.0`.

Отвергнуты: 1B (гонки при N подписках), 1C (push всё равно доставляется, не переносится между устройствами), 2B (затирание настроек при логине), 2C (localStorage не клиентский, рассинхрон), 3B (повторный запрос разрешения), 4B (target обходит флаг рассылок), 5B (спам без маркеров), 5C (маркер по клиенту слипает записи), 6B (лишняя миграция), 7B (мёртвый код), 8B (рассинхрон клиента и Worker).

### Границы решения

- Изменяется **только**: хранение/чтение/применение клиентских предпочтений push и механика второго напоминания. Затрагиваются: `cf-worker/src/index.js` (новые `getPrefs`/`putPrefs`-хелперы и эндпоинт `/preferences`; фильтрация в `handleSend`; переработка `handleCron` под два вида + анти-дубль + TZ), `cf-worker/wrangler.toml` (cron), `docs/modules/push.js` (`getPreferences`/`savePreferences`), `docs/modules/consent.js` (рабочая логика тумблеров + удаление мёртвого кода), `docs/modules/profile.js` (загрузка состояния тумблеров при отрисовке профиля; учёт недоступности push для ФТ-8), `docs/prototype.html` (разметка двух тумблеров, удаление трёх элементов), `docs/sw.js` (`CACHE_VERSION`), `docs/version.json`.
- **Не затрагиваются (вне объёма):** схема ключей подписок `sub:${clientId}:${endpointHash}` и её значение (ADR-006); криптография Web Push (`encryptPayload`/`vapidToken`/`sendOne`); формат VAPID; `subscribePush`/`unsubscribePush` (контракт не меняется — тумблеры их не вызывают); onboarding-экран `s-consent` (`toggleConsent`, `acceptConsent`); сегменты аудитории, серверное планирование, история рассылок, target-UI (объём BA-006, отложено); `/review`-проксирование; манифест, иконки, состав `APP_SHELL`; сценарий записи, отзывы; захардкоженный YCLIENTS-токен (известный долг — не усугублять).
- **Контракт `/subscribe` и `/unsubscribe` не меняется.** Новый контракт — только `/preferences`.

## Последствия

- **Положительные:**
  - Два тумблера в профиле становятся фактически рабочими и независимыми (ФТ-1, ФТ-3, ФТ-4, ФТ-5): «Push-уведомления» управляет `handleSend`, «Напоминания о визитах» — `handleCron`.
  - Состояние per-client, читается с сервера при открытии профиля → достоверно и переносится между устройствами клиента (решение №4, ФТ-6, НФТ-3, НФТ-4).
  - «Оба выключены» не рвёт подписку; повторное включение мгновенно, без повторного запроса разрешения (решение №3).
  - Дефолт «включено» не отключает доставку действующим клиентам (решение №6); ноль кода миграции.
  - Второе напоминание «за ~2 часа» работает автоматически по cron, без участия администратора (ФТ-7, НФТ-5); анти-дубль исключает спам; корректное локальное время студии устраняет промах TZ.
  - Без новых рантайм-зависимостей (НФТ-1); секреты остаются в Worker, `/send` под `X-Admin-Secret` (НФТ-2).
  - Удалён мёртвый код согласий «фото/отзывы» (решение №7).
- **Отрицательные / компромиссы:**
  - Cron учащается до каждых 30 минут (48 запусков/сутки) — больше чтений KV; приемлемо в free-tier при текущей базе. При значительном росте базы — пересмотреть частоту/окно.
  - Погрешность времени `soon`-напоминания ±~30 минут (дискретность cron) — допустимо по решению №5 («примерно за 2 часа»).
  - Смещение TZ зашито константой UTC+3; при переходах/смене региона студии потребуется правка (для РФ переходов на летнее время нет — риск низкий).
  - `handleSend`/`handleCron` делают дополнительные чтения KV (prefs, маркеры) — оправдано требованием.
- **Технический долг:**
  - TZ-смещение константой, а не из конфигурации/IANA — упрощение; вынести в env при необходимости.
  - Сегменты аудитории, серверное планирование, история, target-UI — остаются отложенными (BA-006), эта задача их не вводит.
  - Захардкоженный YCLIENTS-токен — известный долг, не трогается.

## Влияние на код

- **`cf-worker/src/index.js`:**
  - Хелперы `getPrefs(env, clientId)` → `{promo, remind}` с дефолтами `true` (толерантное чтение); `putPrefs(env, clientId, {promo, remind})` → запись `prefs:${clientId}`.
  - Маршрут `/preferences`: `POST` (запись `{clientId, promo, remind}`), `GET` с `?clientId=` (чтение). Без `X-Admin-Secret`.
  - `handleSend` — перед отправкой на устройство получить `clientId` из значения подписки, прочитать prefs (кэш `Map` на время вызова) и пропустить, если `promo === false`. Логирование/410-очистка/счётчики — как в ADR-006.
  - `handleCron` — частый запуск; вычислить локальное «сейчас» студии (UTC+3); обработать вид `day` (записи на завтра, отправка в локальный ~18:00-час) и вид `soon` (записи на сегодня в окне `[now+90мин, now+150мин)`); для каждого `(record, kind)` проверить/поставить маркер `reminded:${recordId}:${kind}` (TTL ~2 суток); для каждого клиента прочитать prefs и слать только при `remind !== false`; рассылка на все подписки клиента (`list({prefix:'sub:${clientId}:'})`, ADR-006); тела push — на русском (`Завтра в HH:MM…` / `Сегодня в HH:MM…`).
  - `scheduled` остаётся `ctx.waitUntil(handleCron(env))`.
- **`cf-worker/wrangler.toml`:** `crons = ["*/30 * * * *"]` вместо `["0 15 * * *"]`.
- **`docs/modules/push.js`:** `getPreferences(clientId)` (GET) и `savePreferences(clientId, {promo, remind})` (POST); экспорт в `window` рядом с существующими. `subscribePush`/`unsubscribePush` — без изменений.
- **`docs/modules/consent.js`:** новая рабочая логика тумблеров профиля (переключают визуальное состояние и вызывают `savePreferences`, без подтверждающей модалки для включения push-типов — формулировку UX уточнить в спецификации); удаление мёртвого кода `photo`/`reviews` (см. Вопрос 7A). `toggleConsent`/`acceptConsent` сохранить.
- **`docs/modules/profile.js`:** при отрисовке профиля вызвать `getPreferences` и выставить состояние двух тумблеров; если push неприменим (нет `PushManager`/разрешения) — отразить это в UI (ФТ-8) без ложного «работает».
- **`docs/prototype.html`:** в «Настройках уведомлений» оставить ровно два тумблера («Push-уведомления», «Напоминания о визитах»); удалить `#photoConsentRow`, строку «Публикация отзывов», блок `#photoDeleteBtn`; убрать осиротевшую разметку модалки удаления материалов (`#deleteModal`/`#deleteToast`), если не используется. Onboarding `s-consent` не трогать.
- **`docs/sw.js`:** `CACHE_VERSION` `v40` → `v41`. Состав `APP_SHELL` без изменений.
- **`docs/version.json`:** `build` 8 → 9, `version` → `v1.6.0` (мейнтейнер обновляет `date`/`time`/`sha` при релизе).
- **Деплой:** Worker — отдельно (`cd cf-worker && npx wrangler deploy`); PWA — `git push origin main`. Выкатывать согласованно (новый эндпоинт `/preferences` нужен новому клиенту).
- **Новые рантайм-зависимости:** нет.
- **Что ломается:** ничего во внешних контрактах `/subscribe`/`/unsubscribe`/`/send`/`/review`. Добавляется `/preferences`. Меняется cron-расписание (Worker деплоится отдельно). Старый кэшированный клиент до обновления SW (`v41`) показывает прежний косметический UI — устраняется подъёмом `CACHE_VERSION`.
