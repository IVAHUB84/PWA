# Требование 026: Убрать дубль в таб-баре, добавить вкладку «Записи»

## Описание
В нижнем таб-баре клиентского приложения две разные кнопки ведут на один и тот же экран каталога услуг (`s-services`):
- центральный приподнятый FAB «Запись»;
- пятая вкладка «Услуги».

Это сбивает: пользователь видит два разных affordance → один результат. Нужно устранить дубль и отдать освободившийся слот под то, что реально нужно постоянным клиентам, — **их собственные записи** (предстоящие + прошлые визиты), которые сейчас закопаны в Профиль → «История посещений».

## Целевое поведение (согласовано с заказчиком)
- Центральный FAB «Запись» остаётся **единственной** точкой входа в сценарий записи (ведёт на `s-services`).
- Пятая вкладка «Услуги» **убирается** (каталог услуг открывается через FAB).
- На её место добавляется вкладка **«Записи»** — список записей клиента (предстоящие и прошлые визиты).
- Итоговый порядок вкладок (слева направо):

```
Сейчас:  Главная · Лента · [＋ Запись] · Профиль · Услуги
                                            └──────────┴─ обе → s-services (дубль)

Стало:   Главная · Лента · [＋ Запись] · Записи · Профиль
```

То есть: «Услуги» удаляется; «Записи» встаёт на 4-ю позицию; «Профиль» сдвигается на 5-ю (крайнюю справа — как принято в iOS).

- Название вкладки — **«Записи»** (утверждено заказчиком, не «Визиты»).
- При нахождении на экране записей вкладка «Записи» должна подсвечиваться как активная (`.tab.active`).

## Текущее состояние (по коду)
- Таб-бар сверстан **инлайн и продублирован в каждом клиентском экране** с навигацией: `s-home` (prototype.html:92–98), `s-services` (127–133), `s-masters` (154–160), `s-slots` (206–212), `s-history` (246–252), `s-upcoming` (266–272), `s-profile` (366–372), `s-feed` (660–666), `s-post` (674–680). Любое изменение состава вкладок нужно вносить **согласованно во все эти копии**.
- Пятая вкладка везде: `<div class="tab" onclick="go('s-services','tab')">` с иконкой-портфелем (briefcase SVG) и подписью «Услуги».
- Центральный FAB везде: `onclick="hapticTap('fab');go('s-services','tab')"` с подписью «Запись» — совпадает по назначению с «Услуги».
- Экраны записей клиента уже существуют:
  - `s-history` — рендерит секцию «Предстоящие · N» (с «Перенести»/«Отменить») и «Прошлые визиты» (с «Повторить»/«Оценить») + фильтр по периоду (`modules/history.js`). Открывается сейчас из Профиль → строка «История посещений» (prototype.html:326).
  - `s-upcoming` — «только предстоящие визиты» (добавлен в req-010, prototype.html:256).
- Админский таб-бар (`s-admin*`, prototype.html:698+) — **отдельный**, требования не касается.

## Открытые вопросы (для BA/архитектора)
- **ОВ-1. Какой экран открывает вкладка «Записи».** Кандидаты: (а) `s-history` — полный список «предстоящие + прошлые», уже готов; (б) `s-upcoming` — только будущие. Логичнее (а): вкладка верхнего уровня «Записи» должна показывать все записи клиента, а не только будущие. Решает архитектор.
- **ОВ-2. Заголовок экрана.** Если открываем `s-history`, его текущий заголовок — «История». Вкладка «Записи» → экран «История» — рассинхрон. Нужно ли переименовать заголовок `s-history` в «Записи» (или «Мои записи»), и не сломает ли это вход из Профиля. База: согласовать заголовок с названием вкладки.
- **ОВ-3. Дублирование входа из Профиля.** После выноса записей в таб-бар строка Профиль → «История посещений» (prototype.html:326) становится вторым входом в тот же экран. Оставить (доп. путь — не вредит) или убрать? База: оставить, как «Все →» на главной уже дублирует.
- **ОВ-4. Иконка вкладки.** Нужна иконка для «Записи» вместо портфеля «Услуги». Кандидат — календарь/чек-лист (в проекте уже есть SVG календаря в `histPeriodBtn`, prototype.html:237, и clipboard у текущей «Услуги»). Выбор иконки — за архитектором/дизайном, в едином stroke-стиле остальных вкладок (24×24, stroke-width 1.8, `currentColor`).
- **ОВ-5. Активное состояние на `s-upcoming`.** Если «Записи» ведёт на `s-history`, а пользователь проваливается в `s-upcoming` (через «Все →» с главной) — какую вкладку считать активной там. База: «Записи».

## Ограничения
- Vanilla HTML/CSS/JS, без новых рантайм-зависимостей; код в `docs/`.
- Изменения согласованно во **всех** инлайн-копиях таб-бара (см. список выше) — не оставить рассинхрон состава/порядка вкладок между экранами.
- FAB и его haptic (`hapticTap('fab')`) не трогать — он остаётся точкой записи.
- Не ломать существующие экраны `s-history` / `s-upcoming` и действия Перенести/Отменить/Повторить/Оценить.
- Каталог услуг должен оставаться доступным (через FAB, кнопку «Записать себя» на главной, блок мастеров) — не потерять доступ к `s-services`.
- При изменении файлов из `APP_SHELL`/`sw.js` — поднять `CACHE_VERSION` в `sw.js`.
- Минимальные изменения по объёму — только состав/порядок вкладок и навигация, без редизайна экранов.

## Критерий готовности
- В нижнем таб-баре нет двух кнопок, ведущих на `s-services`: FAB «Запись» — единственный вход в каталог/запись.
- Пятая вкладка «Услуги» отсутствует; вместо неё вкладка **«Записи»** на 4-й позиции, «Профиль» — 5-й (крайний справа).
- Тап по «Записи» открывает экран со списком записей клиента (предстоящие + прошлые); вкладка подсвечивается активной.
- Состав и порядок вкладок одинаковы на всех клиентских экранах с таб-баром; админский таб-бар не затронут.
- Каталог услуг по-прежнему открывается (FAB / «Записать себя» / мастера).
- В консоли нет ошибок; PWA остаётся устанавливаемой; при изменении `APP_SHELL`/`sw.js` поднята `CACHE_VERSION`.
