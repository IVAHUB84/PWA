# Требование 027: Контакты студии на экране подтверждения и в профиле

## Описание
В приложении нигде нет контактов студии: ни адреса, ни телефона, ни часов работы, ни карты. После записи (`s-confirm`) клиент не знает, куда идти и как доехать; в профиле тоже нет блока «О студии / Контакты». Для офлайн-студии красоты это базовая дыра — особенно для нового клиента, который только что записался и впервые поедет в студию.

Нужно показать контактный блок студии в двух местах:
1. На экране подтверждения записи `s-confirm` — сразу после успешной брони.
2. В профиле `s-profile` — как постоянно доступный блок «О студии».

## Целевое поведение (согласовано с заказчиком)
Контактный блок содержит:
- Название студии.
- Адрес (текстом).
- Кнопка «Построить маршрут» / открыть карту (Яндекс.Карты и/или 2ГИС — ссылки уже есть в проекте).
- Телефон с возможностью позвонить в один тап (`tel:`).
- Часы работы.

Поведение:
- Тап по телефону → `tel:` (системный звонок).
- Тап по адресу/кнопке карты → открыть карты по координатам/ссылке в новой вкладке.
- На `s-confirm` блок — компактный, под карточкой записи, не мешает кнопкам «На главную» и «Добавить услугу к записи».
- В профиле — отдельная секция (в стиле существующих `settings-group`).

## Источник данных (РЕШЕНО с заказчиком — гибрид)
Выбран источник YCLIENTS `/company/{id}`, **но проверка показала, что для этой компании все контактные поля пусты, а partner-токен не имеет прав на запись настроек компании** (PUT `/company/{id}` → `success:false`, «Недопустимый формат запроса»; данные не записываются). Поэтому:

- Код **читает `/company/{YC.company}`** и использует поля `title`, `phone`/`phones`, `address`, `coordinate_lat`, `coordinate_lon`, `schedule`, **если они непустые**.
- Недостающие/пустые поля берутся из хардкод-объекта `STUDIO` (плейсхолдеры) — единый источник дефолтов.
- Когда заказчик заполнит поля в админ-панели YCLIENTS, приложение **автоматически** подхватит живые данные без правки кода (API имеет приоритет над плейсхолдером).

- `modules/constants.js` уже содержит:
  - `REVIEW_URLS['2gis']` — ссылка на 2ГИС с firm id `70000001042765322`.
  - `REVIEW_URLS.yandex` — ссылка на Я.Карты с адресом Троицка и координатами `ll=37.289422,55.466536`.
- `modules/api.js` → `YC` (get/post, company=1940850). Нет функции загрузки компании.
- `app.js` → `initApp()` грузит категории/услуги/мастеров/галерею одним `Promise.all` (app.js:166–171) — естественное место добавить запрос `/company`.
- Экран подтверждения `s-confirm` — `prototype.html:216–228`: `.confirm-content` → `.confirm-icon`/`.confirm-title`/`.confirm-sub` → `.confirm-card #confirmRows` → кнопки «На главную» и «Добавить услугу к записи». `#confirmRows` наполняется в `modules/slots.js` → `updateConfirmScreen()`.
- Экран профиля `s-profile` — `prototype.html:296+`: карточка профиля (имя/телефон/email + `#loyaltyBlock`), далее секции (тема, уведомления, избранный мастер). Рендер — `modules/profile.js` → `renderProfileScreen()`.
- v1.20.0 глобально отключил выделение текста (`user-select:none`) — телефон/адрес должны быть **выделяемыми/кликабельными** (исключение для контактного блока), иначе клиент не сможет скопировать.
- Запись на услуги, цены, лояльность не затрагиваются.

## Открытые вопросы (для BA/архитектора)
- **ОВ-1. Где держать `STUDIO`-дефолты.** Кандидаты: (а) новый объект `STUDIO` в `modules/constants.js` рядом с `REVIEW_URLS`; (б) отдельный `studio.json`. База: (а) — консистентно с уже зашитыми там URL карт, без лишнего сетевого запроса. Решает архитектор.
- **ОВ-2. Когда грузить `/company`.** Кандидаты: (а) в общем `Promise.all` внутри `initApp()` (app.js:166), результат в состояние/кэш; (б) лениво при заходе на `s-confirm`/`s-profile`. База: (а) один раз на старте, кэш в памяти + опционально `localStorage` для оффлайна.
- **ОВ-3. Рендер блока — общий компонент или две копии.** Блок нужен на двух экранах. База: одна функция-рендер контактного блока, переиспользуемая в `updateConfirmScreen()` и `renderProfileScreen()`, чтобы не плодить дубли. Решает архитектор.
- **ОВ-4. Какие карты показывать.** Я.Карты, 2ГИС или обе кнопки. База: кнопка «Маршрут» → Я.Карты (есть координаты), плюс ссылка 2ГИС опционально. Решает архитектор/дизайн.
- **ОВ-5. Часы работы из API.** Поле `schedule` в `/company` — строка свободного формата и сейчас пустое. При заполнении в админке формат может отличаться от плейсхолдера. База: показывать `schedule` как есть, если непустой; иначе плейсхолдер.
- **ОВ-6. Поведение при оффлайне / пустом ответе API.** Если `/company` недоступен — показывать плейсхолдеры из `STUDIO`. Блок не должен исчезать или ломать экран.

## Ограничения
- Vanilla HTML/CSS/JS, без новых рантайм-зависимостей; код в `docs/`.
- Не хардкодить секреты; токен YCLIENTS уже в проекте — не усугублять, новые ключи не вводить.
- Контактный блок — выделяемый/кликабельный текст (локальное исключение из глобального `user-select:none`).
- `tel:` и ссылки на карты — `rel="noopener"` для внешних ссылок, экранирование пользовательских/серверных строк (`esc`).
- Не ломать существующие кнопки и поток на `s-confirm` (брони/перенос/кросс-селл) и секции профиля.
- API-данные имеют приоритет над плейсхолдером; код не должен требовать ручной правки при заполнении админки YCLIENTS.
- При изменении файлов из `APP_SHELL` / `sw.js` — поднять `CACHE_VERSION` в `sw.js`.
- Минимальные изменения по объёму — только добавление контактного блока и загрузки данных компании.

## Критерий готовности
- На `s-confirm` под карточкой записи виден блок контактов: название, адрес, кнопка карты/маршрута, кликабельный телефон (`tel:`), часы работы.
- В профиле `s-profile` есть секция «О студии» с теми же контактами.
- Телефон в контактном блоке можно нажать (звонок) и выделить/скопировать; адрес/кнопка открывает карту по координатам.
- Если `/company` отдаёт непустые поля — показываются они; иначе плейсхолдеры из `STUDIO`. Заполнение полей в админке YCLIENTS подхватывается без правки кода.
- При недоступности API блок показывает плейсхолдеры, экран не ломается.
- Телефон по умолчанию помечен в коде как плейсхолдер (комментарий «заменить на реальный»).
- В консоли нет ошибок; PWA остаётся устанавливаемой; при изменении `APP_SHELL`/`sw.js` поднята `CACHE_VERSION`.
