# Требование 017: Исправление карусели фото в детальной карточке услуги

## Описание
Карусель фотографий в детальной карточке услуги (`s-service`), реализованная в рамках требования 015 / релиза v1.12.0 (ADR-014), **работает некорректно**. Нужно устранить дефекты так, чтобы карусель листалась плавно, слайды были выровнены и обрезаны корректно, а зацикливание не давало визуального рывка.

Подтверждённые пользователем симптомы (на живом устройстве):
1. **Слайды смещены / обрезаны** — фото выровнены неровно, «подглядывание» (peek) соседнего слайда неравномерно по краям, либо текущий слайд обрезан не по центру.
2. **Свайп залипает / не листает** — жест свайпа не перелистывает фото, требует нескольких попыток или застревает между слайдами.
3. **Рывок / перескок при зацикливании** — на стыке «последний → первый» (и обратно) виден скачок/мигание из-за сброса позиции на клон-слайд.

## Текущее состояние (подтверждено кодом)
- Карусель — собственная реализация на `transform: translateX`, **без нативного scroll-snap** (`docs/modules/services.js:167-308`).
- **Разметка** (`services.js:176-181`): трек `#svcTrack` содержит `cloneLast + realSlides + cloneFirst` — клоны крайних слайдов для бесшовного цикла; индикаторы-точки `#svcDots`.
- **Геометрия слайда** (`docs/style.css:822-841`): `.svc-carousel { overflow:hidden }`, `.svc-carousel-track { display:flex; gap:8px; transition: transform .35s ease }`, `.svc-carousel-slide { flex:0 0 calc(100% - 44px); border-radius:16px }`, картинка `height:280px; object-fit:cover`.
- **Расчёт позиции** (`services.js:221-239`): `slideWidth()` читает `track.firstElementChild.offsetWidth`; `translateForIdx(idx) = -((idx+1)*(sw+GAP)) + peek`, где `peek = (carousel.offsetWidth - sw)/2`, `SLIDE_GAP = 8` (комментарий: «must match gap of .svc-carousel-track»), а `44` в фолбэке `slideWidth()` обязан совпадать с `calc(100% - 44px)` в CSS — две «магические» связки CSS↔JS, легко рассинхронизируются.
- **Логика листания** (`services.js:246-280`): `goTo(newIdx)` ставит `transitioning = true`, при входе `if (transitioning) return;` (свайпы во время анимации игнорируются ~350мс + сброс). Завершение — по `transitionend` или сторожевому таймеру 500мс (`transitionGuard`), затем `settle()` переставляет `logIdx` на реальный слайд через `setPos(idx, false)` (мгновенно, без анимации).
- **Жесты** (`services.js:282-306`): `touchstart/move/end`, все `{ passive:true }` (нет `preventDefault`); порог срабатывания `|dx| < 30 || |dx| < |dy|` → игнор; направление → `goTo(logIdx ± 1)`.
- **Начальная установка** (`services.js:259-260`): `setPos(0, false)` + `updateDots()` вызываются синхронно внутри `openServiceCard`, **до** `go('s-service')` (`services.js:310`) — т.е. возможно до того, как экран стал видимым/получил финальные размеры.
- Поведение для **1 фото** (`services.js:182-187`) и **0 фото** (плейсхолдер, `services.js:188-192`) — отдельные ветки, карусель не строится. Это поведение сохранить.

## Открытые вопросы (для BA/архитектора)
- **Корневая причина смещения/обрезки.** Проверить гипотезы: (а) `slideWidth()`/`carousel.offsetWidth` измеряются до того, как `#s-service` стал активным/видимым (измерение в скрытом/трансформированном экране → 0 или неверная ширина → неверный `translateX`); (б) рассинхрон «магических» констант `44px` (CSS) и `SLIDE_GAP=8` с фактической геометрией; (в) отсутствие пересчёта позиции при ресайзе/повороте экрана. Нужен ли пересчёт `setPos` в `onEnter`/после анимации перехода и на `resize`/`orientationchange`.
- **Залипание свайпа.** Это следствие игнора жестов во время `transitioning` (нельзя «доброшенными» свайпами листать подряд), порога `|dx|<30`, отсутствия `preventDefault` (конкуренция с вертикальным скроллом страницы), или незавершения `transitionend` (тогда срабатывает только 500мс-сторож)? Допустимо ли разрешать прерывание/постановку в очередь следующего свайпа.
- **Рывок при зацикливании.** Сейчас `settle()` делает мгновенный `setPos(false)` на клон-границе по `transitionend`. Возможные причины видимого скачка: гонка `transitionend` vs сторож 500мс; reflow `track.getBoundingClientRect()` (`services.js:236`) для сброса transition недостаточен/срабатывает не вовремя. Уточнить целевое поведение: строго бесшовно в обе стороны.
- **Подход к фиксу.** Минимальная правка существующей реализации (предпочтительно — это багфикс, не редизайн) либо переработка геометрии (например, измерять контейнер, а не клон; убрать «магические» 44px в пользу вычисляемого peek). Без рантайм-зависимостей.
- **Объём аффордансы.** Менять ли визуально peek/точки или только чинить механику — по умолчанию **только механику** (внешний вид из v1.12.0 сохранить), подтвердить.

## Ограничения
- Vanilla JS + HTML + CSS, без рантайм-зависимостей; правки только в `docs/modules/services.js` и `docs/style.css` (при необходимости).
- Доработка строго в пределах детальной карточки услуги (`s-service`) и её карусели. Не трогать каталог услуг, выбор мастера, дальнейший поток записи.
- Сохранить поведение для 1 фото (одиночное изображение) и 0 фото (категорийный плейсхолдер).
- Экранирование URL/данных через `esc()` сохранить.
- Корректная работа на тач (свайп) и без ошибок в консоли; производительность при свайпе (без тяжёлых перерисовок).
- Если затрагивается `APP_SHELL`/`sw.js` — поднять `CACHE_VERSION` (для правок только в `services.js`/`style.css` это касается версий модулей в `APP_SHELL`, проверить).
- Минимальные изменения строго по объёму бага.

## Критерий готовности
В детальной карточке услуги с 2+ фото карусель листается свайпом надёжно (без залипания и пропусков жестов), слайды выровнены и обрезаны по центру с равномерным peek по обоим краям, индикатор-точка соответствует текущему фото, а на стыке «последний → первый» и «первый → последний» переход бесшовный — без видимого рывка/мигания. Поведение корректно после открытия карточки сразу (без необходимости «пошевелить» экран) и при повороте/ресайзе. Для 1 фото — одиночное изображение, без фото — плейсхолдер (как раньше). Ошибок в консоли нет; каталог, выбор мастера и поток записи не затронуты.
