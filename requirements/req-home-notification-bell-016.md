# Требование 016: Колокольчик уведомлений на главном экране (инбокс, очистка истории, счётчик непрочитанных)

## Описание
На главном экране (`s-home`), в шапке **сверху справа**, добавить иконку-колокольчик. По нажатию пользователь попадает в экран («инбокс») со списком **всех уведомлений**, которые ему приходили. На этом экране должна быть возможность **очистить историю уведомлений**. Если есть непрочитанные уведомления, на колокольчике **справа внизу** показывается **бейдж со счётчиком** количества непрочитанных. После просмотра уведомлений счётчик должен обнуляться (уведомления становятся прочитанными).

## Текущее состояние (подтверждено кодом)
- **Шапка главного экрана** (`docs/prototype.html:35-53`): градиентный хедер, строка с логотипом и названием студии (`prototype.html:38-44`). Правый верхний угол этой строки **свободен** — естественное место для колокольчика. Ниже — строка поиска (`#homeSearchBar`, `prototype.html:46-52`).
- **Push-уведомления приходят, но НИГДЕ не сохраняются.** SW-обработчик `push` (`docs/sw.js:158-170`) только вызывает `showNotification(title, opts)` (показ системного уведомления ОС) и **не персистит** содержимое. Обработчик `notificationclick` (`docs/sw.js:172-182`) при тапе по системному уведомлению лишь фокусирует окно / открывает приложение — никакой записи в историю.
- **Нет хранилища уведомлений.** IndexedDB в проекте не используется (грепом не найдено). Состояние хранится в `localStorage` (`docs/modules/storage.js`, `state.js`), но `localStorage` **недоступен из Service Worker**, а push-события прилетают именно в SW и в том числе когда приложение закрыто.
- **Внутриприложенческий тост уже есть, но это не инбокс.** Элемент `#pushNotif` (`prototype.html:25`) с обработчиком `handlePushTap()` (`docs/modules/notifications.js:11-14`) при тапе делает `go('s-history','tab')`.
- **`s-history` — это «История записей», а не инбокс уведомлений** (`prototype.html:227-235`, заголовок «История», фильтр по периоду визитов). То есть сейчас тап по уведомлению ведёт в историю визитов как временное решение; отдельного экрана истории уведомлений **нет**.
- **Источники уведомлений (что должно попадать в инбокс):**
  - Ручные push от администратора — `sendAdminPush(title, body, targetClientId)` (`docs/modules/push.js:114-129`) через Cloudflare Worker (`/send`).
  - Возможные напоминания/промо — у пользователя есть тумблеры предпочтений `promo` / `remind` (`push.js:89-112`, требование 007 `req-profile-notif-toggles-007.md`).
  - Payload push: `{ title, body, icon? }` (см. `sw.js:159-167`). На клиенте `_sendNotification(title, body)` (`notifications.js:16-35`) показывает локальное уведомление.
- **Подписка/права:** подписка на push идёт через `subscribePush` (`push.js:31-67`); часть пользователей могут быть не подписаны или не давать разрешение — у них уведомления ОС не приходят, но инбокс/счётчик всё равно должен работать корректно (например, быть пустым, без ошибок).
- **Текущая версия:** `docs/version.json` → build 26, v1.12.1.

## Открытые вопросы (для BA/архитектора)
- **Где и как хранить историю уведомлений (ключевое архитектурное решение).** Push прилетает в SW и при закрытом приложении, а `localStorage` из SW недоступен. Варианты: (а) **IndexedDB** — писать запись в SW-обработчике `push` (доступно из SW), читать из приложения; (б) хранить только то, что приложение получило при открытом окне (через `postMessage` из SW в клиент → `localStorage`), смирившись с потерей уведомлений, пришедших при закрытом приложении; (в) дочитывать историю с Cloudflare Worker (если хранить список отправленных на сервере). Выбрать с учётом «должны попадать ВСЕ уведомления, в т.ч. при закрытом приложении».
- **Что считать «уведомлением» в инбоксе.** Только push (admin/промо/напоминания)? Включать ли системные события приложения? Уточнить набор типов и поля записи (заголовок, текст, дата/время, прочитано/нет, иконка/тип, опц. ссылка/действие при тапе).
- **Связь счётчика с системным badge.** Нужно ли синхронизировать число с иконкой приложения (Badging API / `navigator.setAppBadge`) или только визуальный бейдж на колокольчике внутри приложения? (Скорее только внутри приложения — уточнить.)
- **Логика «прочитано».** Когда уведомления помечаются прочитанными: при открытии экрана инбокса? при просмотре конкретного? Уточнить, обнуляется ли счётчик целиком при заходе в инбокс.
- **«Очистить историю».** Что делает: удаляет все записи или только помечает прочитанными? Нужна ли подтверждающая модалка перед очисткой? Возможность удалить одно уведомление (свайп/кнопка) — в объёме или нет?
- **Экран vs шторка.** Инбокс — это отдельный экран (`s-...` как остальные) с таббаром, или выезжающая панель/модалка поверх главной? Как с него возвращаться.
- **Лимит хранения.** Сколько уведомлений хранить (например, последние N или за период), чтобы не раздувать хранилище? Нужна ли автоочистка старых.
- **Пустое состояние.** Что показывать, когда уведомлений нет (текст/иллюстрация).
- **Поведение для неавторизованных / без разрешения на уведомления.** Показывать ли колокольчик всегда; что в инбоксе, если пользователь не подписан.
- **Дедупликация и порядок.** Сортировка (новые сверху), исключение дублей при повторной доставке.
- **Где ещё показывать колокольчик.** Только на `s-home` или в шапках других вкладок тоже? (По требованию — главный экран; уточнить.)
- **Точное место и вид бейджа.** Кружок-счётчик «справа внизу» колокольчика; формат при больших числах (например, «9+»); цвет/контраст на градиентном хедере.

## Ограничения
- Vanilla JS + HTML + CSS, **без рантайм-зависимостей**; код в `docs/` (`prototype.html`, `docs/modules/*.js`, `docs/style.css`, при необходимости `docs/sw.js`).
- Если для персистентности используется SW-обработчик `push` / меняется `APP_SHELL` — **поднять `CACHE_VERSION`** в `docs/sw.js`.
- Не ломать текущий поток push: показ системного уведомления (`sw.js:169`), `notificationclick` (`sw.js:172-182`), внутриприложенческий тост `#pushNotif` и подписку (`push.js`).
- Экранирование текста уведомлений (заголовок/тело) через `esc()` при рендере в инбоксе (контент приходит извне — admin push).
- Минимальные изменения строго по объёму требования. Не трогать каталог услуг, запись, профиль сверх необходимого.
- Доступность хранилища: учесть, что `localStorage` недоступен из SW; не полагаться на него для приёма push при закрытом приложении.
- Без ошибок в консоли; корректная работа на тач; оффлайн-просмотр уже полученных уведомлений желателен (инбокс читается из локального хранилища).
- API-токены/секреты не хардкодить и не логировать.

## Критерий готовности
На главном экране в правом верхнем углу шапки отображается колокольчик. При наличии непрочитанных уведомлений на нём справа внизу виден бейдж с их количеством. Тап по колокольчику открывает инбокс со списком всех полученных уведомлений (новые сверху, с заголовком, текстом и датой), включая те, что пришли при закрытом приложении. После просмотра инбокса непрочитанные помечаются прочитанными и бейдж обнуляется. На экране инбокса есть действие «Очистить историю», которое очищает список (с понятным пустым состоянием после). Текущий поток push (системное уведомление, тап, тост, подписка) продолжает работать; ошибок в консоли нет; при изменении `APP_SHELL`/`sw.js` поднята `CACHE_VERSION`.
