# Требование 022: Лента в разрезе категорий и услуг YCLIENTS + запись на услугу из поста

## Описание
Сейчас лента студии существует «сама по себе»: посты привязываются к произвольной строке-категории из **хардкод-чипов** админки, не связанной с реальными категориями и услугами YCLIENTS. Из-за этого кнопка «Записаться» в посте работает плохо.

Нужно перевести ленту **в разрез существующих категорий и услуг YCLIENTS**:
- Пост можно привязать к **категории** YCLIENTS или к **конкретной услуге** YCLIENTS.
- Сохранить возможность публиковать **свободные посты** без привязки (новость/анонс).
- Если пост **по услуге** — у клиента должна быть возможность нажать «Записаться» и перейти **на карточку этой услуги** (а не в общий каталог). Пример: администратор публикует пост «Акция на стрижку 20%» → клиент жмёт «Записаться» → открывается карточка услуги «Стрижка».
- В ленте нужен **фильтр по типам постов**.

## Текущее состояние (подтверждено кодом)
- **Модель поста** (`docs/modules/admin.js:110`): `{ id, cat, text, image, date, draft }`. Поле `cat` — это текст активного чипа из админ-формы, **не** id/название реальной категории.
- **Категории в админ-форме поста хардкод** (`docs/prototype.html`, экран `#s-admin-post`): чипы `Брови / Ногти / Лицо / Волосы / Тело / Акции` (`adminCat(this)`), плюс fallback `Другое` (`admin.js:109`).
- **Реальные данные YCLIENTS грузятся при старте** (`docs/app.js:166-200`):
  - Категории: `GET /service_categories/{company}` → `catMap[c.id] = c.title`.
  - Услуги: `GET /services/{company}` → `SERVICES_DATA` = `{ id: String(s.id), name, cat: catMap[s.category_id] || 'Другое', dur, price_min, priceStr, comment, photos }` (`app.js:189-197`). У каждой услуги стабильный `id` и реальная категория `cat`.
  - Реальные категории (пример из статических данных): `Брови, Губы, Глаза, Эпиляция` — **не совпадают** с хардкод-чипами ленты.
- **Кнопка «Записаться →»** в карточке поста (`docs/modules/feed.js:68`) вызывает `_feedGoBook(p.cat)` → `window.filterCategory(cat)` + `go('s-services')` (`feed.js:111-114`). Фильтрует каталог по строке `cat`; при рассинхроне строк → «Ничего не найдено». На конкретную услугу перехода нет.
- **Переход на карточку услуги существует**: `openServiceCard(id)` → экран `s-service` (`docs/modules/services.js:283-344`). `filterCategory(cat)` фильтрует каталог по названию категории (`services.js:61-68`).
- **Фильтр в ленте — по категории-строке** (`feed.js:87-109`): чипы строятся из `new Set(posts.map(p => p.cat))`. Фильтра «по типам постов» нет.
- **Карточка поста рендерится в двух местах** одним `_feedCardHtml` (`feed.js:54-74`): список ленты `s-feed` и одиночный пост `s-post` (`renderPostScreen`, `feed.js:127-137`). Превью на главной — `_renderHomeFeedPreview` (`profile.js`).
- **Синхронизация постов**: посты хранятся в `localStorage` `yc_feed_posts` и синхронизируются через GitHub (`_ghSyncPosts` / `_ghPullToLocal`, `docs/modules/github.js`) → `posts.json`. Новые поля модели поста должны переноситься при публикации/синхронизации.

## Целевое поведение (предлагаемая модель — на согласование с BA)

### Типы постов
1. **Свободный** (новость/анонс) — без привязки к категории/услуге. CTA «Записаться» **не показывается** (или показывается нейтральный CTA — см. ОВ-3).
2. **По категории** — привязан к реальной категории YCLIENTS. CTA ведёт в каталог `s-services`, отфильтрованный по этой категории (`filterCategory(realCat)`).
3. **По услуге** — привязан к конкретной услуге YCLIENTS (по `serviceId`). CTA «Записаться» ведёт на карточку услуги `openServiceCard(serviceId)`. Пример с акцией на стрижку.

### Админ-форма поста
- Заменить хардкод-чипы категорий на выбор **типа поста** + соответствующего объекта:
  - Свободный → ничего выбирать не нужно.
  - По категории → выбор из **реальных категорий** YCLIENTS (источник — `SERVICES_DATA`/`catMap`).
  - По услуге → выбор/поиск из **реальных услуг** YCLIENTS (`SERVICES_DATA`), сохраняется `serviceId` (+ название/категория для отображения).
- Модель поста расширяется (черновик, детали — за архитектором): например `type: 'free'|'category'|'service'`, `categoryId`/`category` (название), `serviceId`/`serviceName`. Существующее поле `cat` — решить судьбу (визуальный градиент/иконка завязаны на `cat` в `feed.js:5-13`).

### Фильтр в ленте по типам постов
- Добавить фильтр по **типу поста** (например: «Все», «Акции/Услуги», «Новости/Свободные», и/или по категории). Точный набор измерений фильтра — ОВ-4.

## Дополнительные варианты «что ещё может потребоваться» (для проработки BA/архитектором)
- **Акция/промо как подтип.** Пост-акция (скидка %, бейдж «−20%») поверх привязки к услуге или категории. Возможно, флаг `promo` + опциональный текст скидки.
- **CTA-метка зависит от типа.** «Записаться →» (услуга), «Смотреть услуги →» (категория), скрыт/нейтрален (свободный).
- **Привязка к мастеру.** Пост о мастере с CTA «Записаться к мастеру» — вероятно вне scope v1 (ОВ-5).
- **Несколько услуг в одном посте.** Вероятно вне scope v1; в посте — одна услуга (ОВ-6).
- **Внешняя ссылка / без действия** для чисто информационных постов.

## Открытые вопросы (для BA/архитектора)
- **ОВ-1. Источник категорий для админки.** Брать реальные категории из `SERVICES_DATA` (`new Set(s.cat)`)/`catMap`. Подтвердить, что отдельный справочник не нужен.
- **ОВ-2. Обратная совместимость существующих постов.** У старых постов в `yc_feed_posts`/`posts.json` есть только `cat` (хардкод-строки, часто не совпадают с реальными категориями). Как мигрировать: трактовать как «свободные», или как «по категории» с маппингом названий, или оставить как есть до перепубликации? Влияет на CTA старых постов.
- **ОВ-3. CTA у свободного поста.** Скрывать кнопку полностью или показывать нейтральный CTA («Все услуги»)?
- **ОВ-4. Измерения фильтра ленты.** Фильтр строго по типу поста, по категории, или комбинированный? Сейчас фильтр по `cat`; нужно решить, заменяется он или дополняется.
- **ОВ-5. Пост, привязанный к мастеру** — входит ли в этот релиз? (предложение — нет, отдельное требование).
- **ОВ-6. Несколько услуг в одном посте** — нужно ли? (предложение — нет, одна услуга на пост).
- **ОВ-7. Промо/скидка** — отдельный подтип/бейдж в этом релизе или позже?
- **ОВ-8. Удалённая/несуществующая услуга или категория.** Поведение CTA, если `serviceId`/категория из поста больше не существует в YCLIENTS (услугу удалили). Не падать, разумный fallback.

## Ограничения
- Vanilla JS/HTML/CSS, без новых рантайм-npm-зависимостей; код в `docs/` (`modules/feed.js`, `modules/admin.js`, `modules/services.js`, `prototype.html`, при необходимости `style.css`).
- Источник категорий/услуг — уже загружаемые `SERVICES_DATA`/`catMap` из YCLIENTS; **не** добавлять новые тяжёлые запросы без необходимости.
- Сохранить синхронизацию постов через GitHub (`posts.json`): новые поля переносятся при публикации и чтении.
- Переход на услугу — через существующий `openServiceCard(serviceId)`; на категорию — через `filterCategory` + `s-services`.
- Не ломать: одиночный пост `s-post`, превью ленты на главной, существующие опубликованные посты (graceful degradation по ОВ-2).
- При изменении файлов из `APP_SHELL`/`sw.js` — поднять `CACHE_VERSION`.
- Минимальные изменения строго по согласованному объёму.

## Критерий готовности
- Администратор может создать пост одного из типов: свободный, по категории YCLIENTS, по конкретной услуге YCLIENTS (категории/услуги берутся из реальных данных, не из хардкода).
- Пост по услуге показывает CTA «Записаться», ведущий на карточку именно этой услуги (`s-service`); пример с акцией на стрижку работает end-to-end.
- Пост по категории ведёт в каталог, отфильтрованный по реальной категории (без «Ничего не найдено» при существующей категории).
- Свободный пост публикуется без привязки; его CTA ведёт себя согласно решению по ОВ-3.
- В ленте работает фильтр по типам постов (согласно ОВ-4).
- Существующие посты не приводят к ошибкам (поведение по ОВ-2); одиночный пост и превью на главной не сломаны.
- В консоли нет ошибок; PWA остаётся устанавливаемой; при изменении `APP_SHELL`/`sw.js` поднята `CACHE_VERSION`.
