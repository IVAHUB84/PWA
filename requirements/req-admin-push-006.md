# Требование 006: Администратор отправляет push-уведомления клиентам из админ-панели

## Описание
Администратор студии должен иметь возможность через админ-панель мобильного приложения (PWA) отправлять push-уведомления клиентам. Функционал частично реализован, но **по факту не работает надёжно** — нужно прогнать его по полному workflow и устранить все дефекты, чтобы он работал сквозно на реальном устройстве.

## Цель
Сделать так, чтобы администратор мог гарантированно доставить push-уведомление выбранной аудитории клиентов из панели приложения. Это полноценный релиз с исправлением всех ошибок, а не документирование существующего поведения.

## Ожидаемое поведение (как должно быть)
- Администратор открывает экран рассылок в админ-панели.
- Выбирает шаблон уведомления или вводит свой текст; видит превью.
- Выбирает аудиторию получателей:
  - Все клиенты (все подписчики);
  - Активные за последние 30 дней;
  - Клиенты с предстоящими записями.
- Отправляет немедленно («Отправить сейчас») **или** планирует на заданное время («Запланировать»).
- Получает достоверный результат: сколько уведомлений реально доставлено, сколько не удалось.
- История рассылок отображает отправленные и запланированные кампании.
- (Желательно) Возможность отправить push конкретному клиенту из списка клиентов.

## Текущее состояние и проблемные места (что предположительно сломано)
Реализация уже есть, но требует ревизии. Известные/подозреваемые дефекты, которые нужно проверить и исправить:
1. **Сегменты аудитории — фиктивны.** Экран даёт выбрать сегмент («Все» / «Активные за 30 дней» / «С записями»), но фактически рассылка всегда уходит **всем подписчикам**. Worker не умеет фильтровать по сегменту.
2. **Планирование — фиктивно.** «Запланированная» кампания только сохраняется в localStorage и **никогда не отправляется**. Серверного планировщика для произвольных рассылок нет (cron шлёт только напоминания о записи на завтра).
3. **Отправка одному клиенту не выведена в UI.** Worker поддерживает `targetClientId`, но кнопки «отправить этому клиенту» в списке клиентов нет.
4. **Подписка клиентов.** Нужно проверить, что обычные клиенты реально подписываются на push (flow `subscribePush`), иначе у рассылки нет получателей и «Отправлено: 0».
5. **Конфигурация.** Отправка зависит от заданных в админке Worker URL + Admin Secret (localStorage). Нужно убедиться, что это надёжно настраивается и понятно администратору, что без этого ничего не работает.
6. **Достоверность обратной связи.** Тосты/счётчики должны показывать реальный результат от Worker, а не «успех» вне зависимости от факта доставки.

## Технический контекст (текущая архитектура)
- Клиентский push-модуль: `docs/modules/push.js` (`subscribePush`, `unsubscribePush`, `sendAdminPush`).
- Админ-логика рассылок: `docs/modules/admin.js` (`renderAdminPush`, `sendNewPush`, выбор аудитории, шаблоны, планирование).
- UI рассылок и настроек Worker: `docs/prototype.html`.
- Бэкенд push: Cloudflare Worker `cf-worker/src/index.js` — эндпоинты `/subscribe`, `/unsubscribe`, `/send` (требует `X-Admin-Secret`), cron-напоминания. KV `SUBSCRIPTIONS`. Worker задеплоен: https://studio-push.ivahub84.workers.dev (на момент написания `/vapid-public-key` отвечает 200, `/send` без секрета → 401).

## Ограничения
- Vanilla JS, без рантайм-зависимостей; код в `docs/`.
- Секреты (Admin Secret, VAPID private, YC-токены) — только в Worker, не в репозитории.
- При изменениях `sw.js` / APP_SHELL — поднимать `CACHE_VERSION`.
- Push работает только по HTTPS (GitHub Pages) или localhost.

## Диагностика (подтверждено в ходе разбора)
Симптом сужен: push **приходит в браузере на десктопе**, но **не приходит на iPhone**. Проверено: на iPhone приложение установлено на «Домой» (standalone), iOS 16.4+, разрешение на уведомления выдано. То есть предпосылки iOS соблюдены — проблема не в способе запуска.

**Подтверждённый корень — коллизия подписок по `clientId`.** Worker хранит одну подписку на клиента (ключ `sub:${clientId}`, `cf-worker/src/index.js:142`). На компьютере и iPhone выполнен вход под одним клиентом → устройство, подписавшееся последним, **затирает** подписку предыдущего. В KV остаётся единственный endpoint → `/send` доставляет только на него; тост показывает «Отправлено 1», хотя подписаны два устройства. iPhone не получает push, так как его endpoint в хранилище отсутствует.

Следствия того же дефекта:
- `handleCron` (напоминания) также берёт одну подписку на клиента → мультидевайс-клиент получает напоминание лишь на одно устройство.
- `/unsubscribe` удаляет по `clientId` (`index.js:154`) → отписка на одном устройстве снесёт подписку другого.
- `handleSend` не логирует статус ответа push-сервиса и не различает причины `failed` → отладка вслепую.

Формат отправки (VAPID JWT ES256, `aud`=origin, `exp` 12ч, aes128gcm) — валиден и для Apple; это не источник проблемы.

## Критерий готовности (определение «работает»)
Администратор на реальном устройстве выбирает аудиторию, отправляет уведомление, и подписанные клиенты этой аудитории действительно получают push на **всех своих устройствах**; счётчик доставленных достоверен; запланированная рассылка приходит в назначенное время.
