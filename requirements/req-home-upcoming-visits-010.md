# Требование 010: Просмотр всех ближайших визитов с главного экрана

## Описание
С главного экрана клиент должен иметь возможность перейти и посмотреть **все ближайшие (будущие) визиты** — не заходя в Профиль → История. Показывать **только предстоящие будущие визиты** (не прошлые). По аналогии с лентой добавить ссылку «Все →».

## Целевое поведение
- На главном экране рядом с блоком «Ближайший визит» появляется affordance «Все →» (как у ленты и у мастеров).
- По нажатию открывается список **всех предстоящих будущих визитов** (отсортированных по дате), без прошлых визитов.
- Для каждого визита доступны действия «Перенести» и «Отменить» (как сейчас для единственного визита на главной).
- Это отдельный путь, не требующий захода в Профиль → «История посещений».

## Текущее состояние (подтверждено кодом)
- На главной `renderHomeHero` (`docs/modules/profile.js:63`) вычисляет **все** будущие визиты (`upcoming`, строки 85-87), но показывает только первый — `next = upcoming[0]` (строка 88). Остальные предстоящие визиты не видны.
- Карточка «Ближайший визит» уже имеет `onclick="go('s-history')"` (profile.js:116) — тап ведёт на экран истории, но это неочевидно (нет явной «Все →») и ведёт в общий экран «История».
- Экран `s-history` (`history.js` `_renderHistoryFromCache`, строки 105-176) **уже** рендерит секцию «Предстоящие · N» со всеми будущими визитами и кнопками «Перенести»/«Отменить» (строки 121-143), а ниже — «Прошлые визиты» с «Повторить»/«Оценить» и фильтром по периоду. То есть данные и рендер списка предстоящих уже существуют, но смешаны с историей прошлых визитов в экране, который концептуально «История» (в профиле).
- Паттерн «Все →»: в ленте — `<button class="btn-ghost" onclick="go('s-feed')">Все →</button>` (prototype.html:72); у мастеров — `onclick="_browseAllMasters()"` (prototype.html:82).

## Открытые вопросы (для BA/архитектора)
- **Отдельный экран или переиспользование?** Сделать отдельный экран «только предстоящие визиты» (например `s-upcoming`) или переиспользовать секцию «Предстоящие» экрана `s-history` (например открывать её и не показывать прошлые при входе с главной)? Пользователь просит «только ближайшие будущие» и «не в Истории» — вероятно, ожидает чистый экран без прошлых визитов.
- **Когда показывать «Все →»?** Всегда при наличии хотя бы одного предстоящего визита, или только когда их 2+ (т.к. один уже показан на главной)?
- **Действия в списке.** Подтвердить, что в списке нужны «Перенести» и «Отменить» для каждого визита (как сейчас), без «Повторить»/«Оценить» (это для прошлых).
- **Пустое состояние.** Что показывать, если предстоящих визитов нет (хотя «Все →» в этом случае, вероятно, не показывается).

## Ограничения
- Vanilla JS, без рантайм-зависимостей; код в `docs/`.
- Источник визитов — `_loadStoredRecords()` (+ серверное слияние `_fetchAndMergeServerRecords`), фильтр будущих и не отменённых — как в существующем коде.
- При изменении `APP_SHELL`/`sw.js` — поднять `CACHE_VERSION`. Минимальные изменения по объёму.
- Не ломать существующий экран `s-history` и действия Перенести/Отменить/Повторить/Оценить.

## Критерий готовности
С главного экрана по «Все →» открывается список всех предстоящих будущих визитов (без прошлых), отсортированных по дате; для каждого доступны «Перенести» и «Отменить»; путь не требует захода в Профиль → История; при отсутствии предстоящих визитов affordance не вводит в заблуждение.
