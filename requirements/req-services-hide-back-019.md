# Требование 019: Убрать кнопку «Назад» на экране услуг при входе с главного экрana

## Описание
При входе в раздел услуг (`s-services`) **с главного экрана** сверху не должно быть кнопки «‹ Назад» — возвращаться некуда (услуги открыты как корневой раздел через нижнюю навигацию). Сейчас кнопка отображается всегда и при входе через таб не работает (визуальный «мусор»).

## Текущее состояние (подтверждено кодом)
- Шапка экрана услуг: `<div class="nav-bar"><button class="back-btn" onclick="back()">‹ Назад</button></div>` (`docs/prototype.html:105`), под ней `page-title` «Выберите услугу» (`prototype.html:106`).
- **Экран `s-services` открывается двумя способами:**
  1. **Как таб (корень)** — `go('s-services','tab')` из нижней навигации (кнопка-FAB «Запись» и таб «Услуги», напр. `prototype.html:95,97,130,132`). Режим `tab` **сбрасывает** `navHistory = ['s-services']` (`docs/modules/navigation.js:67-76`). В этом случае `back()` ничего не делает: `if (navHistory.length < 2) return;` (`navigation.js:103-104`) — **кнопка «Назад» неактивна/бесполезна**.
  2. **Как push** — `go('s-services')` (без `tab`) с кнопки «Записать себя» на главном (`prototype.html:62`). Тогда экран добавляется в `navHistory` поверх `s-home` (`navigation.js:87`), и `back()` корректно возвращает на главную — **здесь кнопка осмысленна**.
- `back-btn` — общий класс кнопки «Назад» на многих экранах; правка стиля класса затронет все экраны, поэтому решение должно касаться только инстанса на `s-services`.
- `onEnter`-обработчики экранов регистрируются через `registerOnEnter` (`navigation.js:6-9,42-45`) — есть точка, где можно показать/скрыть элемент при входе на экран.

## Открытые вопросы (для BA/архитектора)
- **Когда именно скрывать кнопку (ключевой вопрос).** Варианты: (а) **динамически** — скрывать, когда возвращаться некуда (`navHistory.length < 2`, т.е. вход как таб/корень), и показывать, когда есть куда (push с «Записать себя»); (б) **всегда** скрывать на `s-services` (тогда из push-входа «Записать себя» вернуться кнопкой нельзя — только нижней навигацией/жестом). Рекомендация — вариант (а): убирает мёртвую кнопку при входе с главного, сохраняет рабочую при реальном push. Подтвердить трактовку «с главного экрана».
- **Реализация скрытия.** CSS (`display:none`) против условного рендера/переключения класса в `onEnter` для `s-services`. Нужно ли резервировать высоту `nav-bar`, чтобы заголовок «Выберите услугу» не «прыгал» при появлении/исчезновении кнопки.
- **Согласованность с другими табами.** На прочих корневых табах (история, профиль, лента) кнопка «Назад» в шапке — есть ли там та же проблема и нужно ли единообразие (по объёму требования — только `s-services`, подтвердить).

## Ограничения
- Правки в `docs/prototype.html` и/или `docs/modules/*.js` (`navigation.js`/инициализация) и при необходимости `docs/style.css` — **только для экрана `s-services`**; не менять поведение `back-btn` на других экранах.
- Не ломать вход через push «Записать себя» (если выбран вариант с сохранением рабочей кнопки) и переходы из услуг дальше по потоку записи.
- Без ошибок в консоли; корректная вёрстка шапки (без «прыжка» заголовка).
- При изменении версий модулей/`APP_SHELL` в `sw.js` — поднять `CACHE_VERSION`.
- Минимальные изменения строго по объёму.

## Критерий готовности
При открытии услуг с главного экрана через нижнюю навигацию (таб/«Запись») в шапке экрана `s-services` кнопка «‹ Назад» отсутствует, заголовок «Выберите услугу» отображается без смещения. Согласованное с пользователем поведение для входа через «Записать себя» соблюдено (кнопка работает либо отсутствует — по выбранному варианту). Кнопка «Назад» на других экранах и поток записи из услуг не затронуты; ошибок в консоли нет.
