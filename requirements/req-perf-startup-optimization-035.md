# Требование 035: Оптимизация скорости запуска и навигации

## Описание
Ускорить холодный старт и навигацию приложения за счёт устранения избыточных
сетевых запросов и повторных рендеров. Видимое поведение и UI не меняются —
только производительность. Источник — ревью производительности от 2026-05-31.

Объём (по приоритету):
1. **Убрать N+1 запросы цен мастеров** при каждом запуске.
2. **Кэшировать данные каталога** (услуги/мастера/категории) для мгновенного
   холодного рендера (stale-while-revalidate на уровне данных).
3. **Не ходить в сеть при каждом входе на главную** (троттлинг фоновых обновлений).
4. (Опц.) Убрать дублирующиеся ре-рендеры при старте.
5. (Опц.) Ленивая загрузка EmailJS вместо подключения на каждом старте.

## Текущее состояние (подтверждено кодом)

### 1. N+1 запросы цен — `_collectStaffPrices` (app.js:233-266)
На каждый запуск для **каждого** мастера делается отдельный
`YC.get('/book_services/{company}', { staff_id })` (app.js:240). При N мастерах —
N параллельных запросов только ради уточнения диапазона цен услуг
(`servicePriceRange`, `staffServicePrice` в state.js). По завершении — ещё один
полный `renderServices()` (app.js:265). Запускается из `initApp()` (app.js:228)
безусловно при каждом старте.

### 2. Нет кэша API-данных — `initApp` (app.js:162-231)
Пять запросов параллельно при каждом старте (app.js:168-174): `service_categories`,
`services`, `book_staff`, `book_services`, `companies`. Результат пишется в
`SERVICES_DATA`/`MASTERS_DATA` через `setServicesData`/`setMastersData` (state.js),
**но не персистится**. SW намеренно не кэширует `api.yclients.com`
(`NETWORK_ONLY_HOSTS`, sw.js:47). На медленной/отсутствующей сети при холодном
старте каталог пуст до прихода ответа. Каталог услуг — почти статичные данные.

### 3. Сеть при каждом входе на главную — `registerOnEnter('s-home')` (app.js:58-71)
Каждая навигация на `s-home` (не только старт) вызывает:
- `_fetchAndMergeServerRecords(client_id)` (api.js:69) → `GET /records` + ре-сортировка/запись localStorage;
- `_ghRead()` (github.js:9-18) → `fetch(_GH_RAW + '?_=' + Date.now())` — cache-bust, гарантированно мимо SW и HTTP-кэша.
Аналогичный `_ghRead` есть в boot-ветке `checkSession()` (app.js:299-304).

### 4. Дублирующиеся ре-рендеры при старте
`renderServices()` вызывается минимум 4 раза за загрузку: boot (app.js:287),
после маппинга услуг (app.js:205), после маппинга мастеров (app.js:227),
в `_collectStaffPrices` (app.js:265). `renderHomeHero()` — несколько раз
(app.js:226 + onEnter + boot). Каждый `renderServices()` — полный `innerHTML`
rebuild списка (`_renderList`, services.js:77-100).

### 5. EmailJS на каждом старте — prototype.html:18
`<script defer src="https://cdn.jsdelivr.net/npm/@emailjs/browser@4/dist/email.min.js">`
подключается глобально, инициализируется в app.js:157-159. Используется только
в fallback-сценарии email-логина (3rd-party CDN на каждом заходе).

## Пожелания/решения пользователя (со слов)
- Ускорить запуск и навигацию; видимый результат и логика не должны измениться.
- Приоритет — пункты 1-3 (реальный выигрыш на каждом старте/навигации).

## Открытые вопросы (для BA/архитектора)
- **Где хранить кэш данных каталога (п.2)** — localStorage (просто, лимит ~5 МБ,
  синхронный) или IndexedDB (асинхронный, больше места, уже используется для
  inbox)? **Предложение: localStorage** — данные небольшие, нужен синхронный
  мгновенный рендер при старте.
- **TTL/инвалидация кэша (п.1, п.2).** Полагаться только на «отрисовать из кэша,
  обновить в фоне всегда» или ввести TTL (например, цены/каталог — N минут/часов)?
  **Предложение: всегда обновлять в фоне (stale-while-revalidate), TTL не вводить —
  свежесть гарантируется фоновым запросом, кэш только для мгновенного первого
  кадра.**
- **Можно ли убрать N+1 полностью (п.1)** — есть ли в YCLIENTS batch-эндпоинт цен
  по всем мастерам, или оставить пер-мастерные запросы, но только в фоне с
  кэшированием? **Предложение: кэшировать результат `_collectStaffPrices` и
  рендерить из кэша мгновенно; пер-мастерные запросы оставить как фоновое
  обновление (не блокируют первый рендер).**
- **Интервал троттлинга для `s-home` (п.3).** Как часто допустимо реально ходить в
  сеть при повторных входах на главную? **Предложение: не чаще раза в 60 секунд
  (timestamp в памяти модуля), при старте — всегда.**
- **`_ghRead` cache-bust (п.3).** Убрать `?_=Date.now()` и положиться на
  HTTP-кэш/SW, или оставить cache-bust но с троттлингом вызовов?
  **Предложение: убрать cache-bust, добавить троттлинг.**

## Ограничения
- Vanilla JS + HTML + CSS, **без рантайм-зависимостей**; код в `docs/app.js`,
  `docs/modules/*.js`, при необходимости `docs/sw.js`.
- **Никаких изменений видимого поведения/UI** — только производительность.
  Те же экраны, тот же контент, та же логика бронирования.
- **Минимальные изменения строго по объёму.** Не трогать: поток бронирования,
  слоты, инбокс/push, стратегии SW для статики (только при необходимости —
  данные API).
- API-токен YCLIENTS уже захардкожен в коде — **не усугублять** (не плодить новые
  места хранения секретов); кэш данных не должен включать токены.
- `localStorage.setItem` оборачивать в `try/catch` (как уже принято в коде,
  напр. api.js:108) — приватный режим/переполнение не должны ронять старт.
- При изменении `APP_SHELL`/`sw.js` — поднять `CACHE_VERSION`.
- Деградация офлайн: при отсутствии сети и наличии кэша — рендерить из кэша;
  при отсутствии и того, и другого — текущее поведение (пусто/скелетон).

## Критерий готовности
- **Холодный старт с кэшем**: при повторном запуске каталог услуг и список
  мастеров рендерятся мгновенно из кэша, без ожидания сети; сетевой ответ
  обновляет данные в фоне.
- **Цены мастеров (п.1)**: первый рендер не блокируется N пер-мастерными
  запросами; диапазоны цен берутся из кэша и/или дорисовываются в фоне.
- **Главная (п.3)**: повторные входы на `s-home` в пределах окна троттлинга
  **не** инициируют новых `/records` и `_ghRead` запросов (проверяется в
  DevTools → Network); первый вход/старт работает как прежде.
- (Опц.) Число вызовов `renderServices()` при старте сокращено без потери
  корректности отображения.
- (Опц.) EmailJS не грузится при старте; подгружается при входе в email-fallback,
  email-логин работает как прежде.
- **Функционально без регрессий**: логин → услуга → мастер → слот → бронь
  проходит; ноль ошибок в консоли; PWA installable не сломан; офлайн-деградация
  корректна; при изменении `sw.js` поднята `CACHE_VERSION`.
