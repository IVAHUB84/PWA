# Требование 014: Детальная карточка услуги (описание + фото работ + «Выбрать мастера»)

## Описание
Сейчас тап по услуге в каталоге сразу ведёт к выбору мастера. Нужно вставить промежуточный **экран детальной карточки услуги**: клиент сначала изучает услугу (описание + фотографии работ), и только потом по кнопке переходит к выбору мастера.

## Целевой поток (как должно быть)
1. Клиент заходит в **Услуги** → видит каталог услуг: **название**, **изображение** (из YCLIENTS), **диапазон цен**, **длительность**.
   - Фото на карточке каталога берётся **из YCLIENTS** (`image_group.images[]` услуги). Если у услуги **несколько** фото — на карточке каталога показывается **первое**.
   - Если фото у услуги нет — категорийный плейсхолдер (градиент + эмодзи), как сейчас.
2. Тап по карточке услуги → открывается **детальная карточка услуги** с:
   - **описанием** услуги (из YCLIENTS),
   - **фотографиями работ** — их может быть **больше одной** (из YCLIENTS),
   - (цена/длительность — уточнить с BA, см. открытые вопросы).
3. Клиент изучает информацию.
4. Внизу экрана — кнопка **«Выбрать мастера»**.
5. Тап по кнопке → экран **«Выберите мастера»** (`s-masters`).

> Каталог из шага 1 (один столбец, фото, диапазон цен, длительность) уже описан/реализуется требованиями 012 (layout), 013 (диапазон цен), 001 (визуальный каталог). Суть **этого** требования — **новый детальный экран** между каталогом и выбором мастера.

## Текущее состояние (подтверждено кодом)
- Тап по карточке услуги сразу вызывает `selectService(id)` (`docs/modules/services.js:91`, `onclick`), который ставит `state.serviceId` и **сразу** делает переход: `go('s-masters')` (или `go('s-slots')`, если мастер предвыбран) — `services.js:147-156`. Промежуточного экрана услуги **нет**.
- Услуги грузятся из YCLIENTS `/services/{company}` в `app.js:167-176` и маппятся в `SERVICES_DATA` с полями: `id, name, cat, dur, price_min, priceStr`. **Поля `comment` (описание) и `image_group` (фото) сейчас НЕ сохраняются** — для детальной карточки маппинг придётся расширить.
- Описание услуги в YCLIENTS — это поле **`comment`** объекта услуги (не `description`); у части услуг оно уже заполнено (см. memory reference-yclients-api). Может быть пустым.
- Фото услуги в YCLIENTS — массив **`image_group.images[]`** объекта услуги. Загружались через `POST /api/v1/images/service` (ресайз ≈420×280, хранятся на `assets.yclients.com`). **Важно:** повторная загрузка по тому же `service_id` перетирает картинку — на практике у услуги в `image_group` обычно **одна** картинка. Источник для «больше одного фото работ» нужно подтвердить (см. открытые вопросы).
- В каталоге картинка резолвится `resolveServiceImage(s)` (`docs/modules/serviceImages.js:21-31`) через `SERVICE_PHOTO_MAP` (сейчас пустой `{}`) с fallback на категорийный градиент+эмодзи. Прямого использования `image_group` из API пока нет.
- Есть готовый образец экрана-карточки: профиль мастера `s-master` (`_masterCardHtml` в `docs/modules/masters.js`) — стек экранов и навигация `go()` (`docs/modules/navigation.js`) уже поддерживают подобные детальные экраны.
- Существует функция `selectService` и переход к `s-masters` — её нужно «разрезать»: тап по каталогу → детальный экран; кнопка на детальном экране → текущая логика перехода к мастерам.

## Открытые вопросы (для BA/архитектора)
- **Источник нескольких фото работ.** `image_group.images[]` услуги обычно содержит **одно** фото (перезапись при загрузке). Откуда брать «больше одного фото работ»: (а) если у каких-то услуг в `image_group` реально несколько — показывать их; (б) общее портфолио/галерея студии; (в) фото примеров по категории; (г) загрузить несколько фото на услугу заранее. Подтвердить источник и есть ли вообще сейчас услуги с >1 фото.
- **Если фото нет / описания нет.** Что показывать: для фото — категорийный плейсхолдер (градиент+эмодзи, как в каталоге) или прятать галерею; для пустого `comment` — прятать блок описания или показывать заглушку. Уточнить.
- **Состав детальной карточки.** Кроме названия, описания, фото и кнопки — показывать ли на этом экране **цену (диапазон)** и **длительность**? Категорию? Что-то ещё (мастера, выполняющие услугу)?
- **Галерея фото.** Формат при нескольких фото: горизонтальный свайп/карусель, вертикальный список, сетка? Нужен ли зум по тапу. Если фото одно — показывать без листалки.
- **Режим предвыбранного мастера.** Когда мастер уже выбран (`state.masterPreSelected`), сейчас услуга ведёт сразу на `s-slots` (время). Показывать ли детальную карточку и в этом режиме, и какой тогда текст/назначение нижней кнопки («Выбрать время» вместо «Выбрать мастера»)? Уточнить.
- **Кнопка «Назад».** С детальной карточки назад — в каталог услуг (стандартное поведение стека `go`)? Подтвердить.
- **Качество фото.** `image_group` отдаёт ресайз ≈420×280 — приемлемо ли для крупной галереи на детальном экране, или нужен источник покрупнее.

## Ограничения
- Vanilla JS + HTML + CSS, без рантайм-зависимостей; код в `docs/` (`prototype.html`, `docs/modules/*.js`, `docs/style.css`).
- Сохранить существующее поведение каталога: поиск, фильтр по категории, баннер предвыбранного мастера, корректную цену/длительность.
- Не ломать дальнейший поток записи (услуга → мастер → время → подтверждение) и YCLIENTS-интеграцию.
- Не делать тяжёлых дополнительных запросов к API без необходимости — описание/фото по возможности брать из уже загруженного объекта услуги (расширить маппинг в `app.js`), а не отдельными запросами на каждую услугу.
- Минимальные изменения строго по объёму. При изменении `APP_SHELL`/`sw.js` — поднять `CACHE_VERSION`.
- Экранирование пользовательских/внешних данных (`comment`, URL фото) через `esc()` — защита от XSS.

## Критерий готовности
В каталоге услуг на карточках отображается фото услуги из YCLIENTS (первое из `image_group`, если фото несколько; категорийный плейсхолдер — если фото нет). Тап по карточке услуги открывает детальный экран с описанием услуги и фотографиями работ (несколько, если они есть; одно/плейсхолдер — иначе); внизу экрана есть кнопка «Выбрать мастера», по которой клиент попадает на экран «Выберите мастера»; кнопка «Назад» возвращает в каталог; описание/фото берутся из YCLIENTS; поиск, фильтр по категории, режим предвыбранного мастера и дальнейший поток записи продолжают работать без ошибок в консоли.
