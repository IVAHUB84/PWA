# Требование 015: Карусель фото услуги в детальной карточке (зацикленная, со скруглениями, с явной аффордансой)

## Описание
В детальной карточке услуги (`s-service`) нужно доработать горизонтальную карусель фотографий услуги. У услуги в YCLIENTS может быть **до 8 фото**. Требования к карусели:
1. **Скруглённые края** у изображений.
2. **Явная аффорданса**: пользователю должно быть понятно, что это карусель и что свайпом влево/вправо есть ещё фото (видна часть соседнего изображения и/или стрелки/индикаторы).
3. **Зацикливание**: после последнего фото свайп вправо ведёт к первому, и наоборот (бесконечная прокрутка в обе стороны).
4. **Свежесть фото из YCLIENTS**: новое/изменённое изображение услуги, загруженное в YCLIENTS, должно **само появляться у клиента в приложении** — без необходимости закрывать или переустанавливать приложение (без ручной чистки кэша / переустановки PWA).

## Текущее состояние (подтверждено кодом)
- Карусель уже частично реализована в `docs/modules/services.js:167-213` (`openServiceCard`): при `s.photos.length >= 2` рендерится `.svc-carousel` (нативный горизонтальный scroll-snap, `flex: 0 0 100%` — слайд на всю ширину) + индикаторы-точки `.svc-carousel-dots`. Активная точка обновляется по событию `scroll` (`services.js:208-211`).
- Один слайд занимает 100% ширины (`docs/style.css:829-831`), поэтому **соседнее фото не видно** — нет «подглядывания», что есть ещё изображения; единственная подсказка — точки.
- **Скруглений нет**: изображения слайдов (`.svc-carousel-slide img`) и одиночное фото — прямоугольные, `height: 280px; object-fit: cover` (`style.css:812-835`).
- **Зацикливания нет**: это нативный scroll-snap, на краях прокрутка упирается, перехода с последнего на первый нет.
- При **одном** фото — одиночное изображение без листалки (`services.js:178-183`), при **отсутствии** фото — категорийный плейсхолдер (градиент + эмодзи) (`services.js:184-188`). Это поведение сохранить.
- Фото услуги приходят в `s.photos` (массив URL). Источник — YCLIENTS `image_group.images[]`, маппинг расширен в рамках требования 014.
- **Источник нескольких фото ПОДТВЕРЖДЁН на живых данных (2026-05-29).** Несколько фото на услугу грузятся через **галерею в админке YCLIENTS** (загруженные получают путь `gallery_service_image/...`). Тестовая услуга «Маникюр с покрытием гель-лак» (id 29196117) имеет 3 фото. ВАЖНО: эндпойнт, который сейчас использует приложение — `GET /services/{company}` (`app.js:163`) — **схлопывает галерею до 1 фото** (`image_group.images` = объект `{basic}`). Всю галерею массивом отдают: (а) `GET /company/{company}/services/` (management) → `image_group.images` массивом (и `serviceImageUrls()` уже это умеет, `serviceImages.js:29-39`); (б) `GET /book_services/{company}` (публичный booking) → готовое поле `images[]` (массив URL) + `image` (одиночный), но поля отличаются (длительность = `seance_length` в секундах, нет `image_group`). Выбор источника/способа дочитки галереи — решение для ADR.
- URL фото экранируются через `esc()` — сохранить при любой переработке разметки.
- **Кэширование фото (причина «не появляется без переустановки»).** Фото услуг отдаются с кросс-origin хоста YCLIENTS (`assets.yclients.com` / CDN). В Service Worker для не-same-origin запросов используется стратегия **cache-first** (`docs/sw.js:108-118`): раз закэшенный URL картинки отдаётся из `RUNTIME_CACHE` всегда, и обновляется только при смене `CACHE_VERSION`. Поэтому если фото в YCLIENTS заменили (особенно при сохранении того же URL), клиент продолжает видеть старое до переустановки/чистки кэша.
- Список URL фото приходит из ответа API `/services/...` (`api.yclients.com`), а этот хост помечен `NETWORK_ONLY_HOSTS` (`docs/sw.js:43,76`) — т.е. сам список фото всегда свежий из сети. Проблема именно в кэшировании **байтов изображения** по URL, а не списка.

## Открытые вопросы (для BA/архитектора)
- **Способ реализации зацикливания.** Нативный scroll-snap зациклить «честно» нельзя. Варианты: (а) JS-карусель с трансформом/transition и клонами крайних слайдов для бесшовного цикла; (б) при достижении края мгновенно переставлять scrollLeft на противоположный край (может быть заметно); (в) лёгкая собственная реализация на `transform: translateX`. Выбрать подход без рантайм-зависимостей (vanilla).
- **Аффорданса «есть ещё фото».** Что использовать: (а) «подглядывание» — соседние слайды чуть выглядывают по краям (peek), для этого слайд < 100% ширины + отступы; (б) стрелки влево/вправо; (в) усиленные точки/счётчик «3 / 8»; (г) комбинация. Уточнить, и нужно ли это, когда фото всего 2.
- **Радиус скругления.** Какой радиус у фото (например, общий `--radius` проекта)? Скруглять каждый слайд отдельно или контейнер карусели? Поведение при peek (видно, что у соседнего фото тоже скруглены края).
- **Управление с десктопа.** Нужны ли кликабельные стрелки/точки для мыши (на тач — свайп), или достаточно прокрутки/свайпа?
- **Автопрокрутка.** Нужна ли автоматическая смена слайдов по таймеру, или только по жесту пользователя? (Скорее не нужна — уточнить.)
- **Высота/формат фото.** Сейчас 280px, `object-fit: cover`. Сохранить? Фото из `image_group` — ресайз ≈420×280, при peek и скруглениях проверить, что не мылит/не обрезает критично.
- **Поведение точек/счётчика при 8 фото.** 8 точек влезают; нужен ли счётчик вместо/вместе с точками.
- **Стратегия свежести фото YCLIENTS.** Как обеспечить автообновление фото без переустановки. Варианты: (а) для хоста фото YCLIENTS перейти на **stale-while-revalidate** (показать кэш сразу, в фоне дозагрузить свежее и обновить кэш — фото обновится при следующем открытии); (б) **network-first** для этого хоста (всегда тянуть свежее, кэш — fallback оффлайн), ценой трафика/скорости; (в) добавить cache-busting к URL фото (например, версия/таймстемп), если YCLIENTS его поддерживает; (г) ограничить TTL кэша фото. Выбрать баланс свежесть/оффлайн/трафик. Уточнить, нужно ли мгновенное обновление в текущей сессии или достаточно «при следующем заходе».
- **Влияние на оффлайн.** Фото услуг должны оставаться доступны оффлайн (как сейчас) — выбранная стратегия не должна ломать оффлайн-просмотр уже виденных карточек.
- **Идентификация хоста фото.** Подтвердить фактический хост(ы) фото услуг (`assets.yclients.com` и/или иные CDN YCLIENTS), чтобы точечно применить новую стратегию, не задев прочие кросс-origin ресурсы (EmailJS, jsdelivr).

## Ограничения
- Vanilla JS + HTML + CSS, без рантайм-зависимостей; код в `docs/` (`prototype.html`, `docs/modules/*.js`, `docs/style.css`).
- Доработка строго в пределах детальной карточки услуги (`s-service`). Не трогать каталог услуг, выбор мастера, дальнейший поток записи.
- Сохранить поведение для 1 фото (одиночное изображение) и 0 фото (плейсхолдер).
- Экранирование URL/данных через `esc()`.
- Производительность: до 8 фото с `loading="lazy"`, без тяжёлых перерисовок при свайпе.
- Изменение стратегии кэширования фото — только для хоста(ов) фото YCLIENTS; не ломать кэширование APP_SHELL, JS-модулей и прочих CDN (EmailJS, jsdelivr) и оффлайн-режим.
- Минимальные изменения строго по объёму. При изменении `APP_SHELL`/`sw.js` — поднять `CACHE_VERSION`.
- Корректная работа на тач (свайп) и без ошибок в консоли.

## Критерий готовности
В детальной карточке услуги с 2+ фотографиями отображается горизонтальная карусель: изображения со скруглёнными краями; визуально понятно, что это листаемая карусель и что есть соседние фото (peek/стрелки/индикаторы); свайп/прокрутка перелистывает фото и **зациклена** (с последнего фото — на первое и обратно); индикатор текущего фото синхронизирован с активным слайдом. Для услуги с одним фото показывается одиночное изображение без листалки, без фото — категорийный плейсхолдер. Новое/изменённое фото услуги из YCLIENTS появляется у клиента автоматически (без закрытия/переустановки приложения и ручной чистки кэша), при этом оффлайн-просмотр ранее виденных карточек сохраняется. Поведение работает на тач-устройствах, без ошибок в консоли; каталог, выбор мастера и дальнейший поток записи не затронуты.
