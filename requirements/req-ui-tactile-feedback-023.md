# Требование 023: Тактильная отзывчивость интерфейса (press feedback + haptic) и пульсация бейджа уведомлений

## Описание
Интерфейс PWA воспринимается «статичным»: часть интерактивных элементов реагирует на нажатие (масштабирование/прозрачность), часть — нет. Тактильной обратной связи (vibrate) нет. Бейдж непрочитанных уведомлений на колокольчике не привлекает внимание — клиент может его не заметить.

Нужно сделать интерфейс «живее» за счёт трёх дешёвых, не нарушающих визуальный стиль улучшений:
1. **Унифицированный press feedback** на всех интерактивных элементах (имитация «нажатия клавиш»).
2. **Haptic feedback** (`navigator.vibrate`) на ключевых действиях.
3. **Pulse-анимация бейджа** на колокольчике, когда есть непрочитанные уведомления.

## Текущее состояние (подтверждено кодом)

### Press feedback — неоднородный
В `docs/style.css` press feedback есть только у части элементов и реализован по-разному:
- `.tab-center-btn:active .tab-fab` — `scale(0.92)` (`style.css:156`)
- `.btn-primary:active` — `opacity: 0.85` (`style.css:186`)
- `.svc-catalog-card:active` — `scale(0.97)` (`style.css:282`)
- `.master-card:active` — `scale(0.99)` (`style.css:317`)
- `.pin-key:active` — `scale(.91)` (`style.css:804`)
- `.review-ext-platform:active` — `opacity: 0.85` (`style.css:723`)
- `.inbox-bell-btn:active` — `opacity: 0.75` (`style.css:887`)

Многие интерактивные элементы press feedback **не имеют**: вторичные кнопки/чипы (`.feed-cat-chip` — есть `transition`, но нет `:active`-эффекта), переключатели разделов (`.seg-btn`), элементы выбора (`.audience-row`, `.push-template`, `.consent-check`), кнопки-иконки (`.cs-add`), карточки в шапке профиля, элементы записи на слот и т. п. — точный список финализирует архитектор.

### Haptic feedback — отсутствует
`navigator.vibrate(...)` в коде не используется. На мобильных Android клиент не получает тактильного подтверждения важных действий (подтверждение записи, выбор слота/мастера/услуги, отправка формы).

### Бейдж уведомлений — статичный
- Шаблон в шапке: `prototype.html:44-46` — `<button class="inbox-bell-btn">…<span class="inbox-badge" id="inboxBadge">`.
- Логика показа: `modules/inbox.js:12-26` — `updateInboxBadge()` ставит `display:flex` при `count > 0`, иначе `display:none`. Текст — `count > 9 ? '9+' : count`.
- Стиль `.inbox-badge` (`style.css:890-902`) — красный кружок 18×18 с белой обводкой; никаких анимаций нет. Клиент может пропустить, что есть непрочитанное.

## Целевое поведение (черновик — финализирует BA/архитектор)

### 1. Единый press feedback
- Все интерактивные элементы (кнопки, карточки, чипы, переключатели, элементы выбора в списках) реагируют на нажатие единообразно: лёгкое сжатие + лёгкое потемнение/прозрачность, длительность ~80–120 мс.
- Параметры (точные значения scale/opacity/duration) — за архитектором; ориентир: уже используемые в проекте `scale(0.97)` для карточек и `opacity:0.85` для кнопок.
- Реализация — чистый CSS (`:active`), без JS, без новых зависимостей.
- Сохранить уже существующие эффекты, где они есть; не «переделывать ради переделывания» — только унифицировать там, где сейчас ничего нет, и привести в один стандарт явные несоответствия.

### 2. Haptic feedback на ключевых действиях
- Короткая вибрация `navigator.vibrate(<10–15 мс>)` на:
  - Подтверждении записи (финальный шаг бронирования).
  - Выборе слота/мастера/услуги (тап по карточке-варианту).
  - Отправке формы (push-уведомление администратором, публикация поста, оценка/отзыв).
- Точный перечень действий и значения мс — за архитектором (ОВ-1).
- Обёртка-утилита (например `tap()` или `hapticTap()`) в одном месте, чтобы вызовы были единообразны; gracefully no-op на устройствах без поддержки (iOS Safari без жеста, десктоп) — без ошибок в консоли.
- Не вибрировать на каждый тап подряд (только важные действия) — чтобы не раздражать.

### 3. Pulse-анимация бейджа колокольчика
- Когда `inboxBadge` виден (`count > 0`), бейдж пульсирует — деликатная CSS-анимация (например, scale 1.0 → 1.12 → 1.0 каждые ~1.4 с, или мягкое свечение). Точные параметры — за архитектором.
- Когда непрочитанных нет — бейдж скрыт (поведение `updateInboxBadge()` сохраняется), анимация не идёт.
- Анимация не должна «прыгать» при первом появлении бейджа и не должна давать заметной нагрузки на CPU (использовать `transform`/`opacity`, не `width`/`height`).
- Опционально (на усмотрение архитектора): один начальный «акцентный» бамп при появлении нового непрочитанного, дальше — спокойный фоновый пульс или пауза.

## Открытые вопросы (для BA/архитектора)
- **ОВ-1. Список действий с haptic.** Финальный перечень действий, которые вызывают `navigator.vibrate`, и точная длительность. Кандидаты: подтверждение записи, отмена записи, выбор слота, выбор мастера, выбор услуги, нажатие на FAB-кнопку, отправка формы. Нужна минимально достаточная подборка, чтобы не раздражать.
- **ОВ-2. Перечень элементов для press feedback.** Какие классы/селекторы получают новый `:active`-эффект, какие остаются как есть. Полный аудит интерактивов — за архитектором; критерий — «всё, что клиент трогает пальцем, должно реагировать».
- **ОВ-3. Pulse колокольчика — поведение во времени.** Постоянный мягкий пульс пока есть непрочитанные, или только один акцентный бамп при появлении и далее статика? Желательно — мягкий постоянный, чтобы бейдж не «терялся».
- **ОВ-4. Учёт настроек пользователя.** Учитывать ли `prefers-reduced-motion` (отключать pulse и press-анимации) и нужен ли где-то скрытый off-toggle для haptic. Базово: уважать `prefers-reduced-motion` — да, отдельный toggle — нет.
- **ОВ-5. iOS / Safari.** На iOS `navigator.vibrate` не работает. Поведение: тихий no-op без любых индикаторов недоступности (без сообщений пользователю).

## Ограничения
- Vanilla JS/HTML/CSS, без новых рантайм-npm-зависимостей.
- Изменения только в `docs/` — преимущественно `docs/style.css` (press feedback, pulse) + точечно `docs/modules/*.js` (вызовы `vibrate` в местах ключевых действий) + при необходимости `docs/prototype.html` (если потребуется новый класс/атрибут).
- Не ломать существующий press feedback там, где он уже есть и выглядит корректно.
- Не вводить тяжёлых JS-библиотек анимации; всё на CSS-transitions/`@keyframes`.
- Уважать `prefers-reduced-motion: reduce` — отключать pulse и анимации press feedback (или сводить к моментальному изменению без анимации).
- При изменении файлов из `APP_SHELL`/`sw.js` — поднять `CACHE_VERSION`.
- Минимальные изменения строго по согласованному объёму.

## Критерий готовности
- Все интерактивные элементы (по согласованному в BA/ADR перечню) реагируют на тап единообразным press feedback (видно глазом, тестируется в DevTools через эмуляцию touch).
- На ключевых действиях (по перечню в BA/ADR) срабатывает `navigator.vibrate` на устройствах с поддержкой; на iOS/десктопе — без ошибок и без визуальных отличий.
- Бейдж колокольчика мягко пульсирует, когда есть непрочитанные уведомления (`count > 0`), и остаётся скрытым при `count === 0`. Анимация не вызывает дёрганий и не сжирает CPU.
- На устройствах/браузерах с `prefers-reduced-motion: reduce` pulse и press-анимации не запускаются (или сводятся к мгновенной смене состояния).
- В консоли нет ошибок; PWA остаётся устанавливаемой; при изменении `APP_SHELL`/`sw.js` поднята `CACHE_VERSION`.
