# Требование 032: Кликабельные уведомления (диплинки), точечная отметка о прочтении, корзина + «Прочитать всё»

## Описание
Сделать уведомления в инбоксе (`s-inbox`) и системные push **кликабельными с переходом к целевой сущности**:
- клик по **напоминанию о визите** → открывает раздел **«Записи»**;
- клик по **пушу об услуге** → открывает **карточку услуги** (`openServiceCard(id)`);
- клик по уведомлению **без цели** (старые записи истории, произвольные промо-пуши) → **только помечает прочитанным**, никуда не ведёт.

Изменить логику «прочитано»: при заходе в инбокс через колокольчик уведомления **НЕ** помечаются прочитанными скопом — прочитанным становится **только то уведомление, которое пользователь открыл (кликнул)**.

В шапке экрана инбокса **заменить иконку «метёлки» на иконку корзины (мусорки)** — действие остаётся «удалить всю историю». Дополнительно добавить **отдельную текстовую кнопку «Прочитать всё» над списком**, которая помечает все уведомления прочитанными (без удаления).

Чтобы пуши об услуге могли вести к карточке, в **композер пуша в админке добавить выбор цели** (услуга / категория / без цели); цель кладётся в payload и сохраняется в истории уведомления.

## Текущее состояние (подтверждено кодом)
- **Уведомления в инбоксе НЕ кликабельны.** `renderInbox()` (`docs/modules/inbox.js:44-57`) рисует `.inbox-item` только с меню «три точки» (сделать непрочитанным / удалить). На самом блоке нет обработчика перехода.
- **Заход в инбокс помечает ВСЁ прочитанным.** `enterInbox()` (`inbox.js:96-100`) вызывает `markAllRead()` сразу при открытии экрана. Точечной отметки «прочитал именно это» при клике нет.
- **Кнопка в шапке = «метёлка», действие «очистить всю историю».** `prototype.html:874-876` — SVG метёлки, `onclick="clearInboxHistory()"` (`inbox.js:102-107`, с `confirm()`).
- **В записи уведомления нет поля цели.** Схема IDB-записи: `{id, title, body, icon, ts, read}` (`inboxStore.js:31-35`, инлайн-копия контракта в `sw.js:160-228`, `_IDB_*`). Поля типа/цели (target) нет.
- **Push payload не несёт цель.** Админская рассылка: `{title, body, icon}` (`sw.js:229`, `cf-worker/src/index.js:229`). Напоминания (cron): `{title:'Реснички · Напоминание о визите', body, icon}` (`cf-worker/src/index.js:363-367`) — при этом в обработчике **доступны** `rec.id` (record_id) и `clientId` (`index.js:342-345`), но в payload не кладутся.
- **`notificationclick` всегда открывает корень.** `sw.js:262-272`: `url = event.notification.data?.url || './'`; `data` сейчас всегда `{url:'./'}` (`sw.js:240`). Диплинка по типу/цели нет.
- **Диплинк на услугу уже реализуем.** `openServiceCard(id)` существует (`docs/modules/services.js:353`) и уже используется лентой (`docs/modules/feed.js:146-147`).
- **Раздел «Записи» существует** (таб, req-026; сохранённые записи `yc_records`, `docs/modules/booking.js`/`storage.js`). **Отдельного экрана «карточка визита» нет** — целью визита выбран именно раздел «Записи».
- **Композер пуша** (`docs/modules/admin.js:402-494`, `sendNewPush()` ~`admin.js:480`): шаблоны `PUSH_TEMPLATES` (`docs/modules/constants.js`), отправка текст+иконка. **Выбора цели нет.** Отправка на сервер — `sendAdminPush(...)` (`docs/modules/push.js`) → Worker `/send` (`index.js:222-272`).
- **Текущая версия:** `docs/version.json` → build 47, v1.27.0.

## Решения пользователя (зафиксированы, не пересматривать)
1. **Цель «визит» → раздел «Записи»** (отдельной детальной карточки визита не делаем).
2. **Админские рассылки → добавить выбор цели в композер** (услуга / категория / без цели); цель в payload.
3. **Кнопка «Прочитать всё» → текстовая кнопка над списком** (помечает все прочитанными, не удаляет).
4. **Клик по уведомлению без цели → пометить прочитанным, никуда не вести.**
5. **Метёлку заменить на корзину**, действие — «удалить всю историю» (как сейчас `clearInboxHistory`).

## Открытые вопросы (для BA/архитектора)
- **Формат поля цели в payload и в IDB-записи.** Предлагается `target: {type, id}` (например `{type:'service', id:123}`, `{type:'records'}`, `{type:'category', id}`, либо отсутствует/`null` = без цели). Согласовать имена типов и хранить ли вместо объекта строковый `url`-маршрут. Учесть, что контракт IDB продублирован в `inboxStore.js` и `sw.js` (`_IDB_*`) — синхронизировать; решить, нужен ли bump `_IDB_VERSION` (добавление поля в schemaless store обычно не требует).
- **Диплинк-маршрут для `notificationclick` при закрытом приложении.** Как передать цель в уже/заново открываемое окно: через `clients.openWindow('./?n=...')`/hash, через `postMessage` после `focus()`, через query-параметр, разбираемый при старте приложения. Определить единый механизм навигации по цели (общий хелпер, используемый и кликом в инбоксе, и `notificationclick`).
- **Категория как цель.** Есть ли в приложении переход «открыть категорию услуг» по id (фильтр категории в каталоге)? Если нет готового хелпера — это доп. объём; уточнить, входит ли категория в этот релиз или только услуга + визиты.
- **Привязка цели к напоминаниям (cron).** Подтвердить, что в payload напоминания добавляется `target:{type:'records'}` (или с `record_id`, если позже понадобится детальная карточка). Worker деплоится отдельно (`npx wrangler deploy`).
- **UI выбора цели в композере.** Вид контрола (выпадающий список услуг/категорий, поиск), обязательность, дефолт «без цели», связка с существующими шаблонами `PUSH_TEMPLATES` (привязывать ли цель к шаблону автоматически).
- **Поведение клика по уже открытому уведомлению.** Закрывать ли экран инбокса при переходе (через `go()` на целевой экран) — да; уточнить навигацию «назад».
- **Старые записи без поля цели.** Должны корректно трактоваться как «без цели» (клик только помечает прочитанным) — подтвердить отсутствие ошибок на записях старой схемы.
- **Видимость кнопки «Прочитать всё».** Показывать всегда или только при наличии непрочитанных; поведение при пустом списке; обновление бейджа (`updateInboxBadge`).
- **Системное уведомление ОС vs инбокс.** Для `notificationclick` цель берётся из `event.notification.data`; для клика в списке — из IDB-записи. Убедиться, что обе ветки используют один маршрутизатор цели.

## Ограничения
- Vanilla JS + HTML + CSS, **без рантайм-зависимостей**; код в `docs/` (`prototype.html`, `docs/modules/*.js`, `docs/style.css`, `docs/sw.js`) и `cf-worker/src/index.js`.
- **Контракт IDB продублирован** в `docs/modules/inboxStore.js` и `docs/sw.js` (`_IDB_*`) — любые изменения схемы записи синхронизировать в обоих местах.
- При изменении `APP_SHELL` / `sw.js` — **поднять `CACHE_VERSION`** в `docs/sw.js`.
- **Worker деплоится отдельно** от PWA (`cd cf-worker && npx wrangler deploy`); изменения payload напоминаний/рассылок согласовать с клиентом, чтобы старые и новые версии не падали.
- Экранирование текста (`esc()`) при рендере; цель/`id` — валидировать, не доверять внешнему вводу из payload.
- Не ломать текущий поток push: показ системного уведомления, подписку (`push.js`), бейдж (`updateInboxBadge`), меню «три точки», `PUSH_RECEIVED` (`inbox.js:109-119`).
- Минимальные изменения строго по объёму требования. Каталог/запись/профиль не трогать сверх необходимого.
- API-токены/секреты не хардкодить и не логировать.

## Критерий готовности
- Клик по уведомлению-напоминанию о визите (в инбоксе и по системному push) открывает раздел «Записи».
- Клик по уведомлению об услуге открывает карточку этой услуги (`openServiceCard`).
- Клик по уведомлению без цели только помечает его прочитанным и никуда не ведёт; ошибок на записях старой схемы нет.
- Заход в инбокс через колокольчик **больше не** помечает все уведомления прочитанными — прочитанным становится только открытое (кликнутое) уведомление; бейдж обновляется соответственно.
- В шапке инбокса вместо метёлки — иконка корзины; её действие — удалить всю историю (с подтверждением, как сейчас).
- Над списком есть кнопка «Прочитать всё», помечающая все уведомления прочитанными без удаления; бейдж обнуляется.
- В композере пуша админки можно выбрать цель (услуга / категория / без цели); цель доезжает в payload и сохраняется в истории уведомления, обеспечивая корректный диплинк у получателя.
- Текущий поток push (системное уведомление, подписка, тост, дедуп) работает; ошибок в консоли нет; при изменении `APP_SHELL`/`sw.js` поднята `CACHE_VERSION`; Worker задеплоен отдельно.
