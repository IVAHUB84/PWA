# Спецификация релиза v1.24.0

- **Дата планирования:** 2026-05-30
- **Тип релиза:** minor
- **Цели:** Сделать экран новой push-рассылки честным — убрать неработающие
  выбор аудитории и расписание, оставить единственный сценарий «Отправить всем
  сейчас» с фактическим числом получателей. Worker не изменяется.

> Обоснование типа релиза: пользователь-видимое поведение меняется (удаляются
> элементы UI), но публичный контракт (Worker API, формат подписок, схемы данных)
> не ломается, а старые `localStorage`-данные читаются толерантно. Это обратимо
> совместимое изменение фронтенда → **MINOR**.

## Источники

- BA-требования: `ba-req/ba-requirements-029.md` (источник
  `requirements/req-admin-push-honest-028.md`)
- ADR: `adrs/adr-026.md` (основной); контекст — ADR-004, ADR-006, ADR-007

## Объём релиза

### Задача 1: Удалить декоративный UI аудитории и расписания на экране `s-admin-push-new`

- **Что:** Убрать с экрана новой рассылки секцию «Аудитория» (три сегмента),
  секцию «Когда отправить» (переключатель «Сейчас/Запланировать») и фейковый блок
  расписания с захардкоженными датой/временем и нерабочей кнопкой «Изменить».
- **Где (файлы/модули):** `docs/prototype.html`, экран `id="s-admin-push-new"`,
  строки ~832-848.
  - Удалить подпись «Аудитория» (`<div class="label">Аудитория</div>` и её
    обёртку, стр. ~832).
  - Удалить три `div.audience-row` `#naud1`, `#naud2`, `#naud3` (стр. ~833-835).
  - Удалить подпись «Когда отправить» (стр. ~836).
  - Удалить `div.seg-control` с `#segNow` и `#segLater` (стр. ~837-840).
  - Удалить блок `#schedTimeRow` целиком (стр. ~841-848).
  - Оставить: «Шаблон» (chips), «Текст уведомления» (`#pushNewText`),
    «Предпросмотр» (`.push-preview`) и кнопку `#pushNewSendBtn`.
- **Контракт (UI):** на экране нет ни одного интерактивного элемента, не влияющего
  на фактический результат отправки; единственный сценарий — broadcast всем сейчас.
- **Критерии готовности:**
  - В DOM экрана `s-admin-push-new` отсутствуют `#naud1`, `#naud2`, `#naud3`,
    `#segNow`, `#segLater`, `#schedTimeRow` и подписи «Аудитория» / «Когда
    отправить» (КП-1, КП-2).
  - В разметке нет атрибутов `onclick="selectNewAudience(...)"` и
    `onclick="pushSchedToggle(...)"` (КП-3, КП-5).
- **Связь с ADR:** ADR-026, разделы «Решение», «Влияние на код».

### Задача 2: Текст кнопки отправки — «Отправить всем сейчас»

- **Что:** Кнопка `#pushNewSendBtn` всегда отображает честную надпись об отправке
  всем подписчикам немедленно (ранее текст менялся на «Запланировать»).
- **Где:** `docs/prototype.html` (стр. ~851, текст кнопки) — установить «Отправить
  всем сейчас».
- **Контракт (UI):** единственная кнопка действия с фиксированным текстом.
- **Критерии готовности:** кнопка отображает «Отправить всем сейчас» во всех
  состояниях; текст нигде программно не меняется (КП-3).
- **Связь с ADR:** ADR-026, раздел «Решение».

### Задача 3: Упростить `sendNewPush` до сценария немедленной отправки

- **Что:** Убрать из `sendNewPush` логику расписания и вычисление аудитории;
  оставить только немедленную отправку через `sendAdminPush` и тост по `res.sent`.
- **Где:** `docs/modules/admin.js`, функция `sendNewPush` (стр. ~481-513).
  - Удалить чтение `isScheduled` (`#segLater`), чтение `audience` из выбранной
    `.audience-row.sel`.
  - При сохранении кампании в `yc_push_campaigns` не выставлять `scheduled:true`
    (поле либо опускается, либо `false`); поле `audience` фиксировать как
    `'Все клиенты'`.
  - Удалить ветку `if (isScheduled) { ... }`; оставить только ветку с
    `sendAdminPush(title, text)`.
- **Контракт (поведение):** после отправки администратор видит тост с фактическим
  числом получателей на основе `res.sent` (существующая логика:
  `Отправлено ${res.sent} уведомл. ✓`, `Отправлено: 0 — нет подписчиков`,
  ошибка при `!res.ok`).
- **Критерии готовности:**
  - После отправки показывается фактическое число получателей из `res.sent`
    (КП-4).
  - Ни в одном состоянии не показывается адресат/время, не соответствующие
    broadcast всем сейчас (КП-6).
- **Связь с ADR:** ADR-026, разделы «Решение», «Решение по блоку Запланированные».

### Задача 4: Убрать рендер блока «Запланированные» и скорректировать очистку

- **Что:** Перестать отображать секцию «Запланированные» в истории кампаний;
  «Очистить всё» очищает все кампании.
- **Где:** `docs/modules/admin.js`.
  - `renderAdminPush` (стр. ~333-370): удалить переменную `sched` и весь блок
    `if (sched.length) { ... }`; оставить рендер отправленных кампаний.
  - `clearAllSentPush` (стр. ~378-382): заменить `filter(c => c.scheduled)` на
    очистку всех кампаний (сохранять пустой массив).
- **Контракт (данные):** старые записи `yc_push_campaigns` со `scheduled:true`
  читаются толерантно — не отображаются, не вызывают ошибок; принудительная
  миграция не выполняется.
- **Критерии готовности:**
  - Блок «Запланированные» не появляется даже при наличии старых
    `scheduled`-кампаний в `localStorage` (КП-8, НФТ-6).
  - «Очистить всё» очищает всю видимую историю; повторный рендер показывает
    пустое состояние «Рассылок пока нет».
  - Создание/удаление кампаний и список истории работают (КП-8).
- **Связь с ADR:** ADR-026, раздел «Решение по блоку Запланированные».

## Изменения публичного контракта

- **Worker API (`cf-worker`):** не изменяется (КП-7). Контракт `handleSend`
  (broadcast по `sub:` с фильтром `promo`) остаётся как есть.
- **Формат подписок / VAPID / `prefs`:** не изменяется.
- **`localStorage` `yc_push_campaigns`:** формат записи совместим; новые записи не
  имеют `scheduled:true`. Старые записи со `scheduled:true` читаются толерантно
  (игнорируются на рендере, не мигрируются). Миграционный путь не требуется.
- **UI экрана `s-admin-push-new`:** удаляются элементы — несовместимое для UI,
  совместимое для данных и API.

## Риски и митигации

- **Риск:** регрессия отправки push (случайно сломать рабочий broadcast при
  упрощении `sendNewPush`). → **Митигация:** не трогать вызов `sendAdminPush` и
  обработку `res`; изменения ограничить удалением ветки `isScheduled` и чтения
  аудитории; ручная проверка отправки тестовому подписчику (КП-4).
- **Риск:** мёртвые ссылки `onclick` в HTML после неполной чистки (элемент
  удалён, обработчик остался, или наоборот). → **Митигация:** удалять HTML-блоки
  и соответствующие функции/экспорты вместе; `npm run lint` + Grep по
  `selectNewAudience`/`pushSchedToggle`/`naud`/`segNow`/`segLater`/`schedTimeRow`
  должны давать ноль совпадений в `docs/` (КП-3, КП-5).
- **Риск:** старые `scheduled`-кампании в `localStorage` у текущих пользователей
  ломают рендер истории. → **Митигация:** толерантное чтение — `renderAdminPush`
  не рендерит scheduled-блок; `clearAllSentPush` очищает всё. Принудительная
  миграция не выполняется (ADR-026). Проверка с подготовленной старой записью.
- **Риск:** забыть поднять `CACHE_VERSION` → пользователи получают старый
  `prototype.html` из кэша. → **Митигация:** Задача 7 обязательна; проверка
  `CACHE_VERSION === 'v79'`.
- **Риск:** оставленная ссылка на `updatePushAudience` в `app.js` → ошибка в
  консоли при входе на экран. → **Митигация:** Задача 6 (удалить импорт + вызов),
  проверка консоли.
