# Спецификация релиза v1.5.1

- **Дата планирования:** 2026-05-28
- **Тип релиза:** patch (багфикс доставки push; публичный контракт приложения не ломается; меняется внутренний контракт `/unsubscribe` клиент↔Worker и схема ключей KV — оба под нашим контролем)
- **Цели:** Push доставляется на **все** устройства одного клиента (исправлен корень: на iPhone push не приходил, потому что вход с десктопа затирал его подписку). Несколько устройств клиента сосуществуют в KV; адресная отправка, broadcast и cron-напоминания достигают всех устройств; счётчик доставленных достоверен; доставка диагностируема.

## Источники

- BA-требования: `ba-req/ba-requirements-006.md` (объём сужен пользователем до починки доставки; открытые вопросы BA №1, №2, №3, №5, №6, №7 — вне объёма, не блокируют)
- Исходное требование с подтверждённым диагнозом: `requirements/req-admin-push-006.md` (раздел «Диагностика — подтверждено в ходе разбора»)
- ADR: `adrs/adr-006.md`

## Объём релиза

> Все задачи реализуют **только** починку доставки (мультидевайс-подписки). Сегменты аудитории, серверное планирование, история рассылок, push конкретному клиенту из UI, изменения достоверности обратной связи сверх корректного `sent`/`failed` — **вне объёма** (см. ADR-006, «Границы решения»).

### Задача 1: Хелпер хеша endpoint и схема ключей KV (Worker)

- **Что:** ввести детерминированный хеш endpoint и перейти на ключ KV `sub:${clientId}:${endpointHash}` вместо `sub:${clientId}`, чтобы устройства одного клиента не затирали друг друга.
- **Где (файлы/модули):** `cf-worker/src/index.js`.
- **Контракт:**
  - Хелпер `endpointHash(endpoint): Promise<string>` — стабильный base64url-хеш строки `endpoint` через `crypto.subtle.digest('SHA-256', …)` и существующий `b64url(...)`. Без новых зависимостей. Один и тот же endpoint → один и тот же хеш (идемпотентность ключа).
  - Формат ключа подписки: `sub:${clientId}:${endpointHash}`.
  - Префикс выборки подписок клиента: `sub:${clientId}:`. Префикс broadcast: `sub:` (без изменений по смыслу).
- **Критерии готовности:**
  - `endpointHash` детерминирован и не использует внешних зависимостей.
  - Все места, читающие/пишущие подписки (Задачи 2–5), используют новую схему ключей.
- **Связь с ADR:** ADR-006, Вопрос 1 (1A).

### Задача 2: `handleSubscribe` — запись по устройству, идемпотентность (Worker)

- **Что:** записывать подписку под пер-устройственный ключ; повторная подписка того же устройства обновляет свою запись, не плодя дубли.
- **Где (файлы/модули):** `cf-worker/src/index.js`, `handleSubscribe`.
- **Контракт:**
  - Тело запроса `/subscribe` — без изменений: `{ subscription, clientId, phone }`. Валидация прежняя (`subscription.endpoint` и `clientId` обязательны).
  - Запись: `SUBSCRIPTIONS.put('sub:${clientId}:${await endpointHash(subscription.endpoint)}', JSON.stringify({ subscription, clientId: String(clientId), phone: phone || '', endpoint: subscription.endpoint, ts: Date.now() }))`.
  - Значение хранит `endpoint` (для логирования/диагностики) в дополнение к `subscription`.
  - Ответ — без изменений: `{ ok: true }`.
- **Критерии готовности:**
  - Две подписки с разными endpoint одного `clientId` создают **две** записи в KV (не затирают друг друга).
  - Повторная подписка с тем же endpoint обновляет **ту же** запись (один ключ, без дубля).
  - Внешний контракт `/subscribe` (тело, ответ) не изменился.
- **Связь с ADR:** ADR-006, Вопрос 1 (1A).

### Задача 3: `handleSend` — отправка на все подписки клиента / broadcast, логирование, очистка по 410 (Worker)

- **Что:** при `targetClientId` отправлять на **все** подписки клиента; при broadcast — на все подписки (как сейчас); логировать хост endpoint + статус по каждой отправке; при 410 удалять конкретный ключ устройства.
- **Где (файлы/модули):** `cf-worker/src/index.js`, `handleSend`.
- **Контракт:**
  - Авторизация (`X-Admin-Secret`), валидация (`title`, `body`), формирование `payload` — без изменений.
  - **Target-ветка** (`targetClientId` задан): перебрать `SUBSCRIPTIONS.list({ prefix: 'sub:${targetClientId}:', cursor, limit: 100 })` (с пагинацией по cursor, как в broadcast); для каждой записи `sendOne`; считать `sent`/`failed`. Если подписок нет — `sent: 0, failed: 0` (а не `error: 'not_found'`; допускается сохранить `not_found` только если у клиента **нет ни одной** подписки — на усмотрение реализации, но без падения).
  - **Broadcast-ветка** (нет `targetClientId`): `list({ prefix: 'sub:', … })` — по смыслу как сейчас (теперь покрывает все устройства).
  - Для каждой отправки: `console.log` (или `console.warn` при не-2xx) строки вида `[send] host=<new URL(endpoint).host> status=<status>`. **Полный endpoint в лог не писать.**
  - При `status === 410` — удалять **конкретный** ключ устройства (`key.name`), не всю выборку клиента.
  - Толерантно читать значения старого формата (`sub:${clientId}` без хеша) — старая запись валидна как одна подписка.
  - Ответ — `{ ok: true, sent, failed }` (как сейчас); `sent` теперь честно считает по устройствам.
- **Критерии готовности:**
  - Адресная отправка клиенту с двумя устройствами даёт `sent: 2` (оба endpoint получили).
  - В логах Worker видны хост и статус по каждой подписке; полный endpoint не логируется.
  - 410 по одной подписке удаляет только её ключ; остальные подписки остаются.
  - Broadcast по-прежнему перебирает все подписки с пагинацией.
- **Связь с ADR:** ADR-006, Вопрос 2 (2A), Вопрос 5 (5A), Вопрос 6 (6A — толерантное чтение).

### Задача 4: `handleCron` — напоминание на все подписки клиента (Worker)

- **Что:** напоминание о завтрашней записи отправлять на **все** подписки клиента, а не на одну.
- **Где (файлы/модули):** `cf-worker/src/index.js`, `handleCron`.
- **Контракт:**
  - Логика выборки записей из YCLIENTS, фильтрация `!deleted`, формирование `payload` (заголовок/тело/время) — **без изменений**.
  - Вместо `get('sub:${clientId}')` — `list({ prefix: 'sub:${clientId}:' })`; `sendOne` на каждую подписку клиента.
  - При 410 — удалить конкретный ключ устройства; логировать хост+статус (как в Задаче 3).
  - Толерантно читать старый формат значения.
- **Критерии готовности:**
  - Клиент с записью на завтра и двумя устройствами получает напоминание на оба.
  - Механизм напоминаний (когда, о чём, тексты) не изменён — изменён только перебор подписок.
  - 410 в cron удаляет только конкретное устройство.
- **Связь с ADR:** ADR-006, Вопрос 3 (3A), Вопрос 5 (5A).

### Задача 5: `handleUnsubscribe` — отписка конкретного устройства по endpoint (Worker)

- **Что:** удалять подписку только того устройства, которое отписывается (по endpoint), не задевая другие устройства клиента.
- **Где (файлы/модули):** `cf-worker/src/index.js`, `handleUnsubscribe`.
- **Контракт:**
  - Тело запроса `/unsubscribe`: `{ clientId, endpoint }`.
  - Валидация: при отсутствии `clientId` **или** `endpoint` → `{ error: 'missing_fields' }`, 400. (Безопаснее прежнего «снести по clientId».)
  - Удаление: `SUBSCRIPTIONS.delete('sub:${clientId}:${await endpointHash(endpoint)}')` — один ключ.
  - Ответ — `{ ok: true }` (как сейчас).
- **Критерии готовности:**
  - Отписка одного устройства удаляет только его запись; подписки других устройств клиента сохраняются.
  - Запрос без `endpoint` возвращает `missing_fields` (не удаляет ничего).
- **Связь с ADR:** ADR-006, Вопрос 4 (4A).

### Задача 6: `unsubscribePush` — передача endpoint (клиент)

- **Что:** клиент должен передавать endpoint текущей подписки в `/unsubscribe`.
- **Где (файлы/модули):** `docs/modules/push.js`, `unsubscribePush`.
- **Контракт:**
  - Получить текущую подписку и её endpoint из `reg.pushManager.getSubscription()` **до** вызова `sub.unsubscribe()` (после `unsubscribe()` endpoint недоступен).
  - В теле запроса `/unsubscribe` передавать `{ clientId: String(sess.client_id), endpoint: sub.endpoint }`.
  - Если активной подписки нет (`sub` отсутствует) — `/unsubscribe` не вызывать (нечего отписывать); локальный `_KEY_SUB` убрать как сейчас.
  - `subscribePush` — **без изменений** (endpoint уже внутри `subscription.toJSON()`).
- **Критерии готовности:**
  - `unsubscribePush` шлёт `endpoint` в теле; порядок: прочитать endpoint → вызвать серверный `/unsubscribe` → локальный `sub.unsubscribe()` (или согласованный порядок, при котором endpoint гарантированно прочитан до локальной отписки).
  - При отсутствии подписки сетевой запрос не уходит.
- **Связь с ADR:** ADR-006, Вопрос 4 (4A).

### Задача 7: Поднять CACHE_VERSION и версию сборки

- **Что:** обновить кэш SW (изменён `push.js` в `APP_SHELL`) и версию проекта.
- **Где (файлы/модули):** `docs/sw.js`, `docs/version.json`.
- **Контракт:**
  - `docs/sw.js`: `CACHE_VERSION` `v38` → `v39`. Состав `APP_SHELL` **не меняется** (новых файлов нет).
  - `docs/version.json`: `build` 6 → 7, `version` → `v1.5.1`; `date`/`time`/`sha` обновляются при релизе мейнтейнером.
- **Критерии готовности:** `CACHE_VERSION === 'v39'`; `version.json.version === 'v1.5.1'`; состав `APP_SHELL` не изменён.
- **Связь с ADR:** ADR-006, Вопрос 7 (7A).

## Изменения публичного контракта

- **Worker `/unsubscribe` (внутренний контракт клиент↔Worker):** тело меняется с `{ clientId }` на `{ clientId, endpoint }`; без `endpoint` → `missing_fields` (400). Клиент (`push.js`) и Worker обновляются синхронно в этом релизе.
- **Worker `/subscribe`, `/send` (внешне):** тело и ответы **не меняются**. Меняется только внутренняя схема ключей KV (`sub:${clientId}` → `sub:${clientId}:${endpointHash}`) и то, что `/send` доставляет на все устройства.
- **KV `SUBSCRIPTIONS`:** новая схема ключей. **Миграция не выполняется** (ADR-006, Вопрос 6A): старые ключи `sub:${clientId}` самоустаревают (переподписка при логине пишет новый ключ; 410-очистка точечно удаляет мёртвые; допустим разовый ручной сброс неймспейса администратором). Чтение значения толерантно к обоим форматам.
- **Приложение (PWA):** публичный контракт (навигация, экраны, сценарий записи) не меняется. `subscribePush`/`unsubscribePush` остаются на `window`.
- **Криптография Web Push, VAPID, `/review`, манифест, иконки:** без изменений.

## Тесты

Минимальный набор, обязательный в этом релизе. Стиль — как существующие Node-тесты (`post-comment-worker.test.js`): запуск через `node`, моки KV/fetch, без рантайм-зависимостей и реальной сети. Worker — ES-модуль (`cf-worker/package.json` `"type": "module"`), его функции тестируемы при экспорте/инъекции зависимостей.

Файл — например `cf-worker/test/subscriptions.test.js` (или `docs/`-стиль, на усмотрение реализации, но физически добавлен):

- **Схема ключей / идемпотентность подписки:**
  - Две подписки с разными endpoint одного `clientId` создают два разных ключа `sub:${clientId}:…` (мок KV `put` фиксирует ключи).
  - Повторный `put` с тем же endpoint → тот же ключ (один ключ в моке KV, без дубля).
- **`handleSend` target → все устройства:**
  - При двух подписках клиента `sendOne` вызывается дважды; `sent === 2`.
  - При статусе 410 от одной подписки удаляется **только** её ключ (мок KV `delete` получает один конкретный ключ); вторая подписка остаётся.
  - В лог пишется хост (а не полный endpoint) — проверить, что полный endpoint не попадает в залогированную строку.
- **`handleUnsubscribe` по endpoint:**
  - С `{ clientId, endpoint }` удаляется один ключ `sub:${clientId}:${hash}`; другие подписки клиента не трогаются.
  - Без `endpoint` → `missing_fields` (400), `delete` не вызывается.
- **`handleCron` → все устройства клиента** (если cron-логика вынесена/инъектируема): клиент с двумя подписками получает `sendOne` дважды.
- **Толерантность к старому формату** (Вопрос 6A): значение `sub:${clientId}` без хеша корректно читается как одна подписка и не ломает отправку.

> Тесты в этой спецификации **не запускаются** — запуск на этапе `/release` тестировщиком. Требуется лишь физическое наличие тестов, покрывающих критерии выше.

## Риски и митигации

- **Риск:** рассинхрон клиента и Worker — установленный клиент шлёт `/unsubscribe` без endpoint. → **Митигация:** подъём `CACHE_VERSION` v38→v39 (Задача 7) обновляет `push.js` у клиентов; Worker при отсутствии endpoint безопасно отвечает `missing_fields` (ничего не удаляет), а не сносит подписки.
- **Риск:** старые ключи `sub:${clientId}` остаются «осиротевшими» в KV. → **Митигация:** толерантное чтение старого формата + точечная очистка по 410; переподписка при логине пишет новый ключ; допустим разовый ручной сброс KV.
- **Риск:** target-ветка `/send` при отсутствии подписок у клиента ведёт себя иначе, чем прежде (`not_found`). → **Митигация:** допускается `sent:0, failed:0` либо сохранение `not_found` без падения; критерий — не ломать отправку и не показывать ложный успех.
- **Риск:** `crypto.subtle.digest` недоступен/иначе ведёт себя в среде Worker. → **Митигация:** Web Crypto уже используется в Worker (`encryptPayload`, `vapidToken`) — среда поддерживает; хеш покрыт тестом детерминированности.
- **Риск:** Worker задеплоен раньше/позже PWA → краткое окно несовместимости `/unsubscribe`. → **Митигация:** безопасный отказ `missing_fields` (не деструктивный); порядок деплоя зафиксирован ниже.
- **Риск:** забыть отдельный деплой Worker. → **Митигация:** порядок деплоя явно прописан; чек-лист содержит пункт деплоя Worker.

## Порядок деплоя

1. **Worker** (бэкенд новой схемы): `cd cf-worker && npx wrangler deploy` (требует `CLOUDFLARE_API_TOKEN`). Сначала Worker — чтобы новая схема ключей и новый `/unsubscribe` уже работали, когда клиенты обновятся.
2. **PWA:** `git push origin main` → GitHub Pages CD публикует `docs/` (~1 мин). Новый SW (`v39`) обновит `push.js` у клиентов.
3. **Проверка:** на реальном устройстве — вход одним клиентом с десктопа и iPhone; отправка push; push приходит на **оба** устройства; счётчик доставленных соответствует числу устройств. DevTools → Application → Service Workers: SW `v39` активен; ноль ошибок в консоли. Логи Worker (`wrangler tail`): видны хост+статус по каждой подписке.

## Примечания по безопасности

- Секреты (`ADMIN_SECRET`, `VAPID_PRIVATE_KEY_JWK`, `YC_TOKEN`, `YC_USER_TOKEN`) остаются **только** в Worker; не коммитятся. `/send` остаётся под `X-Admin-Secret` (без секрета → 401).
- Полный endpoint подписки в логи **не пишется** — логируется только хост (endpoint считается чувствительным).
- Захардкоженный YCLIENTS-токен в `prototype.html` — известный долг (CLAUDE.md), в этом релизе не трогается.
