# Спецификация релиза v1.12.0

- **Дата планирования:** 2026-05-29
- **Тип релиза:** minor (доработка функциональности — полная галерея фото услуги, скругления, peek-аффорданса, зацикленная карусель, stale-while-revalidate для хоста фото; добавляется один эндпойнт чтения каталога `GET /company/{company}/services/` для слияния поля `photos`, но публичные контракты PWA/CLI/схем данных и поток записи не меняются, рантайм-зависимостей нет. Последний тег — v1.11.0)
- **Цели:** (1) Доставить в детальную карточку услуги `s-service` **все** фото из галереи YCLIENTS (до 8), а не одно. (2) Сделать галерею премиальной и очевидно листаемой: скругления, peek соседних слайдов, зацикливание. (3) Обеспечить автоматическую свежесть фото из YCLIENTS без переустановки приложения (stale-while-revalidate).

## Источники

- BA-требования: `ba-req/ba-requirements-015.md` (статус «согласовано, готов к /adr»; все блокирующие вопросы решены; §8 №1 «зацикливание» и №10 «источник галереи» — инженерные развилки, закрыты ADR-014)
- ADR: `adrs/adr-014.md` (источник галереи G1, скругления/peek H1, зацикливание transform+клоны I1, SWR для хоста фото J1); дорабатывает `adrs/adr-013.md` (детальная карточка `s-service`, карусель E1 — механизм заменяется на transform-ленту); переиспользуются `adrs/adr-012.md` (диапазон цен `priceStr`, не меняется), `adrs/adr-001.md` (категорийные плейсхолдеры)

## Объём релиза

> **Суть релиза:** карусель фото на детальной карточке услуги `s-service` дорабатывается до полноценной галереи: (а) каталог дочитывает **всю** галерею услуги из management-эндпойнта и сливает её точечно в поле `photos`; (б) изображения скругляются (~16px), соседние слайды выглядывают по краям (peek ~14px), стрелок нет; (в) карусель зацикливается в обе стороны через transform-ленту с клонами крайних слайдов, индикатор-точки синхронизирован с активным слайдом; (г) байты фото с `assets.yclients.com` кэшируются по стратегии stale-while-revalidate. Поведение 1 фото (одиночное изображение) и 0 фото (плейсхолдер) сохраняется, добавляется только скругление.
>
> **Вне объёма:** каталог услуг (вёрстка, поиск, фильтр категории, баннер предвыбранного мастера, состав полей кроме точечного слияния `photos`), обложка карточки каталога (продолжает использовать `photos[0]`), выбор мастера, расчёт слотов, бронирование, подтверждение; логика цен ADR-012 (`priceStr` только переиспользуется); состав детальной карточки сверх галереи (ADR-013); зум/лайтбокс; автопрокрутка; счётчик «N/M»; глобальная смена кэш-стратегии SW для всех кросс-origin ресурсов; перенос захардкоженного YCLIENTS-токена; любые операции с веткой `backup`.

### Задача 1: Дочитка полной галереи фото из management-эндпойнта и слияние в `photos`

- **Что:** при разовой загрузке каталога дополнительно запросить `GET /company/{company}/services/`, построить словарь `id → массив URL галереи` и перезаписать поле `photos` каждой услуги полной галереей, не трогая остальные поля.
- **Где (файлы/модули):** `docs/app.js` (`initApp`, `app.js:160-179`); переиспользуется `serviceImageUrls` из `docs/modules/serviceImages.js` (без изменений).
- **Контракт:**
  - В `Promise.all` (`app.js:161-165`) добавляется 4-й запрос: `YC.get(`/company/${YC.company}/services/`)`.
  - Из ответа строится словарь `gallery[String(svc.id)] = serviceImageUrls(svc.image_group)`; обработка ответа защищённая — при `success === false`/отсутствии `data`/исключении словарь остаётся пустым (загрузка каталога не прерывается).
  - После маппинга `SERVICES_DATA` (через `setServicesData`, `app.js:169-177`) для каждой услуги: если `gallery[s.id]` — непустой массив, то `s.photos = gallery[s.id]`; иначе `s.photos` остаётся прежним (значение из `/services/`).
  - Прочие поля маппинга (`id, name, cat, dur, price_min, priceStr, comment`) и агрегация цен (ADR-012) — **без изменений**.
  - Новый запрос — на `api.yclients.com` (тот же `Bearer+User` токен, та же база `api.yclients.com/api/v1` через `YC.get`), помечен `NETWORK_ONLY_HOSTS` → не кэшируется.
- **Критерии готовности:**
  - Услуга id 29196117 (контрольный пример, 3 фото в галерее) на детальной карточке показывает все 3 фото (карусель), а не одно. (КП-0, ФТ-0)
  - Название, категория, длительность, диапазон цен per-master, описание услуги и дальнейший поток записи (выбор мастера → слоты → бронирование → подтверждение) работают как прежде. (КП-16, ФТ-0, НФТ-5)
  - При недоступности/пустом ответе management-эндпойнта каталог не ломается — услуги показываются с одиночными фото из `/services/` (graceful). (риск ADR-014)
  - На открытие каждой карточки доп. запросов за списком фото нет — список из уже загруженного объекта услуги; дочитка галереи — разово при старте. (КП-11, ФТ-12)
- **Связь с ADR:** ADR-014, Вопрос G (Вариант G1).

### Задача 2: Скругления и peek-аффорданса (CSS)

- **Что:** скруглить изображения галереи (~16px) и реализовать peek (соседние слайды выглядывают по краям ~14px) чистым CSS; сохранить точки-индикаторы; без стрелок.
- **Где (файлы/модули):** `docs/style.css` (`.svc-card-gallery`, `.svc-card-single-img`, `.svc-card-placeholder`, `.svc-carousel*`, `style.css:809-848`).
- **Контракт:**
  - `border-radius` ~16px (под радиус карточек проекта) применяется к **каждому** изображению/слайду карусели, к `.svc-card-single-img` и к `.svc-card-placeholder` (визуальная консистентность; при peek у соседнего слайда тоже видны скруглённые края).
  - Peek реализуется **без JS**: слайд получает ширину `< 100%` (~`calc(100% - 28px)`), контейнер карусели — горизонтальный `gap` и боковой `padding`, так что по краям выглядывает ~14px соседнего слайда с каждой стороны; подсказка присутствует и при ровно 2 фото.
  - Высота изображений ~280px и `object-fit: cover` сохраняются; фото не должно критично мылить/обрезаться в новой геометрии (если визуально неприемлемо — эскалация, §8 №5).
  - Точки-индикаторы (`.svc-carousel-dots`, `.svc-carousel-dot`, `.active`) сохраняются; стрелки не добавляются.
- **Критерии готовности:**
  - Изображения галереи, одиночное фото и плейсхолдер имеют скруглённые края, согласованные с интерфейсом. (КП-1, ФТ-1, ФТ-6, КП-5)
  - При 2+ фото визуально понятно, что карусель листается: соседние слайды выглядывают (peek ~14px) + точки; стрелок нет; подсказка есть и при ровно 2 фото. (КП-2, ФТ-2, КП-10, ФТ-11)
- **Связь с ADR:** ADR-014, Вопрос H (Вариант H1).

### Задача 3: Зацикленная карусель на transform-ленте с клонами + синхронизация индикатора

- **Что:** заменить механизм карусели (для N≥2 фото) с нативного scroll-snap на JS-ленту `transform: translateX` с клонами крайних слайдов; обеспечить бесшовное зацикливание в обе стороны и синхронизацию точек.
- **Где (файлы/модули):** `docs/modules/services.js` (`openServiceCard`, ветка `photos.length >= 2`, `services.js:167-213`); CSS-поддержка в `docs/style.css` (Задача 2).
- **Контракт:**
  - Лента слайдов рендерится с клонами крайних: `[clone(last), s0, s1, …, sN-1, clone(first)]`; лента двигается через `transform: translateX` с CSS-`transition`.
  - Текущий **логический** индекс (0..N-1) ведётся в JS-переменной (не вычисляется из `scrollLeft`).
  - Свайп на тач отслеживается лёгкими `touchstart`/`touchmove`/`touchend` (порог сдвига + направление); десктоп — без обязательных стрелок (свайп/трекпад).
  - **Зацикливание:** переход вправо с последнего слайда ведёт на клон первого, влево с первого — на клон последнего; по `transitionend` выполняется **бесшовный сброс** позиции на реальный слайд **без анимации** (снять `transition`, выставить `translateX`, форсировать reflow, вернуть `transition`).
  - **Синхронизация точек:** активная точка подсвечивается по логическому индексу из JS, в том числе при зацикленном переходе через край; старый слушатель `scroll` (`services.js:208-211`) удаляется/заменяется.
  - Все URL фото и `alt` (`s.name`) выводятся через `esc()`; клоны строятся из тех же экранированных данных; `loading="lazy"` сохраняется; `<img onerror>` деградирует на категорийный плейсхолдер (как `services.js:169`).
  - Ветки **1 фото** (одиночное изображение, `services.js:178-183`) и **0 фото** (плейсхолдер, `services.js:184-188`) — **без изменений** (кроме скругления из Задачи 2): без ленты, клонов, точек, зацикливания.
  - Слушатели снимаются корректно при ре-рендере экрана (DOM `#serviceCardContent` пересоздаётся при каждом `openServiceCard`).
- **Критерии готовности:**
  - Свайп/переход с последнего фото ведёт на первое и с первого на последнее (зацикливание в обе стороны); листание не упирается на краях; переход через край визуально бесшовен (без заметного скачка). (КП-3, ФТ-3)
  - Индикатор текущего фото синхронизирован с активным слайдом, в том числе при зацикленном переходе через край. (КП-4, ФТ-4)
  - При 2 фото карусель с зацикливанием работает так же, как при большем числе. (граничный сценарий BA §3)
  - При одном фото — одиночное скруглённое изображение без листалки/стрелок/точек/зацикливания; при отсутствии фото — категорийный плейсхолдер без листалки. (КП-5, ФТ-5, ФТ-6)
  - При битом URL фото слайд деградирует на категорийный плейсхолдер без поломки вёрстки. (КП-6, ФТ-7, НФТ-3)
  - Карусель и зацикливание работают на тач без ошибок в консоли; до 8 фото `loading="lazy"`, свайп без рывков. (КП-12, НФТ-2)
  - URL фото и `alt` экранируются (`esc()`); инъекция HTML невозможна. (КП-13, НФТ-6)
- **Связь с ADR:** ADR-014, Вопрос I (Вариант I1); дорабатывает ADR-013, Вопрос E (E1 → transform-лента).

### Задача 4: stale-while-revalidate для хоста фото `assets.yclients.com` в Service Worker

- **Что:** добавить в `sw.js` точечную ветку stale-while-revalidate для байтов фото услуг YCLIENTS, не затрагивая прочие кросс-origin ресурсы и оффлайн.
- **Где (файлы/модули):** `docs/sw.js` (`fetch`-обработчик, перед общей не-same-origin веткой `sw.js:108-118`; `CACHE_VERSION`, `sw.js:1`).
- **Контракт:**
  - Новая ветка размещается **до** общей не-same-origin ветки (`sw.js:108-118`).
  - Условие срабатывания: `url.hostname === 'assets.yclients.com'` И путь начинается с `main_service_image/` или `gallery_service_image/`.
  - Стратегия SWR: вернуть закэшированный ответ немедленно (если есть); параллельно в фоне `fetch(request)`, при `res.ok` положить клон в `RUNTIME_CACHE`; если кэша нет — вернуть сетевой ответ (и закэшировать); при сетевой ошибке — вернуть кэш (оффлайн-фолбэк).
  - Остальные кросс-origin ресурсы (EmailJS, jsdelivr) идут по **прежней** cache-first ветке; APP_SHELL (cache-first + фон), JS (network-first), `NETWORK_ONLY_HOSTS` (`api.yclients.com`), same-origin (network-first) — **без изменений**.
  - `CACHE_VERSION` поднимается `v57` → `v58` (правило проекта при изменении файлов APP_SHELL, включая `sw.js`).
- **Критерии готовности:**
  - Новое/изменённое фото услуги из YCLIENTS появляется у клиента автоматически без переустановки и ручной чистки кэша: при наличии кэша фото показывается мгновенно, свежее подтягивается в фоне, обновление видно при следующем открытии карточки. (КП-7, ФТ-8)
  - Ранее виденные фото доступны оффлайн; оффлайн-просмотр ранее открытых карточек не сломан. (КП-8, ФТ-9)
  - Изменённая стратегия применена **только** к `assets.yclients.com` (пути `main_service_image/`, `gallery_service_image/`); кэширование APP_SHELL, JS-модулей, EmailJS, jsdelivr и общий оффлайн-режим не изменены. (КП-9, ФТ-10)
  - PWA остаётся устанавливаемой, оффлайн не регрессирует; `CACHE_VERSION` поднята; новых рантайм-зависимостей нет. (КП-14, НФТ-1, НФТ-4)
- **Связь с ADR:** ADR-014, Вопрос J (Вариант J1).

## Изменения публичного контракта

- **Внешний API PWA / CLI / схемы БД:** не меняются.
- **Потребляемый YCLIENTS API:** добавляется один эндпойнт **чтения** — `GET /company/{company}/services/` (management; тот же токен/база) — исключительно для построения словаря галереи и слияния поля `photos`. Эндпойнты `/services/{company}`, `/service_categories/{company}`, `/book_staff/{company}` и поток записи — без изменений. Поле `photos` объекта услуги теперь может содержать **до 8** URL вместо одного (внутренний контракт `SERVICES_DATA`; обратно совместим — карусель/одиночное/плейсхолдер уже ветвятся по длине массива).
- **Миграционный путь:** не требуется (данные эфемерные, грузятся из API при старте).
- **Service Worker:** меняется кэш-стратегия для одного хоста (`assets.yclients.com`); `CACHE_VERSION` `v57 → v58` инвалидирует старые кэши при активации.

## Риски и митигации

- **Риск:** management-эндпойнт `/company/{company}/services/` недоступен/иная форма/несовпадение `id` → **Митигация:** защищённый разбор через `serviceImageUrls()` (любая иная форма → `[]`), слияние только при непустом массиве, мягкая обработка `success === false`; каталог деградирует на одиночные фото. Подтвердить форму и совпадение `id` на живых данных. (Задача 1)
- **Риск:** видимый «скачок» при бесшовном сбросе позиции на краю transform-ленты → **Митигация:** сброс без анимации строго в `transitionend` (снять transition → переставить translateX → reflow → вернуть transition); протестировать на iOS/Android; при заметном скачке — эскалация. (Задача 3)
- **Риск:** peek + центрирование первого/последнего слайда (недокрут/полоса по краю) → **Митигация:** выверить боковой `padding`/ширину слайда; позиция в transform-ленте считается явно. (Задачи 2, 3)
- **Риск:** SWR-ветка перехватит лишние ресурсы или фото уйдёт в cache-first → **Митигация:** узкое условие (`hostname` + префикс пути), ветка строго **до** общей не-same-origin; проверить, что прочие CDN идут прежними ветками. (Задача 4, КП-9)
- **Риск:** старый кэш SW отдаёт прежнюю разметку/CSS/JS/SW после релиза → **Митигация:** поднять `CACHE_VERSION` `v57 → v58` при изменении `style.css`/`services.js`/`app.js`/`sw.js`. (Задача 4, КП-14)
- **Риск:** XSS через URL фото/`alt` → **Митигация:** все внешние данные через `esc()`, включая клоны. (Задача 3, КП-13)
