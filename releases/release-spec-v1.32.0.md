# Спецификация релиза v1.32.0

- **Дата планирования:** 2026-05-31
- **Тип релиза:** minor
- **Цели:** Заменить нативные `alert/confirm` в клиентских сценариях на согласованные in-app примитивы (toast + bottom-sheet), дать пользователю восстановиться после сетевой ошибки (повтор/звонок), показать индикатор офлайна и устаревших данных. Заложить переиспользуемые примитивы UI для v1.33/v1.34.

## Источники
- BA-требования: `ba-req/ba-requirements-037.md` (источник `requirements/req-inapp-feedback-036.md`)
- ADR: `adrs/adr-034.md`

## Объём релиза

### Задача 1: Чистая логика тайминга/выбора toast (`uiQueue.js` + тесты)
- **Что:** модуль чистых функций, определяющих какой toast активен и когда скрыть; таймер инжектируется (без реальных `setTimeout` и DOM).
- **Где:** новый `docs/modules/uiQueue.js`; тесты `docs/modules/uiQueue.test.js`.
- **Контракт:** длительности по умолчанию — `success`/`info` = 3000 мс, `error` = 5000 мс; функция выбора активного toast реализует вытеснение (новый заменяет текущий, см. ADR-034 Вопрос 2).
- **Критерии готовности:** unit-тесты проверяют выбор длительности по типу и логику вытеснения; модуль не импортирует DOM-API.
- **Связь с ADR:** ADR-034, Вопрос 2 / контракт примитивов.

### Задача 2: Примитивы UI (`ui.js`)
- **Что:** реализация `toast`, `confirmSheet`, `errorSheet`, `setOfflineBanner` поверх паттерна `services.js:_openCatFilter`.
- **Где:** новый `docs/modules/ui.js`; DOM-хосты в `docs/prototype.html`; стили в `docs/style.css`.
- **Контракт:** строго как в разделе «Нормативный контракт примитивов» ADR-034:
  - `toast(message, type='info', opts={})` — текст через `esc()`, авто-скрытие, `aria-live="polite"`, одиночный с вытеснением.
  - `confirmSheet({title,message?,confirmText,cancelText='Отмена',danger=false}) → Promise<boolean>` — закрытие по фону и кнопке отмены резолвит `false`.
  - `errorSheet({title,message,onRetry,phone?})` — «Повторить» закрывает sheet и зовёт `onRetry()`; «Позвонить» рендерится только при непустом `phone` (через `_telHref`); «Закрыть».
  - `setOfflineBanner(mode)` — `mode ∈ {'offline','stale',null}`; `offline` приоритетнее `stale`.
- **Критерии готовности:** все элементы используют только токены `var(--…)`; читаемы в светлой и тёмной теме; sheet строится так, что добавление Esc/фокус-ловушки в v1.33 не потребует менять сигнатуры; ни один параметр текста не вставляется в DOM без `esc()`.
- **Связь с ADR:** ADR-034, Вопросы 1, 3 / контракт примитивов.

### Задача 3: Замена нативных диалогов в `booking.js`
- **Что:** заменить `alert` на `toast`/`errorSheet` согласно карте замен.
- **Где:** `docs/modules/booking.js` строки 53, 84, 125, 226, 234; импорт `getCompanyContacts`/`_telHref` из `studio.js`.
- **Контракт:**
  - 53 → `toast('Услуги ещё загружаются…','info')`.
  - 84 (ошибка создания) → `errorSheet({onRetry:()=>startBooking(), phone})`, `message` из `r.meta?.message` через `esc()`.
  - 125 (ошибка переноса) → `errorSheet({onRetry:()=>startBooking(), phone})`.
  - 226 (ошибка отмены, HTTP≥400) → `errorSheet({onRetry:()=>confirmCancel(), phone})`.
  - 234 (нет соединения при отмене) → `errorSheet({onRetry:()=>confirmCancel(), phone})`.
  - `confirmCancel` успех (после `go('s-home')`) → `toast('Запись отменена','success')`.
  - Создание брони успех → без тоста (экран `s-confirm`).
  - `phone = getCompanyContacts().phone`.
- **Критерии готовности:** в `booking.js` не остаётся `alert/confirm`; `_bookInProgress`/`_rescheduleInProgress`/`_cancelInProgress` и переходы `go('s-confirm')`/`go('s-home')` не изменены; «Повторить» работает без потери контекста (`_rescheduleId`/`_cancelId`/поля брони не сброшены до успеха).
- **Связь с ADR:** ADR-034, «Точки замены нативных диалогов».

### Задача 4: Замена нативных диалогов в `review.js`
- **Что:** заменить `alert` на `toast`/`errorSheet`.
- **Где:** `docs/modules/review.js` строки 132, 150, 154.
- **Контракт:**
  - 132 (нет данных визита) → `toast(...,'error')` (без повтора — это валидация).
  - 150 (успех) → `toast('Спасибо! Ваш отзыв отправлен.','success')`, затем `go('s-home')`.
  - 154 (ошибка отправки) → `errorSheet({onRetry:()=>submitReview(), phone})`.
- **Критерии готовности:** в `review.js` нет `alert/confirm`; восстановление кнопки (`btn.disabled`, `origLabel`) не нарушено.
- **Связь с ADR:** ADR-034, «Точки замены».

### Задача 5: Офлайн-баннер и индикатор устаревших данных
- **Что:** показывать баннер «Нет соединения» по `navigator.onLine`/событиям `online`/`offline`; индикатор «Данные могли устареть» при гидрации из снимка + неуспехе фонового revalidate каталога.
- **Где:** `docs/app.js` (`initApp` и boot-секция), узел баннера в `docs/prototype.html`, стили в `docs/style.css`.
- **Контракт (ADR-034 Вопрос 4):**
  - На boot: если `!navigator.onLine` → `setOfflineBanner('offline')`.
  - Слушатели: `offline` → `setOfflineBanner('offline')`; `online` → `setOfflineBanner(null)` (затем при необходимости вернуть `'stale'`).
  - В `initApp`: запомнить, была ли успешная синхронная гидрация (снимок был непустым). Если каталог не обновился (`catch(e)` или `!svcRes.success`, т.е. `writeCatalogSnapshot` не вызван) и гидрация была — `setOfflineBanner('stale')`. При успешном обновлении — `setOfflineBanner(null)` (если онлайн).
- **Критерии готовности:** DevTools→Network→Offline показывает баннер «Нет соединения», возврат online — скрывает; при провале revalidate с показом кэша — индикатор «Данные могли устареть»; `offline` приоритетнее `stale`.
- **Связь с ADR:** ADR-034, Вопрос 4 / решение по баннеру.

### Задача 6: Service Worker и версия
- **Что:** включить новые модули в кэш, поднять версии.
- **Где:** `docs/sw.js`, `docs/version.json`.
- **Контракт:** добавить `./modules/ui.js` и `./modules/uiQueue.js` в `APP_SHELL`; `CACHE_VERSION` `v88` → `v89`; обновить `version` → `1.32.0` и инкремент `build` в `version.json` (выполняется на этапе `/release`).
- **Критерии готовности:** оба модуля в `APP_SHELL`; `CACHE_VERSION` = `v89`; PWA installable (Lighthouse).
- **Связь с ADR:** ADR-034, «Влияние на код».

## Изменения публичного контракта
- Внешний API YCLIENTS, CLI, схемы данных — без изменений.
- Меняется только воспринимаемый UX: клиентские нативные модалки заменены на in-app; добавлены баннеры. Миграция не требуется. Откат — `git revert` коммита релиза (CACHE_VERSION откатится вместе с `sw.js`).

## Тесты
Обязательны к появлению в этом релизе (автотесты — на чистую логику):
- `docs/modules/uiQueue.test.js` — выбор длительности по типу toast; логика вытеснения активного toast (КП-10).
- Тест экранирования: при формировании toast/sheet сообщение с HTML-спецсимволами проходит через `esc()` и не интерпретируется как разметка (КП-9). Допустимо в `uiQueue.test.js` либо отдельным `ui-escape.test.js` (проверка функции формирования текста без DOM).

## Риски и митигации
- **Риск:** «Повторить» теряет контекст брони/отмены. → **Митигация:** не сбрасывать `_rescheduleId`/`_cancelId`/поля брони до успеха; `onRetry` зовёт публичную точку входа (`startBooking`/`confirmCancel`/`submitReview`).
- **Риск:** двойная отправка при повторе. → **Митигация:** существующие флаги `_bookInProgress`/`_cancelInProgress` уже сбрасываются в ветке ошибки до показа sheet; не трогать их.
- **Риск:** телефона нет в данных филиала. → **Митигация:** `errorSheet` скрывает «Позвонить» при пустом `phone`; остальные действия доступны (ФТ-7).
- **Риск:** ложный индикатор «устарело» при медленной сети. → **Митигация:** триггер — только неуспех/исключение, не таймаут (ADR-034 Вопрос 4).
- **Риск:** XSS через `meta.message`. → **Митигация:** весь текст через `esc()`; покрыто тестом КП-9.
- **Риск:** забыли поднять CACHE_VERSION → старые модули из кэша. → **Митигация:** Задача 6, чек-лист.
