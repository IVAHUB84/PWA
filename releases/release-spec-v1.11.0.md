# Спецификация релиза v1.11.0

- **Дата планирования:** 2026-05-29
- **Тип релиза:** minor (новая функциональность — промежуточный экран детальной карточки услуги + реальное фото услуги на карточках каталога; публичные контракты YCLIENTS/CLI/схем данных не меняются; новых эндпоинтов и рантайм-зависимостей нет; дальнейший поток записи сохраняется. Последний тег — v1.10.0)
- **Цели:** (1) Дать клиенту изучить услугу перед записью — новый экран `s-service` с названием, описанием, ценой-диапазоном + длительностью и фотографиями работ (галерея), с кнопкой «Выбрать мастера». (2) Показывать в каталоге реальное фото услуги из YCLIENTS (первое из `image_group`) вместо категорийного плейсхолдера, когда фото есть.

## Источники

- BA-требования: `ba-req/ba-requirements-014.md` (открытые вопросы №1, №3, №4, №5 решены заказчиком 2026-05-29, §8; №2, №6, №7 — значения по умолчанию; остаётся технический риск к проверке: хранит ли YCLIENTS >1 фото на услугу)
- ADR: `adrs/adr-013.md` (отдельный экран `s-service`, рендер при тапе, маппинг `comment`/`image_group`, карусель на CSS scroll-snap); переиспользуется `adrs/adr-012.md` (диапазон цен — `priceStr`, не меняется)

## Объём релиза

> **Суть релиза:** между каталогом услуг (`s-services`) и выбором мастера (`s-masters`) вставляется промежуточный экран детальной карточки услуги `s-service`. Тап по услуге в обычном потоке открывает карточку; кнопка «Выбрать мастера» ведёт к `s-masters`. В режиме предвыбранного мастера карточка пропускается (тап → `s-slots`, как сейчас). Дополнительно: обложка карточки каталога берётся из первого фото `image_group` услуги.
>
> **Вне объёма:** логика выбора мастера, расчёта слотов, бронирования, подтверждения; состав/вёрстка каталога сверх замены источника обложки; логика цен ADR-012 (`_fmtPriceRange`, сбор по мастерам — только переиспользуется `priceStr`); карточка мастера `s-master`/`openMasterCard`; набор эндпоинтов YCLIENTS (новых запросов нет); зум/полноэкранный просмотр фото (лайтбокс); дозагрузка нескольких фото работ в YCLIENTS (отдельная контентная задача); захардкоженный YCLIENTS-токен (известный техдолг — не усугублять).

### Задача 1: Нормализация фото услуги — хелпер `serviceImageUrls`

- **Что:** добавить в `serviceImages.js` защищённый хелпер, нормализующий неоднородную/неполную `image_group` услуги в массив URL-строк.
- **Где (файлы/модули):** `docs/modules/serviceImages.js` (новый экспортируемый `serviceImageUrls`).
- **Контракт:**
  - `serviceImageUrls(image_group): string[]` — экспортируется:
    - формат **массив фото** `image_group.images` (ожидаемый для услуги): для каждого элемента извлечь доступный URL-размер (поле `path` элемента или вложенный объект размеров) — **форму элемента подтвердить на реальном ответе `/services/`** (см. риски ADR-013);
    - формат **объект размеров** (как у мастеров, `image_group.images.norm.path`, `masters.js:168`), если встретится у услуги — извлечь доступный путь как **единственный** URL;
    - отсутствие / `null` / неожиданная форма → пустой массив `[]`.
  - Хелпер не должен бросать исключения и **не должен ничего писать в консоль** при любой форме входных данных.
- **Критерии готовности:**
  - `serviceImageUrls` возвращает массив URL для услуги с фото и `[]` для услуги без фото/с неожиданной формой; ноль ошибок в консоли. (опора для КП-2, КП-6, КП-8, КП-9; НФТ-3)
  - Форма `image_group.images` услуги подтверждена на реальных данных и извлечение URL соответствует ей. (ADR-013 Вопрос F1; риск «форма не подтверждена»)
- **Связь с ADR:** ADR-013, Вопрос F (Вариант F1).

### Задача 2: Расширение маппинга услуги — `comment` и `photos`

- **Что:** при загрузке услуг сохранять описание (`comment`) и нормализованный массив фото (`photos`) в объекте услуги.
- **Где (файлы/модули):** `docs/app.js` (маппинг услуг в `initApp`, стр. 168-174).
- **Контракт:**
  - В маппинг `svcRes.data.map(s => ({ … }))` добавляются два поля:
    - `comment: s.comment || ''` — описание услуги (строка, может быть пустой);
    - `photos: serviceImageUrls(s.image_group)` — нормализованный массив URL фото (Задача 1); `[]` при отсутствии фото.
  - Импорт `serviceImageUrls` из `./modules/serviceImages.js`.
  - Существующие поля (`id, name, cat, dur, price_min, priceStr`) и сбор/агрегация цен (ADR-012) — **без изменений**. Дополнительных запросов на услугу не вводится. (ФТ-11, КП-10)
- **Критерии готовности:**
  - Объект услуги в `SERVICES_DATA` содержит `comment` и `photos`, заполненные из уже загруженного ответа `/services/`, без дополнительных запросов. (ФТ-11, КП-10, НФТ-2)
- **Связь с ADR:** ADR-013, Вопрос D (Вариант D1).

### Задача 3: Обложка каталога из фото услуги

- **Что:** в каталоге показывать первое фото услуги из YCLIENTS; при отсутствии — категорийный плейсхолдер (как сейчас).
- **Где (файлы/модули):** `docs/modules/serviceImages.js` (`resolveServiceImage`, стр. 21-31).
- **Контракт:**
  - `resolveServiceImage(service)`: если `service.photos` непуст (`service.photos?.length`) — вернуть `{ type:'photo', src: service.photos[0], emoji: cat.emoji, grad: cat.grad }` (первое фото — обложка, ФТ-9, КП-8); иначе — **существующая** логика (`SERVICE_PHOTO_MAP[key]` → категорийный плейсхолдер).
  - Приоритет: фото услуги (`photos[0]`) > `SERVICE_PHOTO_MAP` (ручной маппинг, сейчас пустой) > категорийный плейсхолдер.
  - Сигнатура `resolveServiceImage` и формат возврата (`type`, `src`, `emoji`, `grad`) сохраняются; `_renderList` (`services.js:86-99`) **не меняется** — он уже умеет рисовать `type:'photo'` с `onerror`-деградацией на плейсхолдер (`services.js:89`). (ФТ-10, КП-9)
- **Критерии готовности:**
  - В каталоге на карточке услуги с фото отображается первое фото из `image_group`; без фото — категорийный плейсхолдер. (ФТ-9, КП-8)
  - При битом URL фото карточка каталога деградирует на плейсхолдер без поломки вёрстки. (ФТ-10, КП-9, НФТ-3)
- **Связь с ADR:** ADR-013, Вопрос D (Вариант D1).

### Задача 5: CSS детальной карточки и карусели

- **Что:** добавить стили детальной карточки и горизонтальной карусели фото на CSS scroll-snap.
- **Где (файлы/модули):** `docs/style.css` (новые селекторы детальной карточки услуги/галереи).
- **Контракт:**
  - **Карусель:** контейнер слайдов — `overflow-x:auto` + `scroll-snap-type:x mandatory`; слайд — фиксированной ширины (ширина контентной области) с `scroll-snap-align:center`, фото `object-fit:cover`/контейнер с пропорцией под качество ≈420×280 (§8 №7) без видимого размытия в пределах макета; скрытие нативного скроллбара — приём на усмотрение реализации.
  - **Точки-индикаторы:** ряд точек под каруселью; активная точка визуально выделена; стили рассчитаны на произвольное число точек.
  - **Одно фото / плейсхолдер:** статичное изображение/категорийный плейсхолдер без контейнера-карусели и без точек.
  - **Кнопка «Выбрать мастера»:** допустимо переиспользовать существующий `.btn-primary` (как на карточке мастера, `masters.js:213`) — отдельный CSS не обязателен.
  - Стили каталога (`.svc-catalog`, `.svc-cover*`, `.svc-name`, `.svc-meta`) и общие компоненты **не трогаются**. Новые селекторы **дополняют**, а не редактируют существующие.
- **Критерии готовности:**
  - При нескольких фото — горизонтальная карусель со свайпом (нативный scroll-snap) и точками-индикаторами; при одном фото — статичное изображение без листалки; при отсутствии — категорийный плейсхолдер без поломки вёрстки. (ФТ-6, ФТ-7, КП-6)
  - Стили каталога и общих компонентов не изменены (нет регрессии каталога/других экранов).
- **Связь с ADR:** ADR-013, Вопрос E (Вариант E1), Вопрос A.

### Задача 6: Рендер детальной карточки `openServiceCard`

- **Что:** реализовать императивный рендер контента детальной карточки при тапе по услуге.
- **Где (файлы/модули):** `docs/modules/services.js` (новая функция `openServiceCard(id)`; экспорт в `window`).
- **Контракт:**
  - `openServiceCard(id)`: ставит `state.serviceId = id`; формирует `innerHTML` контейнера `#serviceCardContent` из объекта услуги (`getService()` / поиск по `SERVICES_DATA` — данные уже в памяти, **без запросов**, ФТ-11, КП-10); затем `go('s-service')`. По образцу `openMasterCard` (`masters.js:128`).
  - Состав карточки (§8 №3, ФТ-2, ФТ-12, КП-2, КП-15):
    - **галерея фото** из `s.photos` (Задача 5: ≥2 фото → карусель + точки; 1 фото → статичное; 0 → категорийный плейсхолдер из `resolveServiceImage`/`CATEGORY_MAP`); каждое `<img>` с `alt="${esc(s.name)}"` (НФТ-9), `loading="lazy"` (НФТ-2) и `onerror`-деградацией на плейсхолдер (ФТ-10, КП-9);
    - **название** услуги (`esc(s.name)`);
    - **строка цены + длительности** — `s.priceStr` (диапазон, уже агрегирован ADR-012) + `s.dur` мин; формат согласован с каталогом (`{priceStr} · {dur} мин`); логика ADR-012 не вызывается и не меняется;
    - **блок описания** — `esc(s.comment)`: показывается только при непустом `comment`; при пустом — блок **скрывается** (без заглушки, §8 №2, ФТ-8, КП-7);
    - **категория и список мастеров — НЕ показываются** (§8 №3, ФТ-12, КП-15);
    - нижняя **кнопка «Выбрать мастера»** (Задача 7).
  - Все внешние/пользовательские данные (`name`, `comment`, URL фото) выводятся **только через `esc()`** (НФТ-1, КП-12). Вставка непроверенного HTML недопустима.
  - Для карусели — минимальный JS: рендер слайдов + точек, слушатель `scroll` контейнера для подсветки активной точки (индекс по `scrollLeft / slideWidth`); только при ≥2 фото.
- **Критерии готовности:**
  - Карточка отображает название, описание (из `comment`), цену-диапазон + длительность и фото работ; всё экранировано. (ФТ-2, ФТ-12, КП-2, КП-12, КП-15)
  - Описание и фото берутся из уже загруженного объекта услуги; дополнительных запросов к YCLIENTS не выполняется. (ФТ-11, КП-10, НФТ-2)
  - При пустом `comment` блок описания скрыт; остальная карточка корректна. (ФТ-8, КП-7)
  - Галерея/одно фото/плейсхолдер и деградация на плейсхолдер при битом URL — корректны. (ФТ-6, ФТ-7, ФТ-10, КП-6, КП-9)
  - Все тексты экрана — на русском. (ФТ-13, КП-14)
- **Связь с ADR:** ADR-013, Вопрос B (Вариант B1), Вопрос E (Вариант E1), Вопрос F.

- **Где (файлы/модули):** `docs/modules/services.js` (`selectService`, стр. 147-156; новая функция перехода к мастерам или inline `onclick` кнопки).
- **Контракт:**
  - `selectService(id)` перестраивается (ADR-013 Вопрос C1):

    - иначе — `openServiceCard(id)` (Задача 6), ведёт на `s-service` (ФТ-1, КП-1).
  - Кнопка «Выбрать мастера» (на детальной карточке) вызывает отдельный обработчик: сброс `state.masterId = null; state.masterName = null` (как в текущей ветке обычного потока `selectService`, `services.js:152-153`) + `go('s-masters')`. `state.serviceId` уже установлен в `openServiceCard`. (ФТ-3, КП-3)
  - `renderMasters` отрабатывает как сейчас (через `registerOnEnter('s-masters')`, `app.js:47`) — `state.serviceId` корректен. Логика выбора мастера/слотов/бронирования **не меняется**.

  - Кнопка «Выбрать мастера» переводит на `s-masters` с корректно установленной услугой. (ФТ-3, КП-3)
- **Связь с ADR:** ADR-013, Вопрос C (Вариант C1), Вопрос B.

## Изменения публичного контракта

- **Внешний API YCLIENTS / CLI / схема БД** — **не меняются**. Новых эндпоинтов не вводится: описание (`comment`) и фото (`image_group`) берутся из уже загружаемого `/services/{company}` (`app.js:162`).
- **Внутренняя модель услуги (`SERVICES_DATA`)** — расширяется полями `comment` (строка) и `photos` (массив URL) — аддитивно, остальные поля не меняются.
- **Внутренняя поверхность `serviceImages.js`** — добавляется экспортируемый `serviceImageUrls`; `resolveServiceImage` сохраняет сигнатуру и формат возврата (поведение расширяется приоритетом реального фото).
- **Разметка/CSS-контракт** — добавляется экран `s-service` (`#serviceCardContent`) и новые CSS-селекторы детальной карточки/карусели; каталог, общие компоненты и прочие экраны — без изменений.
- **Поведенческое изменение:** в обычном потоке тап по услуге ведёт на новый экран `s-service` вместо прямого перехода к выбору мастера; в режиме предвыбора поведение прежнее. Изменение намеренное, согласовано с BA-014 и ADR-013. Миграция данных не требуется.

## Тесты

> Автотестов в проекте нет (CLAUDE.md). Проверка ручная по сценариям ниже; тестировщик сверяет с КП BA-014. Спецификация фиксирует минимальный обязательный набор проверок для релиза.

- **Т-1 (КП-1, КП-2, КП-15):** В обычном потоке (мастер не предвыбран) тап по карточке услуги открывает экран `s-service` с названием, описанием (для услуги с заполненным `comment`), строкой «цена-диапазон · длительность» и фото работ. Категория и список мастеров на экране отсутствуют.
- **Т-2 (КП-3):** На детальной карточке внизу есть кнопка «Выбрать мастера»; тап по ней открывает `s-masters` для выбранной услуги (список мастеров соответствует услуге); далее обычный поток (мастер → время → подтверждение) работает.
- **Т-3 (КП-4):** Войти в каталог в режиме предвыбранного мастера (с экрана мастера) → тап по услуге ведёт **сразу** на `s-slots` (выбор времени); экран `s-service` **не** показывается.
- **Т-4 (КП-5):** С детальной карточки нажать «Назад» → возврат в каталог `s-services`; ранее применённые поиск/фильтр по категории/баннер предвыбранного мастера сохранены.
- **Т-5 (КП-6):** Услуга с несколькими фото → галерея: горизонтальный свайп листает фото, точки-индикаторы отражают позицию. Услуга с одним фото → статичное изображение без листалки. Услуга без фото → категорийный плейсхолдер (градиент + эмодзи) без поломки вёрстки. *(При фактическом отсутствии услуг с >1 фото в данных — проверить деградацию на 1 фото/плейсхолдер; карусель верифицируется при появлении нескольких фото, см. риск ADR-013.)*
- **Т-6 (КП-7):** Услуга с пустым `comment` → блок описания на детальной карточке скрыт (нет текста-заглушки); остальная карточка (название, цена/длительность, фото/плейсхолдер, кнопка) корректна.
- **Т-7 (КП-8):** В каталоге услуга с фото показывает первое фото из `image_group` как обложку; услуга без фото — категорийный плейсхолдер (как раньше).
- **Т-8 (КП-9, НФТ-3):** Эмулировать битый URL фото (например, подменить `src` в DevTools или взять услугу с недоступным фото) → и карточка каталога, и галерея на детальной карточке деградируют на категорийный плейсхолдер без поломки вёрстки.
- **Т-9 (КП-10, НФТ-2):** Открыть детальную карточку → DevTools → Network: дополнительных запросов к YCLIENTS за описанием/фото **не выполняется** (описание/фото из уже загруженного `/services/`). Прокрутка каталога и открытие карточки не тормозят.
- **Т-10 (КП-11):** Поиск по услугам, фильтр по категории, режим предвыбранного мастера, состояние «Ничего не найдено» работают; пройти полный поток услуга → детальная карточка → мастер → время → подтверждение; DevTools → Console: **ноль ошибок** на всех шагах.
- **Т-11 (КП-12):** Услуга с заведомо «опасным» `comment`/названием (например, содержит `<`, `>`, `"`, `&`) → текст выводится как есть (экранирован), инъекция HTML не срабатывает; URL фото также экранирован.
- **Т-12 (КП-13):** DevTools → Application → Service Workers: после деплоя активна новая версия кэша (`v57`); PWA остаётся устанавливаемой (Lighthouse/installable); офлайн-открытие не регрессирует; новых рантайм-зависимостей нет.
- **Т-13 (КП-14):** Все пользовательские тексты экрана `s-service` (кнопка «Выбрать мастера», «Назад», заголовки/строки) — на русском языке, в едином тоне приложения.

## Риски и митигации

- **Риск:** YCLIENTS не хранит/не отдаёт >1 фото на услугу (`POST /images/service` перезаписывает) → карусель не проявится на реальных данных. → **Митигация:** галерея реализована на массиве `photos` произвольной длины — деградирует на 1 фото/плейсхолдер, карусель «включается сама» при N>1; разработчик подтверждает фактическое содержимое `image_group.images` на реальном ответе `/services/`; вопрос об альтернативном источнике (если только 1 фото) эскалируется заказчику, **вне объёма реализации**. (ADR-013, риск «>1 фото»; §8 №1)
- **Риск:** фактическая форма `image_group` услуги не подтверждена/неоднородна (у мастеров — объект размеров) → пустая галерея/ошибки на валидных данных. → **Митигация:** защищённый `serviceImageUrls` обрабатывает массив и объект размеров и любые отсутствия (→ `[]`), ноль ошибок в консоли; форму элемента разработчик подтверждает на реальных данных (Задача 1). (ADR-013 Вопрос F1; Т-7, Т-8, Т-9)
- **Риск:** XSS через `comment` / URL фото / название. → **Митигация:** все внешние данные — только через `esc()`; вставка непроверенного HTML недопустима (Задача 6). (НФТ-1; Т-11)
- **Риск:** детальная карточка показывается в режиме предвыбранного мастера (нарушение §8 №5) либо тап перестаёт работать. → **Митигация:** единая развилка в `selectService` по `masterPreSelected` (Задача 7, ADR-013 C1); режим предвыбора не трогается. (Т-2, Т-3)
- **Риск:** правка/регрессия каталога, общих CSS-компонентов или логики поиска/фильтра при добавлении карточки. → **Митигация:** `_renderList`/`renderServices`/поиск/фильтр и стили каталога **не трогаются**; новый экран и стили — отдельно, дополняя существующие (Задачи 3, 5, 7). (Т-7, Т-10)
- **Риск:** дополнительные запросы к YCLIENTS при открытии карточки → деградация/нарушение НФТ-2. → **Митигация:** `comment`/`photos` сохраняются при загрузке (Задача 2), карточка рендерится из памяти, без запросов (Задача 6). (Т-9)
- **Риск:** забытое поднятие `CACHE_VERSION` → клиенты не получают обновления. → **Митигация:** единое поднятие `v56`→`v57` (Задача 8). (Т-12)
- **Риск:** слушатель `scroll` карусели остаётся при возврате/повторном открытии. → **Митигация:** экран рендерится заново при каждом тапе по услуге (старый DOM `#serviceCardContent` заменяется `innerHTML`), слушатель снимается вместе с заменённым узлом (ADR-013 Вопрос E1). (Т-5, Т-10)
