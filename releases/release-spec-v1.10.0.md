# Спецификация релиза v1.10.0

- **Дата планирования:** 2026-05-29
- **Тип релиза:** minor (новая функциональность — диапазон цен в каталоге + переработка вёрстки экрана услуг; публичные контракты YCLIENTS/CLI/схем данных не меняются; новых эндпоинтов и рантайм-зависимостей нет; логика записи сохраняется. Последний тег — v1.9.0)
- **Цели:** (1) Каталог услуг — крупные карточки в один столбец с обложкой сверху 16/9 (в духе «Ленты») и заметная строка поиска по услугам. (2) В каталоге показывать клиенту реальный диапазон стоимости услуги «от {min} до {max} ₽», вычисленный из цен мастеров; после выбора мастера — его конкретную цену.

## Источники

- BA-требования: `ba-req/ba-requirements-012.md` (вёрстка экрана услуг; открытые вопросы №1–№4 закрыты заказчиком 2026-05-29, §8), `ba-req/ba-requirements-013.md` (диапазон цен; блокирующий вопрос №1 решён заказчиком 2026-05-29, вопросы №2–№6 — значения по умолчанию)
- ADR: `adrs/adr-011.md` (вёрстка экрана услуг), `adrs/adr-012.md` (схема сбора цен мастеров и агрегации диапазона)

## Объём релиза

> **Группа A (ADR-011, вёрстка):** одностолбцовый каталог услуг с обложкой 16/9, локальное выделение строки поиска `s-services`. **Группа B (ADR-012, цены):** фоновый сбор цен мастеров (`book_services?staff_id=` по каждому мастеру), агрегация диапазона min/max по услуге, новый форматтер `_fmtPriceRange`, конкретная цена мастера на подтверждении/сохранении/в режиме мастера.
>
> **Вне объёма:** логика выбора услуги/мастера, расчёта слотов, бронирования; состав данных карточки услуги; общий класс `.search-bar` и строка поиска на главной `#homeSearchBar`; раздел «Лента» (`.feed-card`); функция `_fmtPrice` (не меняется); мок `SERVICES_DATA` (fallback, не переписывается на диапазоны); карточка мастера `openMasterCard` (свой список услуг); persist кэша цен в `localStorage`; захардкоженный YCLIENTS-токен.

### Задача 1: Одностолбцовый каталог услуг с обложкой 16/9

- **Что:** перевести каталог `s-services` на один столбец с крупной карточкой «обложка сверху + название + мета», обложка `aspect-ratio:16/9` (как `.feed-photo`).
- **Где (файлы/модули):** `docs/style.css` (`.svc-catalog` стр. 261-265, `.svc-cover` стр. 277-283, `.svc-cover-img`/`.svc-cover-placeholder` стр. 284-299, `.svc-name`/`.svc-meta` стр. 303-304); при необходимости минимально `docs/modules/services.js` (`_renderList`, стр. 86-99) — только с сохранением классов/`data-sid`/`onclick`/`loading="lazy"`.
- **Контракт:**
  - `.svc-catalog` — один столбец (без `grid-template-columns: 1fr 1fr`); согласованные `gap`/`padding`.
  - `.svc-cover` — `aspect-ratio:16/9` вместо `padding-top:75%`; `.svc-cover-img`/`.svc-cover-placeholder` корректно заполняют блок 16/9 (при `aspect-ratio` абсолютное позиционирование `inset:0` может быть упрощено — реализация на усмотрение, главное — обложка заполняет область без искажений, `object-fit:cover` сохранён).
  - `.svc-name`/`.svc-meta` — увеличенная типографика пропорционально крупной карточке (BA-012 №4; конкретные значения — деталь реализации).
  - HTML `_renderList` сохраняет: классы `svc-catalog-card`/`svc-cover`/`svc-info`/`svc-name`/`svc-meta`, `data-sid="${esc(s.id)}"`, `onclick="selectService(this.dataset.sid)"`, `coverHtml` с `loading="lazy"`, состав «обложка + название + {priceStr} · {dur} мин».
- **Критерии готовности:**
  - Услуги отображаются в один столбец во всю ширину контентной области (сетки 2 колонки нет). (BA-012 КП-1)
  - Карточки крупнее прежних, обложка сверху 16/9, под ней название и мета. (BA-012 КП-2, КП-3)
  - Тап по карточке выбирает услугу и ведёт к `s-masters`/`s-slots` без ошибок. (BA-012 КП-5)
  - Поиск, фильтр по категории, баннер предвыбранного мастера, «Ничего не найдено» — работают в новой вёрстке. (BA-012 КП-6, КП-7, КП-8)
- **Связь с ADR:** ADR-011, Вопрос A (Вариант A1), Решение.

- **Контракт:**

  - Новый селектор в `style.css` задаёт выделение (контрастный фон / заметная рамка / тень — приём на усмотрение реализации; цель — поле явно различимо на фоне экрана); плейсхолдер «Поиск услуг…» и иконка остаются читаемыми.
  - **Общий класс `.search-bar` (style.css:252-257) и правила `#homeSearchBar` (style.css:258-259) не изменяются.** Новый селектор **дополняет**, а не редактирует общий класс.

- **Связь с ADR:** ADR-011, Вопрос B (Вариант B1), Решение.

### Задача 3: Форматтер диапазона цен `_fmtPriceRange`

- **Что:** добавить в `utils.js` функцию форматирования диапазона/единичной цены, не меняя `_fmtPrice`.
- **Где (файлы/модули):** `docs/modules/utils.js` (рядом с `_fmtPrice`, стр. 17-21).
- **Контракт:**
  - `_fmtPriceRange(min, max): string` — экспортируется:
    - `!min` → `'По запросу'`;
    - `min === max` или `!max` → единичная сумма `f(min)`;
    - `min < max` → `'от ' + f(min) + ' до ' + f(max)` (формат по умолчанию BA-013 №2; допускается `f(min) + ' – ' + f(max)` — но единый во всех местах показа);
    - `f(n) = Math.round(n).toLocaleString('ru-RU') + ' ₽'` — идентично внутреннему форматтеру `_fmtPrice` (`utils.js:19`).
  - `_fmtPrice` (`utils.js:17-21`) **не изменяется** — продолжает использоваться для единичных цен.
- **Критерии готовности:**
  - Диапазон при `min<max`, единичная сумма при равенстве/отсутствии max, «По запросу» при отсутствии цены; форматирование `ru-RU`/«₽»/округление сохранено. (BA-013 КП-2, КП-3, ФТ-4, ФТ-6)
  - `_fmtPrice` физически не изменена. (BA-013 НФТ-3)
- **Связь с ADR:** ADR-012, Вопрос D (Вариант D1).

### Задача 4: Кэш цен мастеров в состоянии

- **Что:** ввести сессионные кэш-структуры под диапазон по услуге и цену мастера на услугу.
- **Где (файлы/модули):** `docs/modules/state.js`.
- **Контракт:**
  - `servicePriceRange` — `{ [serviceId]: { min, max } }` (агрегат по ценам мастеров услуги).
  - `staffServicePrice` — `{ [staffId]: { [serviceId]: price } }` (конкретная цена мастера на услугу).
  - Реализация — как поля `state` или отдельные экспортируемые структуры + сеттер(ы) для наполнения из `app.js`. Хранение **в памяти на сессию**, без persist в `localStorage`.
- **Критерии готовности:**
  - Структуры доступны для наполнения из `initApp` и для чтения из `services.js`/`slots.js`/`booking.js`. (опора для КП-4, КП-6, КП-7)
- **Связь с ADR:** ADR-012, Вопрос A (Вариант A1), «Влияние на код» (`state.js`).

### Задача 5: Фоновый сбор цен мастеров и агрегация диапазона в `initApp`

- **Что:** после загрузки мастеров собрать цены каждого мастера через `book_services?staff_id=`, агрегировать min/max по услуге и цену мастера на услугу, пересчитать `priceStr` каталога и перерисовать — **не блокируя** первичную отрисовку.
- **Где (файлы/модули):** `docs/app.js` (`initApp`, стр. 160-198).
- **Контракт:**
  - Первичная отрисовка каталога (`app.js:167-174`, `priceStr` через `_fmtPrice(s.price_min,s.price_max)`) — **без изменений** (фолбэк-цена, показывается немедленно).
  - После получения мастеров (`staffRes`, стр. 163/176) запускается **фоновый** сбор: `Promise.all` по мастерам `YC.get('/book_services/'+YC.company, { staff_id: <ycId/id мастера> })` (id мастера — как в `masters.js:146`; для статичных мастеров без `ycId` — пропуск).
  - Для каждого ответа: по `data.services[]` накапливать валидные `price_min` (> 0) в `servicePriceRange[service.id]` (обновлять `min`/`max`) и в `staffServicePrice[staffId][service.id]`.
  - По завершении сбора: для услуг с собранным диапазоном пересчитать `priceStr = _fmtPriceRange(range.min, range.max)`; для услуг без собранных цен — оставить фолбэк (`_fmtPrice(price_min,price_max)`); вызвать `renderServices()` (перерисовка с диапазонами).
  - **Graceful degradation (ADR-012 Вопрос E):** провал/пустой ответ по мастеру — пропускается (try/catch); при полном отсутствии собранных цен по услуге — фолбэк; **ноль ошибок в консоли**. Сбор не блокирует первичный показ каталога.
- **Критерии готовности:**
  - Диапазон в каталоге вычислен из цен мастеров (не из `price_min`/`price_max` услуги); схема — один проход по мастерам, без заметной деградации загрузки/прокрутки. (BA-013 КП-4, НФТ-2)
  - При недоступности данных мастера каталог отрисован с фолбэк-ценой, без ошибок в консоли. (BA-013 КП-8)
- **Связь с ADR:** ADR-012, Вопрос A (A1), Вопрос B (B1), Вопрос E (E1).

### Задача 6: Диапазон в каталоге и поиске на главной

- **Что:** вывести диапазон (через `priceStr`, пересчитанный в Задаче 5) в каталоге и в поиске на главной.
- **Где (файлы/модули):** `docs/modules/services.js` (`_renderList`, стр. 95 — вывод `s.priceStr`); `docs/modules/search.js` (стр. 27 — вывод `s.priceStr`).
- **Контракт:**
  - `_renderList` (`services.js:95`) выводит `s.priceStr` — структурно без изменений; значение приходит из `_fmtPriceRange` (Задача 5).
  - Поиск на главной (`search.js:27`) выводит `s.priceStr` — тот же диапазон, что в каталоге (BA-013 №5); структурно без изменений.
- **Критерии готовности:**
  - В каталоге для услуги с различающимися ценами мастеров отображается диапазон «от {min} до {max} ₽»; единичная — для равных; «По запросу» — без цены. (BA-013 КП-1, КП-2)
  - Формат диапазона единообразен в каталоге и поиске на главной. (BA-013 КП-5)
  - Поиск на главной работает со строкой цены без ошибок и артефактов. (BA-013 КП-8)
- **Связь с ADR:** ADR-012, «Границы решения», Вопрос D (D1).

### Задача 7: Режим предвыбранного мастера — конкретная цена мастера

- **Что:** в режиме предвыбранного мастера показывать конкретную цену этого мастера, а не диапазон.
- **Где (файлы/модули):** `docs/modules/services.js` (`renderServices`, ветка мастера, стр. 106-129; формирование цены — стр. 118).
- **Контракт:**
  - Для услуг режима мастера цена берётся из ответа `book_services?staff_id=` (`services.js:111`, поле `s.price_min` соответствующей услуги) либо из `staffServicePrice[masterId][serviceId]` — **без дополнительных запросов**.
  - Текущее «от {price_min} ₽» (`services.js:118`) заменяется на **единичную** конкретную цену мастера согласованного формата (через `_fmtPrice`/общий `f`), не диапазон. (BA-013 №4)
- **Критерии готовности:**
  - В режиме предвыбранного мастера отображается конкретная цена мастера (не общий диапазон). (BA-013 КП-6)
- **Связь с ADR:** ADR-012, Вопрос C (C1), Решение №4.

### Задача 8: Подтверждение и сохранение визита — конкретная цена выбранного мастера

- **Что:** на экране подтверждения «Стоимость» и при сохранении визита использовать конкретную цену выбранного мастера, согласованную между собой.
- **Где (файлы/модули):** `docs/modules/slots.js` (`updateConfirmScreen`, стр. 85); `docs/modules/booking.js` (`_saveBookedRecord`, стр. 30).
- **Контракт:**
  - `slots.js:85`: если выбран конкретный мастер (`getMaster()` не null) и есть `staffServicePrice[masterId][serviceId]` — показать **его** цену единичной суммой; иначе фолбэк на `svc.priceStr`. (BA-013 №3)
  - `booking.js:30` (поле `price` визита): сохранять значение, **согласованное с показанным на подтверждении** (конкретная цена мастера при его выборе; иначе текущий `svc.priceStr`). (BA-013 №3, ФТ-8)
- **Критерии готовности:**
  - На подтверждении при выбранном мастере отображается его конкретная цена (не диапазон). (BA-013 КП-6)
  - Цена в визите (`price`) согласована с показанной на подтверждении. (BA-013 КП-7)
- **Связь с ADR:** ADR-012, Вопрос C (C1), Решение №3.

## Изменения публичного контракта

- **Внешний API YCLIENTS / CLI / схема БД** — **не меняются**. Новых эндпоинтов не вводится: используются уже применяемые `/book_services/{company}?staff_id=` (`masters.js:146`, `services.js:111`) и `/book_staff/{company}` (`app.js:163`).
- **Формат записи визита (`yc_records`)** — поле `price` сохраняется как и прежде (строка цены); меняется лишь **значение** (конкретная цена выбранного мастера вместо каталожного `priceStr`) — поведенческое, согласовано с BA-013 и ADR-012. Схема не меняется, миграция не требуется.
- **Внутренняя поверхность `utils.js`** — расширяется экспортируемым `_fmtPriceRange` (внутренний контракт, не публичный API). `_fmtPrice` не меняется.
- **Внутреннее состояние `state`** — расширяется кэш-структурами `servicePriceRange`/`staffServicePrice` (аддитивно).
- **CSS-контракт** — `.svc-catalog`/`.svc-cover`/`.svc-name`/`.svc-meta` переопределяются (вёрстка экрана услуг); добавляется локальный селектор выделения поиска `s-services`. Общий `.search-bar` и `#homeSearchBar` — **без изменений**. Раздел «Лента» — без изменений.
- **Поведенческие изменения:** (1) каталог услуг — один столбец, обложка 16/9; (2) строка цены каталога — диапазон при различии цен мастеров; (3) подтверждение/сохранение/режим мастера — конкретная цена мастера. Все три — намеренные, согласованы с BA и ADR.

- **Т-6 (BA-013 КП-1, КП-2, КП-3, КП-5):** В каталоге для услуги с разными ценами мастеров отображается диапазон «от {min} до {max} ₽» (форматирование `ru-RU`, «₽», округление); для услуги с равной/единственной ценой — единичная сумма; без цены — «По запросу». Тот же формат — в результатах поиска на главной.
- **Т-7 (BA-013 КП-4, НФТ-2):** При загрузке каталог отрисовывается сразу (фолбэк-цена), затем цены доуточняются до диапазона; прокрутка/отрисовка не деградируют; число дополнительных запросов = число мастеров (DevTools → Network: по одному `book_services?staff_id=` на мастера).
- **Т-8 (BA-013 КП-6):** Выбрать конкретного мастера → на экране подтверждения «Стоимость» показана его конкретная цена (не диапазон). В режиме предвыбранного мастера в каталоге — конкретная цена мастера, не диапазон.
- **Т-9 (BA-013 КП-7):** Завершить запись → в сохранённом визите (`yc_records`, поле `price`) цена совпадает с показанной на подтверждении.
- **Т-10 (BA-013 КП-8):** Эмулировать недоступность цен мастера (например, отключить сеть после первичной загрузки `/services/` или вернуть пустой `book_services`) → каталог отрисован с фолбэк-ценой; поиск на главной, подтверждение, сохранение визита работают; DevTools → Console: ноль ошибок.
- **Т-11 (BA-012 КП-11, BA-013 КП-9):** DevTools → Application → Service Workers: после деплоя активна новая версия кэша (`v55`); PWA остаётся устанавливаемой (Lighthouse/installable); офлайн-открытие не регрессирует; новых рантайм-зависимостей нет.
- **Т-12 (BA-012 КП-12, BA-013 КП-10):** Затронутые пользовательские тексты («от … до … ₽», «По запросу», «Поиск услуг…») — на русском языке.

## Риски и митигации

- **Риск:** правка общего `.search-bar` или `#homeSearchBar` при выделении поиска → регрессия главной. → **Митигация:** выделение через отдельный селектор `s-services` (Задача 2); общий класс физически не редактируется; Т-2.
- **Риск:** изменение `_fmtPrice` ради диапазона → регрессия в сценариях единичной цены. → **Митигация:** отдельная `_fmtPriceRange`, `_fmtPrice` не тронута (Задача 3, ADR-012 D1); Т-6.
- **Риск:** сбор цен мастеров в критическом пути → деградация загрузки каталога. → **Митигация:** первичная отрисовка по `/services/`, сбор фоном с перерисовкой (Задача 5, ADR-012 B1); Т-7.
- **Риск:** недоступность/пустой ответ `book_services` по мастеру → пустые цены/ошибки в консоли. → **Митигация:** graceful degradation, try/catch, фолбэк на цену из `/services/` (Задача 5, ADR-012 E1); Т-10.
- **Риск:** рассинхрон цены между подтверждением и сохранённым визитом. → **Митигация:** оба читают конкретную цену мастера из единого кэша/контекста (Задача 8); Т-8, Т-9.
- **Риск:** диапазон утекает туда, где нужна конкретная цена (подтверждение/сохранение/режим мастера). → **Митигация:** каталог/поиск → `_fmtPriceRange`; подтверждение/сохранение/режим мастера → единичная цена мастера (Задачи 7, 8, ADR-012 C1); Т-8.
- **Риск:** `aspect-ratio:16/9` с абсолютным позиционированием обложки → искажение/пустоты. → **Митигация:** согласовать позиционирование `.svc-cover-img`/`.svc-cover-placeholder` под `aspect-ratio` с `object-fit:cover` (Задача 1); Т-1.
- **Риск:** забытое поднятие `CACHE_VERSION` → клиенты не получают обновления. → **Митигация:** единое поднятие `v54`→`v55` (Задача 9); Т-11.
