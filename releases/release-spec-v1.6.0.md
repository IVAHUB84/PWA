# Спецификация релиза v1.6.0

- **Дата планирования:** 2026-05-28
- **Тип релиза:** minor (новая функциональность: рабочие per-client предпочтения push и второе напоминание; внешние контракты `/subscribe`/`/unsubscribe`/`/send`/`/review` не ломаются, добавляется новый эндпоинт `/preferences`)
- **Цели:** Клиент в настройках профиля получает два **реально работающих** независимых тумблера — «Push-уведомления» (админские рассылки) и «Напоминания о визитах» — с применением на уровне клиента (все его устройства). Добавляется второе напоминание примерно за 2 часа до визита. Удаляются нерелевантные пункты «Публикация фото и видео» и «Публикация отзывов».

## Источники

## Объём релиза

> Все задачи реализуют **только** клиентские предпочтения push и второе напоминание. Схема ключей подписок `sub:${clientId}:${endpointHash}`, криптография, VAPID, `subscribePush`/`unsubscribePush`, onboarding-экран `s-consent`, сегменты/планирование/история рассылок (BA-006) — **вне объёма** (см. ADR-007, «Границы решения»).

### Задача 1: Хранение и чтение предпочтений в Worker (`prefs:${clientId}`)

- **Что:** ввести per-client предпочтения push в KV `SUBSCRIPTIONS` под ключом `prefs:${clientId}` и хелперы их чтения/записи с дефолтом «включено».
- **Где (файлы/модули):** `cf-worker/src/index.js`.
- **Контракт:**
  - Формат ключа: `prefs:${clientId}`. Значение — JSON `{ promo: boolean, remind: boolean, ts: number }`.
  - `getPrefs(env, clientId): Promise<{promo: boolean, remind: boolean}>` — читает ключ; **толерантно**: отсутствие ключа → `{promo:true, remind:true}`; отсутствие конкретного поля → это поле `true` (явное `false` уважается). Никогда не бросает на повреждённом JSON — при ошибке возвращает дефолты.
  - `putPrefs(env, clientId, {promo, remind}): Promise<void>` — пишет `prefs:${clientId}` с обоими булевыми и `ts: Date.now()`; идемпотентная перезапись.
  - Экспортировать `getPrefs`/`putPrefs` из модуля (как `endpointHash` и обработчики) для тестируемости.
- **Критерии готовности:**
  - При отсутствии ключа `getPrefs` возвращает `{promo:true, remind:true}`.
  - Сохранённое `{promo:false}` (без `remind`) читается как `{promo:false, remind:true}`.
  - Префикс `sub:` (подписки) и `prefs:` (предпочтения) не пересекаются: выборки `list({prefix:'sub:'})` не подхватывают ключи `prefs:`.
- **Связь с ADR:** ADR-007, Вопрос 1 (1A), Вопрос 6 (6A).

### Задача 2: Эндпоинт `/preferences` (Worker)

- **Что:** добавить маршрут чтения и записи предпочтений клиента.
- **Где (файлы/модули):** `cf-worker/src/index.js` (роутинг в `fetch` + хелпер-обработчик, например `handlePreferences`).
- **Контракт:**
  - `POST /preferences` — тело `{ clientId, promo, remind }`. Валидация: `clientId` обязателен; `promo`/`remind` приводятся к булевым (значения по умолчанию — `true`, если поле отсутствует). При отсутствии `clientId` → `{ error: 'missing_fields' }`, 400. При успехе → `putPrefs(...)`, ответ `{ ok: true }`.
  - `GET /preferences?clientId=…` — при отсутствии `clientId` → `{ error: 'missing_fields' }`, 400; иначе `getPrefs(...)` и ответ `{ promo, remind }`.
  - Авторизация: **без** `X-Admin-Secret` (клиентский эндпоинт, как `/subscribe`). `/send` остаётся под `X-Admin-Secret` — не менять.
  - CORS: использовать существующий `CORS`/`json(...)`; OPTIONS уже обрабатывается в `fetch`. GET-маршрут добавить в `fetch` рядом с существующими.
- **Критерии готовности:**
  - `POST /preferences {clientId, promo:false, remind:true}` → KV содержит `prefs:${clientId}` с этими значениями; ответ `{ok:true}`.
  - `GET /preferences?clientId=X` без сохранённой записи → `{promo:true, remind:true}`.
  - `POST`/`GET` без `clientId` → `missing_fields`, 400.
  - `/send` по-прежнему требует `X-Admin-Secret` (регресс-проверка не нарушена).
- **Связь с ADR:** ADR-007, Вопрос 2 (2A).

### Задача 3: Учёт `promo` в `handleSend` (Worker)

- **Что:** при отправке админских рассылок пропускать клиентов с `promo === false`; читать prefs один раз на клиента (кэш в пределах вызова).
- **Где (файлы/модули):** `cf-worker/src/index.js`, `handleSend`.
- **Контракт:**
  - Авторизация (`X-Admin-Secret`), валидация (`title`, `body`), формирование `payload`, логирование хост+статус, очистка по 410, счётчики `sent`/`failed`, пагинация — **без изменений** (ADR-006).
  - Перед `sendOne` для каждой подписки: взять `clientId` из значения подписки (`parsed.clientId`), получить prefs через кэш `Map<clientId, {promo,remind}>` (читать KV не более одного раза на clientId за вызов), и **если `promo === false` — пропустить** эту подписку (не отправлять, не увеличивать `sent`/`failed`).
  - Действует и в **broadcast**, и в **target** ветке (target-рассылка клиенту — тоже «Push-уведомления»).
  - Толерантность к старому формату значения подписки сохраняется (ADR-006).
- **Критерии готовности:**
  - У клиента с `promo:false` ни одна его подписка не получает рассылку (`sendOne` не вызывается для его endpoint'ов).
  - У клиента с `promo:true`/без записи — рассылка приходит как прежде.
  - Для мультидевайс-клиента prefs читается из KV один раз (кэш), а не на каждое устройство.
  - Счётчик `sent` считает только фактически отправленные (подавленные не учитываются).
- **Связь с ADR:** ADR-007, Вопрос 4 (4A); ADR-006, Вопрос 2 (2A).

### Задача 4: Второе напоминание + анти-дубли + часовой пояс в `handleCron` (Worker)

- **Что:** добавить вид напоминания `soon` (≈ за 2 часа до визита) к существующему `day` (накануне); защитить от дублей; учитывать локальное время студии; пропускать клиентов с `remind === false`.
- **Где (файлы/модули):** `cf-worker/src/index.js`, `handleCron`.
- **Контракт:**
  - **Часовой пояс:** ввести константу смещения студии `STUDIO_TZ_OFFSET_MIN = 180` (Москва, UTC+3). «Локальное сейчас» = `new Date(Date.now() + STUDIO_TZ_OFFSET_MIN*60000)` (использовать UTC-геттеры на этом сдвинутом значении для получения локальных год/месяц/день/час/минуты). Все сравнения дат/времени визитов (`rec.date` — локальное время студии) выполняются в локальном времени.
  - **Вид `day` (накануне):** сохранить текущее поведение и тексты — записи на **завтра** (локальная дата), тело `Завтра в HH:MM: <услуга>. Ждём вас!`. Отправлять только когда локальный час попадает в прежнее окно (локальные ~18:00, эквивалент прежнего `0 15 * * *` = 18:00 МСК) — например при `localHour === 18`. Анти-дубль (ниже) гарантирует одну отправку, несмотря на несколько cron-тиков в этом часу.
  - **Вид `soon` (≈ за 2 часа):** на каждом запуске брать записи на **сегодня** (локальная дата); для каждой записи вычислить локальное время начала визита; отправлять, если оно попадает в окно `[now+90мин, now+150мин)` (целевое смещение 2 ч, ширина 60 мин покрывает дискретность cron 30 мин). Тело: `Сегодня в HH:MM: <услуга>. Скоро ждём вас!` (русский, единый тон).
  - **Защита от дублей:** ключ `reminded:${recordId}:${kind}` (`kind ∈ {'day','soon'}`) в KV `SUBSCRIPTIONS`. Перед отправкой по записи проверить наличие ключа — если есть, пропустить запись для этого вида. После успешной отправки (хотя бы одной успешной доставки на устройства клиента) записать ключ через `put(key, '1', { expirationTtl: 172800 })` (~2 суток). Маркер ставится **на запись+вид**, один на всю запись (не на устройство).
  - **Предпочтения:** для клиента записи получить prefs (`getPrefs`); если `remind === false` — пропустить запись (оба вида). Кэш prefs в пределах запуска приветствуется (необязателен).
  - **Доставка на устройства:** как в ADR-006 — `list({prefix:'sub:${clientId}:'})`, `sendOne` на каждую подписку клиента; лог хост+статус; 410 → удалить конкретный ключ устройства; толерантность к старому формату.
  - **Выборка YCLIENTS:** для `day` — запрос на завтрашнюю локальную дату (как сейчас, но дата вычисляется в локальном времени); для `soon` — на сегодняшнюю локальную дату. Допустимо делать два запроса (на сегодня и на завтра) либо объединять — на усмотрение реализации, без дублирующих обходов.
  - `scheduled` остаётся `ctx.waitUntil(handleCron(env))`.
- **Критерии готовности:**
  - Запись на завтра у клиента с `remind:true` → одно `day`-напоминание (повторные cron-тики в том же часу не шлют повторно — маркер `reminded:${recordId}:day` стоит).
  - Запись на сегодня, до начала которой ~2 часа, у клиента с `remind:true` → одно `soon`-напоминание; следующий cron-тик в окне повторно не шлёт (маркер `reminded:${recordId}:soon`).
  - У клиента с `remind:false` напоминания (оба вида) не отправляются.
  - Время визита сравнивается в локальном времени студии (UTC+3): запись в 12:00 локального времени даёт `soon` около 10:00 локального, а не со сдвигом 3 часа.
  - Мультидевайс-клиент получает каждый вид один раз на устройство (маркер на запись не блокирует второе устройство в рамках одной отправки, но блокирует повторную отправку того же вида при следующем тике).
  - Тела напоминаний — на русском.
- **Связь с ADR:** ADR-007, Вопрос 5 (5A); ADR-006, Вопрос 3 (3A).

### Задача 6: Клиентские функции предпочтений (`push.js`)

- **Что:** добавить чтение и запись предпочтений с клиента.
- **Где (файлы/модули):** `docs/modules/push.js`.
- **Контракт:**
  - `getPreferences(clientId): Promise<{promo:boolean, remind:boolean}>` — `GET ${workerUrl}/preferences?clientId=…`; при сетевой ошибке/недоступности — возвращать дефолт `{promo:true, remind:true}` (не падать).
  - `savePreferences(clientId, {promo, remind}): Promise<boolean>` — `POST ${workerUrl}/preferences` с `{clientId, promo, remind}`; вернуть `true` при `r.ok`, иначе `false`.
  - Использовать существующий `_workerUrl()`/`WORKER_URL`. Без новых зависимостей.
  - Экспортировать обе функции в `window` рядом с `subscribePush`/`unsubscribePush`. `subscribePush`/`unsubscribePush` — **без изменений**.
- **Критерии готовности:**
  - `getPreferences` возвращает значения сервера; при ошибке сети — `{promo:true, remind:true}`.
  - `savePreferences` отправляет полное состояние обоих флагов; возвращает булев успех.
- **Связь с ADR:** ADR-007, Вопрос 2 (2A), Вопрос 3 (3A).

### Задача 7: Рабочая логика двух тумблеров и удаление мёртвого кода (`consent.js`)

- **Что:** сделать тумблеры «Push-уведомления» и «Напоминания о визитах» рабочими (сохраняют предпочтение per-client), удалить мёртвый код согласий «фото/отзывы».
- **Где (файлы/модули):** `docs/modules/consent.js`.
- **Контракт:**
  - Новая логика тумблеров push: при переключении тумблера визуально обновлять состояние **и** вызывать `savePreferences(clientId, {promo, remind})` с актуальным состоянием **обоих** тумблеров (полное состояние, не частичное). `clientId` берётся из `getSession().client_id`.
    - Маппинг: тумблер «Push-уведомления» → флаг `promo`; тумблер «Напоминания о визитах» → флаг `remind`.
    - Включение push-типа происходит **без** подтверждающей модалки и **не** вызывает `subscribePush`/`unsubscribePush` (подписка уже есть; решение №3 — повторного запроса разрешения нет). Допустимо оставить подтверждающую модалку только при **выключении** (UX-предупреждение «перестанете получать…»), но это не должно блокировать сохранение; конкретный UX согласовать в реализации, не нарушая достоверность (НФТ-3) — отображаемое состояние = сохранённое.
    - При неуспехе `savePreferences` (вернула `false`) — откатить визуальное состояние тумблера, чтобы UI не показывал несохранённое значение (НФТ-3).
  - **Удаление мёртвого кода** (становится неиспользуемым после удаления строк в Задаче 9):
    - `CONSENT_INFO.photo`, `CONSENT_INFO.reviews`.
    - Ветки `type === 'photo'` в `consentToggle`/`confirmConsentOff`.
    - Функции `showDeleteRequest`, `confirmDeleteRequest`, `closeDeleteModal` (модалка удаления материалов).
    - Соответствующие записи в `Object.assign(window, …)`.
    - Перед удалением — проверить grep'ом, что эти идентификаторы не используются нигде, кроме удаляемой разметки.
  - **Сохранить без изменений:** `toggleConsent`, `acceptConsent`, `consentModal`-логику для оставшихся типов (если переиспользуется в выключении push-тумблеров) — onboarding `s-consent` не трогается (решение №1).
- **Критерии готовности:**
  - Переключение «Push-уведомления»/«Напоминания о визитах» вызывает `savePreferences` с полным состоянием обоих флагов.
  - Мёртвый код `photo`/`reviews`/`showDeleteRequest`/`confirmDeleteRequest`/`closeDeleteModal` удалён; `grep` не находит их вызовов вне удалённой разметки.
  - При ошибке сохранения тумблер возвращается в прежнее состояние.
  - `toggleConsent`/`acceptConsent` и onboarding не затронуты.
- **Связь с ADR:** ADR-007, Вопрос 3 (3A), Вопрос 7 (7A).

### Задача 8: Загрузка состояния тумблеров и недоступность push (`profile.js`)

- **Что:** при отрисовке профиля выставлять тумблеры по серверным предпочтениям; при недоступности push честно это отражать (ФТ-8).
- **Где (файлы/модули):** `docs/modules/profile.js` (`renderProfileScreen` и/или `_toggleNotifSettings`).
- **Контракт:**
  - При отрисовке профиля (или при раскрытии «Настроек уведомлений») вызвать `getPreferences(client_id)` и выставить визуальное состояние двух тумблеров согласно ответу (`promo`→«Push-уведомления», `remind`→«Напоминания о визитах»). Источник истины — сервер (решение №4, НФТ-3); localStorage допустим лишь как кэш для мгновенной отрисовки до ответа.
  - **Недоступность push (ФТ-8):** если `!('serviceWorker' in navigator) || !('PushManager' in window)` или `Notification.permission !== 'granted'` — интерфейс не должен утверждать, что управление работает: отобразить пояснение (например подпись «Уведомления недоступны в этой среде» / «Разрешите уведомления») и/или визуально пометить тумблеры как неактивные. Конкретная подача — на усмотрение реализации, но без ложного «работает».
  - Не вводить рантайм-зависимостей; тексты — на русском.
- **Критерии готовности:**
  - После переоткрытия приложения тумблеры показывают ранее сохранённое (серверное) состояние, не сбрасываясь к «всё включено» (КП-5).
  - На втором устройстве клиента тумблеры показывают то же (клиентское) состояние (решение №4).
  - В среде без поддержки/разрешения push UI не создаёт ложного впечатления, что управление действует (ФТ-8, КП-7).
- **Связь с ADR:** ADR-007, Вопрос 2 (2A), Вопрос 3 (3A).

### Задача 10: Поднять CACHE_VERSION и версию сборки

- **Где (файлы/модули):** `docs/sw.js`, `docs/version.json`.
- **Контракт:**
  - `docs/sw.js`: `CACHE_VERSION` `v40` → `v41`. Состав `APP_SHELL` **не меняется** (новых файлов нет).
  - `docs/version.json`: `build` 8 → 9, `version` → `v1.6.0`; `date`/`time`/`sha` обновляются при релизе мейнтейнером.
- **Критерии готовности:** `CACHE_VERSION === 'v41'`; `version.json.version === 'v1.6.0'`; состав `APP_SHELL` не изменён.
- **Связь с ADR:** ADR-007, Вопрос 8 (8A).

## Изменения публичного контракта

- **Новый Worker-эндпоинт `/preferences`:** `POST {clientId, promo, remind}` → `{ok:true}`; `GET ?clientId=…` → `{promo, remind}`. Без `X-Admin-Secret`. При отсутствии `clientId` → `missing_fields` (400). Клиент (`push.js`) и Worker добавляются синхронно в этом релизе.
- **KV `SUBSCRIPTIONS`:** новые префиксы ключей `prefs:${clientId}` (предпочтения) и `reminded:${recordId}:${kind}` (анти-дубли, с TTL). Префикс `sub:` (подписки, ADR-006) — без изменений. **Миграция не выполняется** (ADR-007, Вопрос 6A): отсутствие `prefs` = «оба включены».
- **Worker `/subscribe`, `/unsubscribe`, `/send`, `/review`:** тела и ответы **не меняются**. `/send` дополнительно фильтрует получателей по `promo` (поведение доставки, не контракт).
- **Cron-расписание:** `0 15 * * *` → `*/30 * * * *` (внутреннее, Worker деплоится отдельно).
- **Приложение (PWA):** навигация, экраны, сценарий записи — без изменений. `subscribePush`/`unsubscribePush` остаются на `window`; добавляются `getPreferences`/`savePreferences`.
- **Криптография Web Push, VAPID, манифест, иконки, состав `APP_SHELL`:** без изменений.

## Тесты

Минимальный набор, обязательный в этом релизе. Стиль — как существующий `cf-worker/test/subscriptions.test.js`: запуск через `node`, мок KV/`fetch`, без рантайм-зависимостей и реальной сети. Worker — ES-модуль; тестируемые функции экспортируются/инъектируются (`_sendOne`).

> **Примечание для реализации:** мок KV в `subscriptions.test.js` (`makeMockKV`) реализует `put(key, value)` без третьего аргумента — для тестов анти-дублей расширить мок `put(key, value, options)`, чтобы принимать `{expirationTtl}` (значение можно игнорировать или фиксировать). Для управления «временем» в `handleCron` инъектировать текущее время (параметр/`Date.now` мок) — без этого тест `soon`-окна и TZ нестабилен.

Файл — например `cf-worker/test/preferences.test.js` (новый; физически добавлен):

- **`getPrefs`/`putPrefs` и дефолты:**
  - Отсутствие ключа `prefs:${clientId}` → `getPrefs` возвращает `{promo:true, remind:true}`.
  - `putPrefs(clientId,{promo:false,remind:true})` → KV `prefs:${clientId}` содержит эти значения; повторный `getPrefs` возвращает их.
  - Частичное значение `{promo:false}` без `remind` → `getPrefs` → `{promo:false, remind:true}` (толерантность).
- **Эндпоинт `/preferences`:**
  - `POST` с `{clientId, promo:false, remind:true}` → `{ok:true}` и запись в KV.
  - `GET` без записи → `{promo:true, remind:true}`.
  - `POST`/`GET` без `clientId` → `missing_fields` (400).
- **`handleSend` учитывает `promo`:**
  - Клиент с `prefs.promo=false` → `sendOne` не вызывается для его подписок; `sent` не увеличивается.
  - Клиент с `promo:true`/без записи → `sendOne` вызывается; `sent` растёт.
  - Мультидевайс-клиент (2 подписки) с `promo:true`: prefs из KV читается один раз (проверить через счётчик `get` на ключе `prefs:`), `sendOne` — дважды.
- **`handleCron` — `remind`, второе напоминание, анти-дубли, TZ:**
  - Клиент с `remind:false` → `sendOne` не вызывается для его записей (ни `day`, ни `soon`).
  - Запись на завтра, `remind:true`, локальный час = 18 → одно `day`-напоминание; маркер `reminded:${recordId}:day` записан; повторный вызов `handleCron` в том же часу → `sendOne` не вызывается повторно.
  - Запись на сегодня с временем начала в окне `[now+90,now+150)` локального времени → одно `soon`-напоминание; маркер `reminded:${recordId}:soon` записан; повторный вызов → без повтора.
  - TZ: при `now` UTC, дающем локальные 10:00 (UTC+3), запись на сегодня в 12:00 локального времени попадает в окно `soon` (а запись в 12:00, ошибочно трактованная как UTC, — нет). Проверить, что используется смещение UTC+3.
  - Анти-дубль маркера — на запись+вид (один `put('reminded:…')` на запись, независимо от числа устройств).

> Тесты **не запускаются** в этой спецификации — запуск на этапе `/release` тестировщиком. Требуется лишь физическое наличие тестов, покрывающих критерии выше. Существующий `subscriptions.test.js` должен продолжать проходить (регресс не вносить).

## Риски и митигации

- **Риск:** при частом cron (`*/30`) один клиент получает напоминание многократно. → **Митигация:** маркеры `reminded:${recordId}:${kind}` с TTL (Задача 4) — проверка перед отправкой, запись после; покрыто тестом анти-дублей.
- **Риск:** `soon`-напоминание промахивается на 3 часа из-за UTC vs локальное время студии. → **Митигация:** константа `STUDIO_TZ_OFFSET_MIN=180`, все сравнения времени в локальном времени; покрыто TZ-тестом.
- **Риск:** при логине `subscribePush` или иной код случайно перезатирает пользовательские предпочтения дефолтами. → **Митигация:** предпочтения хранятся отдельным ключом `prefs:` и пишутся **только** через `/preferences` из тумблеров (ADR-007, Вопрос 2A); `subscribePush` их не трогает.
- **Риск:** рассинхрон UI и фактической доставки (НФТ-3). → **Митигация:** тумблеры читают состояние с сервера (`getPreferences`) при отрисовке; при неуспехе `savePreferences` визуальное состояние откатывается (Задача 7, 8).
- **Риск:** случайное отключение доставки действующим клиентам при релизе. → **Митигация:** дефолт «оба включены», миграции нет (ADR-007, Вопрос 6A); отсутствие `prefs` = доставка идёт.
- **Риск:** удаление мёртвого кода согласий заденет onboarding или иные экраны. → **Митигация:** перед удалением — `grep` использований; `toggleConsent`/`acceptConsent`/`s-consent` явно не трогаются (решение №1).
- **Риск:** установленный клиент остаётся на старом косметическом UI. → **Митигация:** подъём `CACHE_VERSION` v40→v41 (Задача 10).
- **Риск:** учащённый cron превышает free-tier или нагружает KV. → **Митигация:** `*/30` (48/сутки) в пределах free-tier; чтения KV ограничены кэшем prefs и текущей базой; при росте — пересмотр частоты (техдолг в ADR-007).
- **Риск:** забыть отдельный деплой Worker. → **Митигация:** порядок деплоя прописан ниже; чек-лист содержит пункт деплоя Worker.

## Порядок деплоя

1. **Worker** (новый `/preferences`, фильтрация `handleSend`, новый `handleCron`, cron `*/30`): `cd cf-worker && npx wrangler deploy` (требует `CLOUDFLARE_API_TOKEN`). Сначала Worker — чтобы `/preferences` уже отвечал, когда клиенты обновятся.
2. **PWA:** `git push origin main` → GitHub Pages CD публикует `docs/` (~1 мин). Новый SW (`v41`) обновит клиентские модули.
3. **Проверка:** на реальном устройстве — открыть профиль → «Настройки уведомлений»: видны два тумблера, без «фото/отзывов». Выключить «Push-уведомления» → отправить админскую рассылку → не приходит; «Напоминания» оставить вкл → cron-напоминание приходит (проверка независимости). Переоткрыть приложение → состояние сохранено. На втором устройстве клиента → то же состояние. DevTools → Application → Service Workers: SW `v41` активен; ноль ошибок. Логи Worker (`wrangler tail`): фильтрация и cron-виды видны.

## Примечания по безопасности

- Секреты (`ADMIN_SECRET`, `VAPID_PRIVATE_KEY_JWK`, `YC_TOKEN`, `YC_USER_TOKEN`) остаются **только** в Worker; не коммитятся. `/send` остаётся под `X-Admin-Secret`. `/preferences` — клиентский, без секрета (только две булевы на `clientId`, чувствительных данных нет — согласовано с `/subscribe`).
- Полный endpoint в логи не пишется (как в ADR-006) — только хост.
- Захардкоженный YCLIENTS-токен в `prototype.html` — известный долг (CLAUDE.md), в этом релизе не трогается.
