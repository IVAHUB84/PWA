# Спецификация релиза v1.3.0

- **Дата планирования:** 2026-05-28
- **Тип релиза:** minor
- **Цели:** Отправлять оценку и текстовый отзыв за завершённый визит в YCLIENTS (через `POST /comments`), чтобы отзывы попадали в основной бэкенд студии и становились видны всем пользователям приложения на карточке мастера (`s-master`), а не сохранялись только локально на устройстве автора.

## Источники

- BA-требования: `ba-req/ba-requirements-003.md`
- ADR: `adrs/adr-003.md`

## Обоснование версии

MINOR: добавляется обратимо-совместимое изменение поведения экрана `s-review` (отправка в YCLIENTS вместо localStorage) и источника признака «оценено» в `s-history`. Публичный контракт навигации (`openRateVisit`, `submitReview`, `back`, `go`), внешний вид экранов, сценарий записи, чтение отзывов на `s-master` — сохраняются. Внешнего API/CLI/схемы БД проекта нет; изменяется механика интеграции с внешним YCLIENTS, что не является ломающим изменением публичного контракта приложения. Миграция данных пользователя не требуется (legacy `yc_reviews` игнорируется).

## Объём релиза

### Задача 1: Функция отправки отзыва в YCLIENTS (`POST /comments`)

- **Что:** Реализовать отправку отзыва в YCLIENTS одним запросом, изолированным в отдельной функции для переиспользования, тестируемости и единообразной обработки исходов. Основной путь — `POST /comments/{company}`; точная форма тела проверяется против реального ответа API.
- **Где (файлы/модули):**
  - `docs/modules/api.js` (новая экспортируемая функция, например `postComment({ rating, text, staffId, recordId })`; имя на усмотрение разработчика)
- **Контракт:**
  - Функция принимает оценку (целое 1–5), текст (строка, может быть пустой), YCLIENTS staff id мастера визита и record id визита.
  - Внутри — один вызов `YC.post('/comments/' + YC.company, body)` с переиспользованием существующего `YC.post` и захардкоженного токена (токен **не трогать**).
  - Тело формируется по аналогии с полями, возвращаемыми `GET /comments` (`rating`, `text`), плюс привязка к мастеру/записи (`staff_id`, `record_id`). Конкретные имена полей и их обязательность разработчик проверяет против реального ответа YCLIENTS; критерий — отзыв создаётся и при последующем `GET /comments?staff_id=N` появляется в списке отзывов мастера. Если API требует дополнительных обязательных полей — добавить минимально необходимое (в рамках «достаточного для записи и привязки»), не добавляя персональных данных автора и не добавляя чаевые. (ADR-003, Вопрос 2)
  - Возвращаемое значение **различает два исхода:** (а) успех — `success:true`/2xx; (б) ошибка — `success:false`/`_status` 4xx-5xx/таймаут. Вызывающий код должен мочь отличить успех от ошибки.
  - Функция **не обращается к DOM**.
- **Критерии готовности:**
  - Один сетевой запрос на вызов.
  - Успех и ошибка различимы вызывающим кодом.
  - Персональные данные автора (имя/телефон/email) и чаевые в тело **не добавляются**. (НФТ-7, ADR-003 Вопрос 2/4)
- **Связь с ADR:** ADR-003, Вопрос 1 (Вариант 1A), Вопрос 2 (Вариант 2A), раздел «Влияние на код».

### Задача 2: Асинхронная отправка отзыва на `s-review` с fail-safe

- **Что:** Переработать `submitReview` (`docs/modules/review.js`): вместо записи в `localStorage('yc_reviews')` — асинхронно отправить отзыв в YCLIENTS (Задача 1), показать индикатор на время запроса и корректно обработать оба исхода. При ошибке отзыв **не теряется**.
- **Где (файлы/модули):**
  - `docs/modules/review.js` (функция `submitReview` — становится асинхронной)
- **Контракт:**
  - **Валидация (без изменений):** если не выбрана ни одна звезда — отправка не выполняется, показывается текущая подсказка «Выберите оценку» (подсветка `#starsRow`, `#_starsHint`). Оценка обязательна, текст опционален. `maxlength="300"` на `#reviewTextarea` сохраняется. (ФТ-2, КП-2, ADR-003 Вопрос 5)
  - **Сбор данных:** `rating` = число выбранных звёзд (`#starsRow .star.on`); `text` = `#reviewTextarea` (trim); привязка — `staff_id` из `state._reviewMasterId` (YCLIENTS staff id записи) и `record_id` из `state._reviewRecordId`. Использовать те же значения, что приходят в `openRateVisit` из `history.js` (`data-mid` = `masterId`, `data-rid` = `recordId`).
  - **Индикатор:** на время сетевого запроса даётся обратная связь (например, блокировка/смена подписи кнопки «Отправить отзыв» или индикатор), интерфейс не «зависает» без отклика. (НФТ-2, КП-9)
  - **Успех:** пользователь получает **явное подтверждение** успешной отправки (русскоязычное; форма на усмотрение разработчика — toast/сообщение/экран успеха, в едином тоне с проектом); `record_id` визита помечается как оценённый (Задача 3); навигация на `s-home` или `s-history` (текущее поведение — `go('s-home','tab')` допустимо сохранить). (ФТ-8, КП-7)
  - **Ошибка (fail-safe):** показывается **понятное русскоязычное сообщение** о неуспехе (допустимо через существующий механизм `alert`, как в `booking.js`); введённые оценка и текст **не теряются** (поля экрана не очищаются, навигация не происходит), кнопка отправки снова доступна для повтора. Отзыв **не** помечается как отправленный. (ФТ-8, НФТ-3, КП-7)
  - **Прекращение записи в `yc_reviews`:** запись отзыва в `localStorage('yc_reviews')` удаляется. (ADR-003 Вопрос 6)
  - **Чаевые — без изменений:** блок чаевых (`tip-btn`, `_selectCustomTip`, «Своя сумма») и его текущая обработка **не входят** в задачу и не меняются по поведению. Если текущий код собирал `tips` для записи в `yc_reviews` — этот код удаляется вместе с записью в `yc_reviews`, но UI чаевых на экране остаётся как есть. (BA «Вне рамок», вопрос 9; ADR-003 «Границы»)
  - **Гонки:** при повторных нажатиях/быстрой навигации не должно происходить двойной отправки одного отзыва (например, блокировка кнопки на время запроса).
- **Критерии готовности:**
  - После выбора звёзд и нажатия «Отправить отзыв» выполняется `POST /comments` (виден во вкладке Network); при успехе — подтверждение и пометка визита оценённым. (КП-3)
  - При ошибке/недоступности API — понятное сообщение, введённый отзыв сохранён, повтор возможен, без «молчаливой» потери. (КП-7)
  - Запись в `yc_reviews` отсутствует (греп по `yc_reviews` в `review.js` не находит записи). (ADR-003 Вопрос 6)
  - UI чаевых на `s-review` визуально и функционально не изменился. (BA вопрос 9)
- **Связь с ADR:** ADR-003, Вопрос 1 (Вариант 1A), Вопрос 5, Вопрос 6, раздел «Влияние на код».

### Задача 3: Признак «оценено» по `record_id` (без `yc_reviews`)

- **Что:** Заменить источник признака «уже оценено» в истории визитов: вместо чтения локального `yc_reviews` — отдельное локальное множество **успешно отправленных** `record_id`. Кнопка «Оценить» скрывается для визита, чей `record_id` в этом множестве.
- **Где (файлы/модули):**
  - `docs/modules/review.js` (запись `record_id` в множество при успешной отправке — Задача 2)
  - `docs/modules/history.js` (чтение множества для `reviewedIds`)
- **Контракт:**
  - Новый ключ `localStorage` (например `yc_reviewed_ids`) хранит множество id записей (`record_id`), по которым отзыв **успешно** отправлен в YCLIENTS. Формат — массив строк id (как сериализуется текущий `reviews.map(r => r.recordId)`).
  - В `submitReview` (Задача 2) при успехе `record_id` добавляется в это множество. При ошибке — **не добавляется**.
  - В `history.js` `reviewedIds` строится из этого множества (а не из `yc_reviews`): `const reviewedIds = new Set(loadReviewedIds())`. Логика скрытия кнопки «Оценить» в `_renderHistoryFromCache` сохраняется (`reviewedIds.has(String(r.id)) ? '' : '<button ...openRateVisit...>'`).
  - Признак — одноразовый: повторная оценка визита, уже присутствующего в множестве, не предлагается (кнопка скрыта). Редактирование/повтор — вне рамок. (ADR-003 Вопрос 3, ФТ-6, КП-6)
  - Признак — клиентский кэш факта, не источник истины об отзыве; хранит **только id**, не текст/оценку.
- **Критерии готовности:**
  - После успешной отправки отзыва визит в «Прошлых визитах» перестаёт показывать кнопку «Оценить». (КП-6)
  - При ошибке отправки кнопка «Оценить» остаётся (визит не помечен оценённым). (КП-7)
  - Признак читается из нового ключа, не из `yc_reviews` (греп по `yc_reviews` в `history.js` не находит использования). (КП-4)
- **Связь с ADR:** ADR-003, Вопрос 3 (Вариант 3A), Вопрос 6, раздел «Влияние на код».

### Задача 4: Прекращение использования `yc_reviews` для рейтинга карточки мастера

- **Что:** Убрать вычисление среднего рейтинга мастера по локальному `yc_reviews` (`_avgRating` в `docs/modules/masters.js`) из строки карточки списка мастеров, т.к. источник `yc_reviews` упраздняется (Задача 2). Рейтинг мастера показывается на `s-master` из YCLIENTS (`m.rating`, `comments_count`) — этот показ не трогаем.
- **Где (файлы/модули):**
  - `docs/modules/masters.js` (функция `_avgRating` и её вызов в `_masterCardHtml`)
- **Контракт:**
  - `_avgRating` либо удаляется, либо возвращает пустую строку; вызов в `_masterCardHtml` (`const rating = _avgRating(m.id);`) перестаёт подмешивать рейтинг из `yc_reviews` в строку роли. Карточка не должна показывать недостоверный/пустой `⭐ X.X`.
  - Чтение и рендер отзывов в `openMasterCard` (`GET /comments/{company}?staff_id=N`, секция «Отзывы») — **без изменений**. Это канал, через который новые отзывы становятся видны всем. (ФТ-5, КП-5)
  - Полная зачистка legacy-кода и ключа `yc_reviews` из `localStorage` — допустимо в рамках этого касания, но не обязательна (ADR-003 «Технический долг»). Минимум — прекратить чтение/запись `yc_reviews` в затронутых модулях.
- **Критерии готовности:**
  - В `_masterCardHtml` нет подмешивания рейтинга из `yc_reviews`; карточка списка не показывает локальный средний рейтинг.
  - Рендер отзывов на `s-master` из `GET /comments` работает без изменений. (КП-5)
- **Связь с ADR:** ADR-003, Вопрос 6, раздел «Влияние на код».

### Задача 5: Service Worker — поднятие `CACHE_VERSION`

- **Что:** Поднять `CACHE_VERSION` в `docs/sw.js`, т.к. изменён код модулей, входящих в `APP_SHELL` (`review.js`, `history.js`, `masters.js`, `api.js`). Состав `APP_SHELL` (перечень файлов) не меняется — новых файлов не создаётся.
- **Где (файлы/модули):**
  - `docs/sw.js` (константа `CACHE_VERSION`, текущее значение `v35`)
- **Контракт:**
  - `CACHE_VERSION` повышается на одну ступень (`v35` → `v36`), чтобы клиенты получили новый код модулей, а не закэшированный старый.
  - Перечень `APP_SHELL` не изменяется (новых модулей нет).
- **Критерии готовности:**
  - `CACHE_VERSION` повышен (значение «до» → «после» зафиксировать в release-notes).
  - PWA устанавливается, офлайн-режим не регрессирует. (КП-10)
- **Связь с ADR:** ADR-003, «Влияние на код» (`docs/sw.js`), НФТ-5.

## Изменения публичного контракта

Публичный контракт приложения не меняется:

- Навигация и переходы (`openRateVisit`, `submitReview`, `openMasterCard`, `back`, `go`) — сигнатуры и точки входа сохраняются.
- Внешний вид экранов `s-review`, `s-history`, `s-master` — сохраняется (точечные правки текстов подтверждения/ошибки/индикатора — без структурных изменений).
- Чтение отзывов на `s-master` (`GET /comments`) — без изменений.
- Сценарий записи, чаевые, внешние ссылки 2ГИС/Яндекс — без изменений.

Изменяется механика интеграции с внешним YCLIENTS (отзыв уходит в `POST /comments` вместо `localStorage`) и источник признака «оценено». Миграция данных пользователя не требуется: legacy `yc_reviews` игнорируется (ADR-003 Вопрос 6).

## Тесты

Минимальный набор, обязательный к появлению в релизе (логика — модульно на чистых/выделяемых функциях, где возможно; рендер/сеть — ручной чек-лист, т.к. в проекте нет UI-раннера):

- **Модульные / логика (там, где функция выделяема и чиста):**
  - Функция Задачи 1 (`postComment`): формирует корректное тело (`rating` 1–5, `text`, `staff_id`, `record_id`), не добавляет ПДн/чаевые; различает успех и ошибку по ответу `YC.post` (мок успеха `success:true` vs ошибки `success:false`/`_status`).
  - Признак «оценено» (Задача 3): при успехе `record_id` попадает в множество; при ошибке — нет; `reviewedIds` строится из нового ключа, не из `yc_reviews`.
- **Ручной чек-лист (DevTools, вкладки Console и Network):**
  - Из «Прошлых визитов» завершённого визита нажать «Оценить» → экран `s-review` открывается с данными визита. (КП-1)
  - Без выбора звёзд нажать «Отправить отзыв» → отправки нет, видна подсказка «Выберите оценку». (КП-2)
  - Выбрать звёзды (+ опц. текст) → «Отправить отзыв» → во вкладке Network виден `POST /comments`; при успехе — подтверждение, визит больше не предлагает «Оценить». (КП-3, КП-6)
  - Проверить на другом устройстве/в режиме чтения карточки этого мастера (`s-master`) → новый отзыв виден в секции «Отзывы» (`GET /comments`). (КП-5)
  - Имитация ошибки API (offline/throttle/блокировка запроса) при отправке → понятное русскоязычное сообщение, введённый отзыв сохранён, кнопка доступна для повтора, визит не помечен оценённым. (КП-7)
  - Индикатор/обратная связь виден на время запроса; интерфейс не зависает; двойного `POST` при повторном нажатии нет. (КП-9)
  - В `localStorage` после отправки запись в `yc_reviews` не появляется; признак «оценено» — в новом ключе. (ADR-003 Вопрос 6, КП-4)
  - Карточка списка мастеров не показывает локальный средний рейтинг из `yc_reviews`. (Задача 4)
  - DevTools → Application → Service Workers: статус active; Lighthouse: PWA installable, офлайн не регрессирует. (КП-10)

## Риски и митигации

- **Риск:** `POST /comments/{company}` не поддерживается partner-токеном (403/404) или требует неизвестных полей → отзыв не создаётся. → **Митигация:** fail-safe (Задача 2): отзыв не теряется, пользователь уведомлён, повтор возможен; функция Задачи 1 проверяется против реального API, тело подстраивается итеративно в пределах «минимально достаточного».
- **Риск:** Сервер не дедуплицирует отзыв по `record_id` → возможен повторный отзыв с другого устройства. → **Митигация:** локальный признак «оценено» (Задача 3) исключает дубль на том же устройстве; кросс-девайс дедуп — осознанный техдолг (ADR-003 «Технический долг»), низкий бизнес-риск.
- **Риск:** Признак «оценено» локален → на новом устройстве кнопка «Оценить» появится для уже оценённого визита. → **Митигация:** принятый компромисс (ADR-003 «Последствия»); серверный признак вне рамок MVP.
- **Риск:** Асинхронная отправка вводит двойную отправку при повторном нажатии. → **Митигация:** Задача 2 требует блокировки кнопки/защиты от повторного запроса.
- **Риск:** Удаление `_avgRating` обнажает зависимости в других местах. → **Митигация:** Задача 4 ограничена `masters.js`; грепом проверить отсутствие других чтений `yc_reviews`, влияющих на UI; чтение отзывов на `s-master` (`GET /comments`) не трогается.
- **Риск:** Забыли поднять `CACHE_VERSION` → клиенты получают старый код модулей. → **Митигация:** Задача 5 явно требует подъёма; пункт в чек-листе.
- **Риск:** ПДн автора утекают в публичный показ. → **Митигация:** Задача 1 запрещает добавлять ПДн в тело; публичный показ — как отдаёт YCLIENTS, без расширения (ADR-003 Вопрос 4, НФТ-7).
