# Спецификация релиза v1.5.0

- **Дата планирования:** 2026-05-28
- **Тип релиза:** minor (добавление функциональности: единый механизм предложения установки PWA; обратимо-совместимо — сценарий записи, отзывы и push не меняются, манифест без изменений)
- **Цели:** При открытии на мобильном устройстве (если не установлено) автоматически и в фирменном стиле предлагать добавить приложение на экран «Домой»: программная установка на Android/Chrome, ручная инструкция на iOS, без повторного раздражения отказавшихся и установивших.

## Объём релиза

### Задача 1: Новый модуль `docs/modules/install.js` — единая логика установки

- **Что:** инкапсулировать перехват `beforeinstallprompt`, детект standalone/платформы, правило повторного показа, состояние в `localStorage`, показ/скрытие фирменного оверлея и управление видимостью `#installBanner`, ветку iOS на `s-install`, обработку `appinstalled`.
- **Где (файлы/модули):** новый `docs/modules/install.js`. Bindings — через `Object.assign(window, { … })`, как в остальных модулях. Константы (дней до повтора, ключ состояния, порог мобильной ширины) — внутри модуля либо в `docs/modules/constants.js`.
- **Контракт (публичные функции и поведение):**
  - `initInstall()` — инициализация: навешивает слушатели `beforeinstallprompt` и `appinstalled` на `window`, вычисляет условия и при необходимости показывает оверлей. Идемпотентна (повторный вызов не дублирует слушатели).
  - `isStandalone(): boolean` — `true`, если `window.matchMedia('(display-mode: standalone)').matches` **или** `navigator.standalone === true`. Чистая функция (от окружения).
  - `isIOS(): boolean` — `true` для iPhone/iPad/iPod по User-Agent, исключая Android. Чистая функция от строки UA (UA инъектируемый параметр для тестируемости, по умолчанию `navigator.userAgent`).
  - `shouldOffer(now?, state?): boolean` — применяет правило повторного показа (Задача 4): `false`, если установлено; `false`, если с момента отказа прошло < 30 дней; иначе `true`. Чистая функция; `now` и `state` инъектируемы для тестов.
  - `triggerInstall()` (или обработчик кнопки «Установить») — на Android/Chrome: вызывает `deferredPrompt.prompt()`, читает `deferredPrompt.userChoice`; при `accepted` помечает установлено; обнуляет `deferredPrompt`. На iOS: ведёт на `s-install` (ручная инструкция). Скрывает оверлей.
  - `dismissInstall()` (или обработчик «Позже»/закрытия) — записывает метку времени отказа в состояние, скрывает оверлей и `#installBanner`.
  - `refreshInstallBanner()` — управляет видимостью `#installBanner` на `s-home`: показывать только если не standalone, `shouldOffer()` истинно, аудитория мобильная и есть что предложить (есть `deferredPrompt` или iOS); иначе скрыть. Вызывается из `profile.js` вместо прежней строки по `yc_install_seen`.
  - Перехват `beforeinstallprompt`: обработчик зовёт `event.preventDefault()` и сохраняет событие в переменную модуля `deferredPrompt`; затем при выполнении условий (Задача 5) реактивно показывает оверлей.
  - Обработка `appinstalled`: помечает состояние «установлено», скрывает оверлей и `#installBanner`.
- **Критерии готовности:**
  - Модуль не обращается к токенам/сети; не выбрасывает ошибок при отсутствии `beforeinstallprompt` (деградация, КП-9).
  - `isStandalone`/`isIOS`/`shouldOffer` — чистые, тестируемые в Node без DOM (инъекция параметров).
  - Bindings зарегистрированы на `window`; функции, вызываемые из разметки (`onclick`), доступны глобально.
- **Связь с ADR:** ADR-005, Вопрос 1 (1A), Вопрос 2 (2A), Вопрос 3 (3A).

### Задача 2: Фирменный оверлей предложения установки (разметка + стили)

- **Что:** добавить фирменный баннер/модал-оверлей предложения установки в стиле приложения (не нативный браузерный баннер). Содержит заголовок, подзаголовок-мотивацию, кнопку «Установить» (вызывает `triggerInstall`) и «Позже»/крестик (вызывает `dismissInstall`).
- **Где (файлы/модули):** `docs/prototype.html` (новая разметка оверлея), `docs/style.css` (стили оверлея, светлая/тёмная тема).
- **Контракт:**
  - Тексты (плейсхолдеры, русский): заголовок «Установите приложение», подзаголовок «Добавьте приложение на экран «Домой» и получайте напоминания о ваших записях», кнопки «Установить» / «Позже».
  - Оверлей по умолчанию скрыт (`display:none`); видимостью управляет только `install.js`.
  - `onclick` кнопок ссылается на функции модуля (через `window`-bindings).
  - Внешний вид согласован с темами; не ломает существующие классы/экраны.
- **Критерии готовности:**
  - Оверлей — фирменный (не нативный), скрыт по умолчанию, управляется модулем.
  - Кнопки «Установить»/«Позже» вызывают `triggerInstall`/`dismissInstall`.
  - Тексты на русском, в едином тоне (КП-10).
- **Связь с ADR:** ADR-005, Вопрос 6 (6A), Вопрос 8 (тексты).

### Задача 3: Унификация `#installBanner`, `s-install`, `yc_install_seen`

- **Что:** перевести существующие механизмы под единый модуль, исключив два конкурирующих предложения и legacy-флаг.
- **Где (файлы/модули):** `docs/modules/profile.js`, `docs/prototype.html`.
- **Контракт:**
  - `profile.js` `renderHomeHero`: удалить прямую строку `installBanner.style.display = localStorage.getItem('yc_install_seen') ? 'none' : 'flex'`; вместо неё вызвать `refreshInstallBanner()` из `install.js`.
  - `s-install` (`prototype.html`): кнопки «Понятно, добавлю» и «Позже» перевести с `localStorage.setItem('yc_install_seen','1')` на функции модуля (например `dismissInstall()` + `go('s-home','tab')`), записывающие отказ в новый ключ состояния. Контент инструкции (3 шага «Поделиться → На экран «Домой» → Добавить») сохраняется как iOS-инструкция.
  - `#installBanner` (`prototype.html`): сохраняется как точка входа; его видимостью управляет модуль; запись/чтение `yc_install_seen` прекращается.
- **Критерии готовности:**
  - Нигде в `docs/` (кроме статичных доков `requirements.html`/`reference.html`/`status-report.html`) не пишется/не читается `yc_install_seen` как источник видимости предложения.
  - Пользователь не видит одновременно оверлей и баннер как два конкурирующих предложения (КП-8).
  - `s-install` переиспользован как iOS-инструкция; контент сохранён (КП-4).
- **Связь с ADR:** ADR-005, Вопрос 4 (4A).

### Задача 4: Состояние и правило повторного показа

- **Что:** ввести единый ключ `localStorage` с различением «отклонил» и «установил»; реализовать правило «30 дней после отказа / никогда после установки».
- **Где (файлы/модули):** `docs/modules/install.js` (логика), ключ `localStorage` (например `yc_install_state`).
- **Контракт:**
  - Структура состояния (JSON в одном ключе): метка времени последнего отказа (`dismissedAt`, ms) и флаг установки (`installed`, boolean). Совместимая форма допустима, но обязаны различаться «отказ» и «установка».
  - Константа окна повтора: `30` дней (в днях, одна именованная константа).
  - `shouldOffer()`: `installed === true` → `false`; иначе если `now - dismissedAt < 30 дней` → `false`; иначе `true`.
  - Запись «отказ» — в `dismissInstall()` и в кнопках `s-install`. Запись «установлено» — в обработчике `appinstalled` и при `userChoice.outcome === 'accepted'`.
  - Состояние переживает перезагрузку (хранится в `localStorage`).
- **Критерии готовности:**
  - После отказа при немедленном повторном открытии предложение не показывается; по истечении 30 дней — может показаться (КП-6).
  - После установки предложение и `#installBanner` не показываются никогда на устройстве (КП-7).
  - «Отклонил» и «установил» хранятся раздельно.
- **Связь с ADR:** ADR-005, Вопрос 5 (5A).

### Задача 5: Условия и момент показа (аудитория, платформа, standalone)

- **Что:** реализовать единое правило показа оверлея: при загрузке и реактивно при позднем приходе `beforeinstallprompt`.
- **Где (файлы/модули):** `docs/modules/install.js`, инициализация из `docs/app.js`.
- **Контракт:**
  - Оверлей показывается, если **все** условия истинны: `!isStandalone()` **и** `shouldOffer()` **и** аудитория мобильная **и** есть что предложить (`deferredPrompt` присутствует **или** `isIOS()`).
  - Аудитория мобильная: узкий экран (`max-width: 767px`) и/или мобильный контекст; на десктопе оверлей не показывается. Порог — именованная константа.
  - Платформенный `prompt()` вызывается **только** из обработчика кнопки «Установить» (жест пользователя), не автоматически.
  - Если `beforeinstallprompt` приходит после первого рендера — модуль повторно проверяет условия и может показать оверлей в рамках того же открытия.
  - Показ не блокирует первичную отрисовку (инициализация после boot; НФТ-6).
- **Где вызывается:** `docs/app.js` — импорт `install.js` и вызов `initInstall()` рано в boot (чтобы перехватить событие), регистрация bindings.
- **Критерии готовности:**
  - В standalone предложение не показывается (КП-2).
  - На Android/Chrome нажатие «Установить» запускает платформенный prompt через перехваченный `beforeinstallprompt` (КП-3).
  - На iOS «Установить»/принять ведёт на `s-install`-инструкцию, не на prompt (КП-4).
  - На десктопе оверлей не показывается; UI не ломается (Вопрос 7A, КП-9).
- **Связь с ADR:** ADR-005, Вопрос 2 (2A), Вопрос 3 (3A), Вопрос 6 (6A), Вопрос 7 (7A).

## Изменения публичного контракта

- **PWA (внутренний):** новый модуль `install.js` с публичными функциями на `window` (`initInstall`, `triggerInstall`, `dismissInstall`, `refreshInstallBanner` и чистые `isStandalone`/`isIOS`/`shouldOffer`). Аддитивно.
- **localStorage:** вводится ключ `yc_install_state` (состояние установки/отказа). Прежний `yc_install_seen` перестаёт использоваться как источник видимости предложения (запись/чтение прекращается). Миграция не требуется — отсутствие нового ключа трактуется как «никогда не предлагали / не отказывались».
- **Разметка:** добавлен фирменный оверлей; `s-install` и `#installBanner` сохраняют id и роль (переиспользуются). Внешний контракт навигации не меняется.
- **Манифест/иконки/Worker/YCLIENTS-запросы:** без изменений.

Минимальный набор тестов, обязательный в этом релизе (стиль — как `post-comment-worker.test.js`/`masters-filter.test.js`: Node-запуск, моки, без рантайм-зависимостей и без реальных сетевых/DOM-вызовов). Файл — например `docs/modules/install.test.js`:

## Риски и митигации

- **Риск:** браузер не сгенерировал `beforeinstallprompt` (свои критерии вовлечённости) → на не-iOS оверлей не показать. → **Митигация:** показ зависит от наличия `deferredPrompt` или iOS; при отсутствии — оверлей не показывается, UI не ломается (ФТ-10). Реактивный показ при позднем приходе события.
- **Риск:** UA-детект iOS может ошибаться на нетипичных UA. → **Митигация:** ошибка приводит максимум к показу/непоказу инструкции, не к поломке; standalone-детект подавляет предложение для уже установивших независимо от UA.
- **Риск:** рассинхрон старого `yc_install_seen` и нового состояния → два предложения. → **Митигация:** единый модуль — единственный источник видимости; прекращение использования `yc_install_seen` (Задача 3); проверка КП-8.
- **Риск:** «30 дней» / порог мобильной ширины не устроят заказчика. → **Митигация:** значения — именованные константы; правка без изменения логики.
- **Риск:** автопоказ замедлит первую отрисовку. → **Митигация:** инициализация после boot, показ не блокирует рендер (НФТ-6).

## Порядок деплоя

1. **PWA:** `git push origin main` → GitHub Pages CD публикует `docs/` (~1 мин). Worker/Secrets/манифест не затрагиваются.
2. **Проверка:** https://ivahub84.github.io/PWA/prototype.html — DevTools → Application → Service Workers: новый SW (`v38`) активируется; ноль ошибок в консоли; на мобильном эмуляторе/устройстве — оверлей при открытии (если не установлено и условия выполнены).

## Примечания по безопасности

- Новый код не использует токены и не делает сетевых запросов; секреты не затрагиваются.
- Захардкоженный `YC.userToken` в `api.js` — известная проблема (CLAUDE.md), в этом релизе не трогается.
