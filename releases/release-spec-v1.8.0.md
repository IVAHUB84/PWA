# Спецификация релиза v1.8.0

- **Дата планирования:** 2026-05-29
- **Тип релиза:** minor (новая функция — отдельный экран «Ближайшие визиты» и affordance «Все →» на главной; публичные контракты YCLIENTS/CLI/схем данных не меняются; существующий `s-history` и действия «Перенести»/«Отменить»/«Повторить»/«Оценить» сохраняются. Последний тег — v1.7.3)
- **Цели:** С главного экрана по явному affordance «Все →» (рядом с блоком «Ближайший визит») открывается **отдельный** экран `s-upcoming` со **всеми** предстоящими будущими визитами (без прошлых), отсортированными по дате; по каждому доступны «Перенести» и «Отменить». Путь не требует захода в Профиль → «История».

## Источники

- BA-требования: `ba-req/ba-requirements-010.md` (открытые вопросы №1–№4 закрыты решениями от 2026-05-29)
- Исходное требование: `requirements/req-home-upcoming-visits-010.md`
- ADR: `adrs/adr-009.md` (фундамент: ADR-008 — контекстный тап по мастеру, не пересматривается)

## Объём релиза

> Изменяются: разметка главной (affordance «Все →» и цель тапа hero-card), извлекается общий хелпер рендера предстоящих из `history.js`, добавляется отдельный экран `s-upcoming`. **Вне объёма:** логика `rescheduleRecord`/`cancelRecord`/`confirmCancel` и YCLIENTS-интеграция; экран `s-history` по сути (секции, фильтр периода, действия «Повторить»/«Оценить»); вёрстка/состав блока «Ближайший визит» по сути; лента, мастера, профиль, push, тема; захардкоженный YCLIENTS-токен (см. ADR-009, «Границы решения»).

### Задача 1: Общий хелпер рендера предстоящих визитов в `history.js`

- **Что:** вынести разметку секции/строки «предстоящий визит» из `_renderHistoryFromCache` (`history.js:121-143`) в переиспользуемый(е) экспортируемый(е) хелпер(ы); перевести `_renderHistoryFromCache` на этот хелпер без изменения внешнего поведения секции «Предстоящие».
- **Где (файлы/модули):** `docs/modules/history.js`.
- **Контракт:**
  - `_upcomingItemHtml(r): string` — HTML одной строки визита: иконка 📅; услуга (`r.svcName`); строка «`r.masterName` · `_fmtDatetime(r.datetime)`»; цена `r.price` при наличии; кнопки «Перенести» (`onclick="event.stopPropagation();rescheduleRecord(this.dataset.rid)"`, `data-rid`) и «Отменить» (`onclick="event.stopPropagation();cancelRecord(this.dataset.cid,this.dataset.chash)"`, `data-cid`/`data-chash`). Все значения проходят через `esc(...)`. Разметка строки **идентична** текущей `history.js:128-140`.
  - `_upcomingListHtml(records, opts): string` — HTML списка визитов, собранный из `_upcomingItemHtml`. `opts.withHeader === true` → добавляется заголовок «Предстоящие · N» (как `history.js:126`) и обёртка `settings-group`; `opts.withHeader` отсутствует/`false` → список без заголовка «Предстоящие · N» (заголовок экрана `s-upcoming` задаётся отдельно). Имена параметров/опции — ориентировочные, допускается иной нейминг при сохранении контракта.
  - `_renderHistoryFromCache` строит `#histUpcoming` через `_upcomingListHtml(upcoming, { withHeader: true })`; при пустом `upcoming` — поведение прежнее (`upEl.innerHTML = ''`). Секция «Прошлые», фильтр периода, действия «Повторить»/«Оценить» — **без изменений**.
  - Логика отбора `upcoming` в `_renderHistoryFromCache` (`history.js:106-111`) — без изменений.
- **Критерии готовности:**
  - В `history.js` присутствует переиспользуемый хелпер рендера предстоящих; разметка строки визита не дублируется (одна реализация, используется и `_renderHistoryFromCache`, и экраном `s-upcoming`).
  - Экран `s-history`: секция «Предстоящие · N» визуально и функционально идентична прежней (услуга/мастер/дата/время/цена, кнопки «Перенести»/«Отменить» работают).
  - Секция «Прошлые», фильтр периода, «Повторить»/«Оценить» не регрессируют.
- **Связь с ADR:** ADR-009, Вопрос 2 (Вариант 2A).

### Задача 2: Экран `s-upcoming` — рендер и onEnter

- **Что:** добавить функцию рендера экрана «Ближайшие визиты», отбирающую только будущие не отменённые визиты и рисующую их через общий хелпер; с серверным слиянием при входе (как `renderHistoryScreen`).
- **Где (файлы/модули):** `docs/modules/history.js` (функция рендера, например `renderUpcomingScreen`); `docs/app.js` (импорт + `registerOnEnter('s-upcoming', ...)`).
- **Контракт:**
  - `renderUpcomingScreen(): void` — экспортируется из `history.js`; импортируется в `app.js`.
  - Отбор визитов **идентичен** `renderHomeHero`/`_renderHistoryFromCache`: `_loadStoredRecords()`, фильтр `r.status !== 'cancelled'` и `new Date(r.datetime.replace(' ','T')) > now`, сортировка по возрастанию `datetime`. Параллельная/иная логика отбора не вводится.
  - Рендер списка — через `_upcomingListHtml(upcoming, { withHeader: false })` в контейнер экрана `s-upcoming`.
  - При пустом списке — простой блок «Нет предстоящих записей» (текст на русском). Отдельный дизайн пустого состояния не требуется (BA №4).
  - Серверное слияние: при наличии сессии (`getSession()` → `client_id`) после первичного рендера из кэша вызвать `_fetchAndMergeServerRecords(client_id)` и перерисовать — по образцу `renderHistoryScreen` (`history.js:94-103`).
  - `registerOnEnter('s-upcoming', () => renderUpcomingScreen())` в `app.js` рядом с регистрацией `s-history` (`app.js:50`).
- **Критерии готовности:**
  - Переход на `s-upcoming` показывает список всех предстоящих будущих визитов, без прошлых; первый элемент совпадает с ближайшим визитом на главной; порядок — по возрастанию даты.
  - Каждый элемент показывает услугу, мастера и дату/время; для каждого доступны «Перенести» и «Отменить»; «Повторить»/«Оценить» отсутствуют.
  - При входе выполняется серверное слияние и перерисовка (актуальные данные); при возврате на экран (`back()`) список перестраивается (onEnter).
  - При отсутствии предстоящих визитов экран показывает «Нет предстоящих записей» (достижимо только прямым открытием).
- **Связь с ADR:** ADR-009, Вопрос 1 (Вариант 1B), Вопрос 3 (Вариант 3A).

- **Контракт:**

  - Навигация на/с экрана анимируется корректно (учитывается `navigation.js`); сброс в `runScenario` (`scenarios.js:198`) обрабатывает экран без ошибок.

### Задача 4: Разметка экрана `s-upcoming` в `prototype.html`

- **Что:** добавить блок экрана «Ближайшие визиты».
- **Где (файлы/модули):** `docs/prototype.html` (новый `<div class="screen" id="s-upcoming">`, рядом с прочими экранами, например после `s-history`).
- **Контракт:**
  - `<div class="screen" id="s-upcoming">` со status-bar (как у других экранов), навигацией «Назад» (`<button class="back-btn" onclick="back()">‹ Назад</button>`, т.к. экран открывается через `go(...)` push), заголовком «Ближайшие визиты», контейнером-списком (`<div class="scroll">` с узлом, который заполняет `renderUpcomingScreen` через `_upcomingListHtml`).
  - Tab-bar — опционально, по аналогии с `s-history`/`s-feed`; если добавляется, вкладки ведут на те же экраны (`go('s-home','tab')` и т.д.).
  - Статический «Все →» для визитов в разметку **главной не добавляется** (рисуется динамически в `renderHomeHero` — Задача 5).
- **Критерии готовности:**
  - Экран `s-upcoming` присутствует в разметке, имеет кнопку «Назад» и контейнер-список под рендер.
  - Открытие `s-upcoming` и возврат «Назад» работают без визуальных артефактов; контейнер заполняется списком.
- **Связь с ADR:** ADR-009, «Влияние на код».

### Задача 5: Affordance «Все →» и цель тапа hero-card в `renderHomeHero`

- **Что:** при 2+ предстоящих визитах добавить на главной ссылку «Все →» (ведёт на `s-upcoming`); тап по hero-card перенаправить с `s-history` на `s-upcoming`.
- **Где (файлы/модули):** `docs/modules/profile.js` (`renderHomeHero`, `profile.js:85-131`).
- **Контракт:**
  - Тап по hero-card: `onclick="go('s-history')"` (`profile.js:116`) → `onclick="go('s-upcoming')"`.
  - При `upcoming.length >= 2` — в разметку hero-card добавляется ссылка-affordance «Все →» (`onclick="event.stopPropagation();go('s-upcoming')"`), визуально согласованная с «Все →» ленты (`prototype.html:72`) и мастеров (`prototype.html:82`). При `upcoming.length < 2` — affordance **не добавляется**.
  - Логика отбора `upcoming`/`next` (`profile.js:85-88`), показ единственного визита, кнопки «Перенести»/«Отменить» на карточке, ветка «нет предстоящих» (`profile.js:92-114`) — **без изменений** по сути.
  - `event.stopPropagation()` на «Все →» и на кнопках карточки сохраняется, чтобы тап по ним не срабатывал как тап по карточке.
- **Критерии готовности:**
  - При 2+ предстоящих на главной виден «Все →»; нажатие открывает `s-upcoming`.
  - При 0–1 предстоящих «Все →» отсутствует.
  - Тап по hero-card открывает `s-upcoming` (не `s-history`).
  - Кнопки «Перенести»/«Отменить» на hero-card работают как прежде (не триггерят переход по карточке).
- **Связь с ADR:** ADR-009, Вопрос 4 (Вариант 4B).

### Задача 6: Поднятие `CACHE_VERSION` в Service Worker

- **Что:** поднять `CACHE_VERSION` на одну ступень, чтобы клиенты получили обновлённые файлы.
- **Где (файлы/модули):** `docs/sw.js`.
- **Контракт:**
  - `const CACHE_VERSION = 'v45';` → `'v46'`.
  - Состав `APP_SHELL` **не меняется** (правятся существующие файлы, уже входящие в `APP_SHELL`: `prototype.html`, `constants.js`, `history.js`, `profile.js`, `app.js`; новых файлов не добавляется).
- **Критерии готовности:**
  - `CACHE_VERSION === 'v46'` в `docs/sw.js`.
  - Список файлов `APP_SHELL` не изменён по сравнению с v45.
- **Связь с ADR:** ADR-009, «Влияние на код» (НФТ-4, правило проекта).

## Изменения публичного контракта

Внешних контрактов (YCLIENTS API, CLI, схемы данных/хранилища) не меняется. Состав запросов к YCLIENTS не расширяется: `s-upcoming` использует существующий `_fetchAndMergeServerRecords` (тот же вызов, что уже делают `s-home` и `s-history`), действия «Перенести»/«Отменить» используют существующие `rescheduleRecord`/`cancelRecord`. Формат записей в `localStorage` (`yc_records`) не меняется. Миграция данных не требуется.

Публичное поведение приложения: добавляется экран `s-upcoming`; affordance «Все →» на главной (при 2+ предстоящих); целевой экран тапа по блоку «Ближайший визит» меняется с `s-history` на `s-upcoming` (намеренно, ADR-009 Вопрос 4).

## Тесты

Автотестов в проекте нет — проверка ручная по критериям приёмки BA (`npx serve docs` → `http://localhost:3000/prototype.html`). Минимальный набор ручных проверок, обязательных для релиза:

- **T1 (КП-1, ФТ-1):** При 2+ предстоящих визитах на главной рядом с блоком «Ближайший визит» виден «Все →», оформленный согласованно с «Все →» ленты/мастеров.
- **T2 (КП-2, КП-8, ФТ-1, ФТ-10):** При 0–1 предстоящих визитах «Все →» не показывается (нет вводящего в заблуждение affordance).
- **T3 (КП-2, КП-3, ФТ-2, ФТ-3, ФТ-7):** Нажатие «Все →» открывает `s-upcoming` со списком всех предстоящих будущих визитов; прошлых визитов в списке нет; путь не требует захода в Профиль → «История».
- **T4 (КП-4, НФТ-3):** Визиты в `s-upcoming` упорядочены по дате по возрастанию; первый элемент совпадает с ближайшим визитом на главной и с первым элементом секции «Предстоящие» в `s-history`.
- **T5 (КП-5, КП-6, ФТ-5, ФТ-6, ФТ-8):** В `s-upcoming` каждый визит показывает услугу, мастера и дату/время; доступны «Перенести» и «Отменить» (выполняют то же, что на главной); «Повторить»/«Оценить» отсутствуют.
- **T6 (КП-7, ФТ-9, НФТ-2, НФТ-3):** После отмены визита из `s-upcoming` и возврата на экран отменённый визит исчезает; после переноса — отображается с новой датой/порядком; состояние согласовано с блоком главного экрана.
- **T7 (КП-9, НФТ-5):** Экран `s-history` не регрессирует: секция «Предстоящие · N» (услуга/мастер/дата/время/цена, «Перенести»/«Отменить»), секция «Прошлые» с «Повторить»/«Оценить», фильтр по периоду работают как прежде.
- **T8 (граничный):** Прямое открытие `s-upcoming` без предстоящих визитов показывает «Нет предстоящих записей» (без ошибок).
- **T9 (КП-10, НФТ-1, НФТ-4):** PWA устанавливаема (Lighthouse), SW активен; `CACHE_VERSION` поднята до `v46`; состав `APP_SHELL` не изменён; новых рантайм-зависимостей нет; DevTools → Console: ноль ошибок.
- **T10 (КП-11, НФТ-6):** Затронутые пользовательские тексты («Ближайшие визиты», «Все →», «Нет предстоящих записей») — на русском, в едином тоне.

## Риски и митигации

- **Риск:** извлечение хелпера ломает разметку секции «Предстоящие» в `s-history` (расхождение HTML). → **Митигация:** разметка строки переносится **дословно** (Задача 1), `withHeader: true` воспроизводит заголовок «Предстоящие · N»; проверка T7.
- **Риск:** список `s-upcoming` расходится с главной/историей (разная логика отбора/сортировки). → **Митигация:** тот же отбор и сортировка, тот же `_loadStoredRecords` + `_fetchAndMergeServerRecords` (Задача 2); проверка T4.
- **Риск:** после отмены/переноса список `s-upcoming` не обновляется. → **Митигация:** действия уводят на `s-slots`/`s-confirm`/`s-cancel`/`s-home`; возврат/повторный вход на `s-upcoming` перерисовывает через onEnter + серверное слияние (ADR-009, «Границы решения»); проверка T6.
- **Риск:** экран не зарегистрирован в `SCREENS` → сбой анимации/сценариев. → **Митигация:** Задача 3; проверка T3/T9 (ноль ошибок в консоли).
- **Риск:** клиенты не получат обновлённые файлы из-за кэша SW. → **Митигация:** Задача 6 (v45 → v46); проверка T9.
- **Риск:** «Все →» или кнопки карточки триггерят тап по hero-card. → **Митигация:** `event.stopPropagation()` на affordance и кнопках (Задача 5); проверка T1/T5.
