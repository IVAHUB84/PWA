# Спецификация релиза v1.13.0

- **Дата планирования:** 2026-05-29
- **Тип релиза:** minor
- **Цели:** дать клиенту инбокс уведомлений на главном экране — единое место, где видны **все** полученные push (включая пришедшие при закрытом приложении), с бейджем непрочитанных на колокольчике и очисткой истории. Повышает информированность, удержание и ценность push-канала.

## Источники

- BA-требования: `ba-req/ba-requirements-016.md` (колокольчик, инбокс, счётчик непрочитанных, очистка истории).
- ADR: `adrs/adr-015.md` (IndexedDB как хранилище истории, запись из SW; экран `s-inbox`; структура записи и дедуп; лимит; бейдж). Связанный: `adrs/adr-006.md` (доставка push — **не пересматривается**).

## Объём релиза

Все задачи реализуются строго в пределах объёма BA-016 (раздел 2) и границ ADR-015. Доставка push, Worker (`cf-worker/`), схема подписок ADR-006, криптография — **не трогаются**. YCLIENTS-токен не трогается. Ветка `backup` — запрещена.

### Задача 1: Модуль хранения истории уведомлений на IndexedDB

- **Что:** тонкий модуль доступа к IndexedDB, инкапсулирующий БД истории push. Должен быть импортируем **и из окна, и из Service Worker** — без обращений к `window`/`document`/`localStorage`/DOM (только `self`/IndexedDB-API).
- **Где (файлы/модули):** новый файл `docs/modules/inboxStore.js`.
- **Контракт:**
  - Запись уведомления: `{ id, title, body, icon?, ts, read }` — `ts` = время получения (мс), `read` = флаг прочитанности (по умолчанию `false`), `id` = детерминированный дедуп-ключ.
  - `id` строится из нормализованного контента (`title|body`) + огрублённого времени получения (временное «ведро», минута) — см. ADR-015 Вопрос B/B1.
  - Объектное хранилище с `keyPath: 'id'`.
  - Экспортируемые функции (имена — на усмотрение разработчика, семантика фиксирована):
    - `addNotification({ title, body, icon, ts })` — вычисляет `id`, вставляет с дедупом (совпадение `id` не порождает дубль), затем обрезает хранилище до лимита (вытеснение самых старых по `ts`).
    - `getAllNotifications()` — возвращает все записи, отсортированные «новые сверху» (по `ts` убыв.).
    - `getUnreadCount()` — число записей с `read === false`.
    - `markAllRead()` — выставляет `read = true` всем записям.
    - `clearAll()` — удаляет все записи.
  - Константа лимита хранения — **50** (настраиваемая; ADR-015 Вопрос C/C1).
  - Все операции защищены try/catch: при недоступности IndexedDB (приватный режим/квота) функции деградируют без выброса (пустой список, счётчик 0), приложение не падает.
- **Критерии готовности:**
  - Модуль не импортирует ничего из `window`/DOM/`localStorage`; импортируется из SW без ошибок.
  - Повторная вставка уведомления с тем же контентом в пределах одного временного «ведра» не создаёт дубль; два разных уведомления в разное время сохраняются как разные.
  - При превышении 50 записей самые старые по `ts` удаляются.
  - `getAllNotifications()` отдаёт «новые сверху».
- **Связь с ADR:** ADR-015, Вопросы A/A1, B/B1, C/C1. **ФТ-5, ФТ-9, ФТ-11; КП-4, КП-8, КП-10, КП-14.**

### Задача 2: Запись истории и оповещение окна в SW-обработчике `push`

- **Что:** в обработчике `push` сохранять каждое приходящее уведомление в IndexedDB (покрывает закрытое приложение) и оповещать открытые окна через `postMessage` для немедленного обновления бейджа. Существующий показ системного уведомления сохраняется без изменений.
- **Где (файлы/модули):** `docs/sw.js` (`push`-обработчик, `sw.js:158-170`).
- **Контракт:**
  - В `event.waitUntil` выполняются параллельно: `showNotification(...)` (как сейчас), запись истории (логика из `inboxStore.js` — динамический `import()` или эквивалент, совместимый с контекстом SW), `postMessage` открытым окнам (`clients.matchAll({ type:'window', includeUncontrolled:true })` + `client.postMessage(...)`).
  - Запись `ts = Date.now()` в момент получения push; поля `title`/`body`/`icon` берутся из распарсенного payload (как сейчас, `sw.js:159-167`).
  - **Ошибка записи истории или `postMessage` не должна блокировать `showNotification`** — показ системного уведомления приоритетен (перехват в catch).
  - `notificationclick` (`sw.js:172-182`), `fetch`-обработчик, `NETWORK_ONLY_HOSTS`, SWR-ветка фото (`sw.js:108-129`) — **без изменений**.
- **Критерии готовности:**
  - При получении push по-прежнему показывается системное уведомление (поведение не регрессирует).
  - Уведомление, доставленное при закрытом приложении, после открытия присутствует в IndexedDB (видно в DevTools → Application → IndexedDB).
  - При открытом окне отправляется `postMessage` (видно в слушателе клиента).
- **Связь с ADR:** ADR-015, Вопросы A/A1, E/E1; раздел «Влияние на код» (`sw.js`). **ФТ-5, ФТ-12; КП-4, КП-11.**

### Задача 3: UI инбокса, бейджа и очистки истории

- **Что:** рендер списка инбокса из IndexedDB; бейдж непрочитанных на колокольчике; пометка прочитанным при входе в инбокс; «Очистить историю» с подтверждением; пустое состояние; слушатель `message` от SW для немедленного обновления бейджа.
- **Где (файлы/модули):** новый файл `docs/modules/inbox.js` (UI-логика). Функции экспонируются на `window` для inline-`onclick` (как в других модулях проекта).
- **Контракт:**
  - Рендер списка: `getAllNotifications()` → элементы «новые сверху», каждый показывает **заголовок, текст и дату/время**; заголовок и текст выводятся **только через `esc()`** (`docs/modules/utils.js:3`) — никакого `innerHTML` с непроверенным контентом.
  - Пустое состояние (нет уведомлений / после очистки): понятный текст на русском (опц. эмодзи в стиле проекта).
  - Бейдж: функция обновления читает `getUnreadCount()`; при `count > 0` показывает кружок-счётчик (формат «9+» при `count > 9`), при `count === 0` бейдж скрыт. Достаточный контраст на градиентном хедере.
  - Пометка прочитанным: при входе в инбокс вызывается `markAllRead()`, после чего бейдж обнуляется.
  - «Очистить историю»: кнопка на экране `s-inbox`; перед очисткой — подтверждение (диалог/`confirm` в стиле проекта); после подтверждения `clearAll()`, список → пустое состояние, бейдж скрыт.
  - Слушатель `navigator.serviceWorker` события `message` (от SW из Задачи 2): при получении — обновить бейдж (пересчёт из `getUnreadCount()`).
  - Все пользовательские тексты — на русском; колокольчик и бейдж имеют aria-метки на русском.
- **Критерии готовности:**
  - Список рендерится «новые сверху» с заголовком/текстом/временем; HTML-спецсимволы в контенте отображаются как текст (экранированы).
  - Бейдж показывает число непрочитанных, «9+» при больших значениях, скрыт при нуле.
  - При входе в инбокс непрочитанные становятся прочитанными, бейдж обнуляется.
  - «Очистить историю» требует подтверждения и полностью очищает список; затем пустое состояние, бейджа нет.
  - При недоступности IndexedDB / отсутствии уведомлений инбокс пуст, без ошибок в консоли.
- **Связь с ADR:** ADR-015, Вопросы D/D1, E/E1; раздел «Влияние на код» (UI-модуль). **ФТ-2, ФТ-3, ФТ-4, ФТ-6, ФТ-7, ФТ-8, ФТ-10, ФТ-13, ФТ-14, ФТ-15; КП-2, КП-3, КП-5, КП-6, КП-7, КП-9, КП-12, КП-13, КП-15.**

### Задача 4: Колокольчик на `s-home` и экран `s-inbox` в разметке

- **Что:** добавить иконку-колокольчик в шапку главного экрана и блок экрана инбокса; зарегистрировать экран в навигации.
- **Где (файлы/модули):** `docs/prototype.html`, `docs/modules/constants.js`, `docs/app.js`, `docs/style.css`.
- **Контракт:**
  - `docs/prototype.html`, шапка `s-home` (`prototype.html:38-44`, правый верхний угол строки логотипа): колокольчик с контейнером бейджа, `onclick="go('s-inbox')"`, aria-метка на русском; **не перекрывает и не ломает** строку поиска `#homeSearchBar` ниже (`:46-52`).
  - `docs/prototype.html`: новый блок `<div class="screen" id="s-inbox">` с заголовком («Уведомления» или эквивалент), кнопкой возврата (`back()` / стрелка), контейнером списка, пустым состоянием и кнопкой «Очистить историю».
  - `docs/modules/constants.js` (`SCREENS`, `:3-9`): добавить `'s-inbox'`.
  - `docs/app.js`: импорт UI-модуля инбокса; `registerOnEnter('s-inbox', …)` — рендер списка + `markAllRead()` + обновление бейджа; в `registerOnEnter('s-home', …)` (`app.js:53`) добавить пересчёт бейджа; (опц.) слушатель `message` от SW.
  - `docs/style.css`: стили колокольчика, бейджа (кружок справа внизу, контраст на градиенте, «9+»), экрана `s-inbox` (список, элемент, пустое состояние).
- **Критерии готовности:**
  - Колокольчик виден в правом верхнем углу шапки `s-home`; строка поиска ниже функциональна и не сломана.
  - Тап по колокольчику открывает `s-inbox`; из инбокса есть понятный возврат назад.
  - `s-inbox` присутствует в `SCREENS` и зарегистрирован через `registerOnEnter` (иначе навигация/сброс сценариев его не учтут — ADR-009).
  - Колокольчик отображается всегда (в т. ч. без подписки на push); при отсутствии уведомлений инбокс пуст без ошибок.
- **Связь с ADR:** ADR-015, Вопросы D/D1, F/F1; раздел «Влияние на код» (`prototype.html`, `constants.js`, `app.js`, `style.css`). **ФТ-1, ФТ-3, ФТ-13; КП-1, КП-3, КП-12.**

## Изменения публичного контракта

- **Внешний API / CLI / схема БД YCLIENTS / Worker:** изменений нет. `cf-worker/src/index.js`, `/subscribe`/`/unsubscribe`/`/send`, схема подписок `sub:${clientId}:${endpointHash}` (ADR-006), формат payload push — **не меняются**.
- **Новое локальное хранилище:** добавляется IndexedDB (БД истории уведомлений) на устройстве клиента. Это клиентское хранилище, не публичный контракт; миграция не требуется (новое хранилище, пустое при первом запуске).
- **Навигация:** добавляется экран `s-inbox` в `SCREENS` (аддитивно). Сигнатуры `go`/`back`/`registerOnEnter` не меняются.
- **SW→клиент:** добавляется `postMessage` из обработчика `push` (новое сообщение; клиенты без слушателя его игнорируют).

## Тесты

Автотестов в проекте нет (CLAUDE.md). Проверка — ручная по критериям приёмки. Тестировщик обязан проверить минимум:

- Получение push при **открытом** приложении → системное уведомление показано, запись появилась в IndexedDB, бейдж обновился немедленно (`postMessage`).
- Получение push при **закрытом** приложении → после открытия запись присутствует в инбоксе и учтена в бейдже (КП-4).
- Открытие инбокса → непрочитанные помечены прочитанными, бейдж обнулён (КП-5).
- «Очистить историю» → подтверждение → список очищен, пустое состояние, бейджа нет (КП-6, КП-7).
- Контент с HTML-спецсимволами (`<`, `>`, `&`, кавычки) в `title`/`body` → отображается как текст, инъекции нет (КП-9).
- Повторная доставка одного push → нет дубля в инбоксе (КП-10).
- Существующий поток push не сломан: системное уведомление, `notificationclick` (фокус/открытие), тост `#pushNotif`, подписка/отписка (КП-11).
- Без разрешения на push / без подписки → колокольчик и инбокс работают, инбокс пуст, бейджа нет, ноль ошибок (КП-12).
- Большое число непрочитанных → бейдж «9+», вёрстка шапки не ломается, контраст читаем (КП-13).
- Превышение лимита 50 → старые вытесняются без ошибок (КП-14).
- Оффлайн → ранее полученные уведомления просматриваются из IndexedDB (КП-8).
- DevTools → Console: ноль ошибок; Application → Service Workers: active; `CACHE_VERSION` = v60 (КП-16).

## Риски и митигации

- **Риск:** модуль `inboxStore.js` зависит от `window`/`localStorage`/DOM → падает при импорте в SW. → **Митигация:** писать только на `self`/IndexedDB-API; проверить импорт и запись из SW в DevTools → Application → IndexedDB.
- **Риск:** запись в IndexedDB не успевает до завершения SW. → **Митигация:** обернуть в `event.waitUntil`; ошибка записи перехватывается и не блокирует `showNotification`.
- **Риск:** дедуп-ключ схлопывает разные уведомления или плодит дубли. → **Митигация:** ключ = `title|body` + временное «ведро» (минута); протестировать повторную доставку и два разных уведомления.
- **Риск:** старый кэш SW отдаёт прежний `sw.js`/разметку/JS. → **Митигация:** `CACHE_VERSION` v59 → v60; новые модули в `APP_SHELL`; `controllerchange`-перезагрузка подхватит новый SW.
- **Риск:** XSS через `title`/`body`. → **Митигация:** рендер только через `esc()`.
- **Риск:** IndexedDB недоступен (приватный режим/квота). → **Митигация:** try/catch вокруг всех операций; деградация в пустой инбокс без ошибок.
- **Риск:** колокольчик ломает шапку/строку поиска. → **Митигация:** разместить в свободном правом верхнем углу строки логотипа, не задевая `#homeSearchBar`; проверить узкие экраны и тач.
