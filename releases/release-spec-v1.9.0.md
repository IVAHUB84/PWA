# Спецификация релиза v1.9.0

- **Дата планирования:** 2026-05-29
- **Тип релиза:** minor (новая функция — отдельный экран одной публикации `s-post` и новое целевое действие тапа по плашке превью ленты; публичные контракты YCLIENTS/CLI/схем данных не меняются; существующий список `s-feed`, бар фильтра, кнопка «Записаться →», ссылка «Все →» и админка ленты сохраняются. Последний тег — v1.8.7)
- **Цели:** Тап по конкретной плашке превью в блоке «Лента студии» на главной открывает **именно выбранную** публикацию (отдельный экран `s-post`: фото/иконка, полный текст, категория, дата, «Записаться →»), адресуя её по устойчивому `p.id`. Ссылка «Все →» сохраняет поведение (полный список `s-feed`). При отсутствии публикации по `id` — мягкий фолбэк на `s-feed` без ошибки.

- Исходное требование: `requirements/req-feed-open-single-post-011.md`

## Объём релиза

> Изменяются: целевое действие тапа по плашке превью ленты (`profile.js`), извлекается общий хелпер рендера карточки публикации из `_paintFeed` (`feed.js`), добавляется отдельный экран одной публикации `s-post` (разметка `prototype.html`, регистрация в `SCREENS` и onEnter), диспетчер открытия публикации по `id` с мягким фолбэком, добавляется аддитивный режим `replace` в `go` (`navigation.js`) для страховочного onEnter-фолбэка. **Вне объёма:** общий список `s-feed` по сути (`_paintFeed` — только извлечение хелпера без смены поведения; бар фильтра `_renderFeedCats`/`_feedFilterCat`); кнопка «Записаться →» (`_feedGoBook`); ссылка «Все →»; превью на главной по сути (количество плашек `slice(0,6)`, вёрстка плашки, условие показа секции); админка ленты (`admin.js`), формирование `id`, синхронизация с GitHub (`_ghReadFn`); состав полей публикации и формат `yc_feed_posts`; захардкоженный YCLIENTS-токен (см. ADR-010, «Границы решения»).

### Задача 1: Общий хелпер рендера карточки публикации в `feed.js`

- **Что:** вынести разметку одной карточки публикации из `_paintFeed` (`feed.js:58-78`) в переиспользуемый экспортируемый хелпер; перевести `_paintFeed` на этот хелпер без изменения внешнего поведения списка `s-feed`.
- **Где (файлы/модули):** `docs/modules/feed.js`.
- **Контракт:**
  - `_feedCardHtml(p): string` — HTML одной карточки публикации: фото/иконка-градиент по категории (`_CAT_GRADS`/`_CAT_ICONS2`, тот же расчёт `safeSrc` и `photoContent`, что `feed.js:62-65`); тег категории (`feed-tag`, `esc(p.cat)`); кнопка «Записаться →» (`feed-book`, `data-cat="${esc(p.cat||'')}"`, `onclick="_feedGoBook(this.dataset.cat)"`); подпись (`feed-caption`, `esc(p.text)` — полный текст); дата (`feed-meta`, `esc(p.date)`). Разметка карточки **идентична** текущей `feed.js:66-78`. Имя функции/параметра — ориентировочное, допускается иной нейминг при сохранении контракта.
  - `_paintFeed(el, posts)` собирает список из `_feedCardHtml(p)`; специфика последнего элемента (`margin-bottom:24px` при `i === posts.length - 1`, `feed.js:61`) сохраняется на уровне `_paintFeed` (через обёртку/параметр), а не внутри `_feedCardHtml`. Пустое состояние списка (`feed.js:54-57`) — без изменений.
  - `_renderFeedCats`/`_feedFilterCat`/`renderClientFeed`/`_feedGoBook` — **без изменений**.
- **Критерии готовности:**
  - В `feed.js` присутствует переиспользуемый хелпер рендера одной карточки публикации; разметка карточки не дублируется (одна реализация, используется и `_paintFeed`, и рендером `s-post`). (КП-4)
  - Экран `s-feed`: карточки визуально и функционально идентичны прежним (фото/иконка, тег категории, текст, дата, кнопка «Записаться →» работает; `margin-bottom` последнего элемента сохранён). (КП-8)
  - Бар фильтра категорий и фильтрация по категориям не регрессируют. (КП-8)
- **Связь с ADR:** ADR-010, Вопрос A (Вариант A1).

### Задача 2: Диспетчер открытия публикации по `id` с мягким фолбэком

- **Что:** добавить тонкий глобальный диспетчер, который по `id` находит публикацию среди `!p.draft`, записывает её `id` в `state` и открывает `s-post`; при отсутствии — открывает `s-feed`.
- **Где (файлы/модули):** `docs/modules/feed.js` (диспетчер + экспорт в `window`); `docs/modules/state.js` (поле под `id` открываемой публикации — при необходимости).
- **Контракт:**
  - `_openFeedPost(id): void` — имя/сигнатура ориентировочные. Читает `yc_feed_posts` из `localStorage`, фильтрует `!p.draft`, ищет публикацию по `p.id` (сравнение с учётом того, что `id` приходит из `data`-атрибута строкой, а хранится числом `Date.now()` — сравнивать через приведение `String(p.id) === String(id)`).
  - Если публикация найдена → записать её `id` в общее состояние (например `state._openPostId`) и вызвать `go('s-post')`.
  - Если не найдена → вызвать `go('s-feed')` (мягкий фолбэк). **Не должно** возникать ошибки/исключения в консоли.
  - Диспетчер экспортируется в `window` (как `_feedGoBook`, `feed.js:111`) для вызова из `onclick` плашки превью.
- **Критерии готовности:**
  - Вызов диспетчера с существующим `id` приводит к открытию `s-post` именно с этой публикацией (адресация по `p.id`, не по индексу). (КП-2)
  - Вызов с несуществующим/удалённым `id` приводит к открытию `s-feed`, без ошибки в консоли. (КП-7)
- **Связь с ADR:** ADR-010, Вопрос B (Вариант B1), Вопрос C (Вариант C1).

### Задача 2А: Режим `replace` в навигации (правка по итогам код-ревью)

- **Что:** добавить в `go(id, mode)` аддитивный режим `mode === 'replace'`, который проигрывает анимацию перехода как обычный push, но **замещает** верхний элемент `navHistory` вместо добавления нового. Нужен страховочному onEnter-фолбэку `renderPostScreen` (Задача 3): иначе `go('s-feed')` из onEnter `s-post` становится push поверх `s-post` (push в `navHistory` делается синхронно, а onEnter — отложенно через `setTimeout(DUR+20)`, `navigation.js:84-92`), что порождает петлю «Назад» `s-feed`↔`s-post` и блокирует возврат на `s-home`.
- **Где (файлы/модули):** `docs/modules/navigation.js` (`go`, push-ветка, строки 77-93).
- **Контракт:**
  - При `mode === 'replace'`: анимация — идентична обычному push (slide-in нового экрана справа, сдвиг текущего влево); по завершении отложенного `onEnterScreen` — тот же guard `navHistory[navHistory.length - 1] !== targetId`. Отличие от push **только** в обновлении истории: `navHistory[navHistory.length - 1] = id` вместо `navHistory.push(id)`.
  - Существующее поведение **не меняется**: вызовы `go(id)` (push по умолчанию) и `go(id, 'tab')` работают как прежде; сигнатуры `go`/`back`/`registerOnEnter` неизменны; `back`, `_clearAnimTimers`, `_updateDotsFn` не затрагиваются. Имя режима — `'replace'` (строкой, как `'tab'`).
- **Критерии готовности:**
  - `go('s-feed', 'replace')`, вызванный когда верх `navHistory` = `'s-post'`, приводит к истории `[…, 's-home', 's-feed']` (без `'s-post'` на вершине). (КП-7)
  - Существующие переходы (push, `'tab'`) и анимация не регрессируют. (КП-9)
- **Связь с ADR:** ADR-010, Вопрос C (Вариант C1, редакция 2026-05-29); «Влияние на код» (`docs/modules/navigation.js`).

### Задача 3: Экран `s-post` — разметка, регистрация, рендер и onEnter

- **Что:** добавить отдельный экран одной публикации `s-post`: разметку в `prototype.html`, запись в `SCREENS`, функцию рендера, читающую `id` из `state` и рисующую публикацию через общий хелпер, регистрацию onEnter в `app.js`.
- **Где (файлы/модули):** `docs/prototype.html` (блок экрана), `docs/modules/constants.js` (`SCREENS`), `docs/modules/feed.js` (функция рендера, например `renderPostScreen`), `docs/app.js` (импорт + `registerOnEnter`).
- **Контракт:**
  - `docs/modules/constants.js`: в массив `SCREENS` (`constants.js:3-9`) добавляется `'s-post'`.
  - `docs/prototype.html`: новый `<div class="screen" id="s-post">` по образцу `s-feed` (`prototype.html:642-656`): status-bar; nav-bar с заголовком «Публикация» и кнопкой «Назад» (`back-btn` `onclick="back()"`); контейнер `<div class="scroll" id="postCard">` (или иной id) под вывод карточки. **Бара фильтра категорий (`feed-cats`/`feedCatBar`) НЕТ** (BA решение №2). Tab-bar — опционально, по аналогии с другими экранами.
  - `renderPostScreen(): void` — экспортируется из `feed.js`, импортируется в `app.js`. Читает `id` из состояния (`state._openPostId`), ищет публикацию среди `!p.draft` в `yc_feed_posts`; при наличии — рисует `_feedCardHtml(p)` в контейнер экрана; при отсутствии — вызывает `go('s-feed', 'replace')` (страховочный фолбэк от гонки обновления данных, через **замещение** `s-post` в истории — см. Задача 2А и ADR-010 Вопрос C, редакция 2026-05-29), без ошибки в консоли. **Не** `go('s-feed')` (обычный push): к моменту onEnter `s-post` уже на вершине `navHistory`, и push дал бы петлю «Назад».
  - `docs/app.js`: `registerOnEnter('s-post', () => renderPostScreen())` (рядом с `s-feed`, `app.js:66`); импорт `renderPostScreen` из `feed.js`.
- **Критерии готовности:**
  - `'s-post'` зарегистрирован в `SCREENS`; анимация перехода `go('s-post')` и `back()` работают штатно. (КП-9, ADR-010 «ограничение из ADR-009»)
  - На экране `s-post` отображены фото/иконка категории, полный текст, категория и дата выбранной публикации. (КП-4)
  - На экране `s-post` присутствует кнопка «Записаться →», ведущая к записи по категории поста (через `_feedGoBook` → `go('s-services')`). (КП-5)
  - В `s-post` **нет** бара фильтра категорий. (BA решение №2)
  - Кнопка «Назад» возвращает на `s-home` (за счёт push-перехода и `back()`). (КП-6, BA решение №3)
  - При входе на `s-post` без валидной публикации (исчезла между переходом и onEnter) — открывается `s-feed` через `go('s-feed', 'replace')`, без ошибки. (КП-7)
  - После такого страховочного фолбэка «Назад» из `s-feed` ведёт на `s-home` (петли `s-feed`↔`s-post` нет; `s-post` не остаётся в истории). (КП-7)
- **Связь с ADR:** ADR-010, Решение (форма `s-post`), Вопрос A (рендер через `_feedCardHtml`), Вопрос C (страховочный фолбэк, редакция 2026-05-29 — через `replace`).

## Изменения публичного контракта

- **Внешний API YCLIENTS / CLI / схема БД / формат `yc_feed_posts`** — **не меняются**. Новых запросов к GitHub/YCLIENTS не вводится: `s-post` рисуется из уже наполненного кэша `yc_feed_posts` (наполняется на входе в `s-home`, `app.js:58-63`, и `s-feed`, `feed.js:38-43`).
- **Реестр экранов `SCREENS`** — расширяется новым значением `'s-post'` (аддитивно, существующие экраны не затрагиваются).
- **Контракт навигации `go(id, mode)`** — расширяется аддитивным режимом `mode === 'replace'` (внутренний контракт навигации, не публичный API приложения). Сигнатура не меняется; существующие режимы (push по умолчанию, `'tab'`) и вызовы не затрагиваются. Миграционный путь не требуется.
- **Внутренняя поверхность модуля `feed.js`** — расширяется экспортируемыми `_feedCardHtml`, `renderPostScreen`, `_openFeedPost` (внутренний контракт, не публичный API приложения). Миграционный путь не требуется.
- **Поведенческое изменение:** целевое действие тапа по плашке превью ленты (`go('s-feed')` → открытие конкретной публикации) — намеренное, согласовано с BA и ADR-010.

Проверить именно страховочный onEnter-путь (гонка: пост исчезает между тапом по плашке и входом на `s-post`), а не только диспетчерный.
- **Т-8 (КП-8):** Админка ленты (создание/редактирование/удаление поста, синхронизация с GitHub) работает как прежде.
- **Т-9 (КП-9):** DevTools → Application → Service Workers: после деплоя активна новая версия кэша (`v54`); PWA остаётся устанавливаемой (Lighthouse/installable); офлайн-открытие приложения не регрессирует; новых рантайм-зависимостей не добавлено.
- **Т-10 (КП-10):** Все затронутые пользовательские тексты («Публикация», «Назад», существующие тексты карточки) — на русском языке.

## Риски и митигации

- **Риск:** регрессия списка `s-feed` при извлечении хелпера из `_paintFeed` (потеря `margin-bottom` последнего элемента, расхождение разметки). → **Митигация:** `_feedCardHtml` повторяет разметку `feed.js:66-78` дословно; специфика последнего элемента остаётся в `_paintFeed`; визуальная сверка карточек `s-feed` до/после (Т-3, Т-8, КП-8).
- **Риск:** несовпадение типов при сравнении `id` (строка из `data`-атрибута vs число `Date.now()`) → пост «не находится», уход в `s-feed`. → **Митигация:** сравнивать через `String(p.id) === String(id)` (Задача 2); Т-1/Т-2 проверяют успешное открытие.
- **Риск:** новый экран `s-post` не добавлен в `SCREENS` → ломается анимация навигации/сброс сценариев (ограничение из ADR-009). → **Митигация:** Задача 3 включает запись в `SCREENS`; Т-9 (переходы/анимация).
- **Риск:** забытое поднятие `CACHE_VERSION` → клиенты не получают обновлённые файлы. → **Митигация:** Задача 6 (`v53`→`v54`); Т-9 (проверка активной версии SW).
- **Риск:** гонка — публикация исчезает между тапом и onEnter `s-post` → пустой экран. → **Митигация:** двойной фолбэк (диспетчер + onEnter): диспетчер → `go('s-feed')` (push из `s-home`), onEnter → `go('s-feed', 'replace')` (замещение `s-post` в истории) (Задачи 2, 2А, 3; Т-7).
- **Риск:** страховочный onEnter-фолбэк делает `go('s-feed')` (push), но `s-post` уже на вершине `navHistory` (push синхронен, onEnter отложен) → история `[…, 's-home', 's-post', 's-feed']`, петля «Назад» `s-feed`↔`s-post`, возврат на `s-home` недоступен (консоль чистая — деградация незаметна формальной проверке КП-7). → **Митигация:** onEnter использует режим `replace` (Задача 2А), а не обычный push; Т-7 явно проверяет «Назад» → `s-home` после фолбэка.
- **Риск:** режим `replace` затрагивает существующие переходы (push/`'tab'`) → регрессия навигации всех экранов. → **Митигация:** `replace` строго аддитивен, ветвление только при `mode === 'replace'`; сигнатуры неизменны; Т-9 (переходы/анимация), Т-1…Т-6 (штатные переходы по сценарию).
- **Риск:** случайное изменение поведения «Все →» или кнопки «Записаться →». → **Митигация:** Задача 5 (контрольная, без изменений); `_feedGoBook` не трогается; Т-3, Т-5.
