# Спецификация релиза v1.31.0

- **Дата планирования:** 2026-05-31
- **Тип релиза:** minor (вводится новая архитектура персистентного кэширования данных каталога и цен; публичный контракт — API YCLIENTS, поток бронирования, видимый UI — не меняется. Больше, чем patch: вытесняется решение ADR-012 о memory-only кэше и добавляется новый модуль)
- **Цели:** Ускорить ощущаемый запуск и навигацию PWA без изменения видимого поведения: мгновенный первый кадр каталога и цен из персистентного кэша (SWR без TTL), меньше избыточных запросов при повторных входах на главную.

## Источники

- BA-требования: `ba-req/ba-requirements-036.md`
- ADR: `adrs/adr-033.md` (вытесняет ADR-012 **только** в части persist кэша цен; наследует схему сбора/форматирования цен из ADR-012)

## Объём релиза

Затрагиваемые файлы: `docs/app.js`, `docs/modules/state.js`,
`docs/modules/github.js`, новый `docs/modules/dataCache.js`, `docs/sw.js`,
`docs/version.json`. Обоснование решений — в ADR-033; здесь не дублируется.

Включено: основные задачи (ФТ-1..ФТ-10, НФТ) + опциональная ФТ-11.
**Исключено из релиза:** ФТ-12 (ленивая загрузка EmailJS) — см. ADR-033, Вопрос 6

не изменяется.

### Задача 1: Модуль персистентного кэша `dataCache.js`

- **Что:** Новый минимальный модуль чтения/записи снимка каталога и цен в
  localStorage, с версионированием ключей и обёрткой записи в `try/catch`.
- **Где:** новый файл `docs/modules/dataCache.js`.
- **Контракт (публичные функции модуля):**
  - `readCatalogSnapshot()` → `{ services, masters } | null` (синхронно; `null`,
    если снимка нет/повреждён/иная версия схемы).
  - `writeCatalogSnapshot(services, masters)` → `void` (в `try/catch`).
  - `readPricesSnapshot()` → `{ servicePriceRange, staffServicePrice } | null`.
  - `writePricesSnapshot(servicePriceRange, staffServicePrice)` → `void` (в `try/catch`).
  - Ключи в неймспейсе `yc_*` с суффиксом версии схемы (например,
    `yc_catalog_cache_v1`, `yc_prices_cache_v1`); при несовпадении версии снимок
    игнорируется.
- **Критерии готовности:**
  - Снимок **не содержит** `YC.token`/`userToken`/секретов/новых персональных
    данных (КП-9, НФТ-4).
  - Любая запись обёрнута в `try/catch`; ошибка/переполнение не пробрасывается
    (КП-8).
  - Чтение синхронное (localStorage), без `await`.
- **Связь с ADR:** ADR-033, «Модель кэша», Вопрос 1, Вопрос 2.

### Задача 2: Гидрация каталога и цен из снимка при старте + запись снимка

- **Что:** При старте до сетевых запросов синхронно восстановить
  `SERVICES_DATA`/`MASTERS_DATA` и `servicePriceRange`/`staffServicePrice` из
  снимка и отрисовать каталог; параллельно/после — сетевое обновление (SWR);
  после успешного сетевого ответа перезаписать соответствующий снимок.
- **Где:** `docs/app.js` (boot перед `initApp`/в `initApp`,
  `_collectStaffPrices` ~233-266); `docs/modules/state.js` (сеттеры/гидрация
  `servicePriceRange`/`staffServicePrice`).
- **Контракт:**
  - При наличии снимка каталог и список мастеров (включая диапазоны цен)
    рендерятся **до** завершения сетевых запросов.
  - Снимок каталога пишется **после успешного** `Promise.all` в `initApp` (когда
    `svcRes`/`staffRes` валидны).
  - Снимок цен пишется в конце `_collectStaffPrices` (после фонового сбора).
  - Сетевые данные при приходе **заменяют** отображённые (stale-while-revalidate);
    видимого мигания сверх текущего поведения нет.
  - `state.js`: способ заполнить/очистить `servicePriceRange`/`staffServicePrice`
    из снимка (мутация существующих module-level объектов или сеттер).
- **Критерии готовности:**
  - КП-1: при повторном холодном старте с заполненным кэшем каталог и мастера
    видны мгновенно, до прихода сетевого ответа (DevTools → Network).
  - КП-2: после рендера данные обновляются из сети в фоне; обновление без
    видимого мигания сверх текущего.
  - КП-3: каталог рендерится без ожидания N пер-мастерных запросов цен; диапазоны
    есть сразу из кэша и/или дорисовываются в фоне.
  - Первый запуск (нет снимка): поведение как сегодня (сеть → рендер).
- **Связь с ADR:** ADR-033, Вопрос 2 (SWR), Вопрос 3 (N+1 = кэш + фон), ОВ-1.

### Задача 3: Офлайн и устойчивость хранилища

- **Что:** При наличии снимка приложение рендерит каталог/мастеров/цены офлайн;
  при недоступном/переполненном хранилище старт деградирует до режима «без кэша».
- **Где:** `docs/modules/dataCache.js`, `docs/app.js`.
- **Контракт:**
  - Offline + есть снимок → каталог, мастера, цены рендерятся из снимка.
  - Offline без снимка (хранилище очищено) → текущее поведение (пусто/скелетон).
  - Сбой `setItem`/`getItem`/JSON.parse → приложение работает без кэша, без
    необработанных ошибок.
- **Критерии готовности:**
  - КП-4: офлайн с кэшем — рендер из кэша; офлайн без кэша — как сегодня.
  - КП-8: при эмуляции ошибки `setItem`/приватного режима старт не падает; ноль
    необработанных ошибок в консоли.
- **Связь с ADR:** ADR-033, «Модель кэша» (запись/чтение, деградация).

### Задача 4: Троттлинг фоновых обновлений на `s-home`

- **Что:** Повторные входы на `s-home` в пределах 60 с не инициируют новые
  `_fetchAndMergeServerRecords` и `_ghRead`; первый вход и холодный старт
  обновляют как прежде.
- **Где:** `docs/app.js` (обработчик `registerOnEnter('s-home')` ~58-71).
- **Контракт:**
  - Timestamp последнего фонового обновления хранится **в памяти модуля**
    (не в localStorage).
  - В пределах 60 с от последнего обновления повторный вход на `s-home`
    пропускает оба сетевых запроса; `renderHomeHero`/`updateInboxBadge` и рендер
    из локального состояния выполняются как обычно.
  - Холодный старт (`checkSession`/`initApp`) и первый вход — обновляют всегда.
- **Критерии готовности:**
  - КП-5: повторные входы на `s-home` в пределах 60 с не порождают новых запросов
    `/records` и `_ghRead` (DevTools → Network); первый вход/старт запросы делает.
- **Связь с ADR:** ADR-033, Вопрос 4.

### Задача 5: Убрать cache-bust в `_ghRead`

- **Что:** Удалить `?_=Date.now()` из URL в `_ghRead`; актуальность регулируется
  троттлингом (Задача 4), а не cache-busting.
- **Где:** `docs/modules/github.js` (`_ghRead`, строка 11).
- **Контракт:** запрос GitHub-данных идёт по URL без параметра `?_=<timestamp>`.
- **Критерии готовности:**
  - КП-6: в запросе `_ghRead` отсутствует `?_=<timestamp>` (DevTools → Network).
- **Связь с ADR:** ADR-033, Вопрос 5.

### Задача 6 (опц., ФТ-11): Сокращение дублирующих ре-рендеров каталога при старте

- **Что:** Убрать избыточные вызовы `renderServices()` при старте (сейчас ≥4) без
  изменения итогового отображения.
- **Где:** `docs/app.js` (`initApp` ~205/227/`_collectStaffPrices` 265, boot 287).
- **Контракт:** итоговое состояние каталога идентично текущему; число полных
  ре-рендеров при старте сокращено.
- **Критерии готовности:**
  - КП-10 (опц.): число полных ре-рендеров при старте сокращено относительно
    текущего (≥4) без визуальных отличий (наблюдение через временное
    инструментирование при разработке).
- **Связь с ADR:** ADR-033, Вопрос 6 (6A — ФТ-11 включён).

- **Контракт:**

  - `CACHE_VERSION` поднята `v87` → `v88`. Стратегии SW и `NETWORK_ONLY_HOSTS` не

    `sw.js` нет.
- **Связь с ADR:** ADR-033, «Влияние на код».

## Изменения публичного контракта

Внешний API YCLIENTS, поток бронирования, видимый UI, состав экранов и контент
**не меняются** (ФТ-8, НФТ-2). Внутренне добавляются: новый модуль
`dataCache.js` и новые ключи localStorage (`yc_catalog_cache_v1`,
`yc_prices_cache_v1`). Миграция не требуется — снимок при несовпадении версии
схемы игнорируется (forward-compatible). Снимок не содержит токенов/секретов.

## Тесты

Автотестов в проекте нет — приёмка ручная (DevTools Network/Console/Application).
Обязательные ручные проверки на основном сценарии:
- Повторный холодный старт с кэшем → мгновенный каталог/цены до ответа сети (КП-1, КП-3).
- Фоновое обновление без видимого мигания (КП-2).
- Офлайн с кэшем / без кэша (КП-4).
- Эмуляция ошибки `setItem` / приватный режим → старт не падает, ноль ошибок (КП-8).
- Повторные входы на `s-home` < 60 с → нет новых `/records` и `_ghRead` (КП-5).
- URL `_ghRead` без `?_=` (КП-6).
- Local Storage не содержит токена YCLIENTS (КП-9).
- Сквозной сценарий логин → услуга → мастер → слот → бронь без регрессий, ноль
  ошибок в консоли, PWA installable (КП-7).
- (опц.) Сокращение ре-рендеров (КП-10).
- e2e smoke `npm run e2e` проходит (если применимо к изменённым путям).

## Риски и митигации

- **Риск:** Снимок с устаревшими ценами/услугами виден до прихода сети.
  → **Митигация:** SWR без TTL — сеть всегда догоняет и заменяет; данные почти
  статичны (ADR-033, Вопрос 2).
- **Риск:** Переполнение/недоступность localStorage ломает старт.
  → **Митигация:** запись в `try/catch`, деградация до «без кэша» (КП-8, ФТ-9).
- **Риск:** Случайное попадание токена в снимок.
  → **Митигация:** кэшируются только нормализованные поля каталога/цен; проверка
  Local Storage в приёмке (КП-9); ревью diff.
- **Риск:** Троттлинг скрывает свежие записи дольше нужного.
  → **Митигация:** окно 60 с; старт и первый вход всегда обновляют; рендер идёт из
  локального состояния.
- **Риск:** Забытое поднятие `CACHE_VERSION` → пользователи на старом SW.
  → **Митигация:** Задача 7, КП-12; пункт чек-листа.
