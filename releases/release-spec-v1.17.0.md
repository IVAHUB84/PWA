# Спецификация релиза v1.17.0

- **Дата планирования:** 2026-05-29
- **Тип релиза:** minor (обратимо-совместимое добавление функциональности: типизированные посты + новый фильтр + маршрутизация CTA; публичный контракт `posts.json` расширяется обратносовместимо).
- **Цели:** перевести ленту студии в разрез реальных категорий/услуг YCLIENTS; дать пост «по услуге» с CTA «Записаться» на карточку этой услуги; добавить фильтр ленты по типу поста.

## Источники

- BA-требования: `ba-req/ba-requirements-022.md`
- ADR: `adrs/adr-019.md` (модель типизированного поста, маршрутизация CTA, фильтр по типу). Переиспользуются `adrs/adr-010.md` (`_feedCardHtml` для `s-feed`/`s-post`), `adrs/adr-013.md` (`openServiceCard`/`s-service`).

## Объём релиза

### Задача 1: Модель типизированного поста

- **Что:** расширить модель поста полем `type` и привязкой к услуге; сохранить строковое `cat` для отображения/фильтра каталога.
- **Где (файлы/модули):** `docs/modules/admin.js` (`publishPost`, `admin.js:102-134`).
- **Контракт (модель поста):**
  ```
  {
    id: Number,
    type: 'free' | 'category' | 'service',
    cat: String,            // category/service → реальное название категории YCLIENTS; free → ''
    serviceId: String,      // только type === 'service' (== SERVICES_DATA.id)
    serviceName: String,    // только type === 'service'
    text: String,
    image: String | null,
    date: String,
    draft: Boolean
  }
  ```
  Отсутствие поля `type` (старые посты) трактуется как `'free'` везде при чтении.
- **Критерии готовности:**
  - При публикации `free` — пишутся `type:'free'`, `cat:''`, без `serviceId`/`serviceName`.
  - При публикации `category` — `type:'category'`, `cat` = выбранная реальная категория.
  - При публикации `service` — `type:'service'`, `serviceId`/`serviceName` из `SERVICES_DATA`, `cat` = категория этой услуги (`SERVICES_DATA.find(...).cat`).
  - `posts.json` после публикации содержит новые поля; старые посты в файле не повреждаются.
- **Связь с ADR:** ADR-019, Решение 1 (Вариант A).

### Задача 2: Админ-форма выбора типа и привязки

- **Что:** заменить хардкод-чипы категорий на переключатель типа поста + условные контролы (селект реальной категории, поиск/выбор реальной услуги). Источник — `SERVICES_DATA`/`catMap`, без новых запросов.
- **Где:** `docs/prototype.html` (экран `#s-admin-post`, строки ~741-749 — блок «Категория»/чипы); `docs/app.js` (хелпер заполнения списков из `SERVICES_DATA`/`catMap`; пересмотр/замена `adminCat`, `app.js:304`); при необходимости `docs/style.css`.
- **Контракт (UI):**
  - 3 чипа типа: «Свободный» / «По категории» / «По услуге» (тексты на русском).
  - «По категории»: селект/чипы реальных категорий (`new Set(SERVICES_DATA.map(s => s.cat))` или `Object.values(catMap)`), сортировка по алфавиту.
  - «По услуге»: поле поиска + список услуг из `SERVICES_DATA` (фильтр по `name`); выбор фиксирует `serviceId`+`serviceName`+`cat`.
  - Условные контролы показываются/скрываются по выбранному типу; для «Свободный» привязка не выбирается.
- **Критерии готовности:**
  - Списки категорий/услуг наполнены реальными данными YCLIENTS (не хардкод).
  - Новых запросов к YCLIENTS не добавлено (используются уже загруженные `SERVICES_DATA`/`catMap`).
  - При пустом обязательном выборе (категория/услуга для соответствующего типа) — публикация блокируется с понятным сообщением на русском.
- **Связь с ADR:** ADR-019, Решение 1, §«Влияние на код».

### Задача 3: Маршрутизация CTA по типу поста + graceful fallback

- **Что:** CTA «Записаться» ведёт себя по типу: услуга → карточка услуги; категория → отфильтрованный каталог; свободный → CTA скрыт. При удалённой привязке — фолбэк без ошибок.
- **Где:** `docs/modules/feed.js` (`_feedCardHtml`, `feed.js:54-74`; `_feedGoBook`, `feed.js:111-114`).
- **Контракт:**
  - В `_feedCardHtml`: для `free` (вкл. посты без `type`) кнопка `.feed-book` **не рендерится**. Для `service`/`category` кнопка несёт `data-type` и ссылку (`data-ref` = `serviceId` для услуги, `cat` для категории).
  - `_feedGoBook(type, ref)`:
    - `service`: `SERVICES_DATA.find(s => s.id === ref)` существует → `openServiceCard(ref)` (сам делает `go('s-service')`); иначе фолбэк.
    - `category`: `ref` присутствует среди категорий `SERVICES_DATA` → `filterCategory(ref)` + `go('s-services')`; иначе фолбэк.
    - фолбэк (ОВ-8): `filterCategory('Все')` + `go('s-services')`, без ошибок в консоли.
- **Критерии готовности:**
  - Пост «по услуге» открывает `s-service` именно этой услуги.
  - Пост «по категории» с существующей категорией не даёт «Ничего не найдено».
  - Свободный пост — без кнопки «Записаться».
  - Удалённый `serviceId`/категория не вызывают ошибок в консоли; CTA ведёт в общий каталог.
  - Метка CTA на русском («Записаться →»).
- **Связь с ADR:** ADR-019, Решение 2 (Вариант A); ADR-013 (`openServiceCard`), ADR-010 (единообразие `s-feed`/`s-post`).

### Задача 4: Фильтр ленты по типу поста

- **Что:** заменить текущий фильтр по строке `cat` на фильтр по типу поста (3 фиксированных чипа).
- **Где:** `docs/modules/feed.js` (`_renderFeedCats`, `feed.js:87-96`; `_feedFilterCat`, `feed.js:98-109`).
- **Контракт:**
  - Чипы: «Все» / «Акции и услуги» / «Новости» (русский).
  - «Акции и услуги» = `type ∈ {service, category}`; «Новости» = `type === 'free'` (вкл. посты без `type`).
  - Набор чипов фиксирован (не строится из данных).
- **Критерии готовности:**
  - Выбор чипа фильтрует список `feedList` по типу; «Все» показывает все опубликованные посты.
  - Старые посты (без `type`) попадают в «Новости».
- **Связь с ADR:** ADR-019, Решение 3 (Вариант A); решение ОВ-4.

### Задача 5: Совместимость рендера старых постов и превью на главной

- **Что:** гарантировать, что посты без `type`/с пустым `cat` рендерятся без ошибок в `s-feed`, `s-post` и превью на главной; тег/иконка не падают.
- **Где:** `docs/modules/feed.js` (`_feedCardHtml`, `renderPostScreen`); `docs/modules/profile.js` (`_renderHomeFeedPreview`, `profile.js:135-163`); `docs/modules/admin.js` (`_renderAdminFeedFromCache`, `admin.js:149-175`).
- **Контракт:** при отсутствии `cat` тег скрывается/пустой, иконка — generic (`📝`); при отсутствии `type` поведение = `free`.
- **Критерии готовности:** старые посты отображаются во всех трёх местах без ошибок в консоли; превью на главной не сломано.
- **Связь с ADR:** ADR-019, решение ОВ-2; ADR-010.

### Задача 6: Поднятие CACHE_VERSION и bump build

- **Что:** так как затронуты файлы `APP_SHELL` (`feed.js`, `admin.js`, `profile.js`, `prototype.html`, возможно `app.js`/`style.css`), поднять `CACHE_VERSION` в `docs/sw.js` (`v65` → `v66`) и обновить `docs/version.json` (build `32` → `33`, `version: v1.17.0`, дата).
- **Где:** `docs/sw.js` (строка 1, `CACHE_VERSION`); `docs/version.json`.
- **Критерии готовности:** `CACHE_VERSION` = `v66`; `version.json` = `{"build":33,"version":"v1.17.0","date":"2026-05-29"}`.
- **Связь с ADR:** правило проекта (CLAUDE.md, НФТ-6).

## Изменения публичного контракта

- **`posts.json` (GitHub-синхронизация):** добавляются необязательные поля `type`, `serviceId`, `serviceName`; обязательное `cat` сохраняется (для `free` = `''`). Изменение обратносовместимо: посты, записанные старой версией (только `cat`), читаются как `free`; новые поля валидатором `_VALID_POST` (`github.js:4-7`) не отбрасываются. Валидатор `_VALID_POST` по существу не меняется (проверка `typeof p.cat === 'string'` пропускает `''`); при правке — только в сторону терпимости, не ломая приём старых постов.
- **Миграция данных:** не требуется (graceful, ОВ-2). Старые посты перепубликовываются администратором при необходимости.

## Тесты

Автотестов в проекте нет; ниже — обязательный набор ручных проверок (для тестировщика):

- **end-to-end «Акция на стрижку 20%»:** создать пост `type:'service'` с услугой «Стрижка», текстом «Акция на стрижку 20%» → в ленте/на главной/в `s-post` нажать «Записаться» → открывается карточка услуги «Стрижка» (`s-service`).
- Пост `type:'category'` с существующей категорией → CTA открывает каталог, отфильтрованный по ней, без «Ничего не найдено».
- Свободный пост → кнопки «Записаться» нет.
- Удалить/подменить `serviceId` поста на несуществующий → CTA ведёт в общий каталог, в консоли нет ошибок.
- Кросс-девайс: опубликовать пост по услуге, прочитать `posts.json` на другом устройстве → поля `type`/`serviceId`/`serviceName` пришли, CTA работает.
- Фильтр ленты: «Все»/«Акции и услуги»/«Новости» фильтруют корректно; старый пост — в «Новости».
- Старые посты (только `cat`) отображаются в `s-feed`, `s-post`, превью на главной без ошибок.
- DevTools → Console: ноль ошибок; Application → Service Workers: active; PWA installable.

## Риски и митигации

- **Риск:** `_VALID_POST` отбросит новые посты при синхронизации, если убрать `cat`. → **Митигация:** `cat` сохраняется всегда (для `free` = `''`); валидатор не ужесточать.
- **Риск:** реальные категории отсутствуют в словарях градиентов/иконок (`feed.js:5-13`) → блёклый плейсхолдер. → **Митигация:** generic-фолбакк допустим (не регрессия); расширение словарей — отдельный техдолг (ADR-019).
- **Риск:** регрессия превью на главной (отдельный дубль рендера). → **Митигация:** Задача 5 явно проверяет `_renderHomeFeedPreview` на постах без `cat`/`type`.
- **Риск:** удалённая привязка ломает CTA. → **Митигация:** фолбэк ОВ-8 в `_feedGoBook`.
