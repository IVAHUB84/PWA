# Спецификация релиза v1.2.0

- **Дата планирования:** 2026-05-28
- **Тип релиза:** minor
- **Цели:** Показывать на экране мастеров `s-masters` (в режиме перехода от выбранной услуги) только тех мастеров, которые реально оказывают эту услугу по данным YCLIENTS, чтобы исключить заведомо некорректные записи и снизить путаницу при выборе.

## Источники

- BA-требования: `ba-req/ba-requirements-002.md`
- ADR: `adrs/adr-002.md`

## Обоснование версии

MINOR: добавляется обратимо-совместимое уточнение функциональности (фильтрация состава списка мастеров) без ломающих изменений публичного контракта. Навигация (`selectMaster`, `selectAnyMaster`, `openMasterCard`), внешний вид карточек, логика бронирования, режимы `masterPreSelected` и «Все мастера студии» — сохраняются. Источник данных услуг и мастеров (YCLIENTS) не меняется. Меняется только **состав** списка на `s-masters` в режиме «от услуги» — это поведение экрана, не публичный API/CLI/схема данных.

## Объём релиза

### Задача 1: Получение мастеров по услуге из YCLIENTS

- **Что:** Реализовать получение списка мастеров, оказывающих конкретную услугу, через единственный запрос к YCLIENTS, параметризованный YCLIENTS-ID услуги. Запрос и его деградация изолируются в одной функции для переиспользования и тестируемости.
- **Где (файлы/модули):**
  - `docs/modules/api.js` (новая экспортируемая функция, например `fetchStaffByService(serviceId)`; имя на усмотрение разработчика)
- **Контракт:**
  - Функция принимает YCLIENTS-ID услуги (значение `state.serviceId`, которое уже равно `String(s.id)` из `setServicesData`).
  - Внутри — один вызов `YC.get('/book_staff/' + YC.company, { 'service_ids[]': serviceId })`. Конкретная форма передачи параметра (`service_ids[]` либо иной поддерживаемый YCLIENTS вариант) проверяется разработчиком против реального ответа API; критерий — YCLIENTS возвращает именно мастеров этой услуги.
  - Возвращаемое значение различает три исхода: (а) успех с непустым списком — массив staff-объектов YCLIENTS; (б) успех с пустым списком — пустой массив; (в) ошибка/таймаут/`success:false` — явный маркер ошибки (например, `null`), отличимый от пустого списка. Это нужно, чтобы вызывающий код различал «нет подходящих мастеров» (пустое состояние) и «API недоступен» (fail-open).
  - Функция не обращается к DOM.
- **Критерии готовности:**
  - Один сетевой запрос на вызов (не N запросов по мастерам).
  - Три исхода (непустой / пустой / ошибка) различимы вызывающим кодом.
  - Переиспользует существующий `YC.get` и захардкоженный токен — токен не трогается.
- **Связь с ADR:** ADR-002, Вопрос 1 (Вариант A), раздел «Влияние на код».

### Задача 2: Кэш результата фильтрации на время сессии

- **Что:** Кэшировать ответ Задачи 1 по ключу `serviceId` в памяти на время сессии, чтобы повторное открытие `s-masters` для той же услуги не делало повторный сетевой запрос.
- **Где (файлы/модули):**
  - `docs/modules/masters.js` либо `docs/modules/state.js` (структура кэша в памяти — module-scope или поле в `state`; на усмотрение разработчика, но не `localStorage` — данные должны быть свежими каждую сессию)
- **Контракт:**
  - Ключ кэша — `serviceId`. Значение — результат Задачи 1 (список staff-ID или маркер исхода).
  - Кэш живёт в памяти, не персистится; сбрасывается при перезагрузке страницы.
  - Исход «ошибка» (fail-open) **не кэшируется** как окончательный — при следующем открытии допускается повторная попытка запроса (чтобы временный сбой API не залипал на сессию).
- **Критерии готовности:**
  - Повторное открытие `s-masters` для той же услуги в рамках сессии не инициирует повторный запрос (проверяется по вкладке Network).
  - Кэш не использует `localStorage`/`sessionStorage`.
- **Связь с ADR:** ADR-002, Вопрос 2 (Вариант 2A), НФТ-2.

### Задача 3: Асинхронная фильтрация в `renderMasters` (режим «от услуги»)

- **Что:** Переработать ветку «от услуги» в `renderMasters` (`docs/modules/masters.js`): вместо синхронной фильтрации по `svc.cat`/`m.cats` — показать индикатор загрузки, получить мастеров по услуге (Задача 1 + кэш Задачи 2), построить итоговый список как **пересечение** ответа YCLIENTS с `MASTERS_DATA` по `id`, применить деградацию и пустое состояние. Ветки `browseAll` и `masterPreSelected` не затрагиваются.
- **Где (файлы/модули):**
  - `docs/modules/masters.js` (функция `renderMasters` — становится асинхронной в ветке «от услуги»)
- **Контракт:**
  - **Источник состава:** мастера из ответа Задачи 1, сопоставленные с `MASTERS_DATA` по `id` (чтобы переиспользовать подготовленные поля карточки: `avatar`, `grad`, `fav`, `role`, `exp`). Мастер, присутствующий в ответе YCLIENTS, но отсутствующий в `MASTERS_DATA`, — добавляется из staff-объекта ответа (минимально достаточные поля для `_masterCardHtml`) либо отбрасывается; выбор на усмотрение разработчика, но список не должен содержать «битых» карточек.
  - **Критерий фильтрации — только данные YCLIENTS.** Прежний код фильтрации по `svc.cat` и `m.cats` (`const bycat = masters.filter(m => m.cats && m.cats.includes(svcCat)); ...`) удаляется. Поле `m.cats` больше не участвует в формировании состава `s-masters`. (ФТ-2, ФТ-3, КП-2)
  - **Индикатор загрузки:** на время запроса в контейнере `#mastersList` (или эквивалентном месте экрана) показывается русскоязычный индикатор/состояние загрузки; экран не «зависает» без обратной связи. (НФТ-2, КП-8) Если результат уже в кэше (Задача 2) — индикатор можно не показывать.
  - **Fail-open:** если Задача 1 вернула маркер ошибки — показать **всех** мастеров из `MASTERS_DATA` (как в режиме без фильтрации), кнопка «Любой мастер» остаётся видимой. (BA вопрос 1, НФТ-3, КП-9)
  - **Пустое состояние:** если Задача 1 вернула **успешный пустой** список — в `#mastersList` показывается понятное русскоязычное сообщение (текст на усмотрение разработчика, в едином тоне с остальными экранами, например «На эту услугу сейчас нет доступных мастеров»), кнопка «Любой мастер» **скрывается** (как `anyMasterEl.style.display = 'none'`). (ФТ-6, BA вопрос 2, КП-5)
  - **Кнопка «Любой мастер» и заголовки:** в обычном (непустом, не fail-open) случае поведение `anyMasterEl`/`masterLabelEl`/`mastersSub` сохраняется как сейчас (показ подзаголовка `${svc.name} · ${svc.dur} мин`). (ФТ-4, КП-3)
  - **Сортировка и карточки:** сортировка по избранному (`fav` вперёд), вызов `_masterCardHtml`, дедуп по `dataset.fp` — сохраняются без изменений. Затенение недоступных мастеров (`avail:false`, `opacity:0.55`) — текущее поведение, не меняется (BA вопрос 3, ФТ-7, КП-6).
  - **Гонки рендера:** при быстрой смене услуги/экрана асинхронный результат не должен перезаписывать актуальный список устаревшими данными (например, проверка актуального `state.serviceId` перед записью в DOM). 
- **Критерии готовности:**
  - После выбора услуги на `s-masters` отображаются только мастера из ответа YCLIENTS для этой услуги; не оказывающие услугу отсутствуют. (КП-1)
  - Состав следует за YCLIENTS, а не за статической таблицей: фильтрация по `m.cats` удалена. (КП-2)
  - При ошибке API показываются все мастера, «Любой мастер» виден. (КП-9)
  - При пустом успешном ответе показано сообщение, «Любой мастер» скрыт. (КП-5)
  - Режимы `browseAll` (ветка `state._mastersAll`) и `masterPreSelected` не затронуты и работают как прежде. (КП-4, КП-7)
- **Связь с ADR:** ADR-002, Вопрос 1 (Вариант A), Вопрос 2 (Вариант 2A), раздел «Границы решения».

### Задача 4: Прекращение использования `cats` как критерия фильтрации

- **Что:** Зафиксировать, что поле `cats` (`docs/modules/state.js`, заполняется в `docs/app.js` по совпадению имени) более не является критерием фильтрации мастеров. Удаление самого поля из модели данных — НЕ в объёме релиза (чтобы не задеть смежный код).
- **Где (файлы/модули):**
  - `docs/modules/masters.js` (удаление ветки фильтрации по `cats` — реализуется в Задаче 3)
  - `docs/modules/state.js` и `docs/app.js` (поле `cats` остаётся как есть; не добавлять новых зависимостей от него)
- **Контракт:**
  - В коде, формирующем состав `s-masters`, нет обращений к `m.cats`/`svc.cat` как к критерию фильтрации.
  - Поле `cats` в `MASTERS_DATA` и его наполнение в `app.js` не удаляются в этом релизе (технический долг, ADR-002 «Последствия»).
- **Критерии готовности:**
  - Поиск по `cats` в `docs/modules/masters.js` не находит использования в логике фильтрации списка.
- **Связь с ADR:** ADR-002, Вопрос 2 (Вариант 2A), «Технический долг».

### Задача 5: Service Worker — поднятие `CACHE_VERSION` при необходимости

- **Что:** Проверить, изменился ли состав кэшируемых ресурсов `APP_SHELL` в `docs/sw.js`. В этом релизе новых файлов не создаётся (изменяются существующие модули `masters.js`/`api.js`/`state.js`, уже входящие в `APP_SHELL`). Поднять `CACHE_VERSION` для гарантированной доставки обновлённого кода клиентам.
- **Где (файлы/модули):**
  - `docs/sw.js` (константа `CACHE_VERSION`)
- **Контракт:**
  - Если содержимое модулей из `APP_SHELL` изменилось (а оно изменилось — `masters.js`, `api.js`), `CACHE_VERSION` повышается на одну ступень, чтобы клиенты получили новый код, а не закэшированный старый.
  - Состав `APP_SHELL` (перечень файлов) не меняется — новых файлов не добавляется.
- **Критерии готовности:**
  - `CACHE_VERSION` повышен (значение «до» → «после» зафиксировать в release-notes).
  - PWA устанавливается, офлайн-режим не регрессирует. (КП-10)
- **Связь с ADR:** ADR-002, «Влияние на код» (`docs/sw.js`), НФТ-5.

## Изменения публичного контракта

Публичный контракт не меняется:

- Источник и поля данных мастеров и услуг (YCLIENTS) — без изменений.
- Навигация и переходы (`selectMaster`, `selectAnyMaster`, `openMasterCard`, `bookServiceFromMaster`, режим `masterPreSelected`, «Все мастера студии») — без изменений.
- Внешний вид карточек мастера (`_masterCardHtml`) и логика бронирования — без изменений.

Меняется только **состав** списка мастеров на `s-masters` в режиме перехода от услуги. Миграция данных не требуется.

## Тесты

Минимальный набор, обязательный к появлению в релизе (логика фильтрации/деградации — модульно на чистых функциях, где возможно; рендер/сеть — ручной чек-лист, т.к. в проекте нет UI-раннера):

- **Модульные / логика (там, где функция выделяема и чиста):**
  - Функция Задачи 1 различает три исхода: непустой ответ → массив; пустой успешный ответ → пустой массив; ошибка/`success:false`/таймаут → маркер ошибки.
  - Построение состава: пересечение ответа YCLIENTS с `MASTERS_DATA` по `id` даёт корректный список; staff-ID, отсутствующих в `MASTERS_DATA`, обрабатываются без «битых» карточек.
  - Кэш (Задача 2): второй вызов для того же `serviceId` не инициирует запрос; исход «ошибка» не кэшируется как окончательный.
- **Ручной чек-лист (DevTools, вкладки Console и Network):**
  - Выбор услуги → `s-masters` показывает только мастеров этой услуги; один запрос `book_staff` с `service_ids`. (КП-1, КП-2)
  - Имитация ошибки API (offline/throttle/блокировка запроса) → показаны все мастера, «Любой мастер» виден. (КП-9)
  - Услуга без мастеров (или имитация пустого ответа) → русскоязычное сообщение, «Любой мастер» скрыт. (КП-5)
  - «Любой мастер» (непустой случай) → продолжение записи без выбора мастера. (КП-3)
  - Переход с карточки мастера (`masterPreSelected`) → фильтрация не применяется, поведение прежнее. (КП-4)
  - «Все мастера студии» (`_browseAllMasters`) → полный список без фильтрации. (КП-7)
  - Индикатор загрузки виден на время запроса; повторное открытие той же услуги — без запроса (кэш). (КП-8, НФТ-2)
  - Внешний вид карточек и бронирование после выбора — без изменений. (КП-6)
  - DevTools → Application → Service Workers: статус active; Lighthouse: PWA installable, офлайн не регрессирует. (КП-10)

## Риски и митигации

- **Риск:** `/book_staff` не поддерживает `service_ids` в ожидаемой форме → отфильтрованный список некорректен. → **Митигация:** Задача 1 требует проверки формы параметра против реального ответа API; при неуспехе срабатывает fail-open (показ всех), запись не блокируется.
- **Риск:** Асинхронный рендер вводит гонку при быстрой смене услуги/экрана → устаревший список перезаписывает актуальный. → **Митигация:** Задача 3 требует проверки актуальности `state.serviceId` перед записью в DOM.
- **Риск:** Задержка на открытии `s-masters` ухудшает UX (НФТ-2). → **Митигация:** один запрос (Вариант A), индикатор загрузки, кэш на сессию (Задача 2).
- **Риск:** Мастер из ответа YCLIENTS отсутствует в `MASTERS_DATA` → «битая» карточка. → **Митигация:** Задача 3 требует корректной обработки таких ID (достраивание полей или отбрасывание без поломки).
- **Риск:** Забыли поднять `CACHE_VERSION` → клиенты получают старый код модулей. → **Митигация:** Задача 5 явно требует подъёма версии; пункт в чек-листе.
- **Риск:** Остаточная зависимость от `cats` где-то ещё → неконсистентность. → **Митигация:** Задача 4 требует грепа по `cats` в `masters.js`; удаление поля отложено как осознанный техдолг (ADR-002).
