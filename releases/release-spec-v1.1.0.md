# Спецификация релиза v1.1.0

- **Дата планирования:** 2026-05-28
- **Тип релиза:** minor
- **Цели:** Заменить текстовый список услуг на экране `s-services` визуальным премиум-каталогом карточек с фотографиями, чтобы повысить воспринимаемую ценность услуг и конверсию в начало записи.

## Источники

- BA-требования: `ba-req/ba-requirements-001.md`
- ADR: `adrs/adr-001.md`

## Обоснование версии

MINOR: добавляется визуальная функциональность без ломающих изменений публичного контракта. Навигация (`selectService`), источник данных услуг (YCLIENTS, поля название/цена/длительность/категория), поиск и фильтр категорий сохраняются. Меняется только внешний вид экрана `s-services` — это не публичный контракт.

## Объём релиза

### Задача 1: Модуль данных и резолвер сопоставления «услуга → фото»

- **Что:** Создать модуль, содержащий данные сопоставления фотографий и чистую функцию резолвинга. Резолвер по объекту услуги возвращает либо ресурс фотографии, либо дескриптор плейсхолдера. Приоритет резолвинга: точное название услуги (после нормализации) → категория → плейсхолдер. Привязка не использует YCLIENTS-ID.
- **Где (файлы/модули):**
  - Новый файл: `docs/modules/serviceImages.js`
- **Контракт:**
  - Экспорт карты точных сопоставлений по названию: ключ — нормализованное название услуги, значение — путь к фото в `docs/images/services/`.
  - Экспорт карты категорий: для каждой категории из `SERVICES_DATA` (`Брови`, `Губы`, `Глаза`, `Эпиляция`) — опциональное фото-по-умолчанию, эмодзи и CSS-градиент плейсхолдера. Должна присутствовать дефолтная запись на случай неизвестной/пустой категории.
  - Функция нормализации названия: приведение к нижнему регистру, схлопывание повторных пробелов, trim, унификация типографики (`ё`→`е`, длинное/короткое тире, кавычки) — конкретный набор правил на усмотрение разработчика, но детерминированный и применяемый одинаково к ключам карты и к названиям услуг.
  - Чистая функция-резолвер `resolveServiceImage(service)` (имя на усмотрение разработчика), возвращающая объект вида: `{ type: 'photo', src }` либо `{ type: 'placeholder', emoji, grad }`. Функция не обращается к DOM и к сети.
  - Резолвер — единственная точка, знающая способ получения URL фото (для будущей миграции на S3, см. ADR-001 «Путь миграции»).
- **Критерии готовности:**
  - Для услуги с точным совпадением названия резолвер возвращает `type: 'photo'` с корректным путём.
  - Для услуги без совпадения по названию, но с известной категорией, резолвер возвращает фото категории (если задано) или плейсхолдер этой категории.
  - Для услуги с пустой/неизвестной категорией возвращается дефолтный плейсхолдер без исключений.
  - Резолвер не зависит от поля `id` услуги.
- **Связь с ADR:** ADR-001, Вопрос 2 (Вариант 2A), Вопрос 1 (контракт резолвера как точка миграции).

### Задача 2: Каталог фотографий в репозитории

- **Что:** Создать каталог для оптимизированных фотографий услуг и положить стартовый набор (допустимы временные/стоковые фото согласно BA, допущение 6.1). Минимум — по одному фото на каждую категорию (для фолбэка) и точечные фото под наиболее ценные услуги.
- **Где (файлы/модули):**
  - Новый каталог: `docs/images/services/`
- **Контракт:**
  - Единое соотношение сторон обложки для всех фото (рекомендуется 4:3 или 3:2; зафиксировать одно значение и применять в CSS Задачи 4).
  - Изображения оптимизированы по весу (целевой ориентир — до ~150 КБ на фото для типового мобильного соединения, НФТ-2). Формат — `webp` либо оптимизированный `jpg`.
  - Имена файлов — латиницей/транслитом, стабильные, соответствующие ключам в `serviceImages.js`.
- **Критерии готовности:**
  - Все пути, на которые ссылается `serviceImages.js`, существуют физически.
  - Для каждой категории из `SERVICES_DATA` доступен либо точечный, либо категорийный фолбэк-ресурс (фото или плейсхолдер).
- **Связь с ADR:** ADR-001, Вопрос 1 (Вариант 1A), НФТ-2.

### Задача 3: Рендер визуальной карточки услуги

- **Что:** Переработать рендер карточки в `services.js` с текстовой строки на карточку-обложку: область изображения сверху (фото или плейсхолдер), под ней — название, цена, длительность. Применить резолвер из Задачи 1 в обеих ветках `renderServices`: обычной и `masterPreSelected` (услуги мастера приходят из YCLIENTS и также должны проходить через резолвер). Сохранить без изменений `selectService`, обработку поиска, фильтра категорий, баннера мастера и пустого состояния.
- **Где (файлы/модули):**
  - `docs/modules/services.js` (функции `_renderList`, `renderServices`; импорт резолвера)
- **Контракт:**
  - Каждая карточка содержит: область изображения (фото или плейсхолдер), название услуги, цену (`priceStr`), длительность (`dur` мин). (ФТ-2)
  - Фото рендерится как `<img>` с атрибутом `loading="lazy"`, обязательным `alt` равным названию услуги (НФТ-7), и `onerror`-обработчиком, переключающим карточку на плейсхолдер без поломки вёрстки (НФТ-3).
  - Плейсхолдер рендерится как блок с градиентом и эмодзи категории из дескриптора резолвера (ФТ-5).
  - Карточка с фото и карточка с плейсхолдером структурно идентичны (одинаковая область обложки), визуально консистентны (КП-4).
  - Тап по карточке вызывает существующий `selectService` с `id` услуги — поведение перехода не меняется (ФТ-3).
  - Все пользовательские строки экранируются через существующий `esc` (XSS-безопасность сохраняется).
  - Пустой результат поиска/фильтра показывает существующее состояние «Ничего не найдено» (ФТ-8, КП-8).
- **Критерии готовности:**
  - На `s-services` услуги отображаются карточками с обложкой; прежняя текстовая строка `svc-line`/`svc-body`/`svc-right` более не используется (КП-1).
  - Услуги без сопоставленного фото показывают градиентный плейсхолдер с эмодзи категории (КП-4).
  - Ветка `masterPreSelected` рендерит карточки тем же способом и через резолвер.
  - Поиск (КП-6), фильтр категорий (КП-7), баннер мастера и переходы (КП-3) работают как прежде.
- **Связь с ADR:** ADR-001, Вопрос 2 и Вопрос 3 (Вариант 3A), раздел «Влияние на код».

### Задача 4: Стили визуальной карточки

- **Что:** Добавить/переработать CSS для карточки-обложки и плейсхолдера, согласованно со светлой и тёмной темами. Сохранить премиум-вид: единые отступы, скругления, тени, консистентная область изображения.
- **Где (файлы/модули):**
  - `docs/style.css` (новые классы карточки; существующие `.svc-card`/`.svc-line`/`.svc-body`/`.svc-right`/`.svc-name`/`.svc-meta`/`.svc-price`/`.svc-cta` переработать или заменить)
- **Контракт:**
  - Область изображения с фиксированным соотношением сторон (то же, что в Задаче 2), `object-fit: cover`, без искажений.
  - Плейсхолдер занимает ту же область, центрирует эмодзи поверх градиента.
  - Использовать существующие CSS-переменные темы (`--bg`, `--surface`, `--text-2`, `--accent`, `--border`, `--shadow`); корректный вид в `[data-theme="dark"]`.
  - Не нарушать `tap-highlight`/`:active`-поведение, заданное для `.svc-card`.
- **Критерии готовности:**
  - Карточки выглядят консистентно в светлой и тёмной темах.
  - Нет горизонтального переполнения и сдвига вёрстки при отсутствии фото (КП-4, НФТ-3).
- **Связь с ADR:** ADR-001, НФТ-1, «Влияние на код».

### Задача 5: Service Worker — регистрация модуля и кэширование фото

- **Что:** Добавить новый модуль данных в APP_SHELL; обеспечить, что фото обслуживаются как same-origin ресурсы с ленивым кэшированием в RUNTIME_CACHE (специальная ветка для фото не обязательна, если текущая same-origin-логика это покрывает). Поднять `CACHE_VERSION`. Фото в APP_SHELL не добавлять.
- **Где (файлы/модули):**
  - `docs/sw.js` (массив `APP_SHELL`, константа `CACHE_VERSION`)
- **Контракт:**
  - `./modules/serviceImages.js` добавлен в `APP_SHELL`.
  - `CACHE_VERSION` повышен (`v33` → `v34`).
  - Фотографии из `docs/images/services/` не входят в `APP_SHELL`; кэшируются лениво (через существующую same-origin-ветку `fetch`-обработчика в RUNTIME_CACHE).
- **Критерии готовности:**
  - PWA устанавливается, офлайн-режим не регрессирует (КП-11).
  - Уже просмотренные фото доступны при повторном открытии офлайн; непросмотренные деградируют до плейсхолдера (НФТ-3).
- **Связь с ADR:** ADR-001, Вопрос 3 (Вариант 3A), НФТ-5.

## Изменения публичного контракта

Публичный контракт не меняется:

- Источник и поля данных услуги (YCLIENTS: название, цена, длительность, категория) — без изменений (ФТ-9).
- Навигация и переходы (`selectService`, режим `masterPreSelected`, баннер мастера) — без изменений (ФТ-3).
- Поиск и фильтр категорий — без изменений в поведении (ФТ-6, ФТ-7).

Меняется только визуальное представление экрана `s-services` (внутренняя вёрстка карточек), что не является публичным контрактом. Миграция данных не требуется.

## Тесты

Минимальный набор тестов, обязательный к появлению в релизе (привязка фото — чистая логика, тестируется модульно; рендер/вёрстка — ручной чек-лист в браузере, т.к. в проекте нет UI-раннера):

- **Модульные (резолвер из Задачи 1):**
  - Точное совпадение названия → возвращается `type: 'photo'` с ожидаемым путём.
  - Нет совпадения по названию, известная категория → фото категории либо плейсхолдер этой категории.
  - Пустая/неизвестная категория → дефолтный плейсхолдер, без исключения.
  - Нормализация: названия, отличающиеся регистром/двойными пробелами/`ё`-`е`/типографикой, резолвятся к одному ресурсу.
  - Резолвер не использует поле `id` (изменение `id` не меняет результат).
- **Ручной чек-лист (DevTools, светлая и тёмная темы):**
  - Карточки с фото и с плейсхолдером структурно идентичны (КП-4).
  - `onerror`: подмена битого пути фото → карточка показывает плейсхолдер без поломки вёрстки (КП-10, НФТ-3).
  - Поиск и фильтр категорий фильтруют каталог карточек (КП-6, КП-7); пустой результат → «Ничего не найдено» (КП-8).
  - Тап по карточке → выбор мастера; в режиме `masterPreSelected` → выбор слота (КП-3).
  - Lighthouse: PWA installable, офлайн не регрессирует (КП-11).

## Риски и митигации

- **Риск:** Названия услуг в YCLIENTS расходятся с ключами карты (опечатки, варианты написания) → услуга не получит точечное фото. → **Митигация:** детерминированная нормализация названия + обязательный фолбэк на фото/плейсхолдер категории; ни одна карточка не остаётся без обложки.
- **Риск:** Тяжёлые фото замедляют экран и прокрутку (НФТ-2). → **Митигация:** оптимизация изображений (целевой вес, фиксированное соотношение сторон), `loading="lazy"`, фото вне APP_SHELL.
- **Риск:** Бинарные фото раздувают git-репозиторий. → **Митигация:** жать перед коммитом; в будущем — миграция на S3 (ADR-001, технический долг).
- **Риск:** Забыли поднять `CACHE_VERSION` → у пользователей старый кэш, новый вид не применится. → **Митигация:** Задача 5 явно требует подъёма `v33`→`v34`; пункт в чек-листе.
- **Риск:** Ветка `masterPreSelected` рендерит фото иначе (услуги из YCLIENTS, иногда `cat: ''`). → **Митигация:** Задача 3 требует прогонять обе ветки через резолвер; дефолтный плейсхолдер покрывает пустую категорию.
