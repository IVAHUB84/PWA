- **Дата планирования:** 2026-05-30
- **Тип релиза:** minor (обратимо-совместимое добавление функциональности: новое необязательное поле `target` в payload/IDB, новый диплинк-поток; публичный контракт push расширяется без слома старых клиентов)
- **Цели:** Превратить уведомления в кликабельные точки входа (диплинки): напоминание → «Записи», промо об услуге/категории → карточка/каталог. ## Объём релиза

### Задача 1: Модуль маршрутизации цели `inboxTarget.js`
- **Что:** Единый клиентский модуль работы с целью уведомления: нормализация/валидация, диспетчер перехода, разбор query-параметра при старте.
- **Где (файлы/модули):** новый `docs/modules/inboxTarget.js`; импортируется в `docs/app.js`, `docs/modules/inbox.js`; добавить в `APP_SHELL` (`docs/sw.js`).
- **Контракт:**
  - `normalizeTarget(raw) → {type, id?} | null` — принимает только `type ∈ {service, records, category}`; для `service` требует числовой `id`, для `category` — непустую строку `id`, для `records` — без `id`. Любой другой случай (отсутствие, `null`, неизвестный `type`, невалидный `id`) → `null` (= «без цели»).
  - `routeToTarget(target)` — `service` → `openServiceCard(id)`; `records` → `go('s-history','tab')`; `category` → `filterCategory(id); go('s-services','tab')` (id = имя категории; если имени нет в `SERVICES_DATA` — ничего не делать); `null`/прочее → ничего, без ошибок.
  - `parseTargetParam()` — читает `?n=` из `location.search`, `JSON.parse` в try/catch, прогоняет через `normalizeTarget`, при валидной цели вызывает `routeToTarget`, затем очищает параметр через `history.replaceState`.
- **Критерии готовности:** одинаковая цель даёт одинаковый переход из любого вызова; невалидный/отсутствующий `?n=` не вызывает ошибок в консоли; функции экспонированы для использования из `inbox.js`/`app.js`.
- **Связь с ADR:** ADR-030, Вопросы A (A1), B (B1), C (C1).

### Задача 2: Точечная отметка о прочтении и кликабельность инбокса
- **Что:** Клик по телу карточки помечает прочитанной именно её и переходит по цели; вход в инбокс больше не помечает всё прочитанным.
- **Где (файлы/модули):** `docs/modules/inbox.js`.
- **Контракт:**
  - `enterInbox()` — убрать вызов `markAllRead()`; оставить `renderInbox()` + `updateInboxBadge()`.
  - `renderInbox()` — у `.inbox-item` тело карточки кликабельно: при клике `setRead(item.id, true)` → `routeToTarget(item.target)` → `updateInboxBadge()` → перерисовка/снятие признака непрочитанного. Клик по «три точки» не должен триггерить переход (меню уже использует `stopPropagation`).
  - `inboxMarkAllRead()` (новый экспорт) — `markAllRead()` + `renderInbox()` + `updateInboxBadge()`.
- **Критерии готовности:**
  - Вход в инбокс через колокольчик не обнуляет бейдж и не помечает всё прочитанным (ФТ-7).
  - Клик по карточке помечает прочитанным только её; бейдж уменьшается на 1 (ФТ-6).
  - Меню «три точки» (сделать непрочитанным / удалить) и `PUSH_RECEIVED` работают как прежде.
- **Связь с ADR:** ADR-030, Вопрос D (D1).

### Задача 3: Переходы по целям (визит / услуга / категория / без цели)
- **Что:** Реализация веток `routeToTarget` поверх существующих переходов; деградация неизвестной цели в «без цели».
- **Где (файлы/модули):** `docs/modules/inboxTarget.js` (использует `openServiceCard`, `filterCategory` из `docs/modules/services.js`, `go` из `docs/modules/navigation.js`).
- **Контракт:**
  - `records` → раздел «Записи» = `go('s-history','tab')`.
  - `service` → `openServiceCard(id)`; если услуги с `id` нет в `SERVICES_DATA` — деградация в «без цели» (без ошибки).
  - `category` → имя сверяется со списком `[...new Set(SERVICES_DATA.map(s=>s.cat))]`; есть → `filterCategory(имя)`+`go('s-services','tab')`; нет → «без цели».
- **Критерии готовности:** КП-1 (визит → «Записи»), КП-2 (услуга → карточка), КП-10 (категория → каталог с фильтром либо «без цели» без ошибок), КП-3 (без цели → только пометить прочитанным).
- **Связь с ADR:** ADR-030, Вопросы A, C.

### Задача 4: Шапка инбокса — корзина + кнопка «Прочитать всё»
- **Что:** Заменить иконку метёлки на корзину (действие «удалить всё» сохраняется); добавить текстовую кнопку «Прочитать всё» над списком.
- **Где (файлы/модули):** `docs/prototype.html` (блок `s-inbox`, шапка `:874-876` и над `#inboxList` `:884`); `docs/style.css`; обработчик `inboxMarkAllRead` из Задачи 2.
- **Контракт:**
  - SVG метёлки → SVG корзины; `onclick` остаётся `clearInboxHistory()`; aria-метка на русском.
  - Над `#inboxList` — текстовая кнопка «Прочитать всё», `onclick="inboxMarkAllRead()"`. Скрыта/`disabled` при отсутствии непрочитанных и при пустом списке.
- **Критерии готовности:** КП-5 (корзина удаляет всю историю с подтверждением), КП-6 (кнопка помечает все прочитанными без удаления, бейдж обнуляется), достаточная тач-зона (НФТ-8).
- **Связь с ADR:** ADR-030, Вопрос D (D1).

### Задача 5: Поле `target` в IDB-записи (синхронизация двух контрактов)
- **Что:** Расширить запись инбокса необязательным полем `target`; синхронно в `inboxStore.js` и `_IDB_*` в `sw.js`; без bump версии.
- **Где (файлы/модули):** `docs/modules/inboxStore.js`, `docs/sw.js` (`_IDB_*`).
- **Контракт:**
  - Запись: `{id, title, body, icon?, ts, read, target?}`; `target` = нормализованный объект или `null`.
  - `addNotification`/`_idbAddNotification` принимают `target` и сохраняют его в записи.
  - `DB_VERSION`/`_IDB_VERSION` остаются `1`. Записи без `target` читаются как «без цели».
  - Комментарий-контракт схемы обновлён в обоих файлах одинаково.
- **Критерии готовности:** КП-7 (схема содержит `target`, синхронизирована в обоих файлах; старые записи без `target` читаются без ошибок).
- **Связь с ADR:** ADR-030, Вопросы A (A1), E (E1).

### Задача 6: Цель в SW — `push` и `notificationclick`
- **Что:** Прокинуть `target` из payload в системное уведомление и в IDB; реализовать диплинк-навигацию в `notificationclick` через единый маршрутизатор.
- **Где (файлы/модули):** `docs/sw.js` (`push`, `notificationclick`), `docs/app.js` (слушатель сообщения + разбор `?n=`).
- **Контракт:**
  - `push`: `target` из `data.target` → `showNotification(title, {..., data:{target}})` и → `_idbAddNotification({title, body, icon, ts, target})`. Сбой записи `target` не ломает показ уведомления.
  - `notificationclick`: взять `target` из `event.notification.data`; есть открытое окно → `client.focus()` + `client.postMessage({type:'NOTIFICATION_TARGET', target})`; нет окна → `clients.openWindow('./?n=' + encodeURIComponent(JSON.stringify(target)))` (при `null`/отсутствии цели — открыть `'./'` как сейчас).
  - `docs/app.js`: слушатель `navigator.serviceWorker` `message` для `NOTIFICATION_TARGET` → `routeToTarget`; при инициализации — `parseTargetParam()`.
- **Критерии готовности:** КП-8 (цель доезжает в payload и IDB, используется и при клике в инбоксе, и в `notificationclick` через единый механизм; одинаковая цель → одинаковый переход), КП-11 (существующий поток push работает, ошибок нет).
- **Связь с ADR:** ADR-030, Вопрос B (B1).

### Задача 7: Выбор цели в композере админки
- **Что:** В композере пуша добавить выбор цели (услуга / категория / без цели, дефолт «без цели»); передать `target` в рассылку.
- **Где (файлы/модули):** `docs/modules/admin.js` (рендер композера + `sendNewPush`), `docs/modules/push.js` (`sendAdminPush`), `docs/prototype.html` (экран `s-admin-push-new`), `docs/style.css`.
- **Контракт:**
  - Контрол выбора цели: «без цели» (дефолт), «услуга» (выбор из `SERVICES_DATA` по `id`), «категория» (выбор имени из `[...new Set(SERVICES_DATA.map(s=>s.cat))]`).
  - `sendNewPush()` формирует `target` (`{type:'service',id}` / `{type:'category',id:имя}` / `null`) и передаёт в `sendAdminPush(title, text, targetClientId, target)`.
  - `sendAdminPush` кладёт `target` в payload `/send` при наличии (необязательно).
- **Критерии готовности:** КП-9 (можно выбрать цель, дефолт «без цели»; цель доезжает в payload и сохраняется в IDB-записи получателя; диплинк работает).
- **Связь с ADR:** ADR-030, Вопрос A; расширение ADR-026.

### Задача 8: Цель напоминаний в Cloudflare Worker (деплой отдельно)

- **Где (файлы/модули):** `cf-worker/src/index.js` (`processReminder`, `handleSend`).
- **Контракт:**
  - `processReminder`: payload напоминания включает `target:{type:'records'}` (без `record_id`).
  - `handleSend`: принять `target` из тела запроса, положить в payload как есть (Worker — транспорт, без валидации формата).
- **Критерии готовности:** КП-1 (напоминание → «Записи»), КП-13 (Worker задеплоен отдельно; выкат не ломает старые/новые версии клиента — старый SW игнорирует `target`, новый клиент при старом Worker трактует отсутствие `target` как «без цели»).
- **Связь с ADR:** ADR-030, Вопрос F (F1).

## Изменения публичного контракта
- **Payload push** (рассылки и напоминания): добавляется необязательное поле `target:{type,id?}` (`type ∈ {service, records, category}`). Старые клиенты игнорируют его (читают `title/body/icon`). Миграция не требуется.
- **IDB-запись инбокса** (`studio-inbox`, store `notifications`, version `1`): запись расширяется необязательным полем `target`. `DB_VERSION` не меняется; старые записи без `target` валидны (= «без цели»). Контракт продублирован в `docs/modules/inboxStore.js` и `docs/sw.js` — синхронизировать.
- **`sendAdminPush(title, body, targetClientId, target?)`** — новый необязательный 4-й параметр.
- **Worker `/send`** — принимает необязательное поле `target` в теле.
- **Глубокая ссылка** `./?n=<encoded target>` — новый query-параметр запуска приложения; разбирается и очищается при старте.

## Тесты
Автотестов в проекте нет (CLAUDE.md). Тестировщик проверяет вручную; этот релиз обязан включить проверки:
- Клик по напоминанию о визите (в инбоксе и по системному push) → раздел «Записи» (КП-1).
- Клик по уведомлению об услуге (в инбоксе и по системному push) → карточка услуги (КП-2).
- Клик по уведомлению без цели → только пометка прочитанным, без перехода, без ошибок на записях старой схемы (КП-3).
- Вход в инбокс не помечает всё прочитанным; клик помечает только кликнутое; бейдж корректен (КП-4).
- Корзина в шапке удаляет всю историю с подтверждением (КП-5).
- Кнопка «Прочитать всё» помечает все прочитанными без удаления, бейдж обнуляется (КП-6).
- IDB-запись содержит `target`, читается в обоих контрактах; старые записи без `target` — «без цели» (КП-7).
- Единый маршрутизатор: одинаковая цель → одинаковый переход из инбокса и из `notificationclick` (КП-8).
- Композер: выбор цели (услуга/категория/без цели, дефолт «без цели»); цель доезжает и сохраняется у получателя (КП-9).
- Категория: клик открывает каталог с фильтром, либо «без цели» без ошибок, если категория исключена/имя отсутствует (КП-10).
- Существующий поток push работает; в консоли ноль ошибок (КП-11).
- `target`/`id` валидируются, строки экранируются `esc()`; диф в пределах объёма (КП-12).
- `CACHE_VERSION` поднята; Worker задеплоен отдельно (КП-13).

## Риски и митигации
- **Риск:** `notificationclick` при открытом приложении только фокусирует окно, не доставляя цель → **Митигация:** для открытого окна `postMessage({type:'NOTIFICATION_TARGET'})` + слушатель в `app.js`; для закрытого — `openWindow('./?n=...')` + `parseTargetParam()`.
- **Риск:** рассинхрон контракта IDB между `inboxStore.js` и `sw.js` → **Митигация:** правка поля `target` и комментария одновременно в обоих файлах; ревью сверяет оба.
- **Риск:** несовместимость выката клиент/Worker → **Митигация:** `target` необязателен в обе стороны; рекомендованный порядок — сначала PWA, затем Worker; проверить старый SW + новый Worker и новый клиент + старый Worker.
- **Риск:** XSS/мусор из `target`/`id` payload → **Митигация:** `normalizeTarget` валидирует тип и `id`; рендер через `esc()`; невалидная цель = «без цели» без ошибок.
- **Риск:** клик по телу карточки конфликтует с меню «три точки» → **Митигация:** меню использует `stopPropagation`; обработчик тела не срабатывает на кнопке меню.
- **Риск:** категория по имени-строке деградирует при переименовании → **Митигация:** сверка имени со списком категорий; отсутствие → «без цели» без ошибок.
- **Риск:** старый кэш SW отдаёт прежний `sw.js`/разметку → **Митигация:** `CACHE_VERSION` v83 → v84; новый модуль в `APP_SHELL`; `controllerchange`-перезагрузка подхватит новый SW.
