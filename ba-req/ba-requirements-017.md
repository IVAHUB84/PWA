# BA-требования 017: Исправление дефектов карусели фото в детальной карточке услуги

- **Источник:** `requirements/req-service-carousel-fix-017.md`
- **Дата:** 2026-05-29
- **Статус:** согласовано (ОВ-5 = фиксим только механику, вид v1.12.0 сохраняем; утверждено заказчиком 2026-05-29; блокеров нет)

## 1. Контекст и бизнес-цель

Studio PWA (студия красоты, онлайн-запись поверх YCLIENTS) — устанавливаемое статическое приложение на vanilla JS/HTML/CSS, отдаётся из `docs/` на GitHub Pages, без рантайм-зависимостей. В рамках требования 015 / релиза v1.12.0 (ADR-014, решение I1) карусель фото на детальной карточке услуги `s-service` была переведена с нативного CSS scroll-snap (ADR-013) на собственную transform-карусель с клонами крайних слайдов и зацикливанием, ведущую логический индекс в JS.

На живом устройстве пользователь подтвердил, что эта карусель **работает некорректно**. Наблюдаются три дефекта (детали в §3):

1. слайды визуально смещены/обрезаны, «подглядывание» (peek) соседних слайдов неравномерно по краям;
2. свайп залипает / не перелистывает, требует нескольких попыток или застревает между слайдами;
3. на стыке зацикливания «последний → первый» (и обратно) виден рывок / перескок / мигание.

Из-за этих дефектов галерея фото работ — ключевой элемент, который должен «продавать» услугу и информировать выбор клиента перед записью, — воспринимается как сломанная, что снижает доверие к приложению и обесценивает функциональность, ранее заявленную в BA-015 (премиальная, очевидно листаемая, надёжная карусель).

**Бизнес-цель:** устранить три наблюдаемых дефекта так, чтобы карусель на детальной карточке услуги листалась надёжно и плавно, слайды были выровнены и обрезаны корректно с равномерным peek по обоим краям, индикатор соответствовал текущему фото, а зацикливание было визуально бесшовным в обе стороны — то есть фактически довести до соответствия критериям приёмки BA-015 (КП-2, КП-3, КП-4, КП-12), которые на живом устройстве не выполняются. Это багфикс ранее принятой функциональности, а не новая фича и не редизайн.

**Стейкхолдеры:**

- Клиент студии — пользователь PWA, просматривает фото работ услуги перед записью; страдает от неработающей листалки.
- Заказчик / владелец студии — заинтересован в корректной, премиальной подаче галереи (уже оплачено решением 015/v1.12.0).
- Бизнес-аналитик / архитектор / разработчик — оформление, диагностика корневых причин, исправление.

## 2. Область (scope)

### В рамках задачи

- Исправление **дефекта смещения/обрезки слайдов**: слайды карусели выровнены, текущий слайд обрезан корректно (по центру кадра без неожиданного среза), peek соседних слайдов равномерен по обоим краям.
- Исправление **дефекта залипания свайпа**: жест свайпа надёжно перелистывает фото, без необходимости повторных попыток и без застревания между слайдами; подряд идущие свайпы листают предсказуемо.
- Исправление **дефекта рывка при зацикливании**: переход через стык «последний → первый» и «первый → последний» визуально бесшовен — без видимого скачка/мигания при сбросе позиции на реальный слайд.
- Корректное поведение **сразу после открытия карточки** (без необходимости «пошевелить» экран, проскроллить или повернуть устройство) и при **повороте/изменении размера экрана**.
- Сохранение существующего корректного поведения: **1 фото** — одиночное скруглённое изображение без листалки; **0 фото** — категорийный плейсхолдер; **битый URL фото** — деградация слайда на плейсхолдер без поломки вёрстки.
- Сохранение существующего внешнего вида галереи из v1.12.0 (скругления, peek-аффорданса, точки-индикаторы, высота/обрезка фото) — по умолчанию исправляется **только механика**, не визуальный дизайн (см. §6, допущение Д-1).
- Отсутствие ошибок в консоли при работе карусели; сохранение производительности свайпа (без тяжёлых перерисовок).

### Вне рамок (explicitly out of scope)

- **Любые изменения за пределами детальной карточки услуги `s-service` и её карусели**: каталог услуг, обложки карточек, выбор мастера, расчёт слотов, бронирование, подтверждение — не затрагиваются.
- **Изменение источника/способа получения галереи фото** (`GET /company/{company}/services/`, слияние поля `photos`, ADR-014 решение G1) — не пересматривается; задача про **отображение/механику** уже полученных фото.
- **Стратегия кэширования фото (stale-while-revalidate в `sw.js`, ADR-014 решение J1)** — не пересматривается, если её правка не требуется собственно для устранения трёх дефектов.
- **Редизайн галереи**: изменение визуального оформления peek, точек, скруглений, высоты, добавление стрелок/счётчика/зума/лайтбокса/автопрокрутки — вне объёма (если только устранение дефекта смещения не потребует скорректировать геометрию — это решает архитектор в рамках сохранения прежнего внешнего вида).
- **Перенос захардкоженного YCLIENTS-токена** (известный техдолг — «не усугублять»).

## 3. Действующие лица и сценарии использования

**Клиент студии** — пользователь PWA (преимущественно мобильный, тач), открывший детальную карточку услуги в рамках сценария записи. Цель — рассмотреть все фото работ услуги, чтобы оценить её перед записью.

Основной сценарий (целевое поведение после исправления):

1. Клиент тапом по услуге в каталоге открывает детальную карточку `s-service`.
2. Сразу видит галерею с выровненными слайдами: текущее фото обрезано корректно, по обоим краям равномерно выглядывают соседние слайды (peek), под галереей — точки-индикаторы.
3. Свайпом листает фото — каждый свайп надёжно перелистывает на одно фото; листание не залипает и не застревает между слайдами.
4. Активная точка соответствует текущему фото.
5. Долистав до последнего фото и продолжив свайп в ту же сторону, попадает на первое (и наоборот) — переход через стык бесшовный, без видимого рывка/мигания.

Текущее (дефектное) поведение, подтверждённое на живом устройстве:

- **Дефект 1 — смещение/обрезка.** Слайды выровнены неровно: peek соседнего слайда неравномерен по левому и правому краю, либо текущий слайд обрезан не по центру.
- **Дефект 2 — залипание свайпа.** Жест свайпа не перелистывает фото с первой попытки, требует нескольких попыток или застревает между слайдами; подряд идущие быстрые свайпы не листают.
- **Дефект 3 — рывок при зацикливании.** На стыке «последний → первый» (и обратно) виден скачок/мигание из-за сброса позиции на клон-слайд.

Граничные сценарии (поведение сохраняется как сейчас, должно остаться корректным после фикса):

- **Два фото** — карусель с зацикливанием и peek работает так же, как при большем числе фото (граничный минимум — 2).
- **Восемь фото** — все слайды и все точки отображаются и листаются корректно.
- **Одно фото** — одиночное скруглённое изображение без листалки, точек и зацикливания.
- **Нет фото** — категорийный плейсхолдер без листалки.
- **Битый URL фото** — слайд деградирует на категорийный плейсхолдер без поломки вёрстки.
- **Поворот / изменение размера экрана** — после поворота слайды остаются выровнены, peek и позиция текущего слайда пересчитаны корректно.

## 4. Функциональные требования

- **ФТ-1.** В детальной карточке услуги `s-service` с двумя и более фото слайды карусели выровнены: текущий слайд обрезан корректно (без неожиданного среза кадра), peek соседних слайдов **равномерен по левому и правому краю**.
- **ФТ-2.** Свайп на тач **надёжно перелистывает** карусель на одно фото в направлении жеста; карусель не застревает между слайдами и не игнорирует осмысленные свайпы; несколько свайпов подряд листают предсказуемо (без потери жестов).
- **ФТ-3.** Зацикливание **визуально бесшовно** в обе стороны: переход со свайпом с последнего фото на первое и с первого на последнее происходит без видимого рывка, перескока или мигания.
- **ФТ-4.** Активная точка-индикатор соответствует фактически отображаемому фото, в том числе после прохождения стыка зацикливания.
- **ФТ-5.** Карусель отображается и листается корректно **сразу после открытия карточки** — без необходимости предварительно проскроллить, «пошевелить» или повернуть экран.
- **ФТ-6.** При **изменении размеров области отображения** (поворот устройства, resize, смена ориентации) выравнивание слайдов, peek и позиция текущего слайда остаются корректными.
- **ФТ-7.** Сохраняется существующее поведение по числу фото: **1 фото** — одиночное скруглённое изображение без листалки/точек/зацикливания; **0 фото** — категорийный плейсхолдер без листалки.
- **ФТ-8.** Сохраняется деградация при ошибке загрузки фото: слайд/изображение с битым/недоступным URL переходит на категорийный плейсхолдер без поломки вёрстки.
- **ФТ-9.** Сохраняется существующий внешний вид галереи из v1.12.0 (скругления, наличие peek, точки-индикаторы, высота и тип обрезки фото) — исправляется механика и геометрия выравнивания, а не визуальный дизайн (см. Д-1).

## 5. Нефункциональные требования

- **НФТ-1 (Совместимость стека).** Решение — vanilla JS/HTML/CSS, код в `docs/` (правки ожидаются в `docs/modules/services.js` и `docs/style.css`); без новых рантайм-npm-зависимостей; без сторонних карусель-библиотек.
- **НФТ-2 (Производительность).** Свайп и перелистывание — плавные, без рывков и тяжёлых перерисовок; синхронизация индикатора не вызывает заметных лагов; пересчёт геометрии при ресайзе не приводит к фризам.
- **НФТ-3 (Отсутствие ошибок).** При работе карусели (открытие карточки, свайп, зацикливание, поворот) в консоли браузера нет ошибок.
- **НФТ-4 (PWA/офлайн не регрессируют).** Устанавливаемость PWA и оффлайн-поведение не ухудшаются; при изменении файлов `APP_SHELL` (`services.js`, `style.css` и иные, входящие в `APP_SHELL`/`sw.js`) поднимается `CACHE_VERSION` в `docs/sw.js` (правило проекта).
- **НФТ-5 (Минимальность изменений).** Изменения строго в пределах детальной карточки `s-service` и её карусели, в объёме, необходимом для устранения трёх дефектов; каталог, выбор мастера, слоты, бронирование, источник фото (G1) и стратегия кэширования (J1) не рефакторятся без необходимости.
- **НФТ-6 (Безопасность / XSS).** URL фото и название услуги (`alt`) выводятся только через `esc()`; клоны слайдов строятся из тех же экранированных данных; вставка непроверенного HTML недопустима.
- **НФТ-7 (Совместимость поведения).** Поведение по числу фото (0/1/N), деградация по `onerror`, скругления, peek и точки остаются консистентными после фикса; никакая ранее работавшая часть детальной карточки (название, цена-диапазон, длительность, описание, кнопка «Выбрать мастера») не ломается.

## 6. Ограничения и допущения

**Ограничения:**

- Релевантный код (для информации; корневые причины и способ исправления определяет архитектор):
  - Карусель — `docs/modules/services.js:167-308` (`openServiceCard`): ветка `photos.length >= 2`, рендер трека `#svcTrack` = `cloneLast + realSlides + cloneFirst`, точки `#svcDots`; расчёт позиции `slideWidth()`/`translateForIdx()`/`setPos()` (`services.js:221-239`); логика листания `goTo()`/`settle()`/`transitionend`/сторож 500 мс (`services.js:246-280`); тач-жесты `touchstart/move/end`, все `{ passive:true }`, порог `|dx|<30`, без `preventDefault` (`services.js:282-306`); начальная установка `setPos(0,false)` + `updateDots()` синхронно **до** `go('s-service')` (`services.js:259-260,310`).
  - Стили — `docs/style.css:822-841`: `.svc-carousel { overflow:hidden }`, `.svc-carousel-track { display:flex; gap:8px; transition: transform .35s ease }`, `.svc-carousel-slide { flex:0 0 calc(100% - 44px); border-radius:16px }`, фото `height:280px; object-fit:cover`.
- В коде присутствуют две «магические» связки CSS↔JS, которые легко рассинхронизируются: `SLIDE_GAP = 8` (JS) ↔ `gap:8px` (CSS) и фолбэк `carousel.offsetWidth - 44` (JS) ↔ `calc(100% - 44px)` (CSS). Исправление должно устранить или надёжно синхронизировать эти связки, но это техническая деталь для архитектора.
- Vanilla JS/HTML/CSS, без рантайм-зависимостей; правки только в `docs/modules/services.js` и `docs/style.css` (при необходимости).
- Доработка строго в пределах детальной карточки `s-service` и её карусели; каталог, выбор мастера, поток записи не трогать.
- Экранирование URL/данных через `esc()` сохранить.
- При изменении файлов `APP_SHELL`/`sw.js` — поднять `CACHE_VERSION`.
- Минимальные изменения строго по объёму бага; предпочтительна правка существующей реализации, а не редизайн (выбор подхода — за архитектором, см. §8 ОВ-4).

**Допущения:**

- **Д-1.** По умолчанию исправляется **только механика и геометрия выравнивания** карусели; визуальный внешний вид из v1.12.0 (peek, точки, скругления, высота, обрезка) сохраняется без редизайна. Требует подтверждения заказчиком (см. §8 ОВ-5). Если устранение дефекта смещения потребует скорректировать значения геометрии (например, пересчитываемый peek вместо «магических» констант), это допустимо в рамках сохранения прежнего визуального впечатления.
- **Д-2.** Три симптома воспроизводимы и относятся к реализации transform-карусели из ADR-014 (решение I1); они не связаны с источником фото (G1) или стратегией кэширования (J1), а с механикой/геометрией отображения и обработкой жестов.
- **Д-3.** Целевое поведение зацикливания — **строго бесшовное в обе стороны** (как и заявлено в BA-015 ФТ-3/КП-3), без допущения «небольшого скачка».
- **Д-4.** Фото услуги уже корректно попадают в `s.photos` (массив URL) — дефект не в данных, а в их отображении.

## 7. Критерии приёмки

- **КП-1.** В карточке услуги с 2+ фото слайды выровнены: текущий слайд обрезан корректно, peek соседних слайдов равномерен по левому и правому краю (проверяется визуально на 2, 3 и 8 фото). (ФТ-1)
- **КП-2.** Свайпом фото перелистывается надёжно с первой попытки в направлении жеста; карусель не застревает между слайдами; серия из нескольких свайпов подряд листает без потери жестов. (ФТ-2)
- **КП-3.** Переход со свайпом с последнего фото на первое и с первого на последнее визуально бесшовен — без видимого рывка, перескока или мигания. (ФТ-3)
- **КП-4.** Активная точка-индикатор всегда соответствует текущему фото, в том числе сразу после прохождения стыка зацикливания. (ФТ-4)
- **КП-5.** Карусель выровнена и листается корректно сразу при открытии карточки — без предварительного скролла, «шевеления» или поворота экрана. (ФТ-5)
- **КП-6.** После поворота устройства / изменения размера окна слайды остаются выровнены, peek равномерен, позиция текущего слайда сохранена/корректно пересчитана. (ФТ-6)
- **КП-7.** При 1 фото — одиночное скруглённое изображение без листалки/точек/зацикливания; при 0 фото — категорийный плейсхолдер без листалки (поведение не изменилось). (ФТ-7)
- **КП-8.** При битом URL фото слайд/изображение деградирует на категорийный плейсхолдер без поломки вёрстки. (ФТ-8)
- **КП-9.** Внешний вид галереи (скругления, peek, точки, высота/обрезка фото) визуально совпадает с v1.12.0 — изменилась только корректность механики/выравнивания, не дизайн. (ФТ-9, Д-1)
- **КП-10.** При открытии карточки, свайпе, зацикливании и повороте в консоли нет ошибок; свайп плавный, без рывков и заметных лагов. (НФТ-2, НФТ-3)
- **КП-11.** URL фото и `alt` экранируются (`esc()`); инъекция HTML невозможна, в том числе в клонах слайдов. (НФТ-6)
- **КП-12.** PWA остаётся устанавливаемой, оффлайн-поведение не регрессирует; при изменении файлов `APP_SHELL`/`sw.js` поднята `CACHE_VERSION`; новые рантайм-зависимости не добавлены. (НФТ-1, НФТ-4)
- **КП-13.** Каталог услуг, выбор мастера, расчёт слотов, бронирование, подтверждение и остальная часть детальной карточки (название, цена-диапазон, длительность, описание, кнопка «Выбрать мастера») работают как прежде. (НФТ-5, НФТ-7)

> Блокирующих открытых вопросов нет — наблюдаемое поведение и критерии приёмки определены без них; документ готов к /adr. Ниже перечислены **неблокирующие** инженерные развилки для диагностики/проектирования (архитектор в ADR) и один пункт-дефолт для подтверждения заказчиком. Противоречий с принятыми ADR не выявлено: это исправление дефекта в рамках уже принятого ADR-014 (решение I1), а не пересмотр его решений.

- **ОВ-1 (диагностика, для архитектора). Корневая причина смещения/обрезки.** Гипотезы: (а) `slideWidth()`/`carousel.offsetWidth` измеряются **до** того, как `#s-service` стал активным/видимым (`setPos(0,false)` вызывается до `go('s-service')`) → измерение в скрытом/трансформированном экране даёт неверную ширину → неверный `translateX`; (б) рассинхрон «магических» констант `44px` (CSS) и `SLIDE_GAP=8` (JS) с фактической геометрией; (в) отсутствие пересчёта позиции при ресайзе/повороте. Какой подход к измерению/пересчёту выбрать — за архитектором. *Влияет на ФТ-1, ФТ-5, ФТ-6.*
- **ОВ-2 (диагностика, для архитектора). Причина залипания свайпа.** Гипотезы: игнор жестов во время `transitioning` (нельзя листать «доброшенными» свайпами подряд); порог `|dx|<30`; отсутствие `preventDefault` (конкуренция с вертикальным скроллом страницы при `{passive:true}`); незавершение `transitionend` (срабатывает только сторож 500 мс). Допустимо ли разрешать прерывание/постановку в очередь следующего свайпа — за архитектором. *Влияет на ФТ-2, НФТ-2.*
- **ОВ-3 (диагностика, для архитектора). Причина рывка при зацикливании.** Гипотезы: гонка `transitionend` vs сторож 500 мс; недостаточный/несвоевременный reflow (`track.getBoundingClientRect()`) при бесшовном сбросе на клон-границе. *Влияет на ФТ-3, ФТ-4.*
- **ОВ-4 (подход, для архитектора). Минимальная правка vs переработка геометрии.** Чинить точечно существующую реализацию (предпочтительно — это багфикс) либо переработать геометрию (измерять контейнер вместо клона, убрать «магические» 44px в пользу вычисляемого peek), без рантайм-зависимостей. Выбор — за архитектором в рамках НФТ-5. *Влияет на §6, НФТ-5.*
- **ОВ-5 (подтверждение заказчиком, неблокирующее — есть дефолт). Объём аффордансы.** По умолчанию (Д-1) меняется **только механика**, внешний вид из v1.12.0 сохраняется. Если заказчик хочет заодно подправить peek/точки визуально — это нужно зафиксировать; иначе работаем по дефолту «только механика». *Влияет на ФТ-9, КП-9.*
