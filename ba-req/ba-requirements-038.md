# BA-требования 038: Доступность (a11y) и клавиатурная навигация

- **Источник:** `requirements/req-accessibility-pass-037.md`
- **Дата:** 2026-05-31
- **Статус:** черновик

> **Примечание о нумерации.** Номер исходного требования — 037, но `ba-req/ba-requirements-037.md` уже занят (оформление требования `req-inapp-feedback-036.md`, см. ADR-034). Поэтому данному документу присвоен следующий свободный номер — **038**. Это не связано с требованием `requirements/req-cross-platform-layout-038.md`. Источник настоящего документа — именно `req-accessibility-pass-037.md`.

## 1. Контекст и бизнес-цель

UX-ревью от 2026-05-31 выявило, что приложение фактически недоступно для пользователей клавиатуры и скринридеров и некомфортно для людей с вестибулярной чувствительностью:

- треть интерактивных элементов реализована через `div[onclick]` (76 против 89 `<button>`) и недостижима по Tab/Enter, невидима для скринридеров;
- по всему приложению всего 4 `aria-label` и 1 `role`;
- перенос фокуса при смене экрана почти отсутствует (`.focus()` только в `auth.js`, `review.js`), из-за чего скринридер «теряется»;
- Esc закрывает оверлеи лишь в 2 местах; большинство модалок/sheet/оверлеев не закрываются клавиатурой;
- `prefers-reduced-motion` учтён минимально (1 правило), слайды экранов, карусель фото и scale-анимации не заглушены;
- нет видимого focus-ring для навигации с клавиатуры.

**Цель:** сделать основной клиентский поток полностью проходимым с клавиатуры и понятным скринридеру, корректным для пользователей с вестибулярной чувствительностью, **без изменения видимого поведения** для пользователей мыши/тача. Это снижает барьеры использования, расширяет аудиторию и приводит приложение к базовым ожиданиям WCAG.

**Заказчик/стейкхолдеры:** владелец продукта (по итогам UX-ревью); конечные пользователи, использующие клавиатуру и/или скринридеры; пользователи с чувствительностью к движению.

Релиз — **v1.33.0**. Технический долг по доступности sheet-примитивов (Esc, фокус-ловушка, ARIA-роли) был осознанно отложен из ADR-034 («Технический долг») именно в этот релиз.

## 2. Область (scope)

### В рамках задачи
- Конвертация **критичных клиентских** `div[onclick]` в доступные интерактивные элементы (`<button>` либо `role="button"` + `tabindex="0"` + обработка Enter/Space — выбор по контексту, см. ФТ-1).
- Закрытие по Esc для оверлеев/bottom-sheet/модалок клиентского потока, **на основе примитивов из v1.32.0/ADR-034** (`ui.js`: `confirmSheet`, `errorSheet`, общий sheet-хост) и оверлея выбора категории (`services.js`).
- Перенос фокуса при смене экрана (`go()` в `navigation.js`) — на заголовок нового экрана с фолбэком на контейнер.
- ARIA: `aria-label` иконочным кнопкам клиентского потока; `aria-live` для тостов/ошибок (опираясь на контейнер `aria-live` из ADR-034); необходимые роли.
- Учёт `prefers-reduced-motion: reduce` — заглушение анимаций переходов экранов, карусели фото и scale-анимаций.
- Видимый focus-ring через `:focus-visible` (виден при навигации с клавиатуры, отсутствует при кликах мышью), корректный в светлой и тёмной теме.

### Вне рамок (explicitly out of scope)
- **Весь админский слой** (`admin.js`, диалоги `github.js`/`prompt`) — согласовано с ADR-034 (вариант 5A). Конвертация админских `div[onclick]` и Esc для админских модалок не требуется, релиз не блокируют.
- Любые изменения **видимого поведения** для пользователей мыши/тача (визуальный дизайн, тексты, потоки, layout).
- Изменение бизнес-логики бронирования/отмены/переноса, защиты от двойной отправки, переходов на экраны успеха.
- Контракт YCLIENTS API, стратегии Service Worker, секреты/токен.
- Полный аудит WCAG AA с формальной сертификацией (фокус релиза — основной клиентский поток и перечисленные пункты).
- Дублирование sheet/Esc-логики — переиспользуются примитивы ADR-034, новые sheet-реализации не создаются.

## 3. Действующие лица и сценарии использования

**Пользователь клавиатуры** — взаимодействует без мыши (Tab/Shift+Tab/Enter/Space/Esc). Цель: пройти основной поток «логин → выбор услуги → выбор мастера → выбор слота → бронирование» и закрывать оверлеи/sheet, не используя мышь.

**Пользователь скринридера** — полагается на озвучивание ролей, меток и динамических сообщений. Цель: понимать, на каком экране находится (фокус после перехода), что делает каждый интерактивный элемент (метки/роли), и слышать результат действий (тосты/ошибки через `aria-live`).

**Пользователь с вестибулярной чувствительностью** — включил в ОС «уменьшить движение». Цель: пользоваться приложением без раздражающих анимаций переходов, карусели и scale-эффектов.

**Пользователь мыши/тача (обычный)** — не должен заметить изменений: focus-ring не «висит» после клика, поведение и вид прежние.

## 4. Функциональные требования

- **ФТ-1.** Критичные клиентские `div[onclick]` (бронь, выбор услуги/мастера/слота, навигация) должны быть достижимы с клавиатуры и доступны скринридеру. Реализация по контексту: где элемент по смыслу кнопка — `<button>`; где элемент визуально является строкой списка/карточкой — `role="button"` + `tabindex="0"` + обработка Enter и Space, с сохранением существующих стилей. Активация с клавиатуры (Enter/Space) выполняет то же действие, что и клик мышью.
- **ФТ-2.** Оверлеи, bottom-sheet и модалки клиентского потока закрываются клавишей Esc. Esc-закрытие реализуется в общих sheet-примитивах (`ui.js`), а не дублируется в каждом вызывающем модуле; оверлей выбора категории (`services.js`) также закрывается по Esc.
- **ФТ-3.** При смене экрана (через `go()` в `navigation.js`) фокус программно переносится на заголовок нового экрана; при отсутствии заголовка — на контейнер экрана (`tabindex="-1"` + focus). Перенос не должен ломать существующие точечные `.focus()` (`auth.js`, `review.js`).
- **ФТ-4.** Иконочные кнопки клиентского потока (без текстовой подписи) снабжаются `aria-label`, описывающим действие.
- **ФТ-5.** Динамические сообщения — тосты и сообщения об ошибках — озвучиваются скринридером через регион `aria-live` (контейнер `aria-live="polite"` определён в ADR-034; настоящее требование лишь подтверждает корректность озвучивания и применимость к sheet ошибок).
- **ФТ-6.** При системной настройке `prefers-reduced-motion: reduce` анимации заглушаются: переходы экранов (`transform: translateX`), карусель фото услуг, scale-анимации. Заглушение означает мгновенную смену состояния без движения; функциональность сохраняется.
- **ФТ-7.** Видимый focus-ring отображается при навигации с клавиатуры (через `:focus-visible`) и **не** отображается при кликах мышью/тачем. Focus-ring различим и в светлой, и в тёмной теме.
- **ФТ-8.** Изменения не затрагивают видимое поведение для пользователей мыши/тача: визуальный вид, layout, тексты и потоки остаются прежними.

## 5. Нефункциональные требования

- **НФТ-1 (без зависимостей).** Только vanilla JS + HTML + CSS; никаких новых рантайм-зависимостей. Код в `docs/prototype.html`, `docs/modules/*.js` (в т. ч. `navigation.js`, `ui.js`, `services.js`), `docs/style.css`.
- **НФТ-2 (переиспользование).** Esc-закрытие, фокус-ловушка и ARIA для sheet строятся на примитивах ADR-034 без изменения их публичных сигнатур (`toast`, `confirmSheet`, `errorSheet`, `setOfflineBanner`) и без дублирования разметки sheet.
- **НФТ-3 (совместимость тем).** Все добавляемые стили (focus-ring и пр.) корректны в светлой и тёмной теме, используют токены `var(--…)`.
- **НФТ-4 (стабильность).** Ноль ошибок в консоли; PWA остаётся installable; при изменении `APP_SHELL`/`sw.js` поднимается `CACHE_VERSION`.
- **НФТ-5 (минимальность).** Изменения строго в пределах объёма §2; чувствительные слои (бронирование, защита от двойной отправки, API, SW-стратегии) не трогаются.

## 6. Ограничения и допущения

**Ограничения:**
- Статический хостинг GitHub Pages из `docs/`; vanilla-стек без сборки фреймворков.
- Sheet/Esc/`aria-live`-инфраструктура уже существует после v1.32.0 (ADR-034) и должна быть переиспользована.
- Запрещены любые операции с веткой `backup`.
- При изменении `sw.js` обязателен подъём `CACHE_VERSION`.

**Допущения (приняты как решения для автономного прогона):**
- **Допущение-1 (объём `div[onclick]`).** В объём входят только **критичные клиентские** `div[onclick]` (бронь, выбор услуги/мастера/слота, навигация). Админские интерактивные `div` — вне объёма и не блокируют релиз (согласовано с ADR-034, вариант 5A). Решение по открытому вопросу исходника.
- **Допущение-2 (button vs role).** Выбор между `<button>` и `role="button"`+`tabindex` определяется контекстом (см. ФТ-1): `<button>` — где элемент кнопка по смыслу; `role`+`tabindex` — где элемент визуально строка/карточка и полная замена тега риском задеть вёрстку/стили. Решение по открытому вопросу исходника.
- **Допущение-3 (перенос фокуса).** Фокус переносится на **заголовок** нового экрана; фолбэк — контейнер экрана с `tabindex="-1"`. Решение по открытому вопросу исходника.
- **Допущение-4.** Контейнер `aria-live` и хост sheet, заложенные в ADR-034 и реализованные в v1.32.0, присутствуют и пригодны для расширения Esc/фокус-ловушкой без правки публичных сигнатур (прямое следствие закладки расширяемости в ADR-034 §«Нормативный контракт»).

## 7. Критерии приёмки

- **КП-1 (клавиатурная проходимость потока).** Основной клиентский поток «логин → услуга → мастер → слот → бронь» полностью проходим только с клавиатуры (Tab/Shift+Tab/Enter/Space): все критичные интерактивные элементы получают фокус и активируются. *(покрывает ФТ-1; ручная проверка клавиатурой)*
- **КП-2 (Esc).** Каждый оверлей/bottom-sheet/модалка клиентского потока (включая оверлей выбора категории, `confirmSheet`, `errorSheet`) закрывается по Esc. *(покрывает ФТ-2; ручная проверка + возможен unit-тест чистого обработчика Esc, если логика вынесена без DOM)*
- **КП-3 (перенос фокуса).** После каждого перехода `go()` фокус оказывается на заголовке нового экрана (или контейнере при отсутствии заголовка); существующие `.focus()` в `auth.js`/`review.js` не сломаны. *(покрывает ФТ-3; unit-тест чистой функции выбора цели фокуса + ручная проверка скринридером/Tab)*
- **КП-4 (aria-label).** Все иконочные кнопки клиентского потока имеют непустой `aria-label`, корректно описывающий действие; проверяется DevTools/скринридером. *(покрывает ФТ-4; ручная проверка + DevTools Accessibility tree)*
- **КП-5 (aria-live).** Тосты и сообщения об ошибках озвучиваются скринридером (регион `aria-live`); сообщение читается при появлении. *(покрывает ФТ-5; ручная проверка скринридером)*
- **КП-6 (reduced-motion).** При `prefers-reduced-motion: reduce` (эмуляция в DevTools) анимации переходов экранов, карусели фото и scale заглушены — смена состояния мгновенная, функциональность сохранена. *(покрывает ФТ-6; ручная проверка DevTools Rendering)*
- **КП-7 (focus-visible).** Focus-ring виден при навигации Tab и отсутствует при кликах мышью; различим в светлой и тёмной теме. *(покрывает ФТ-7, НФТ-3; ручная проверка)*
- **КП-8 (без регрессии для мыши/тача).** Видимое поведение, вид, layout, тексты и потоки для пользователя мыши/тача не изменились. *(покрывает ФТ-8; ручная проверка)*
- **КП-9 (тех. целостность).** Ноль ошибок в консоли; PWA installable; при изменении `sw.js` поднята `CACHE_VERSION`; новых рантайм-зависимостей нет. *(покрывает НФТ-1, НФТ-4; ручная проверка + lint)*
- **КП-10 (без дублирования).** Esc/фокус-ловушка sheet реализованы в общих примитивах `ui.js`, публичные сигнатуры из ADR-034 не изменены; дублирующих sheet-реализаций не добавлено. *(покрывает НФТ-2; ревью кода)*

## 8. Открытые вопросы

Блокирующих вопросов нет — три открытых вопроса исходника зафиксированы как решения (Допущения 1-3, §6). Некритичные уточнения, не блокирующие проектирование:

- **ОВ-1.** Полный перечень «критичных клиентских» иконочных кнопок и `div[onclick]` для конвертации — детализируется архитектором/разработчиком на этапе ADR/реализации по фактическому коду `prototype.html`; граница «критичные клиентские vs админские» уже задана (Допущение-1).
- **ОВ-2.** Точные тексты `aria-label` для иконочных кнопок (формулировки) — на усмотрение реализации; критерий — действие должно быть понятно из метки (проверка скринридером в КП-4).
- **ОВ-3.** Конкретные параметры focus-ring (толщина, цвет-токен, offset) и точный приём заглушения каждой анимации — на усмотрение архитектора/разработчика; критерии заданы поведенчески (КП-6, КП-7).

Противоречий с действующими ADR не выявлено: ADR-034 явно отнёс a11y sheet (Esc, фокус-ловушка, ARIA) к техдолгу для v1.33.0 и заложил расширяемость примитивов без смены сигнатур.
