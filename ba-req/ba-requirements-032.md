## 1. Контекст и бизнес-цель

Контактный блок студии («О студии») введён в релизе v1.23.0 (req-027 /
BA-028) и показывается на экране подтверждения записи `s-confirm` и в профиле
`s-profile`. Боевая проверка от 2026-05-30 выявила, что блок вместо помощи
вводит клиента в заблуждение:

1. **Кнопка «Маршрут» ведёт на чужую студию.** Поскольку endpoint
   `/company/{id}` отдаёт нулевые координаты (`0,0`), срабатывает фолбэк на
   `REVIEW_URLS.yandex`, а это ссылка «оставить отзыв» о другой организации
   («БьютиФлайБар», Троицк) — не маршрут и не та студия.
2. **Телефон фейковый.** Partner-токен не отдаёт телефон компании, поэтому
   показывается плейсхолдер `+7 495 123-45-67` как кликабельный `tel:` —
   звонок уходит в никуда.
3. **Город теряется.** Показывается только `address` без `city`, хотя это
   разные поля API; клиент не видит, в каком городе студия.

Корень всех трёх дефектов — гибридная модель данных из BA-028 (API + хардкод-
плейсхолдеры `STUDIO` + ссылки `REVIEW_URLS`), которая при пустых данных API
подставляет недостоверные значения. Бизнес-цель ревизии — сделать плашку
**достоверным источником контактов**: показывать только реальные данные
студии из YCLIENTS, честно обозначать отсутствие данных прочерком, дать
рабочую навигацию к студии с выбором картографического приложения.

**Заказчик / стейкхолдеры:** конечные клиенты студии (особенно новые,
впервые едущие в студию), владелец студии (заинтересован, чтобы клиент дошёл
и видел корректные контакты), владелец продукта (доверие к данным).

## 2. Область (scope)

### В рамках задачи
- Переработка контактной плашки «О студии» на обоих экранах: `s-profile` и
  `s-confirm` (единый рендер, два места показа).
- Переключение источника данных компании с `GET /company/{id}` на
  `GET /companies?id={id}` — он отдаёт реальные координаты и все нужные поля
  одним запросом.
- Переход на принцип «данные только из YCLIENTS (API + кэш `localStorage`
  `yc_company`)»: полный отказ от хардкод-плейсхолдеров как источника.
- Замена пустого значения любого поля на прочерк `—` при сохранении метки
  строки (в т. ч. для заголовка).
- Прекращение использования объекта `STUDIO` и ссылок `REVIEW_URLS` в плашке
  (REVIEW_URLS остаётся только в модуле отзывов `review.js`).
- Новый состав плашки: заголовок (`public_title`/`title`), адрес
  (`city + ', ' + address`), телефон, часы, сайт, Telegram.
- Замена двух ссылок «Маршрут» + «2ГИС» на одну кнопку «Показать на карте» с
  bottom-sheet выбора приложения: Apple Maps / Google Maps / Yandex Navigator
  / Yandex Maps + «Отмена».
- Построение ссылок карт по реальным координатам компании; фолбэк на поиск по
  тексту адреса при отсутствии координат.
- Новое аккуратное визуальное оформление плашки (иконки строк, карточка,
  bottom-sheet). Конкретика визуала — за архитектором/дизайном.
- Сохранение расширенного набора полей компании в кэш `localStorage`
  (`yc_company`) для оффлайна.
- Сохранение локального исключения из глобального `user-select:none` для
  выделяемости/копируемости контактов.

### Вне рамок (explicitly out of scope)
- Запись данных компании в YCLIENTS (PUT `/company/{id}`) — недоступна по
  правам partner-токена.
- Вывод **метро** — поля метро в API YCLIENTS нет (проверено на `/company`,
  `/companies`, `include[]=metro`); не выводим.
- Прочие соцсети/мессенджеры, кроме Сайта и Telegram (в т. ч. `social.viber`,
  хотя он заполнен) — по решению заказчика только Сайт + Telegram (см. ОВ-4).
- Кликабельный `tel:`-вызов по телефону: при отсутствии телефона в API
  ссылка не строится; формат отображения телефона при его появлении в API —
  см. ОВ-2.
- Изменение существующих кнопок и потока на `s-confirm` (бронь, «На главную»,
  кросс-селл) и прочих секций профиля.
- Многостудийность / выбор филиала, редактирование контактов из приложения.
- Ввод новых секретов/токенов; рефакторинг захардкоженного токена YCLIENTS.
- Техническая реализация (структура DOM, выбор/построение компонента
  bottom-sheet, момент загрузки запроса) — задача архитектора.

## 3. Действующие лица и сценарии использования

**Новый клиент после записи (`s-confirm`).** Завершил бронь, видит под
карточкой записи плашку «О студии»: название, адрес с городом, часы, телефон
(если есть), сайт, Telegram. Нажимает «Показать на карте» → выбирает удобное
приложение (Apple Maps / Google Maps / Yandex Navigator / Yandex Maps) →
строится маршрут/открывается точка студии по реальным координатам.

**Постоянный клиент в профиле (`s-profile`).** В секции «О студии» в любой
момент находит актуальные контакты студии, может скопировать адрес/телефон,
открыть сайт или Telegram, построить маршрут.

**Клиент при недоступном интернете / пустом ответе API.** Видит данные из
кэша `yc_company`, а при отсутствии данных — метки с прочерком `—`; экран не
ломается.

## 4. Функциональные требования

- **ФТ-1. Источник данных — только YCLIENTS.** Все значения плашки берутся
  исключительно из ответа `GET /companies?id={id}` и/или кэша `localStorage`
  `yc_company`. Хардкод-плейсхолдеры в качестве значений не используются.
- **ФТ-2. Смена endpoint.** Загрузка данных компании выполняется через
  `GET /companies?id={id}` (вместо `GET /company/{id}`). Этот ответ
  предоставляет `title`, `public_title`, `address`, `city`, `schedule`,
  `site`, `social`, а также реальные координаты `coordinate_lat`,
  `coordinate_lon`.
- **ФТ-3. Пустое значение → прочерк.** Если поле в API пустое, строка и её
  метка остаются, а значение отображается как прочерк `—` (например
  `Телефон: —`, `Часы: —`). Это правило распространяется и на заголовок:
  пустой заголовок → `—`.
- **ФТ-4. Заголовок.** Заголовок плашки = `public_title` (имя для онлайн-
  записи) с фолбэком на `title`; при обоих пустых → `—`.
- **ФТ-5. Адрес.** Адрес = склейка `city + ', ' + address`
  (например «Люберцы, улица Генерала Кузнецова, 22»). При пустом адресе/городе
  применяется правило прочерка (ФТ-3) к итоговому значению.
- **ФТ-6. Телефон.** Отображается значение телефона из API; при пустом
  значении — `—` без `tel:`-ссылки. (Хардкод-плейсхолдер телефона не
  используется.)
- **ФТ-7. Часы.** Отображается `schedule` «как есть»; при пустом значении —
  `—`.
- **ФТ-8. Сайт.** Если `site` непустой — отображается ссылкой; текст ссылки —
  без префикса `https://` (например `albertivanov.com`). При пустом `site`
  строка сайта не показывается (соцсети/сайт при отсутствии — скрываются,
  см. ФТ-9).
- **ФТ-9. Telegram.** Если `social.telegram` непустой — отображается ссылкой
  `https://t.me/<handle без @>`, текст `@handle`. Пустые соцсети не
  показываются (строка отсутствует, а не выводится с прочерком).
- **ФТ-10. Метро не выводится.** Поля/строки метро в плашке нет.
- **ФТ-11. Объект `STUDIO` и `REVIEW_URLS` не используются в плашке.** Плашка
  не обращается к объекту `STUDIO` как источнику данных и не использует
  `REVIEW_URLS` для построения ссылок карт/контактов. `REVIEW_URLS` остаётся
  доступным только модулю отзывов `review.js`.
- **ФТ-12. Единая кнопка «Показать на карте».** Вместо ссылок «Маршрут» и
  «2ГИС» в плашке одна кнопка «Показать на карте».
- **ФТ-13. Bottom-sheet выбора приложения.** По нажатию кнопки открывается
  лист выбора с четырьмя вариантами — Apple Maps, Google Maps, Yandex
  Navigator, Yandex Maps — и пунктом «Отмена». Все четыре варианта
  показываются всегда (на desktop, где применимо, открываются веб-версии).
- **ФТ-14. Ссылки карт по координатам.** При наличии координат ссылки
  строятся по `coordinate_lat`/`coordinate_lon`:
  - Apple Maps: `https://maps.apple.com/?ll=<lat>,<lon>&q=<title>`
  - Google Maps: `https://www.google.com/maps/search/?api=1&query=<lat>,<lon>`
  - Yandex Maps: `https://yandex.ru/maps/?pt=<lon>,<lat>&z=17&l=map`
  - Yandex Navigator: `yandexnavi://build_route_on_map?lat_to=<lat>&lon_to=<lon>`
    (построение маршрута к точке).
- **ФТ-15. Фолбэк и скрытие кнопки.** Если координат нет (API вернул `0`), но
  адрес есть — ссылки строятся как поиск по тексту адреса (Yandex `?text=`,
  Navigator `map_search?text=` и т. п.). Если нет ни координат, ни адреса —
  кнопка «Показать на карте» скрывается.
- **ФТ-16. Единый рендер для двух экранов.** Плашка отображается и на
  `s-confirm`, и на `s-profile`; вариативность оформления между экранами
  допускается через параметр варианта рендера, без дублирования логики данных.
- **ФТ-17. Устойчивость к недоступности API.** При ошибке/пустом ответе
  `/companies?id=` показываются данные из кэша `yc_company`; при отсутствии
  кэша — метки с прочерком `—`; экраны `s-confirm` и `s-profile` не ломаются.
- **ФТ-18. Кэширование расширенного набора.** В кэш `yc_company` сохраняется
  расширенный набор полей: координаты, `site`, `social`, `city` (в дополнение
  к ранее кэшируемым), для корректной работы плашки и карты в оффлайне.
- **ФТ-19. Выделяемость контактов.** Текст плашки (адрес, телефон и др.)
  доступен для выделения и копирования, несмотря на глобальный
  `user-select:none`.
- **ФТ-20. Безопасность вывода.** Все серверные строки экранируются перед
  вставкой в разметку; внешние http(s)-ссылки имеют `rel="noopener"` (и, как
  правило, `target="_blank"`); телефон и handle нормализуются перед
  подстановкой в href; app-схемы (`yandexnavi://`) используются как обычный
  `href`.

## 5. Нефункциональные требования

- **НФТ-1 (Совместимость / стек).** Vanilla HTML/CSS/JS, без новых рантайм-
  зависимостей; код в `docs/`.
- **НФТ-2 (Безопасность).** Внешние ссылки — `rel="noopener"`; экранирование
  отображаемых строк (`esc`); новые секреты не вводятся, действующий токен
  YCLIENTS не «усугубляется».
- **НФТ-3 (Устойчивость / оффлайн).** Отказ/задержка/пустой ответ
  `/companies?id=` не блокирует и не ломает `s-confirm` и `s-profile`; плашка
  всегда отображается (минимум — метки с прочерком).
- **НФТ-4 (Достоверность данных).** В плашке не отображаются выдуманные
  значения под видом реальных контактов; отсутствие данных обозначается явным
  прочерком `—`.
- **НФТ-5 (Минимальность изменений).** Изменения ограничены переработкой
  плашки и источника её данных; существующая функциональность и потоки
  `s-confirm`/`s-profile` не переписываются.
- **НФТ-6 (Доступность / UX).** Кнопка карты, ссылки и пункты bottom-sheet —
  доступные интерактивные элементы (читаемые, с достаточной тач-зоной);
  плашка оформлена аккуратно (иконки строк, карточка, стилизованный лист
  выбора).
- **НФТ-7 (Кэширование SW).** При изменении файлов из `APP_SHELL` / `sw.js`
  повышается `CACHE_VERSION`.
- **НФТ-8 (Отсутствие ошибок).** В консоли браузера нет ошибок; PWA остаётся
  устанавливаемой (Lighthouse).

## 6. Ограничения и допущения

**Ограничения (подтверждённые факты):**
- Endpoint `GET /company/{id}` для компании `1940850` отдаёт нулевые
  координаты (`0,0`) и не отдаёт телефон (`phone:""`, `phones:[]`). Поэтому
  источник переключается на `GET /companies?id={id}`, отдающий реальные
  координаты (`coordinate_lat=55.686006`, `coordinate_lon=37.85368`) и
  остальные поля.
- В `/companies?id=` присутствуют `title`, `public_title`, `address`, `city`
  («Люберцы»), `schedule`, `site`, `social` (включая `telegram` и `viber`).
  Поля метро нет ни в одном из проверенных endpoint.
- PUT `/company/{id}` partner-токеном недоступен (`success:false`) — запись
  настроек невозможна, поэтому редактирование контактов вне scope.
- Глобальный `user-select:none` (v1.20.0) требует локального исключения для
  плашки (исключение уже существует в CSS).
- Затрагиваемый код (справочно, для архитектора): `docs/modules/constants.js`
  (`STUDIO`, `REVIEW_URLS`), `docs/modules/studio.js` (`setCompanyData`,
  `getCompanyContacts`, `renderStudioContacts`), `docs/app.js:167-174`
  (`initApp()`, текущий запрос `/company/${YC.company}`),
  `docs/modules/slots.js:97-98` (`updateConfirmScreen`),
  `docs/modules/profile.js:63-64` (`renderProfileScreen`),
  `docs/style.css:1107-1169` (`.studio-contacts*`), `docs/sw.js`
  (`studio.js` в `APP_SHELL`, `CACHE_VERSION`).

**Допущения:**
- `schedule` — строка свободного формата, показывается «как есть»; при пустом
  значении — прочерк.
- Реальные координаты студии `55.686006, 37.85368` стабильны и пригодны для
  построения ссылок/маршрутов во всех четырёх картографических приложениях.
- Bottom-sheet с четырьмя вариантами + «Отмена» приемлем как паттерн выбора
  приложения карты; на desktop веб-версии открываются корректно (где
  применимо), app-схема Navigator на desktop может не сработать — это
  ожидаемо и допустимо.
- Кэш `yc_company` можно безопасно расширить дополнительными полями без
  поломки совместимости со старым форматом кэша.

## 7. Критерии приёмки

- **КП-1.** (ФТ-1, ФТ-2) Плашка показывает данные только из `/companies?id=`
  и/или кэша `yc_company`; запрос компании выполняется к `/companies?id=`, а
  не к `/company/{id}`.
- **КП-2.** (ФТ-4, ФТ-5, ФТ-6, ФТ-7) В плашке отображаются: заголовок
  (`public_title`/`title`), адрес с городом (`city + ', ' + address`),
  телефон (или `—`), часы (или `—`).
- **КП-3.** (ФТ-3) Пустое значение любого поля показывается как `—` с
  сохранением метки, в т. ч. для заголовка; деления/поломки нет.
- **КП-4.** (ФТ-8, ФТ-9) Сайт отображается ссылкой без префикса `https://`;
  Telegram — ссылкой `https://t.me/<handle>` с текстом `@handle`; пустые
  соцсети/сайт не показываются.
- **КП-5.** (ФТ-10) Строки/поля метро в плашке отсутствуют.
- **КП-6.** (ФТ-11) Объект `STUDIO` не используется как источник данных
  плашки; `REVIEW_URLS` в плашке не используется (остаётся только в

- **КП-7.** (ФТ-12, ФТ-13) Вместо «Маршрут» + «2ГИС» — одна кнопка «Показать

- **КП-8.** (ФТ-14) При наличии координат ссылки строятся по реальным

- **КП-11.** (ФТ-17, ФТ-18, НФТ-3) При недоступном/пустом ответе API плашка
  показывает данные из кэша `yc_company`, иначе прочерки; кэш содержит
  расширенный набор (координаты, `site`, `social`, `city`); экраны не
  ломаются.

  глобальный `user-select:none`.
- **КП-13.** (ФТ-20, НФТ-2) Серверные строки экранируются; внешние ссылки —
  `rel="noopener"`; телефон/handle нормализуются перед подстановкой в href;

- **КП-14.** (НФТ-6) Плашка оформлена аккуратно: иконки строк, карточка,

- **КП-15.** (НФТ-1, НФТ-5, НФТ-7, НФТ-8) Диф ограничен переработкой плашки и
  источника данных; в консоли нет ошибок; PWA остаётся устанавливаемой; при
  изменении `APP_SHELL`/`sw.js` повышена `CACHE_VERSION`.

## 8. Открытые вопросы

Блокирующих открытых вопросов нет: ключевые решения (источник данных,
endpoint, состав плашки, выбор карты, отказ от `STUDIO`/`REVIEW_URLS`, отказ
от метро) согласованы заказчиком и зафиксированы в исходном требовании.
Требование можно передавать на этап `/adr`. Ниже — некритичные вопросы,
относящиеся к проектированию/реализации.

- **ОВ-1. Реализация bottom-sheet выбора карты.** Новый компонент или
  переиспользование существующего паттерна модалок/листов проекта. Внешние
  ссылки — `target="_blank" rel="noopener"`, app-схемы (`yandexnavi://`) —
  обычный `href`. Решает архитектор.
- **ОВ-2. Формат отображения телефона при его появлении в API.** Сейчас
  телефон в API пуст. Когда он появится — уточнить формат вывода и нужна ли
  кликабельная `tel:`-ссылка (исходное требование задаёт телефон «без
  `tel:`-ссылки»; подтвердить, остаётся ли это правило при непустом
  телефоне). База: показывать значение из API как текст, при пустом — `—`.
- **ОВ-3. Поведение Yandex Navigator без координат.** База: при наличии
  координат — построение маршрута (`build_route_on_map`); при отсутствии
  координат, но наличии адреса — поиск по тексту (`map_search?text=`).
  Подтверждение схемы — за архитектором.
- **ОВ-4. Прочие соцсети.** В API заполнен `social.viber=@Alb0203`, но
  заказчик просил только Сайт + Telegram. База (принято): показывать только
  Сайт и Telegram, остальные соцсети — вне scope.
- **ОВ-5. Расширение и совместимость кэша `yc_company`.** Уточнить структуру
  расширенного кэша и поведение при чтении старого формата (после релиза
  v1.23.0 в кэше может лежать прежний набор полей). База: при отсутствии поля
  в кэше — прочерк/скрытие соответствующей строки.

> Связь с ранее принятыми решениями: данное требование **сознательно отменяет**
> часть решений BA-028 (req-027) — гибридный источник «API → кэш → `STUDIO`»,
> использование `REVIEW_URLS` в плашке, плейсхолдер телефона
> `+7 495 123-45-67`, источник `/company/{id}` и вывод по адресу Троицка.
> Это не противоречие с действующим ADR, а целенаправленная ревизия по итогам
> боевой проверки. Архитектору при создании ADR следует явно отметить
> вытеснение прежнего решения по контактному блоку.
