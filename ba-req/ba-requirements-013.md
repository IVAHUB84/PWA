# BA-требования 013: Диапазон цен услуги в каталоге (от min до max)

- **Источник:** `requirements/req-services-price-range-013.md`
- **Дата:** 2026-05-29
- **Статус:** черновик

## 1. Контекст и бизнес-цель

Studio PWA (студия красоты, онлайн-запись поверх YCLIENTS) — устанавливаемое статическое приложение, отдаётся из `docs/` на GitHub Pages. На экране каталога услуг (`s-services`, `_renderList`, `docs/modules/services.js:86-99`) для каждой услуги показывается строка вида `{priceStr} · {dur} мин`. Строка цены `priceStr` формируется один раз при загрузке услуг из YCLIENTS `/services/{company}` через `_fmtPrice(s.price_min, s.price_max)` (`docs/app.js:172`).

Одну и ту же услугу могут выполнять разные мастера по разной цене (например, «Стрижка»: Иван — 2 000 ₽, Мария — 5 000 ₽). Сейчас, когда минимальная и максимальная цена услуги различаются, `_fmtPrice` **намеренно отбрасывает верхнюю границу** и выводит «от 2 000 ₽» (`docs/modules/utils.js:17-21`). Для клиента это скрывает реальный потолок стоимости: формулировка «от» создаёт впечатление, что услуга «начинается» от минимума и может стоить сколько угодно, тогда как фактический разброс ограничен (2 000–5 000 ₽).

**Бизнес-цель:** в каталоге услуг показывать клиенту понятный **диапазон стоимости** «от минимальной до максимальной цены», когда цены у разных мастеров отличаются. Это повышает прозрачность ценообразования, снижает неопределённость на этапе выбора услуги и уменьшает риск расхождения ожиданий клиента с итоговой ценой записи.

**Стейкхолдеры:**
- Заказчик / владелец студии — заинтересован в честной и понятной подаче цен, снижении вопросов «а почему дороже, чем написано».
- Клиент студии — выбирает услугу и хочет заранее видеть реальные границы стоимости.
- Бизнес-аналитик / архитектор / разработчик — оформление, проектирование, реализация.

## 2. Область (scope)

### В рамках задачи
- В карточке услуги **в каталоге** при различающихся ценах услуги отображается диапазон стоимости («от минимальной до максимальной»), а не только нижняя граница «от {min}».
- **Источник диапазона — цены мастеров (решение заказчика, см. §8 №1):** минимум/максимум приложение вычисляет само из цен мастеров, выполняющих услугу (поля `price_min`/`price_max` из `/services/` для этого не подходят). Сбор цен по мастерам и агрегация диапазона входят в объём; конкретную схему загрузки/кэширования определяет архитектор отдельным ADR.
- Когда цена единственная (нижняя и верхняя границы совпадают либо верхняя отсутствует) — отображается единичная сумма, как сейчас.
- Когда цена отсутствует — сохраняется текущее поведение «По запросу».
- Сохраняется текущее форматирование сумм: локаль `ru-RU` (разряды), символ «₽», округление (`Math.round`) — как в `_fmtPrice`.
- Поведение приводится в соответствие для всех путей формирования цены каталога: и для общего каталога (`app.js:172` → `_fmtPrice`), и для режима предвыбранного мастера, где цена услуги сейчас собирается отдельно как «от {price_min} ₽» (`docs/modules/services.js:118`) — с учётом решения по открытому вопросу №4.
- Места, где цена используется уже **после выбора конкретного мастера** (экран подтверждения «Стоимость» — `slots.js:85`; сохранение цены в визит — `booking.js:30`), приводятся в соответствие с продуктовым решением по открытому вопросу №3 (там логична конкретная цена выбранного мастера, а не диапазон).

### Вне рамок (explicitly out of scope)
- **Изменение бизнес-смысла цены и состава отображаемых данных** сверх перехода от «от {min}» к диапазону «min–max» (и конкретной цены мастера после его выбора).
- **Изменение вёрстки/дизайна карточки услуги, экрана подтверждения и результатов поиска** сверх замены текста строки цены.
- **Логика выбора мастера, расчёта слотов, самого бронирования** — не затрагивается; меняется только представление/значение строки цены.
- **Мок-данные `SERVICES_DATA`** (`docs/modules/state.js:2-14`) хранят готовые строки цены как fallback до загрузки из API — их переписывание на диапазоны не входит в задачу (они отображаются только до прихода данных из API); при необходимости — отдельным решением.
- **Перенос захардкоженного YCLIENTS-токена** из `prototype.html` (известный техдолг — «не усугублять», не трогать).

## 3. Действующие лица и сценарии использования

**Клиент студии** — пользователь PWA, открывший каталог услуг (или нашедший услугу через поиск на главной), чтобы выбрать услугу и записаться. Цель — заранее понимать стоимость.

Основной сценарий (каталог):
1. Клиент открывает каталог услуг.
2. Для услуги, которую разные мастера выполняют по разной цене, в строке метаданных карточки отображается **диапазон** стоимости (от минимума до максимума) с сохранением форматирования сумм.
3. Для услуги с единственной ценой отображается одна сумма; для услуги без цены — «По запросу».

Сценарий «предвыбранный мастер»:
1. Клиент входит в каталог в режиме предвыбранного мастера (услуги отфильтрованы под конкретного мастера, `services.js:106-129`).
2. Для услуг показывается цена, релевантная контексту (по решению открытого вопроса №4 — предположительно конкретная цена в рамках выбранного мастера, а не общий диапазон).

Сценарий «после выбора мастера» (подтверждение и запись):
1. Клиент выбрал конкретного мастера и слот, попал на экран подтверждения.
2. В строке «Стоимость» (`slots.js:85`) показывается цена, релевантная факту выбора мастера (по решению открытого вопроса №3 — предположительно конкретная цена, а не диапазон).
3. При сохранении записи в визит записывается цена (`booking.js:30`, поле `price`) согласованно с отображённой на подтверждении.

## 4. Функциональные требования

- **ФТ-1.** В каталоге услуг для услуги, у которой минимальная и максимальная цена различаются, строка цены отображается как диапазон «от минимальной до максимальной стоимости» (вместо текущего «от {min}»).
- **ФТ-2.** Для услуги с единственной ценой (нижняя и верхняя границы совпадают либо верхняя граница отсутствует) строка цены отображается как единичная сумма.
- **ФТ-3.** Для услуги без цены строка отображается как «По запросу» (текущее поведение сохраняется).
- **ФТ-4.** Форматирование сумм во всех вариантах сохраняется: локаль `ru-RU` (разделение разрядов), символ «₽», округление до целого — единообразно с текущей `_fmtPrice`.
- **ФТ-5.** Диапазон определяется приложением как минимум/максимум по ценам мастеров, выполняющих услугу (а не из `price_min`/`price_max` услуги `/services/`). Способ получения цен мастеров, агрегации и кэширования определяет архитектор отдельным ADR (см. §8 №1).
- **ФТ-6.** Формат отображения диапазона (разделитель и обрамление: например «2 000 – 5 000 ₽», «от 2 000 до 5 000 ₽» или иной) единый во всех местах, где диапазон показывается, и фиксируется решением по открытому вопросу №2.
- **ФТ-7.** В местах, где услуга показывается уже привязанной к конкретному мастеру (экран подтверждения «Стоимость» — `slots.js:85`; режим предвыбранного мастера — `services.js:106-129`), отображается цена, релевантная выбранному мастеру, а не общий диапазон по услуге — согласно решениям по открытым вопросам №3 и №4.
- **ФТ-8.** При сохранении записи (`booking.js:30`, поле `price` визита) сохраняется значение цены, согласованное с тем, что показано клиенту на экране подтверждения (т. е. не «голый» диапазон, если на подтверждении выбран конкретный мастер) — согласно решению по открытому вопросу №3.
- **ФТ-9.** Изменение не должно ломать прочие места, использующие строку цены (`priceStr`): каталог (`services.js:95`), поиск на главной (`search.js:27`), подтверждение (`slots.js:85`), сохранение визита (`booking.js:30`) — каждое из них продолжает корректно отображать/сохранять цену.
- **ФТ-10.** Все пользовательские тексты, затронутые изменением, — на русском языке, в едином тоне приложения.

## 5. Нефункциональные требования

- **НФТ-1 (Совместимость стека).** Решение остаётся в рамках стека: vanilla JS/HTML/CSS, код в `docs/`; статический хостинг на GitHub Pages; без новых рантайм-npm-зависимостей.
- **НФТ-2 (Производительность сбора цен).** Сбор цен мастеров для агрегации диапазона требует дополнительных запросов к YCLIENTS (поскольку решено вычислять диапазон из per-master цен, §8 №1). Схема должна минимизировать число запросов и их влияние на время загрузки каталога (например, один проход по мастерам с агрегацией и кэшированием, а не запрос на каждую услугу); конкретику определяет архитектор в ADR. Прокрутка/отрисовка каталога не должны деградировать.
- **НФТ-3 (Минимальность изменений).** Изменения строго в объёме задачи. `_fmtPrice` — общая утилита (`docs/modules/utils.js`); правка не должна изменить поведение в сценариях, не входящих в задачу, без явного решения.
- **НФТ-4 (PWA/офлайн не регрессируют).** Изменения не ломают устанавливаемость PWA и офлайн-поведение Service Worker; при изменении состава кэшируемых файлов `APP_SHELL` (например, `utils.js`, `services.js`, `app.js`, `slots.js`, `booking.js`, `search.js`) поднимается `CACHE_VERSION` в `docs/sw.js` (правило проекта).
- **НФТ-5 (Локализация).** Все пользовательские тексты — на русском языке; формат сумм — `ru-RU`.

## 6. Ограничения и допущения

**Ограничения:**
- Релевантный код: формирование цены услуг — `docs/app.js:160-174`; утилита форматирования — `docs/modules/utils.js:17-21` (`_fmtPrice`); вывод цены в каталоге — `docs/modules/services.js:95`, режим предвыбранного мастера — `docs/modules/services.js:106-129` (отдельное формирование цены — `services.js:118`); переиспользование `priceStr` — `docs/modules/search.js:27`, `docs/modules/slots.js:85`, `docs/modules/booking.js:30`; fallback-мок — `docs/modules/state.js:2-14`.
- `_fmtPrice(min, max)` — общая функция, вызываемая (как минимум) из `app.js:172`; перед изменением её поведения необходимо проверить все вызовы, чтобы не затронуть сценарии вне задачи.
- Запрещены рантайм-npm-зависимости; запрещён коммит секретов (включая YCLIENTS-токен); запрещены любые операции с веткой `backup`.
- При изменении файлов из `APP_SHELL` — поднятие `CACHE_VERSION` в `docs/sw.js`.

**Допущения:**
- **Ключевое (подтверждено заказчиком):** `price_min`/`price_max` услуги из `/services/` **не** отражают разброс по мастерам; диапазон вычисляется приложением из цен мастеров. Способ сбора/агрегации/кэширования цен мастеров — за архитектором (новый ADR, см. §8 №1).
- «Диапазон» считается уместным только когда минимум и максимум по мастерам различаются; при равенстве (или единственном мастере) — единичная сумма; при отсутствии цены — «По запросу».
- После выбора конкретного мастера (подтверждение, сохранение записи) и в режиме предвыбранного мастера показывается конкретная цена этого мастера, а не диапазон (§8 №3, №4).

## 7. Критерии приёмки

- **КП-1.** В каталоге услуг для услуги с различающимися ценами отображается диапазон «от минимальной до максимальной стоимости», а не «от {min}». (ФТ-1, ФТ-6)
- **КП-2.** В каталоге для услуги с единственной ценой отображается единичная сумма; для услуги без цены — «По запросу». (ФТ-2, ФТ-3)
- **КП-3.** Во всех вариантах строки цены сохранено форматирование `ru-RU` с символом «₽» и округлением до целого. (ФТ-4)
- **КП-4.** Диапазон вычисляется приложением как min/max по ценам мастеров услуги (не из `price_min`/`price_max` услуги); схема сбора цен соответствует ADR и не приводит к заметной деградации загрузки/прокрутки каталога. (ФТ-5, НФТ-2)
- **КП-5.** Формат диапазона (разделитель/обрамление) единообразен во всех местах его показа. (ФТ-6)
- **КП-6.** На экране подтверждения «Стоимость» и в режиме предвыбранного мастера отображается цена, релевантная выбранному мастеру (не общий диапазон) — в соответствии с решениями по открытым вопросам №3, №4. (ФТ-7)
- **КП-7.** Цена, сохранённая в визит при бронировании (`price`), согласована с ценой, показанной клиенту на экране подтверждения. (ФТ-8)
- **КП-8.** Поиск на главной, экран подтверждения и сохранение визита продолжают корректно работать со строкой цены — без ошибок в консоли и без визуальных артефактов. (ФТ-9)
- **КП-9.** PWA остаётся устанавливаемой, офлайн-поведение не регрессирует; при изменении файлов `APP_SHELL` поднята `CACHE_VERSION`; новые рантайм-зависимости не добавлены. (НФТ-1, НФТ-4)
- **КП-10.** Затронутые пользовательские тексты — на русском языке. (ФТ-10, НФТ-5)

## 8. Открытые вопросы

> **Блокирующий вопрос №1 РЕШЁН заказчиком (2026-05-29).** Решение принято в «дорогом» направлении — диапазон вычисляет приложение из цен мастеров; `price_min`/`price_max` из `/services/` для этого не подходят. Это требует архитектурного решения по схеме загрузки цен (новый ADR). См. ниже.

1. **(РЕШЕНО) Источник диапазона — цены мастеров, не `/services/`.** Заказчик подтвердил: у услуги **нет** готового min/max — есть услуга и несколько мастеров с разными ценами на неё; **минимум и максимум приложение должно определять само** по ценам мастеров. Следствия:
   - Поля `price_min`/`price_max` из `/services/{company}` **не используются** как источник диапазона (тривиальная правка `_fmtPrice` отменяется как достаточное решение).
   - Диапазон строится агрегированием цен по мастерам, выполняющим услугу. Цены мастера на услуги доступны через `/book_services/{company}?staff_id=` (как в `masters.js:146`, `services.js:111`); список мастеров услуги — через `/book_staff/{company}?service_ids[]=` (как в `api.js:132`). **Конкретную схему и стоимость запросов определяет архитектор в новом ADR** (например: один проход по всем мастерам с агрегацией min/max по услуге при загрузке, с кэшированием — против запроса на каждую услугу).
   - **Это вход для архитектора и предмет нового ADR** (меняется схема загрузки данных каталога). ФТ-5, НФТ-2 пересматриваются (см. ниже).
   *Влияет на ФТ-1, ФТ-5, НФТ-2, всю оценку трудоёмкости.*

Оставшиеся вопросы — **непродуктовые/UX-уточнения, не блокируют `/adr`**; зафиксированы значения по умолчанию, архитектор/разработчик действуют по ним, если заказчик не возразит:

2. **Формат диапазона — по умолчанию «от {min} до {max} ₽»** (словесная форма, как сформулировал заказчик в исходном требовании «от 2000 до 5000»), с форматированием сумм `ru-RU` и символом «₽». Допустима замена на «{min} – {max} ₽» на усмотрение реализации. *Влияет на ФТ-6, КП-1, КП-5.*

3. **Подтверждение и сохранение записи — по умолчанию конкретная цена выбранного мастера** (не диапазон). На экране подтверждения (`slots.js:85`) и при сохранении визита (`booking.js:30`) мастер уже выбран — показывается и сохраняется его конкретная цена. Способ получения цены конкретного мастера — в рамках той же схемы загрузки цен (вопрос №1, ADR). *Влияет на ФТ-7, ФТ-8, КП-6, КП-7.*

4. **Режим предвыбранного мастера в каталоге — по умолчанию конкретная цена этого мастера** (не диапазон), `services.js:106-129`. *Влияет на ФТ-7, КП-6.*

5. **Поиск на главной — по умолчанию тот же диапазон, что и в каталоге** (`search.js:27`), для единообразия. *Влияет на ФТ-9, КП-8.*

6. **Граница «диапазон vs единичная цена» — диапазон только при различии min и max**; при равенстве (или единственном мастере) — единичная сумма; при отсутствии цены — «По запросу». *Влияет на ФТ-1, ФТ-2, КП-2.*

**Влияние на ADR: задача теперь ТРЕБУЕТ нового ADR.** Решение №1 меняет схему загрузки данных каталога услуг (сбор цен по мастерам и агрегация диапазона), что выходит за рамки текущей загрузки `/services/` (`app.js:160-174`). Противоречий с действующими `adrs/adr-001`…`adr-010` не выявлено, но архитектор должен оформить новое решение по загрузке/агрегации цен мастеров (стоимость запросов, кэширование, поведение при недоступности данных мастера).
