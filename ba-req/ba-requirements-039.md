# BA-требования 039: Кросс-платформенный лейаут (десктоп / планшет / мышь)

- **Источник:** `requirements/req-cross-platform-layout-038.md`
- **Дата:** 2026-05-31
- **Статус:** черновик

> Примечание о нумерации: исходное требование имеет номер 038, но каталог `ba-req/` уже содержит `ba-requirements-038.md` (соответствует другому требованию — `req-accessibility-pass-037.md`). Поэтому данному документу присвоен следующий свободный номер **039**. Связь с источником `requirements/req-cross-platform-layout-038.md` зафиксирована явно.

## 1. Контекст и бизнес-цель

Studio PWA — онлайн-запись клиентов студии красоты поверх YCLIENTS. Приложение исторически свёрстано как мобильный телефон: контент помещён в фиксированный «телефон-макет» (`.phone` 393×852px со скруглением и тенью), с фейковым iOS-хромом (статус-бар «9:41», dynamic island). На десктопе и планшете это выглядит как незавершённый прототип: маленький «телефон» по центру экрана, нет hover-обратной связи для мыши, заблокирован зум, текст не выделяется/не копируется.

UX-ревью от 2026-05-31 зафиксировало, что часть аудитории открывает запись с десктопа/планшета (рабочий компьютер, большой экран), и текущий вид снижает доверие к студии и затрудняет копирование контактов/цен.

**Бизнес-цель:** сделать приложение визуально завершённым и удобным на больших экранах и при работе мышью, не ухудшив основной (мобильный) сценарий и не нарушив ранее принятых решений по a11y (v1.33.0) и in-app фидбэку (v1.32.0).

**Стейкхолдеры:** владелец студии (бренд, доверие), клиенты на десктопе/планшете (удобство), команда разработки (минимизация рефакторинга).

**Целевой релиз:** v1.34.0.

## 2. Область (scope)

### В рамках задачи

Зафиксированный по умолчанию объём — вариант «оформленное устройство + полировка взаимодействия» (минимальные изменения), согласно предложению автора требования. Финальное решение по объёму (см. развилку в разделе 8) остаётся за архитектором.

- **Оформление на больших экранах (≥768px).** Приложение перестаёт выглядеть «телефоном-прототипом»: осознанно оформленное «устройство» на брендовом фоне (а не одинокий мелкий телефон по центру). Многоколоночные адаптивные сетки в рамках текущего объёма не требуются.
- **Скрытие фейкового iOS-хрома вне iOS.** Статус-бар «9:41» и dynamic island не отображаются на десктопе и не-iOS-платформах.
- **Hover-состояния для мыши.** Интерактивные элементы (карточки услуг/мастеров, строки списков, кнопки) дают визуальную обратную связь при наведении мышью.
- **Снятие блокировки зума.** Масштабирование (pinch / Ctrl+=) доступно на всех платформах.
- **Ревизия запрета выделения текста.** Контентный текст (цены, описания, адрес, телефон, имена) выделяется и копируется; запрет выделения остаётся только на «кнопочных»/служебных элементах.
- **Десктоп-установка PWA.** Формулировки и affordance установки адекватны для Chrome/Edge на десктопе (не мобильно-ориентированный текст «Добавить на экран»/iOS-overlay).

### Вне рамок (explicitly out of scope)

- **Настоящий адаптивный многоколоночный лейаут** (сетки услуг/мастеров в 2-3 колонки на ≥768px, перекомпоновка экранов, крупный рефактор CSS/сеток) — выносится в отдельное требование, если будет признано необходимым (см. развилку, раздел 8).
- Изменения мобильного вида (≤500px) — он не должен меняться (только не ухудшаться).
- Изменение бизнес-логики записи, интеграции с YCLIENTS, API-контрактов.
- Изменение примитивов `ui.js` (toast/sheet, v1.32.0) и поведения a11y (v1.33.0) — данный релиз не должен затрагивать их публичные сигнатуры и поведение.

## 3. Действующие лица и сценарии использования

- **Клиент на десктопе (Chrome/Edge, мышь).** Открывает приложение на большом экране, ожидает аккуратный вид, наводит мышь на карточки и видит реакцию, копирует адрес/телефон/цену, при желании устанавливает PWA через понятную affordance.
- **Клиент на планшете.** Альбомная/портретная ориентация; ожидает корректный вид и удобные касания, не «мелкий телефон».
- **Клиент со сниженным зрением (любая платформа).** Использует зум браузера/системы для увеличения текста — масштабирование не должно быть заблокировано.
- **Клиент на телефоне (основной кейс).** Видит привычный мобильный вид без изменений; iOS-клиент в standalone-режиме по-прежнему видит оформление под iOS, где это уместно.

## 4. Функциональные требования

- **ФТ-1. Оформление на больших экранах.** При ширине окна ≥768px приложение отображается как осознанно оформленное «устройство» на брендовом фоне, без впечатления незавершённого прототипа. Контент остаётся читаемым и центрированным; пустое пространство вокруг оформлено брендово.
- **ФТ-2. Условный показ фейкового iOS-хрома.** Фейковый статус-бар «9:41» и dynamic island отображаются только когда приложение запущено как standalone-PWA на платформе iOS. Во всех прочих случаях (десктоп, браузер, не-iOS) этот хром не отображается.
- **ФТ-3. Определение iOS-режима без UA-снифинга.** Признак «iOS standalone» определяется по сочетанию `display-mode: standalone` и признака платформы iOS, а не по парсингу строки user-agent. (Конкретный механизм определения платформы — за архитектором; ограничение: не полагаться на нестабильный UA-снифинг.)
- **ФТ-4. Hover-состояния для мыши.** Интерактивные элементы (карточки услуг и мастеров, строки списков, кнопки) при наведении указателем мыши дают визуальную обратную связь. Состояния применяются только для устройств с точным указателем (`@media (hover:hover)`), чтобы не залипать на тач-устройствах.
- **ФТ-5. Снятие блокировки зума.** Из метатега viewport удаляются `user-scalable=no` и `maximum-scale=1.0`; масштабирование (pinch-zoom и Ctrl+=/−) работает на всех платформах.
- **ФТ-6. Выделяемость контентного текста.** Контентный текст — цены, описания услуг, адрес, телефон, имена мастеров — выделяется и копируется. Запрет выделения (`user-select: none`) сохраняется только на «кнопочных», иконочных и служебных элементах. Существующие 21 правило `user-select` (наследие req-025) подлежат ревизии под это условие.
- **ФТ-7. Десктоп-установка PWA.** Affordance и тексты установки PWA адекватны для десктопных браузеров Chrome/Edge (обработка `beforeinstallprompt` уже работает): формулировка не использует мобильно-ориентированный «Добавить на экран»/iOS-overlay там, где это не соответствует платформе.
- **ФТ-8. Сохранность мобильного и тематического вида.** Мобильный вид (≤500px) визуально не изменяется. Светлая и тёмная темы, а также смена ориентации корректны на всех целевых платформах.

## 5. Нефункциональные требования

- **НФТ-1. Совместимость без регрессий.** Изменения не затрагивают публичные сигнатуры и поведение примитивов `ui.js` (v1.32.0) и a11y-поведение (v1.33.0). Существующие сценарии записи проходят без изменений.
- **НФТ-2. Доступность.** Снятие блокировки зума соответствует требованиям доступности (WCAG: пользователь должен иметь возможность масштабировать контент). Возможность копировать контент (адрес/телефон/цены) повышает доступность.
- **НФТ-3. Без новых зависимостей.** Только vanilla JS/HTML/CSS; изменения локализованы в `docs/prototype.html`, `docs/style.css`, при необходимости `docs/modules/install.js`. Никаких рантайм-зависимостей.
- **НФТ-4. Чистота консоли.** Ноль ошибок в консоли браузера на всех целевых платформах.
- **НФТ-5. Кэширование.** При изменении `APP_SHELL`/`docs/sw.js` поднимается `CACHE_VERSION`.
- **НФТ-6. Локализация.** Все пользовательские тексты — на русском языке.

## 6. Ограничения и допущения

**Ограничения:**
- Технологический стек: vanilla JS + HTML + CSS, без зависимостей; деплой статикой из `docs/` на GitHub Pages.
- Затрагиваемые файлы ограничены: `docs/prototype.html`, `docs/style.css`, при необходимости `docs/modules/install.js` (и `docs/sw.js` при изменении APP_SHELL).
- Мобильный вид (≤500px) — основной кейс, ухудшать запрещено.
- Корректность в обеих темах и при смене ориентации обязательна.
- Запрещены любые операции с веткой `backup`.
- Минимальные изменения строго в пределах объёма, согласованного с архитектором.

**Допущения:**
- «Оформленное устройство» (а не полный многоколоночный адаптив) — приемлемый для бизнеса результат на v1.34.0; полный адаптив при необходимости выносится отдельно.
- Зум снимается полностью (без `maximum-scale`), а не ограничивается значением вроде `maximum-scale=5` — принято как дефолт из требования.
- Определение iOS через `display-mode: standalone` + платформа покрывает целевые сценарии; крайние случаи (iPadOS, выдающий себя за десктоп) допустимо обрабатывать в пользу скрытия фейкового хрома (лучше не показать «9:41», чем показать его на не-iOS).
- На десктопе фейковый iOS-хром не несёт функциональной нагрузки и его скрытие не ломает раскладку.

## 7. Критерии приёмки

- **КП-1.** (ФТ-1) На десктопе и планшете приложение не выглядит «мелким телефоном-прототипом»: видно осознанно оформленное устройство на брендовом фоне. *Проверка: ручная, десктоп-браузер + планшет.*
- **КП-2.** (ФТ-2) Фейковый статус-бар «9:41» и dynamic island не отображаются на десктопе и в обычном браузере; на iOS standalone — отображаются как прежде. *Проверка: ручная, десктоп + iOS standalone (или эмуляция режима).*
- **КП-3.** (ФТ-3) Логика определения «iOS standalone» вынесена в чистую функцию (вход — флаги режима/платформы, выход — булево) и покрыта unit-тестами; UA-парсинг для этого решения не используется. *Проверка: unit-тест + код-ревью.*
- **КП-4.** (ФТ-4) При наведении мыши на карточки услуг/мастеров, строки списков и кнопки видна визуальная hover-реакция; на тач-устройстве hover не «залипает». *Проверка: ручная, мышь на десктопе + тач на мобильном.*
- **КП-5.** (ФТ-5) Зум работает: pinch-zoom на тач-устройстве и Ctrl+=/Ctrl+− в десктоп-браузере увеличивают/уменьшают контент; масштабирование не заблокировано. *Проверка: ручная, десктоп + мобильный.*
- **КП-6.** (ФТ-6) На десктопе выделяются и копируются цена, адрес, телефон, имя мастера, описание услуги; «кнопочные»/служебные элементы не выделяются. *Проверка: ручная, выделение мышью + копирование.*
- **КП-7.** (ФТ-7) Affordance/текст установки PWA в Chrome/Edge на десктопе адекватен платформе (не «Добавить на экран»/iOS-overlay). *Проверка: ручная, Chrome/Edge десктоп.*
- **КП-8.** (ФТ-8, НФТ-4) Мобильный вид (≤500px) визуально не изменился; светлая и тёмная темы корректны на всех платформах; смена ориентации корректна; ноль ошибок в консоли. *Проверка: ручная, мобильный + переключение тем + ориентация + консоль.*
- **КП-9.** (НФТ-1) Основной сценарий записи (логин → услуга → мастер → слот → бронирование) проходит без регрессий; поведение toast/sheet (v1.32.0) и a11y (v1.33.0) не изменилось. *Проверка: ручная, сквозной сценарий.*
- **КП-10.** (НФТ-5) Если изменён `APP_SHELL`/`sw.js` — `CACHE_VERSION` поднята. *Проверка: код-ревью.*

## 8. Открытые вопросы

> Эти вопросы **не блокируют** оформление BA-документа: для всех зафиксированы дефолты (решения для архитектора). Главный из них — развилка объёма — оставлен на финальное решение архитектора при подготовке ADR.

- **ОВ-1 (главный — объём, для архитектора). Настоящий адаптив vs «оформленное устройство».** Дефолт, принятый в этом документе: на v1.34.0 делаем «оформленное устройство» + hover + зум + select + скрытие iOS-хрома (минимальные изменения). Полный многоколоночный адаптив (сетки на ≥768px, рефактор) выносится отдельным требованием. Архитектор подтверждает или меняет объём в ADR; при расширении объёма раздел 2 («Вне рамок») должен быть пересмотрен.
- **ОВ-2. Способ определения платформы iOS** без UA-снифинга (например, доступные платформенные API/признаки) — конкретная реализация за архитектором; зафиксировано лишь требование «не полагаться на UA-парсинг» (ФТ-3).
- **ОВ-3. Точный перечень элементов с сохраняемым `user-select: none`.** Дефолт: запрет только на кнопках/иконках/служебных элементах, весь контентный текст выделяем. Финальный список «кнопочных» классов уточняется при ревизии 21 правила (наследие req-025) на этапе реализации.
- **ОВ-4. Визуальный язык «брендового фона/оформленного устройства»** (цвета, паддинги, тень рамки на десктопе) — дизайн-решение, согласуется с владельцем/архитектором; должен использовать существующие токены тем.

Противоречий с действующими ADR не выявлено: данный релиз сознательно не затрагивает контракты `ui.js` (ADR-034) и a11y-поведение (ADR-035).
