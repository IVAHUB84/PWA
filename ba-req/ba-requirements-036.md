## 1. Контекст и бизнес-цель

Холодный старт и навигация PWA онлайн-записи воспринимаются медленными:
приложение делает избыточные сетевые запросы при каждом запуске и при каждом
заходе на главную, а каталог услуг/мастеров перерисовывается несколько раз за
одну загрузку. Источник — ревью производительности от 2026-05-31, подтверждённое
кодом (`docs/app.js`, `docs/modules/*.js`).

Проблема, которую решаем:
1. При каждом старте на каждого мастера уходит отдельный сетевой запрос ради
   диапазонов цен (паттерн N+1, `_collectStaffPrices`, `app.js:233-266`).
2. Данные каталога (услуги/мастера/категории) не сохраняются между запусками,
   поэтому на медленной или отсутствующей сети при холодном старте каталог пуст
   до прихода ответа сети (`initApp`, `app.js:162-231`; SW намеренно не кэширует
   `api.yclients.com` — `NETWORK_ONLY_HOSTS`, `sw.js:47`).
3. Каждый вход на главную (не только старт) безусловно идёт в сеть за записями
   клиента и за GitHub-данными, причём `_ghRead` принудительно обходит кэш через
   `?_=Date.now()` (`app.js:58-71`, `github.js`).

Бизнес-цель — ускорить ощущаемый запуск и навигацию (мгновенный первый кадр
каталога, меньше лишних запросов на каждом заходе), **не меняя видимое поведение,
UI и логику бронирования**. Это улучшает удержание на первом экране и общее
восприятие качества записи, особенно на мобильной/нестабильной сети.

**Заказчик / стейкхолдеры:** владелец студии (как заказчик PWA), клиенты студии
(конечные пользователи онлайн-записи).

## 2. Область (scope)

### В рамках задачи

**Приоритетные (1-3):**
- **(1) Устранение N+1 запросов цен мастеров при старте.** Первый рендер каталога
  не должен блокироваться N пер-мастерными запросами; диапазоны цен должны
  отображаться мгновенно из сохранённого кэша, а актуализация — выполняться в
  фоне.
- **(2) Персистентный кэш данных каталога** (услуги, мастера, категории) для
  мгновенного холодного рендера по модели stale-while-revalidate на уровне данных
  приложения: при старте рендерим из кэша, параллельно обновляем из сети и
  заменяем данные по приходу ответа.
- **(3) Троттлинг фоновых обновлений на главной.** Повторные входы на `s-home` в
  пределах окна троттлинга не должны инициировать новые запросы
  `_fetchAndMergeServerRecords` (`/records`) и `_ghRead`; первый вход и старт
  работают как прежде. Убрать принудительный обход кэша `?_=Date.now()` в
  `_ghRead`.

**Опциональные (4-5, реализуются при наличии времени/без риска регрессий):**
- **(4) Сокращение дублирующихся ре-рендеров** каталога при старте (сейчас
  `renderServices()` вызывается не менее 4 раз за загрузку) — без изменения
  итогового отображения.
- **(5) Ленивая загрузка EmailJS** — не подключать сторонний CDN-скрипт при
  каждом старте; подгружать его только при входе в сценарий email-fallback, с
  сохранением работоспособности email-логина.

### Вне рамок (explicitly out of scope)
- Любые изменения видимого UI, экранов, контента, текстов, дизайна.
- Изменение логики и потока бронирования (выбор услуги → мастер → слот →
  подтверждение), экранов слотов, инбокса/push.
- Изменение серверной части и набора эндпоинтов YCLIENTS API.
- Изменение стратегий Service Worker для статики/APP_SHELL и для хоста фото
  (`assets.yclients.com`, ADR-014) — затрагивается только при технической
  необходимости и только в части данных API.
- Введение новых рантайм-зависимостей.
- Изменение способа хранения секретов / захардкоженного YCLIENTS-токена.
- Кэширование данных, содержащих токены или персональные данные сверх того, что
  уже хранится в localStorage сегодня.

## 3. Действующие лица и сценарии использования

**Клиент студии** — пользователь PWA. Цель — быстро открыть приложение и
записаться, не дожидаясь сети.

Сценарии:
1. **Повторный холодный старт (есть кэш).** Клиент открывает уже использованное
   приложение → каталог услуг и список мастеров (включая диапазоны цен)
   отображаются мгновенно из кэша → в фоне данные обновляются из сети и при
   изменениях бесшовно заменяются.
2. **Первый запуск (кэша нет) / очищенное хранилище.** Поведение как сегодня:
   данные грузятся из сети, до прихода ответа — текущее пустое/скелетон-состояние.
3. **Повторные заходы на главную в рамках сессии.** Клиент несколько раз
   переходит на `s-home` за короткий промежуток → в пределах окна троттлинга
   повторные сетевые запросы за записями и GitHub-данными не отправляются;
   данные берутся из уже загруженного состояния.
4. **Офлайн.** Сеть недоступна, но кэш есть → каталог рендерится из кэша. Сети и
   кэша нет → текущее поведение (пусто/скелетон).
5. **Email-fallback логин (опц. п.5).** Клиент входит через email → EmailJS
   подгружается в этот момент → отправка кода работает как прежде.

## 4. Функциональные требования

- **ФТ-1.** При повторном холодном старте при наличии сохранённого кэша каталог
  услуг и список мастеров отображаются мгновенно из кэша, без ожидания сетевого
  ответа.
- **ФТ-2.** После рендера из кэша данные каталога обновляются из сети в фоне и
  при изменении бесшовно заменяют отображённые (stale-while-revalidate на уровне
  данных). Свежесть обеспечивается фоновым запросом, а не TTL (см. §6, §8).
- **ФТ-3.** Первый рендер каталога не блокируется N пер-мастерными запросами цен:
  диапазоны цен берутся из кэша при наличии и/или дорисовываются в фоне после
  завершения сбора.
- **ФТ-4.** Результат сбора цен мастеров (диапазоны/цены, аналог
  `servicePriceRange` / `staffServicePrice`) сохраняется в персистентном кэше и
  используется для мгновенного первого кадра при последующих запусках.
- **ФТ-5.** При отсутствии сети, но наличии кэша, приложение рендерит каталог,
  мастеров и цены из кэша. При отсутствии и сети, и кэша сохраняется текущее
  поведение (пустое/скелетон-состояние).
- **ФТ-6.** Повторные входы на `s-home` в пределах окна троттлинга не инициируют
  новые сетевые запросы `_fetchAndMergeServerRecords` (`/records`) и `_ghRead`.
  Первый вход на `s-home` и холодный старт выполняют обновление как прежде.
- **ФТ-7.** Принудительный обход кэша `?_=Date.now()` в `_ghRead` устранён;
  актуальность данных регулируется троттлингом частоты обращений (ФТ-6), а не
  cache-busting.
- **ФТ-8.** Видимое поведение, состав экранов, контент и логика бронирования не
  изменяются: те же экраны, тот же контент, тот же поток
  логин → услуга → мастер → слот → бронь.
- **ФТ-9.** Запись в персистентный кэш обёрнута в `try/catch`; недоступность или
  переполнение хранилища (приватный режим, лимит) не приводит к ошибке старта —
  приложение деградирует до поведения «без кэша».
- **ФТ-10.** Персистентный кэш не содержит секретов (в т. ч. YCLIENTS-токена).
- **ФТ-11 (опц.).** Число вызовов полного ре-рендера каталога при старте
  сокращено относительно текущего (≥4) без изменения итогового отображения.
- **ФТ-12 (опц.).** Скрипт EmailJS не загружается при старте; он подгружается
  только при входе в сценарий email-fallback, при этом email-логин (отправка и
  проверка кода) работает как прежде.

## 5. Нефункциональные требования

- **НФТ-1 (Производительность).** Ощущаемое время до первого кадра каталога при
  повторном холодном старте сокращается: первый кадр не ждёт сети при наличии
  кэша. Число сетевых запросов при старте и при повторных входах на `s-home`
  снижается (проверяется в DevTools → Network).
- **НФТ-2 (Без регрессий поведения/UI).** Видимый результат, контент и логика
  идентичны текущим; визуальных изменений нет.
- **НФТ-3 (Устойчивость).** Сбои/недоступность/переполнение хранилища кэша не
  ломают запуск (ФТ-9); ноль ошибок в консоли на основном сценарии.
- **НФТ-4 (Безопасность / приватность).** Кэш не содержит токенов; не вводятся
  новые места хранения секретов; объём кэшируемых данных не выходит за рамки уже
  хранимых сегодня сущностей каталога.
- **НФТ-5 (Технологические ограничения).** Только vanilla JS/HTML/CSS, без новых
  рантайм-зависимостей; затрагиваемые файлы — `docs/app.js`, `docs/modules/*.js`
  (в т. ч. `state.js`, `api.js`, `github.js`), при необходимости
  `docs/prototype.html` (п.5) и `docs/sw.js` (только данные API при крайней
  необходимости).
- **НФТ-6 (PWA / кэш статики).** Установка PWA не должна сломаться; при изменении
  `APP_SHELL` или `sw.js` обязательно поднять `CACHE_VERSION`.

## 6. Ограничения и допущения

Ограничения:
- Видимое поведение, UI, контент и логика бронирования не меняются (только
  производительность).
- Минимальные изменения строго по объёму; не трогать поток бронирования, слоты,
  инбокс/push, стратегии SW для статики и хоста фото — кроме случаев реальной
  необходимости в части данных API.
- Без новых рантайм-зависимостей; код в `docs/app.js`, `docs/modules/*.js`, при
  необходимости `docs/prototype.html`, `docs/sw.js`.
- `localStorage.setItem` (и любая запись в персистентный кэш) — в `try/catch`,
  как уже принято в коде (например, `api.js:108`).
- Кэш не должен включать токены; не усугублять проблему захардкоженного токена.
- При изменении `APP_SHELL`/`sw.js` — поднять `CACHE_VERSION`.
- Проект статический, без автотестов — приёмка ручная (DevTools, сквозной
  сценарий).

Допущения (приняты из исходного требования; см. §8 — требуют подтверждения
архитектора):
- Хранилище кэша каталога — **localStorage** (синхронный мгновенный рендер при
  старте, малый объём данных), а не IndexedDB.
- **TTL не вводится**; используется чистая модель stale-while-revalidate: кэш
  нужен только для мгновенного первого кадра, актуальность гарантирует фоновый
  запрос.
- N+1 устраняется не за счёт нового batch-эндпоинта, а за счёт кэширования
  результата сбора цен и фонового (неблокирующего) обновления существующими
  пер-мастерными запросами.
- Окно троттлинга для `s-home` — **60 секунд** (timestamp в памяти модуля); при
  старте обновление выполняется всегда.
- Данные каталога (услуги/мастера/категории) — почти статичны, что и делает
  кэширование с фоновым обновлением безопасным с точки зрения свежести.

## 7. Критерии приёмки

- **КП-1.** При повторном холодном старте с заполненным кэшем каталог услуг и
  список мастеров видны мгновенно, до прихода сетевого ответа (DevTools →
  Network: рендер происходит раньше завершения запросов каталога). (ФТ-1)
- **КП-2.** После мгновенного рендера данные обновляются из сети в фоне; при
  изменении данных отображение обновляется без видимого «мигания» сверх текущего
  поведения. (ФТ-2)
- **КП-3.** При старте каталог рендерится без ожидания N пер-мастерных запросов
  цен; диапазоны цен присутствуют сразу (из кэша) и/или дорисовываются в фоне.
  (ФТ-3, ФТ-4)
- **КП-4.** Офлайн с кэшем: при выключенной сети (DevTools → Network: Offline)
  каталог, мастера и цены рендерятся из кэша. Офлайн без кэша (хранилище
  очищено): поведение совпадает с текущим (пусто/скелетон). (ФТ-5)
- **КП-5.** Повторные входы на `s-home` в пределах 60 с не порождают новых
  запросов `/records` и `_ghRead` (DevTools → Network: при повторной навигации
  на главную новых запросов нет); первый вход/старт запросы выполняет. (ФТ-6)
- **КП-6.** В запросе GitHub-данных отсутствует параметр `?_=<timestamp>`
  (DevTools → Network: URL `_ghRead` без cache-bust). (ФТ-7)
- **КП-7.** Сквозной сценарий логин → услуга → мастер → слот → бронь проходит без
  регрессий; экраны, контент и логика идентичны прежним; ноль ошибок в консоли;
  PWA installable не сломан (Lighthouse/Application). (ФТ-8, НФТ-2)
- **КП-8.** При недоступном/переполненном хранилище (эмуляция: приватный режим
  или принудительная ошибка `setItem`) старт не падает, приложение работает в
  режиме «без кэша»; ноль необработанных ошибок в консоли. (ФТ-9, НФТ-3)
- **КП-9.** Содержимое персистентного кэша (DevTools → Application →
  Local Storage) не содержит YCLIENTS-токена и иных секретов. (ФТ-10, НФТ-4)
- **КП-10 (опц.).** Число полных ре-рендеров каталога при старте сокращено
  относительно текущего (≥4) без визуальных отличий итогового состояния
  (наблюдение через лог/инструментирование при разработке). (ФТ-11)
- **КП-11 (опц.).** При старте скрипт EmailJS не запрашивается (DevTools →
  Network: нет запроса к CDN EmailJS на старте); при входе в email-fallback
  скрипт подгружается и email-логин работает как прежде. (ФТ-12)
- **КП-12.** При изменении `sw.js`/`APP_SHELL` поднята `CACHE_VERSION`; иначе
  правок `sw.js` нет. (НФТ-6)

> Соответствие требований и критериев: ФТ-1→КП-1; ФТ-2→КП-2; ФТ-3,ФТ-4→КП-3;
> ФТ-5→КП-4; ФТ-6→КП-5; ФТ-7→КП-6; ФТ-8→КП-7; ФТ-9→КП-8; ФТ-10→КП-9;
> ФТ-11→КП-10; ФТ-12→КП-11; НФТ-6→КП-12.

## 8. Открытые вопросы

Блокирующих вопросов нет. По всем продуктовым развилкам исходное требование
зафиксировало предложения автора; ниже они приведены как **рекомендации-дефолты**,
требующие подтверждения архитектора на этапе ADR (но не блокирующие
проектирование). Отдельно отмечено пересечение с ранее принятым ADR-012.

- **ОВ-1. Пересмотр решения ADR-012 о persist кэша цен (важно для архитектора).**
  ADR-012 (диапазон цен услуг) **сознательно постановил** хранить кэш цен
  (`servicePriceRange`, `staffServicePrice`) **только в памяти на сессию**, а
  persist в `localStorage` **не вводить** (отнесено в технический долг: «persist
  кэша цен и его инвалидация — не делается; при необходимости — отдельным
  решением»). Настоящее требование (ФТ-4) **именно вводит этот persist**. Это не
  противоречие, а целенаправленный пересмотр отложенного техдолга. *Рекомендация:*
  в новом ADR явно отметить вытеснение решения ADR-012 в части хранения кэша цен
  и зафиксировать инвалидацию (модель stale-while-revalidate без TTL — ОВ-3).

- **ОВ-2. Хранилище кэша каталога.** localStorage (синхронный, мгновенный рендер,
  ~5 МБ) vs IndexedDB (асинхронный, больше места, уже используется для inbox).
  *Рекомендация-дефолт (требует подтверждения):* **localStorage** — данные малы,
  нужен синхронный первый кадр.

- **ОВ-3. TTL / инвалидация кэша.** Чистый stale-while-revalidate (всегда
  обновлять в фоне, без TTL) vs введение TTL. *Рекомендация-дефолт:* **без TTL** —
  кэш только для первого кадра, свежесть обеспечивает фоновый запрос.

- **ОВ-4. Полное устранение N+1.** Есть ли в YCLIENTS batch-эндпоинт цен по всем
  мастерам, либо оставить пер-мастерные запросы фоновыми с кэшированием.
  *Рекомендация-дефолт:* **batch не вводить**; кэшировать результат сбора цен и
  рендерить из кэша мгновенно, пер-мастерные запросы оставить как неблокирующее
  фоновое обновление (согласуется со сложившимся подходом ADR-012, A1/B1).

- **ОВ-5. Окно троттлинга `s-home`.** Как часто допустимо реально ходить в сеть
  при повторных входах. *Рекомендация-дефолт:* **не чаще 1 раза в 60 с**
  (timestamp в памяти модуля); при старте — всегда.

- **ОВ-6. `_ghRead` cache-bust.** Убрать `?_=Date.now()` и положиться на
  HTTP-кэш/SW + троттлинг, либо оставить cache-bust с троттлингом.
  *Рекомендация-дефолт:* **убрать cache-bust + добавить троттлинг** (ФТ-7).

- **ОВ-7. Объём опциональных пунктов (4-5).** Включать ли сокращение ре-рендеров
  и ленивую загрузку EmailJS в этот же релиз или вынести отдельно. *Решение —*
  за архитектором/lead при планировании релиза; на приёмку влияет лишь
  соответствующими опциональными КП.
