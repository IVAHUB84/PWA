# BA-требования 002: Фильтрация мастеров по выбранной услуге

- **Источник:** `requirements/req-master-filter-002.md`
- **Дата:** 2026-05-28
- **Статус:** готово

## 1. Контекст и бизнес-цель

Студия красоты «Реснички, сестрички» (направления: брови, ресницы, макияж, перманентный макияж) принимает онлайн-записи через PWA поверх YCLIENTS. Сценарий записи: клиент выбирает услугу (`s-services`) → попадает на экран мастеров (`s-masters`) → выбирает мастера или «Любого мастера» → выбирает слот.

Сейчас на экране мастеров показываются **все** мастера студии, независимо от того, оказывают ли они выбранную услугу. Из-за этого клиент видит, например, мастера по бровям при выборе услуги по ресницам. Это вводит в заблуждение и приводит к ошибкам бронирования (запись к мастеру, который не выполняет выбранную услугу).

Технически частичная фильтрация в коде уже есть, но она опирается на **статические** данные мастеров (поле `cats` — категории) из `docs/modules/state.js`, а не на реальные данные YCLIENTS о том, какие услуги оказывает конкретный мастер. Для мастеров, не совпавших по имени со статической таблицей, привязка категорий пустая, и фильтрация фактически не срабатывает.

**Бизнес-цель:** показывать на экране мастеров только тех, кто действительно оказывает выбранную услугу (по данным YCLIENTS), чтобы исключить заведомо некорректные записи и снизить путаницу при выборе.

**Стейкхолдеры:**
- Заказчик / владелец студии — заинтересован в корректных записях и отсутствии «холостых» переходов клиента.
- Клиенты студии — конечные пользователи экрана выбора мастера.
- Бизнес-аналитик / архитектор / разработчик — оформление и реализация.

## 2. Область (scope)

### В рамках задачи
- На экране `s-masters` (режим перехода от выбранной услуги) показывать только тех мастеров, которые оказывают выбранную услугу.
- Источником данных о соответствии «мастер ↔ услуга» должен быть **YCLIENTS API**, а не статическая таблица категорий из кода.
- Мастера, не оказывающие выбранную услугу, **не отображаются** в списке (не показываются серыми/заблокированными — просто отсутствуют).
- Понятное пустое состояние: если ни один мастер не найден для выбранной услуги — показать сообщение, объясняющее ситуацию.
- Сохранение режима «Любой мастер» (кнопка в начале списка) без изменений.
- Сохранение режима предвыбранного мастера (переход с карточки мастера) без изменений.

### Вне рамок (explicitly out of scope)
- Внешний вид и структура карточек мастера (вёрстка, аватары, бейджи, кнопка «избранное») — не меняются.
- Логика бронирования после выбора мастера (выбор слота, создание записи) — не меняется.
- Режим «Все мастера студии» (просмотр всего списка вне привязки к услуге, `state._mastersAll`) — остаётся как есть, без фильтрации по услуге.
- Сортировка/приоритизация мастеров (избранные вперёд и т. п.) — не меняется.
- Доступность слотов мастера на конкретную дату/время (поле `avail`/`availText`) — отдельная тема, не входит в эту задачу.
- Изменение источника или набора данных об услугах (`s-services`) — не затрагивается.
- Визуальный каталог услуг (см. `ba-req/ba-requirements-001.md`) — не затрагивается.

## 3. Действующие лица и сценарии использования

**Клиент студии** — выбирает услугу и хочет записаться к подходящему мастеру.

Основной сценарий:
1. Клиент выбирает услугу на экране `s-services` (например, «Коррекция бровей»).
2. Переходит на экран мастеров `s-masters`.
3. Видит **только** мастеров, оказывающих выбранную услугу, плюс кнопку «Любой мастер».
4. Выбирает мастера (или «Любой мастер») и продолжает запись.

Граничный сценарий (нет подходящих мастеров):
- Для выбранной услуги ни один мастер не доступен → экран показывает понятное сообщение вместо пустого списка (например, что на эту услугу сейчас нет доступных мастеров). Поведение кнопки «Любой мастер» при этом определяется в открытых вопросах.

Альтернативный сценарий (предвыбранный мастер):
- Клиент пришёл с карточки конкретного мастера (`masterPreSelected`); список мастеров с фильтрацией по услуге в этом режиме не показывается (клиент идёт сразу к выбору слота). Поведение этого режима не меняется.

Альтернативный сценарий (просмотр всех мастеров):
- Клиент открывает «Все мастера студии» (`_browseAllMasters`) вне контекста выбора услуги → видит полный список без фильтрации по услуге. Поведение не меняется.

## 4. Функциональные требования

- **ФТ-1.** При переходе на экран мастеров `s-masters` после выбора услуги список содержит только тех мастеров, которые оказывают выбранную услугу.
- **ФТ-2.** Признак «мастер оказывает услугу» определяется по данным **YCLIENTS API**, а не по статическому полю категорий мастера в коде. Статическая привязка по категориям не должна оставаться единственным или основным критерием фильтрации.
- **ФТ-3.** Мастера, не оказывающие выбранную услугу, отсутствуют в списке (не отображаются как заблокированные/затенённые элементы).
- **ФТ-4.** Кнопка «Любой мастер» в начале списка сохраняется и работает как прежде.
- **ФТ-5.** Режим предвыбранного мастера (`masterPreSelected`, переход с карточки мастера) сохраняется без изменений: фильтрация списка к нему не применяется.
- **ФТ-6.** Если для выбранной услуги нет ни одного подходящего мастера, экран показывает понятное текстовое сообщение об этом, а не пустой список без пояснения.
- **ФТ-7.** Внешний вид карточек мастера и логика последующего бронирования не изменяются — фильтрация затрагивает только состав списка.
- **ФТ-8.** Режим «Все мастера студии» (просмотр полного списка вне привязки к услуге) продолжает показывать всех мастеров без фильтрации по услуге.

## 5. Нефункциональные требования

- **НФТ-1 (Корректность данных).** Фильтрация должна отражать актуальное состояние услуг мастера в YCLIENTS; рассинхрон со статической таблицей кода недопустим как источник истины.
- **НФТ-2 (Производительность/отзывчивость).** Получение данных для фильтрации не должно заметно задерживать открытие экрана мастеров на мобильном устройстве; на время загрузки допустимо показывать индикатор, но переход не должен «зависать» без обратной связи.
- **НФТ-3 (Деградация при ошибке API).** При недоступности или ошибке YCLIENTS API экран не должен «ломаться» или показывать пустоту без объяснения; требуется предсказуемое поведение (см. открытый вопрос об отказе API).
- **НФТ-4 (Совместимость).** Решение остаётся в рамках стека: vanilla JS/HTML/CSS, статический хостинг из `docs/` на GitHub Pages, без новых рантайм-зависимостей; единственный внешний источник — YCLIENTS API.
- **НФТ-5 (PWA/офлайн).** Поведение Service Worker и устанавливаемость PWA не должны регрессировать; влияние на `CACHE_VERSION` (`docs/sw.js`) — предмет проектирования.
- **НФТ-6 (Локализация).** Все пользовательские сообщения, включая пустое состояние, — на русском языке, в едином тоне с остальными экранами.

## 6. Ограничения и допущения

**Ограничения:**
- Приложение — статический PWA в `docs/`, обслуживается GitHub Pages; собственного бэкенда нет, единственный внешний источник — YCLIENTS API.
- Запрещены рантайм-npm-зависимости (только devDependencies).
- При изменении состава кэшируемых ресурсов (`APP_SHELL`) требуется поднятие `CACHE_VERSION` в `docs/sw.js`.
- YCLIENTS API токен сейчас захардкожен в коде — известная проблема, не усугублять и не затрагивать в рамках этой задачи.

**Допущения:**
- YCLIENTS предоставляет данные о соответствии мастеров и услуг: по каждому мастеру можно получить список его услуг (`GET /book_services/{company}?staff_id=N`), а по услуге — список оказывающих её мастеров (`GET /book_staff/{company}?service_ids=...`). Конкретный способ получения выбирает архитектор.
- У выбранной услуги доступен идентификатор/признак, по которому YCLIENTS сопоставляет её с мастерами (после перехода проекта на загрузку услуг из YCLIENTS у услуги есть её YCLIENTS-ID).
- Кнопка «Любой мастер» концептуально означает «запись на услугу без привязки к конкретному мастеру» и остаётся доступной в основном сценарии.
- Текущая навигация, состояние (`state`) и контракт перехода между экранами остаются актуальными и переиспользуются.

## 7. Критерии приёмки

- **КП-1.** После выбора услуги на экране `s-masters` отображаются только мастера, оказывающие эту услугу; мастера, её не оказывающие, в списке отсутствуют. (ФТ-1, ФТ-3)
- **КП-2.** Состав отфильтрованного списка соответствует данным YCLIENTS об услугах мастеров, а не статической таблице категорий из кода: при расхождении статики и YCLIENTS список следует за YCLIENTS. (ФТ-2, НФТ-1)
- **КП-3.** Кнопка «Любой мастер» присутствует и при нажатии ведёт к продолжению записи без выбора конкретного мастера, как прежде. (ФТ-4)
- **КП-4.** Переход с карточки конкретного мастера (`masterPreSelected`) работает без изменений и без применения фильтрации списка. (ФТ-5)
- **КП-5.** Если для выбранной услуги нет подходящих мастеров, на экране показывается понятное русскоязычное сообщение вместо пустого списка. (ФТ-6, НФТ-6)
- **КП-6.** Внешний вид карточек мастера и логика бронирования после выбора мастера остаются прежними. (ФТ-7)
- **КП-7.** Режим «Все мастера студии» по-прежнему показывает полный список мастеров без фильтрации по услуге. (ФТ-8)
- **КП-8.** Открытие экрана мастеров остаётся отзывчивым: на время получения данных пользователю даётся обратная связь (индикатор загрузки), переход не зависает без реакции. (НФТ-2)
- **КП-9.** При ошибке/недоступности YCLIENTS API экран ведёт себя предсказуемо и не показывает необъяснённую пустоту (конкретная стратегия — по решению из открытого вопроса). (НФТ-3)
- **КП-10.** PWA остаётся устанавливаемой, офлайн-поведение не регрессирует, новые рантайм-зависимости не добавляются. (НФТ-4, НФТ-5)

## 8. Открытые вопросы

1. **Поведение при отказе API.** ✅ *Решено:* fail-open — показывать всех мастеров, не блокировать запись.
2. **«Любой мастер» при пустом результате.** ✅ *Решено:* скрывать кнопку «Любой мастер» когда список мастеров пуст.
3. **Учитывать ли `avail`.** ✅ *Решено:* фильтрация только по факту оказания услуги, независимо от доступности слотов сегодня. Мастера без окон остаются в списке (их карточки отображаются как недоступные — текущее поведение сохраняется).
4. **Судьба поля `cats`.** ✅ *Решено:* на усмотрение архитектора — сохранить как legacy-фолбэк или удалить.

Противоречий с принятыми ADR не выявлено: единственный принятый `adrs/adr-001.md` относится к визуальному каталогу услуг и не затрагивает фильтрацию мастеров.
