# BA-требования 011: Открытие конкретного поста из превью ленты на главной

- **Источник:** `requirements/req-feed-open-single-post-011.md`
- **Дата:** 2026-05-29
- **Статус:** черновик

## 1. Контекст и бизнес-цель

Studio PWA (студия красоты, онлайн-запись поверх YCLIENTS) — устанавливаемое приложение, отдаётся статически из `docs/` на GitHub Pages. На главном экране есть блок «Лента студии» с горизонтальной прокруткой превью последних публикаций (`_renderHomeFeedPreview`, `docs/modules/profile.js:132-160`) и ссылкой «Все →» рядом с заголовком секции (`docs/prototype.html:72`).

Сейчас тап по **конкретной** плашке превью выполняет `onclick="go('s-feed')"` (`profile.js:154`) — тот же самый переход, что и у ссылки «Все →» (`prototype.html:72`). То есть тап по отдельному посту открывает **общий список всех публикаций** (`s-feed`, `_paintFeed`, `docs/modules/feed.js:53-80`), а не выбранную публикацию. Это расходится с пользовательским ожиданием: тап по конкретной плашке должен открывать **именно этот пост**.

**Бизнес-цель:** сделать так, чтобы тап по конкретной плашке превью ленты на главной открывал именно выбранную публикацию, при этом ссылка «Все →» сохраняла текущее поведение (полный список ленты). Это устраняет вводящее в заблуждение поведение и приводит превью ленты в соответствие с уже сложившимся в приложении паттерном «тап по конкретному элементу → его представление» (ср. ADR-008 для мастеров, ADR-009 для визитов).

**Стейкхолдеры:**
- Заказчик / владелец студии — заинтересован в том, чтобы публикации ленты прочитывались целиком (вовлечение, переход к записи через кнопку «Записаться →» внутри поста).
- Клиент студии — пользователь, который тапает по конкретной плашке, ожидая увидеть именно её содержимое.
- Бизнес-аналитик / архитектор / разработчик — оформление, проектирование, реализация.

## 2. Область (scope)

### В рамках задачи
- Тап по **конкретной** плашке превью ленты на главном экране открывает представление **именно выбранной** публикации (а не общий список).
- Адресация выбранной публикации выполняется по её устойчивому идентификатору `p.id` (создаётся в `admin.js:111` как `Date.now()`), а не по позиции в списке: список может меняться (новые публикации, удаление, пагинация превью `slice(0,6)`).
- Ссылка «Все →» (`prototype.html:72`) сохраняет текущее поведение — открывает полный список ленты `s-feed`.
- Из представления выбранной публикации клиент может вернуться назад обычной навигацией (конкретная точка возврата — открытый вопрос №3).
- Корректная обработка ситуации, когда публикации с переданным `id` уже нет (например, удалена через админку): без ошибки в консоли и без «пустого экрана без объяснения» (поведение — открытый вопрос №5).
- Источник публикаций — существующий: `yc_feed_posts` в `localStorage` с обновлением из GitHub через `_ghReadFn` (`feed.js:38`); видимыми считаются только не-черновики (`!p.draft`), как в текущем коде.

### Вне рамок (explicitly out of scope)
- **Изменение поведения ссылки «Все →»** и общего списка ленты `s-feed` (`_paintFeed`, бар фильтра категорий `_renderFeedCats`/`_feedFilterCat`) — сохраняются как есть; вид и логика фильтрации общего списка не дорабатываются.
- **Изменение содержимого/вёрстки самой публикации** (текст, изображение, категория, дата) и поведения кнопки «Записаться →» внутри карточки поста (`_feedGoBook`, `feed.js:73,106-109`) — переиспользуются как есть.
- **Создание/редактирование/удаление публикаций**, админка ленты (`admin.js`), формирование `id`, синхронизация с GitHub — не затрагиваются (используются как источник данных).
- **Изменение состава полей публикации** или формата хранения `yc_feed_posts`.
- **Доработка превью на главной по сути** (количество плашек `slice(0,6)`, их вёрстка, условие показа секции) — меняется только целевое действие тапа по плашке.
- **Перенос захардкоженного YCLIENTS-токена** из `prototype.html` (известный техдолг — «не усугублять», не трогать).

## 3. Действующие лица и сценарии использования

**Клиент студии** — пользователь PWA, открывший главный экран, где в блоке «Лента студии» показано до 6 превью последних публикаций. Цель — прочитать конкретную заинтересовавшую публикацию целиком.

Основной сценарий:
1. Клиент на главном экране видит блок «Лента студии» с превью публикаций и ссылкой «Все →».
2. Клиент тапает по **конкретной** плашке превью.
3. Открывается представление **именно этой** публикации (её изображение/иконка, полный текст, категория, дата; кнопка «Записаться →» доступна, как в карточке списка).
4. Клиент возвращается назад обычной навигацией (к главной или к ленте — открытый вопрос №3).

Альтернативный сценарий:
1. Клиент тапает по ссылке «Все →».
2. Открывается полный список ленты `s-feed` со всеми опубликованными постами и баром фильтра категорий — поведение не меняется.

Граничные сценарии:
- **Публикация по `id` отсутствует** (удалена через админку / отсутствует после обновления из GitHub) к моменту перехода — клиент не должен попасть на пустой экран без объяснения и не должно возникать ошибки (поведение — открытый вопрос №5).
- **Публикация стала черновиком** (`draft === true`) после обновления данных — приравнивается к «недоступна» (видимыми считаются только `!p.draft`).

## 4. Функциональные требования

- **ФТ-1.** Тап по конкретной плашке превью ленты на главном экране открывает представление **именно выбранной** публикации, а не общий список всех публикаций.
- **ФТ-2.** Выбранная публикация адресуется по устойчивому идентификатору `p.id`, а не по позиции элемента в списке.
- **ФТ-3.** Ссылка «Все →» рядом с заголовком «Лента студии» сохраняет текущее поведение — открывает полный список ленты (`s-feed`).
- **ФТ-4.** В представлении выбранной публикации отображается её содержимое в объёме, уже доступном для поста в списке ленты: изображение или иконка категории, полный текст, категория, дата.
- **ФТ-5.** В представлении выбранной публикации сохраняется возможность перейти к записи (кнопка «Записаться →» с тем же поведением, что в карточке списка, — `_feedGoBook` для категории поста).
- **ФТ-6.** Из представления выбранной публикации доступен возврат назад обычной навигацией приложения (точка возврата — по решению открытого вопроса №3).
- **ФТ-7.** Если публикации с переданным `id` нет среди видимых (`!p.draft`) на момент открытия, поведение не приводит к ошибке в консоли и к пустому экрану без объяснения (конкретика — по решению открытого вопроса №5).
- **ФТ-8.** Видимыми для открытия считаются только опубликованные посты (`!p.draft`), из текущего источника (`yc_feed_posts` в `localStorage` с обновлением из GitHub через `_ghReadFn`).
- **ФТ-9.** Все пользовательские тексты, затронутые изменением, — на русском языке, в едином тоне приложения.

## 5. Нефункциональные требования

- **НФТ-1 (Совместимость стека).** Решение остаётся в рамках стека: vanilla JS/HTML/CSS, код в `docs/`; статический хостинг на GitHub Pages; без новых рантайм-npm-зависимостей.
- **НФТ-2 (Единый источник данных).** Представление выбранной публикации строится из тех же данных, что превью и список ленты (`yc_feed_posts` + `_ghReadFn`); дополнительные поля и запросы не вводятся.
- **НФТ-3 (Минимальность изменений).** Изменения строго в объёме задачи; общий список ленты `s-feed`, бар фильтра категорий, кнопка «Записаться →» и админка ленты не должны сломаться.
- **НФТ-4 (PWA/офлайн не регрессируют).** Изменения не ломают устанавливаемость PWA и офлайн-поведение Service Worker; при изменении состава `APP_SHELL`/кэшируемых файлов (`prototype.html`, `profile.js`, `feed.js`, `constants.js` и т. п.) поднимается `CACHE_VERSION` в `docs/sw.js` (правило проекта).
- **НФТ-5 (Локализация).** Все пользовательские тексты — на русском языке.

## 6. Ограничения и допущения

**Ограничения:**
- Приложение — статический PWA в `docs/` на GitHub Pages; релевантный код — `docs/modules/profile.js` (`_renderHomeFeedPreview`), `docs/modules/feed.js` (`_paintFeed`, `renderClientFeed`, фильтр категорий), разметка экранов — `docs/prototype.html`, реестр экранов — `docs/modules/constants.js` (`SCREENS`), навигация — `docs/modules/navigation.js` (`go`).
- Источник публикаций и фильтрация (`!p.draft`) уже существуют и должны переиспользоваться; параллельную/иную логику отбора вводить не следует.
- Идентификатор поста `p.id` — `Date.now()` (`admin.js:111`); уникальность в пределах данных не гарантируется формально, но это единственный стабильный адресный признак (надёжнее индекса). Достаточность `p.id` как адреса подтверждает архитектор.
- Запрещены рантайм-npm-зависимости; запрещён коммит секретов; запрещены любые операции с веткой `backup`.
- При изменении файлов из `APP_SHELL` — поднятие `CACHE_VERSION` в `docs/sw.js`.

**Допущения:**
- «Доступная для открытия публикация» = пост со `!p.draft` в текущем `yc_feed_posts` (как в коде превью и списка).
- Данных публикации (`image`, `text`, `cat`, `date`, `id`) достаточно для представления одного поста в том же объёме, что и в карточке списка; дополнительных полей/запросов не требуется. **Подтверждает архитектор** при проектировании.
- До решения заказчика по открытым вопросам №1–№5 окончательная форма представления (отдельный экран `s-post`/модалка либо прокрутка с подсветкой в `s-feed`), судьба бара фильтра в этом представлении, точка возврата «Назад» и поведение при отсутствующем `id` не фиксируются на уровне реализации.

## 7. Критерии приёмки

- **КП-1.** Тап по конкретной плашке превью ленты на главной открывает именно выбранную публикацию, а не общий список всех публикаций. (ФТ-1)
- **КП-2.** Открытая публикация соответствует тапнутой плашке (адресация по `p.id`); при изменении состава/порядка списка открывается по-прежнему именно выбранный пост, а не сосед по позиции. (ФТ-2)
- **КП-3.** Ссылка «Все →» по-прежнему открывает полный список ленты `s-feed` со всеми опубликованными постами. (ФТ-3)
- **КП-4.** В представлении выбранной публикации отображены её изображение/иконка категории, полный текст, категория и дата. (ФТ-4)
- **КП-5.** В представлении выбранной публикации доступна кнопка «Записаться →», ведущая к записи по категории поста (как в карточке списка). (ФТ-5)
- **КП-6.** Из представления выбранной публикации возможен возврат назад обычной навигацией. (ФТ-6)
- **КП-7.** При открытии поста, которого уже нет среди видимых по `id`, не возникает ошибки в консоли и клиент не остаётся на пустом экране без объяснения. (ФТ-7)
- **КП-8.** Общий список ленты `s-feed`, бар фильтра категорий и админка ленты не регрессируют. (НФТ-3)
- **КП-9.** PWA остаётся устанавливаемой, офлайн-поведение не регрессирует; при изменении файлов `APP_SHELL` поднята `CACHE_VERSION`; новые рантайм-зависимости не добавлены. (НФТ-1, НФТ-4)
- **КП-10.** Затронутые пользовательские тексты — на русском языке. (ФТ-9, НФТ-5)

## 8. Открытые вопросы

> **РЕШЕНО заказчиком 2026-05-29** (перед запуском `/adr`):
> 1. **Форма представления — отдельный экран `s-post`** (регистрируется в `SCREENS`, `constants.js`; по аналогии с ADR-009 `s-upcoming`).
> 2. **Бар фильтра категорий в представлении поста не нужен** (следствие выбора отдельного экрана `s-post`).
> 3. **«Назад» возвращает на главную** (`s-home`, откуда был тап).
> 4. **Состав представления — как в карточке списка**: фото/иконка категории, полный текст, категория, дата, кнопка «Записаться →» (тот же набор, что в `_paintFeed`).
> 5. **При отсутствующем/удалённом посте по `id`** (нет среди `!p.draft`) — **открыть общий список ленты `s-feed`** (мягкий фолбэк, без ошибки).
>
> Эти решения фиксируют форму реализации для архитектора. Ниже — исходные формулировки вопросов (для контекста).

Перечисленные ниже вопросы — продуктовые/UX-решения. Они **не блокируют** оформление настоящего BA-документа, но **должны быть решены до запуска `/adr`**, так как определяют форму реализации. Архитектор не должен выбирать вариант самостоятельно без согласования с заказчиком.

1. **Как именно «открыть пост»?** Отдельное представление одной публикации (новый экран `s-post`, регистрируемый в `SCREENS`, `constants.js`, — по аналогии с ADR-009 `s-upcoming`; или модальное окно) **или** открыть `s-feed` с автопрокруткой к выбранному посту и его подсветкой? Формулировка требования («именно этот пост, а не все») склоняется к **чистому представлению одной публикации**. *Влияет на раздел 2, ФТ-1, ФТ-4. Продуктовое решение заказчика; механику — архитектор.*

2. **Судьба бара фильтра категорий в представлении одной публикации.** Если выбран вариант отдельного представления — бар фильтра категорий (`feedCatBar`, `_renderFeedCats`) в нём, вероятно, не нужен; подтвердить. Если выбран вариант «прокрутка+подсветка в `s-feed`» — бар, очевидно, остаётся. *Влияет на ФТ-4, КП-4. Продуктовое решение заказчика.*

3. **Навигация «Назад».** Куда возвращает «Назад» из открытого поста — на главную (`s-home`, откуда был тап) или в полный список ленты (`s-feed`)? Требование допускает оба («к главной/ленте»). *Влияет на ФТ-6, КП-6. Продуктовое решение заказчика.*

4. **Набор содержимого/действий в представлении поста.** Подтвердить, что в представлении одной публикации показывается тот же набор, что в карточке списка (изображение/иконка, текст, категория, дата, кнопка «Записаться →»), без дополнительных элементов. Настоящий документ исходит из этого (ФТ-4, ФТ-5); требуется явное подтверждение. *Влияет на ФТ-4, ФТ-5, КП-4, КП-5. Продуктовое решение заказчика.*

5. **Поведение при отсутствующем/удалённом посте по `id`.** Что показывать, если публикации с переданным `id` уже нет (удалена через админку или стала черновиком после обновления из GitHub): открыть общий список ленты `s-feed`, показать сообщение «Публикация недоступна», вернуть на главную или иное? Требование задаёт лишь нижнюю границу — «без ошибки и без пустого экрана без объяснения». *Влияет на ФТ-7, КП-7. Продуктовое решение заказчика.*

**Противоречий с принятыми ADR (`adrs/adr-001`…`adr-009`) не выявлено.** Задача касается навигации и представления уже имеющихся данных ленты и согласуется по духу с ADR-008 (контекстный тап по мастеру) и ADR-009 (отдельный экран `s-upcoming` с переиспользованием рендера) — но эти ADR не предписывают конкретного решения для поста ленты. Если на этапе проектирования выбранный вариант (например, создание нового экрана `s-post`) затронет действующий ADR, это должно быть отражено в новом/обновлённом ADR. Отмечу также общесистемное ограничение из ADR-009: любой новый экран обязан быть зарегистрирован в `SCREENS` (`constants.js`), иначе анимация навигации и сброс сценариев его не учтут — это вход для архитектора, не блокирующий BA.
