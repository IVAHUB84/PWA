# BA-требования 034: Экран «Выберите время» — редизайн под мобильное приложение YCLIENTS

- **Источник:** `requirements/req-slots-time-picker-yclients-033.md`
- **Дата:** 2026-05-30
- **Статус:** черновик

> Примечание о нумерации: исходное требование имеет номер `033`, однако
> `ba-req/ba-requirements-033.md` уже занят другим требованием (кликабельные
> уведомления, источник `req-inbox-clickable-notifications-032.md`). Поэтому
> данному документу присвоен следующий свободный номер — `034`. Содержательно
> он соответствует `requirements/req-slots-time-picker-yclients-033.md`.

## 1. Контекст и бизнес-цель

Экран выбора времени (`s-slots`) — ключевой шаг воронки онлайн-записи: клиент
выбирает дату и слот перед бронированием. Сейчас дата представлена короткой
горизонтальной лентой ближайших доступных дней, а слоты — плоской сеткой без
группировки. Это ограничивает обзор расписания (не видно весь месяц) и
затрудняет навигацию по большому числу слотов.

Цель — привести экран к привычному для пользователей виду мобильного приложения
YCLIENTS (полноценный месячный календарь + слоты, сгруппированные по периодам
суток), сохранив при этом нашу подпись кнопки действия («Записаться») и убрав
стороннее брендирование «Работает на yclients». Это повышает узнаваемость,
удобство выбора даты в пределах месяца и общее качество восприятия записи.

**Заказчик / стейкхолдеры:** владелец студии (как заказчик PWA), клиенты студии
(конечные пользователи онлайн-записи).

## 2. Область (scope)

### В рамках задачи
- Замена горизонтальной ленты дат на полноценный раскрывающийся месячный
  календарь: заголовок месяца, стрелки навигации «‹ ›», строка дней недели,
  сетка чисел месяца.
- Реальная подгрузка соседних месяцев при навигации стрелками (дополнительные
  запросы `book_dates`).
- Визуальные состояния дат: выбранная — тёмный заполненный кружок; «сегодня» —
  подчёркнуто; недоступные/прошедшие — приглушённые и некликабельные; доступные
  (из `book_dates`) — активные.
- Группировка слотов по периодам суток (Утро `< 12:00` / День `12:00–16:59` /
  Вечер `≥ 17:00`) со сворачиваемыми секциями (шеврон).
- Выбранный слот — тёмная заливка.
- Удаление карточки мастера (`#masterStrip`) с этого экрана.
- Изменение подписи нижней кнопки на «Записаться» (и сопутствующих текстов
  «Записываем…» / сброса состояния).
- Удаление надписи «Работает на yclients».
- Сохранение текущего потока бронирования и перехода на экран подтверждения
  `s-confirm` («Запись подтверждена!»).

### Вне рамок (explicitly out of scope)
- Изменение логики самого бронирования (`startBooking`), экрана подтверждения
  `s-confirm`, структуры `state` (`slot` / `dateISO` / `dateFull`).
- Добавление отдельного промежуточного шага подтверждения между выбором слота и
  экраном `s-confirm`.
- Изменение потока выбора услуги и мастера (предшествующие экраны).
- Изменение серверной части / YCLIENTS API и набора используемых эндпоинтов.
- Запоминание состояния раскрытия секций/календаря между заходами на экран.
- Расширение горизонта доступных дат за пределы, который отдаёт API.

## 3. Действующие лица и сценарии использования

**Клиент студии** — пользователь PWA, который записывается на услугу.

Основной сценарий:
1. Клиент дошёл до экрана выбора времени (услуга и мастер уже выбраны на
   предыдущих шагах).
2. Видит календарь-карточку текущего (или ближайшего с доступными датами)
   месяца. Первая доступная дата автоматически выбрана.
3. При необходимости листает месяцы стрелками «‹ ›» — подгружаются доступные
   даты соседнего месяца.
4. Выбирает доступную дату — загружаются слоты на эту дату, сгруппированные по
   периодам суток.
5. При необходимости сворачивает/разворачивает секции периодов суток.
6. Выбирает слот (выделяется тёмной заливкой). В инфо-строке отображается
   контекст: дата · слот · услуга.
7. Нажимает «Записаться» — система бронирует через API и показывает экран
   «Запись подтверждена!» (`s-confirm`).

Альтернативный сценарий (перенос записи): пользователь переносит существующую
запись (`state._rescheduleId`), кнопка ведёт на экран «Запись перенесена!».

## 4. Функциональные требования

- **ФТ-1.** Экран `s-slots` отображает дату в виде месячного календаря-карточки:
  заголовок месяца, стрелки навигации «‹ ›», строка дней недели «Пн…Вс», сетка
  чисел месяца. Горизонтальная лента дат удалена.
- **ФТ-2.** Стрелка «вперёд» / «назад» переключает отображаемый месяц и реально
  подгружает доступные даты этого месяца через `book_dates`.
- **ФТ-3.** Каждое число месяца имеет одно из состояний: доступное
  (кликабельное), недоступное/прошедшее (приглушённое, некликабельное),
  выбранное (тёмный заполненный кружок), «сегодня» (подчёркнуто). «Сегодня» и
  «выбранное» могут совпадать.
- **ФТ-4.** Выбор доступной даты перезагружает слоты на эту дату
  (`book_times`).
- **ФТ-5.** При открытии экрана автоматически выбирается первая доступная дата и
  первый доступный слот (сохранение текущего поведения).
- **ФТ-6.** Слоты времени сгруппированы по периодам суток: Утро `< 12:00`,
  День `12:00–16:59`, Вечер `≥ 17:00`. Группировка определяется временем слота.
- **ФТ-7.** Пустые секции периодов суток скрываются. Непустые секции по
  умолчанию раскрыты.
- **ФТ-8.** Каждая непустая секция имеет заголовок с шевроном, по нажатию на
  который секция сворачивается/разворачивается. Состояние раскрытия живёт только
  в рамках текущего сеанса экрана и не сохраняется между заходами.
- **ФТ-9.** Выбранный слот отображается тёмной заливкой.
- **ФТ-10.** Карточка мастера (`#masterStrip`) на экране `s-slots` отсутствует.
- **ФТ-11.** Нижняя кнопка действия подписана «Записаться». По нажатию система
  бронирует запись через API и переходит на экран `s-confirm` («Запись
  подтверждена!»). Промежуточный шаг подтверждения не добавляется.
- **ФТ-12.** Сопутствующие текстовые состояния кнопки/процесса («Записываем…» и
  сброс) приведены в соответствие с новой подписью.
- **ФТ-13.** Надпись «Работает на yclients» на экране отсутствует.
- **ФТ-14.** Инфо-строка над кнопкой (дата · слот · услуга) сохраняется как
  контекст перед бронированием.
- **ФТ-15.** Состояния загрузки и ошибок сохраняются: загрузка дат/времени,
  фолбэк при отсутствии данных от API, состояние «нет окон», ошибка сети.
- **ФТ-16.** Для сценария переноса записи (`state._rescheduleId`) сохраняется
  корректный переход на экран «Запись перенесена!»; подпись кнопки переноса —
  по решению из раздела 8.

## 5. Нефункциональные требования

- **НФТ-1 (Безопасность).** Сохранить санитизацию времени до формата `HH:MM`
  перед вставкой в обработчик `onclick`. Любой вставляемый в разметку текст
  экранируется (`esc()`).
- **НФТ-2 (Совместимость / тема).** Экран корректно отображается в светлой и
  тёмной теме; цвета берутся из токенов `var(--…)`, без хардкода. «Тёмная
  заливка» выбранного состояния опирается на `var(--primary)` /
  `var(--on-primary)`.
- **НФТ-3 (Отзывчивость UI).** Сохранить тактильный отклик (`hapticTap`) при
  выборе даты и слота.
- **НФТ-4 (Технологические ограничения).** Реализация только на vanilla
  JS/HTML/CSS, без рантайм-зависимостей; затрагиваемые файлы — `docs/prototype.html`,
  `docs/modules/slots.js`, `docs/style.css`.
- **НФТ-5 (PWA).** Установка PWA не должна сломаться; при изменении `APP_SHELL`
  или `sw.js` обязательно поднять `CACHE_VERSION`.
- **НФТ-6 (Качество).** Ноль ошибок в консоли браузера на основном сценарии.

## 6. Ограничения и допущения

Ограничения:
- Используемые эндпоинты неизменны: `GET /book_dates/{company}` и
  `GET /book_times/{company}/{staffId}/{iso}`.
- Минимальные изменения строго по объёму требования; поток бронирования
  (`startBooking`, `s-confirm`, `state.slot/dateISO/dateFull`, перенос
  `_rescheduleId`) не трогать сверх необходимого.
- Деплой — статический хостинг GitHub Pages из `docs/`.

Допущения:
- Горизонт доступных для записи дат, отдаваемый API, ограничен (ориентировочно
  около месяца вперёд); навигация стрелками должна корректно вести себя на
  границах этого горизонта (детализация — раздел 8).
- Формат ответа `book_times` (`r.data` — массив `{time|datetime}`) и
  `book_dates` остаётся прежним; группировка по периодам суток выполняется на
  клиенте по значению времени.
- Референсом целевого вида служит скриншот `IMG_20260530_230100.png`,
  приложенный к исходному требованию.

## 7. Критерии приёмки

- **КП-1.** На экране `s-slots` дата представлена месячным календарём-карточкой
  (заголовок месяца, стрелки «‹ ›», дни недели, сетка чисел); горизонтальной
  ленты дат нет. (ФТ-1)
- **КП-2.** Стрелки «‹ ›» переключают месяц и подгружают доступные даты соседнего
  месяца через `book_dates`. (ФТ-2)
- **КП-3.** Выбранная дата показана тёмным кружком, «сегодня» подчёркнуто,
  недоступные/прошедшие даты приглушены и некликабельны, доступные —
  кликабельны. (ФТ-3)
- **КП-4.** Выбор доступной даты перезагружает слоты этой даты. (ФТ-4)
- **КП-5.** При открытии экрана автоматически выбраны первая доступная дата и
  первый слот. (ФТ-5)
- **КП-6.** Слоты разбиты по периодам: Утро `< 12:00`, День `12:00–16:59`,
  Вечер `≥ 17:00`. (ФТ-6)
- **КП-7.** Пустые секции скрыты, непустые раскрыты по умолчанию; шеврон
  сворачивает/разворачивает секцию; состояние не сохраняется между заходами.
  (ФТ-7, ФТ-8)
- **КП-8.** Выбранный слот отображается тёмной заливкой. (ФТ-9)
- **КП-9.** Карточка мастера `#masterStrip` на экране отсутствует. (ФТ-10)
- **КП-10.** Нижняя кнопка подписана «Записаться»; нажатие бронирует через API и
  ведёт на `s-confirm` «Запись подтверждена!»; промежуточного шага нет. (ФТ-11,
  ФТ-12)
- **КП-11.** Надпись «Работает на yclients» отсутствует. (ФТ-13)
- **КП-12.** Инфо-строка (дата · слот · услуга) присутствует над кнопкой. (ФТ-14)
- **КП-13.** Работают загрузка дат/времени, фолбэки и состояния «нет окон» /
  ошибка сети; время санитизировано до `HH:MM`. (ФТ-15, НФТ-1)
- **КП-14.** Сценарий переноса записи ведёт на «Запись перенесена!»; подпись
  кнопки соответствует согласованному в разделе 8 решению. (ФТ-16)
- **КП-15.** Экран корректен в светлой и тёмной теме (токены, не хардкод);
  работает тактильный отклик при выборе; ноль ошибок в консоли; PWA installable
  не сломан; при изменении `sw.js` поднята `CACHE_VERSION`. (НФТ-2, НФТ-3,
  НФТ-5, НФТ-6)

## 8. Открытые вопросы

Блокирующих вопросов нет. Ниже — три некритичных вопроса; по каждому
зафиксирована **рекомендация-дефолт**, которая требует подтверждения заказчиком
или архитектором (но не блокирует начало проектирования).

- **ОВ-1. Подпись кнопки при переносе записи (`state._rescheduleId`).**
  *Рекомендация (требует подтверждения):* для сценария переноса оставить подпись
  «Перенести запись» — семантика отличается от первичной записи, успех ведёт на
  «Запись перенесена!». На «Записаться» меняется только подпись в обычном
  сценарии записи.

- **ОВ-2. Поведение стрелок навигации на границах горизонта дат.**
  Что делать, когда `book_dates` за выбранный месяц вернул пусто или достигнут
  горизонт записи. *Рекомендация (требует подтверждения):* стрелка «назад»
  неактивна на текущем месяце (не уходим в прошлое); стрелка «вперёд» неактивна,
  когда соседний месяц вне горизонта доступных дат. Месяц без доступных дат, в
  который пользователь всё же попал, показывать с приглушёнными числами и
  нейтральной подсказкой об отсутствии свободных дат.

- **ОВ-3. Анимация раскрытия секций и календаря.**
  *Рекомендация (требует подтверждения):* лёгкая анимация раскрытия/сворачивания
  по аналогии с существующими паттернами интерфейса (например, инбокс); при
  технических сложностях допустимо мгновенное переключение без анимации.
