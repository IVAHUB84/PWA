# BA-требования 006: Надёжная отправка push-уведомлений клиентам из админ-панели

- **Источник:** `requirements/req-admin-push-006.md`
- **Дата:** 2026-05-28
- **Статус:** черновик

## 1. Контекст и бизнес-цель

Studio PWA (студия «Реснички, сестрички», Троицк) — устанавливаемое PWA онлайн-записи поверх YCLIENTS, отдаётся статически из `docs/` на GitHub Pages. Push-инфраструктура уже существует: клиентский модуль `docs/modules/push.js`, админ-логика рассылок `docs/modules/admin.js`, UI в `docs/prototype.html`, бэкенд — Cloudflare Worker `cf-worker/src/index.js` (эндпоинты `/subscribe`, `/unsubscribe`, `/send` с `X-Admin-Secret`, cron-напоминания о записи на завтра, KV `SUBSCRIPTIONS`). Worker задеплоен на https://studio-push.ivahub84.workers.dev (на момент написания `/vapid-public-key` → 200, `/send` без секрета → 401).

Функционал рассылок «частично реализован, но фактически не работает надёжно». Эта задача — **полноценный релиз с исправлением дефектов**, а не документирование текущего поведения. Цель: администратор студии гарантированно доставляет push-уведомление выбранной аудитории подписанных клиентов из админ-панели и видит достоверный результат доставки.

**Подтверждённое кодом текущее состояние** (фактическая ревизия, не предположения):

- **Сегменты аудитории фиктивны.** UI даёт выбрать сегмент («Все клиенты» / «Активные за 30 дней» / «С записями»), счётчики получателей считаются на клиенте из записей YCLIENTS (`updatePushAudience`, `admin.js`). Но `sendNewPush` (`admin.js`) вызывает `sendAdminPush(title, text)` **без передачи сегмента**, а Worker `/send` (`cf-worker/src/index.js`) умеет либо отправить одному `targetClientId`, либо **всем подписчикам** из KV — фильтрации по сегменту нет. Реально рассылка всегда уходит всем.
- **Планирование фиктивно.** При выборе «Запланировать» (`sendNewPush`, ветка `isScheduled`) кампания только сохраняется в `localStorage` (`yc_push_campaigns`) и показывается тост «Рассылка запланирована ✓». Серверного планировщика произвольных рассылок нет: cron Worker (`scheduled` → `handleCron`) шлёт только напоминания о завтрашних записях. Запланированная кампания **никогда не отправляется**.
- **Отправка одному клиенту не выведена в UI.** Worker и `sendAdminPush` поддерживают `targetClientId`, но в списке клиентов админки (`renderAdminClients` / `_renderClientRows`) нет кнопки «отправить этому клиенту».
- **Подписка клиентов под вопросом.** `subscribePush` для обычного клиента вызывается при логине (`app.js`, `auth.js`), но фактически зависит от наличия Worker URL в `localStorage` клиентского устройства, поддержки PushManager и предоставленного разрешения на уведомления. Если подписок нет — у рассылки нет получателей и результат «Отправлено: 0».
- **Конфигурация ручная и хрупкая.** Отправка работает только при заданных в админке Worker URL (`yc_worker_url`) и Admin Secret (`yc_push_secret`) в `localStorage` (поля в `docs/prototype.html`, кнопка «Подписать этот браузер» → `adminSubscribeSelf`). Без них `sendAdminPush` возвращает `no_config` и ничего не отправляется; администратору это явно не объяснено.
- **Недостоверная обратная связь.** Для немедленной отправки тост уже показывает реальный `res.sent` от Worker — это работает. Но: ветка планирования показывает «успех» без отправки; вдобавок существует неиспользуемая функция-дубликат `sendPush()` (`admin.js`), показывающая «Рассылка отправлена ✓» безусловно — потенциальный источник ложной обратной связи.

**Стейкхолдеры:**
- Заказчик / владелец студии — администратор, отправляющий рассылки; заинтересован в гарантированной доставке и достоверной статистике.
- Клиенты студии — получатели push (подписанные на уведомления).
- Бизнес-аналитик / архитектор / разработчик — оформление, проектирование, реализация.

## 2. Область (scope)

### В рамках задачи
- Достоверная **отправка push выбранной аудитории**: сегмент, выбранный администратором в UI, должен фактически определять круг получателей (а не уходить всем).
- **Сегменты аудитории**, заявленные в требовании: «Все клиенты (все подписчики)», «Активные за последние 30 дней», «Клиенты с предстоящими записями». Каждый сегмент должен реально фильтровать получателей; счётчик получателей в UI должен соответствовать тому, кому уйдёт рассылка.
- **Отправка «сейчас»** — немедленная доставка подписанным клиентам выбранного сегмента.
- **Планирование («Запланировать»)** — запланированная на заданное время рассылка должна фактически отправиться в назначенное время (не только сохраниться локально).
- **Достоверный результат доставки**: сколько реально доставлено, сколько не удалось; обратная связь администратору отражает фактический ответ бэкенда, а не безусловный «успех».
- **История рассылок** — отображение отправленных и запланированных кампаний; запланированные, которые уже ушли, корректно отражают свой статус.
- **Отправка push конкретному клиенту** из списка клиентов админки (требование помечает как «желательно» — приоритет уточняется, см. «Открытые вопросы»).
- **Надёжность и понятность конфигурации**: администратор должен понимать, что без корректно заданных Worker URL и Admin Secret рассылка не работает; некорректная/отсутствующая конфигурация не должна давать ложноположительный результат.
- **Проверка сквозной подписки клиентов**: убедиться, что обычные клиенты реально становятся подписчиками (иначе у рассылки нет получателей).
- **Сквозная проверка на реальном устройстве**: выбор аудитории → отправка → фактическое получение push подписанными клиентами этого сегмента.

### Вне рамок (explicitly out of scope)
- Изменение механизма cron-напоминаний о записи на завтра (`handleCron`) — отдельный действующий функционал; затрагивается только если планировщик произвольных рассылок будет переиспользовать инфраструктуру (решение архитектора).
- Предложение установки PWA и логика установки (`docs/modules/install.js`, ADR-005) — смежная, но отдельная тема; не затрагивается, кроме связи «установка/подписка — предпосылка наличия получателей».
- Сценарий онлайн-записи (услуга → мастер → слот → подтверждение), отзывы, лента публикаций — не затрагиваются.
- Изменение схемы хранения данных YCLIENTS и состава токенов; перенос захардкоженного YCLIENTS-токена из `prototype.html` (известный техдолг — «не усугублять», но и не предмет этой задачи).
- Аналитика рассылок сверх счётчиков доставлено/не доставлено (открываемость push, CTR, A/B тексты) — возможная отдельная задача.
- Редактирование/повторная отправка ранее отправленной кампании, RFM-сегментация, произвольные пользовательские сегменты сверх трёх заявленных.
- Двухсторонняя синхронизация истории рассылок между устройствами администраторов (история сейчас локальна в `localStorage`; единый бэкенд-журнал — предмет «Открытых вопросов», не обязателен по требованию).

## 3. Действующие лица и сценарии использования

**Администратор студии** — авторизован в админ-панели PWA, имеет настроенные Worker URL и Admin Secret.

Основной сценарий (немедленная рассылка сегменту):
1. Администратор открывает экран рассылок, нажимает «Новая».
2. Выбирает шаблон уведомления или вводит свой текст; видит превью.
3. Выбирает сегмент аудитории; видит достоверное число получателей в этом сегменте.
4. Выбирает «Отправить сейчас» и подтверждает.
5. Система отправляет push **только** подписанным клиентам выбранного сегмента и возвращает фактический результат: доставлено N, не доставлено M.
6. Администратор видит достоверный результат; кампания появляется в истории как отправленная.

Сценарий (запланированная рассылка):
1. Администратор так же выбирает текст, сегмент, затем «Запланировать» и задаёт время.
2. Кампания сохраняется как запланированная и отображается в истории в разделе запланированных.
3. В назначенное время рассылка **фактически** отправляется подписанным клиентам выбранного сегмента.
4. Статус кампании в истории отражает факт отправки/результат (а не остаётся «вечно запланированной»).

Сценарий (push конкретному клиенту — приоритет уточняется):
1. Администратор открывает список клиентов, находит клиента.
2. Инициирует отправку push этому клиенту (с текстом/шаблоном).
3. Если клиент подписан — push доставляется и администратор видит результат; если не подписан — система понятно сообщает, что доставка невозможна (нет подписки), без ложного «успеха».

Сценарий (клиент-получатель):
1. Клиент авторизуется в приложении (на устройстве с поддержкой push).
2. Клиент становится подписчиком (предоставляет разрешение на уведомления), его подписка регистрируется в бэкенде.
3. При рассылке, охватывающей сегмент клиента, он получает push на устройстве.

Граничные сценарии:
- **Нет конфигурации** (не задан Worker URL или Admin Secret) — отправка не выполняется; администратору явно сообщается причина (не «успех»).
- **Нет подписчиков в сегменте** — результат честно показывает «доставлено 0»; администратор понимает, что получателей нет.
- **Часть доставок не удалась** (устаревшие/отозванные подписки) — результат отражает доставлено/не доставлено; недоставленные по причине устаревшей подписки обрабатываются без ложного успеха.

## 4. Функциональные требования

- **ФТ-1.** Администратор может составить уведомление: выбрать шаблон **или** ввести произвольный текст, с превью того, как уведомление будет выглядеть у клиента.
- **ФТ-2.** Администратор может выбрать сегмент аудитории из трёх: «Все клиенты (все подписчики)», «Активные за последние 30 дней», «Клиенты с предстоящими записями».
- **ФТ-3.** Перед отправкой администратор видит число получателей выбранного сегмента, и это число соответствует фактическому кругу, которому будет доставлена рассылка (с учётом того, что фактически доставка идёт только **подписанным** клиентам сегмента — см. ФТ-5, НФТ-4).
- **ФТ-4.** При отправке рассылка доставляется **только** подписанным клиентам выбранного сегмента, а не всем подписчикам без разбора. Выбор сегмента в UI фактически влияет на круг получателей.
- **ФТ-5.** Администратор может отправить рассылку немедленно («Отправить сейчас»): подписанные клиенты выбранного сегмента получают push.
- **ФТ-6.** Администратор может запланировать рассылку на заданное время. Запланированная рассылка **фактически отправляется** в назначенное время подписанным клиентам выбранного сегмента (а не только сохраняется локально).
- **ФТ-7.** После отправки (немедленной или по расписанию) система предоставляет достоверный результат: количество фактически доставленных и количество недоставленных уведомлений, основанный на ответе бэкенда, а не на безусловном «успехе».
- **ФТ-8.** История рассылок отображает отправленные и запланированные кампании; запланированная кампания после фактической отправки корректно меняет статус/отражает результат и не остаётся бессрочно «запланированной».
- **ФТ-9.** Администратор может отправить push конкретному клиенту из списка клиентов админки. *(Приоритет — «желательно»; обязательность для этого релиза уточняется, см. «Открытые вопросы».)* При отсутствии подписки у клиента система понятно сообщает о невозможности доставки, без ложного «успеха».
- **ФТ-10.** Если обязательная конфигурация отправки (Worker URL и/или Admin Secret) отсутствует или некорректна, отправка не выполняется, и администратору явно сообщается причина; ложноположительный результат («успех») не показывается.
- **ФТ-11.** Обычный клиент приложения, давший разрешение на уведомления на поддерживаемом устройстве, фактически регистрируется как подписчик и потенциально является получателем рассылок (сквозной flow подписки работоспособен).
- **ФТ-12.** Уведомления и пользовательские тексты администратора и клиента отображаются на русском языке.

## 5. Нефункциональные требования

- **НФТ-1 (Совместимость стека).** Решение остаётся в рамках стека: vanilla JS/HTML/CSS, статический хостинг из `docs/` на GitHub Pages, бэкенд — существующий Cloudflare Worker; без новых рантайм-npm-зависимостей в приложении.
- **НФТ-2 (Безопасность секретов).** Admin Secret, VAPID private key, YCLIENTS-токены остаются **только** в Worker / в локальной конфигурации администратора, не в репозитории. Эндпоинт `/send` остаётся защищён `X-Admin-Secret` (без секрета → 401). Никакие секреты не коммитятся.
- **НФТ-3 (Достоверность обратной связи).** Любой показанный администратору результат отправки должен отражать фактический ответ бэкенда; ложноположительные сообщения недопустимы (поддерживает ФТ-7, ФТ-10).
- **НФТ-4 (Целостность сегментации).** Показанное число получателей и фактический круг доставки должны быть согласованы и объяснимы (например, расхождение «клиентов в сегменте» и «подписанных клиентов в сегменте» должно быть понятно администратору, а не выглядеть как сбой). Точная семантика — предмет проектирования и «Открытых вопросов».
- **НФТ-5 (Надёжность доставки по расписанию).** Механизм запланированной отправки должен срабатывать без присутствия администратора онлайн (т.е. не зависеть от того, открыто ли приложение администратора в назначенное время). Допустимая точность времени срабатывания — предмет «Открытых вопросов».
- **НФТ-6 (Обработка устаревших подписок).** Недоставка из-за отозванной/устаревшей подписки не должна засчитываться как доставка и не должна ломать рассылку остальным; такие подписки корректно учитываются как недоставленные.
- **НФТ-7 (PWA/офлайн не регрессируют).** Изменения не должны ломать устанавливаемость PWA и офлайн-поведение Service Worker; при изменении состава `APP_SHELL` / кэшируемых ресурсов — поднятие `CACHE_VERSION` в `docs/sw.js` (по правилу проекта).
- **НФТ-8 (Локализация).** Все пользовательские тексты (UI администратора, тосты, тела push) — на русском языке, в едином тоне приложения.
- **НФТ-9 (Ограничение среды push).** Push работает только по HTTPS (GitHub Pages) или localhost — это учитывается при тестировании сквозного сценария на реальном устройстве.

## 6. Ограничения и допущения

**Ограничения:**
- Приложение — статический PWA в `docs/` на GitHub Pages; единственный управляемый бэкенд — Cloudflare Worker (`cf-worker/src/index.js`), деплоится отдельно от PWA (`npx wrangler deploy`).
- Подписки хранятся в KV `SUBSCRIPTIONS` по ключу `sub:<clientId>` и не содержат истории визитов/записей клиента; данные о визитах и записях доступны через YCLIENTS API (который Worker уже использует в cron).
- Push-доставка идёт только подписанным клиентам; неподписанный клиент получателем быть не может, какой бы сегмент ни был выбран.
- Push работает только по HTTPS или localhost; на iOS — только для приложения, установленного на экран «Домой» (ограничение платформы).
- Конфигурация отправки (Worker URL, Admin Secret) хранится в `localStorage` устройства администратора; при смене устройства/браузера её нужно задать заново.
- При изменении `APP_SHELL` / `sw.js` требуется поднятие `CACHE_VERSION`. При изменении Worker — отдельный деплой Worker.
- Запрещены рантайм-npm-зависимости в приложении (только devDependencies); запрещён коммит секретов; запрещены операции с веткой `backup`.

**Допущения:**
- Криптография Web Push (RFC 8291, VAPID) в Worker (`encryptPayload`, `vapidToken`, `sendOne`) работает корректно — это не предмет ревизии, доработка касается маршрутизации/сегментации/планирования, а не шифрования.
- Worker остаётся источником истины о подписчиках (KV `SUBSCRIPTIONS`); сегментация по визитам/записям, если потребуется на стороне Worker, опирается на те же данные YCLIENTS, что и cron.
- Существующий вызов `subscribePush` при логине клиента (`app.js`, `auth.js`) — рабочая основа подписки; задача — проверить и довести его до надёжного срабатывания, а не вводить подписку с нуля.
- Немедленная отправка уже частично достоверна (тост показывает реальный `res.sent`); доработка достоверности касается прежде всего планирования и краевых случаев конфигурации/нуля подписчиков.
- Тексты/шаблоны уведомлений (`PUSH_TEMPLATES`) — приемлемая отправная точка; финальные формулировки может скорректировать заказчик.
- История рассылок может оставаться локальной (`localStorage`) для этого релиза, если заказчик не потребует единый журнал — см. «Открытые вопросы».

## 7. Критерии приёмки

- **КП-1.** Администратор может выбрать шаблон или ввести свой текст и увидеть превью уведомления. (ФТ-1)
- **КП-2.** Доступны три сегмента: «Все клиенты», «Активные за 30 дней», «С предстоящими записями», и для выбранного сегмента отображается число получателей. (ФТ-2, ФТ-3)
- **КП-3.** При отправке сегменту, отличному от «Все», рассылка доставляется **только** подписанным клиентам этого сегмента; контрольная проверка показывает, что клиент вне сегмента push не получает. (ФТ-4, НФТ-4)
- **КП-4.** «Отправить сейчас» доставляет push на реальное устройство подписанного клиента выбранного сегмента. (ФТ-5, НФТ-9)
- **КП-5.** Запланированная на ближайшее время рассылка фактически приходит подписанным клиентам сегмента в назначенное время **без** того, чтобы приложение администратора было открыто. (ФТ-6, НФТ-5)
- **КП-6.** После отправки администратор видит достоверный результат: число доставленных и недоставленных, соответствующее ответу Worker (для нулевого числа подписчиков — честное «доставлено 0»). (ФТ-7, НФТ-3)
- **КП-7.** История рассылок показывает отправленные и запланированные кампании; после фактической отправки запланированная кампания меняет статус и не остаётся бессрочно «запланированной». (ФТ-8)
- **КП-8.** При отсутствии/некорректности Worker URL или Admin Secret отправка не выполняется и администратор видит понятную причину, а не сообщение об успехе. (ФТ-10, НФТ-3)
- **КП-9.** Из списка клиентов администратор может отправить push конкретному клиенту; подписанный получает уведомление, для неподписанного система честно сообщает о невозможности доставки. *(При подтверждении, что ФТ-9 в объёме релиза.)* (ФТ-9)
- **КП-10.** Сквозной flow подписки работоспособен: вновь авторизованный клиент на поддерживаемом устройстве, давший разрешение, появляется среди подписчиков и получает адресованную ему рассылку. (ФТ-11)
- **КП-11.** Недоставка из-за отозванной/устаревшей подписки не засчитывается как доставка и не прерывает рассылку остальным получателям. (НФТ-6)
- **КП-12.** Все пользовательские тексты (UI, тосты, тела push) — на русском языке. (ФТ-12, НФТ-8)
- **КП-13.** PWA остаётся устанавливаемой, офлайн-поведение не регрессирует; при изменении `APP_SHELL` поднята `CACHE_VERSION`; новые рантайм-зависимости в приложении не добавлены; секреты не закоммичены. (НФТ-1, НФТ-2, НФТ-7)

## 8. Открытые вопросы

1. **Семантика сегмента «С предстоящими записями» и согласование со списком подписчиков.** Сегменты «Активные за 30 дней» и «С записями» определяются по данным YCLIENTS (визиты/записи), а доставка возможна только подписанным клиентам (KV `SUBSCRIPTIONS`). Нужно подтвердить ожидаемую семантику: число получателей сегмента = «клиенты сегмента, которые подписаны»? Или показывать «в сегменте N, из них подписаны M»? *Влияет на ФТ-3, ФТ-4, НФТ-4. Реализацию определяет архитектор; продуктовое намерение нужно подтвердить заказчику.*

2. **Источник данных для серверной сегментации.** Сейчас счётчики сегментов считаются на клиенте админки из YCLIENTS, а Worker о визитах/записях ничего не знает (хранит только подписки). Чтобы рассылка фактически фильтровалась по сегменту, бэкенду нужны данные о принадлежности клиента к сегменту. Допустимо ли, чтобы Worker сам обращался к YCLIENTS для определения сегмента (как в cron), или сегмент-список клиентов формируется на стороне админки и передаётся в `/send`? *Архитектурный вопрос; для BA важно подтвердить, что заказчик согласен на любой из подходов при сохранении достоверности. Не блокирует оформление BA, но блокирует переход к ADR без ответа на вопрос №1.*

3. **Механизм и точность планирования.** Требуется отправка по расписанию без участия администратора (НФТ-5). Текущий cron Worker рассчитан на ежедневные напоминания о завтрашних записях. Какая допустимая точность срабатывания запланированной рассылки (с точностью до минуты / до часа / «в течение N минут после назначенного времени»)? Где хранятся запланированные кампании, чтобы пережить закрытие приложения администратора? *Влияет на ФТ-6, ФТ-8, НФТ-5. Конкретику решает архитектор, но требование к точности должен задать заказчик.*

4. **Приоритет «push конкретному клиенту» (ФТ-9).** В требовании помечено как «желательно». Включаем в объём этого релиза или выносим в следующий? *Влияет на ФТ-9, КП-9.*

5. **Хранение истории рассылок.** История сейчас локальна (`localStorage`, ключ `yc_push_campaigns`) и не синхронизируется между устройствами/администраторами. Достаточно ли локальной истории для этого релиза, или нужен единый серверный журнал рассылок (доставлено/не доставлено, статус запланированных)? *Влияет на ФТ-8; единый журнал — потенциально вне рамок, решение заказчика стоит зафиксировать.*

6. **Подтверждение перед массовой отправкой.** Нужно ли явное подтверждение администратору («Отправить N клиентам?») перед немедленной рассылкой большому сегменту, чтобы избежать случайной отправки всем? *Влияет на ФТ-5; не критично, но определяет UX-защиту от ошибок.*

7. **Поведение «нет подписчиков».** При выборе сегмента, в котором подписанных клиентов нет, как себя ведёт кнопка отправки — блокируется заранее или отправка проходит с результатом «доставлено 0»? *Влияет на ФТ-3, ФТ-7, КП-6.*

Противоречий с принятыми ADR не выявлено. ADR-005 (единый механизм install prompt) явно относит push-логику (`docs/modules/push.js`, Worker `/subscribe`/`/send`/cron) к **вне рамок** того релиза и не меняет её — настоящая задача дорабатывает именно push-логику и с ADR-005 не конфликтует (затрагиваются разные подсистемы; при изменении `APP_SHELL` обе соблюдают правило `CACHE_VERSION`). ADR-001–ADR-004 (каталог, фильтрация мастеров, отзывы) данной темы не затрагивают.
