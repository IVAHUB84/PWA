# BA-требования 018: Оформление рамок плашек в каталоге услуг и слайдов карусели детальной карточки

- **Источник:** `requirements/req-service-card-borders-018.md`
- **Дата:** 2026-05-29
- **Статус:** согласовано (визуальный стиль утверждён заказчиком 2026-05-29; конкретные значения и решения по ОВ-1…ОВ-5 — в §8)

## 1. Контекст и бизнес-цель

Studio PWA (студия красоты, онлайн-запись поверх YCLIENTS) — устанавливаемое статическое приложение на vanilla JS/HTML/CSS, отдаётся из `docs/` на GitHub Pages, без рантайм-зависимостей. Раздел услуг состоит из двух экранов: каталог услуг `s-services` (сетка карточек `.svc-catalog-card`) и детальная карточка услуги `s-service`, где фото работ показываются каруселью (слайды `.svc-carousel-slide`).

Текущее визуальное оформление границ в этих двух местах разное:

- **Карточка каталога** `.svc-catalog-card` (`docs/style.css:273-282`) имеет границу: `border: 1px solid var(--border)`, `border-radius: 16px`, тень. При этом в блоке «CARD POLISH» (`docs/style.css:693`) тень ослаблена до `box-shadow: 0 1px 4px rgba(0,0,0,0.05)` — итоговый край = тонкий бордер `--border` + очень лёгкая тень.
- **Слайд карусели** `.svc-carousel-slide` (`docs/style.css:834-838`) рамки как таковой не имеет: только `border-radius: 16px` + `overflow: hidden`, без бордера и тени — край фото просто скруглён.

Заказчик субъективно оценивает текущие края как «плоские/невыразительные» и хочет «красивые рамки у плашек» — визуально аккуратные, выразительные границы карточек каталога и слайдов карусели.

**Бизнес-цель:** повысить визуальную аккуратность и воспринимаемое качество раздела услуг за счёт более выразительного и единообразного оформления границ карточек каталога и слайдов карусели, в согласованном с заказчиком стиле, корректно работающем в светлой и тёмной теме, без нарушения уже принятой геометрии карусели и обрезки фото. Это чисто визуальная (UI) доработка, а не функциональное изменение.

**Стейкхолдеры:**

- Заказчик / владелец студии — носитель эстетического ожидания «красиво»; именно он задаёт визуальный ориентир (референс/значения), без которого требование не формализуемо.
- Клиент студии — пользователь PWA, воспринимает аккуратность интерфейса; прямого нового сценария не получает.
- Бизнес-аналитик / архитектор / разработчик — оформление, выбор конкретных значений стиля, реализация.

## 2. Область (scope)

### В рамках задачи

- Изменение **визуального оформления границ** карточек каталога услуг `.svc-catalog-card` (экран `s-services`).
- Изменение **визуального оформления границ** слайдов карусели `.svc-carousel-slide` на детальной карточке услуги `s-service`.
- Достижение **единообразного** (согласованного между каталогом и каруселью) и **выразительного** оформления края, корректного в **светлой и тёмной** теме.
- Сохранение всех существующих функциональных свойств этих элементов: обрезка фото (`overflow: hidden` + скругление), `aspect-ratio` обложки каталога, высота и `object-fit: cover` фото, геометрия карусели из BA-017 (равномерный peek, выравнивание слайдов), состояние нажатия `:active` карточки каталога.

### Вне рамок (explicitly out of scope)

- **Любые функциональные изменения** механики карусели, логики листания, источника фото, кэширования (BA-015 / BA-017 / ADR-013 / ADR-014) — задача только про оформление границ.
- **Изменение геометрии карусели** (ширина слайда, gap, peek, высота фото) — кроме случая, когда выбранное оформление рамки физически требует микрокоррекции (например, чтобы бордер не съедал peek); такое решение принимает архитектор, сохраняя равномерность peek из BA-017.
- **Изменение оформления других карточек приложения**: карточки мастеров `.master-card`, истории визитов `.hist-visit`, ленты/постов, hero-карточек, confirm/cs-карточек и пр. (хотя у них в блоке «CARD POLISH» похожая тень — они вне объёма).
- **Изменение одиночного фото / категорийного плейсхолдера детальной карточки** (`.svc-card-single-img`, `.svc-card-placeholder`, `docs/style.css:812-820`) — по умолчанию вне объёма; включается в scope только при явном указании заказчика (см. §8 ОВ-2).
- **Изменение содержимого карточек** (название, цена, длительность, описание, обложки как таковые) — не затрагивается.
- **Редизайн раздела услуг** сверх оформления границ.

## 3. Действующие лица и сценарии использования

**Клиент студии** — пользователь PWA (преимущественно мобильный, тач). Новый функциональный сценарий не появляется; меняется только воспринимаемый внешний вид.

Наблюдаемый результат (целевой):

1. Клиент открывает раздел услуг и видит каталог: карточки услуг имеют аккуратную, выразительную рамку в согласованном стиле; край карточки читается чётко, не «плоско».
2. Клиент открывает детальную карточку услуги с несколькими фото: слайды карусели имеют рамку, единообразную по стилю с карточками каталога; текущий и выглядывающие соседние слайды (peek) сохраняют корректное скругление и обрезку фото.
3. При переключении системной/приложенческой темы (светлая ↔ тёмная) рамки в обоих местах остаются аккуратными и достаточно контрастными — не «теряются» на фоне и не выглядят грубо.

Граничные сценарии (поведение визуально сохраняется):

- **Нажатие на карточку каталога** — анимация `:active` (`transform: scale(0.97)`) сохраняется и не конфликтует с новой рамкой.
- **Слайд с битым фото** (плейсхолдер вместо картинки) — рамка/скругление слайда не ломают деградацию из BA-015/BA-017.
- **1 фото / 0 фото на детальной карточке** — поведение одиночного фото и плейсхолдера не изменяется (если ОВ-2 не расширит scope на них).

## 4. Функциональные требования

> Здесь «функциональные» = наблюдаемые визуальные свойства. **Конкретные значения утверждены заказчиком (стиль «мягкая приподнятая», 2026-05-29):** карточка каталога — `border-radius: 18px`, `border: 1px solid var(--border)`, `box-shadow: 0 4px 16px rgba(0,0,0,.08)`; слайд карусели — `border-radius: 18px`, `box-shadow: inset 0 0 0 1px rgba(0,0,0,.06)` (тонкая внутренняя обводка поверх фото). Это ориентир для архитектора; точные итоговые значения (в т.ч. тему-зависимые для тёмной темы) фиксирует ADR в пределах этого стиля. См. §8.

- **ФТ-1.** Карточки каталога услуг `.svc-catalog-card` (экран `s-services`) имеют рамку стиля «мягкая приподнятая»: увеличенный радиус (18px), тонкий бордер `1px var(--border)` и выраженную мягкую тень (`0 4px 16px rgba(0,0,0,.08)`), заменяющую ослабленную тень из блока «CARD POLISH» (`style.css:693`) для этой карточки.
- **ФТ-2.** Слайды карусели `.svc-carousel-slide` на детальной карточке услуги `s-service` имеют **единообразный по впечатлению** край того же стиля, реализованный для фото как внутренняя обводка `inset 0 0 0 1px rgba(0,0,0,.06)` + радиус 18px (обводка поверх изображения, не «съедающая» peek/ширину слайда из BA-017).
- **ФТ-3.** Оформление рамок корректно отображается в **светлой и тёмной** теме: граница достаточно контрастна и аккуратна в обеих темах, не «исчезает» и не выглядит грубо (целевой контраст/цвет для тёмной темы — по ОВ-4).
- **ФТ-4.** Существующая обрезка фото сохраняется: скругление и `overflow: hidden` продолжают корректно обрезать обложку каталога (`.svc-cover`) и фото слайда; `aspect-ratio: 16/9` обложки и `height/object-fit: cover` фото не нарушаются.
- **ФТ-5.** Геометрия карусели из BA-017 сохраняется: ширина слайда, gap и **равномерный peek по левому и правому краю** не нарушаются добавлением рамки; выравнивание слайдов остаётся корректным.
- **ФТ-6.** Состояние нажатия карточки каталога (`:active` → `transform: scale(0.97)`) сохраняется и визуально согласуется с новой рамкой.
- **ФТ-7.** Изменения локализованы в разделе услуг (каталог + слайды карусели детальной карточки); прочие карточки и экраны приложения визуально не меняются (см. §2 «вне рамок»).

## 5. Нефункциональные требования

- **НФТ-1 (Совместимость стека).** Решение — vanilla CSS, правки ожидаются в `docs/style.css`; при необходимости минимальная правка разметки в `docs/modules/services.js` с сохранением `esc()`. Без новых рантайм-npm-зависимостей.
- **НФТ-2 (Производительность).** Оформление рамок (бордер/тень/обводка) не вызывает заметных лагов при скролле каталога и свайпе карусели; не вводит тяжёлых перерисовок.
- **НФТ-3 (Отсутствие ошибок).** В консоли браузера нет ошибок при просмотре каталога, открытии детальной карточки и листании карусели.
- **НФТ-4 (Темы).** Оформление консистентно в светлой и тёмной теме (есть `[data-theme="dark"]` с переопределением `--border`, `--surface`, `--shadow`; блок «CARD POLISH» переопределяет тени карточек) — рамки должны учитывать эти переопределения, а не вводить захардкоженные цвета, ломающие тёмную тему (без явного основания).
- **НФТ-5 (PWA/офлайн не регрессируют).** Устанавливаемость PWA и оффлайн-поведение не ухудшаются; при изменении файлов, входящих в `APP_SHELL` (`style.css`, и при правке — `services.js`), поднимается `CACHE_VERSION` в `docs/sw.js` (правило проекта).
- **НФТ-6 (Безопасность / XSS).** Если потребуется правка разметки слайдов/карточек — URL фото и `alt` выводятся только через `esc()`; вставка непроверенного HTML недопустима.
- **НФТ-7 (Минимальность изменений).** Изменения строго в пределах оформления границ карточек каталога и слайдов карусели; прочие карточки, механика карусели, источник фото и кэширование не рефакторятся.

## 6. Ограничения и допущения

**Ограничения:**

- Релевантный код (для информации; реализацию определяет архитектор/разработчик):
  - Карточка каталога — `.svc-catalog-card` (`docs/style.css:273-282`): `border: 1px solid var(--border)`, `border-radius: 16px`, `box-shadow: var(--shadow)`, `overflow: hidden`; `:active` → `scale(0.97)` (`style.css:282`); тень ослаблена в «CARD POLISH» до `0 1px 4px rgba(0,0,0,0.05)` (`style.css:693`). Обложка `.svc-cover` — `aspect-ratio: 16/9`, `overflow: hidden` (`style.css:283-289`). Разметка — `docs/modules/services.js:91-96`.
  - Слайд карусели — `.svc-carousel-slide` (`docs/style.css:834-838`): `flex: 0 0 calc(100% - 44px)`, `border-radius: 16px`, `overflow: hidden`, **без бордера**; фото `height: 280px; object-fit: cover`. Трек `.svc-carousel-track` — `gap: 8px`. Разметка — `docs/modules/services.js:168-169`.
  - Тема — `:root` и `[data-theme="dark"]` (`docs/style.css:10-39`): `--border` (светлая `#E5E5EA` / тёмная `#3A3A3C`), `--surface`, `--shadow` (тёмная тень сильнее), `--accent` (`#8B3558`). Единой переменной `--radius` в проекте нет — карточки используют конкретные значения (каталог и слайды — 16px).
- Только CSS-изменения (`docs/style.css`); при необходимости — минимальная правка `docs/modules/services.js` с сохранением `esc()`.
- Нельзя нарушать `overflow: hidden` (скругление обрезает фото), `aspect-ratio` обложки каталога, высоту/`object-fit` фото и геометрию карусели из BA-017 (равномерный peek, расчёт ширины слайда — рамка не должна «съедать» peek/ширину).
- Изменения строго в пределах раздела услуг (каталог + слайды карусели детальной карточки). Карточки мастеров, истории, ленты и пр. — не трогать.
- Корректность в светлой и тёмной теме; без ошибок в консоли.
- При изменении файлов `APP_SHELL`/`sw.js` — поднять `CACHE_VERSION`.
- Минимальные изменения строго по объёму; запрет коммита секретов; `--force` в `main` запрещён; любые операции с веткой `backup` запрещены.

**Допущения:**

- **Д-1.** Требование чисто визуальное: меняется только оформление границ, без изменения данных, логики и поведения карусели.
- **Д-2.** До получения ответа на ОВ-1 числовые параметры стиля (цвет/толщина/тень/радиус) считаются **неопределёнными** и не выдумываются; они фиксируются после согласования с заказчиком и при необходимости отражаются дополнением к §4.
- **Д-3.** По умолчанию (до ответа на ОВ-3) оформление каталога и карусели делается **единообразным по визуальному впечатлению**; технически это может быть выражено по-разному (вокруг всей плашки каталога vs обводка поверх фото слайда), но воспринимаемый стиль один.
- **Д-4.** По умолчанию (до ответа на ОВ-2) одиночное фото и категорийный плейсхолдер детальной карточки **не входят** в объём.

## 7. Критерии приёмки

> Критерии проверяются по наблюдаемому результату. «Согласованный стиль» = утверждённая «мягкая приподнятая» рамка (значения — см. §4: карточка radius 18px + border 1px `--border` + тень `0 4px 16px rgba(0,0,0,.08)`; слайд radius 18px + `inset 0 0 0 1px rgba(0,0,0,.06)`).

- **КП-1.** На экране каталога услуг карточки `.svc-catalog-card` имеют рамку в согласованном с заказчиком стиле (по ОВ-1); край карточки выглядит аккуратно и выразительно, не «плоско». (ФТ-1)
- **КП-2.** На детальной карточке услуги слайды карусели `.svc-carousel-slide` имеют рамку, единообразную по стилю с карточками каталога (или согласованно отличную по ОВ-3). (ФТ-2)
- **КП-3.** Рамки в каталоге и карусели аккуратны и достаточно контрастны как в **светлой**, так и в **тёмной** теме (визуальная проверка переключением темы). (ФТ-3, НФТ-4)
- **КП-4.** Фото в обложке каталога и в слайдах остаются корректно скруглены и обрезаны (`overflow: hidden`), `aspect-ratio: 16/9` обложки и высота/`object-fit: cover` фото не нарушены. (ФТ-4)
- **КП-5.** Геометрия карусели не нарушена: слайды выровнены, peek соседних слайдов равномерен по левому и правому краю (проверка на 2, 3 и 8 фото — согласно BA-017 КП-1). (ФТ-5)
- **КП-6.** Нажатие на карточку каталога сохраняет анимацию `:active` (`scale(0.97)`) и визуально согласуется с рамкой. (ФТ-6)
- **КП-7.** Оформление других карточек приложения (мастера, история, лента, hero, confirm/cs) и прочих экранов визуально не изменилось. (ФТ-7, НФТ-7)
- **КП-8.** При просмотре каталога, открытии детальной карточки и листании карусели в консоли нет ошибок; скролл/свайп без заметных лагов. (НФТ-2, НФТ-3)
- **КП-9.** Если правилась разметка — URL фото и `alt` экранируются (`esc()`); инъекция HTML невозможна. (НФТ-6)
- **КП-10.** PWA остаётся устанавливаемой, оффлайн-поведение не регрессирует; при изменении файлов `APP_SHELL`/`sw.js` поднята `CACHE_VERSION`; новые рантайм-зависимости не добавлены. (НФТ-1, НФТ-5)

## 8. Согласованные решения (закрытые открытые вопросы)

> Все открытые вопросы согласованы с заказчиком 2026-05-29. Блокеров для `/adr` нет.

- **ОВ-1 (РЕШЕНО). Стиль «красивой рамки» — «мягкая приподнятая».** Утверждено:
  - **Карточка каталога** `.svc-catalog-card`: `border-radius: 18px`; `border: 1px solid var(--border)`; `box-shadow: 0 4px 16px rgba(0,0,0,.08)` (выраженная мягкая «карточная» тень; для этой карточки заменяет ослабленную тень из «CARD POLISH», `style.css:693`).
  - **Слайд карусели** `.svc-carousel-slide`: `border-radius: 18px`; `box-shadow: inset 0 0 0 1px rgba(0,0,0,.06)` (тонкая внутренняя обводка поверх фото).
  - Тип края: тонкий бордер + мягкая тень у карточки, inset-обводка у слайда; увеличенный радиус 18px у обоих. Цвет — нейтральный `--border`/полупрозрачный чёрный, **без** акцентного цвета.
- **ОВ-2 (РЕШЕНО). Объём — только `.svc-catalog-card` и `.svc-carousel-slide`.** Одиночное фото `.svc-card-single-img` и категорийный плейсхолдер `.svc-card-placeholder` в объём **не входят** (как в Д-4). §2 «вне рамок» в силе.
- **ОВ-3 (РЕШЕНО). Единый стиль по впечатлению, реализация разная.** Каталог — рамка (border) + тень вокруг плашки; слайд — inset-обводка поверх фото. Воспринимаемый стиль один (как Д-3).
- **ОВ-4 (РЕШЕНО, с инженерной заметкой для ADR). Тёмная тема — через тему-зависимые переменные.** Бордер карточки использует `var(--border)` (адаптируется к тёмной теме автоматически). **Заметка архитектору:** inset-обводка слайда `rgba(0,0,0,.06)` на тёмном фоне почти не видна — в тёмной теме допустимо/желательно заменить её на светлую полупрозрачную (например, `rgba(255,255,255,.08)`) через `[data-theme="dark"]`, чтобы край слайда оставался читаемым. Это деталь реализации в пределах утверждённого стиля, не меняет ФТ/КП.
- **ОВ-5 (РЕШЕНО). Отдельных состояний рамки нет.** Сохраняется существующий `:active scale(0.97)` карточки каталога; новых hover/selected-рамок не вводим.
