# BA-требования 019: Скрыть кнопку «Назад» на экране услуг при входе с главного экрана

- **Источник:** `requirements/req-services-hide-back-019.md`
- **Дата:** 2026-05-29
- **Статус:** согласовано (ОВ-1 = вариант «а», динамическое скрытие; утверждено заказчиком 2026-05-29; см. §8)

## 1. Контекст и бизнес-цель

Studio PWA (студия красоты, онлайн-запись поверх YCLIENTS) — устанавливаемое статическое приложение на vanilla JS/HTML/CSS, отдаётся из `docs/` на GitHub Pages. Экран услуг `s-services` — каталог услуг, точка входа в поток записи.

Сейчас в шапке экрана `s-services` всегда отображается кнопка «‹ Назад» (`docs/prototype.html:105`). При этом экран открывается двумя способами, и в одном из них кнопка бесполезна:

1. **Как корневой раздел (таб)** — через нижнюю навигацию: центральная FAB-кнопка «Запись» и таб «Услуги» (`go('s-services','tab')`, `docs/prototype.html:95,97,130,132`). Режим `tab` сбрасывает историю навигации `navHistory = ['s-services']` (`docs/modules/navigation.js:75`), после чего `back()` не делает ничего из-за защиты `if (navHistory.length < 2) return;` (`navigation.js:103-104`). В этом случае кнопка «Назад» — мёртвый, неработающий элемент: возвращаться некуда, услуги открыты как корень.
2. **Как push с главного экрана** — кнопка «Записать себя» вызывает `go('s-services')` без режима `tab` (`docs/prototype.html:62`). Тогда `s-services` кладётся в `navHistory` поверх `s-home` (`navigation.js:87`), и `back()` корректно возвращает на главный экран — кнопка осмысленна и работает.

**Бизнес-цель:** убрать визуальный «мусор» — нерабочую кнопку «‹ Назад» в шапке экрана услуг, когда возвращаться некуда (вход с главного как корневой раздел), не нарушив навигацию в остальных сценариях и на остальных экранах. Это точечная UX-правка одного экрана, а не редизайн навигации.

**Стейкхолдеры:**

- Клиент студии — пользователь PWA; видит нерабочую кнопку при входе в услуги с главного, что воспринимается как недоработка.
- Заказчик / владелец студии — заинтересован в аккуратном, предсказуемом интерфейсе.
- Бизнес-аналитик / архитектор / разработчик — оформление, выбор подхода, реализация.

## 2. Область (scope)

### В рамках задачи

- Поведение кнопки «‹ Назад» в шапке (`nav-bar`) **только экрана `s-services`** (`docs/prototype.html:105`).
- Скрытие этой кнопки при входе на `s-services` с главного экрана как корневого раздела (через нижнюю навигацию: таб «Услуги» и FAB «Запись»), когда `back()` неактивна (`navHistory.length < 2`).
- Корректная вёрстка шапки `s-services`: заголовок «Выберите услугу» (`prototype.html:106`) не должен «прыгать» при наличии/отсутствии кнопки.

### Вне рамок (explicitly out of scope)

- Поведение и стиль общего класса `back-btn` на **любых других экранах** (детальная карточка услуги, выбор мастера, слоты, бронирование, OTP, профиль, история, лента и т. д.) — не затрагивается. Правка должна касаться только инстанса кнопки на `s-services`.
- Любые изменения логики `go()` / `back()` / `navHistory` для других экранов и сценариев, не нужные строго для данной задачи.
- Кнопки «Назад» в шапках прочих корневых табов (лента, профиль, история) — единообразие по ним вне объёма данного требования (зафиксировано в открытых вопросах, см. §8 ОВ-2).
- Поток записи из услуг дальше (выбор услуги → мастер → слот → бронирование) — не меняется.
- Перенос захардкоженного YCLIENTS-токена (известный техдолг — «не усугублять»).

## 3. Действующие лица и сценарии использования

**Клиент студии** — пользователь PWA (преимущественно мобильный, тач). Цель — открыть каталог услуг и выбрать услугу для записи.

Сценарии входа на экран услуг:

- **С1. Вход как корневой раздел (таб).** Клиент на любом экране нажимает таб «Услуги» или центральную FAB «Запись» в нижней навигации. Открывается `s-services` как корень; история сброшена; возвращаться кнопкой некуда. Целевое поведение: кнопка «‹ Назад» в шапке **отсутствует**, заголовок «Выберите услугу» отображается корректно без смещения.
- **С2. Вход push с главного экрана.** Клиент на главном (`s-home`) нажимает «Записать себя». Открывается `s-services` поверх главного; `back()` ведёт на главный. Целевое поведение для этого сценария — предмет блокирующего открытого вопроса (см. §8 ОВ-1): кнопка либо остаётся рабочей (вариант «а»), либо тоже скрыта, а возврат — только нижней навигацией/жестом (вариант «б»).
- **С3. Дальнейшая навигация из услуг.** Из `s-services` клиент уходит дальше по потоку записи и возвращается назад — на этих экранах кнопка «Назад» работает как прежде (вне объёма правки).

## 4. Функциональные требования

- **ФТ-1.** При входе на экран `s-services` как корневой раздел через нижнюю навигацию (таб «Услуги» или FAB «Запись», т. е. когда возврат невозможен — `navHistory.length < 2`) кнопка «‹ Назад» в шапке экрана `s-services` **не отображается**.
- **ФТ-2.** Заголовок «Выберите услугу» и остальное содержимое шапки/экрана `s-services` отображаются корректно как при наличии, так и при отсутствии кнопки «Назад» — без визуального смещения («прыжка») при появлении/исчезновении кнопки.
- **ФТ-3.** Поведение кнопки «Назад» (класс `back-btn`) на **всех остальных экранах** приложения не изменяется.
- **ФТ-4.** При входе на `s-services` push-переходом с главного экрана («Записать себя», `go('s-services')`, когда `navHistory.length >= 2`) кнопка «‹ Назад» **отображается и по нажатию возвращает на главный экран** (согласованный вариант «а», см. §8 ОВ-1). Условие показа кнопки — есть куда возвращаться (`navHistory.length >= 2`).
- **ФТ-5.** Переходы из экрана услуг дальше по потоку записи (выбор услуги, мастера, слота, бронирование) и возврат по ним работают как прежде.

## 5. Нефункциональные требования

- **НФТ-1 (Совместимость стека).** Решение — vanilla JS/HTML/CSS, код в `docs/`; без новых рантайм-npm-зависимостей. Точка для условного показа/скрытия при входе на экран — механизм `registerOnEnter` (`docs/modules/navigation.js:6-9,42-45`); способ реализации (CSS-класс, `display`, условный рендер) определяет архитектор.
- **НФТ-2 (Локальность изменений).** Изменения затрагивают только инстанс кнопки на `s-services` и не меняют поведение/стиль `back-btn` на других экранах; минимальный объём строго по задаче.
- **НФТ-3 (Отсутствие ошибок).** При входе на `s-services` обоими способами, при переключении табов и при переходах в консоли браузера нет ошибок; вёрстка шапки корректна.
- **НФТ-4 (PWA/офлайн не регрессируют).** Устанавливаемость и оффлайн-поведение PWA не ухудшаются; при изменении файлов из `APP_SHELL` (например, `navigation.js`, `prototype.html`, `style.css`, если входят в `APP_SHELL`/`sw.js`) поднимается `CACHE_VERSION` в `docs/sw.js` (правило проекта).

## 6. Ограничения и допущения

**Ограничения:**

- Релевантный код (для информации; способ реализации определяет архитектор):
  - Шапка экрана услуг: `<div class="nav-bar"><button class="back-btn" onclick="back()">‹ Назад</button></div>` (`docs/prototype.html:105`); заголовок «Выберите услугу» — `prototype.html:106`.
  - Вход как таб: `go('s-services','tab')` (`prototype.html:95,97,130,132`); режим `tab` сбрасывает `navHistory = [id]` (`navigation.js:75`); `back()` неактивна при `navHistory.length < 2` (`navigation.js:103-104`).
  - Вход push: `go('s-services')` с кнопки «Записать себя» (`prototype.html:62`); экран добавляется в историю (`navigation.js:87`).
  - Хук входа на экран: `registerOnEnter(screenId, fn)` / `onEnterScreen` (`navigation.js:6-9,42-45`); регистрируется в инициализации приложения.
- `back-btn` — общий класс кнопки «Назад» на многих экранах: правка стиля самого класса затронет все экраны и недопустима; решение должно адресовать только инстанс на `s-services`.
- Vanilla JS/HTML/CSS, без рантайм-зависимостей; правки в `docs/prototype.html` и/или `docs/modules/*.js` и при необходимости `docs/style.css`.
- При изменении файлов из `APP_SHELL`/`sw.js` — поднять `CACHE_VERSION`.
- Минимальные изменения строго по объёму.

**Допущения:**

- **Д-1.** Признак «возвращаться некуда» технически выражается как `navHistory.length < 2` (та же проверка, по которой `back()` уже сейчас ничего не делает). При входе через таб это значение равно 1 (история сброшена).
- **Д-2.** Под «входом с главного экрана» в названии требования понимается прежде всего вход в услуги как корневой раздел через нижнюю навигацию (С1); трактовка push-входа «Записать себя» (С2) — предмет блокирующего открытого вопроса ОВ-1.
- **Д-3.** Скрытие кнопки не влияет на работу нижней навигации (табов) и системного жеста «назад» — возврат с `s-services` остаётся доступен через нижнюю навигацию в любом случае.

## 7. Критерии приёмки

- **КП-1.** При открытии услуг через нижнюю навигацию (таб «Услуги» или FAB «Запись») кнопка «‹ Назад» в шапке `s-services` отсутствует. (ФТ-1)
- **КП-2.** При отсутствии кнопки «Назад» заголовок «Выберите услугу» отображается без смещения/«прыжка» относительно состояния с кнопкой; вёрстка шапки корректна. (ФТ-2)
- **КП-3.** Кнопка «Назад» на других экранах (детальная карточка услуги, выбор мастера, слоты, бронирование, OTP, профиль, лента, история) работает и выглядит как прежде. (ФТ-3)
- **КП-4.** При входе на `s-services` через «Записать себя» поведение кнопки «Назад» соответствует согласованному варианту из ОВ-1 (вариант «а» — кнопка возвращает на главный; вариант «б» — кнопки нет, возврат через нижнюю навигацию/жест). (ФТ-4)
- **КП-5.** Возврат с `s-services` на главный экран доступен (кнопкой и/или нижней навигацией согласно выбранному варианту); переходы из услуг дальше по потоку записи и обратно работают как прежде. (ФТ-4, ФТ-5)
- **КП-6.** При входе на `s-services` обоими способами, переключении табов и переходах в консоли нет ошибок. (НФТ-3)
- **КП-7.** PWA остаётся устанавливаемой, оффлайн-поведение не регрессирует; при изменении файлов из `APP_SHELL`/`sw.js` поднята `CACHE_VERSION`; новые рантайм-зависимости не добавлены. (НФТ-1, НФТ-4)

## 8. Согласованные решения и открытые вопросы

> ОВ-1 согласован с заказчиком 2026-05-29 — выбран **вариант «а» (динамическое скрытие)**. Блокеров для `/adr` нет. Противоречий с принятыми ADR не выявлено.

- **ОВ-1 (РЕШЕНО — вариант «а», динамически).** Кнопка «‹ Назад» на `s-services` скрывается **только когда возвращаться некуда** — вход как таб/корень (`navHistory.length < 2`). При push-входе «Записать себя» (`go('s-services')`, `navHistory.length >= 2`) кнопка **отображается и работает** (ведёт на главный). Логика показа: кнопка видна ⇔ `navHistory.length >= 2`. Это фиксирует ФТ-1, ФТ-4, КП-1, КП-4, КП-5, Д-2.

- **ОВ-2 (неблокирующий, уточнение объёма). Единообразие с другими корневыми табами.** На прочих корневых табах (лента, профиль, история) кнопка «Назад» в шапке может иметь ту же проблему «мёртвой кнопки». По текущему объёму требования правка касается **только `s-services`**. Нужно ли распространять то же поведение на остальные табы — отдельным требованием. По умолчанию — нет, только `s-services`. Влияет на §2 (scope).

- **ОВ-3 (неблокирующий, для архитектора). Способ скрытия и стабильность вёрстки.** Как именно скрывать (условный класс/`display:none`/условный рендер в `onEnter` для `s-services`) и нужно ли резервировать высоту `nav-bar`, чтобы заголовок «Выберите услугу» не смещался при появлении/исчезновении кнопки. Выбор подхода — за архитектором в рамках ФТ-2/КП-2; на формулировку требований не влияет.
