# BA-требования 010: Просмотр всех предстоящих визитов с главного экрана

- **Источник:** `requirements/req-home-upcoming-visits-010.md`
- **Дата:** 2026-05-29
- **Статус:** черновик

## 1. Контекст и бизнес-цель

Studio PWA (студия красоты, онлайн-запись поверх YCLIENTS) — устанавливаемое приложение, отдаётся статически из `docs/` на GitHub Pages. На главном экране в блоке «Ближайший визит» (`renderHomeHero`, `docs/modules/profile.js:63`) клиенту показывается только один — самый ближайший — будущий визит (`next = upcoming[0]`, `profile.js:88`), хотя все будущие визиты уже вычисляются (`upcoming`, `profile.js:85-87`). Если у клиента несколько предстоящих записей, увидеть их все с главного экрана нельзя: тап по карточке ведёт на экран «История посещений» (`onclick="go('s-history')"`, `profile.js:116`), где предстоящие визиты смешаны с прошлыми, а сам переход не обозначен явной ссылкой.

**Бизнес-цель:** дать клиенту быстрый и очевидный путь с главного экрана к списку **всех своих предстоящих (будущих) визитов** — без прошлых визитов и без захода в Профиль → «История посещений», — с возможностью по каждому визиту выполнить действия «Перенести» и «Отменить». Affordance — ссылка «Все →» рядом с блоком ближайшего визита, по аналогии с уже существующими «Все →» у ленты (`go('s-feed')`, `prototype.html:72`) и у мастеров (`_browseAllMasters()`, `prototype.html:82`).

**Стейкхолдеры:**
- Заказчик / владелец студии — заинтересован в удобном управлении записями и снижении числа неявок.
- Клиент студии — пользователь, у которого может быть несколько будущих визитов; хочет видеть их все и управлять ими в одном месте.
- Бизнес-аналитик / архитектор / разработчик — оформление, проектирование, реализация.

## 2. Область (scope)

### В рамках задачи
- На главном экране рядом с блоком «Ближайший визит» появляется явный affordance «Все →» (визуально и по поведению согласованный с «Все →» ленты и мастеров).
- По нажатию «Все →» открывается список **всех предстоящих (будущих) визитов** клиента, отсортированных по дате (по возрастанию, ближайший — первым), **без прошлых визитов**.
- Для каждого визита в этом списке доступны действия «Перенести» (`rescheduleRecord`) и «Отменить» (`cancelRecord`) — то же поведение, что у действий для единственного визита на главной.
- Путь к списку предстоящих визитов не требует захода в Профиль → «История посещений».
- Источник данных — те же визиты, что уже используются на главной и в истории: `_loadStoredRecords()` с серверным слиянием `_fetchAndMergeServerRecords`; отбор «будущие и не отменённые» — по существующей логике (`status !== 'cancelled'` и `datetime > now`).
- Определение пустого состояния и условий показа «Все →» (см. открытые вопросы №2 и №4).

### Вне рамок (explicitly out of scope)
- **Изменение логики переноса и отмены визита** (`rescheduleRecord`, `cancelRecord`) и их интеграции с YCLIENTS — переиспользуются как есть, не дорабатываются.
- **Действия «Повторить» и «Оценить»** — относятся к прошлым визитам; в список предстоящих визитов не добавляются (см. открытый вопрос №3).
- **Раздел «Прошлые визиты», фильтр по периоду и предложение «оценить прошлый визит»** на экране истории / на главной — не затрагиваются (кроме случая, прямо вытекающего из решения по открытому вопросу №1).
- **Изменение блока «Ближайший визит» по сути** (какой визит показывать первым, его вёрстка, кнопки) — меняется только добавление affordance «Все →»; показ первого визита сохраняется.
- **Доработка ленты, списка мастеров, профиля, push, темы оформления** — не затрагиваются.
- **Перенос захардкоженного YCLIENTS-токена** из `prototype.html` (известный техдолг — «не усугублять», не трогать).

## 3. Действующие лица и сценарии использования

**Клиент студии** — авторизованный пользователь PWA, у которого есть одна или несколько будущих записей. Цель — быстро увидеть все предстоящие визиты и при необходимости перенести или отменить любой из них, не углубляясь в раздел «История».

Основной сценарий:
1. Клиент на главном экране видит блок «Ближайший визит» (ближайшая по дате будущая запись) и рядом — affordance «Все →».
2. Клиент нажимает «Все →».
3. Открывается список **всех** его предстоящих (будущих) визитов, отсортированных по дате; прошлые визиты не показаны.
4. По любому визиту клиент нажимает «Перенести» или «Отменить» — выполняется существующее действие переноса/отмены.
5. После отмены/переноса список и блок главного экрана отражают актуальное состояние.

Граничные сценарии:
- **Ровно один предстоящий визит** — он уже показан на главной; поведение affordance «Все →» в этом случае определяется открытым вопросом №2.
- **Нет предстоящих визитов** — на главной показывается существующее состояние «Нет предстоящих записей» (или предложение оценить недавний визит, `profile.js:92-114`); affordance «Все →» в этом случае не должен вводить в заблуждение (открытый вопрос №4).
- **Визит отменён или перенесён из списка** — список перестраивается на актуальных данных; отменённый визит исчезает, перенесённый отображается с новой датой/порядком.

## 4. Функциональные требования

- **ФТ-1.** На главном экране рядом с блоком «Ближайший визит» отображается affordance «Все →», визуально согласованный с «Все →» ленты и мастеров.
- **ФТ-2.** По нажатию «Все →» открывается представление со списком **всех** предстоящих (будущих) визитов клиента.
- **ФТ-3.** В этом списке отображаются **только** будущие, не отменённые визиты; прошлые визиты не показываются.
- **ФТ-4.** Визиты в списке отсортированы по дате/времени по возрастанию (ближайший — первым).
- **ФТ-5.** Для каждого визита в списке доступны действия «Перенести» и «Отменить», выполняющие то же поведение, что и соответствующие действия на главном экране (`rescheduleRecord`, `cancelRecord`).
- **ФТ-6.** Действия «Повторить» и «Оценить» в списке предстоящих визитов **не** отображаются.
- **ФТ-7.** Достижение списка предстоящих визитов не требует перехода в Профиль → «История посещений».
- **ФТ-8.** Каждый элемент списка отображает достаточную для идентификации визита информацию (как минимум: услуга, мастер, дата/время — в объёме, уже показываемом для предстоящих визитов сегодня).
- **ФТ-9.** После отмены или переноса визита из списка отображаемое состояние (список и/или блок главного экрана) приводится в соответствие с актуальными данными визитов.
- **ФТ-10.** При отсутствии предстоящих визитов поведение affordance «Все →» не вводит клиента в заблуждение (конкретика — по решению открытого вопроса №4).
- **ФТ-11.** Все пользовательские тексты, затронутые изменением, — на русском языке, в едином тоне приложения.

## 5. Нефункциональные требования

- **НФТ-1 (Совместимость стека).** Решение остаётся в рамках стека: vanilla JS/HTML/CSS, код в `docs/`; статический хостинг на GitHub Pages; без новых рантайм-npm-зависимостей.
- **НФТ-2 (Единый источник данных).** Список предстоящих визитов строится из тех же данных, что главный экран и история (`_loadStoredRecords()` + `_fetchAndMergeServerRecords`); состав и набор запросов к YCLIENTS не расширяется.
- **НФТ-3 (Достоверность и согласованность).** Состав и порядок визитов в списке согласованы с блоком «Ближайший визит» (первый элемент списка совпадает с показанным на главной ближайшим визитом) и с секцией «Предстоящие» экрана истории; одни и те же визиты не должны различаться между представлениями.
- **НФТ-4 (PWA/офлайн не регрессируют).** Изменения не ломают устанавливаемость PWA и офлайн-поведение Service Worker; при изменении состава `APP_SHELL` / кэшируемых файлов (`prototype.html`, `profile.js`, `history.js`, `constants.js` и т. п.) поднимается `CACHE_VERSION` в `docs/sw.js` (правило проекта).
- **НФТ-5 (Минимальность изменений).** Изменения строго в объёме задачи; существующий экран `s-history` и действия «Перенести»/«Отменить»/«Повторить»/«Оценить» не должны сломаться.
- **НФТ-6 (Локализация).** Все пользовательские тексты — на русском языке.

## 6. Ограничения и допущения

**Ограничения:**
- Приложение — статический PWA в `docs/` на GitHub Pages; релевантный код — `docs/modules/profile.js` (`renderHomeHero`), `docs/modules/history.js` (`_renderHistoryFromCache`), разметка экранов — `docs/prototype.html`, реестр экранов — `docs/modules/constants.js` (`SCREENS`).
- Логика отбора будущих и не отменённых визитов уже существует и должна переиспользоваться (`profile.js:85-87`, `history.js:106-111`); вводить параллельную/иную логику отбора не следует.
- Действия «Перенести»/«Отменить» (`rescheduleRecord`, `cancelRecord`) — существующие, переиспользуются без изменения поведения.
- Запрещены рантайм-npm-зависимости; запрещён коммит секретов; запрещены любые операции с веткой `backup`.
- При изменении файлов из `APP_SHELL` — поднятие `CACHE_VERSION` в `docs/sw.js`.

**Допущения:**
- «Предстоящий / будущий визит» = запись со статусом `!= 'cancelled'` и `datetime > now` (как в текущем коде). Прошлым считается визит с `datetime <= now`.
- Сортировка списка предстоящих — по возрастанию даты (как в текущем `upcoming.sort`, `profile.js:87` / `history.js:111`).
- Данные о визитах достаточны для отображения и действий в новом списке в том же объёме, что уже используется секцией «Предстоящие» истории и блоком главного экрана; дополнительных полей/запросов не требуется. **Подтверждает архитектор** при проектировании.
- До решения заказчика по открытым вопросам №1–№4 окончательная форма представления (отдельный экран `s-upcoming` или переиспользование секции «Предстоящие» из `s-history`), условие показа «Все →» и пустое состояние не фиксируются на уровне реализации.

## 7. Критерии приёмки

- **КП-1.** На главном экране рядом с блоком «Ближайший визит» присутствует affordance «Все →», оформленный согласованно с «Все →» ленты/мастеров. (ФТ-1, НФТ-6)
- **КП-2.** Нажатие «Все →» открывает список всех предстоящих (будущих) визитов клиента, не требуя перехода в Профиль → «История посещений». (ФТ-2, ФТ-7)
- **КП-3.** В открывшемся списке отображаются только будущие не отменённые визиты; прошлых визитов в нём нет. (ФТ-3)
- **КП-4.** Визиты в списке упорядочены по дате по возрастанию; первый элемент совпадает с ближайшим визитом, показанным на главной. (ФТ-4, НФТ-3)
- **КП-5.** Для каждого визита в списке доступны «Перенести» и «Отменить», и они выполняют то же действие, что соответствующие кнопки на главном экране; «Повторить»/«Оценить» в списке отсутствуют. (ФТ-5, ФТ-6)
- **КП-6.** Каждый элемент списка показывает услугу, мастера и дату/время визита. (ФТ-8)
- **КП-7.** После отмены или переноса визита из списка отображаемое состояние (список и блок главного экрана) соответствует актуальным данным; отменённый визит исчезает, перенесённый — отображается с новой датой/порядком. (ФТ-9, НФТ-2, НФТ-3)
- **КП-8.** При отсутствии предстоящих визитов клиент не попадает по «Все →» на пустое/вводящее в заблуждение представление (поведение — по решению открытого вопроса №4). (ФТ-10)
- **КП-9.** Существующий экран «История посещений» (`s-history`) и его действия «Перенести»/«Отменить»/«Повторить»/«Оценить», фильтр по периоду — не регрессируют. (НФТ-5)
- **КП-10.** PWA остаётся устанавливаемой, офлайн-поведение не регрессирует; при изменении файлов `APP_SHELL` поднята `CACHE_VERSION`; новые рантайм-зависимости не добавлены. (НФТ-1, НФТ-4)
- **КП-11.** Затронутые пользовательские тексты — на русском языке. (ФТ-11, НФТ-6)

## 8. Открытые вопросы

Перечисленные ниже вопросы — продуктовые (выбор UX-решения); они **не блокируют** оформление BA-документа, но **должны быть решены до запуска `/adr`**, так как определяют форму реализации. Архитектор не должен выбирать вариант самостоятельно без согласования с заказчиком.

1. **Отдельный экран или переиспользование секции «Предстоящие» из `s-history`?** Создать отдельный экран «только предстоящие визиты» (например `s-upcoming`, регистрируется в `SCREENS`, `constants.js`) **или** переиспользовать уже существующую секцию «Предстоящие» экрана `s-history` (`history.js:121-143`), открывая её так, чтобы при входе с главной показывались только будущие визиты (без блока «Прошлые визиты»)? Формулировка требования («только ближайшие будущие», «не в Истории») склоняется к **чистому представлению без прошлых визитов**; технически данные и рендер списка предстоящих уже существуют в `history.js`. *Влияет на раздел 2, ФТ-2, ФТ-3, ФТ-7. Продуктовое решение заказчика; механику — архитектор.*

2. **Когда показывать «Все →»?** Показывать affordance всегда при наличии хотя бы одного предстоящего визита **или** только когда предстоящих визитов 2+ (поскольку один уже отображён на главной как «Ближайший визит»)? *Влияет на ФТ-1, ФТ-10, КП-1, КП-8. Продуктовое решение заказчика.*

3. **Набор действий в списке.** Подтвердить, что в списке предстоящих визитов нужны только «Перенести» и «Отменить» (как у визита на главной), без «Повторить»/«Оценить» (это действия для прошлых визитов). В требовании предполагается именно такой набор — настоящий документ исходит из этого (ФТ-5, ФТ-6); требуется явное подтверждение. *Влияет на ФТ-5, ФТ-6, КП-5. Продуктовое решение заказчика.*

4. **Пустое состояние.** Что показывать, если предстоящих визитов нет? Базовое предположение: при отсутствии будущих визитов affordance «Все →» не показывается (согласуется с решением по вопросу №2), поэтому отдельный экран пустого состояния не требуется. Если же заказчик хочет показывать «Все →» всегда — нужно определить текст/вид пустого представления. *Влияет на ФТ-10, КП-8. Продуктовое решение заказчика.*

### Решения по открытым вопросам (приняты 2026-05-29)

1. **Отдельный экран** «Ближайшие визиты» (только предстоящие, без прошлых) — например `s-upcoming`. Архитектор **переиспользует** существующий рендер секции «Предстоящие» из `history.js` (вынести в общий хелпер, использовать и в `s-history`, и в новом экране), чтобы не дублировать разметку/логику.
2. **«Все →» показывается только при 2+ предстоящих визитах** (один уже отображён на главной как «Ближайший визит»). При 0–1 предстоящих affordance не показывается.
3. **Действия в списке — «Перенести» и «Отменить»** по каждому визиту. «Повторить»/«Оценить» (для прошлых) не нужны.
4. **Пустое состояние:** при отсутствии (или единственном) предстоящем визите «Все →» не показывается, экран недостижим — отдельное пустое представление не требуется (если экран всё же открыт напрямую — простое «Нет предстоящих записей»).

Противоречий с принятыми ADR (`adrs/adr-001`…`adr-008`) не выявлено: задача касается навигации и представления уже имеющихся данных о визитах на главном экране и не пересматривает ранее зафиксированных архитектурных решений. Если на этапе проектирования обнаружится, что выбранный вариант (например создание нового экрана `s-upcoming`) затрагивает действующий ADR, это должно быть отражено в новом/обновлённом ADR.
