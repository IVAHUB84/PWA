# BA-требования 022: Лента в разрезе категорий и услуг YCLIENTS + запись на услугу из поста

- **Источник:** `requirements/req-feed-categories-services-022.md`
- **Дата:** 2026-05-29
- **Статус:** черновик (блокирующих открытых вопросов нет; продуктовые развилки ОВ-2/ОВ-3/ОВ-4/ОВ-7 — решить до `/adr`, см. §8)

## 1. Контекст и бизнес-цель

Studio PWA — устанавливаемое приложение клиента студии красоты (онлайн-запись поверх YCLIENTS API), vanilla JS/HTML/CSS, отдаётся статически из `docs/` на GitHub Pages.

В приложении есть **лента студии**: список публикаций (`s-feed`, `_paintFeed`, `docs/modules/feed.js`), карточка одной публикации (`s-post`, `renderPostScreen`, см. ADR-010), превью последних постов на главной (`_renderHomeFeedPreview`, `docs/modules/profile.js`). Публикации создаются администратором (`docs/modules/admin.js`), хранятся в `localStorage` `yc_feed_posts` и синхронизируются между устройствами через GitHub (`posts.json`, `_ghSyncPosts`/`_ghPullToLocal`, `docs/modules/github.js`).

Сейчас лента живёт **отдельно от реального каталога**. Пост привязывается к произвольной строке-категории из **хардкод-чипов** админ-формы (`Брови / Ногти / Лицо / Волосы / Тело / Акции`, fallback `Другое`; поле `cat` в модели поста, `admin.js:110`). Эти строки **не совпадают** с реальными категориями YCLIENTS, которые приложение уже грузит при старте (`catMap` из `GET /service_categories/{company}`, `SERVICES_DATA` из `GET /services/{company}`, `docs/app.js:166-200`, реальные категории — например `Брови, Губы, Глаза, Эпиляция`). Из-за рассинхрона строк кнопка «Записаться →» в посте (`_feedGoBook(p.cat)` → `filterCategory(cat)` + `go('s-services')`, `feed.js:68,111-114`) часто приводит к экрану «Ничего не найдено». Перехода на конкретную услугу из поста нет вовсе.

**Бизнес-цель:** перевести ленту в разрез **реальных** категорий и услуг YCLIENTS, чтобы публикации были связаны с каталогом, а призыв к действию работал предсказуемо и доводил клиента до записи. В частности — дать администратору публиковать пост по конкретной услуге так, чтобы клиент по кнопке «Записаться» попадал **на карточку именно этой услуги** (`s-service`, ADR-013). Эталонный кейс: пост «Акция на стрижку 20%» → «Записаться» → карточка услуги «Стрижка». Дополнительно — добавить в ленту фильтр по типам постов.

**Стейкхолдеры:**
- Заказчик / владелец студии — заинтересован в том, чтобы публикации вели к записи (конверсия из контента в визит), а контент-планирование опиралось на реальный каталог.
- Администратор студии — автор публикаций; ему нужно привязывать пост к реальной категории/услуге без ручного ввода строк.
- Клиент студии — пользователь PWA; ожидает, что «Записаться» из поста ведёт к релевантной услуге/каталогу, а не в тупик.
- Бизнес-аналитик / архитектор / разработчик — оформление, проектирование, реализация.

## 2. Область (scope)

### В рамках задачи
- Три **типа поста**: свободный (без привязки), по **категории** YCLIENTS, по **услуге** YCLIENTS.
- Админ-форма создания/редактирования поста: вместо хардкод-чипов — выбор типа поста и соответствующего объекта (категория или услуга) из **реальных данных** YCLIENTS (источник — уже загруженные `SERVICES_DATA`/`catMap`, ОВ-1).
- Расширение модели поста новыми полями (тип, ссылка на категорию/услугу) с сохранением их **при публикации и при GitHub-синхронизации** (`posts.json`).
- Поведение CTA в карточке поста в зависимости от типа:
  - по услуге → «Записаться» ведёт на карточку этой услуги (`openServiceCard(serviceId)` → `s-service`);
  - по категории → ведёт в каталог `s-services`, отфильтрованный по реальной категории (`filterCategory(realCat)`);
  - свободный → поведение CTA по решению ОВ-3.
- Корректное поведение CTA, если привязанная услуга/категория **больше не существует** в YCLIENTS (graceful degradation, ОВ-8).
- **Фильтр по типам постов** в ленте `s-feed` (измерения — ОВ-4).
- Обратная совместимость **уже опубликованных** постов (старое поле `cat`, ОВ-2) — без ошибок в существующих местах рендера: список `s-feed`, одиночный пост `s-post`, превью на главной.

### Вне рамок (explicitly out of scope)
- **Пост, привязанный к мастеру** («Записаться к мастеру») — отдельное требование (ОВ-5).
- **Несколько услуг в одном посте** — одна услуга на пост (ОВ-6).
- **Промо/скидка как самостоятельный подтип с бейджем** — в этом релизе только если решено по ОВ-7; иначе позже.
- **Внешняя ссылка / произвольный CTA без действия** для информационных постов — не входит (свободный пост покрывает информационный сценарий).
- Изменение источника данных каталога, добавление новых тяжёлых запросов к YCLIENTS — запрещено (используем уже загружаемые `SERVICES_DATA`/`catMap`).
- Изменение механизма GitHub-синхронизации как такового — переиспользуется (`_ghSyncPosts`/`_ghPullToLocal`), меняется только состав переносимых полей.
- Изменение логики экрана услуги `s-service` (ADR-013) и фильтрации каталога `filterCategory` (`services.js:61-68`) — переиспользуются как есть.
- Перенос захардкоженного YCLIENTS-токена из `prototype.html` (известный техдолг — «не усугублять»).

## 3. Действующие лица и сценарии использования

**Администратор студии** — создаёт публикации через админ-экран (`#s-admin-post`).

Сценарий A (пост по услуге — эталон):
1. Администратор создаёт пост, выбирает тип «По услуге».
2. Выбирает/ищет реальную услугу из каталога YCLIENTS (например «Стрижка»), вводит текст («Акция на стрижку 20%») и изображение.
3. Публикует. У поста сохраняется ссылка на конкретную услугу (`serviceId`) и переносится при синхронизации.

Сценарий B (пост по категории):
1. Администратор выбирает тип «По категории», выбирает реальную категорию YCLIENTS.
2. Публикует пост.

Сценарий C (свободный пост):
1. Администратор выбирает тип «Свободный», вводит текст/изображение без привязки.
2. Публикует.

**Клиент студии** — читает ленту и переходит к записи.

Сценарий D (запись с поста по услуге — эталон):
1. Клиент видит пост «Акция на стрижку 20%» в ленте / на главной / в одиночной карточке.
2. Жмёт «Записаться».
3. Открывается карточка **именно услуги «Стрижка»** (`s-service`).

Сценарий E (переход с поста по категории):
1. Клиент жмёт CTA на посте по категории.
2. Открывается каталог `s-services`, отфильтрованный по этой реальной категории; результат не пуст (если категория существует).

Сценарий F (фильтр ленты):
1. Клиент открывает `s-feed`, выбирает фильтр по типу поста (набор — ОВ-4).
2. Лента показывает только посты соответствующего типа/среза.

Граничные сценарии:
- Привязанная услуга/категория удалена в YCLIENTS — CTA не падает и ведёт себя по разумному фолбэку (ОВ-8).
- Старый пост с хардкод-строкой `cat` — отображается без ошибок, CTA по правилу совместимости (ОВ-2).
- Свободный пост — CTA по решению ОВ-3.

## 4. Функциональные требования

- **ФТ-1.** Администратор может создать пост одного из трёх типов: свободный (без привязки), по категории YCLIENTS, по услуге YCLIENTS.
- **ФТ-2.** Для типа «по категории» категория выбирается из **реальных категорий** YCLIENTS, а не из хардкод-чипов; источник — уже загруженные при старте данные (`catMap`/`SERVICES_DATA`).
- **ФТ-3.** Для типа «по услуге» услуга выбирается/ищется из **реального каталога** YCLIENTS (`SERVICES_DATA`); у поста сохраняется устойчивый идентификатор услуги (`serviceId` из `SERVICES_DATA.id`).
- **ФТ-4.** Модель поста расширяется полями, однозначно задающими тип и привязку (тип поста + ссылка на категорию и/или услугу с данными для отображения). Конкретный состав полей и судьба существующего поля `cat` (от него зависит визуальный градиент/иконка, `feed.js:5-13`) определяются архитектором.
- **ФТ-5.** Новые поля поста сохраняются при публикации, переносятся в GitHub-синхронизации (`posts.json`, `_ghSyncPosts`) и корректно читаются при загрузке (`_ghPullToLocal`).
- **ФТ-6.** В карточке поста **по услуге** CTA «Записаться» ведёт на карточку **именно этой** услуги (`openServiceCard(serviceId)` → `s-service`).
- **ФТ-7.** В карточке поста **по категории** CTA ведёт в каталог `s-services`, отфильтрованный по соответствующей реальной категории (`filterCategory(realCat)`), без ложного «Ничего не найдено» при существующей категории.
- **ФТ-8.** В карточке поста **свободного** типа CTA ведёт себя согласно решению ОВ-3 (скрыт либо нейтральный CTA).
- **ФТ-9.** Текст/метка CTA соответствует типу поста (например «Записаться →» для услуги; формулировка для категории/свободного — продуктовое решение в рамках ОВ-3/ОВ-4); все тексты на русском языке.
- **ФТ-10.** Если привязанная услуга или категория отсутствует в текущих данных YCLIENTS, CTA не вызывает ошибку в консоли и применяет разумный фолбэк (ОВ-8).
- **ФТ-11.** В ленте `s-feed` доступен фильтр по типам постов; набор измерений фильтра (только тип / тип + категория / комбинированный) и судьба текущего фильтра по `cat` определяются решением ОВ-4.
- **ФТ-12.** Уже опубликованные посты (только старое поле `cat`) отображаются во всех текущих местах (`s-feed`, `s-post`, превью на главной) без ошибок; их трактовка по типу/CTA — по решению ОВ-2.
- **ФТ-13.** Поведение, описанное в ФТ-6…ФТ-8, единообразно во всех местах рендера карточки поста (общий `_feedCardHtml`: список `s-feed` и одиночный `s-post`, ADR-010), а также не ломает превью на главной (`_renderHomeFeedPreview`).

## 5. Нефункциональные требования

- **НФТ-1 (Совместимость стека).** Решение остаётся в рамках vanilla JS/HTML/CSS, код в `docs/` (`modules/feed.js`, `modules/admin.js`, `modules/services.js`, `prototype.html`, при необходимости `style.css`); статический хостинг на GitHub Pages; без новых рантайм-npm-зависимостей.
- **НФТ-2 (Единый источник данных).** Категории и услуги для админ-формы берутся из уже загружаемых `SERVICES_DATA`/`catMap`; новые/тяжёлые запросы к YCLIENTS не вводятся.
- **НФТ-3 (Совместимость синхронизации).** Формат `posts.json` расширяется обратносовместимо: новые поля добавляются так, чтобы посты, опубликованные старой версией, оставались читаемыми, а новые — корректно синхронизировались между устройствами.
- **НФТ-4 (Минимальность изменений).** Изменения строго в согласованном объёме; экран услуги `s-service`, фильтрация каталога `filterCategory`, одиночный пост `s-post`, превью на главной и админка не должны регрессировать.
- **НФТ-5 (Устойчивость).** Несуществующие/удалённые привязки не приводят к ошибкам в консоли и пустым экранам без объяснения (ФТ-10, ОВ-8).
- **НФТ-6 (PWA/офлайн не регрессируют).** Изменения не ломают устанавливаемость PWA и офлайн-поведение; при изменении состава `APP_SHELL`/кэшируемых файлов (`prototype.html`, `feed.js`, `admin.js`, `services.js`, `style.css` и т. п.) поднимается `CACHE_VERSION` в `docs/sw.js` (правило проекта).
- **НФТ-7 (Локализация).** Все пользовательские тексты — на русском языке, в едином тоне приложения.

## 6. Ограничения и допущения

**Ограничения:**
- Стек и размещение: vanilla JS/HTML/CSS, код в `docs/`; статический хостинг GitHub Pages; запрещены рантайм-npm-зависимости, коммит секретов, любые операции с веткой `backup`.
- Источник категорий/услуг — уже загружаемые при старте `SERVICES_DATA`/`catMap` (`app.js:166-200`); добавлять новые запросы к YCLIENTS нельзя.
- Переход на услугу — через существующий `openServiceCard(serviceId)` (`services.js:283-344`, ADR-013); на категорию — через `filterCategory` + `s-services` (`services.js:61-68`).
- Синхронизация постов — через существующий механизм GitHub (`posts.json`); меняется только состав полей.
- Общий рендер карточки поста (`_feedCardHtml`, `feed.js:54-74`) используется и в `s-feed`, и в `s-post` (ADR-010) — изменения CTA/полей затрагивают оба места одновременно.
- При изменении файлов из `APP_SHELL`/`sw.js` — поднятие `CACHE_VERSION`.

**Допущения:**
- `SERVICES_DATA.id` — устойчивый идентификатор услуги, пригодный для адресации привязки поста (как в ADR-013); подтверждает архитектор.
- Реальные категории однозначно адресуются названием (`cat`/`title` из `catMap`), как этого требует существующий `filterCategory(cat)`; стабильность по `category_id` при необходимости подтверждает архитектор.
- Объём данных категорий/услуг (десятки-сотни) пригоден для выбора/поиска в админ-форме без пагинации и без дополнительных запросов.
- Окончательная форма UI админ-формы (переключатель типа, селект категории, поиск услуги), точный набор полей модели и судьба поля `cat` фиксируются архитектором после решения продуктовых развилок §8.

## 7. Критерии приёмки

- **КП-1.** Администратор может создать пост каждого из трёх типов: свободный, по категории, по услуге; категории и услуги выбираются из реальных данных YCLIENTS, а не из хардкод-чипов. (ФТ-1, ФТ-2, ФТ-3)
- **КП-2.** Пост «по услуге» («Акция на стрижку 20%») по кнопке «Записаться» открывает карточку именно услуги «Стрижка» (`s-service`) — сценарий работает end-to-end. (ФТ-6, ФТ-13)
- **КП-3.** Пост «по категории» по CTA открывает каталог, отфильтрованный по реальной категории; при существующей категории результат не пуст («Ничего не найдено» не появляется). (ФТ-7)
- **КП-4.** Свободный пост публикуется без привязки; его CTA ведёт себя согласно решению ОВ-3. (ФТ-8)
- **КП-5.** Новые поля поста сохраняются при публикации, корректно переносятся через GitHub-синхронизацию (`posts.json`) и читаются на другом устройстве. (ФТ-4, ФТ-5, НФТ-3)
- **КП-6.** В ленте `s-feed` работает фильтр по типам постов согласно решению ОВ-4. (ФТ-11)
- **КП-7.** Если привязанная услуга/категория удалена в YCLIENTS, CTA не вызывает ошибки в консоли и применяет разумный фолбэк. (ФТ-10, НФТ-5)
- **КП-8.** Уже опубликованные (старые) посты отображаются в `s-feed`, `s-post` и превью на главной без ошибок; их CTA ведёт себя по правилу совместимости ОВ-2. (ФТ-12, ФТ-13)
- **КП-9.** Поведение карточки поста единообразно в `s-feed` и `s-post`; превью ленты на главной не сломано. (ФТ-13, НФТ-4)
- **КП-10.** PWA остаётся устанавливаемой, офлайн-поведение не регрессирует; при изменении файлов `APP_SHELL` поднята `CACHE_VERSION`; новые рантайм-зависимости не добавлены; в консоли нет ошибок. (НФТ-1, НФТ-6)
- **КП-11.** Все затронутые пользовательские тексты — на русском языке. (ФТ-9, НФТ-7)

## 8. Открытые вопросы

Блокирующих оформление настоящего документа вопросов **нет**. Перечисленные ниже вопросы — продуктовые/UX-развилки и технические уточнения для архитектора. **ОВ-1, ОВ-5, ОВ-6, ОВ-8** имеют достаточные рекомендации и **не блокируют** запуск `/adr` (требуют лишь подтверждения/проектирования архитектором). **ОВ-2, ОВ-3, ОV-4, ОВ-7** — продуктовые решения заказчика, определяющие поведение и объём; их **рекомендуется зафиксировать до запуска `/adr`**, поскольку они влияют на scope и UX (заказчик может согласовать предложенные рекомендации либо изменить их).

- **ОВ-1. Источник категорий для админки.** *Рекомендация:* брать реальные категории из уже загруженных `SERVICES_DATA` (`new Set(s.cat)`) / `catMap`, отдельный справочник не вводить (соответствует НФТ-2). *Блокирует `/adr`:* нет (подтверждение архитектора при проектировании).
- **ОВ-2. Обратная совместимость существующих постов.** Старые посты имеют только `cat` (хардкод-строки, часто не совпадают с реальными категориями). *Рекомендация:* трактовать старые посты как «свободные» (CTA по ОВ-3) до перепубликации администратором — это безопаснее, чем угадывать категорию маппингом строк, и исключает ложный переход. Опционально архитектор может предложить мягкий маппинг точного совпадения строки `cat` с реальной категорией. *Блокирует `/adr`:* нет, но **решить до `/adr`** — определяет поведение CTA уже опубликованных постов. *Влияет на ФТ-12, КП-8.*
- **ОВ-3. CTA у свободного поста.** Скрывать кнопку полностью или показывать нейтральный CTA («Все услуги» → `s-services` без фильтра)? *Рекомендация:* **скрывать CTA** у свободного поста (информационный пост не должен навязывать запись; меньше визуального шума). *Блокирует `/adr`:* нет, но **решить до `/adr`** — определяет ФТ-8/ФТ-9. *Влияет на ФТ-8, ФТ-9, КП-4.*
- **ОВ-4. Измерения фильтра ленты.** Фильтр строго по типу поста, по категории или комбинированный? Текущий фильтр строится из `new Set(posts.map(p => p.cat))` (`feed.js:87-109`). *Рекомендация (v1):* заменить текущий фильтр на **фильтр по типу поста** («Все» / «Услуги и акции» / «Новости»); комбинацию с категорией отложить, чтобы не раздувать объём. *Блокирует `/adr`:* нет, но **решить до `/adr`** — определяет ФТ-11 и судьбу текущего фильтра. *Влияет на ФТ-11, КП-6.*
- **ОВ-5. Пост, привязанный к мастеру.** *Рекомендация:* **вне scope** этого релиза, оформить отдельным требованием. *Блокирует `/adr`:* нет (зафиксировано в §2 «Вне рамок»).
- **ОВ-6. Несколько услуг в одном посте.** *Рекомендация:* **вне scope**, одна услуга на пост. *Блокирует `/adr`:* нет (зафиксировано в §2 «Вне рамок»).
- **ОВ-7. Промо/скидка как подтип/бейдж.** Вводить ли в этом релизе отдельный флаг промо (`promo` + текст скидки, бейдж «−20%») поверх привязки? *Рекомендация:* **в v1 не вводить отдельный механизм промо** — эталонный кейс «Акция на стрижку 20%» полностью покрывается типом «по услуге» + текстом поста; бейдж/подтип акции отложить в отдельное требование. *Блокирует `/adr`:* нет, но **решить до `/adr`** — определяет объём (наличие/отсутствие полей промо). *Влияет на §2, ФТ-4.*
- **ОВ-8. Удалённая/несуществующая услуга или категория.** Поведение CTA, если привязка из поста больше не существует в YCLIENTS. *Рекомендация:* не падать; **фолбэк — открыть каталог `s-services` без фильтра** (для поста по услуге/категории), либо скрыть CTA — точную механику определяет архитектор; пост при этом отображается нормально. *Блокирует `/adr`:* нет (нижняя граница задана ФТ-10/НФТ-5; механику выбирает архитектор).

**Противоречий с принятыми ADR не выявлено.** Задача опирается на действующие решения и согласуется с ними: ADR-010 (отдельный экран одной публикации `s-post` с общим рендером карточки и мягким фолбэком) — изменения CTA/полей должны единообразно работать в `s-feed` и `s-post`; ADR-013 (детальная карточка услуги `s-service`, `openServiceCard`) — целевой экран для CTA поста по услуге; ADR-001 (визуальный каталог услуг) и ADR-002 (источник связи «мастер↔услуга») — источник данных каталога. Если при проектировании изменение модели поста или формата `posts.json` затронет действующие решения по ленте/синхронизации, это должно быть отражено в новом/обновлённом ADR (вход архитектора, не блокирующий BA).
