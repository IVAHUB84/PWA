# BA-требования 030: Drill-down записей и клиентов в админке (детали + быстрые действия)

- **Источник:** `requirements/req-admin-drilldown-029.md`
- **Дата:** 2026-05-30
- **Статус:** черновик

> Примечание о нумерации: исходное требование имеет номер 029, но в `ba-req/` номер 029 уже занят (`ba-requirements-029.md`). Поэтому данный документ оформлен под следующим свободным номером — **030**. Это нормальная для проекта ситуация: номер BA-документа не обязан совпадать с номером `req-*`.

## 1. Контекст и бизнес-цель

Администратор студии работает с двумя экранами админ-панели PWA:
- **«Сводка»** (`s-admin`) — список записей на сегодня (`renderAdminDashboard`).
- **«Клиенты»** (`s-admin-clients`) — база клиентов с поиском (`renderAdminClients` / `_renderClientRows`).

Сейчас оба экрана — «только для чтения»: строки записей (`.booking-row`) и клиентов (`.client-row`) некликабельны. Чтобы выполнить любое действие — позвонить клиенту, посмотреть полные детали записи — администратор вынужден уходить в веб-кабинет YCLIENTS. Для оперативной работы «на ходу» (телефон в руке) это тупиковые экраны.

**Бизнес-цель:** сократить путь администратора от «вижу запись/клиента в PWA» до «связался с клиентом» до одного-двух тапов, не выходя из приложения. В этом релизе — только просмотр деталей и связь (звонок); управление записями (отмена/перенос) намеренно вне объёма.

**Стейкхолдеры:** администратор студии (основной пользователь админ-панели); владелец студии (как заказчик функции).

## 2. Область (scope)

### В рамках задачи
- Строки записей в «Сводке» становятся кликабельными → открывают bottom-sheet с деталями записи.
- Строки клиентов в «Базе» становятся кликабельными → открывают bottom-sheet с деталями клиента.
- В шите записи — детали: клиент, услуга, мастер, время, стоимость; быстрое действие: звонок клиенту (`tel:`), если телефон доступен.
- В шите клиента — детали: имя, телефон, число визитов; быстрое действие: звонок (`tel:`).
- Телефон в шите — выделяемый и копируемый (обход глобального `user-select:none`, по аналогии с блоком контактов из BA-027).
- Закрытие шита тапом по фону и по явной кнопке/жесту.

### Вне рамок (explicitly out of scope)
- Отмена, перенос, создание или удаление записи (любые деструктивные/изменяющие операции через API) — отдельное требование.
- Редактирование карточки клиента.
- Отправка сообщений (SMS/мессенджеры).
- Глубокая интеграция/deep-link в YCLIENTS (действие «Открыть в YCLIENTS», ФТ-7) — **исключена из v1.25.0** (см. ОВ-2 РЕШЁН): возможность сформировать стабильный URL не подтверждена, требует отдельного исследования.
- Дополнительные сетевые запросы на каждую строку записи ради дозагрузки телефона (ограничение Огр-3). Телефон уже приходит в `/records` (ОВ-1 РЕШЁН), доп. запросы не нужны.

## 3. Действующие лица и сценарии использования

**Действующее лицо:** администратор студии, авторизованный в админ-панели PWA, как правило с мобильного устройства.

**Сценарий A — звонок по записи (Сводка):**
1. Администратор открывает «Сводку», видит записи на сегодня.
2. Тапает по строке записи.
3. Снизу выезжает шит с деталями записи (клиент, услуга, мастер, время, стоимость).
4. Если у клиента есть телефон — тапает «Позвонить» → системный диалер с номером.
5. Закрывает шит тапом по фону.

**Сценарий B — звонок клиенту (База):**
1. Администратор открывает «Клиенты», при необходимости ищет клиента (существующий поиск сохраняется).
2. Тапает по строке клиента.
3. Снизу выезжает шит с деталями (имя, телефон, число визитов).
4. Тапает «Позвонить» → системный диалер; либо выделяет/копирует номер вручную.
5. Закрывает шит.

**Сценарий C — клиент без телефона:** в данных нет номера → кнопка «Позвонить» не отображается (или неактивна), детали показываются, шит остаётся информативным, без «мёртвой» кнопки.

## 4. Функциональные требования

- **ФТ-1.** Строка записи в «Сводке» реагирует на тап и открывает bottom-sheet с деталями этой записи.
- **ФТ-2.** Шит записи отображает: имя клиента, название услуги, имя мастера, время, стоимость. Поля, отсутствующие в данных, отображаются плейсхолдером «—», без ошибок рендера.
- **ФТ-3.** (обязательное; зависимость от ОВ-1 снята — телефон подтверждён в `/records`, см. ОВ-1 РЕШЁН) Шит записи показывает действие «Позвонить» со ссылкой `tel:` на нормализованный номер `rec.client.phone`. Если телефон у конкретной записи пуст или отсутствует — действие звонка не показывается (graceful degradation, см. Сценарий C). Дополнительные запросы (`/clients`) не выполняются.
- **ФТ-4.** Строка клиента в «Базе» реагирует на тап и открывает bottom-sheet с деталями клиента.
- **ФТ-5.** Шит клиента отображает: имя, телефон, число визитов; и действие «Позвонить» (`tel:`) при наличии телефона.
- **ФТ-6.** В обоих шитах телефонный номер можно выделить и скопировать (текст телефона исключён из глобального `user-select:none`).
- **ФТ-7. (ВНЕ ОБЪЁМА v1.25.0; решение по ОВ-2).** Действие «Открыть в YCLIENTS» (deep-link на запись/клиента в веб-кабинете) в данном релизе **не реализуется**: возможность сформировать стабильный URL не подтверждена и требует отдельного исследования; включение противоречит принципу минимального объёма. Требование зафиксировано как кандидат на будущий релиз.
- **ФТ-8.** Шит закрывается тапом по затемнённому фону и по явному элементу закрытия (кнопка/жест). Закрытие не нарушает прокрутку страницы и работу нижней навигации (tab-bar).
- **ФТ-9.** Существующий поиск клиентов (`filterAdminClients`) и текущий вид строк сохраняются: добавление кликабельности не ломает фильтрацию и повторный рендер строк.
- **ФТ-10.** Все пользовательские данные в шите (имя, телефон, услуга, мастер) экранируются перед выводом; внешние ссылки (если будут) открываются безопасно (`rel="noopener"`).

## 5. Нефункциональные требования

- **НФТ-1. Совместимость / отсутствие зависимостей.** Реализация на vanilla HTML/CSS/JS, без новых рантайм-зависимостей; код в пределах `docs/`.
- **НФТ-2. Переиспользование UI.** Использовать уже сложившийся в проекте паттерн bottom-sheet (динамически создаваемый overlay-блок, выезжающий снизу, закрытие тапом по фону — как в `services.js`, `search.js`, `history.js`), а не вводить новый параллельный компонент. См. ОВ-3 и раздел «Ограничения и допущения».
- **НФТ-3. Доступность ввода/звонка.** Телефон в шите — кликабельный (`tel:`) и одновременно выделяемый/копируемый; визуально читаемый в светлой и тёмной теме.
- **НФТ-4. PWA-целостность.** После изменений приложение остаётся устанавливаемым; в консоли нет ошибок. При изменении `APP_SHELL` / `sw.js` поднимается `CACHE_VERSION`.
- **НФТ-5. Производительность.** Открытие шита не инициирует тяжёлых сетевых операций по умолчанию; недопустимы дополнительные запросы на каждую строку при рендере списка (см. Огр-3).

## 6. Ограничения и допущения

**Ограничения:**
- **Огр-1.** Только просмотр + связь. Любые деструктивные/изменяющие операции (отмена, перенос, удаление записи) исключены без отдельного согласования.
- **Огр-2.** Не вводить новый компонент модалки/шита, если применим существующий паттерн проекта. CSS-классы `.sheet` / `.sheet-overlay` в `style.css` относятся к демо-сценариям (десктопная media-query, модуль `scenarios.js`) и **не** являются переиспользуемым мобильным компонентом; целевой паттерн — динамический overlay-блок из `history.js` / `services.js` / `search.js`.
- **Огр-3.** Запрещены лишние запросы: телефон для шита записи берётся из того, что уже есть в ответе `/records`; не делать запрос `/clients` на каждую запись.
- **Огр-4.** Глобальное правило `body { user-select:none }` (введено в v1.20.0) требует точечного исключения для телефона в шите — по образцу `.studio-contacts { user-select:text }` (req-027 / BA-027).

**Допущения:**
- **Доп-1.** Структура `rec.client` в ответе `/records` содержит `id`, `name` и `phone` (подтверждено на боевом API 2026-05-30, ОВ-1 РЕШЁН): телефон — полный номер, напр. `"79268513443"`. Шит записи всё равно проектируется устойчиво к пустому/отсутствующему `phone` у отдельных записей (graceful degradation).
- **Доп-2.** Для шита клиента телефон гарантированно доступен: `renderAdminClients` уже маппит `phone: c.phone || ''` из `/clients`.
- **Доп-3.** Число визитов клиента доступно (`visits_count` → `visits`), уже используется в рендере строк.

## 7. Критерии приёмки

- **КП-1.** (ФТ-1, ФТ-2) Тап по строке записи в «Сводке» открывает шит, в котором видны клиент, услуга, мастер, время и стоимость; отсутствующие поля показаны как «—», ошибок в консоли нет.
- **КП-2.** (ФТ-3) В шите записи с непустым `client.phone` присутствует кнопка «Позвонить», тап по которой вызывает системный диалер с корректным номером (без дополнительных сетевых запросов). Если у записи телефон пуст/отсутствует — кнопка звонка отсутствует, остальное содержимое шита отображается корректно.
- **КП-3.** (ФТ-4, ФТ-5) Тап по строке клиента в «Базе» открывает шит с именем, телефоном и числом визитов и рабочей кнопкой «Позвонить» (`tel:`).
- **КП-4.** (ФТ-6, НФТ-3) Телефон в обоих шитах можно выделить и скопировать, а также нажать для звонка.
- **КП-5.** (ФТ-8) Шит закрывается тапом по фону и по элементу закрытия; после закрытия прокрутка страницы и нижняя навигация работают как прежде.
- **КП-6.** (ФТ-9) Поиск клиентов продолжает работать; кликабельность строк сохраняется после фильтрации и повторного рендера.
- **КП-7.** (ФТ-10, НФТ-1) Данные экранируются; новых рантайм-зависимостей нет.
- **КП-8.** (НФТ-4) PWA остаётся устанавливаемой; в консоли нет ошибок; при изменении `APP_SHELL`/`sw.js` поднята `CACHE_VERSION`.
- **КП-9.** (ФТ-7) Действие «Открыть в YCLIENTS» в v1.25.0 отсутствует — это соответствует решению по ОВ-2 (вне объёма) и не считается дефектом.

## 8. Открытые вопросы

- **ОВ-1 — РЕШЁН (2026-05-30, проверено на боевом API).** Телефон клиента доступен прямо в ответе `/records`: `GET /api/v1/records/1940850` возвращает в каждой записи `client.phone` (полный номер, напр. `"79268513443"`) наряду с `client.id` и `client.name`. Дополнительный запрос `/clients` не требуется, ограничение Огр-3 не нарушается. **Следствие:** ФТ-3 переведено в обязательное; зависимость снята; блокировки `/adr` нет.
- **ОВ-2 — РЕШЁН (2026-05-30).** Действие «Открыть в YCLIENTS» (deep-link) **не включаем в v1.25.0**: возможность сформировать стабильный URL записи/клиента не подтверждена и требует отдельного исследования, а релиз держим в минимальном объёме. ФТ-7 помечено как вне объёма; зафиксировано как кандидат на будущий релиз. Не блокирует.
- **ОВ-3 — рекомендация (не блокирует).** Переиспользовать существующий динамический overlay-паттерн (`history.js` / `services.js` / `search.js`), не вводить новый компонент и не задействовать демо-классы `.sheet`. Финальное решение и возможное выделение общего хелпера — за архитектором.

> **Блокирующих открытых вопросов нет** — переход к `/adr` разрешён. Противоречий с действующими ADR не выявлено. Требование развивает линию req-027 (контактный блок, кликабельный/копируемый телефон) и не затрагивает деструктивные операции, ограниченные политикой проекта.
