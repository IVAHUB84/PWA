# BA-требования 008: Прямой переход с выбора мастера на выбор времени

- **Источник:** `requirements/req-booking-skip-master-card-008.md`
- **Дата:** 2026-05-28
- **Статус:** черновик

## 1. Контекст и бизнес-цель

Studio PWA (студия «Реснички, сестрички», Троицк) — устанавливаемое приложение онлайн-записи поверх YCLIENTS, отдаётся статически из `docs/` на GitHub Pages. Сценарий записи: услуга → мастер → время → подтверждение.

**Подтверждено кодом.** В потоке записи после выбора услуги клиент попадает на список мастеров `s-masters`. Тап по мастеру вызывает `openMasterCard` (`docs/modules/masters.js:16`) и открывает промежуточный экран-профиль мастера `s-master`; только оттуда по кнопке «Записаться к этому мастеру» (`bookFromMaster` → `bookWithMaster`) клиент идёт дальше. То есть между «выбрал мастера» и «выбираю время» вставлен лишний экран.

Бизнес-цель: сократить путь записи — тап по мастеру в списке должен вести **сразу** на экран «Выберите время» (`s-slots`), без промежуточной карточки-профиля. Это уменьшает число шагов до брони и приводит поведение «конкретный мастер» в соответствие с уже работающим «Любой мастер» (`selectAnyMaster`, `masters.js:236`), который и сейчас ведёт прямо на `s-slots`.

**Подтверждено кодом (технических предпосылок достаточно):**
- Функция прямого перехода уже существует: `selectMaster(id)` (`masters.js:112`) ставит `state.masterId` и делает `go('s-slots')`, но в списке не используется.
- Экраны `s-slots` (`slots.js:15`), подтверждения `s-confirm` (`slots.js:67`) и отмены (`booking.js:175`) берут данные мастера через `getMaster()` (`state.js:56`), который восстанавливает мастера из `MASTERS_DATA` по `state.masterId`. То есть для корректного показа имени/аватара/роли на этих экранах достаточно `state.masterId`; поля `state.masterName/masterAvatar/masterGrad` им не требуются.

**Стейкхолдеры:**
- Заказчик / владелец студии — заинтересован в более коротком и быстром сценарии записи.
- Клиент студии — пользователь сценария записи; получает меньше «лишних» экранов на пути к брони.
- Бизнес-аналитик / архитектор / разработчик — оформление, проектирование, реализация.

## 2. Область (scope)

### В рамках задачи
- В сценарии записи (услуга уже выбрана) тап по доступному мастеру в списке `s-masters` ведёт **сразу** на экран «Выберите время» (`s-slots`), без промежуточного показа карточки-профиля `s-master`.
- На экране «Выберите время» и на экране подтверждения при таком прямом переходе корректно отображаются имя, роль и аватар выбранного мастера (паритет с прежним путём через карточку).
- Поведение варианта «Любой мастер» сохраняется: он по-прежнему ведёт сразу на `s-slots` без выбора конкретного мастера.

### Вне рамок (explicitly out of scope)
- **Изменение содержимого экрана «Выберите время»** (`s-slots`), логики загрузки дат/слотов (`loadDates`, YCLIENTS `book_dates`/`book_times`), экрана подтверждения и самой брони (`book_record`) — не затрагиваются, меняется только точка входа на `s-slots`.
- **Фильтрация состава списка мастеров по услуге** (ADR-002, `renderMasters`, пересечение с `MASTERS_DATA`) — не пересматривается.
- **Режим просмотра «Все мастера студии»** (`_browseAllMasters`/`browseAll`) и точки входа в карточку мастера вне потока записи (главный экран «Наши мастера» `profile.js:73`, поиск `search.js:19`) — изменяются только в части, определяемой решением по открытым вопросам №1 и №2; по умолчанию (см. допущения) их поведение сохраняется.
- **Функция чаевых, отзывов, push, фид, тема оформления** — не затрагиваются.
- **Перенос захардкоженного YCLIENTS-токена** из `prototype.html` (известный техдолг — «не усугублять», не трогать).

## 3. Действующие лица и сценарии использования

**Клиент студии** — авторизован (или авторизуется по ходу брони) в PWA, проходит сценарий онлайн-записи.

Основной сценарий (запись к конкретному мастеру):
1. Клиент выбирает услугу → открывается список мастеров `s-masters` для этой услуги.
2. Клиент тапает по доступному мастеру.
3. Сразу открывается экран «Выберите время» (`s-slots`); промежуточная карточка-профиль `s-master` не показывается.
4. На `s-slots` видно имя, роль и аватар выбранного мастера; клиент выбирает дату и время и идёт к подтверждению.
5. На экране подтверждения мастер отображается корректно.

Сценарий «Любой мастер»:
1. На списке мастеров клиент выбирает «Любой мастер».
2. Сразу открывается `s-slots` без привязки к конкретному мастеру (как сейчас).

Граничные сценарии:
- **Недоступный мастер** в списке (карточка с `opacity` и без обработчика тапа, `masters.js:14,16`) — не выбирается; поведение не меняется.
- **Просмотр мастера вне записи** (главный экран / поиск / «Все мастера студии», когда услуга не выбрана) — клиент хочет посмотреть профиль и отзывы мастера, а не записаться немедленно; ожидаемое поведение тапа здесь определяется открытыми вопросами №1 и №2.

## 4. Функциональные требования

- **ФТ-1.** В сценарии записи (услуга выбрана) тап по доступному мастеру в списке `s-masters` открывает экран «Выберите время» (`s-slots`) напрямую, без промежуточного показа экрана-профиля `s-master`.
- **ФТ-2.** При таком прямом переходе на экране «Выберите время» корректно отображаются имя, роль и аватар выбранного мастера (визуально эквивалентно прежнему пути через карточку).
- **ФТ-3.** При таком прямом переходе на экране подтверждения корректно отображаются имя и роль выбранного мастера.
- **ФТ-4.** Бронь, созданная через прямой переход, привязана к тому же мастеру, что был выбран в списке (тот же `staff_id` уходит в YCLIENTS, что и при прежнем пути).
- **ФТ-5.** Вариант «Любой мастер» сохраняет текущее поведение: ведёт на `s-slots` без выбора конкретного мастера, и далее по потоку отображается как «Любой свободный мастер».
- **ФТ-6.** Просмотр профиля мастера (с отзывами) остаётся доступен из точек входа вне сценария записи в объёме, определяемом решениями по открытым вопросам №1 и №2 (по умолчанию — сохраняется без изменений).
- **ФТ-7.** Все пользовательские тексты, затронутые изменением, — на русском языке, в едином тоне приложения.

## 5. Нефункциональные требования

- **НФТ-1 (Совместимость стека).** Решение остаётся в рамках стека: vanilla JS/HTML/CSS, код в `docs/` (`docs/modules/masters.js` и связанные), статический хостинг на GitHub Pages; без новых рантайм-npm-зависимостей.
- **НФТ-2 (Целостность YCLIENTS-интеграции).** Изменение не должно ломать получение мастеров по услуге (ADR-002), получение дат/слотов и создание записи; состав запросов к YCLIENTS в сценарии записи не меняется.
- **НФТ-3 (Достоверность отображения мастера).** Имя/роль/аватар мастера, показанные на `s-slots` и `s-confirm` при прямом переходе, должны соответствовать фактически выбранному и фактически бронируемому мастеру (без рассинхрона между UI и `state.masterId`).
- **НФТ-4 (PWA/офлайн не регрессируют).** Изменения не ломают устанавливаемость PWA и офлайн-поведение Service Worker; при изменении состава `APP_SHELL` / кэшируемых файлов (`masters.js`, `prototype.html`, `constants.js` и т. п.) поднимается `CACHE_VERSION` в `docs/sw.js` (правило проекта).
- **НФТ-5 (Локализация).** Все пользовательские тексты — на русском языке.
- **НФТ-6 (Минимальность изменений).** Изменения строго в объёме задачи; не затрагивать сценарии, отмеченные «вне рамок».

## 6. Ограничения и допущения

**Ограничения:**
- Приложение — статический PWA в `docs/` на GitHub Pages; код сценария — в `docs/modules/masters.js`, `slots.js`, `booking.js`, `state.js`, разметка экранов — в `docs/prototype.html`.
- Список мастеров на `s-masters` в режиме «от услуги» строится из пересечения ответа YCLIENTS с `MASTERS_DATA` (ADR-002) — то есть выбранный в этом списке мастер гарантированно присутствует в `MASTERS_DATA`, откуда `getMaster()` берёт имя/роль/аватар для `s-slots`/`s-confirm`.
- Карточка-профиль `s-master` (`openMasterCard`) — самостоятельный экран, читающий профиль мастера, его услуги и **отзывы** через YCLIENTS (`GET /comments`, ADR-003); используется не только в потоке записи, но и из главного экрана (`profile.js:73`), поиска (`search.js:19`) и режима «Все мастера студии» (`browseAll`).
- Экран `s-master` зарегистрирован в `SCREENS` (`constants.js:8`).
- Запрещены рантайм-npm-зависимости; запрещён коммит секретов; запрещены любые операции с веткой `backup`.
- При изменении файлов в `APP_SHELL` — поднятие `CACHE_VERSION` в `docs/sw.js`.

**Допущения:**
- Для корректного отображения мастера на `s-slots`/`s-confirm` достаточно установки `state.masterId` (как делает существующий `selectMaster`), поскольку эти экраны читают мастера через `getMaster()` из `MASTERS_DATA`; дозаполнение `state.masterName/masterAvatar/masterGrad` для них не обязательно. Эти поля нужны лишь баннеру входа «от мастера» на `s-services` (`services.js:15`), который данной задачей не затрагивается. **Достаточность подтверждает архитектор** при проектировании (см. открытый вопрос №3).
- По умолчанию (до решения заказчика по вопросам №1/№2) поведение тапа по мастеру **вне** сценария записи (услуга не выбрана: главный экран, поиск, «Все мастера студии») сохраняется — открывается карточка-профиль `s-master`. Прямой переход на время вводится только там, где услуга уже выбрана.
- Решение по «судьбе» экрана `s-master` (оставить как просмотр профиля или удалить совсем) принимает заказчик/архитектор (вопрос №1); при отсутствии явного решения экран **сохраняется** для просмотра вне записи, чтобы не потерять чтение отзывов на карточке (ADR-003).

## 7. Критерии приёмки

- **КП-1.** В сценарии записи после выбора услуги тап по доступному мастеру в списке открывает экран «Выберите время» напрямую; экран-профиль `s-master` в этом потоке не показывается. (ФТ-1)
- **КП-2.** На открывшемся экране «Выберите время» отображаются имя, роль и аватар именно выбранного мастера. (ФТ-2, НФТ-3)
- **КП-3.** На экране подтверждения отображаются имя и роль выбранного мастера; созданная запись привязана к этому мастеру (корректный `staff_id` в YCLIENTS). (ФТ-3, ФТ-4, НФТ-2, НФТ-3)
- **КП-4.** Вариант «Любой мастер» по-прежнему ведёт сразу на «Выберите время» и далее отображается как «Любой свободный мастер». (ФТ-5)
- **КП-5.** Просмотр профиля мастера с отзывами доступен из точек входа вне записи в объёме принятого решения по вопросам №1/№2 (по умолчанию — работает как прежде). (ФТ-6)
- **КП-6.** Получение мастеров по услуге, дат/слотов и создание записи не регрессируют; состав запросов к YCLIENTS в сценарии не изменился. (НФТ-2)
- **КП-7.** PWA остаётся устанавливаемой, офлайн-поведение не регрессирует; при изменении файлов `APP_SHELL` поднята `CACHE_VERSION`; новые рантайм-зависимости не добавлены. (НФТ-1, НФТ-4)
- **КП-8.** Затронутые пользовательские тексты — на русском языке. (ФТ-7, НФТ-5)

## 8. Открытые вопросы

1. **Судьба экрана-профиля `s-master`.** Карточка убирается только из потока записи (остаётся доступной для просмотра мастера вне записи — главный экран / поиск / «Все мастера студии») или удаляется как экран совсем — вместе со связанным кодом (`openMasterCard`, `bookFromMaster`, регистрация `s-master` в `SCREENS`)? Важно: на `s-master` сейчас читаются и показываются **отзывы** мастера (`GET /comments`, ADR-003) и список его услуг; при полном удалении эта функция просмотра/чтения отзывов исчезнет. *Влияет на ФТ-6, КП-5, раздел 2. Продуктовое решение заказчика; механику — архитектор.*

2. **Поведение тапа по мастеру вне сценария записи (услуга не выбрана).** В режиме «Все мастера студии» (`browseAll`), а также из главного экрана «Наши мастера» (`profile.js:73`) и поиска (`search.js:19`) тап сейчас открывает профиль `s-master`. Должно ли это поведение сохраниться (просмотр профиля) или измениться? Прямой переход «на время» вне выбранной услуги невозможен (нет услуги для брони), поэтому по умолчанию предлагается сохранить открытие профиля. Подтвердить ожидание. *Влияет на ФТ-6, раздел 2; согласуется с ADR-002 (режим «Все мастера» — без изменений). Продуктовое решение заказчика.*

3. **Достаточность `state.masterId` для отображения мастера при прямом переходе.** Требование отмечает, что прежний путь через карточку дозаполнял `state.masterName/masterAvatar/masterGrad`. Анализ кода показывает, что `s-slots`/`s-confirm`/`s-cancel` берут мастера через `getMaster()` из `MASTERS_DATA` по `state.masterId`, и для них достаточно `state.masterId` (как в существующем `selectMaster`). Нужно, чтобы архитектор подтвердил, что для всех целевых экранов прямого перехода данных достаточно и не требуется дозаполнять `masterName/masterAvatar/masterGrad` (или явно зафиксировать, что дозаполнить их нужно для единообразия/будущих экранов). *Влияет на ФТ-2, ФТ-3, НФТ-3, КП-2, КП-3. Техническое подтверждение архитектора.*

### Решения по открытым вопросам (приняты 2026-05-28)

1. **Экран `s-master` сохраняется** (НЕ удаляется). Карточка-профиль + отзывы остаётся доступной для **просмотра вне записи** (режим «Все мастера», главный экран, поиск). Из потока записи она лишь **исключается** (тап ведёт на время).
2. **Поведение вне записи не меняется:** тап по мастеру в режиме просмотра (услуга не выбрана) по-прежнему открывает профиль `s-master`. Меняется только тап в потоке записи (услуга выбрана) → сразу `s-slots`. Реализация должна быть контекстно-зависимой.
3. (Техническое, подтверждает архитектор) Ожидается, что `state.masterId` достаточно (`getMaster()` читает из `MASTERS_DATA`); архитектор подтверждает или фиксирует необходимость дозаполнения `masterName/masterAvatar/masterGrad`.

Противоречий с принятыми ADR не выявлено: ADR-002 фиксирует фильтрацию состава `s-masters` и неизменность режима «Все мастера» — настоящая задача меняет только действие при тапе в потоке записи, не трогая состав списка; ADR-003 описывает чтение отзывов на `s-master` — оно затрагивается только при выборе «удалить экран совсем» (вопрос №1). ADR-001/004/005/006 данной темы не касаются.
