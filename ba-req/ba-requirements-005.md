# BA-требования 005: Предложение установить приложение на экран «Домой»

- **Источник:** `requirements/req-pwa-install-prompt-005.md`
- **Дата:** 2026-05-28
- **Статус:** черновик

## 1. Контекст и бизнес-цель

Studio PWA (студия «Реснички, сестрички», Троицк) — устанавливаемое PWA для онлайн-записи поверх YCLIENTS, отдаётся статически из `docs/` на GitHub Pages. Установка приложения на экран «Домой» даёт студии главный канал удержания: запуск в standalone-режиме, иконка на главном экране смартфона и фундамент для push-уведомлений о записи (push-инфраструктура уже существует — Cloudflare Worker, `docs/modules/push.js`).

Текущее состояние (фактическое, из кода):
- В приложении уже есть экран-инструкция `s-install` (`docs/prototype.html`) с **ручными шагами для iOS/Safari** («Поделиться → На экран «Домой» → Добавить»).
- На главной (`s-home`) есть баннер `#installBanner`, ведущий на `s-install`. Его видимость управляется флагом `localStorage` `yc_install_seen` (`docs/modules/profile.js`): баннер показывается, пока пользователь не нажал «Понятно, добавлю».

Чего сейчас **нет** (это и есть суть требования):
- Нет программного предложения установки для Android/Chrome через событие `beforeinstallprompt` (платформенный prompt не перехватывается и не используется).
- Нет проверки, что приложение уже запущено как установленное (standalone) — предложение не подавляется для уже установивших.
- Нет автоматического показа предложения при открытии — пользователь видит лишь пассивный баннер на главной.
- Нет различения платформ (Android-prompt против iOS-инструкции) и нет правила «не показывать повторно либо показывать раз в N дней».

**Бизнес-цель:** при открытии приложения на мобильном устройстве (если оно ещё не установлено) автоматически и в фирменном стиле предлагать добавить приложение на экран «Домой», доводя пользователя до установки на каждой поддерживаемой платформе, не раздражая повторными показами тех, кто уже установил или осознанно отказался.

**Стейкхолдеры:**
- Заказчик / владелец студии — заинтересован в росте числа установок (удержание, канал для push).
- Клиенты студии (женщины 20–40 лет) — конечные пользователи, видят предложение.
- Бизнес-аналитик / архитектор / разработчик — оформление и реализация.

## 2. Область (scope)

### В рамках задачи
- Автоматический показ предложения установить приложение при открытии на мобильном устройстве, если приложение **не установлено** (не запущено в standalone-режиме).
- Подавление предложения, если приложение уже установлено / открыто как PWA (standalone).
- Поведение по платформам:
  - **Android / Chrome (и браузеры, поддерживающие `beforeinstallprompt`)** — предложение, инициирующее **программную** установку через перехваченное событие `beforeinstallprompt`.
  - **iOS / Safari (нет программного API)** — предложение с **ручной инструкцией** «Поделиться → На экран «Домой»» (переиспользует существующий экран/контент `s-install`).
  - **Прочие браузеры** — по возможности (предложение/инструкция в пределах поддержки; при невозможности — корректная деградация без поломки UI).
- Возможность пользователю принять предложение либо отклонить его.
- Правило повторных показов: после отклонения предложение не показывается повторно (либо показывается не чаще, чем раз в N дней — конкретное правило см. «Открытые вопросы»).
- Согласование нового автоматического предложения с уже существующими механизмами (`s-install`, `#installBanner`, флаг `yc_install_seen`), чтобы не было двух конкурирующих предложений и противоречивых состояний «показано/скрыто».
- Внешний вид предложения — в стиле приложения (не нативный браузерный баннер браузера).

### Вне рамок (explicitly out of scope)
- Push-уведомления и логика подписки на них (`docs/modules/push.js`, Cloudflare Worker) — не входят; установка лишь создаёт почву для них.
- Изменение сценария записи (услуга → мастер → слот → подтверждение) и его экранов.
- Изменение содержимого Web App Manifest (`docs/manifest.json`) и иконок — если только это не окажется необходимым для устанавливаемости (тогда минимальные правки, но не редизайн манифеста).
- Десктопная установка PWA как отдельная задача оптимизации — основной фокус требования мобильный («на мобильном устройстве»); десктоп не должен ломаться, но не является целью.
- A/B-тестирование текстов/частоты показа, аналитика конверсии установок — отдельная возможная задача.
- Принудительная установка/блокирование функциональности до установки — приложение остаётся полностью работоспособным без установки.

## 3. Действующие лица и сценарии использования

**Клиент студии (мобильный, не установил приложение)** — открывает сайт в браузере смартфона.

Основной сценарий (Android / Chrome):
1. Клиент открывает приложение в браузере на смартфоне; приложение не установлено (не standalone).
2. Браузер сообщает о возможности установки (событие `beforeinstallprompt`); приложение показывает собственное предложение в фирменном стиле.
3. Клиент нажимает «Установить» → запускается программный установочный prompt платформы; клиент подтверждает → приложение добавляется на экран «Домой».
4. Если клиент отклоняет предложение — оно скрывается и далее не показывается повторно (с учётом правила «раз в N дней», см. «Открытые вопросы»).

Основной сценарий (iOS / Safari):
1. Клиент открывает приложение в Safari на смартфоне; приложение не установлено (не standalone).
2. Программного API установки нет → приложение показывает предложение с ручной инструкцией «Поделиться → На экран «Домой»» (контент `s-install`).
3. Клиент следует инструкции и добавляет приложение, либо отклоняет предложение («Позже») — оно скрывается с учётом правила повторного показа.

Сценарий «уже установлено» (любая платформа):
- Приложение открыто как установленное (standalone) → предложение установки **не показывается** ни в каком виде.

Граничный сценарий (платформа без поддержки):
- Браузер не поддерживает программную установку и не является поддерживаемым iOS/Safari → предложение либо не показывается, либо показывается доступная инструкция; UI не ломается, основной функционал доступен.

Граничный сценарий (повторное открытие после отказа):
- Клиент ранее отклонил предложение → при последующих открытиях предложение не показывается повторно (либо не чаще раза в N дней).

## 4. Функциональные требования

- **ФТ-1.** При открытии приложения на мобильном устройстве, если приложение **не установлено** (не запущено в standalone-режиме), приложение автоматически показывает предложение установить его на экран «Домой».
- **ФТ-2.** Если приложение уже установлено / открыто в standalone-режиме, предложение установки **не показывается** ни на одной платформе.
- **ФТ-3.** На платформах, поддерживающих `beforeinstallprompt` (Android/Chrome и совместимые), предложение использует перехваченное событие для **программного** запуска установки: по действию пользователя «Установить» вызывается платформенный установочный prompt.
- **ФТ-4.** На iOS/Safari, где программного API установки нет, предложение показывает **ручную инструкцию** добавления на экран «Домой» («Поделиться → «На экран «Домой»» → «Добавить»»), переиспользуя существующий контент экрана `s-install`.
- **ФТ-5.** Пользователь может **принять** предложение (инициировать установку/перейти к инструкции) или **отклонить** его; оба действия доступны и понятны.
- **ФТ-6.** После отклонения предложения оно **не показывается повторно** при последующих открытиях (либо показывается не чаще, чем раз в N дней — параметр уточняется, см. «Открытые вопросы»). Состояние отказа сохраняется между сессиями на устройстве.
- **ФТ-7.** После успешной установки приложения предложение и связанный с ним пассивный баннер (`#installBanner`) на главной перестают показываться.
- **ФТ-8.** Внешний вид предложения выдержан в стиле приложения (фирменный баннер/подсказка), а не является нативным браузерным баннером установки.
- **ФТ-9.** Новый автоматический механизм предложения согласован с существующими (`s-install`, `#installBanner`, флаг `yc_install_seen`): пользователь не видит двух конкурирующих предложений одновременно, а состояния «показано / отклонено / установлено» не противоречат друг другу.
- **ФТ-10.** На платформах, где установка невозможна или не поддерживается, отсутствие предложения (или показ доступной инструкции) не нарушает работу приложения; основной функционал доступен без установки.

## 5. Нефункциональные требования

- **НФТ-1 (Совместимость стека).** Решение остаётся в рамках стека: vanilla JS/HTML/CSS, статический хостинг из `docs/` на GitHub Pages, без новых рантайм-npm-зависимостей.
- **НФТ-2 (PWA/устанавливаемость и офлайн).** Приложение должно оставаться устанавливаемым по критериям PWA; поведение Service Worker и офлайн-режима не должно регрессировать; влияние на `CACHE_VERSION` (`docs/sw.js`) при правке кэшируемых ресурсов — предмет проектирования.
- **НФТ-3 (Ненавязчивость).** Предложение не должно мешать первому полезному действию пользователя и не должно появляться повторно после отказа/установки; частота показов контролируема (см. ФТ-6).
- **НФТ-4 (Локализация).** Все пользовательские тексты предложения и инструкции — на русском языке, в едином тоне с остальными экранами.
- **НФТ-5 (Деградация).** При отсутствии поддержки установки в браузере приложение не показывает «битых» элементов и не выдаёт ошибок в консоли; UI остаётся консистентным (НФТ-2/ФТ-10).
- **НФТ-6 (Производительность).** Логика показа предложения не должна заметно задерживать загрузку и первичную отрисовку на мобильном устройстве.

## 6. Ограничения и допущения

**Ограничения:**
- Приложение — статический PWA в `docs/`, обслуживается GitHub Pages; собственного бэкенда для PWA нет (Worker — отдельный артефакт для push/прокси).
- PWA-установка и `beforeinstallprompt` работают только по HTTPS (GitHub Pages) или на localhost.
- Программный prompt установки доступен только в браузерах, поддерживающих `beforeinstallprompt` (на момент написания — Chromium-семейство). На iOS/Safari программного API установки нет — только ручная инструкция.
- Браузер сам решает, когда генерировать `beforeinstallprompt` (свои критерии вовлечённости/устанавливаемости) — приложение не может гарантировать его срабатывание на каждом открытии.
- При изменении состава кэшируемых ресурсов (`APP_SHELL`) требуется поднятие `CACHE_VERSION` в `docs/sw.js`.
- Запрещены рантайм-npm-зависимости (только devDependencies).

**Допущения:**
- Манифест и Service Worker уже обеспечивают базовую устанавливаемость PWA (`docs/manifest.json`: `display: standalone`, иконки 192/512; `docs/sw.js` регистрируется) — задача добавляет предложение установки, а не вводит устанавливаемость с нуля.
- Существующий контент `s-install` (iOS-инструкция) пригоден для переиспользования в новом предложении (возможны точечные правки текста, без редизайна).
- Признак «уже установлено» определим на клиенте через standalone-режим (например, media-query `display-mode: standalone` и/или платформенный признак iOS) — конкретику определяет архитектор.
- Состояние отказа/установки хранится локально на устройстве (`localStorage`, по образцу существующего `yc_install_seen`) — без бэкенда; на новом устройстве предложение может появиться снова (приемлемо).

## 7. Критерии приёмки

- **КП-1.** При первом открытии на мобильном устройстве в браузере, поддерживающем установку, приложение автоматически показывает фирменное (не нативное браузерное) предложение установить его. (ФТ-1, ФТ-8)
- **КП-2.** Когда приложение запущено в standalone-режиме (уже установлено), предложение установки не показывается. (ФТ-2)
- **КП-3.** На Android/Chrome нажатие «Установить» в предложении запускает платформенный установочный prompt (через перехваченный `beforeinstallprompt`), и при подтверждении приложение добавляется на экран «Домой». (ФТ-3)
- **КП-4.** На iOS/Safari предложение показывает ручную инструкцию «Поделиться → «На экран «Домой»»» (контент `s-install`), а не программный prompt. (ФТ-4)
- **КП-5.** Из предложения доступны и работают оба действия: принять (установка/инструкция) и отклонить. (ФТ-5)
- **КП-6.** После отклонения предложение не показывается при следующем открытии (или не чаще раза в N дней по согласованному правилу); состояние сохраняется между сессиями. (ФТ-6, НФТ-3)
- **КП-7.** После установки приложения предложение и пассивный баннер `#installBanner` на главной больше не показываются. (ФТ-7, ФТ-9)
- **КП-8.** Пользователь не видит одновременно нового автоматического предложения и старого баннера/экрана как двух конкурирующих предложений; состояния «показано/отклонено/установлено» непротиворечивы. (ФТ-9)
- **КП-9.** В браузере без поддержки установки приложение не показывает «битых» элементов и не выдаёт ошибок в консоли; основной функционал доступен без установки. (ФТ-10, НФТ-5)
- **КП-10.** Все тексты предложения и инструкции — на русском языке, в едином стиле. (НФТ-4)
- **КП-11.** PWA остаётся устанавливаемой по критериям PWA, офлайн-поведение не регрессирует, новые рантайм-зависимости не добавляются. (НФТ-1, НФТ-2)

## 8. Открытые вопросы

1. **Правило повторного показа.** Исходное требование допускает два варианта: «после отказа не показывать совсем» **или** «показывать раз в N дней». Какой выбрать и чему равно N (например, 7/14/30 дней)? Различать ли «отклонил предложение» и «уже установил» (после установки — не показывать никогда)? *Влияет на ФТ-6; для старта возможен дефолт, но требуется подтверждение заказчика.*
2. **Момент показа.** Требование говорит «показывать сразу при открытии (не ждать взаимодействия)». Однако `beforeinstallprompt` срабатывает по решению браузера и может прийти не мгновенно; программный install prompt браузеры разрешают вызывать только в ответ на жест пользователя. Подтвердить ожидаемое поведение: показывать **наше** фирменное предложение сразу при открытии, а **платформенный** prompt — по нажатию кнопки внутри него (это совместимо с политиками браузеров). Согласовать, считается ли это «сразу при открытии».
3. **Судьба существующего баннера `#installBanner` и экрана `s-install`.** Заменяет ли новое автоматическое предложение пассивный баннер на главной, дополняет его или они сосуществуют? Переиспользуется ли `s-install` целиком как iOS-инструкция, или предложение — отдельный оверлей/модал? Сохраняется ли флаг `yc_install_seen` или вводится новое состояние? *Влияет на ФТ-9; нужно решение, чтобы не плодить конкурирующие механизмы.* (Реализационную форму определяет архитектор, но продуктовое намерение «один связный механизм» нужно подтвердить.)
4. **Только мобильные или и десктоп.** Требование акцентирует «на мобильном устройстве». Показывать ли предложение на десктопе (где Chrome тоже поддерживает установку), или строго на мобильных? *Влияет на условие показа в ФТ-1.*
5. **Текст и оформление предложения.** Какой текст/заголовок/иконка у фирменного предложения (мотивация установки: напоминания о визите, персональные предложения)? Существующие тексты `s-install`/`#installBanner` — отправная точка; нужно подтверждение заказчика по финальной формулировке и виду (НФТ-4, ФТ-8).
6. **Метрика успеха.** Нужно ли заказчику измерять конверсию в установку (количество показов/принятий/отказов)? Сейчас аналитики установок нет. *Аналитика — вне рамок, но решение заказчика стоит зафиксировать, чтобы не закрывать возможность.*

Противоречий с принятыми ADR не выявлено. `adrs/adr-001.md` (визуальный каталог), `adrs/adr-002.md` (фильтрация мастеров), `adrs/adr-003.md` и `adrs/adr-004.md` (отзывы через YCLIENTS/Worker) данной темы не затрагивают. Требование НФТ-2 (не регрессировать устанавливаемость/офлайн PWA) согласуется с ограничениями этих ADR относительно `CACHE_VERSION` и стека.
