# BA-требования 014: Детальная карточка услуги (описание + фото работ + «Выбрать мастера»)

- **Источник:** `requirements/req-service-detail-card-014.md`
- **Дата:** 2026-05-29
- **Статус:** черновик

## 1. Контекст и бизнес-цель

Studio PWA (студия красоты «Реснички, сестрички», онлайн-запись поверх YCLIENTS) — устанавливаемое статическое приложение, отдаётся из `docs/` на GitHub Pages. Экран выбора услуги (`s-services`) — обязательный шаг основного сценария записи. Сейчас тап по карточке услуги в каталоге сразу вызывает `selectService(id)` (`docs/modules/services.js:147-156`), который ставит `state.serviceId` и **немедленно** переходит к выбору мастера (`go('s-masters')`) либо, если мастер предвыбран, к выбору времени (`go('s-slots')`). Промежуточного экрана, на котором клиент мог бы изучить услугу, **нет**.

Студия позиционируется в премиум-сегменте и «продаёт эстетику». Клиент выбирает услугу «вслепую» — по названию, цене и одной обложке в каталоге, без описания процедуры и без примеров работ. Это снижает информированность выбора и воспринимаемую ценность услуги перед записью.

**Бизнес-цель:** дать клиенту возможность изучить услугу перед записью — посмотреть описание и фотографии работ на отдельном детальном экране, и только осознанно перейти к выбору мастера. Цель — повысить информированность выбора, воспринимаемую ценность услуги и конверсию в запись. Дополнительно — показать в каталоге реальное фото услуги из YCLIENTS вместо текущего категорийного плейсхолдера, когда фото есть.

**Стейкхолдеры:**
- Заказчик / владелец студии — заинтересован в премиальной, информативной подаче услуг и поставляет фото/описания в YCLIENTS.
- Клиент студии — пользователь, выбирающий услугу; хочет до записи понять, что за процедура и как выглядит результат.
- Бизнес-аналитик / архитектор / разработчик — оформление, проектирование, реализация.

## 2. Область (scope)

### В рамках задачи

- **Новый промежуточный экран — детальная карточка услуги** между каталогом услуг (`s-services`) и выбором мастера (`s-masters`).
- Изменение поведения тапа по карточке в каталоге: тап **открывает детальную карточку услуги**, а не сразу ведёт к выбору мастера.
- Состав детальной карточки: **название** услуги, **описание** услуги (поле `comment` объекта услуги из YCLIENTS), **цена (диапазон, ADR-012) и длительность** (§8 №3), **фотографии работ** (из YCLIENTS, может быть больше одной — см. §8 №1) и нижняя кнопка **«Выбрать мастера»**.
- По кнопке «Выбрать мастера» выполняется переход к экрану выбора мастера `s-masters`. В режиме предвыбранного мастера (`masterPreSelected`) детальная карточка не показывается вовсе — тап по услуге ведёт сразу к выбору времени `s-slots`, как сейчас (§8 №5).
- Кнопка «Назад» с детальной карточки возвращает в каталог услуг (стандартный стек навигации `back()`/`go`, см. §8 №6).
- **Фото услуги на карточках каталога из YCLIENTS:** в каталоге обложка карточки берётся из фото услуги YCLIENTS; если у услуги несколько фото — на карточке каталога показывается **первое**; если фото нет — текущий категорийный плейсхолдер (градиент + эмодзи).
- Расширение маппинга объекта услуги при загрузке (`docs/app.js:167-176`): сохранение полей `comment` (описание) и фото услуги (`image_group`), которые сейчас отбрасываются. Описание и фото берутся из уже загруженного объекта услуги — **без дополнительных запросов к API на каждую услугу** (см. §6, ограничение по запросам).
- Сохранение существующего поведения каталога: поиск (`filterSearch`/`#serviceSearch`), фильтр по категории (`filterCategory`/`#catFilterBtn`), баннер предвыбранного мастера (`_updateMasterBanner`), состояние «Ничего не найдено», корректная цена/длительность (включая диапазон цен по ADR-012) и дальнейший поток записи (услуга → мастер → время → подтверждение).

### Вне рамок (explicitly out of scope)

- **Изменение логики выбора мастера, расчёта слотов, самого бронирования и подтверждения** — детальная карточка лишь вставляется перед выбором мастера; дальнейший поток переиспользуется как есть.
- **Изменение источника данных и набора эндпоинтов YCLIENTS** сверх сохранения уже приходящих полей `comment` и `image_group` из `/services/{company}`; новые тяжёлые запросы (фото/описание по каждой услуге отдельно) не вводятся.
- **Изменение вёрстки/состава каталога и строки поиска** сверх замены источника обложки на фото из YCLIENTS (одностолбцовые крупные карточки — задача BA-012; диапазон цен — BA-013/ADR-012; визуальный каталог — BA-001).
- **Изменение логики ценообразования и строки цены** (диапазон/конкретная цена) — задаётся ADR-012; настоящая задача её не пересматривает (показывать ли цену/длительность на детальной карточке — §8 №3, без изменения самой логики).
- **Загрузка/подготовка нескольких фото работ на стороне студии** (в YCLIENTS или ином источнике) — поставка контента вне объёма реализации; в рамках задачи показываются те фото, что фактически доступны (§8 №1).
- **Зум/полноэкранный просмотр фото, лайтбокс** — только при подтверждении в §8 №4; по умолчанию не входит.
- **Привязка фото к конкретным мастерам, портфолио мастера, отзывы** — вне объёма.
- **Перенос захардкоженного YCLIENTS-токена** из `prototype.html` (известный техдолг — «не усугублять», не трогать).

## 3. Действующие лица и сценарии использования

**Клиент студии** — пользователь PWA, выбирающий услугу в рамках сценария записи. Цель — изучить услугу (что за процедура, как выглядит результат) и затем записаться.

Основной сценарий:
1. Клиент открывает каталог услуг `s-services`; видит карточки с названием, фото услуги из YCLIENTS (или плейсхолдером), диапазоном цен и длительностью.
2. Клиент тапает по карточке услуги → открывается **детальная карточка услуги** с названием, описанием и фотографиями работ.
3. Клиент изучает описание и фото.
4. Клиент нажимает нижнюю кнопку **«Выбрать мастера»** → переходит на экран «Выберите мастера» (`s-masters`).
5. Далее — обычный поток записи (мастер → время → подтверждение).

Альтернативные / граничные сценарии:
- **Услуга без описания и/или без фото.** Детальная карточка корректно отображается: пустое описание/отсутствие фото обрабатываются согласно §8 №2 (по умолчанию — скрыть пустой блок описания; для фото — показать категорийный плейсхолдер). Кнопка «Выбрать мастера» доступна в любом случае.
- **Несколько фото работ.** Если у услуги фактически больше одного фото (§8 №1), они показываются галереей; формат галереи — §8 №4. Если фото одно — без листалки.
- **Предвыбранный мастер.** Клиент пришёл с экрана мастера (`masterPreSelected`). Детальная карточка **не показывается** — тап по услуге ведёт сразу к выбору времени (`s-slots`), как сейчас (§8 №5).
- **Возврат назад.** С детальной карточки кнопка «Назад» возвращает в каталог услуг (§8 №6), состояние каталога (фильтр/поиск/баннер мастера) сохраняется.

## 4. Функциональные требования

- **ФТ-1.** В каталоге услуг (`s-services`), **когда мастер не предвыбран**, тап по карточке услуги открывает **детальную карточку услуги** (новый экран), а не выполняет немедленный переход к выбору мастера.
- **ФТ-2.** Детальная карточка услуги отображает: **название** услуги, **описание** услуги (из поля `comment` объекта услуги YCLIENTS), **цену (диапазон, ADR-012) и длительность** (§8 №3) и **фотографии работ** услуги (из YCLIENTS).
- **ФТ-3.** В нижней части детальной карточки расположена кнопка **«Выбрать мастера»**; тап по ней выполняет переход к выбору мастера (`s-masters`) с установленным `state.serviceId` выбранной услуги.
- **ФТ-4.** В режиме предвыбранного мастера (`state.masterPreSelected`) детальная карточка **не показывается** — сохраняется текущее поведение: тап по услуге ведёт сразу к выбору времени (`s-slots`) с установленным `state.serviceId` (§8 №5).
- **ФТ-5.** Кнопка «Назад» на детальной карточке возвращает клиента в каталог услуг `s-services` (стандартное поведение стека навигации); состояние каталога (фильтр по категории, поисковый запрос, баннер предвыбранного мастера) при возврате сохраняется (§8 №6).
- **ФТ-6.** Если у услуги **несколько** фотографий, они отображаются **горизонтальной каруселью со свайпом и точками-индикаторами** (§8 №4); если фотография **одна** — она показывается без элементов перелистывания.
- **ФТ-7.** Если у услуги **нет** фотографий — на детальной карточке отображается категорийный плейсхолдер (градиент + эмодзи), как в каталоге (поведение по умолчанию, §8 №2), без поломки вёрстки.
- **ФТ-8.** Если у услуги **пустое** описание (`comment` отсутствует/пустой) — блок описания обрабатывается согласно §8 №2 (по умолчанию — скрывается; заглушка не показывается), остальная карточка отображается корректно.
- **ФТ-9.** В каталоге услуг обложка карточки берётся из фотографии услуги YCLIENTS; при наличии **нескольких** фото — на карточке каталога показывается **первое**; при отсутствии фото — категорийный плейсхолдер (как сейчас).
- **ФТ-10.** При ошибке загрузки фотографии (битый URL, недоступность) карточка каталога и детальная карточка корректно деградируют на категорийный плейсхолдер без поломки вёрстки.
- **ФТ-11.** Описание и фотографии услуги берутся из **уже загруженного** объекта услуги (`/services/{company}`, маппинг расширяется в `docs/app.js:167-176` сохранением `comment` и `image_group`); дополнительные запросы к API на каждую услугу для получения описания/фото **не выполняются**.
- **ФТ-12.** На детальной карточке показываются **цена (диапазон) и длительность** услуги (§8 №3); цена/длительность согласованы с уже принятой логикой (диапазон цен — ADR-012), без её изменения. Категория и список мастеров на детальной карточке не показываются.
- **ФТ-13.** Все пользовательские тексты экрана (кнопки, заголовки, заглушки) — на русском языке, в едином тоне приложения.
- **ФТ-14.** Поиск по услугам, фильтр по категории, режим предвыбранного мастера и состояние «Ничего не найдено» в каталоге продолжают работать без изменения логики и без ошибок в консоли.

## 5. Нефункциональные требования

- **НФТ-1 (Безопасность / XSS).** Внешние и пользовательские данные, выводимые на экранах (описание `comment`, URL фотографий, название услуги), экранируются через `esc()` — защита от XSS; вставка непроверенного HTML не допускается.
- **НФТ-2 (Производительность).** Открытие детальной карточки не должно выполнять тяжёлых дополнительных запросов к YCLIENTS (описание/фото — из уже загруженного объекта услуги); загрузка и прокрутка каталога не должны деградировать. Фотографии подгружаются без блокировки взаимодействия (`loading="lazy"` сохраняется).
- **НФТ-3 (Деградация).** При отсутствии/ошибке фото или описания экран остаётся консистентным: плейсхолдер вместо фото, скрытие пустого блока описания; кнопка «Выбрать мастера» всегда доступна.
- **НФТ-4 (Совместимость стека).** Решение — в рамках стека: vanilla JS/HTML/CSS, код в `docs/` (`prototype.html`, `docs/modules/*.js`, `docs/style.css`); статический хостинг на GitHub Pages; без новых рантайм-npm-зависимостей.
- **НФТ-5 (Минимальность изменений).** Изменения строго в объёме задачи; логика выбора мастера/слотов/бронирования и состав каталога (вне замены источника обложки) не рефакторятся.
- **НФТ-6 (PWA/офлайн не регрессируют).** Изменения не ломают устанавливаемость PWA и офлайн-поведение Service Worker; при изменении состава кэшируемых файлов `APP_SHELL` (`prototype.html`, `style.css`, `services.js`, `app.js` и т. п.) поднимается `CACHE_VERSION` в `docs/sw.js` (правило проекта).
- **НФТ-7 (Качество фото).** Фотографии услуги из `image_group` отдаются YCLIENTS в ресайзе ≈420×280. Приемлемость такого размера для крупной галереи на детальном экране (или необходимость более качественного источника) — §8 №7. По умолчанию используется доступный из `image_group` размер с корректным масштабированием без видимого размытия в пределах макета.
- **НФТ-8 (Локализация).** Все пользовательские тексты — на русском языке, в едином тоне с существующими экранами.
- **НФТ-9 (Доступность контента).** Фотографии имеют текстовую альтернативу (название услуги в `alt`), чтобы карточка оставалась понятной при недоступности изображения.

## 6. Ограничения и допущения

**Ограничения:**

- Релевантный код (для информации; реализацию определяет архитектор): загрузка и маппинг услуг — `docs/app.js:159-176` (поля `comment` и `image_group` сейчас НЕ сохраняются); рендер каталога и тап по карточке — `docs/modules/services.js:77-100, 91, 147-156`; резолв обложки — `docs/modules/serviceImages.js:21-31` (через пустой `SERVICE_PHOTO_MAP` + категорийный плейсхолдер, `image_group` из API не используется); навигация и стек экранов — `docs/modules/navigation.js` (`go`/`back`); образец детального экрана-карточки — профиль мастера `s-master`/`_masterCardHtml` (`docs/modules/masters.js:10`).
- Описание услуги в YCLIENTS — поле **`comment`** (не `description`), может быть пустым.
- Фотографии услуги в YCLIENTS — структура `image_group` объекта услуги. На практике у услуги обычно **одно** фото: повторная загрузка через `POST /api/v1/images/service` по тому же `service_id` перетирает предыдущую картинку. Источник «нескольких фото работ» не подтверждён (§8 №1). Дополнительно: структура `image_group` в YCLIENTS неоднородна между сущностями (например, у мастеров `image_group.images.norm.path` — объект размеров, см. `masters.js:168`), поэтому фактический формат фото услуги (массив `images[]` против объекта размеров) необходимо подтвердить на реальных данных при проектировании.
- Запрещены рантайм-npm-зависимости; запрещён коммит секретов (включая YCLIENTS-токен); запрещены любые операции с веткой `backup`; `--force` в `main` запрещён.
- При изменении файлов из `APP_SHELL` — поднятие `CACHE_VERSION` в `docs/sw.js`.
- Не делать тяжёлых дополнительных запросов к API без необходимости — описание/фото по возможности из уже загруженного объекта услуги (расширение маппинга в `app.js`), а не отдельными запросами на каждую услугу.

**Допущения:**

- Объект услуги из `/services/{company}` содержит поля `comment` (описание) и `image_group` (фото), пригодные для сохранения в данных услуги без дополнительных запросов; конкретную структуру `image_group` (массив фото или объект размеров) **подтверждает архитектор/разработчик** на реальных данных при проектировании.
- Стек экранов и навигация (`go`/`back`, `navHistory`) уже поддерживают подобные детальные экраны (образец — `s-master`), поэтому добавление экрана `s-services` → детальная карточка → `s-masters`/`s-slots` укладывается в существующий механизм без его переделки.
- У части услуг описание (`comment`) уже заполнено; у части — пустое; фото может отсутствовать. Граничные случаи (нет описания / нет фото / одно фото / несколько фото) должны обрабатываться корректно.
- Каталог уже эволюционирует параллельными требованиями (BA-001 визуальный каталог, BA-012 одностолбцовый layout, BA-013/ADR-012 диапазон цен); настоящая задача не противоречит им и переиспользует их результат (обложка, цена/длительность, поиск/фильтр).

## 7. Критерии приёмки

- **КП-1.** В каталоге услуг тап по карточке открывает детальную карточку услуги (новый экран), а не сразу ведёт к выбору мастера/времени. (ФТ-1)
- **КП-2.** Детальная карточка отображает название, описание (из `comment`) и фотографии работ услуги из YCLIENTS. (ФТ-2, ФТ-11)
- **КП-3.** На детальной карточке внизу есть кнопка «Выбрать мастера», по которой клиент попадает на экран «Выберите мастера» (`s-masters`) с корректно установленной услугой. (ФТ-3)
- **КП-4.** В режиме предвыбранного мастера детальная карточка не показывается — тап по услуге ведёт сразу к выбору времени (`s-slots`), как в текущем поведении. (ФТ-4)
- **КП-5.** Кнопка «Назад» с детальной карточки возвращает в каталог услуг; ранее применённые поиск/фильтр/баннер мастера сохраняются. (ФТ-5)
- **КП-6.** При нескольких фото они показываются галереей (формат — §8 №4); при одном фото — без перелистывания; при отсутствии фото — категорийный плейсхолдер без поломки вёрстки. (ФТ-6, ФТ-7)
- **КП-7.** При пустом описании блок описания обрабатывается согласно §8 №2 (по умолчанию скрыт), остальная карточка корректна. (ФТ-8)
- **КП-8.** В каталоге на карточке отображается фото услуги из YCLIENTS (первое из доступных при нескольких); при отсутствии фото — категорийный плейсхолдер. (ФТ-9)
- **КП-9.** При ошибке загрузки фото каталог и детальная карточка деградируют на плейсхолдер без поломки вёрстки. (ФТ-10, НФТ-3)
- **КП-10.** Открытие детальной карточки не выполняет дополнительных запросов к YCLIENTS за описанием/фото; описание и фото берутся из уже загруженного объекта услуги. (ФТ-11, НФТ-2)
- **КП-11.** Поиск по услугам, фильтр по категории, режим предвыбранного мастера, состояние «Ничего не найдено» и дальнейший поток записи (услуга → мастер → время → подтверждение) работают без ошибок в консоли. (ФТ-14)
- **КП-12.** Описание (`comment`), URL фото и название услуги экранируются (`esc()`); инъекция HTML невозможна. (НФТ-1)
- **КП-13.** PWA остаётся устанавливаемой, офлайн-поведение не регрессирует; при изменении файлов `APP_SHELL` поднята `CACHE_VERSION`; новые рантайм-зависимости не добавлены. (НФТ-4, НФТ-6)
- **КП-14.** Все пользовательские тексты экрана — на русском языке. (ФТ-13, НФТ-8)
- **КП-15.** Состав детальной карточки сверх «название/описание/фото/кнопка» (цена, длительность и т. д.) соответствует решению §8 №3; при показе цены/длительности они согласованы с действующей логикой (ADR-012) и не меняют её. (ФТ-12)

## 8. Открытые вопросы

> Вопросы №1, №3, №4, №5 **подтверждены заказчиком 2026-05-29** (решения ниже). Для №2, №6, №7 действуют значения по умолчанию (заказчик не возражал). Остаётся один технический риск к проверке архитектором — поддерживает ли YCLIENTS хранение >1 фото на услугу (см. №1).

1. **Источник нескольких фотографий работ — РЕШЕНО (2026-05-29): дозагрузить фото в YCLIENTS.** Целевой источник фото работ — поле `image_group` объекта услуги; студия предварительно зальёт по несколько фото на услуги (это **отдельная контентная задача, вне объёма реализации PWA**). PWA должна показывать **все** фото из `image_group` услуги (одно → без галереи; несколько → горизонтальная карусель, см. №4). Варианты (б) общее портфолио и (в) фото по категории — **отклонены**.
   - **Технический риск (к проверке архитектором/при дозагрузке):** на практике `POST /api/v1/images/service` по тому же `service_id` **перезаписывает** фото — не подтверждено, что YCLIENTS вообще хранит и отдаёт >1 фото на услугу (возможно, несколько фото добавляются только через админку, а не API). Перед/во время дозагрузки нужно подтвердить, что `image_group.images` реально содержит массив из нескольких фото. Если YCLIENTS ограничивает услугу одним фото — поднять вопрос об альтернативном источнике. Пока несколько фото фактически недоступны, детальная карточка показывает одно фото/плейсхолдер; карусель включается автоматически, когда фото станет несколько. *Влияет на ФТ-2, ФТ-6, ФТ-9, КП-2, КП-6, КП-8.*

2. **Поведение при пустом описании / отсутствии фото — значение по умолчанию (заказчик не возражал).** При пустом `comment` — **скрывать** блок описания (без текста-заглушки); при отсутствии фото — показывать **категорийный плейсхолдер** (градиент + эмодзи), как в каталоге. *Влияет на ФТ-7, ФТ-8, КП-6, КП-7.*

3. **Состав детальной карточки — РЕШЕНО (2026-05-29): цена (диапазон) + длительность.** На детальной карточке сверх названия/описания/фото/кнопки показываются **цена (диапазон, согласно ADR-012) и длительность** (для целостности с каталогом). Категорию и список мастеров — **не** показывать (выбор мастера — следующий шаг). *Влияет на ФТ-12, КП-15.*

4. **Формат галереи — РЕШЕНО (2026-05-29): горизонтальная карусель.** При нескольких фото — горизонтальная карусель со свайпом (точки-индикаторы). При одном фото — статичное изображение без листалки. Зум/полноэкранный просмотр (лайтбокс) — **вне объёма** (добавляется отдельным решением при необходимости). *Влияет на ФТ-6, КП-6.*

5. **Режим предвыбранного мастера — РЕШЕНО (2026-05-29): пропускать детальную карточку.** Когда мастер уже выбран (`state.masterPreSelected`), сохраняется **текущее** поведение: тап по услуге в каталоге ведёт **сразу к выбору времени** (`s-slots`), детальная карточка услуги **не показывается**. Детальная карточка вставляется только в обычном потоке (мастер не предвыбран). *Влияет на ФТ-1, ФТ-4, КП-4.*

6. **Поведение кнопки «Назад» — значение по умолчанию (подтверждается).** «Назад» с детальной карточки возвращает в каталог услуг `s-services` (стандартное поведение стека `go`/`back`, `navigation.js:103-130`); состояние каталога сохраняется. Отдельной логики возврата не требуется. *Влияет на ФТ-5, КП-5.*

7. **Качество фотографий — значение по умолчанию (заказчик не возражал).** `image_group` отдаёт ресайз ≈420×280. Использовать доступный из `image_group` размер с корректным масштабированием в пределах макета; более качественный источник — по запросу заказчика. *Влияет на НФТ-7.*

**Противоречий с принятыми ADR (`adrs/adr-001`…`adr-012`) не выявлено.** Настоящая задача добавляет промежуточный экран и расширяет маппинг услуги (сохранение `comment`/`image_group`) и источник обложки каталога; логика цен (диапазон, ADR-012), логика выбора мастера/слотов и бронирования не затрагиваются. Если на этапе проектирования выбранное решение по галерее/фото/составу карточки затронет действующий ADR, это должно быть отражено в новом/обновлённом ADR (вероятно, потребуется новый ADR на схему детального экрана и работу с `image_group`).
