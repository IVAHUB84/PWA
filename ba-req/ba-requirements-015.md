# BA-требования 015: Карусель фото услуги в детальной карточке — полная галерея, скругления, аффорданса, зацикливание, свежесть фото

- **Источник:** `requirements/req-service-card-carousel-015.md`
- **Дата:** 2026-05-29
- **Статус:** согласовано (готов к /adr)

## 1. Контекст и бизнес-цель

Studio PWA (студия красоты «Реснички, сестрички», онлайн-запись поверх YCLIENTS) — устанавливаемое статическое приложение, отдаётся из `docs/` на GitHub Pages, стек vanilla JS/HTML/CSS без рантайм-зависимостей. В рамках требования 014 / ADR-013 добавлен детальный экран услуги `s-service`, на котором при наличии нескольких фото работ показывается горизонтальная карусель (нативный CSS scroll-snap + точки-индикаторы, `docs/modules/services.js:167-213`). У услуги в YCLIENTS может быть до 8 фото работ.

Текущая реализация карусели не «продаёт» эстетику студии и плохо считывается как листаемый компонент:

- **Изображения прямоугольные** (`height:280px; object-fit:cover`, `docs/style.css:812-835`), без скруглений — диссонанс с остальным интерфейсом, где карточки и кнопки скруглены (14–16px, `docs/style.css`).
- **Нет визуальной подсказки**, что фото больше одного и что компонент листается: слайд занимает 100% ширины (`flex:0 0 100%`), соседнее фото не «выглядывает», единственный признак — мелкие точки.
- **Нет зацикливания**: нативный scroll-snap упирается на краях, перехода с последнего фото на первое (и обратно) нет.
- **Фото «застревают»**: новое/изменённое в YCLIENTS изображение услуги не появляется у клиента, пока он не переустановит приложение или вручную не очистит кэш. Причина — кэширование Service Worker (см. §6): список URL фото приходит из API `api.yclients.com` (помечен `NETWORK_ONLY_HOSTS`, всегда свежий), но сами байты картинок отдаются с кросс-origin хоста YCLIENTS-CDN и попадают в ветку «не-same-origin → cache-first без ревалидации» (`docs/sw.js:108-118`), поэтому закэшированный URL отдаётся из кэша до смены `CACHE_VERSION`.

Отдельно подтверждена принципиальная достижимость нескольких фото на услугу (ключевой риск требования снят, проверка живого API YCLIENTS компании 1940850, 2026-05-29): несколько фото добавляются через **галерею услуги в админке YCLIENTS** — подтверждено на реальной услуге «Маникюр с покрытием гель-лак» (id 29196117, 3 фото). При этом эндпойнт, который приложение использует сейчас — `GET /services/{company}` (`docs/app.js:163`) — **схлопывает галерею до одного фото** (`image_group.images` приходит объектом `{basic}`, а не массивом), то есть карусель в текущем виде на проде физически не получит больше одного фото. Полную галерею массивом отдают другие эндпойнты YCLIENTS (см. §6) — поэтому в scope входит требование «получать все фото услуги».

**Бизнес-цель:** сделать галерею фото работ на детальной карточке услуги визуально премиальной и очевидно листаемой, чтобы клиент уверенно просматривал все фото услуги перед записью; обеспечить, чтобы в карточку фактически попадали **все** фото услуги из галереи YCLIENTS (а не одно); и гарантировать, что обновлённый студией фотоконтент доходит до клиента автоматически — без переустановки и ручной чистки кэша. Это повышает воспринимаемую ценность услуги, информированность выбора и доверие к актуальности контента.

**Стейкхолдеры:**

- Заказчик / владелец студии — поставляет фото работ в YCLIENTS, ожидает, что обновлённые фото быстро видны клиентам; заинтересован в премиальной подаче.
- Клиент студии — пользователь PWA, просматривает фото работ услуги перед записью.
- Бизнес-аналитик / архитектор / разработчик — оформление, проектирование, реализация.

## 2. Область (scope)

### В рамках задачи

- **Получение всех фото услуги (до 8) из галереи YCLIENTS.** Каталог и детальная карточка должны получать **все** фото услуги, а не одно — источником служит галерея услуги в YCLIENTS. Текущий эндпойнт `GET /services/{company}` отдаёт лишь одно фото, поэтому требуется обеспечить дочитку полной галереи (конкретный эндпойнт/способ — инженерное решение архитектора, §8 №10). Условие: смена эндпойнта/способа загрузки фото **не должна сломать** существующие поля каталога (название, категория, длительность, диапазон цен, описание) и дальнейший поток записи.
- **Скругления** изображений карусели и одиночного фото на детальной карточке услуги `s-service`, согласованные с радиусами интерфейса проекта (§8 №3).
- **Явная аффорданса листаемости (РЕШЕНО, §8 №2):** соседние слайды чуть выглядывают по краям (**peek, ~14px**) + сохранённые точки-индикаторы. Без стрелок (приложение мобильное, свайп — основной жест; peek — самый сильный сигнал «листается» и реализуется чистым CSS). Счётчик «N / 8» — необязателен.
- **Зацикливание** карусели: свайп/переход вправо с последнего фото ведёт на первое, влево с первого — на последнее (бесконечная прокрутка в обе стороны). Способ реализации — рекомендация архитектору (§8 №1), не блокер.
- **Свежесть фото из YCLIENTS (РЕШЕНО, §8 №7):** стратегия **stale-while-revalidate** для хоста фото — кэш показывается мгновенно, в фоне дотягивается свежее и кладётся в кэш, обновлённое фото появляется при следующем открытии карточки. Не ломает оффлайн, экономнее по трафику, чем network-first. Изменение стратегии кэширования — **точечно для хоста фото услуг YCLIENTS `assets.yclients.com`** (пути `main_service_image/...` и `gallery_service_image/...`, §8 №8), не затрагивая APP_SHELL, JS-модули и прочие CDN (EmailJS, jsdelivr).
- **Синхронизация индикатора** текущего фото с активным слайдом — сохраняется и корректно работает при зацикливании.
- Сохранение существующего поведения: **1 фото** — одиночное изображение без листалки; **0 фото** — категорийный плейсхолдер (градиент + эмодзи). Скругления применяются и к одиночному фото, и (по §8 №4) к плейсхолдеру.
- Корректная работа на тач-устройствах (свайп), без ошибок в консоли; экранирование URL/данных через `esc()` сохраняется.

### Вне рамок (explicitly out of scope)

- **Изменения за пределами детальной карточки услуги `s-service`**: каталог услуг (`s-services`, обложка карточки), выбор мастера, расчёт слотов, бронирование, подтверждение — не затрагиваются.
- **Изменение состава детальной карточки** (название, описание, цена-диапазон, длительность, кнопка «Выбрать мастера») сверх доработки галереи — определено ADR-013, не пересматривается.
- **Зум / полноэкранный просмотр / лайтбокс** фото — вне объёма (как и в BA-014 §8 №4); добавляется отдельным решением при необходимости.
- **Автопрокрутка** карусели по таймеру — вне объёма (см. §8 №6).
- **Наполнение галереи услуг фотографиями в админке YCLIENTS** — отдельная контентная задача владельца студии (BA-014 §8 №1, ADR-013); PWA отображает то, что фактически лежит в галерее услуги. Карусель «включается сама» при N>1 фото. (При этом получение всей галереи приложением — **в рамках** этой задачи, см. выше.)
- **Глобальная смена стратегии кэширования** Service Worker для всех кросс-origin ресурсов — меняется только для хоста фото YCLIENTS `assets.yclients.com`.
- **Перенос захардкоженного YCLIENTS-токена** из `prototype.html` (известный техдолг — «не усугублять»).

## 3. Действующие лица и сценарии использования

**Клиент студии** — пользователь PWA, открывший детальную карточку услуги в рамках сценария записи. Цель — рассмотреть фото работ, чтобы оценить услугу перед записью.

Основной сценарий:

1. Клиент открывает детальную карточку услуги `s-service` (тапом по услуге в каталоге, обычный поток).
2. Видит галерею со **всеми** фото работ услуги (до 8), а не с одним: изображения со скруглёнными краями; по краю выглядывает соседнее фото (peek) и есть точки-индикаторы — сразу понятно, что фото несколько и их можно листать.
3. Свайпом (тач) листает фото; индикатор текущего фото синхронизирован с активным слайдом.
4. Долистав до последнего фото, продолжает свайп в ту же сторону — попадает на первое (и наоборот) — листание не «упирается».

Граничные сценарии:

- **Одно фото.** Показывается одиночное изображение со скруглением, без листалки, стрелок, точек и зацикливания.
- **Нет фото.** Показывается категорийный плейсхолдер (градиент + эмодзи) со скруглением, без листалки.
- **Два фото.** Карусель с зацикливанием и аффордансой работает так же, как при большем числе (граничный минимум для карусели — 2).
- **Битый URL фото.** Слайд деградирует на категорийный плейсхолдер без поломки вёрстки (как уже сделано, `services.js:169`).
- **Студия заменила фото в YCLIENTS.** Клиент, у которого приложение уже открыто/установлено, при следующем открытии этой карточки (или после фонового обновления) видит новое фото без переустановки и ручной чистки кэша; ранее просмотренные карточки остаются доступны оффлайн.
- **Оффлайн.** Ранее виденные фото доступны из кэша; новые/изменённые подтянутся при появлении сети.

## 4. Функциональные требования

- **ФТ-0.** Каталог и детальная карточка услуги получают **все** фото услуги (до 8) из галереи YCLIENTS, а не одно. Источником служит галерея услуги в YCLIENTS; конкретный эндпойнт/способ дочитки — задача архитектора (§8 №10). Получение полной галереи **не должно сломать** существующие поля каталога (название, категория, длительность, диапазон цен, описание) и дальнейший поток записи (выбор мастера, слоты, бронирование).
- **ФТ-1.** На детальной карточке услуги `s-service` изображения галереи (слайды карусели и одиночное фото) отображаются со **скруглёнными краями**, согласованными с радиусами интерфейса проекта (§8 №3).
- **ФТ-2.** При **двух и более** фото галерея визуально считывается как листаемая карусель: соседние слайды частично выглядывают по краям (**peek, ~14px**), плюс сохранены точки-индикаторы. Стрелки не добавляются. Подсказка (peek + точки) присутствует и при ровно двух фото.
- **ФТ-3.** Карусель **зациклена**: переход/свайп вправо с последнего фото ведёт на первое, влево с первого — на последнее; листание не упирается на краях ни в одну сторону.
- **ФТ-4.** Свайп/прокрутка перелистывает фото; **индикатор** текущего фото (точки и/или счётчик) синхронизирован с фактически активным слайдом, в том числе при зацикленном переходе через край.
- **ФТ-5.** При **одном** фото отображается одиночное скруглённое изображение **без** листалки, стрелок, точек и зацикливания (поведение сохранено).
- **ФТ-6.** При **отсутствии** фото отображается категорийный плейсхолдер (градиент + эмодзи) без листалки; скругление плейсхолдера — согласно §8 №4.
- **ФТ-7.** При ошибке загрузки фото (битый/недоступный URL) слайд/изображение деградирует на категорийный плейсхолдер без поломки вёрстки (сохранение текущего `onerror`-поведения).
- **ФТ-8.** Новое или изменённое фото услуги, загруженное в YCLIENTS, **появляется у клиента автоматически** — без закрытия/переустановки приложения и без ручной чистки кэша. Целевое поведение — **stale-while-revalidate** (§8 №7): закэшированное фото показывается мгновенно, свежее дотягивается в фоне и кладётся в кэш, обновлённое фото появляется **при следующем открытии** карточки. Мгновенное обновление в текущем открытом просмотре не требуется.
- **ФТ-9.** Ранее просмотренные фото услуг остаются доступны для просмотра **оффлайн**; стратегия stale-while-revalidate не ломает оффлайн-просмотр ранее виденных карточек (кэш — основной источник показа, обновление — фоновое).
- **ФТ-10.** Изменение стратегии кэширования применяется **только** к хосту фото услуг YCLIENTS `assets.yclients.com` (пути `main_service_image/...` и `gallery_service_image/...`); кэширование APP_SHELL, JS-модулей, прочих CDN (EmailJS, jsdelivr) и общий оффлайн-режим не изменяются.
- **ФТ-11.** Управление с указателя (десктоп/мышь): **кликабельные стрелки не добавляются** (§8 №2). Перелистывание — свайпом/скроллом; точки-индикаторы при текущем решении остаются индикатором без обязательной кликабельности. Если кликабельность точек реализуется, она не мешает свайпу на тач.
- **ФТ-12.** На **открытие каждой карточки** доработка не выполняет дополнительных запросов к API за списком фото сверх существующих — список фото берётся из уже загруженного объекта услуги (как в ADR-013). Получение полной галереи (ФТ-0) выполняется в рамках существующей разовой загрузки каталога при старте, без поштучных запросов на каждую карточку; стратегия кэширования меняется только для **байтов** изображений.
- **ФТ-13.** Все пользовательские тексты (если добавляются — например, счётчик «3 / 8», подписи стрелок/aria-метки) — на русском языке, в едином тоне приложения.

## 5. Нефункциональные требования

- **НФТ-1 (Совместимость стека).** Решение — vanilla JS/HTML/CSS, код в `docs/` (`prototype.html`, `docs/modules/*.js`, `docs/style.css`, `docs/sw.js`); статический хостинг GitHub Pages; **без новых рантайм-npm-зависимостей**. Зацикливание реализуется без сторонних карусель-библиотек.
- **НФТ-2 (Производительность).** До 8 фото, `loading="lazy"` сохраняется; перелистывание/свайп без тяжёлых перерисовок и без рывков; синхронизация индикатора не вызывает заметных лагов при свайпе.
- **НФТ-3 (Деградация).** При отсутствии/ошибке фото галерея остаётся консистентной (плейсхолдер вместо фото); карусель деградирует по числу фото (0 → плейсхолдер, 1 → одиночное, N>1 → карусель) без поломки вёрстки.
- **НФТ-4 (PWA/офлайн не регрессируют).** Устанавливаемость PWA и оффлайн-поведение Service Worker не ухудшаются; при изменении файлов `APP_SHELL` (включая `docs/sw.js`, `style.css`, `services.js`, `prototype.html`) поднимается `CACHE_VERSION` в `docs/sw.js` (правило проекта).
- **НФТ-5 (Минимальность изменений).** Изменения строго в пределах детальной карточки `s-service` и точечной правки кэш-стратегии для хоста фото; каталог, выбор мастера, слоты, бронирование, прочие ветки кэширования не рефакторятся.
- **НФТ-6 (Безопасность / XSS).** URL фото и название услуги (`alt`) выводятся только через `esc()`; вставка непроверенного HTML недопустима.
- **НФТ-7 (Качество фото).** `image_group` отдаёт ресайз ≈420×280; при peek и скруглениях изображение масштабируется без видимого размытия/критичной обрезки в пределах макета (формат/высота — §8 №5).
- **НФТ-8 (Трафик / экономия данных).** Выбранная стратегия свежести не должна приводить к неоправданной повторной загрузке неизменившихся фото при каждом открытии (баланс свежесть/оффлайн/трафик — §8 №7); приоритет — показать кэш мгновенно, обновлять в фоне, если иное не решит заказчик.
- **НФТ-9 (Локализация и доступность).** Добавляемые тексты — на русском; интерактивные элементы управления (стрелки/точки, если есть) имеют понятные текстовые/aria-метки; фото сохраняют `alt` (название услуги).

## 6. Ограничения и допущения

**Ограничения:**

- Релевантный код (для информации; реализацию определяет архитектор):
  - Карусель и галерея — `docs/modules/services.js:160-216` (`openServiceCard`): рендер слайдов/точек, слушатель `scroll` для активной точки (`services.js:204-213`).
  - Стили галереи/карусели — `docs/style.css:809-849` (`.svc-card-gallery`, `.svc-card-single-img`, `.svc-card-placeholder`, `.svc-carousel*`); слайд `flex:0 0 100%` (нет peek), фото `height:280px; object-fit:cover`, без скруглений.
  - Service Worker — `docs/sw.js`: ветка не-same-origin CDN — **cache-first без ревалидации** (`sw.js:108-118`); `NETWORK_ONLY_HOSTS=['api.yclients.com']` (`sw.js:43`); JS — network-first (`sw.js:83-92`); shell — cache-first + фоновое обновление (`sw.js:94-105`).
  - Механизм обновления приложения — `docs/app.js:294-313` (`reg.update()` по таймеру и `visibilitychange`, `skipWaiting`, `controllerchange` → `reload`); он освежает APP_SHELL/JS при смене `CACHE_VERSION`, но **не** освежает уже закэшированные байты кросс-origin фото в `RUNTIME_CACHE`.
- **Текущий эндпойнт каталога схлопывает галерею.** `GET /services/{company}` (`docs/app.js:163`) возвращает `image_group.images` объектом `{basic}` (одно фото). Полную галерею массивом отдают (выбор источника — за архитектором, §8 №10):
  - `GET /company/{company}/services/` (management) → `image_group.images` массивом объектов с `.path` (хелпер `serviceImageUrls()`, `docs/modules/serviceImages.js:29-39`, уже корректно обрабатывает массивы);
  - `GET /book_services/{company}` (публичный booking) → готовое поле `images[]` (массив URL-строк) + `image` (одиночный основной); набор полей отличается (длительность — `seance_length` в секундах, поля `image_group` нет) — при выборе этого источника нужен дополнительный маппинг полей.
- **Подтверждённый хост байтов фото** — `assets.yclients.com`, пути `main_service_image/...` (основное фото) и `gallery_service_image/...` (галерейные). Имя файла содержит таймстемп (например, `..._20260529180821.png`), то есть при перезаливке URL, вероятно, меняется. Stale-while-revalidate корректно покрывает оба случая — и стабильный URL, и меняющийся.
- В проекте **нет** единой CSS-переменной `--radius`; используются конкретные значения (кнопки/инпуты — 14px, карточки — 16px, и др.). Радиус скругления фото нужно выбрать согласованным с этим (§8 №3).
- Vanilla JS/HTML/CSS, без рантайм-npm-зависимостей; запрет коммита секретов (включая YCLIENTS-токен); запрет любых операций с веткой `backup`; `--force` в `main` запрещён.
- При изменении файлов `APP_SHELL` — поднятие `CACHE_VERSION` в `docs/sw.js`.
- Сохранить экранирование внешних данных через `esc()` при любой переработке разметки.

**Допущения:**

- Список URL фото услуги приходит свежим из API (`api.yclients.com`, `NETWORK_ONLY`) и сохраняется в объекте услуги (ADR-013, поле `photos`); проблема свежести — в кэшировании **байтов** изображения по URL, а не списка. Подтверждено, что имя файла фото содержит таймстемп и URL при перезаливке, вероятно, меняется; stale-while-revalidate выбрана как стратегия, покрывающая и случай меняющегося URL, и случай сохранения того же URL.
- Несколько фото на услугу **достижимы** через галерею в админке YCLIENTS (подтверждено на услуге id 29196117 — 3 фото). Карусель работает на нормализованном массиве URL произвольной длины (ADR-013, хелпер `serviceImageUrls`): 0 → плейсхолдер, 1 → одиночное, N>1 → карусель — эта логика переиспользуется при условии, что в `photos` действительно попадает вся галерея (ФТ-0).
- Существующий нативный scroll-snap не позволяет «честное» зацикливание без JS — потребуется доработка механизма карусели (конкретный подход — рекомендация архитектору, §8 №1, не блокер).

## 7. Критерии приёмки

- **КП-0.** Услуга, у которой в галерее YCLIENTS несколько фото (контрольный пример — id 29196117, 3 фото), отображается на детальной карточке со всеми этими фото (карусель), а не с одним; при этом название, категория, длительность, диапазон цен, описание и дальнейший поток записи остаются корректными. (ФТ-0)
- **КП-1.** На детальной карточке услуги изображения галереи (слайды и одиночное фото) имеют скруглённые края, согласованные с радиусами интерфейса (§8 №3). (ФТ-1)
- **КП-2.** При 2+ фото визуально понятно, что это листаемая карусель: соседние слайды выглядывают по краям (peek, ~14px) и присутствуют точки-индикаторы; стрелок нет; подсказка присутствует и при ровно двух фото. (ФТ-2)
- **КП-3.** Свайп/переход с последнего фото ведёт на первое и с первого на последнее (зацикливание в обе стороны); листание не упирается на краях. (ФТ-3)
- **КП-4.** Индикатор текущего фото синхронизирован с активным слайдом, в том числе при зацикленном переходе через край. (ФТ-4)
- **КП-5.** При одном фото — одиночное скруглённое изображение без листалки/стрелок/точек/зацикливания; при отсутствии фото — категорийный плейсхолдер без листалки. (ФТ-5, ФТ-6)
- **КП-6.** При битом URL фото слайд/изображение деградирует на категорийный плейсхолдер без поломки вёрстки. (ФТ-7, НФТ-3)
- **КП-7.** Новое/изменённое фото услуги из YCLIENTS появляется у клиента автоматически — без закрытия/переустановки приложения и без ручной чистки кэша. По stale-while-revalidate: при наличии кэша фото показывается мгновенно, свежее подтягивается в фоне, обновление видно при следующем открытии карточки. (ФТ-8)
- **КП-8.** Ранее виденные фото доступны оффлайн; оффлайн-просмотр ранее открытых карточек не сломан stale-while-revalidate. (ФТ-9)
- **КП-9.** Изменённая стратегия кэширования применена только к хосту фото YCLIENTS `assets.yclients.com` (пути `main_service_image/...`, `gallery_service_image/...`); кэширование APP_SHELL, JS-модулей, прочих CDN (EmailJS, jsdelivr) и общий оффлайн-режим не изменены. (ФТ-10)
- **КП-10.** Кликабельные стрелки отсутствуют; перелистывание работает свайпом/скроллом. Если кликабельность точек реализована — она не мешает свайпу на тач. (ФТ-11)
- **КП-11.** Доработка не добавляет запросов за списком фото на каждое открытие карточки сверх существующих; список фото — из уже загруженного объекта услуги. (ФТ-12)
- **КП-12.** Карусель и зацикливание работают на тач-устройствах без ошибок в консоли; до 8 фото `loading="lazy"`, свайп без рывков. (НФТ-2, общий критерий)
- **КП-13.** URL фото и `alt` экранируются (`esc()`); инъекция HTML невозможна. (НФТ-6)
- **КП-14.** PWA остаётся устанавливаемой, оффлайн-поведение не регрессирует; при изменении файлов `APP_SHELL`/`sw.js` поднята `CACHE_VERSION`; новые рантайм-зависимости не добавлены. (НФТ-1, НФТ-4)
- **КП-15.** Добавленные пользовательские тексты — на русском языке; интерактивные элементы имеют текстовые/aria-метки. (ФТ-13, НФТ-9)
- **КП-16.** Каталог услуг (название, категория, длительность, диапазон цен, описание), выбор мастера, расчёт слотов, бронирование и подтверждение работают как прежде — в том числе после возможной смены источника/эндпойнта получения галереи (ФТ-0). (НФТ-5, ФТ-0)

## 8. Открытые вопросы

> **Все блокирующие вопросы РЕШЕНЫ** (пользователь + проверка живого API YCLIENTS компании 1940850, 2026-05-29). Решения ниже помечены **РЕШЕНО** с зафиксированным ответом; исходный контекст сохранён для прослеживаемости. Оставшиеся пункты — несрочные значения по умолчанию и инженерные развилки для архитектора, **дальнейшую работу (/adr) не блокируют**.

1. **Способ реализации зацикливания — инженерная развилка для архитектора (не блокирует BA).** Нативный scroll-snap «честно» зациклить нельзя. Варианты: (а) JS-карусель на `transform: translateX` с клонами крайних слайдов для бесшовного цикла; (б) при достижении края мгновенно переставлять `scrollLeft` на противоположный край в обработчике `scroll`/`scrollend` (риск визуального «скачка»); (в) полноценная собственная JS-лента на трансформах. **Рекомендация BA (к решению архитектора в ADR):** выбрать минимальный по коду вариант без рантайм-зависимостей, при котором переход через край визуально бесшовен и сохраняется плавный нативный/инерционный свайп на тач (приоритет — отсутствие заметного «скачка» и плавность на мобильном). Окончательный выбор — за архитектором. *Влияет на ФТ-3, ФТ-4, НФТ-1, НФТ-2.*

2. **Набор аффордансы листаемости — РЕШЕНО.** Был блокирующим UX-вопросом. **Решение:** peek — соседние слайды чуть выглядывают по краям (**~14px**) + сохранить существующие точки-индикаторы. **Без стрелок** (приложение мобильное, свайп — основной жест; peek — самый сильный сигнал «листается» и реализуется чистым CSS: ширина слайда < 100% + gap, без JS). Счётчик «N / 8» — **не обязателен**. Это снимает и подвопрос про «ровно два фото»: peek + точки применяются единообразно при N ≥ 2. *Влияет на ФТ-2, ФТ-11, КП-2, КП-10.*

3. **Радиус скругления — значение по умолчанию (заказчик может скорректировать).** В проекте нет переменной `--radius`; карточки используют 16px. **Предложение BA:** радиус фото галереи ≈ 16px (под стиль карточек), применять к каждому слайду/изображению (а не к контейнеру), чтобы при peek у соседнего фото тоже были видны скруглённые края; плейсхолдер скруглять так же. Точное значение и объект скругления (слайд vs контейнер) — на усмотрение реализации в этих рамках. *Влияет на ФТ-1, ФТ-6, КП-1, КП-5.*

4. **Скругление плейсхолдера и одиночного фото — значение по умолчанию.** Скруглять одиночное фото и категорийный плейсхолдер тем же радиусом, что и слайды карусели (визуальная консистентность). *Влияет на ФТ-5, ФТ-6, КП-5.*

5. **Высота/формат фото — значение по умолчанию.** Сохранить высоту ≈280px и `object-fit: cover`. При введении peek слайд становится < 100% ширины — проверить при реализации, что фото не мылит и не обрезает критично в новой геометрии. Если визуально неприемлемо — эскалировать (возможен иной aspect-ratio/высота). *Влияет на НФТ-7.*

6. **Автопрокрутка — значение по умолчанию: не нужна.** Смена слайдов — только по жесту/клику пользователя, без таймера (исходное требование склоняется к этому). *Влияет на scope, НФТ-2.*

7. **Момент обновления фото (свежесть vs трафик/оффлайн) — РЕШЕНО.** Был блокирующим продуктово-архитектурным вопросом. **Решение: stale-while-revalidate** для хоста фото — кэш показывается мгновенно, в фоне дотягивается свежее и кладётся в кэш, обновлённое фото появляется **при следующем открытии** карточки. Мгновенное обновление в текущем открытом просмотре **не требуется**. Стратегия не ломает оффлайн и экономнее network-first по трафику. (Отклонённые альтернативы: network-first — дороже по трафику/медленнее на слабой сети; cache-busting — избыточен, т.к. SWR покрывает и меняющийся, и стабильный URL; TTL — менее предсказуем.) Архитектор фиксирует SWR в ADR. *Влияет на ФТ-8, ФТ-9, ФТ-10, НФТ-8, КП-7, КП-8.*

8. **Идентификация хоста фото YCLIENTS — РЕШЕНО.** Был блокирующим для реализации кэш-стратегии. **Подтверждено на живых данных:** байты фото услуг отдаются с `assets.yclients.com`, пути `main_service_image/...` (основное фото) и `gallery_service_image/...` (галерейные). Правило свежести (SWR) в `sw.js` применять **точечно к этому хосту**, не задевая EmailJS/jsdelivr/APP_SHELL/JS-модули. Нюанс: имя файла содержит таймстемп (`..._20260529180821.png`) — при перезаливке URL, вероятно, меняется; SWR закрывает оба случая. *Влияет на ФТ-8, ФТ-10, КП-7, КП-9.*

9. **Поведение индикатора при 8 фото — значение по умолчанию.** 8 точек помещаются; **предложение BA** — оставить точки; счётчик «N / M» как дополнение необязателен (см. №2). *Влияет на ФТ-4, КП-4.*

10. **Источник/способ получения полной галереи фото — инженерная развилка для архитектора (РАЗРЕШЁН риск, не блокирует BA).** Был **ключевым риском всего требования** и теперь снят: несколько фото на услугу достижимы через галерею YCLIENTS (подтверждено на услуге id 29196117 — 3 фото). Однако текущий эндпойнт `GET /services/{company}` (`docs/app.js:163`) схлопывает галерею до одного фото. Полную галерею массивом отдают два эндпойнта (см. §6): `GET /company/{company}/services/` (`image_group.images` массивом, совместим с существующим `serviceImageUrls()`) и `GET /book_services/{company}` (поле `images[]`, но иной набор полей — длительность `seance_length` в секундах, нет `image_group`). **Решение по выбору источника и способу дочитки — за архитектором (ADR).** На уровне BA зафиксировано (ФТ-0): каталог/карточка должны получать **все** фото услуги (до 8) из галереи YCLIENTS, и смена эндпойнта/способа не должна сломать существующие поля каталога и поток записи. *Влияет на ФТ-0, КП-0, КП-16.*

**Блокирующих открытых вопросов нет — документ готов к /adr.** Противоречий с принятыми ADR не выявлено. Задача доработывает галерею детальной карточки, спроектированной в **ADR-013** (вариант E1 — карусель на CSS scroll-snap + точки): зацикливание и аффорданса (peek) расширяют вариант E1, изменение источника фото меняет/дополняет маппинг каталога, а stale-while-revalidate для `assets.yclients.com` затрагивает ветку Service Worker `sw.js:108-118` — всё это новые архитектурные решения, которые должны быть зафиксированы в новом/обновлённом ADR. Если выбранный подход к зацикливанию, к источнику фото или к кэшированию изменит решения ADR-013, это отражается в новом ADR.
