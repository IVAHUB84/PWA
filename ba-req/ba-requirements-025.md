# BA-требования 025: Запрет выделения текста и контекстного меню/перетаскивания изображений (анти-selection / анти-callout)

- **Источник:** `requirements/req-disable-text-selection-025.md`
- **Дата:** 2026-05-30
- **Статус:** черновик (блокирующих открытых вопросов нет; ключевые продуктовые развилки уже согласованы заказчиком, см. §1 и §8)

## 1. Контекст и бизнес-цель

Studio PWA — устанавливаемое приложение клиента студии красоты (онлайн-запись поверх YCLIENTS API), vanilla JS/HTML/CSS, отдаётся статически из `docs/` на GitHub Pages.

Сейчас приложение ведёт себя как обычная веб-страница, а не как нативное приложение, при долгом удержании пальца (long-press) на мобильном:

- выделяется текст (подсветка + маркеры выделения) на кнопках, карточках, чипах, метках;
- по изображениям (фото услуг, карусель, аватары мастеров, иконки) на iOS Safari всплывает нативное контекстное меню сохранения/копирования (`-webkit-touch-callout`);
- картинку можно «утащить» (drag) на десктопе.

Эффект усиливает недавно добавленный press feedback (req-023, v1.19.x): пользователь удерживает элемент управления, ожидая визуальной реакции «нажатия», а вместо этого получает системное выделение текста.

Текущее состояние кода: глобального `user-select: none` нет, правило задано лишь точечно (`.tab` — `docs/style.css:142`, `.pin-key` — `docs/style.css:819`, слайды карусели — `docs/style.css:852-856`, где также уже стоит `user-drag: none` и `pointer-events: none` на `img`). `-webkit-touch-callout` нигде не задан. Функционала копирования через `navigator.clipboard`/`execCommand` в кодовой базе нет (подтверждено поиском) — значит сейчас единственный способ скопировать текст (телефон/адрес студии, промокод) — это нативное выделение, которое и будет запрещено (см. ОВ-2 в §8).

**Бизнес-цель:** убрать «вебовость» интерфейса — запретить выделение текста по всему UI (кроме полей ввода) и отключить нативное контекстное меню/перетаскивание по изображениям, чтобы взаимодействие ощущалось как в нативном приложении, не ломая ввод в формах и функциональные жесты. Реализация — чистый CSS, без новых зависимостей.

**Стейкхолдеры:**

- Клиент студии — основной выгодоприобретатель: интерфейс перестаёт «прыгать» в выделение при удержании, фото нельзя случайно вытащить в системное меню.
- Заказчик / владелец студии — заинтересован в нативном ощущении приложения; ключевые продуктовые решения по этой задаче уже принял (см. §2, §8).
- Бизнес-аналитик / архитектор / разработчик — оформление, проектирование, реализация.

## 2. Область (scope)

### В рамках задачи

- **Глобальный запрет выделения текста** во всём UI приложения через `user-select: none` (с вендорными префиксами) на корневом контейнере приложения. Конкретный корневой селектор определяет архитектор по `prototype.html` (приложение размещено непосредственно в `<body>`, отдельной `#app`-обёртки нет — см. ОВ-1).
- **Исключение для полей ввода:** возврат выделяемости (`user-select: text` с префиксами) для `input`, `textarea` и `[contenteditable]` — чтобы сохранить ввод, выделение и вставку (PIN/OTP-коды, телефон, имя, e-mail, тексты отзывов, формы админа push). На момент оформления в `prototype.html` присутствуют `input`/`textarea`; `[contenteditable]` не используется, но добавляется в перечень исключений на будущее.
- **Отключение нативного контекстного меню (callout) на iOS** через `-webkit-touch-callout: none` — long-press по изображениям/элементам не открывает системное меню сохранения.
- **Запрет выделения и перетаскивания изображений:** для `img` — запрет drag (`-webkit-user-drag: none` / `user-drag: none`) и невозможность выделить картинку. Существующее точечное поведение на слайдах карусели (`docs/style.css:852-856`) сохраняется или обобщается без регресса.
- **Реализация — чистый CSS**, преимущественно в `docs/style.css`; точечно `docs/prototype.html` (атрибуты на `img`) — минимально, только если CSS не покрывает кейс.
- При изменении файлов из `APP_SHELL` (`style.css`, при необходимости `prototype.html`) — поднятие `CACHE_VERSION` в `docs/sw.js` (правило проекта).

### Вне рамок (explicitly out of scope)

- Любая JS-логика блокировки (перехват `contextmenu`, `selectstart`, `dragstart` обработчиками) как основное средство — базово только CSS. Точечный JS-обработчик допустим лишь как запасной вариант для конкретного движка по решению архитектора, но не входит в плановый объём.
- Введение нового функционала копирования (кнопки «Скопировать» для телефона/адреса/промокода) — не в этой задаче (см. ОВ-2: при запрете выделения теряется текущая возможность скопировать текст вручную; решение по компенсации — за заказчиком/архитектором, по умолчанию НЕ вводим).
- Защита контента от сохранения как функция безопасности/DRM — задача косметическая (нативное ощущение), а не защита от копирования; обход тривиален и не является целью.
- Ослабление запрета на десктопе через `@media`/`pointer` — по умолчанию НЕ делаем, поведение единое (см. ОВ-4).
- Изменение визуального стиля, press feedback (req-023), логики карусели — не трогаются (кроме сохранения совместимости).
- Перенос захардкоженного YCLIENTS-токена из `prototype.html` — известный техдолг, «не усугублять».
- Любые операции с веткой `backup`.

## 3. Действующие лица и сценарии использования

**Клиент студии** — пользователь PWA на мобильном (Android/iOS) или десктопе.

- **С1. Long-press по элементу управления (кнопка / карточка / чип / иконка / текстовая метка) на мобильном.** Текст НЕ выделяется (нет подсветки и маркеров); пользователь видит только press feedback элемента.
- **С2. Long-press по изображению (фото услуги, слайд карусели, аватар мастера, иконка) на iOS Safari / Android Chrome.** Нативное контекстное меню сохранения НЕ появляется; картинка не выделяется и не перетаскивается.
- **С3. Перетаскивание изображения мышью на десктопе.** Drag-«призрак» картинки не появляется, картинку нельзя «утащить».
- **С4. Ввод в поле формы** (PIN/OTP-код, телефон, имя, e-mail, текст отзыва, поля админа push). Каретка, выделение текста, копирование/вставка (в т. ч. вставка кода из SMS) работают штатно — поле осталось выделяемым.
- **С5. Функциональные жесты.** Свайп карусели фото услуг, скролл списков, тапы и press feedback работают без регрессий — глобальный `user-select: none` не блокирует pointer-события.

**Администратор студии** — пользователь админ-раздела PWA (формы push, поиск клиентов). Те же правила: интерфейс не выделяется, но все его поля ввода (`input`/`textarea`) остаются полностью работоспособными (С4).

## 4. Функциональные требования

- **ФТ-1.** Во всём UI приложения выделение текста запрещено: long-press / drag-select по любому неинтерактивному и интерактивному элементу (кнопки, карточки, чипы, метки, заголовки) НЕ приводит к выделению текста. Реализуется глобальным `user-select: none` (с вендорными префиксами `-webkit-/-moz-/-ms-`) на корневом контейнере приложения (ОВ-1).
- **ФТ-2.** Поля ввода и редактируемые элементы — `input`, `textarea`, `[contenteditable]` — сохраняют выделяемость (`user-select: text` с префиксами): в них работают каретка, выделение, копирование и вставка. Это исключение имеет приоритет над глобальным правилом из ФТ-1.
- **ФТ-3.** На iOS long-press по изображениям и элементам интерфейса НЕ открывает нативное callout-меню («Сохранить изображение», «Копировать», «Поделиться») — за счёт `-webkit-touch-callout: none` на корне (и/или явно на `img`).
- **ФТ-4.** Изображения (`img`) во всём UI нельзя перетащить (drag) — `-webkit-user-drag: none` / `user-drag: none` (и при необходимости атрибут `draggable="false"`); картинку нельзя выделить.
- **ФТ-5.** Существующее точечное поведение слайдов карусели (`pointer-events: none` и `user-drag: none` на `img`, `docs/style.css:852-856`) сохраняется без регресса; свайп карусели продолжает работать.
- **ФТ-6.** Существующие точечные `user-select: none` (`.tab`, `.pin-key`) не конфликтуют с глобальным правилом; после обобщения дублирующие точечные объявления могут быть оставлены или убраны на усмотрение архитектора без изменения наблюдаемого поведения.
- **ФТ-7.** Решение не нарушает функциональные жесты и события указателя: свайп карусели, скролл, тапы, press feedback (req-023) работают как прежде; `user-select: none` не подавляет pointer/touch-события.
- **ФТ-8.** Реализация выполняется средствами CSS без новых рантайм-npm-зависимостей; изменения преимущественно в `docs/style.css`, точечно в `docs/prototype.html` (атрибуты на `img`) — минимально.

## 5. Нефункциональные требования

- **НФТ-1 (Совместимость стека).** Решение в рамках vanilla CSS/HTML, код в `docs/`. Без новых рантайм-зависимостей.
- **НФТ-2 (Кроссплатформенность).** Запрет выделения и callout проверяется как минимум на iOS Safari и Android Chrome (мобильные движки WebKit/Blink) — это целевые платформы для проявления проблемы; вендорные префиксы (`-webkit-`, `-moz-`, `-ms-`) задаются для совместимости.
- **НФТ-3 (Отсутствие регрессий ввода).** Ни одно поле ввода или редактируемая область не теряет возможности ввода/выделения/вставки (критично: PIN/OTP, телефон, формы записи, формы админа push).
- **НФТ-4 (Производительность/стабильность).** Изменения чисто декларативные (CSS-свойства), не влияют на производительность; в консоли — ноль ошибок.
- **НФТ-5 (Минимальность изменений).** Только описанный объём; визуальный стиль, press feedback и логика карусели не меняются.
- **НФТ-6 (PWA/офлайн не регрессируют).** Устанавливаемость PWA и офлайн-поведение не ухудшаются; при изменении файлов из `APP_SHELL` поднимается `CACHE_VERSION` в `docs/sw.js`.

## 6. Ограничения и допущения

**Ограничения:**

- Стек и размещение: vanilla CSS/HTML, код в `docs/`, статический хостинг GitHub Pages; запрещены рантайм-npm-зависимости, коммит секретов, любые операции с веткой `backup`.
- Реализация — преимущественно CSS; JS-перехват событий как основное средство не используется (только запасной точечный вариант по решению архитектора).
- Не сломать ввод/выделение в полях (`input`, `textarea`, `[contenteditable]`).
- Не сломать функциональные жесты: свайп карусели (req-015/017), скролл, тапы, press feedback (req-023).
- При изменении файлов из `APP_SHELL`/`sw.js` — обязательное поднятие `CACHE_VERSION`.
- Известный техдолг (захардкоженный YCLIENTS-токен в `prototype.html`) — не усугублять.

**Допущения:**

- **Д-1.** Приложение размещено непосредственно в `<body>` (отдельной `#app`-обёртки в `prototype.html` нет); корневым селектором для глобального правила может выступить `body` или иной фактический корень — выбирает архитектор (ОВ-1).
- **Д-2.** Функционала копирования через `navigator.clipboard`/`execCommand` и кнопок «Скопировать» в кодовой базе нет (подтверждено поиском). Следовательно, запрет выделения убирает текущую (ручную) возможность скопировать телефон/адрес/промокод; по умолчанию это принимается как осознанная потеря (ОВ-2), компенсирующий функционал в этой задаче не вводится.
- **Д-3.** `[contenteditable]` сейчас в приложении не используется, но включается в перечень исключений ФТ-2 на будущее без риска регресса.
- **Д-4.** Целевая платформа — в первую очередь мобильная; единое поведение запрета на десктопе принято заказчиком как база (ОВ-4), отдельная адаптация под десктоп не требуется.

## 7. Критерии приёмки

- **КП-1.** На мобильном (эмуляция touch в DevTools + реальное устройство) long-press по кнопкам, карточкам, чипам, иконкам и текстовым меткам НЕ вызывает выделения текста. (ФТ-1)
- **КП-2.** Поля ввода (PIN/OTP-код, телефон, имя, e-mail, текст отзыва, поля админа push) сохраняют возможность ввода, выделения, копирования и вставки (в т. ч. вставку кода из SMS). (ФТ-2, НФТ-3)
- **КП-3.** Long-press по изображениям (фото услуг, слайд карусели, аватары мастеров, иконки) на iOS Safari и Android Chrome НЕ открывает нативное контекстное меню сохранения. (ФТ-3)
- **КП-4.** Изображения нельзя выделить и нельзя перетащить (drag) — на десктопе drag-«призрак» картинки не появляется. (ФТ-4)
- **КП-5.** Свайпы карусели фото услуг, скролл списков, тапы и press feedback (req-023) работают без регрессий. (ФТ-5, ФТ-7)
- **КП-6.** В консоли при взаимодействии — ноль ошибок; PWA остаётся устанавливаемой; новые рантайм-зависимости не добавлены. (НФТ-4, НФТ-6)
- **КП-7.** При изменении файлов из `APP_SHELL`/`sw.js` поднята `CACHE_VERSION` в `docs/sw.js`. (НФТ-6)

## 8. Открытые вопросы

Блокирующих оформление настоящего документа вопросов **нет**. Ключевые продуктовые развилки уже согласованы заказчиком (см. §«Целевое поведение» и §«Открытые вопросы» исходного требования). Ниже — те же пункты с зафиксированным решением «по умолчанию»; **ОВ-2** рекомендуется явно подтвердить заказчику перед `/adr` (есть осознанная потеря функциональности).

- **ОВ-1. Корневой селектор для глобального правила (техническое, не блокирует).** На что вешать `user-select: none` — на `body` или конкретный корневой контейнер. *Решение:* приложение размещено в `<body>` без отдельной `#app`-обёртки; фактический корень определяет архитектор по `prototype.html`. *Влияет на ФТ-1.*
- **ОВ-2. Полнота исключений / потеря копирования (рекомендуется подтвердить заказчику).** Заказчик выбрал «весь UI, кроме полей ввода» — дополнительных исключений для выделения по умолчанию НЕ делаем. **Важно:** функционала копирования (`clipboard`/кнопок «Скопировать») в приложении нет, поэтому запрет выделения убирает единственный текущий способ скопировать **телефон и адрес студии и промокоды вручную**. *Решение по умолчанию:* принимаем потерю, компенсирующий функционал в этой задаче не вводим. *Рекомендация BA:* подтвердить у заказчика, что ручное копирование контактов/промокода действительно не требуется; если требуется — это отдельная задача (кнопка «Скопировать» либо точечное исключение `user-select: text` на конкретных текстовых блоках), вне рамок req-025. *Влияет на §2 «Вне рамок», ФТ-1.*
- **ОВ-3. Поля PIN/OTP и формы (контроль, не блокирует).** Экран ввода PIN/OTP (`input.otp-box` × 4, `prototype.html:406-409`) и все формы должны остаться работоспособными (каретка, выделение, вставка кода из SMS). *Решение:* покрывается исключением ФТ-2; обязательно проверяется в КП-2.
- **ОВ-4. Десктоп — единое или ослабленное поведение (согласовано).** *Решение заказчика (база):* единое поведение, запрет выделения действует и на десктопе, без усложнения через `@media`/`pointer`. Ослабление вынесено «вне рамок» (§2). *Влияет на §2 «Вне рамок».*

**Противоречий с принятыми ADR не выявлено.** Задача — косметическая CSS-полировка ощущения нативности; затрагиваемые механизмы (`user-select`, `-webkit-touch-callout`, `user-drag`, существующее поведение карусели и press feedback) не пересекаются с архитектурными решениями принятых ADR. Если архитектор сочтёт нужным добавить класс/атрибут в `prototype.html` или обобщить точечные стили карусели/`.tab`/`.pin-key` — это решается в рамках данного документа без выхода за его границы.
