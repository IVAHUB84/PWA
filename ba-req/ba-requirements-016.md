# BA-требования 016: Колокольчик уведомлений на главном экране — инбокс, счётчик непрочитанных, очистка истории

- **Источник:** `requirements/req-home-notification-bell-016.md`
- **Дата:** 2026-05-29
- **Статус:** черновик

## 1. Контекст и бизнес-цель

Studio PWA (студия красоты «Реснички, сестрички», онлайн-запись поверх YCLIENTS) — устанавливаемое статическое приложение, отдаётся из `docs/` на GitHub Pages; стек vanilla JS/HTML/CSS без рантайм-зависимостей. Push-инфраструктура существует: клиентский `docs/modules/push.js`, бэкенд — Cloudflare Worker (`cf-worker/src/index.js`, KV `SUBSCRIPTIONS`, ключ устройства `sub:${clientId}:${endpointHash}` — ADR-006), который шлёт админские рассылки (`handleSend`) и автоматические напоминания о визитах (`handleCron`).

**Подтверждено кодом:** приходящие push-уведомления **нигде не сохраняются**. SW-обработчик `push` (`docs/sw.js:158-170`) только показывает системное уведомление ОС (`showNotification`) и не персистит его содержимое. Обработчик `notificationclick` (`docs/sw.js:172-182`) при тапе по системному уведомлению лишь фокусирует/открывает приложение. Внутри приложения есть тост `#pushNotif` (`docs/prototype.html:25`), тап по которому (`handlePushTap()`, `docs/modules/notifications.js:11-14`) ведёт в `go('s-history','tab')` — но `s-history` — это «История записей» (визитов), а не история уведомлений (`docs/prototype.html:227-235`). Отдельного экрана истории уведомлений нет. IndexedDB в проекте не используется; состояние хранится в `localStorage` (`docs/modules/storage.js`, `state.js`), который **недоступен из Service Worker**, тогда как push прилетает именно в SW, в том числе при закрытом приложении.

В результате клиент не может посмотреть, что ему приходило: уведомление живёт только как системное оповещение ОС и теряется после закрытия. Это снижает ценность push-канала (акции, напоминания) и доверие к нему.

**Бизнес-цель:** дать клиенту единое место — «инбокс» уведомлений на главном экране — где он видит **все** полученные уведомления (включая пришедшие при закрытом приложении), с понятным индикатором непрочитанных на колокольчике и возможностью очистить историю. Это повышает информированность клиента (он не пропустит акцию/напоминание), удержание и воспринимаемую ценность push-канала.

**Стейкхолдеры:**

- Заказчик / владелец студии — отправляет push (акции, объявления, напоминания); заинтересован, чтобы клиент мог вернуться к полученным сообщениям и не пропускал их.
- Клиент студии — пользователь PWA, получатель уведомлений; хочет видеть историю и понимать, что есть непрочитанное.
- Бизнес-аналитик / архитектор / разработчик — оформление, проектирование, реализация.

## 2. Область (scope)

### В рамках задачи

- **Колокольчик в шапке главного экрана `s-home`**, в правом верхнем углу строки логотипа/названия (свободное место, `docs/prototype.html:38-44`), не конфликтуя со строкой поиска ниже (`#homeSearchBar`, `docs/prototype.html:46-52`).
- **Бейдж непрочитанных** на колокольчике (визуально — справа внизу): кружок-счётчик количества непрочитанных уведомлений; формат при больших числах ограничивается (например, «9+», §8 №11).
- **Инбокс уведомлений** — экран/представление со списком **всех** полученных уведомлений (новые сверху), каждое с заголовком, текстом и датой/временем; доступен по тапу на колокольчик; есть возврат назад.
- **Сохранение истории уведомлений локально на устройстве** так, чтобы в инбокс попадали **все** уведомления, **включая пришедшие при закрытом приложении** (push прилетает в SW). Конкретный механизм персистентности (где и как хранить, как писать из SW) — техническое решение архитектора (§8 №1); на уровне BA фиксируется требование полноты истории, а не способ его достижения.
- **Логика «прочитано»**: при открытии инбокса все непрочитанные уведомления помечаются прочитанными, и бейдж-счётчик обнуляется.
- **«Очистить историю»** — действие на экране инбокса, удаляющее все записи истории, с подтверждением перед очисткой и понятным пустым состоянием после.
- **Пустое состояние** инбокса (когда уведомлений нет / после очистки) — понятный текст (опц. эмодзи/иллюстрация в стиле проекта).
- **Оффлайн-просмотр** ранее полученных уведомлений (читаются из локального хранилища без сети).
- **Экранирование** заголовка и тела уведомления через `esc()` при рендере в инбоксе (контент приходит извне — admin push).
- **Сохранение текущего поведения push:** показ системного уведомления (`docs/sw.js:169`), `notificationclick` (`docs/sw.js:172-182`), внутриприложенческий тост `#pushNotif` и подписка (`docs/modules/push.js`) продолжают работать.

### Вне рамок (explicitly out of scope)

- **Колокольчик/инбокс на других экранах** кроме `s-home` (выбор услуги, профиль, история визитов и т. д.) — по требованию только главный экран (§8 №10).
- **Синхронизация счётчика с системным значком приложения** (Badging API / `navigator.setAppBadge`) — вне рамок; бейдж только визуальный внутри приложения (§8 №3).
- **Удаление отдельного уведомления** (свайп/кнопка на элементе) — вне рамок; «Очистить историю» работает целиком (§8 №5).
- **Действия/переходы по тапу на конкретное уведомление** (deep-link в услугу/запись) — вне рамок этого релиза; элемент инбокса информационный (§8 №2).
- **Серверная история уведомлений / синхронизация между устройствами** (единый журнал на Worker, чтение истории с сервера) — вне рамок; история ведётся локально на устройстве (§8 №1). Если архитектор выберет серверное хранилище как механизм персистентности — это его решение в ADR, но как продуктовая функция «история на сервере / синхронизация устройств» в объём не входит.
- **Изменение состава, семантики и доставки push** (`handleSend`, `handleCron`, схема подписок ADR-006, криптография Web Push, тумблеры предпочтений из BA-007) — не пересматриваются; задача только сохраняет и отображает то, что приходит.
- **Изменение экрана «История записей» `s-history`** (визиты) сверх того, что тап по тосту/уведомлению перенаправляется в новый инбокс вместо `s-history` (если такое перенаправление принимается при реализации, §8 №2).
- **Перенос захардкоженного YCLIENTS-токена** из `prototype.html` (известный техдолг — «не усугублять»).

## 3. Действующие лица и сценарии использования

**Клиент студии** — пользователь PWA на главном экране. Цель — видеть, какие уведомления ему приходили, и понимать, есть ли непрочитанное.

Основной сценарий (есть непрочитанные):

1. Клиент открывает главный экран `s-home`; в правом верхнем углу шапки виден колокольчик, на нём справа внизу — бейдж с числом непрочитанных уведомлений.
2. Тапает по колокольчику — открывается инбокс со списком всех полученных уведомлений, новые сверху, каждое с заголовком, текстом и датой/временем.
3. После открытия инбокса все уведомления помечаются прочитанными; при возврате на главный бейдж обнулён (не показывается).

Сценарий (очистка истории):

1. На экране инбокса клиент нажимает «Очистить историю».
2. Приложение запрашивает подтверждение.
3. После подтверждения список очищается; показывается понятное пустое состояние; бейдж отсутствует.

Сценарий (уведомление пришло при закрытом приложении):

1. Клиенту приходит push, когда приложение закрыто (он видит системное уведомление ОС).
2. Клиент позже открывает приложение — на колокольчике виден бейдж, а в инбоксе присутствует это уведомление (заголовок/текст/время).

Граничные сценарии:

- **Уведомлений нет** — колокольчик отображается без бейджа; инбокс показывает пустое состояние; ошибок нет.
- **Клиент не подписан на push / не дал разрешение ОС** — системные уведомления ему не приходят, но колокольчик и инбокс работают корректно (пусты, без ошибок).
- **Оффлайн** — ранее полученные уведомления доступны для просмотра из локального хранилища.
- **Много уведомлений** — список не «раздувает» хранилище сверх разумного лимита (§8 №6); число на бейдже отображается компактно при больших значениях (§8 №11).
- **Повторная доставка одного уведомления** — дубли в инбоксе не накапливаются (дедупликация, §8 №9).

## 4. Функциональные требования

- **ФТ-1.** На главном экране `s-home`, в правом верхнем углу строки логотипа/названия шапки, отображается иконка-колокольчик; она не перекрывает и не ломает существующую строку поиска ниже.
- **ФТ-2.** При наличии непрочитанных уведомлений на колокольчике (визуально справа внизу) отображается бейдж-счётчик с количеством непрочитанных; при отсутствии непрочитанных бейдж не отображается.
- **ФТ-3.** Тап по колокольчику открывает инбокс — представление со списком **всех** полученных уведомлений, отсортированных «новые сверху»; из инбокса есть понятный возврат назад.
- **ФТ-4.** Каждый элемент инбокса отображает как минимум заголовок, текст и дату/время получения уведомления.
- **ФТ-5.** В инбокс попадают уведомления, полученные в том числе **при закрытом приложении** (push, доставленный в Service Worker), а не только полученные при открытом окне.
- **ФТ-6.** При открытии инбокса все непрочитанные уведомления помечаются прочитанными; после этого бейдж-счётчик на колокольчике обнуляется (не отображается).
- **ФТ-7.** На экране инбокса доступно действие «Очистить историю», которое после подтверждения пользователем удаляет все записи истории уведомлений.
- **ФТ-8.** После очистки истории (или когда уведомлений нет) инбокс показывает понятное пустое состояние; бейдж на колокольчике отсутствует.
- **ФТ-9.** Ранее полученные уведомления доступны для просмотра **оффлайн** (читаются из локального хранилища без обращения к сети).
- **ФТ-10.** Заголовок и текст уведомления выводятся в инбоксе только через `esc()` (контент приходит из admin push — внешний источник); вставка непроверенного HTML недопустима.
- **ФТ-11.** Уведомления в инбоксе не дублируются при повторной доставке одного и того же уведомления (дедупликация по устойчивому признаку — §8 №9).
- **ФТ-12.** Существующий поток push не ломается: показ системного уведомления (`docs/sw.js:169`), обработчик `notificationclick` (`docs/sw.js:172-182`), внутриприложенческий тост `#pushNotif` и подписка (`docs/modules/push.js`) продолжают работать как прежде.
- **ФТ-13.** Колокольчик и инбокс работают корректно (пустой список, без ошибок) и в случае, когда клиент не подписан на push или не дал разрешение ОС на уведомления.
- **ФТ-14.** При большом числе непрочитанных значение на бейдже отображается компактно (например, «9+») — без переполнения/искажения вёрстки шапки (§8 №11).
- **ФТ-15.** Все пользовательские тексты (заголовок инбокса, кнопка «Очистить историю», текст подтверждения очистки, пустое состояние, aria-метки колокольчика/бейджа) — на русском языке, в едином тоне приложения.

## 5. Нефункциональные требования

- **НФТ-1 (Совместимость стека).** Решение — vanilla JS/HTML/CSS, код в `docs/` (`prototype.html`, `docs/modules/*.js`, `docs/style.css`, при необходимости `docs/sw.js`); статический хостинг GitHub Pages; **без новых рантайм-npm-зависимостей**.
- **НФТ-2 (PWA/офлайн не регрессируют).** Устанавливаемость PWA и оффлайн-поведение Service Worker не ухудшаются; при изменении файлов `APP_SHELL` (включая `docs/sw.js`) поднимается `CACHE_VERSION` в `docs/sw.js` (правило проекта). Если для персистентности используется/меняется SW-обработчик `push` — `CACHE_VERSION` поднимается обязательно.
- **НФТ-3 (Безопасность / XSS).** Контент уведомлений (заголовок/тело), приходящий извне, выводится только через `esc()`; инъекция HTML невозможна. API-токены и секреты не хардкодятся и не логируются.
- **НФТ-4 (Целостность данных истории).** Хранилище истории не должно неограниченно расти: применяется разумный лимит (последние N записей или иной предел — §8 №6) с автоочисткой самых старых сверх лимита; переполнение хранилища не приводит к ошибкам в работе приложения.
- **НФТ-5 (Минимальность изменений).** Изменения строго в пределах объёма: колокольчик/инбокс/счётчик/история. Каталог услуг, выбор мастера, слоты, бронирование, профиль, доставка push (`handleSend`/`handleCron`, ADR-006) не рефакторятся сверх необходимого.
- **НФТ-6 (Отсутствие ошибок и тач-совместимость).** Ноль ошибок в консоли; корректная работа на тач-устройствах (тап по колокольчику, прокрутка списка, кнопка очистки).
- **НФТ-7 (Локализация и доступность).** Все добавляемые пользовательские тексты — на русском; колокольчик и бейдж имеют понятные текстовые/aria-метки; бейдж имеет достаточный контраст на градиентном хедере (§8 №11).
- **НФТ-8 (Согласованность с push-инфраструктурой).** Сохранение истории не нарушает доставку push и схему подписок ADR-006 (`sub:${clientId}:${endpointHash}`); если архитектор выберет серверную дочитку истории — она использует существующие механизмы Worker без пересмотра криптографии и формата подписок.

## 6. Ограничения и допущения

**Ограничения:**

- `localStorage` **недоступен из Service Worker**; на него нельзя полагаться для приёма и сохранения push при закрытом приложении (push приходит в SW). Это ключевое ограничение для механизма персистентности.
- Push-уведомления приходят и при закрытом приложении — именно в SW (`docs/sw.js:158-170`); полнота истории (ФТ-5) достижима только хранилищем, доступным из SW (например, IndexedDB) либо дочиткой с сервера. Конкретику определяет архитектор (§8 №1).
- Релевантный код (для информации; реализацию определяет архитектор): шапка `s-home` (`docs/prototype.html:35-53`, свободный правый верхний угол `:38-44`, строка поиска `#homeSearchBar` `:46-52`); SW push/notificationclick (`docs/sw.js:158-182`); тост `#pushNotif` (`docs/prototype.html:25`) и `docs/modules/notifications.js:11-35` (`handlePushTap`, `_sendNotification`); экран «История записей» `s-history` (`docs/prototype.html:227-235`) — это история визитов, не уведомлений; источники push — `sendAdminPush` (`docs/modules/push.js:114-129`), напоминания (`handleCron`), payload push `{ title, body, icon? }` (`docs/sw.js:159-167`); подписка `subscribePush` (`docs/modules/push.js:31-67`).
- Vanilla JS/HTML/CSS, без рантайм-npm-зависимостей; запрет коммита секретов (включая YCLIENTS-токен); запрет любых операций с веткой `backup`; `--force` в `main` запрещён. Минимальные изменения строго по объёму.
- При изменении файлов `APP_SHELL` (включая `docs/sw.js`, новые JS-модули, если попадают в shell, `prototype.html`, `style.css`) — поднятие `CACHE_VERSION` в `docs/sw.js`.
- Текущая версия: `docs/version.json` → build 26, v1.12.1.

**Допущения:**

- Push-инфраструктура (доставка, подписка, формат payload) корректна и стабильна (ADR-006); задача только сохраняет/отображает приходящие уведомления, не меняя их доставку.
- Под «уведомлением» в инбоксе понимаются push-сообщения (админские рассылки, акции/промо, напоминания о визитах) с полями `title`/`body` (+ опц. `icon`); внутренние системные события приложения в инбокс не попадают (§8 №2).
- Дефолтное состояние «прочитано» — при открытии инбокса всё помечается прочитанным целиком (требование «после просмотра инбокса счётчик обнуляется», §8 №4).
- «Очистить историю» удаляет записи (а не просто помечает прочитанными), с подтверждением (§8 №5).
- Инбокс реализуется как отдельный экран в существующей навигационной модели приложения (как прочие `s-...`) либо как накладная панель — выбор подхода не меняет объём (§8 №7); ключевое — наличие списка, бейджа и очистки.
- Колокольчик отображается на `s-home` всегда (в т. ч. без подписки на push); это не вводит в заблуждение, т. к. инбокс при отсутствии уведомлений пуст (§8 №8).

## 7. Критерии приёмки

- **КП-1.** На главном экране `s-home` в правом верхнем углу шапки отображается колокольчик; строка поиска ниже не ломается и остаётся функциональной. (ФТ-1)
- **КП-2.** При наличии непрочитанных на колокольчике справа внизу виден бейдж с их количеством; при отсутствии непрочитанных бейдж не отображается. (ФТ-2, ФТ-6)
- **КП-3.** Тап по колокольчику открывает инбокс со списком всех полученных уведомлений, новые сверху; каждый элемент показывает заголовок, текст и дату/время; из инбокса есть возврат назад. (ФТ-3, ФТ-4)
- **КП-4.** Уведомление, доставленное при **закрытом** приложении, присутствует в инбоксе после открытия приложения (и учтено в бейдже до открытия инбокса). (ФТ-5)
- **КП-5.** После открытия инбокса непрочитанные становятся прочитанными и бейдж на колокольчике обнуляется (не отображается). (ФТ-6)
- **КП-6.** На экране инбокса есть «Очистить историю»; действие требует подтверждения и после него полностью очищает список. (ФТ-7)
- **КП-7.** После очистки (или когда уведомлений нет) инбокс показывает понятное пустое состояние; бейдж отсутствует. (ФТ-8)
- **КП-8.** Ранее полученные уведомления просматриваются **оффлайн** (без сети), из локального хранилища. (ФТ-9, НФТ-2)
- **КП-9.** Заголовок и текст уведомления, содержащие HTML-спецсимволы, отображаются как текст (экранированы через `esc()`); инъекция HTML невозможна. (ФТ-10, НФТ-3)
- **КП-10.** Повторная доставка одного и того же уведомления не создаёт дубль в инбоксе. (ФТ-11)
- **КП-11.** Существующий поток push не сломан: при получении push по-прежнему показывается системное уведомление; тап по нему фокусирует/открывает приложение; внутриприложенческий тост `#pushNotif` работает; подписка/отписка работают. (ФТ-12)
- **КП-12.** В среде без разрешения на push / без подписки колокольчик и инбокс работают без ошибок (инбокс пуст, бейджа нет). (ФТ-13)
- **КП-13.** При большом числе непрочитанных бейдж отображается компактно (например, «9+») без поломки вёрстки шапки и с достаточным контрастом на градиенте. (ФТ-14, НФТ-7)
- **КП-14.** История уведомлений ограничена разумным пределом (§8 №6): при превышении старые записи вытесняются без ошибок; хранилище не растёт неограниченно. (НФТ-4)
- **КП-15.** Все добавленные пользовательские тексты — на русском; колокольчик/бейдж имеют текстовые/aria-метки. (ФТ-15, НФТ-7)
- **КП-16.** PWA остаётся устанавливаемой, оффлайн не регрессирует; при изменении файлов `APP_SHELL`/`docs/sw.js` (в т. ч. правка SW-обработчика `push` ради персистентности) поднята `CACHE_VERSION`; новые рантайм-зависимости не добавлены; секреты не закоммичены. (НФТ-1, НФТ-2, НФТ-3, НФТ-5)

## 8. Открытые вопросы

> Блокирующих открытых вопросов нет: критерий готовности требования жёстко фиксирует полноту истории («все уведомления, включая пришедшие при закрытом приложении»), что исключает «потерять закрытые» как продуктовый компромисс. Ниже — одна архитектурная развилка, делегированная архитектору, и значения по умолчанию, принятые BA для некритичных вопросов. Документ готов к `/adr`.

1. **Механизм персистентности истории — архитектурная развилка для архитектора (не блокирует BA).** `localStorage` недоступен из SW, а push прилетает в SW и при закрытом приложении. Варианты: (а) **IndexedDB** — запись в SW-обработчике `push`, чтение из приложения (доступно из SW, покрывает «закрытое приложение»); (б) только то, что получено при открытом окне через `postMessage` SW→клиент в `localStorage` — **отклонено BA как не удовлетворяющее ФТ-5/КП-4** (теряет уведомления при закрытом приложении); (в) дочитка истории с Cloudflare Worker (если хранить отправленные на сервере). **Решение и продуктовый компромисс на уровне BA:** история ведётся **локально на устройстве**, полнота (включая закрытое приложение) обязательна (ФТ-5). Вариант (а) выглядит минимальным под этот критерий без новых зависимостей и без серверной разработки; вариант (в) допустим как решение архитектора, но «серверная история / синхронизация устройств» как функция в объём не входит (раздел 2). Окончательный выбор хранилища и способа записи из SW — за архитектором (ADR). *Влияет на ФТ-5, ФТ-9, ФТ-11, НФТ-2, НФТ-8, КП-4, КП-8.*

2. **Что считать «уведомлением» и поведение по тапу на элемент — значение по умолчанию.** В инбокс попадают push-сообщения (админские рассылки/акции/промо, напоминания о визитах) с полями `title`/`body` (+ опц. `icon`/тип). Внутренние системные события приложения в инбокс **не** добавляются. Элемент инбокса информационный — без deep-link/действий по тапу в этом релизе. Текущий тап по тосту `#pushNotif`, ведущий в `s-history` (история визитов), при реализации логично перенаправить в новый инбокс — допустимо в рамках задачи, но не обязательно. *Влияет на раздел 2, ФТ-4. Может скорректировать заказчик; не блокирует.*

3. **Связь счётчика с системным значком приложения — значение по умолчанию: нет.** Бейдж только визуальный внутри приложения (на колокольчике). Синхронизация с Badging API / `navigator.setAppBadge` — вне рамок (поддержка платформ неоднородна, особенно iOS). *Влияет на раздел 2, ФТ-2.*

4. **Логика «прочитано» — значение по умолчанию: всё при открытии инбокса.** При открытии инбокса все непрочитанные помечаются прочитанными, бейдж обнуляется целиком (этого требует критерий готовности «после просмотра инбокса счётчик обнуляется»). Пометка «прочитано» по конкретному элементу — не вводится. *Влияет на ФТ-6, КП-5.*

5. **«Очистить историю» — значение по умолчанию: удаляет все записи, с подтверждением; удаление одного — вне рамок.** Действие удаляет историю целиком (не «пометить прочитанными»), с подтверждающей модалкой/диалогом перед очисткой. Свайп/удаление отдельного уведомления — вне объёма этого релиза. *Влияет на ФТ-7, ФТ-8, КП-6.*

6. **Лимит хранения — значение по умолчанию: последние N (предложение — 50), автовытеснение старых.** Чтобы хранилище не росло неограниченно, хранить ограниченный объём (например, последние 50 уведомлений), вытесняя самые старые. Точное N и/или период — на усмотрение реализации в этих рамках; при необходимости корректирует заказчик. *Влияет на НФТ-4, КП-14.*

7. **Экран vs накладная панель — значение по умолчанию: отдельный экран в существующей навигации.** Инбокс реализуется как отдельный экран `s-...` (консистентно с остальными экранами и навигационной моделью проекта) либо как накладная панель/модалка — на усмотрение реализации; обязателен понятный возврат назад. Выбор подхода не меняет объём. *Влияет на ФТ-3. Конкретику UI подтверждает заказчик/архитектор при проектировании.*

8. **Поведение для неавторизованных / без разрешения на push — значение по умолчанию: колокольчик всегда виден, инбокс пуст без ошибок.** Колокольчик показывается на `s-home` всегда; при отсутствии уведомлений инбокс пуст, ошибок нет; без подписки/разрешения новые уведомления просто не приходят. *Влияет на ФТ-13, КП-12.*

9. **Дедупликация и порядок — значение по умолчанию: новые сверху, дедуп по устойчивому признаку.** Сортировка — по времени получения, новые сверху. Дедупликация — по устойчивому идентификатору уведомления (например, комбинация времени и контента или сквозной id, если он есть/добавляется в payload). Конкретный ключ дедупликации — за реализацией. *Влияет на ФТ-11, КП-10.*

10. **Где ещё показывать колокольчик — значение по умолчанию: только `s-home`.** По требованию — главный экран. Колокольчик в шапках других вкладок — вне объёма. *Влияет на раздел 2, ФТ-1. Может расширить заказчик отдельным требованием.*

11. **Вид и контраст бейджа — значение по умолчанию.** Кружок-счётчик справа внизу колокольчика; при больших числах — «9+»; цвет/контраст подобрать читаемым на градиентном хедере. Точное оформление — на усмотрение реализации в этих рамках. *Влияет на ФТ-2, ФТ-14, НФТ-7, КП-13.*

**Непротиворечие принятым ADR.** Прямых конфликтов не выявлено. ADR-006 (мультидевайс-подписки `sub:${clientId}:${endpointHash}`, `handleSend`/`handleCron`) — фундамент доставки, который эта задача **не пересматривает**: история — клиентская надстройка над приходящими push. Если архитектор выберет серверную дочитку истории (вариант 1в), решение должно опираться на существующий Worker без изменения криптографии/формата подписок. ADR-005 (install prompt; push вне рамок того релиза) и ADR-013 (детальная карточка услуги) данной темы не затрагивают. Введение IndexedDB и/или записи из SW-обработчика `push` — **новое архитектурное решение**, которое должно быть зафиксировано в новом ADR (с подъёмом `CACHE_VERSION` при правке `docs/sw.js`/`APP_SHELL`).
