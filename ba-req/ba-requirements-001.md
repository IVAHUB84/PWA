# BA-требования 001: Визуальный каталог услуг

- **Источник:** `requirements/req-services-catalog-001.md`
- **Дата:** 2026-05-28
- **Статус:** готово

## 1. Контекст и бизнес-цель

Студия красоты «Реснички, сестрички» (направления: брови, ресницы, макияж, перманентный макияж; Троицк, Москва) принимает онлайн-записи через PWA поверх YCLIENTS. Сейчас экран выбора услуги (`s-services`) — это текстовый список карточек: название, длительность, категория, цена, кнопка «Записаться →».

Студия позиционируется в премиум-сегменте и «продаёт эстетику». Текстовый список не передаёт визуальный результат процедур и, по мнению заказчика, занижает воспринимаемую ценность услуг.

**Бизнес-цель:** повысить конверсию из просмотра каталога в начало записи за счёт визуальной подачи — карточек с фотографиями процедур/результатов, оформленных «дорого и красиво» по образцу beauty-приложений (Fresha, Booksy) и маркетплейсов (Airbnb-стиль карточек).

**Стейкхолдеры:**
- Заказчик / владелец студии — инициатор, поставщик фотоконтента, оценивает «премиальность» вида.
- Клиенты студии (женщины 20–40 лет) — конечные пользователи экрана.
- Бизнес-аналитик / архитектор / разработчик — реализация.

## 2. Область (scope)

### В рамках задачи
- Замена текстового списка услуг на экране `s-services` визуальным каталогом из карточек с крупной фотографией.
- На каждой карточке: фотография, название услуги, цена, длительность.
- Механизм привязки фотографии к услуге по **названию услуги** или по **категории** (не по ID из YCLIENTS).
- Градиентный плейсхолдер с эмодзи категории для услуг без сопоставленной фотографии.
- Сохранение существующей навигации: тап по карточке ведёт к выбору мастера (или к слотам, если мастер уже предвыбран) — поведение не меняется.
- Сохранение поиска по услугам и фильтра по категориям в новом визуальном виде.
- Определение места и формата хранения фотографий (каталог в репозитории `docs/images/` и/или внешние URL) — на уровне требования, без выбора технической реализации.

### Вне рамок (explicitly out of scope)
- Изменение логики выбора услуги, перехода к мастеру и к выбору слота.
- Изменение источника данных об услугах (YCLIENTS API) и набора полей услуги (название, цена, длительность, категория).
- Изменение других экранов (мастера, слоты, профиль, лента, главная).
- Административный интерфейс/CMS для загрузки и привязки фотографий силами сотрудников студии.
- Подбор и закупка финального фотоконтента (контент поставляет заказчик; для старта допустимы стоковые фото).
- Привязка фотографий к конкретным мастерам.

## 3. Действующие лица и сценарии использования

**Клиент студии** — заходит в PWA, открывает экран записи/услуг.

Основной сценарий:
1. Клиент попадает на экран `s-services` и видит каталог карточек услуг с фотографиями.
2. При желании сужает выбор: вводит запрос в поиск или выбирает категорию (горизонтальные чипы / текущий фильтр категорий).
3. Визуально оценивает услуги по фотографиям, выбирает нужную и нажимает на карточку.
4. Переходит к выбору мастера (или к выбору времени, если мастер был предвыбран).

Альтернативный сценарий (предвыбранный мастер):
- Клиент пришёл с экрана конкретного мастера; каталог показывает только услуги этого мастера; тап по карточке ведёт сразу к выбору слота. Поведение баннера мастера и фильтрации сохраняется.

Граничный сценарий (нет фотографии):
- Для услуги, к которой не сопоставлена фотография, карточка показывает градиентный плейсхолдер с эмодзи соответствующей категории, оставаясь визуально консистентной с остальным каталогом.

## 4. Функциональные требования

- **ФТ-1.** Экран `s-services` отображает услуги в виде каталога карточек с крупной фотографией вместо текстового списка.
- **ФТ-2.** Каждая карточка содержит: фотографию (или плейсхолдер), название услуги, цену, длительность.
- **ФТ-3.** Тап по карточке услуги выбирает услугу и инициирует переход по той же логике, что и сейчас: к выбору мастера, а при предвыбранном мастере — к выбору слота. Логика выбора и переходов не меняется.
- **ФТ-4.** Фотография сопоставляется услуге по названию услуги или по её категории. Сопоставление не должно зависеть от ID услуги из YCLIENTS (ID считаются нестабильными).
- **ФТ-5.** Если для услуги не найдена фотография, карточка отображает градиентный плейсхолдер с эмодзи, соответствующим категории услуги.
- **ФТ-6.** На экране сохраняется поиск по услугам: ввод запроса фильтрует каталог по названию (и категории — как в текущем поведении вне режима предвыбранного мастера).
- **ФТ-7.** На экране сохраняется фильтр по категориям; выбранная категория сужает каталог до услуг этой категории.
- **ФТ-8.** При пустом результате поиска/фильтрации экран показывает понятное состояние «ничего не найдено» (как сейчас).
- **ФТ-9.** Источник и набор полей услуги остаются прежними (название, цена, длительность, категория из YCLIENTS); фотография — дополнительный визуальный слой поверх этих данных.
- **ФТ-10.** Каталог сопоставления «услуга/категория → фотография» должен быть редактируемым силами разработки без изменения логики экрана (добавление новой услуги или фото не должно требовать переписывания кода рендера).

## 5. Нефункциональные требования

- **НФТ-1 (Визуальное качество).** Каталог должен соответствовать премиум-восприятию: единый стиль карточек, аккуратные отступы, консистентность фотографий и плейсхолдеров. Финальная приёмка вида — за заказчиком.
- **НФТ-2 (Производительность).** Загрузка фотографий не должна заметно замедлять открытие экрана и прокрутку на мобильном устройстве; изображения должны быть оптимизированы по весу/размеру и подгружаться без блокировки взаимодействия (приёмлемое восприятие на типовом мобильном соединении).
- **НФТ-3 (Деградация).** При недоступности или ошибке загрузки фотографии карточка должна корректно показывать плейсхолдер, не «ломая» вёрстку.
- **НФТ-4 (Совместимость).** Решение остаётся в рамках стека проекта: vanilla JS/HTML/CSS, статический хостинг из `docs/` на GitHub Pages, без новых рантайм-зависимостей.
- **НФТ-5 (PWA/офлайн).** Поведение Service Worker и офлайн-режима не должно регрессировать; необходимость кэширования фотографий и влияние на `CACHE_VERSION` — вопрос проектирования (см. «Открытые вопросы»).
- **НФТ-6 (Локализация).** Интерфейс — на русском языке, в едином тоне с существующими экранами.
- **НФТ-7 (Доступность контента).** Фотографии должны иметь текстовую альтернативу (название услуги), чтобы карточка оставалась понятной при недоступности изображения.

## 6. Ограничения и допущения

**Ограничения:**
- Приложение — статический PWA в `docs/`, обслуживается GitHub Pages; бэкенд отсутствует, кроме внешнего YCLIENTS API.
- Запрещены рантайм-npm-зависимости (только devDependencies).
- При изменении состава кэшируемых ресурсов (APP_SHELL) требуется поднятие `CACHE_VERSION` в `docs/sw.js`.
- ID услуг из YCLIENTS считаются изменяемыми и не могут служить ключом привязки фотографий.

**Допущения:**
- Заказчик предоставит фотоконтент (или согласует стоковые фото) до релиза; на этапе разработки допустимы временные изображения.
- Набор категорий услуг ограничен и известен (Брови, Губы, Глаза, Эпиляция и т. п.), что позволяет задать эмодзи-плейсхолдер на каждую категорию.
- Названия услуг в YCLIENTS относительно стабильны и пригодны для сопоставления; редкие расхождения покрываются привязкой по категории и плейсхолдером.
- Текущая навигация и состояние (`state`, режим предвыбранного мастера, баннер мастера) остаются актуальными и переиспользуются.

## 7. Критерии приёмки

- **КП-1.** На экране `s-services` услуги отображаются карточками с фотографией; текстовый список более не используется. (ФТ-1, ФТ-2)
- **КП-2.** Каждая карточка показывает фотографию (или плейсхолдер), название, цену и длительность услуги. (ФТ-2, ФТ-9)
- **КП-3.** Тап по карточке услуги ведёт к выбору мастера, а в режиме предвыбранного мастера — к выбору слота, без изменения прежней логики. (ФТ-3)
- **КП-4.** Для услуги без сопоставленного фото отображается градиентный плейсхолдер с эмодзи её категории, и карточка визуально не отличается по структуре от карточек с фото. (ФТ-5, НФТ-3)
- **КП-5.** Фотография сопоставляется по названию или категории услуги; смена ID услуги в YCLIENTS не нарушает привязку. (ФТ-4)
- **КП-6.** Поиск по услугам работает и фильтрует каталог карточек. (ФТ-6)
- **КП-7.** Фильтр по категориям работает и сужает каталог карточек до выбранной категории. (ФТ-7)
- **КП-8.** При отсутствии результатов поиска/фильтра показывается состояние «ничего не найдено». (ФТ-8)
- **КП-9.** Добавление новой услуги или новой привязки фотографии не требует изменения логики рендера экрана. (ФТ-10)
- **КП-10.** Изображения оптимизированы и не блокируют отрисовку/прокрутку; при ошибке загрузки фото показывается плейсхолдер без поломки вёрстки. (НФТ-2, НФТ-3)
- **КП-11.** PWA остаётся устанавливаемой, офлайн-поведение не регрессирует, стек не пополняется рантайм-зависимостями. (НФТ-4, НФТ-5)
- **КП-12.** Заказчик подтверждает соответствие вида каталога премиум-восприятию. (НФТ-1)

## 8. Открытые вопросы

1. **Хранение фотографий.** ✅ *Решено заказчиком:* хранить в S3-совместимом хранилище (внешние URL). Конкретный провайдер и credentials — на усмотрение архитектора.
2. **Поставка и объём контента.** Кто и в какие сроки предоставит фотографии? Требуется ли единый визуальный стандарт (кадрирование, соотношение сторон)? *(Некритично для реализации — для старта допустимы временные фото.)*
3. **Приоритет привязки.** ✅ *Решено заказчиком:* привязка по **точному названию услуги** — приоритет выше привязки по категории. Нормализацию (регистр, пробелы) определяет архитектор.
4. **Эмодзи-плейсхолдеры по категориям.** Нужно зафиксировать соответствие «категория → эмодзи и градиент» для всех актуальных категорий. *(Определяет разработчик/архитектор по существующим данным.)*
5. **Категорийные чипы vs текущий фильтр.** ✅ *Решено заказчиком:* оставить текущий механизм фильтра без изменений.
6. **Офлайн/Service Worker.** ✅ *Решено:* кэширование фотографий — на усмотрение архитектора.

Примечание: противоречий с существующими ADR не выявлено (каталог `adrs/` не содержит принятых решений по данной теме).
