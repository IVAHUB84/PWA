Баги — обнаружены и исправлены
В ходе текущей сессии
BUG-7
runScenario затирал window.history
Строка history = [startScreen] перезаписывала нативный window.history. Исправлено: переименовано в navHistory.
Исправлено
BUG-8
back() не вызывал onEnterScreen
При нажатии «Назад» не обновлялись данные экрана (список мастеров, история, профиль). Добавлен вызов onEnterScreen(prev) в setTimeout после анимации.
Исправлено
BUG-2
Маркетинговые согласия pre-checked (нарушение 152-ФЗ)
Все 4 чекбокса в экране согласий имели класс on. Исправлено: "Акции", "Фото/видео", "Отзывы", "Оферта" теперь unchecked по умолчанию.
Исправлено
BUG-4
registerClient() не проверял r.success
При ошибке создания клиента в YCLIENTS API (400/403/409) пользователь всё равно получал сессию с client_id: null. Добавлена проверка с показом ошибки.
Исправлено
BUG-12
_findClientByEmail() возвращала первого клиента из БД
YCLIENTS API игнорирует параметр ?email= и возвращает всех клиентов. r.data[0] мог быть чужим клиентом. Добавлена клиентская фильтрация: r.data.find(c => c.email === email).
Исправлено
BUG-13
Существующие клиенты видели экран согласий при каждом входе
После OTP при нахождении клиента в YCLIENTS шёл переход на s-consent. Исправлено: существующие клиенты переходят на s-home. Согласия — только при первой регистрации.
Исправлено
BUG-5
confirmCancel() обновлял UI до проверки успеха API
Запись удалялась из localStorage и пользователь уходил на главный экран, потом получал alert об ошибке. Исправлено: при ошибке API пользователь остаётся на экране отмены, localStorage не меняется.
Исправлено
BUG-6
Верификация отмены использовала /record/ вместо /records/
GET-запрос проверки статуса после DELETE использовал неверный эндпоинт (/record/{company}/{id} вместо /records/{company}/{id}). Исправлено.

Критичные нерешённые проблемы
MVP-функции, требующие доработки
AUTH
Вход по email вместо телефона (отклонение от F2.1)
Требование: вход по номеру телефона + SMS. Реализовано: вход по email + OTP через EmailJS. YCLIENTS использует телефон как основной идентификатор, что создаёт несоответствие при поиске клиентов.
Требует решения
PUSH
Реальные push-уведомления не реализованы (F6.1–F6.4)
Service Worker не подписан на push. Нет серверного компонента для планирования и отправки уведомлений. Все push в демо — имитация. Для реализации нужен backend (Cloudflare Workers / Firebase / суперсервер).
Критично для MVP
RENT
Отсутствует multi-tenant архитектура (F0.1–F0.3)
Данные одного салона захардкожены. Для SaaS нужен: конфиг-сервер или URL-параметры, изоляция данных, динамический манифест.
Критично для платформы
MOVE
Нет переноса записи (F3.5)
Клиент может только отменить запись. Перенос — обязательная функция (требование MVP). Нужен отдельный флоу: выбор новой даты/времени без отмены.

Итоговая оценка
Готовность к запуску первых клиентов
Базовый флоу записи — работает
Клиент может зайти по email, найти услугу, выбрать мастера, выбрать слот, создать запись, увидеть её в истории, отменить. 8 критичных багов исправлены в этой сессии.
Для реального запуска не хватает:
  • Push-уведомления — напоминания о визитах (основная ценность продукта)
  • Multi-tenant — сейчас один жёстко прописанный салон
  • Перенос записи — клиент может только отменить
  • Авторизация по телефону — требование YCLIENTS и 152-ФЗ
Оценка готовности: ~55% MVP. Подходит для демонстрации инвесторам и ранних клиентам как интерактивный прототип. Для production-запуска требуется реализация push-инфраструктуры и multi-tenant архитектуры.
